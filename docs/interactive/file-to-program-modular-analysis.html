File → Program Analysis

How KindScript aggregates file-level analysis into program-level results today, how JastAdd / Silver / Infer solve this problem, and what our options are for introducing a file-level evaluation boundary.

KindScript JastAdd Silver Facebook Infer

The Core Tension

Many analyses need cross-file information (a kind defined in file A is used in file B). But whole-program evaluation has costs: latency, memory, and tight coupling.

W

Whole-Program

Build one tree, evaluate everything at once. Simple and correct, but every file change re-evaluates the world. Can't serve single-file IDE queries without paying full-program cost.

KindScript today

M

Modular / File-Level

Evaluate each file independently, producing a summary. Compose summaries for program-level results. File-level results are cacheable, parallelisable, and independently useful.

Goal

KindScript's kind-checking analysis has a natural file-level boundary that isn't being exploited:

  • Kind definitions (kindDefs) are per-CompilationUnit — each file's definitions are independent
  • Local violations — checking a variable against its annotation only requires the kind definition, not the entire tree
  • IDE use case — when editing one file, we only need that file's violations plus the global definition environment
  • Watch mode — when one file changes, re-evaluating all files is wasted work
  • Large projects — 100+ files means 100× wasted re-evaluation for each keystroke

KindScript Today: Whole-Program Evaluation

The complete data flow from N source files to program-level diagnostics.

1. TypeScript Parse

ts.createProgram(files) — TypeScript parses all N source files into a single ts.Program.

2. AST Translation

tsToAstTranslatorAdapter.convert(tsProgram) — iterates every source file, converts each to a CompilationUnit, wraps them all in a single KSProgram root.

function buildKSTree(tsProgram: ts.Program): KSTree {
  const compilationUnits: KSCompilationUnit[] = [];
  for (const sf of tsProgram.getSourceFiles()) {
    if (sf.isDeclarationFile) continue;
    compilationUnits.push(convertSourceFile(ctx, sf));  // per-file
  }
  const root: KSProgram = {
    kind: 'Program', compilationUnits, children: compilationUnits,
  };
  return { root };
}

3. Build AG Tree (No File Boundary)

evaluator.buildTree(root) — walks the entire KSProgram recursively, wrapping every AST node in an AGNode with lazy attribute caching. No file-level gates.

4. Evaluate Attributes

evaluator.evaluate(root) — triggers evaluation of projections on the root AGNode, which demand-drives the full attribute computation across all files.

5. Projections Aggregate

Projections iterate all CompilationUnit children of the root to collect definitions and violations across the whole program.

The Bottleneck

The inherited attribute defEnv is computed at the Program root by iterating all CompilationUnits' kindDefs. This single equation creates a whole-program dependency: every node that needs to look up a kind definition transitively depends on every file's definitions.

// eq_defEnv_root — the cross-file bottleneck
function eq_defEnv_root(ctx: Ctx): Map<string, KindDefinition> {
  const map = new Map();
  for (const cuCtx of ctx.children) {          // ALL CompilationUnits
    for (const def of cuCtx.attr('kindDefs')) { // from every file
      map.set(def.name, def);
    }
  }
  return map;  // inherited down to every node in every file
}

How Others Solve This

Three systems, three fundamentally different strategies for crossing file boundaries.

Strategy: Demand-Driven Evaluation + Reference Attributes

JastAdd builds one unified AST for the whole program, like KindScript. But attributes are evaluated lazily on demand with fine-grained caching and incremental re-evaluation.

In JastAdd, no attribute is computed until someone asks for it. After parsing, the AST is completely unevaluated — no attribute values exist, no rewrites have been applied. When node.kindDefs() is called, the evaluator:

  1. Checks the node's attribute cache — if already computed, return it
  2. Executes the equation, which may trigger demand for other attributes
  3. Caches the result on the node

This means if you only query violations for file A, attributes in files B–Z may never be computed at all. The evaluation boundary emerges naturally from the query pattern. There is zero cost for defining attributes that are never accessed — enabling rich specifications (hundreds of attributes) where only a subset is relevant to any given query.

// JastAdd-style (Java)
// Attribute is only computed when accessed
syn lazy List<KindDef> CompilationUnit.kindDefs() {
  List<KindDef> defs = new List<>();
  for (Statement s : getStatements()) {
    if (s instanceof TypeAlias) defs.add(extractKindDef(s));
  }
  return defs;
}

// Inherited attribute — only computed for nodes that ask
inh Map<String, KindDef> ASTNode.defEnv();
eq Program.getChild().defEnv() {
  // This only runs when a child actually accesses defEnv()
  Map<String, KindDef> env = new HashMap<>();
  for (CompilationUnit cu : getCompilationUnits()) {
    for (KindDef d : cu.kindDefs()) env.put(d.name(), d);
  }
  return env;
}

Key insight: KindScript's AGNode already does lazy caching (via _cache in the AGNode class). The difference is that KindScript's evaluate() eagerly triggers projections on the root, which demand-drives the entire tree. JastAdd would let you call cuNode.attr('kindDefs') without triggering cross-file computation.

In ExtendJ (JastAdd's Java compiler), library classes are not parsed upfront. Instead, they are loaded lazily using Nonterminal Attributes (NTAs) — attributes whose values are new AST subtrees:

// Program has a parameterized NTA for library classes
syn lazy CompilationUnit Program.getLibCompilationUnit(String typeName) {
  // Only parsed when first demanded during type resolution
  return parseFromClasspath(typeName);
}

// Type lookup first checks source, then triggers NTA for libraries
syn TypeDecl Program.lookupType(String pkg, String name) {
  TypeDecl d = lookupSourceType(pkg, name);     // source CUs
  if (d != null) return d;
  return lookupLibraryType(pkg, name);           // triggers NTA on demand
}

This means the tree grows on demand. If a library class is never referenced, its CompilationUnit is never loaded or evaluated. The tree structure is determined by what the analysis needs, not by what exists on disk.

For KindScript: We could use a similar pattern — build CompilationUnit nodes lazily as cross-file references are resolved, rather than converting all files upfront.

JastAdd's distinctive feature is reference attributes — attributes whose values are pointers to other AST nodes. This enables cross-file name resolution without whole-tree inherited attributes.

// Instead of passing a Map down via inheritance...
syn KindDef Identifier.lookupKind(String name) {
  // Reference attribute: returns a pointer to a node elsewhere in the tree
  return program().globalScope().lookup(name);
}

// The reference attribute creates a direct edge in the dependency graph
// Only the specific definition node is depended on, not the entire defEnv

Reference attributes make the dependency graph sparse. Changing a definition in file A only invalidates attributes in other files that referenced that specific definition, not all attributes that use any definition.

JastAdd tracks fine-grained dependencies between attribute instances. When a file changes:

  1. The changed subtree's cached attributes are invalidated
  2. Attributes in other subtrees that depended on the invalidated ones are also invalidated (transitively)
  3. On next access, only the invalidated attributes are recomputed

This is fine-grained incremental evaluation: the unit of re-evaluation is a single attribute instance, not a file or a tree. It requires runtime dependency tracking (which JastAdd does automatically).

File A changes:
  ┌─────────────┐
  │ CU(A)       │ ← invalidate: kindDefs, allViolations
  │  └─ TypeDef │
  └─────────────┘
          │ depended on by
          ▼
  ┌──────────────┐
  │ Program root │ ← invalidate: defEnv (aggregates kindDefs)
  └──────────────┘
          │ inherited by
          ▼
  ┌─────────────┐
  │ CU(B)       │ ← invalidate: only nodes using defEnv for A's kinds
  │  └─ VarDecl │ ← if @SafeID and SafeID was in file A → recompute
  │  └─ VarDecl │ ← if @OtherKind (defined in C) → cache still valid!
  └─────────────┘

JastAdd's collection attributes accumulate contributions from anywhere in the tree. They're declared with a root, an initial value, and a combining operation. Any node can contribute to any collection attribute.

coll Set<Diagnostic> Program.allDiagnostics()
  [new HashSet<>()]    // initial value
  with addAll;          // combining operation

// Any node can contribute
Identifier contributes violations()
  when !violations().isEmpty()
  to Program.allDiagnostics();

This is similar to KindScript's collection attributes, but with one key difference: JastAdd collection attributes are demand-driven. They aren't computed until accessed, and contributions are gathered lazily.

JastAdd supports Circular Reference Attributed Grammars (CRAGs) — attributes that can reference themselves directly or indirectly. These are evaluated via fixed-point iteration:

// Liveness analysis — inherently circular
syn Set<String> Stmt.liveVars() circular [emptySet()];
eq Stmt.liveVars() = use().union(succ().liveVars().minus(def()));

// Evaluation:
// 1. Initialize to bottom value (emptySet())
// 2. Evaluate equation — may re-encounter liveVars → use current iteration value
// 3. Compare new value to previous — if CHANGED, iterate again
// 4. Repeat until fixed point (no change on any attribute in the cycle)

Requirements: Values must form a lattice of finite height with monotonic equations. Common pattern: set-valued lattices with empty set as bottom and union as the combining operation.

For KindScript: Currently not needed (no circular dependencies in kind-checking), but relevant if future analyses add data-flow properties that cross file boundaries.

Strategy: Modular Well-Definedness + Higher-Order Attributes + Forwarding

Silver focuses on modular extensibility — ensuring that independently-developed language extensions compose safely. Its approach to modularity is about grammar composition, not file-level analysis boundaries.

Silver's key innovation is forwarding: a production can "forward" to another tree, providing default attribute values. This allows extensions to add new syntax without defining every attribute — unhandled attributes fall through to the forwarded-to tree.

-- Silver syntax
-- A new "enhanced for" loop forwards to a regular for loop
abstract production enhancedFor
top::Stmt ::= var::Name collection::Expr body::Stmt
{
  -- Forward to desugared form
  forwards to
    regularFor(
      declareIterator(collection),
      hasNext(iterator),
      next(iterator, var),
      body
    );

  -- Attributes not explicitly defined here are computed
  -- on the forwarded-to tree instead
  top.prettyPrint = "for (" ++ var ++ " : " ++ collection ++ ") " ++ body;
}

This is about modular language composition, not file-level analysis. But the principle is relevant: forwarding defines an abstraction boundary where detailed implementation is hidden behind a simpler interface.

Silver's MDA guarantees that independently-developed extensions compose safely. Each extension is checked in isolation against a modular analysis that ensures:

  • Completeness: every attribute has an equation on every production (via forwarding defaults)
  • Non-interference: extensions don't break each other's attribute definitions
  • Type safety: composed grammars remain well-typed

This is a compile-time guarantee about grammar composition. The analogy to file-level analysis: MDA ensures that "file-level" grammar modules can be composed without whole-program analysis of the grammar itself.

Silver supports higher-order attributes — attributes whose values are themselves decorated ASTs. This allows tree transformations to be expressed as attributes:

-- A higher-order attribute that builds a transformed tree
syn Decorated Expr with {env, errors} transformedExpr =
  desugar(originalExpr);

-- The transformed tree can have its own attributes evaluated
-- This creates a "virtual" tree boundary

Higher-order attributes enable a form of file-level isolation: a CompilationUnit could have a higher-order attribute that produces a summary tree — a simplified version of the file that only exposes its public interface. Other files would analyze this summary tree instead of the full file.

Silver's collection attributes (contribution system) are similar to JastAdd's but with static checking. Contributions must be declared with an explicit contributes to clause that is checked at compile time.

-- Declare a collection on the root
synthesized attribute allErrors :: [Error] with ++ ;

-- Any production can contribute
aspect production variableDecl
top::Stmt ::= name::String expr::Expr
{
  top.allErrors <- checkType(name, expr);  -- contributes via <-
}

The static checking of contributions means Silver knows at compile time which productions contribute to which collections — enabling potential per-file pre-computation.

Critical Distinction: Specification vs Program Modularity

Silver's modularity is about the AG specification being modular — grammar modules, attributes, and productions can be developed independently and composed safely. But once composed, evaluation is whole-program on the target AST. Silver does not provide incremental per-file evaluation of the target program.

This is a fundamentally different decomposition: Silver makes the analysis definition modularly composable while maintaining whole-program analysis execution. For per-file incremental evaluation of the target program, JastAdd's incremental evaluation or Infer's summary architecture are more directly relevant.

Strategy: Compositional Summaries via Separation Logic

Infer takes a radically different approach: analyze each procedure independently, producing a summary (precondition/postcondition pair). Summaries compose bottom-up — no unified tree is ever built.

Infer's foundational technique is bi-abduction: given a call foo(x), Infer simultaneously infers:

  • Anti-frame: what additional resources the caller must provide (precondition)
  • Frame: what resources the caller retains unchanged (frame rule)
// Analyzing bar() which calls foo():
void foo(int* p) { *p = 42; }      // Summary: {p↦_} foo() {p↦42}

void bar(int* p, int* q) {
  foo(p);        // Apply foo's summary:
  *q = 10;       //   Need: p↦_ (from foo's pre)
}                //   Have: p↦_, q↦_ (from bar's params)
                 //   Frame: q↦_ (untouched by foo)
                 // Bar's summary: {p↦_, q↦_} bar() {p↦42, q↦10}

The key property: to analyze bar(), Infer only needs foo()'s summary, not its implementation. If foo's implementation changes but its summary doesn't, bar doesn't need re-analysis.

Each procedure produces a summary — a compact description of its observable behavior:

┌──────────────────────────────────────────────────┐
│  Procedure Summary                                │
│                                                    │
│  Precondition:   {x↦v * v > 0}                   │
│  Postcondition:  {x↦(v+1) * return↦v}            │
│  Errors:         [NullDeref if x=null]            │
│                                                    │
│  ✓ Complete description of observable behavior     │
│  ✓ Implementation details are forgotten            │
│  ✓ Can be computed once, cached, and reused        │
│  ✓ Invalidated only when implementation changes    │
└──────────────────────────────────────────────────┘

File-level summary = union of procedure summaries:
┌──────────────────────────────────────────────────┐
│  File: utils.ts                                   │
│  Exports: foo(summary₁), bar(summary₂)           │
│  Internal: helper(summary₃) — not exported        │
│  File-level errors: [...]                         │
└──────────────────────────────────────────────────┘

This is the most decoupled approach: files don't share an AST at all. Each file is a self-contained analysis unit. Cross-file dependencies are mediated entirely through summaries.

Infer's --reactive mode (and earlier --diff) achieves incrementality through summary caching:

  1. Maintain a database of procedure summaries from the last full analysis
  2. When files change, identify changed procedures
  3. Re-analyze changed procedures, producing new summaries
  4. Identify procedures that called changed procedures (reverse dependency)
  5. Re-analyze those callers using the new summaries
  6. Repeat until a fixed point (summaries stabilize)

The granularity is procedure-level, but file-level works too: a file's "summary" is the union of its procedure summaries. If no public summary changes, no downstream re-analysis is needed.

O'Hearn's "Continuous Reasoning" (LICS 2018) revealed a striking insight about Infer's deployment at Meta:

Batch mode (overnight runs, emailing results): 0% fix rate

Diff-time mode (commenting on code reviews as they happen): 70%+ fix rate

Same analysis, radically different impact. Compositionality enables diff-time deployment.

The continuous reasoning loop: developer submits a diff → Infer runs incrementally (leveraging cached summaries for unchanged code) → reports regressions as code review comments → developer fixes before landing. File-level summaries make this fast enough to run on every diff.

For KindScript: This is directly relevant. Kind-checking results delivered at edit-time (in IDE) are vastly more impactful than results from a full program build. File-level evaluation is the enabler.

The mathematical foundation is separation logic's frame rule:

If   {P} C {Q}   holds    (C transforms P into Q)
Then {P * R} C {Q * R}    (irrelevant state R is preserved)

The * (separating conjunction) means the states are disjoint — they don't interfere. This is what makes summaries composable: a procedure's summary describes only the heap it touches, and the frame rule guarantees it doesn't affect anything else.

For KindScript: The analog would be that a file's "kind-checking summary" (definitions exported, violations found) is separable from other files' internals. The only shared state is the definition environment — and that's essentially a read-only, monotonically-growing set.

Our Options

Four approaches for introducing file-level evaluation in KindScript, ordered from least to most disruptive.

Option A: Two-Phase Evaluation (Infer-inspired)
Evaluate each file independently in Phase 1, then compose file summaries into a program view in Phase 2.
Recommended

Split evaluation into two explicit phases:

Phase 1: File-Level (parallel, cacheable)
═══════════════════════════════════════════
  For each CompilationUnit independently:
    ┌─────────────────────────────────────────────┐
    │  Input: CompilationUnit AST                 │
    │                                             │
    │  Compute:                                   │
    │    • kindDefs      (synthesized, local)      │
    │    • kindAnnotations (synthesized, local)    │
    │    • nodeCount     (collection, local)       │
    │                                             │
    │  Output: FileSummary {                       │
    │    fileName: string                         │
    │    exportedDefs: KindDefinition[]            │
    │    localAnnotations: KindAnnotation[]        │
    │    nodeCount: number                        │
    │  }                                          │
    └─────────────────────────────────────────────┘

Phase 2: Program-Level (lightweight composition)
═══════════════════════════════════════════
  Input: FileSummary[] from all files
    ┌─────────────────────────────────────────────┐
    │  1. Build defEnv from all summaries         │
    │     defEnv = union(summaries.map(s =>       │
    │       s.exportedDefs))                      │
    │                                             │
    │  2. For each file, evaluate with defEnv:    │
    │     • contextFor   (needs defEnv)           │
    │     • violationFor (needs contextFor)       │
    │     • allViolations (aggregates above)      │
    │                                             │
    │  3. Compose: program diagnostics =          │
    │     flatMap(files, f => f.violations)       │
    └─────────────────────────────────────────────┘

What changes in the codebase:

// New: FileSummary type (in libs/analysis/ or libs/evaluator/)
interface FileSummary<K extends string, P> {
  fileName: string;
  exportedDefs: KindDefinition[];
  localAnnotations: Map<string, KindAnnotation>;
  nodeCount: number;
  // ... other file-local synthesized results
}

// New: evaluateFile() — runs Phase 1 on a single CompilationUnit
function evaluateFile(cu: ASTNode, config: EvaluatorConfig): FileSummary;

// Modified: evaluate() — now runs both phases
function evaluate(root: ASTNode, config: EvaluatorConfig): ProgramResult {
  // Phase 1: evaluate each CU independently
  const summaries = root.children.map(cu => evaluateFile(cu, config));

  // Phase 2: build program-level environment, evaluate cross-file attrs
  const defEnv = buildDefEnv(summaries);
  const diagnostics = evaluateCrossFile(root, defEnv, config);

  return { definitions: summaries.flatMap(s => s.exportedDefs), diagnostics };
}
  • File-level results are independently useful (IDE, watch mode)
  • Phase 1 is embarrassingly parallel
  • File summaries are cacheable — skip Phase 1 for unchanged files
  • Minimal disruption to existing architecture
  • The analysis spec can declare which attrs are "file-local" vs "cross-file"
  • Phase 2 still processes all files (but only for cross-file attrs)
  • Need to classify each attribute as file-local or cross-file
  • Two evaluation modes means more testing surface
Option B: Lazy Projection Boundaries (JastAdd-inspired)
Keep the unified tree but add query-level entry points that only trigger evaluation where needed.
Low effort

KindScript's evaluator already has lazy caching (AGNode._cache). The issue is that evaluate() triggers projections on the root, which demand-drives everything. The fix: expose file-level query points.

// New: evaluateFile() — builds tree but only queries one CU's attributes
function evaluateFile(root: ASTNode, fileName: string, config: EvaluatorConfig) {
  const agTree = buildAGTree(root, config);  // still builds full tree (cheap)
  const cuNode = agTree.children.find(c => c.raw.fileName === fileName);

  // Only demand attributes on this CU — lazy caching means other
  // CUs' attributes are only computed if this CU's equations need them
  return {
    kindDefs: cuNode.attr('kindDefs'),        // file-local, fast
    violations: cuNode.attr('allViolations'),  // may trigger defEnv (cross-file)
  };
}

// Existing evaluate() still works for full program analysis
function evaluate(root: ASTNode, config: EvaluatorConfig) {
  const agTree = buildAGTree(root, config);
  return config.projections(agTree);  // triggers everything
}
  • Almost zero changes — just expose new entry points
  • Lazy caching already exists in AGNode
  • File-local attrs (kindDefs) won't trigger cross-file computation
  • Cross-file attrs (violationFor via defEnv) still trigger whole-tree evaluation
  • No parallelism — evaluation is single-threaded
  • No caching across re-evaluations (tree is rebuilt each time)
  • Good first step that can evolve into Option A
Option C: Summary Attributes (Silver-inspired)
Add a new attribute direction — "summary" — that computes a per-CompilationUnit digest for cross-file consumption.
Elegant

Introduce a new AttrDecl direction, summary, that acts like a "higher-order synthesized attribute" at file boundaries. A summary attribute is synthesized on CompilationUnit nodes and provides the interface that cross-file attributes consume.

// New AttrDecl variant
interface SummaryAttr<K> extends BaseAttrDecl<K> {
  direction: 'summary';
  // Only defined on rootKind (CompilationUnit)
  equation: (ctx: Ctx) => T;
  // How summaries compose at the program level
  compose: (summaries: T[]) => ProgramLevelT;
}

// In the analysis spec:
const fileSummary: SummaryAttr<TSNodeKind> = {
  name: 'fileSummary',
  direction: 'summary',
  type: 'FileSummary',
  equation: (ctx: KindCtx<KSCompilationUnit>) => ({
    defs: ctx.attr('kindDefs'),
    annotations: ctx.attr('kindAnnotations'),
  }),
  compose: (summaries: FileSummary[]) => {
    const defEnv = new Map();
    for (const s of summaries) {
      for (const d of s.defs) defEnv.set(d.name, d);
    }
    return { defEnv };
  },
};

// The evaluator knows about summary attributes:
// 1. Evaluate file-local attrs + summary on each CU
// 2. Compose summaries
// 3. Use composed result as the "inherited" context for cross-file attrs
  • Explicit, declarative file-boundary in the analysis spec
  • The compose function is a natural abstraction barrier
  • Analysis spec authors choose what crosses the file boundary
  • Fits naturally into the existing AttrDecl discriminated union
  • New concept for analysis authors to learn
  • More machinery in the evaluator to handle summary evaluation order
  • Need to decide: does summary replace inh root values, or coexist?
Option D: Incremental Attribute Tracking (JastAdd-full)
Add runtime dependency tracking to AGNode so attributes can be incrementally re-evaluated when the tree changes.
High effort

Modify AGNode to track which attribute instances depend on which other attribute instances. When a file changes, invalidate only the affected attributes and recompute on next access.

class AGNode {
  private _cache = new Map<string, unknown>();
  private _deps = new Map<string, Set<{node: AGNode, attr: string}>>();

  attr(name: string, param?: unknown): unknown {
    const key = cacheKey(name, param);
    if (this._cache.has(key) && !this._invalid.has(key)) {
      return this._cache.get(key);
    }
    // Track which attributes are being computed (call stack)
    const caller = evaluationStack.peek();
    if (caller) {
      caller.deps.add({ node: this, attr: key });
    }
    evaluationStack.push({ node: this, attr: key, deps: new Set() });
    const result = this._dispatch(name, param);
    const frame = evaluationStack.pop();
    this._deps.set(key, frame.deps);
    this._cache.set(key, result);
    this._invalid.delete(key);
    return result;
  }

  invalidate(attr: string) {
    this._invalid.add(attr);
    // Transitively invalidate dependents
    for (const [key, deps] of this._deps) {
      if (deps.has({ node: this, attr })) {
        this.invalidate(key);
      }
    }
  }
}
  • True fine-grained incrementality — optimal re-evaluation
  • No need to classify attributes as file-local vs cross-file
  • Works for any tree change, not just file-level
  • Significant implementation complexity
  • Runtime overhead for dependency tracking (memory + time)
  • Need tree-diffing or tree-patching to update the AG tree
  • Circular attribute handling becomes much more complex
  • Overkill for KindScript's current needs
★

Recommended Path: B → A → C

Step 1 (Option B): Add file-level query entry points to the evaluator. Near-zero effort, gives IDE and watch mode a way to query single files. File-local attributes (kindDefs) skip cross-file computation entirely.
Step 2 (Option A): Introduce two-phase evaluation. Phase 1 computes file-local attributes and produces FileSummary objects. Phase 2 composes summaries into the program-level defEnv and evaluates cross-file attributes. File summaries are cacheable and Phase 1 is parallelisable.
Step 3 (Option C): If the two-phase split proves valuable, formalize it as a summary attribute direction in the analysis spec DSL. This makes the file boundary explicit and declarative, and lets the evaluator optimize evaluation order automatically.

Option D (full incremental tracking) is likely overkill for KindScript. The program's definition environment is small (dozens of kinds, not thousands), so the cost of re-computing defEnv on each change is negligible compared to the cost of fine-grained dependency tracking.

Attribute Classification for Two-Phase

Each attribute in the kind-checking spec falls naturally into Phase 1 (file-local) or Phase 2 (cross-file).

Attribute Direction Phase Rationale
kindDefs syn Phase 1 Only computed on CompilationUnit — purely file-local synthesis
kindAnnotations syn Phase 1 Per VariableDeclaration — only reads the annotation, not defEnv
nodeCount collection Phase 1 Purely structural — 1 + children, no cross-file deps
defEnv inh Phase 2 Root equation aggregates all files' kindDefs — the cross-file bridge
defLookup syn Phase 2 Closure over defEnv — depends on cross-file definitions
contextFor inh (param) Phase 2 Needs kindAnnotations (Phase 1) + defEnv (Phase 2) for lookup
violationFor syn (param) Phase 2 Checks node against contextFor — depends on cross-file context
allViolations syn Phase 2 Aggregates violationFor — transitively cross-file

Key observation: The Phase 1 → Phase 2 boundary is exactly at defEnv. Phase 1 attributes are synthesized bottom-up within each file. Phase 2 attributes depend on the composed definition environment. This is a natural seam — the two-phase split doesn't require rethinking any equations, just controlling evaluation order.

The two-phase approach generalizes if we make the phase boundary declarative:

interface AnalysisSpec<K, P> {
  attrs: AttrDecl<K>[];

  // NEW: declare which attributes form the file summary
  fileSummary?: {
    attrs: string[];   // e.g., ['kindDefs', 'kindAnnotations', 'nodeCount']
    compose: (summaries: FileSummary[]) => CrossFileContext;
  };

  projections: { ... };
}

Different analyses would declare different file summaries. A complexity analysis might export functionComplexity per-file, then compose into a program-wide complexity report. A module-dependency analysis might export imports and exports per-file.

Review

Click each card to reveal the answer.

What is the single equation in KindScript that creates the whole-program dependency?
Click to reveal
eq_defEnv_root — the inherited attribute equation that iterates all CompilationUnits' kindDefs to build a global Map<string, KindDefinition>. Every node that looks up a kind definition transitively depends on every file's definitions through this equation.
How does JastAdd avoid whole-tree evaluation without file-level boundaries?
Click to reveal
Demand-driven (lazy) evaluation — attributes are only computed when accessed. If you query violations for one file, only the attributes actually needed for that query are computed. Combined with reference attributes that create sparse cross-tree dependencies, most of the tree's attributes may never be evaluated.
What mathematical principle enables Facebook Infer's compositional file-level analysis?
Click to reveal
The frame rule from separation logic: if {P} C {Q} holds, then {P * R} C {Q * R}. This means a procedure's summary captures everything about its behavior, and unrelated state (the "frame" R) is guaranteed untouched. Summaries compose because they describe disjoint portions of the program state.
In the two-phase approach, which KindScript attributes belong to Phase 1 (file-local)?
Click to reveal
kindDefs (synthesized per CompilationUnit), kindAnnotations (synthesized per VariableDeclaration), and nodeCount (collection, purely structural). These have no dependencies on cross-file information and can be computed independently per file.
What is Silver's "forwarding" and why is it relevant to file-level analysis?
Click to reveal
Forwarding lets a production delegate to a simpler tree for unhandled attributes — it defines an abstraction boundary. For file-level analysis, the analog is a CompilationUnit that "forwards" to a summary tree: cross-file consumers see only the summary's attributes, not the full internal structure.
Why is Option D (full incremental tracking) likely overkill for KindScript?
Click to reveal
KindScript's cross-file dependency is small and simple: the definition environment is a Map of dozens of kind definitions. Re-computing this Map on each change is negligible (microseconds). Fine-grained dependency tracking adds runtime overhead (memory for dep graphs, time for tracking) that would exceed the cost of just recomputing the small cross-file state.
