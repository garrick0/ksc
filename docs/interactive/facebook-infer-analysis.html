Facebook Infer

How Infer combines Cousot's abstract interpretation, Reynolds's separation logic, O'Hearn's bi-abduction, and incorrectness logic into an industrial-scale static analyzer

Does Infer Use Cousot's Abstract Interpretation?

YES — Abstract Interpretation is a foundational pillar
PLUS — It goes far beyond classical AI with three additional theories

Infer is not a "pure" Cousot-style abstract interpreter. It is a hybrid system that combines four distinct theoretical foundations, each contributing a critical capability:

1. Abstract Interpretation Cousot & Cousot, 1977

Provides the fixpoint computation engine — loop invariants, convergence via widening/narrowing, abstract domains with join/meet/widen operations. This is the computational backbone.

2. Separation Logic Reynolds & O'Hearn, 2002

Provides heap reasoning — the separating conjunction (∗) lets Infer reason about disjoint memory regions, enabling local reasoning about pointer-manipulating programs.

3. Bi-Abduction Calcagno et al., POPL 2009

Provides compositional scalability — automatically infers pre/post specifications for each procedure independently, enabling incremental analysis of million-line codebases.

4. Incorrectness Logic O'Hearn, 2020

Provides true bug finding — under-approximate reasoning that guarantees reported bugs are reachable, reducing false positives. Powers the modern Pulse analyzer.

Key insight: Cousot's abstract interpretation provides the how (fixpoint computation over abstract domains), while separation logic provides the what (the abstract domain itself — symbolic heaps). Bi-abduction provides the scale (compositional procedure summaries), and incorrectness logic provides the precision (under-approximate bug reports).

The Four-Theory Stack

Cousot's Abstract Interpretation

Abstract interpretation, introduced by Patrick Cousot and Radhia Cousot in their landmark 1977 POPL paper, provides a unified lattice model for static analysis by constructing or approximating fixpoints of semantic equations.

Core Ideas

The Rule of Signs — AI in 30 Seconds

The simplest example of abstract interpretation: to determine the sign of -1515 * 17, you don't compute the actual result. Instead, you work in the abstract domain {(+), (-), (±)} using the rule of signs:

Concrete -1515 × 17 = -25755
Abstract (-) × (+) = (-)

The abstract computation is sound (the sign is correct) and fast (no big multiplication needed). This is abstract interpretation in miniature.

Concrete Domain (C)

The actual program semantics — potentially infinite sets of program states, memory configurations, execution traces. Uncomputable in general.

Abstract Domain (A)

A simplified representation — intervals, signs, pointer graphs, shapes. Must form a lattice with partial order ⊑, join ⊔, meet ⊓.

The Galois Connection

The formal bridge between concrete and abstract worlds:

Galois Connection (C, ⊆) ⇆ (A, ⊑)    via    α : C → A  (abstraction)    γ : A → C  (concretization)
Soundness Condition ∀ c ∈ C, a ∈ A :   α(c) ⊑ a  ⟺  c ⊆ γ(a)
How Fixpoint Computation Works ▶

Programs with loops create recursive equations. Abstract interpretation computes their solutions:

  1. Initialize: Start with ⊥ (bottom of the lattice)
  2. Iterate: Apply abstract transfer function F♯ repeatedly: ⊥, F♯(⊥), F♯(F♯(⊥)), ...
  3. Converge: When F♯ⁿ(⊥) = F♯ⁿ⁺¹(⊥), we have found the fixpoint
  4. Widen if needed: If the lattice has infinite ascending chains, apply widening (∇) to force convergence
Widening x ∇ y ⊒ x ⊔ y     (overshoots the join to guarantee termination)

The result is a sound over-approximation of all possible program behaviors.

Separation Logic

Separation logic, developed by John C. Reynolds and Peter O'Hearn (building on work by Rod Burstall), extends Hoare logic with operators for reasoning about mutable heap structures. It is the abstract domain that makes Infer's memory analyses possible.

Why Hoare Logic Isn't Enough

Classical Hoare logic can't distinguish between:

// Are x and y aliases? Hoare logic can't express this distinction. *x = 5; // Does this also change *y? *y = 10; // Does this also change *x?

Without knowing whether x and y point to the same memory cell, reasoning about pointer programs is unsound.

The Separating Conjunction (∗)

Core Operator A ∗ B    holds when the heap splits into two disjoint parts, one satisfying A and one satisfying B

Points-to: x ↦ v

Cell x exists in the heap and contains value v. Describes exactly one allocated cell.

Separation: x ↦ v ∗ y ↦ w

Two distinct cells exist: x holds v, y holds w, and x ≠ y is guaranteed by the separating conjunction.

Visual: How ∗ Partitions the Heap ▶

The Frame Rule

The most important inference rule in separation logic — it enables local reasoning:

Frame Rule If   {P} C {Q}   then   {P ∗ Frame} C {Q ∗ Frame}

Meaning: If program C only needs heap described by P, then any additional heap (Frame) remains completely untouched. You can reason about C using only the memory it accesses — ignoring the rest of a potentially gigabyte-sized heap.

Why it matters for Infer: This is why Infer can analyze individual functions without knowing the full program state. Each function's spec mentions only the heap it touches.

Separation Logic as an Abstract Domain

The connection to Cousot: Separation logic formulae (symbolic heaps) serve as the abstract domain in Infer's abstract interpretation. The Galois connection maps concrete heap states to symbolic heap formulas. Transfer functions map SL pre-conditions to SL post-conditions. Loop invariants are computed as fixpoints over this SL domain using widening.

Bi-Abduction

Bi-abduction is the algorithmic breakthrough that makes Infer scale to millions of lines. Introduced by Calcagno, Distefano, O'Hearn, and Yang at POPL 2009 (Gödel Prize 2016), it enables fully automatic, compositional analysis.

The Problem

Traditional whole-program analysis requires analyzing all code at once. For Facebook's mobile apps (tens of millions of lines), this is infeasible. We need per-procedure analysis that can run on code diffs in minutes.

What is Abduction?

Deduction (forward)

A ⊢ B    "A implies B"

Given A, conclude B. Standard logic.

Abduction (backward)

A ∗ ?X ⊢ B    "what's missing?"

Given A and B, find what additional facts X are needed to make A imply B. Inference of missing preconditions.

Bi-Abduction = Abduction + Frame Inference

Bi-Abduction Query A ∗ ?antiframe ⊢ B ∗ ?frame

Antiframe ?antiframe

The missing precondition — what additional heap state must exist before the operation for it to succeed. Becomes part of the procedure's pre-spec.

Frame ?frame

The unchanged remainder — heap state that passes through the operation untouched. Carried forward to the post-spec.

Worked Example: Inferring Specs Automatically ▶

Consider analyzing this code without any annotations:

closeResource(r1); // spec: {r ↦ open} close(r) {r ↦ closed} closeResource(r2);

Step 1: Analyze first call. Current state is empty (emp).

emp ∗ ?anti₁ ⊢ r1 ↦ open ∗ ?frame₁

Solution: anti₁ = r1 ↦ open, frame₁ = emp

After call: state is r1 ↦ closed

Step 2: Analyze second call. Current state is r1 ↦ closed.

r1 ↦ closed ∗ ?anti₂ ⊢ r2 ↦ open ∗ ?frame₂

Solution: anti₂ = r2 ↦ open, frame₂ = r1 ↦ closed

After call: state is r1 ↦ closed ∗ r2 ↦ closed

Result: Infer automatically discovered:

Inferred Specification {r1 ↦ open ∗ r2 ↦ open}   code   {r1 ↦ closed ∗ r2 ↦ closed}

No programmer annotations required. The antiframes accumulate into preconditions, the frames carry through to postconditions.

Infer's Architecture

High-Level Pipeline

Implementation Details

Written in OCaml

Infer's core engine is implemented in OCaml — a natural fit for symbolic computation, pattern matching on ASTs, and implementing abstract domains. OCaml's type system helps ensure the soundness of abstract domain implementations.

Frontend translators (for Java via javac, C/C++/ObjC via Clang) convert source code into Infer's intermediate representation.

SIL — Smallfoot Intermediate Language

All source languages are translated to SIL, a simple imperative language with:

  • Variable assignments, loads, stores
  • Heap operations (alloc, dealloc, field access)
  • Control flow (branches, loops)
  • Procedure calls

Named after Smallfoot, an early separation logic verifier by Berdine, Calcagno & O'Hearn that proved the feasibility of automated SL reasoning.

A newer "Textual" format allows frontends to emit SIL as text or JSON, enabling support for new languages (e.g., Infer# for C#/.NET).

A general-purpose framework for building new checkers. You provide:

The framework then handles: CFG construction, worklist iteration, fixpoint convergence, widening at loop heads, and interprocedural summary composition.

Compositional Interprocedural Analysis

Infer's interprocedural analysis is bottom-up and modular:

  1. Build a call graph of the program
  2. Analyze leaf procedures first (no callees)
  3. For each procedure, compute a summary (pre/post Hoare triple in SL)
  4. When analyzing a caller, instantiate callee summaries at call sites
  5. Bi-abduction infers the missing anti-frame for each call
  6. Summaries are cached — only re-computed when code changes
Key property: Summaries are context-independent — they describe procedure behavior for any calling context. This is what makes the analysis truly compositional and incremental.

Infer's Checkers

Infer ships with multiple specialized checkers, each targeting a different class of bugs. Some use the full SL+bi-abduction machinery, others use simpler abstract domains via the Infer.AI framework.

Pulse Primary ISL

Memory safety — null dereferences, use-after-free, memory leaks, uninitialized values. The flagship analyzer, using incorrectness separation logic for under-approximate reasoning. Replaced the original Biabduction checker.

  • Abstract domain: symbolic heaps + value history
  • Tracks object lifetimes and ownership
  • Reports only true bugs (no false positives goal)

Biabduction Legacy

Original memory checker — the classic SL + bi-abduction analysis. Over-approximate (sound but may have false positives). Produces pre/post summaries for every procedure. Being gradually replaced by Pulse.

RacerD Concurrency

Data race detection for Java/Android. Uses a custom abstract domain tracking lock state, thread identity, and shared memory accesses. Compositional — each method gets a "thread-safety summary".

  • Models Java's synchronized, ReentrantLock, etc.
  • Ownership-based reasoning reduces false positives

Starvation Concurrency

Deadlock and starvation detection. Tracks lock acquisition orders across threads and reports potential deadlocks when conflicting orders are found.

Topl Temporal

User-defined temporal properties. You define a state machine representing a property (e.g., "file must be closed after open"), and Topl monitors it using Pulse as its underlying memory model. Essentially a typestate checker.

Litho Android

Android-specific thread safety. Checks that Litho component lifecycle methods don't access state from the wrong thread. Domain-specific variant of thread-safety analysis.

Incorrectness Logic

Incorrectness Logic (IL), introduced by Peter O'Hearn in 2020, is a radical departure from traditional program verification. While Hoare Logic proves the absence of bugs (over-approximation), incorrectness logic proves the presence of bugs (under-approximation).

Over-Approximation vs Under-Approximation

Classical (Hoare Logic / AI)

Over-approximate: considers all possible behaviors, including impossible ones.

Hoare Triple {P} C {Q}

"If P holds before, then Q holds after every execution."

  • Proves absence of bugs
  • May report false positives (spurious warnings)
  • Sound: if it says "no bugs," there are no bugs

Incorrectness Logic

Under-approximate: considers only actual reachable behaviors.

Incorrectness Triple [P] C [Q]

"There exists an execution starting in P that reaches Q."

  • Proves presence of bugs
  • No false positives — every report is a real bug
  • May miss bugs (false negatives are acceptable)
The shift in perspective: Traditional verification asks "can anything go wrong?" and over-approximates to be safe. Incorrectness logic asks "does something definitely go wrong?" and under-approximates to avoid false alarms. For a developer tool at scale, eliminating false positives matters more than catching every bug — because developers ignore noisy tools.

Incorrectness Separation Logic (ISL)

O'Hearn and collaborators combined incorrectness logic with separation logic to get local reasoning about the presence of bugs in heap-manipulating programs:

ISL Frame Rule If   [P] C [Qerr]   then   [P ∗ Frame] C [Qerr ∗ Frame]

This means: if there's a bug reachable from state P, adding more heap (Frame) doesn't eliminate the bug. ISL is what powers Pulse, Infer's modern memory analyzer.

Pulse: ISL in Practice

Real Results

Pulse runs on every code diff at Meta (Facebook, Instagram, WhatsApp). It uses ISL to ensure reported issues are true positives — each report includes a concrete witness execution trace showing how the bug is triggered.

Research extensions (Pulse-X) found 15 real bugs in OpenSSL that were confirmed and fixed by maintainers, demonstrating the effectiveness of under-approximate reasoning for security-critical code.

Abstract Interpretation vs Separation Logic

A common misconception is that Infer uses "either AI or SL." In reality, they operate at different levels and are deeply complementary.

Aspect Abstract Interpretation (Cousot) Separation Logic (Reynolds/O'Hearn)
What it is A theory of computing fixpoints over abstract domains A logic for reasoning about heap-manipulating programs
Role in Infer The computation engine — how to iterate, converge, and compose The abstract domain — what to track about memory
Answers "How do we compute a sound approximation?" "What properties about the heap do we approximate?"
Key operations ⊑ (order), ⊔ (join), ∇ (widen), fixpoint iteration ∗ (separating conjunction), ↦ (points-to), frame rule
Without the other Could analyze programs but not reason about heap aliasing Could specify heap properties but not compute them automatically
Level Meta-theory (framework for building analyses) Object-level theory (specific domain of discourse)

How They Combine in Infer

Analogy

Think of it like a game engine and physics rules:

  • Abstract Interpretation = the engine (renders frames, handles the game loop, manages state transitions)
  • Separation Logic = the physics rules (defines how objects interact, what collisions mean, what forces apply)
  • Bi-Abduction = the level-loading system (lets you load and analyze individual rooms without the whole world)
  • Incorrectness Logic = the bug reporter (only flags issues a player can actually encounter, not theoretical edge cases)

You need all of them for a working game (analyzer). None is sufficient alone.

Q1: Does Facebook Infer use Cousot's abstract interpretation?
Click to reveal
Yes. Abstract interpretation provides the fixpoint computation engine that every Infer checker uses. The Infer.AI framework is a direct implementation of the AI paradigm: abstract domains with ⊑, ⊔, ∇, transfer functions, and worklist-based fixpoint iteration. However, Infer goes well beyond classical AI by using separation logic as its abstract domain for heap reasoning, bi-abduction for compositional scaling, and incorrectness logic for precision.
Q2: What does the separating conjunction (∗) mean?
Click to reveal
A ∗ B means the heap can be divided into two disjoint sub-heaps, one satisfying A and one satisfying B. This guarantees no aliasing between the two parts. For example, x ↦ 1 ∗ y ↦ 2 means x and y point to different cells.
Q3: What problem does bi-abduction solve?
Click to reveal
Bi-abduction solves the problem of compositional analysis at scale. Given a current state A and a required precondition B, bi-abduction simultaneously discovers: (1) the antiframe — what additional state is needed (becomes the procedure's precondition), and (2) the frame — what state is unchanged (carried to the postcondition). This lets each procedure be analyzed independently, enabling incremental analysis on code diffs.
Q4: What is the difference between Hoare triples and incorrectness triples?
Click to reveal
Hoare triple {P} C {Q}: for ALL executions starting in P, if they terminate, they end in Q. Over-approximate — may include impossible behaviors → false positives possible.

Incorrectness triple [P] C [Q]: there EXISTS an execution from P that reaches Q. Under-approximate — only describes reachable behaviors → no false positives, but may miss bugs.
Q5: Why did Infer transition from Biabduction checker to Pulse?
Click to reveal
The original Biabduction checker was over-approximate (sound but noisy). At Meta's scale, false positives caused developers to ignore warnings. Pulse uses incorrectness separation logic for under-approximate reasoning — it only reports bugs it can prove are actually reachable. This dramatically improves signal-to-noise ratio, even at the cost of potentially missing some bugs.
Q6: What role does the Frame Rule play in Infer's scalability?
Click to reveal
The frame rule states: if {P} C {Q} then {P ∗ Frame} C {Q ∗ Frame}. This means a procedure only needs to mention the heap it actually accesses. Everything else (Frame) passes through unchanged. This enables local reasoning — analyzing a function without knowing the entire program's memory layout — which is essential for scaling to millions of lines of code.
Q7: What is SIL and why is it important?
Click to reveal
SIL (Smallfoot Intermediate Language) is Infer's intermediate representation. All source languages (Java, C, C++, Objective-C) are translated to SIL during the "capture" phase. This means all checkers only need to work with one language. Named after Smallfoot, an early automated separation logic verifier that proved SL-based analysis is practical.
Q8: How does abstract interpretation handle infinite loops?
Click to reveal
When the abstract domain has infinite ascending chains (e.g., the interval domain for integers), naive fixpoint iteration may not converge. The widening operator (∇) forces convergence by over-approximating: x ∇ y ⊒ x ⊔ y. It "overshoots" the least upper bound to guarantee termination in finitely many steps, at the cost of precision. Subsequent narrowing can recover some lost precision.
