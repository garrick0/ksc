Part 18 — KindScript Research Series

How JavaScript Engines Actually Work

From source text to machine code: the fundamental objects, intermediate representations, and execution pipeline inside V8, SpiderMonkey, and JavaScriptCore.

1. JavaScript's Fundamental Entities

JavaScript has a remarkably simple value universe. There are seven primitives and one compound type (Object). Everything else — arrays, functions, dates, regex, maps — is just an Object with extra internal slots. The entire runtime is built on one data structure: the mutable property dictionary.

7 Primitive Types

// Immutable. Compared by value. No properties of their own.

number     // IEEE 754 double (64-bit float). 42, 3.14, NaN, Infinity
string     // UTF-16 sequence. "hello"
boolean    // true or false. That's it.
bigint     // Arbitrary precision integers. 9007199254740993n
symbol     // Unique identifier. Symbol("description")
null       // Intentional absence. One value: null.
undefined  // Uninitialized. One value: undefined.

// When you write "hello".length, JS temporarily wraps
// the primitive in a String object, accesses .length,
// then discards the wrapper. The primitive itself has
// no properties.

Primitives are the atoms. They have no internal structure the language exposes. They're immutable — you can't change the number 5 or mutate the string "hello". You can only create new values.

Objects — The One Compound Type

// An object is fundamentally:
// 1. A mutable map from property keys (string|symbol) → values
// 2. A [[Prototype]] link to another object (or null)
// 3. A set of "internal slots" (engine-level hidden state)

const obj = { name: "Alice", age: 30 };

// In the spec's model, this object has:
//   Own properties: { "name" → "Alice", "age" → 30 }
//   [[Prototype]] → Object.prototype
//   Internal slots: [[Extensible]] = true

// Property access walks the prototype chain:
obj.toString()
//   1. Check obj's own properties for "toString" → not found
//   2. Follow [[Prototype]] to Object.prototype
//   3. Check Object.prototype for "toString" → found ✓

Every non-primitive value in JavaScript is an Object. The prototype chain creates inheritance. Internal slots add special behaviors that you can't replicate in user code.

Functions — Callable Objects

// A function is an Object with two extra internal slots:
//   [[Call]]      → the code to execute when called
//   [[Construct]] → the code to execute with 'new' (optional)

function greet(name) { return "Hello " + name; }

// 'greet' is an Object that has:
//   Own properties: { "name" → "greet", "length" → 1, "prototype" → {...} }
//   [[Prototype]] → Function.prototype
//   [[Call]] → the compiled code for the function body
//   [[Environment]] → reference to the outer scope (closure)

// You can add arbitrary properties to functions:
greet.version = "1.0";
// This works because functions are objects.

// The dual nature: greet is both:
//   - A VALUE you can store, pass, compare (object)
//   - A COMPUTATION you can trigger (callable)

This duality is exactly why typing functions is hard. The object part has shape (properties). The callable part has behavior (effects, termination, purity). TypeScript describes the former. KindScript describes the latter.

Arrays — Objects with Integer Keys

// An array is an Object with:
//   - Integer-indexed properties (elements)
//   - A magic "length" property that auto-updates
//   - [[Prototype]] → Array.prototype (gives .push, .map, etc.)

const arr = ["a", "b", "c"];

// Internally this is roughly:
//   Own properties: { "0" → "a", "1" → "b", "2" → "c", "length" → 3 }
//   [[Prototype]] → Array.prototype

// This is why arr["0"] === arr[0] === "a"
// Property keys are always strings (or symbols).
// The number 0 gets coerced to the string "0".

// Engines optimize: V8 stores dense arrays as contiguous
// memory (like C arrays), not as hash maps. But the
// spec model is just "object with numeric keys."

The ECMAScript Spec Model

// The spec defines ABSTRACT OPERATIONS — functions that
// describe what JS means, not how to implement it.

// When you write: x + y
// The spec says:
//   1. Call ToPrimitive(x)
//   2. Call ToPrimitive(y)
//   3. If either result is a string, concatenate
//   4. Otherwise, call ToNumber on both and add

// The spec also defines:
//   Execution Context  — scope + variable bindings
//   Lexical Environment — mapping of names to values
//   Environment Record  — the actual storage
//   Realm              — global object + built-ins
//   Agent              — thread of execution
//   Job Queue           — async scheduling

// These are ABSTRACT. Engines implement them however
// they want, as long as observable behavior matches.

The spec is Layer 2 (denotation) for JavaScript. It defines what programs mean without prescribing implementation. Engines can and do deviate wildly from the spec's internal structure — they just have to produce the same observable results.

Source Text

function add(a, b) {
  return a + b;
}
add(2, 3);

Just characters in a file. The engine reads this as a UTF-8/UTF-16 byte stream. Everything starts here — both TypeScript's type checker and V8's compiler begin by turning these characters into structured data.

Parser → AST

// V8's parser is a recursive-descent parser.
// It reads tokens and builds an Abstract Syntax Tree.
// V8 also does LAZY PARSING: inner functions are
// pre-parsed (syntax check only) and fully parsed
// only when first called. This speeds up startup.

// Two parsers:
//   PreParser  — fast, just checks syntax, no AST
//   Parser     — full parse, builds AST nodes

// For: function add(a, b) { return a + b; }
// If add() is never called, the PreParser handles it.
// If add() IS called, the full Parser kicks in.

V8's AST is structurally very similar to TypeScript's AST (both represent the same language). But V8's parser is optimized for speed, not analysis — it drops information TypeScript would keep (like exact source positions for every sub-expression).

The AST

// The AST for: function add(a, b) { return a + b; }
FunctionDeclaration {
  name: "add"
  params: [Identifier("a"), Identifier("b")]
  body: Block {
    statements: [
      ReturnStatement {
        expression: BinaryExpression {
          op: "+"
          left: Identifier("a")
          right: Identifier("b")
        }
      }
    ]
  }
}

// THIS is the shared substrate.
// TypeScript walks this structure to assign types.
// KindScript walks this structure to check properties.
// V8 walks this structure to generate bytecode.
// Same tree, three different consumers.

Ignition Compiler (AST → Bytecode)

// Ignition walks the AST and emits register-based bytecode.
// The AST is DISCARDED after this step — only bytecode remains.

// The bytecode uses:
//   - An implicit ACCUMULATOR register
//   - Numbered registers: r0, r1, r2, ...
//   - A constant pool for literals
//   - Feedback slots for type profiling

// Ignition is both a COMPILER (AST → bytecode)
// and an INTERPRETER (executes bytecode directly).
// This is the first tier of execution.

Key insight: the AST is an intermediate representation that gets consumed and discarded. V8 doesn't keep the AST around. TypeScript doesn't keep it either (after checking). The AST is a stepping stone, not a permanent structure.

Bytecode

// Bytecode for: function add(a, b) { return a + b; }

    Ldar    a0        // Load argument 0 (a) into accumulator
    Add     a1, [0]   // Add argument 1 (b) to accumulator
                       // [0] = feedback slot index
    Return            // Return accumulator value

// More complex example: if (x > 0) return x; else return -x;

    Ldar    a0        // Load x into accumulator
    TestGreaterThan [0], Smi(0)  // Compare with 0
    JumpIfFalse @else // Branch
    Ldar    a0        // Load x
    Return
  @else:
    Ldar    a0        // Load x
    Negate  [1]       // Negate accumulator
    Return

// Key: [0], [1] are FEEDBACK SLOT indices.
// The interpreter records what types flow through
// each slot. This feeds the JIT compilers.

Bytecode is a flat, linear instruction sequence — no tree structure. Each instruction is a few bytes. The entire bytecode for a function is typically much smaller than the AST was. This is the representation that actually executes (initially).

JIT Compilation Tiers

As functions get called repeatedly, they get promoted through optimization tiers:

T0 Ignition (Interpreter) — all code starts here
Executes bytecode instruction by instruction. Slow but starts instantly (no compilation delay). Records type feedback in the feedback vector: "at this Add instruction, I've seen number + number 47 times."
T1 Sparkplug (Baseline Compiler) — warm code
Translates bytecode directly to machine code without optimization. One-to-one mapping: each bytecode instruction becomes a fixed sequence of machine instructions. Compiles in ~10μs per function. ~5-10x faster than interpretation. Kicks in after a function is called a few times.
T2 Maglev (Mid-tier Compiler) — warmer code
Uses feedback vector data to specialize. If Add always saw numbers, compiles it as a raw machine ADD with a type guard. Builds a graph IR, does basic optimizations (dead code elimination, common subexpression elimination). ~10x slower to compile than Sparkplug but ~10x better code.
T3 TurboFan (Optimizing Compiler) — hot code
Full optimizing compiler. Builds a "Sea of Nodes" / Turboshaft IR. Performs: function inlining, escape analysis, loop invariant code motion, dead code elimination, range analysis, type specialization. The output is highly optimized machine code. Compilation takes milliseconds but produces code rivaling hand-written C for numeric work.
⚠ Deoptimization — when assumptions fail 
// TurboFan compiled add() assuming numbers:
add(2, 3)     // ✓ fast path: machine ADD
add(4, 5)     // ✓ fast path: machine ADD
add("a", "b") // ✗ STRINGS! Type guard fails.
                //   → DEOPTIMIZE: discard machine code
                //   → Fall back to Ignition bytecode
                //   → Re-collect feedback (now polymorphic)
                //   → Maybe re-optimize with broader types

Deoptimization is the safety net. The JIT can speculate aggressively because it can always bail out. This is fundamentally different from static type checking — TypeScript rejects the program; V8 accepts it but adapts at runtime.

Machine Code

// TurboFan output for: function add(a, b) { return a + b; }
// (when specialized for Smi integers)

// x86-64 assembly (simplified):
  test   rdi, 0x1       ; check: is 'a' a Smi? (tag bit)
  jnz    deopt          ; if not, deoptimize
  test   rsi, 0x1       ; check: is 'b' a Smi?
  jnz    deopt          ; if not, deoptimize
  add    rax, rdi, rsi  ; actual addition (one instruction!)
  jo     deopt          ; if overflow, deoptimize
  ret                   ; return result

// The type guards (test/jnz) are the cost of dynamic typing.
// In a statically typed language, they wouldn't be needed.
// The actual work is one ADD instruction.

This is the final form. Raw machine instructions. The CPU executes these directly. The journey from function add(a,b) { return a+b } to a single ADD instruction is what the entire pipeline achieves — but only for hot, type-stable code.

3. How V8 Represents Objects in Memory

3.1 Hidden Classes (Maps)

The key optimization: Instead of treating objects as hash maps (slow), V8 gives each object a "hidden class" (internally called a Map) that describes its shape. Objects with the same sequence of properties share the same Map. Property access becomes a direct memory read at a known offset, not a dictionary lookup.

Click each step to see how Maps evolve:

1 const obj = {} — empty object
obj →
Map pointer → Map₀ (empty)

V8 creates a JSObject with a Map describing "no properties." The Map stores: property count = 0, no property descriptors.

2 obj.x = 1 — add property "x"
obj →
Map pointer → Map₁ (x at offset 0)
+8 →
property "x" = 1

V8 transitions from Map₀ to Map₁. Map₁ says: "property 'x' is at byte offset 0 in the object." The value 1 is stored directly in the object (in-object property). Map₀ records the transition: "if you add 'x', go to Map₁."

3 obj.y = 2 — add property "y"
obj →
Map pointer → Map₂ (x@0, y@1)
+8 →
property "x" = 1
+16 →
property "y" = 2

Transition to Map₂. Now the Map says: "x is at offset 0, y is at offset 1." Any other object created with the same property sequence { x, y } will share Map₂. This is why V8 loves constructor functions — they produce objects with consistent shapes.

4 Another object with same shape shares the Map
obj →
Map pointer → Map₂
+8 →
1
+16 →
2

obj2 →
Map pointer → Map₂ (same!)
+8 →
10
+16 →
20

obj2 = {}; obj2.x = 10; obj2.y = 20; — same transition path, same Map₂. V8 only stores the shape description once. This is analogous to TypeScript's structural typing: both objects have "the same type" because they have the same shape.

3.2 Inline Caches (ICs)

The performance trick: When code accesses obj.x, V8 doesn't do a hash lookup every time. After the first access, it caches: "objects with Map₂ have 'x' at byte offset 0." Subsequent accesses of the same shape are a single pointer dereference with a type guard.

UNINITIALIZED First encounter: obj.x
The IC slot has no cached data. V8 performs a full property lookup: check obj's Map, find "x" in the Map's descriptor table, get the offset. Record: (Map₂, offset 0) in the IC slot.
MONOMORPHIC Same shape again: obj2.x 
// Generated IC check (pseudocode):
if (obj2.map === Map₂) {
  // FAST PATH: direct memory read at offset 0
  result = obj2[offset_0];   // one load instruction!
} else {
  // SLOW PATH: full property lookup
}

Monomorphic = only one shape seen. This is the fastest possible property access — one comparison + one load. V8 loves monomorphic call sites.

POLYMORPHIC Different shape: obj3.x (where obj3 has Map₇) 
// IC now caches 2-4 shapes:
if (obj.map === Map₂) result = obj[offset_0];
else if (obj.map === Map₇) result = obj[offset_2];
else { /* slow path */ }

Polymorphic = 2-4 shapes seen. Still fast — a small linear search through cached shapes. This happens when a function receives objects of slightly different shapes.

MEGAMORPHIC Too many shapes (>4)
The IC gives up on caching specific shapes and falls back to a hash-table-based lookup. Much slower. This is the performance cliff you hit when code is too polymorphic — when a function receives objects with wildly different shapes.

Lesson for KindScript: Monomorphic code is fast code. A static analysis that could detect megamorphic call sites would be practically useful, not just theoretically interesting.

4. Functions, Closures & Scope Chains

What a Function IS in Memory

func →
Map pointer → FunctionMap
+8 →
SharedFunctionInfo → {name, bytecode, source position}
+16 →
Context → captured variables (closure)
+24 →
Feedback Vector → {IC data, type profiles}
+32 →
Code → pointer to bytecode or machine code

SharedFunctionInfo is shared across all closures created from the same source function. The bytecode and metadata are identical. Context is unique per closure — it captures the specific variable values from the enclosing scope.

Scope Chain

function outer() {
  const x = 10;
  function middle() {
    const y = 20;
    function inner() {
      return x + y;  // accesses both outer scopes
    }
    return inner;
  }
  return middle();
}

// When inner() runs, variable lookup for 'x':
//   1. Check inner's own scope → not found
//   2. Follow context pointer to middle's scope → not found
//   3. Follow context pointer to outer's scope → found! x = 10

// This chain of context pointers IS the scope chain.

Context Objects in Memory

// V8 creates a Context object when a scope has captured variables.

// For outer():
Context_outer = {
  parent: GlobalContext,
  slots: { x: 10 }           // x is captured by inner
}

// For middle():
Context_middle = {
  parent: Context_outer,       // points UP the chain
  slots: { y: 20 }            // y is captured by inner
}

// inner's JSFunction has:
inner.context → Context_middle
// To reach x: inner.context.parent.slots.x
// That's two pointer dereferences for x, one for y.

// Optimization: V8 can compute the exact depth at
// compile time. It doesn't "search" — it knows x is
// at context[1].slot[0] (depth 1, slot 0).

What Engines Actually Do

// Key optimization: NOT all variables go on the heap.

function example() {
  const a = 1;      // NOT captured → stays on STACK
  const b = 2;      // captured by closure → goes on HEAP
  const c = 3;      // NOT captured → stays on STACK
  return () => b;   // only captures b
}

// V8 analyzes which variables are captured (scope analysis)
// and only allocates Context objects for those.
// Uncaptured variables live in registers/stack — fast and
// automatically cleaned up when the function returns.

// This is actually a form of static analysis that V8 does!
// It's analyzing the AST to determine variable capture.
// KindScript does something similar when it checks
// whether a function accesses outer scope variables.

V8 (Chrome, Node.js, Deno, Bun) is the most widely deployed. 4-tier pipeline optimized for server and desktop workloads. Aggressive speculation with deoptimization safety net. Uses "Sea of Nodes" IR (transitioning to Turboshaft CFG IR).

SpiderMonkey (Firefox) uses a 3-tier approach. Key difference: Warp's optimizations are built directly on top of Inline Cache data rather than a separate speculation system. Bailouts reconstruct the Baseline Interpreter's stack frame to resume execution.

~6 calls

JavaScriptCore (Safari, Bun) uses 4 tiers. LLInt is written in custom assembly ("offlineasm"). DFG (Data Flow Graph) does mid-level optimization. FTL (Faster Than Light) uses LLVM for peak optimization. Optimized for mobile memory constraints.

The pattern across all three engines: Parse → AST → Bytecode → Interpret → Profile types → JIT compile with type specialization → Deoptimize if wrong. The details differ (IR design, tier thresholds, optimization strategies) but the architecture is fundamentally the same. All three do runtime type inference through observation + speculation — the dynamic counterpart to TypeScript's static type inference through analysis.

6. The ECMAScript Spec vs What Engines Do

What the Spec Says

// For: obj.x
// Spec algorithm (simplified):
1. GetValue(Reference(obj, "x"))
2. ToObject(obj)
3. OrdinaryGetOwnProperty(obj, "x")
4. If not found, recurse on
   obj.[[Prototype]]
5. If found, return value
6. If accessor, call [[Get]]

The spec defines abstract operations — step-by-step algorithms. These are mathematical definitions of meaning (Layer 2: denotation). They tell you what the answer should be.

What V8 Does

// For: obj.x (monomorphic IC)
// Actual execution:
1. Load obj's Map pointer
2. Compare to cached Map
3. If match: load from
   cached byte offset
   (ONE memory read)
4. If mismatch: slow path
   (full spec algorithm)

The engine takes shortcuts that produce the same observable result. The IC skips steps 2-6 of the spec algorithm for the fast path. This is legal because the result is identical.

The contract: Engines can do whatever they want as long as observable behavior matches the spec. "Observable" means: return values, thrown exceptions, side effects (property reads/writes, function calls). Internal representation, execution order of non-observable operations, memory layout — all implementation freedom.

7. What This Means for KindScript

Shared Substrate

TypeScript, KindScript, and V8 all start from the same place: source text → AST. The AST is the common substrate. TypeScript walks it to assign types. KindScript walks it to check properties. V8 walks it to generate bytecode. Same tree, three different analyses, three different outputs.

Dimension TypeScript KindScript V8
Input AST (from parser) AST (from ts.createProgram) AST (from V8 parser)
Walks Every expression Function bodies Every node (for bytecode gen)
Assigns Types (shape predicates) Properties (behavior predicates) Bytecode + feedback slots
Models Sets of possible values Behavioral characteristics Machine instructions
When Static (before execution) Static (before execution) Dynamic (during execution)
On error Reject program Report diagnostic Deoptimize & adapt

The V8 analogy for properties

V8's feedback vector is interesting because it does runtime property inference. At each Add instruction, V8 observes: "I've seen numbers 100% of the time here." This is a runtime-inferred behavioral property — "this operation is always numeric." It's the dynamic equivalent of what KindScript does statically.

V8 uses this to optimize. KindScript could use its property inference to verify contracts. Same fundamental idea — observe/analyze properties of computations — but one does it at runtime through observation, the other at compile time through analysis.
