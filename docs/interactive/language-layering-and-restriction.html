Language Layering & Building Paradigms from JS

How higher-level languages restrict lower-level ones, and how you can build functional/effect-typed/total-pure languages on top of JavaScript.

1 The layer cake

Every program you run ultimately becomes machine code — raw CPU instructions that can do anything: read any memory address, overwrite the OS, brick the hardware. But you don't have those powers in JavaScript. Where did they go?

The pattern at every layer: The lower layer can do dangerous things. The higher layer removes your ability to express those dangerous things. You can't corrupt memory in JS not because the CPU can't — but because the V8 engine never generates the machine instructions that would corrupt memory in response to your JS code. The restriction happens at the translation boundary.

2 Four mechanisms of restriction

Higher languages restrict lower ones through exactly four mechanisms. Click each to see how it works:

"Do we just have to assume some assertions hold?" — Mostly no. The restrictions are enforced, not assumed. V8's GC isn't an assumption — it's a real mechanism that physically prevents use-after-free. TypeScript's type checker isn't an assumption — it actually rejects programs with type errors. The only assumptions are at the edges: "the V8 engine itself doesn't have bugs" and "the OS scheduler is fair." Those are trusted computing base assumptions that every system makes.

5 The trust boundary question

You asked: "If they're still using C-like capabilities under the hood, do we just have to assume some assertions hold?" Here's the precise answer:

There are three levels of trust:

1. Hardware trust: You trust the CPU executes instructions correctly. Everyone makes this assumption. If Intel's microcode has a bug, all bets are off at every level. (This has actually happened: Spectre, Meltdown.)

2. Runtime trust: You trust V8/SpiderMonkey/etc. correctly implements the JS spec. If V8 has a GC bug, memory safety breaks. This is the trusted computing base — it must be correct for the layer above to have its guarantees. It's a small, heavily-tested surface.

3. Library/lint trust: You trust that eslint-plugin-functional actually catches all mutations, or that Effect-TS's type encoding is sound. This is the weakest level — escape hatches exist. But it's the most practical level for building new paradigms.

6 The complete picture

What you can build from JS/TS today, with what trust level:

Functional: ✅ Done. Elm (compiles to JS, language-level guarantee), eslint-plugin-functional (lint-level), Ramda/fp-ts/Effect-TS (library-level). Trust: high for Elm, medium for others.

Effect-typed: ✅ Done. Effect-TS gives you Effect<A, E, R> tracking. Trust: medium — you can escape the Effect wrapper, but the type system catches most violations.

Capability-based: ✅ Partially done. SES (Secure ECMAScript)/Hardened JS (Agoric) removes ambient authority. Deno's permission model. Trust: high for SES (runtime-enforced), medium for Deno.

Total/pure: ⚠️ Approximated. No JS-targeting language guarantees termination for all programs. You can approximate with loop bans + recursion limits (eslint rules + runtime guards). Trust: medium — heuristic, not proof.

Dependently typed: ⚠️ Exists but niche. Idris compiles to JS. PureScript has some dependent type features. Trust: high (language-level), but tiny ecosystem.

The fundamental insight: you don't need to change the engine to change the language. JS's C++ runtime provides a uniform computational substrate. On top of it, you can build any paradigm by controlling what the programmer can express — via compilation (Elm → JS), types (Effect-TS), lint rules (eslint-plugin-functional), or runtime wrappers (SES/Hardened JS). The guarantees are real — they just live at different levels of the stack.
