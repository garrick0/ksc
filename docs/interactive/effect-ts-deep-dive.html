Effect-TS Deep Dive

A thorough analysis of how Effect-TS actually works — compilation, type encoding, runtime internals, what it checks statically, and where it stops.

1. The Headline: No Magic

The most important thing to understand about Effect-TS: there is no compiler plugin, no Babel transform, no SWC plugin, no AST manipulation, no custom type checker. It is a pure TypeScript library. Standard tsc compiles it. Every guarantee it provides comes from TypeScript's existing type system — generics, conditional types, variance annotations, branded types.

This means: everything Effect-TS "checks" is checked by tsc's normal generic resolution. Everything it "enforces" is enforced by type errors from tsc. It doesn't read your AST, doesn't do abstract interpretation, doesn't run a separate analysis pass. It makes TypeScript's type checker do the work by encoding information in generic type parameters.

The Core Type Definition

Effect<A, E, R> is the central type. The three parameters encode everything:

A — Success Value

What the computation produces on success. Like Promise<A> but with more tracking.

E — Error Channel

What can go wrong. A union type that accumulates as you compose. never means "cannot fail."

R — Requirements

Services needed to run. An intersection type that accumulates. never means "needs nothing."

What's the actual runtime representation?

At runtime, an Effect is a Plain Old JavaScript Object (POJO) — a discriminated union of primitive instructions. Not a function, not a class instance. A data structure with a tag field that the runtime pattern-matches on:

// Simplified — the actual instruction set is larger
type EffectPrimitive =
  | { _tag: "Succeed";  value: any }
  | { _tag: "Fail";     cause: Cause }
  | { _tag: "Sync";     thunk: () => any }
  | { _tag: "Async";    register: (cb: (_: any) => void) => void }
  | { _tag: "FlatMap";  first: Effect; andThen: (a: any) => Effect }
  | { _tag: "OnFailure"; first: Effect; onFailure: (e: any) => Effect }

// When you write:
Effect.succeed(42)
// You get: { _tag: "Succeed", value: 42 }

// When you write:
Effect.flatMap(getUser, user => getAddress(user))
// You get: { _tag: "FlatMap", first: getUserEffect, andThen: user => getAddressEffect }

This is the Freer monad pattern: separate the description of what to do from the interpretation of how to do it. The runtime is the interpreter.

Variance Annotations

TypeScript 4.7+ introduced in/out modifiers for explicit variance control. Effect-TS uses these to tell TypeScript how to handle subtype relationships:

// Simplified from Effect's actual definition:
interface Effect<out A, out E = never, out R = never> {
  // out = covariant: subtypes are compatible
}

What covariance means in practice:

A is covariant (out)

Effect<Dog> is assignable to Effect<Animal> because Dog extends Animal. If you produce a more specific type, that's fine where a less specific type is expected.

E is covariant (out)

Effect<A, never> (cannot fail) is assignable to Effect<A, Error> (might fail). An effect that never fails is a valid substitute for one that might — it's strictly better.

R is covariant (out)

Effect<A, E, never> (needs nothing) is assignable to Effect<A, E, Database> (needs Database). An effect with no requirements satisfies any requirement.

Without these annotations, TypeScript would default to structural checking, which can give wrong results for complex generic compositions. The variance annotations prevent subtle bugs where effects with different requirement profiles get mixed up.

How pipe Enables Type Inference

pipe is defined as ~20 overloaded signatures, each with progressive generic parameters:

// Each overload adds one more transformation step:
function pipe<A>(a: A): A
function pipe<A, B>(a: A, ab: (a: A) => B): B
function pipe<A, B, C>(a: A, ab: (a: A) => B, bc: (b: B) => C): C
function pipe<A, B, C, D>(a: A, ab: (a: A) => B, bc: (b: B) => C, cd: (c: C) => D): D
// ... up to ~20 steps

TypeScript's generic inference resolves each step. Here's what happens in a real chain:

const result = pipe(
  getUser(id),                    // A = Effect<User, NotFound, Database>
  Effect.flatMap(getAddress),     // B = Effect<Address, NotFound | NoAddr, Database>
  Effect.catchTag("NoAddr",       // C = Effect<Address, NotFound, Database>
    () => Effect.succeed(fallback)),
  Effect.timeout("5 seconds"),   // D = Effect<Address, NotFound | TimeoutException, Database>
);

At each step, TypeScript infers the new A, E, R. Errors union together when composed. Errors narrow when handled. Requirements accumulate as intersections. All from standard generic resolution — no custom inference engine.

Context.Tag — Service Identity

How does Effect-TS track which services a computation needs? Through Context.Tag, which uses a clever nominal typing trick:

// Define a service
class Database extends Context.Tag("Database")<
  Database,                               // Self type (recursive)
  {                                       // Service interface
    query: (sql: string) => Effect<Row[], DbError>
  }
>() {}

// The string "Database" is the unique identifier
// The Self type parameter creates NOMINAL typing
// Two tags with identical interfaces but different names are DIFFERENT services

At runtime, Context is essentially a Map<Tag, Implementation>. At the type level, the R parameter accumulates all Tags needed:

// When you use a service:
const getUsers = Effect.flatMap(
  Effect.service(Database),    // R picks up Database
  db => db.query("SELECT *")
);
// Type: Effect<Row[], DbError, Database>
//                              ^^^^^^^^ tracked here

// When you provide the service:
const runnable = Effect.provide(getUsers, DatabaseLive);
// Type: Effect<Row[], DbError, never>
//                              ^^^^^ satisfied!

The Tag creates a unique type-level "key." TypeScript's structural typing would normally merge two services with the same interface. The Self type parameter plus the string identifier prevent this — it's nominal typing within a structural system.

Branded Types

TypeScript uses structural typing — two types with the same shape are interchangeable. This is dangerous when you want semantic distinction:

// Without branding: these are the same type!
type UserId = string;
type OrderId = string;

function getUser(id: UserId) { ... }
getUser(orderId); // TypeScript allows this! Both are strings.
// With Brand: these are now distinct types
type UserId = string & Brand.Brand<"UserId">
type OrderId = string & Brand.Brand<"OrderId">

function getUser(id: UserId) { ... }
getUser(orderId); // TypeScript ERROR: OrderId is not assignable to UserId

The trick: Brand.Brand<"UserId"> adds a phantom property to the type — a property that exists in the type system but has no runtime representation. Two differently branded types have different phantom properties, so TypeScript considers them structurally distinct.

Effect-TS offers two flavors:

Brand.nominal

No validation. Just a type-level tag. Use when you trust the source of the value.

Brand.refined

Includes runtime validation. Values are checked when branded. Use for domain constraints (positive numbers, email formats, etc.).

The Interpretation Loop

When you call Effect.runPromise(program), the fiber runtime starts an interpretation loop:

// Simplified pseudocode of the core loop:
function runLoop(effect: Effect): Exit {
  let current = effect;
  let continuationStack: Array<(value) => Effect> = [];
  let opCount = 0;

  while (true) {
    opCount++;
    if (opCount > 2048) {       // yield threshold
      reschedule(current);      // put back in work queue
      return;                   // give other fibers a turn
    }

    switch (current._tag) {
      case "Succeed":
        if (continuationStack.length === 0) return Exit.succeed(current.value);
        current = continuationStack.pop()(current.value);
        break;

      case "Fail":
        // Walk up continuation stack looking for error handler
        // If none found, return Exit.fail(current.cause)
        break;

      case "Sync":
        current = { _tag: "Succeed", value: current.thunk() };
        break;

      case "Async":
        current.register(result => resumeFiber(result));
        return;  // suspend — will resume when callback fires

      case "FlatMap":
        continuationStack.push(current.andThen);
        current = current.first;  // evaluate the first effect
        break;
    }
  }
}

Key properties of this loop:

Stack-safe

Continuations live on the heap (in the array), not on the JavaScript call stack. Deeply nested flatMap chains don't cause stack overflow.

Cooperative

After ~2048 operations, the fiber yields. Other fibers get a turn. This prevents one computation from monopolizing the event loop.

Suspendable

On async boundaries, the fiber suspends. The continuation stack is preserved on the heap. When the async callback fires, the fiber resumes exactly where it left off.

The Effect Instruction Set

Every Effect value is one of these primitive instructions. Complex programs are trees of these:

InstructionWhat it meansCreated by
SucceedImmediate value — doneEffect.succeed(42)
FailImmediate failure with a CauseEffect.fail(err)
SyncRun this synchronous thunkEffect.sync(() => Date.now())
AsyncRegister a callback, suspend until it firesEffect.async(cb => ...)
FlatMapRun first, feed result to andThenEffect.flatMap(e, f)
OnFailureRun first, if it fails run onFailureEffect.catchAll(e, f)
OnSuccessRun first, if it succeeds run handlerEffect.tap(e, f)
ForkStart a new fiber for this effectEffect.fork(e)
ProvideAttach a context (service map) to this effectEffect.provide(e, layer)

A real program is a tree of these. For example:

pipe(
  getUser(id),             // Async (register fetch callback)
  Effect.flatMap(          // FlatMap
    user => getAddress(user) //   andThen: Async (another fetch)
  ),
  Effect.catchTag("NotFound", // OnFailure
    () => Effect.succeed(fallback) //   recovery: Succeed
  ),
)
// Becomes: OnFailure(FlatMap(Async(...), user => Async(...)), ...)

Fibers — Lightweight Green Threads

A fiber is the runtime's unit of concurrent execution — like a thread but managed entirely in JavaScript, not by the OS. Each fiber has:

Its own continuation stack

Each fiber tracks where it is in its Effect tree independently. Suspending one fiber doesn't affect others.

An interruptible status

Fibers can be interrupted (cancelled). The runtime checks for interruption between instructions. This enables timeouts, races, and resource cleanup.

A scope

Each fiber belongs to a scope. When a scope closes (parent completes or fails), child fibers are interrupted. This is structured concurrency — no orphaned background work.

// Fork a fiber
const program = Effect.gen(function*() {
  const fiber = yield* Effect.fork(longComputation);
  // fiber is running concurrently now

  const result = yield* Fiber.join(fiber);
  // wait for it to complete

  // Or: yield* Fiber.interrupt(fiber)  → cancel it
});

// Race two effects — first one to complete wins, loser is interrupted
const fast = Effect.race(fetchFromCache, fetchFromDB);

The Scheduler

The scheduler manages which fiber runs when:

Your Effect Programs — descriptions of work
Fiber Runtime — interprets instructions, manages continuation stacks
Scheduler — work queue, op-count yielding (~2048 ops), cooperative multitasking
JavaScript Event Loop — the actual execution engine

The scheduler uses cooperative yielding: each fiber maintains an opCount. After ~2048 primitive operations, the fiber voluntarily yields, the scheduler puts it back in the work queue, and picks up the next fiber. This prevents starvation and ensures responsiveness.

For async boundaries (Async instructions), the fiber suspends naturally — it registers a callback and returns control to the event loop. When the callback fires, the scheduler resumes the fiber.

5. What tsc Misses — The Gaps

This is the critical section. Effect-TS has real blind spots that come from being a library rather than a language feature or a static analyzer:

Summary of What tsc Misses

ScenarioCaught?Why not

as any type assertionsNOIntentional escape hatch in TypeScript itself
Calling effectful code without wrappingNONothing forces you to use Effect for every side effect
Throwing inside Effect.sync (defects)NOEffect.sync assumes no failures; throws become "defects" not tracked in E

Non-determinism (Math.random)NONo effect tracking for randomness unless you wrap it

The fundamental limit

TypeScript's type checker doesn't analyze what happens inside function bodies. It checks the signature — the contract at the boundary. If Effect.sync(() => ...) says it returns Effect<number, never, never>, tsc believes it. It doesn't peek inside the callback to verify there's no IO, no mutation, no throws. This is the gap that a static analyzer like KindScript could fill.

What TypeScript Infers Automatically

Effect-TS is designed so you rarely need explicit type annotations. TypeScript's inference handles:

// All three parameters are inferred — no annotation needed:
const getUser = (id: string) => pipe(
  Effect.service(Database),                  // R = Database
  Effect.flatMap(db => db.query(`...`)),     // A = Row[], E = DbError
  Effect.flatMap(rows =>
    rows.length > 0
      ? Effect.succeed(rows[0])               // A narrows to Row
      : Effect.fail(new NotFound(id))          // E adds NotFound
  ),
);
// TypeScript infers: Effect<Row, DbError | NotFound, Database>
// You didn't write any of that — it all came from the pipe chain.

When you add a new service dependency, the R parameter updates automatically. When you add error-producing operations, E updates. When you handle errors, E narrows. All inferred.

Effect.gen — Generator-Based Syntax

The generator syntax gives the best inference and the most readable code:

const program = Effect.gen(function*() {
  const db   = yield* Effect.service(Database);   // db: DatabaseService
  const rows = yield* db.query("SELECT *");      // rows: Row[]

  if (rows.length === 0) {
    yield* Effect.fail(new NotFound());        // adds NotFound to E
  }

  const logger = yield* Effect.service(Logger);  // adds Logger to R
  yield* logger.info(`Found ${rows.length}`);

  return rows[0];
});
// Inferred: Effect<Row, NotFound | DbError, Database | Logger>

Each yield* "unwraps" an Effect, extracting the success value. TypeScript infers the type of each intermediate variable. The overall Effect type is inferred from all the yield* calls — errors union, requirements union, return type is A.

Requires "downlevelIteration": true or "target": "ES2015"+ in tsconfig.

What You Must Annotate

Error types from external libraries

// tsc can't know what fetch might throw
const fetchData = Effect.tryPromise({
  try: () => fetch("/api"),
  catch: (e) => new NetworkError(String(e))  // YOU define the error type
});

Service Tag definitions

// The Self type parameter must be explicit:
class Database extends Context.Tag("Database")<
  Database,             // ← this Self type is required
  { query: ... }
>() {}

Recursive or complex type boundaries

When inference can't follow the recursion, or when you want to publish a clean API, you add an explicit return type annotation. Effect-TS recommends leaving return types off internal functions and adding them to public API boundaries.

10. KindScript Could Fill the Gaps

The gaps in section 5 all share a common cause: tsc doesn't analyze function bodies for effects. KindScript does exactly that. Here's where they'd complement each other:

GapEffect-TS StatusKindScript Could
Side effects in Effect.sync Invisible Walk the AST inside the callback, flag IO/mutation/console
Unwrapped effectful calls Invisible Detect bare fetch(), fs.*, console.* outside Effect wrappers
Mutation inside closures Invisible Detect ++, =, delete in any function body
Purity verification Trust the wrapper Verify that "pure" functions actually are pure
Effect signature inference Only for wrapped code Infer Koka-like effect signatures from plain TS code
Architectural constraints Not in scope noImports, maxFanOut, noCycles, composites

The ideal combination: Effect-TS for runtime execution (fibers, retry, timeout, structured concurrency) + KindScript for static verification (actually checking that code does what the types claim).
