Custom Logic Integration

How KSC, JastAdd, Silver, Infer, CodeQL, and Datalog analyzers handle procedural code that doesn't fit into declarative specs — and what our options are.

The Problem

Core Tension
KSC's two-functor architecture separates declarative specs from generic machinery. But some code is neither — it's procedural runtime logic specific to a particular grammar target. Where does it live?

Consider what happens when we convert a TypeScript AST into our KSC grammar. For most fields, conversion is mechanical:

// Mechanical: just read the property from the TS node
escapedText: n.escapedText ?? "",
statements:  findChildrenOf(children, n.statements),
name:        findChild(children, n.name)!,

But some fields require real computation — calling into the TypeScript compiler API, walking parent chains, resolving symbols:

// Custom: requires TypeScript checker, symbol resolution, parent walking
resolvesToImport:  isImportReference(node),      // 25 lines of logic
isDefinitionSite:  checkIsDefinitionSite(node),  // 24 lines of parent checks
symIsVariable:     hasSymFlag(node, ts.SymbolFlags.Variable),
typeString:        getTypeString(node),           // needs type checker
resolvedFileName: getResolvedFileName(node),     // alias resolution

This procedural code is:

  • Not spec data — it's imperative TypeScript, not declarative configuration
  • Not generic machinery — it's specific to the TypeScript grammar target
  • Not generated — it's hand-written helper functions
  • Essential — without it, stamped fields would be empty

How KSC Currently Handles It

▶ Architecture: The Data Flow Current

KSC splits the problem into two halves that meet at code generation time:

The spec provides expression strings — references to functions that will exist at runtime. The compiler provides the function implementations — hardcoded in generateConvert().

▶ What The Spec Declares (extractors.ts)

Extractors.ts is pure data — a map from [nodeKind][fieldName] to expression strings:

export const BASE_EXTRACTORS: Record<string, Record<string, string>> = {
  PrefixUnaryExpression: {
    operator: 'prefixUnaryOperatorMap[n.operator]'      // ← string!
  },
  Identifier: {
    resolvesToImport: 'isImportReference(node)',         // ← string!

    symIsVariable: 'hasSymFlag(node, ts.SymbolFlags.Variable)',

  ImportDeclaration: {
    resolvedModulePath: 'getResolvedModulePath(node as any)'
  },
};

// Rule-based expansion: one expression applied to many kinds
kindRules: [
  { kinds: EXPORTED_DECLARATION_KINDS,   // 10 node kinds
    fieldName: 'isExported',
    expression: 'isNodeExported(node)' },
  { kinds: SCOPE_CONTAINER_KINDS,        // 11 node kinds

    expression: 'getLocalCount(node)' },

Key design
Specs only name the helpers they want to call. They don't define them. The expressions are opaque strings — the spec has no idea what isImportReference does internally.
▶ What The Compiler Hardcodes (compile.ts)

The generateConvert() function in grammar/compile.ts has ~200 lines of TypeScript-specific helper functions hardcoded as template strings. These get emitted verbatim into convert.ts:

// Inside generateConvert() in grammar/compile.ts:
L.push(`
function hasSymFlag(node: ts.Node, flag: ts.SymbolFlags): boolean {
  if (!_checker) return false;
  const sym = _checker.getSymbolAtLocation(node);
  return sym ? !!(sym.flags & flag) : false;
}

function checkIsDefinitionSite(node: ts.Node): boolean {
  if (!ts.isIdentifier(node)) return false;
  const parent = node.parent;
  if (!parent) return false;
  if (ts.isVariableDeclaration(parent) && parent.name === node) return true;
  if (ts.isParameter(parent) && parent.name === node) return true;
  // ... 15 more checks
  return false;
}

function isImportReference(node: ts.Node): boolean {
  if (!_checker || !ts.isIdentifier(node)) return false;
  const sym = _checker.getSymbolAtLocation(node);
  // ... 25 lines of import resolution logic
}

// ... operator maps, comment extraction, type resolution, etc.
`);
The problem
These ~200 lines are TypeScript-specific but live in generic grammar machinery. If we added a Python AST grammar, none of these helpers would apply. Yet grammar/compile.ts would still emit them into every grammar's convert.ts.

▶ Why This Matters

Today, KSC only has one grammar target (TypeScript AST). So the coupling is invisible. But the moment we add a second grammar target, the problem surfaces:

  1. compile.ts would emit TS helpers into non-TS grammars — a Python AST grammar doesn't need hasSymFlag or ts.SyntaxKind
  2. Adding helpers for a new grammar means modifying shared machinery — violating the open-closed principle
  3. No separation between "generic convert infrastructure" and "TS-specific stamping" — the WeakMap, child lookup, and dispatch are generic; the symbol flags and operator maps are not

The spec's expression strings and the compiler's function bodies are coupled by convention: if a spec references 'isImportReference(node)', the compiler must emit a function with that exact name. No type checking, no validation — just string matching at code generation time.

How JastAdd Handles Custom Logic

▶ JastAdd's Aspect System JastAdd

JastAdd is an attribute grammar system for Java that uses aspect-oriented programming to attach attributes and methods to AST nodes. It has three file types:

Grammar declarations (.ast)

Pure declarative node structure — similar to KSC's nodes.ts:

// TypeDecl.ast
abstract TypeDecl ::= <Name:String>;
ClassDecl : TypeDecl ::= SuperType:Access Body:BodyDecl*;
InterfaceDecl : TypeDecl ::= SuperInterface:Access* Body:BodyDecl*;

This is the grammar spec. It declares node kinds, inheritance, children, and token fields. No behavior whatsoever.

Declarative aspects (.jrag)

Attribute declarations and equations — the AG semantics layer. Equation bodies are arbitrary Java code:

// NameAnalysis.jrag
aspect NameAnalysis {
  // Attribute declaration
  inh TypeDecl Expr.lookupType(String name);

  // Equation — contains real Java code
  eq CompilationUnit.getChild().lookupType(String name) {
    for (TypeDecl t : getTypeDecls()) {
      if (t.getName().equals(name)) return t;
    }
    return unknownType();     // ← Java method call
  }

  syn boolean TypeDecl.isSubtypeOf(TypeDecl other) {
    if (this == other) return true;
    // Walk supertype chain — imperative Java
    TypeDecl sup = getSuperType().decl();
    while (sup != null) {
      if (sup == other) return true;
      sup = sup.getSuperType().decl();
    }
    return false;
  }
}
Key insight
In JastAdd, the equation body IS the custom logic. There's no separate "helper function" concept — the Java code in the equation IS the implementation. This is possible because equations are compiled into methods on the generated Java AST classes.

Imperative aspects (.jadd)

.jadd files add arbitrary methods, fields, and constructors to AST classes — essentially inter-type declarations:

// Helpers.jadd
aspect Helpers {
  // Add a method to ClassDecl
  public boolean ClassDecl.hasField(String name) {
    for (BodyDecl b : getBodyDecls()) {
      if (b instanceof FieldDecl && ((FieldDecl)b).getName().equals(name))
        return true;
    }
    return false;
  }

  // Add a field to CompilationUnit
  private Map<String, TypeDecl> CompilationUnit.typeCache = new HashMap<>();

  // Static helper
  public static String ASTNode.formatTypeName(String raw) {
    return raw.replace("$", ".");
  }
}

These methods and fields become part of the generated AST classes, available for equation bodies to call. This is how JastAdd handles "custom logic" that doesn't fit the equation pattern — you add helper methods to the nodes that need them.

JastAdd's answer
Custom logic lives IN the AST node classes as methods. Equations can call arbitrary Java code. Helper functions are added via inter-type declarations in .jadd aspects. Everything is woven into the generated classes at compile time.

How Silver Handles Custom Logic

▶ Silver's Three-Layer System Silver

Silver is a standalone AG language that compiles to Java. It separates declarations, equations, and helper functions into distinct language constructs:

Attribute declarations

Silver separates declaration from binding — what the attribute is vs. which nonterminals have it:

-- Declaration: the attribute exists
synthesized attribute pp :: String;
inherited attribute env :: Env;

-- Binding: which nonterminals have it
attribute pp occurs on Expr, Stmt, Decl;
attribute env occurs on Expr, Stmt;

This is purely declarative — names, types, and scoping. No behavior.

Equations in productions

Equations are defined inside production bodies. They can call any function in scope:

abstract production expr
top::Expr ::=

concrete production addExpr
top::Expr ::= l::Expr '+' r::Expr
{
  top.pp = l.pp ++ " + " ++ r.pp;
  l.env = top.env;
  r.env = top.env;

  -- Custom logic via function call
  top.errors = checkTypes(l.typerep, r.typerep, top.env)
               ++ l.errors ++ r.errors;
}

concrete production varRef
top::Expr ::= n::Name
{
  -- Calling a helper function
  top.typerep = lookupType(n, top.env);

  -- Complex logic inline
  top.errors =
    if containsKey(n.name, top.env)
    then []
    else [err(n.location, "Undefined variable: " ++ n.name)];
}

User-defined functions

Silver has first-class functions that can contain arbitrary logic. These serve as the equivalent of KSC's helper functions:

-- Pure function (no access to attributes)
function lookupType
Type ::= n::Name  env::Env
{
  return case lookup(n.name, env) of
    | just(t) -> t
    | nothing() -> errorType()
  end;
}

-- Function with pattern matching and recursion
function checkTypes
[Message] ::= left::Type  right::Type  env::Env
{
  return case left, right of
    | intType(), intType()     -> []
    | floatType(), floatType() -> []
    | _, _ -> [err("Type mismatch")]
  end;
}

-- IO function (side-effecting, explicit token)
function readSourceFile
IOVal<String> ::= path::String  ioin::IOToken
{
  return readFile(path, ioin);
}
Key insight
Silver functions are pure by default. Side effects require an explicit IOToken parameter — making purity a language-level guarantee. KSC's depth-aware helpers (getTypeString reads global _checker) would need to be threaded differently in Silver's model.

Foreign functions

Silver can call Java code via foreign function interface — this is how it integrates with existing tools (like a Java-based parser or type checker):

-- Foreign type: wraps a Java class
foreign type JavaTypeChecker;

-- Foreign function: calls into Java
function getTypeString
String ::= checker::JavaTypeChecker  node::JavaNode
{
  return foreign("com.example.TypeHelper.getTypeString",
                 checker, node);
}

-- Using it in an equation
concrete production typedExpr
top::Expr ::= e::Expr ':' t::Type
{
  top.typeString = getTypeString(top.checker, top.javaNode);
}

This is how Silver would handle the equivalent of KSC's TypeScript compiler API calls — wrapping them in foreign functions that equation bodies can call.

Silver's answer
Custom logic lives in standalone functions. Equations call them. Functions can be pure Silver or foreign (Java). Side effects are tracked at the type level. There's no special mechanism for "helper code" — it's just functions.

How Infer / Abstract Interpreters Handle It

Abstract interpretation frameworks face the same separation problem but at a different level: how to separate the analysis algorithm (fixpoint iteration, CFG traversal) from language-specific semantics (what does "+" mean? how do imports work?). Their solutions provide a powerful lens on KSC's problem.

▶ Facebook Infer: IR + Domain Functors Infer

Infer is Facebook's static analyzer for Java, C/C++, and Objective-C. Its architecture solves the custom logic problem with two clean separations:

Separation 1: Language frontends → SIL (intermediate representation)

Infer's frontend pipeline: Java bytecode ──→ Java frontend ──╲ C/C++ source ──→ Clang frontend ──→ SIL ──→ Analysis Framework Obj-C source ──→ Clang frontend ──╱ (Smallfoot IL) (language-agnostic) Hack source ──→ Textual format ──╱ .NET bytecode ──→ Infer# frontend ──╱ All language-specific concerns are resolved BEFORE analysis begins. Symbol resolution, type checking, import handling → all done by the frontend.

SIL (Smallfoot Intermediate Language) is a CFG-based IR that all frontends translate into. Once code is in SIL, the analysis framework has zero knowledge of the source language. This is the nuclear option for separation — language-specific code is physically unreachable from the analysis.

Separation 2: Analysis framework ← Domain + Transfer Functions

To write a new checker in Infer, you provide exactly two things — via OCaml functors:

(* The checker author provides: *)

(* 1. An abstract domain — type + lattice operations *)
module Domain : AbstractDomain.S = struct
  type t = ResourceSet.t          (* e.g., set of acquired resources *)
  let leq ~lhs ~rhs = ...        (* partial order *)
  let join a b = ...              (* least upper bound *)
  let widen ~prev ~next ~num_iters = ...
end

(* 2. Transfer functions — one handler per SIL instruction kind *)
module TransferFunctions : TransferFunctions.SIL = struct
  module CFG = ProcCfg.Normal     (* forward analysis *)
  module Domain = Domain

  let exec_instr (state : Domain.t) _analysis_data _node _idx (instr : Sil.instr) =
    match instr with
    | Call (_, procname, args, _, _) when is_acquire procname ->
        ResourceSet.add (resource_of args) state
    | Call (_, procname, args, _, _) when is_release procname ->
        ResourceSet.remove (resource_of args) state
    | _ -> state                  (* no effect *)
end

(* The framework provides everything else: *)
module Analyzer = AbstractInterpreter.MakeRPO(TransferFunctions)
(* → fixpoint iteration, CFG traversal, interprocedural summaries,
      worklist algorithm, widening schedule, error reporting *)
The OCaml functor pattern
AbstractInterpreter.MakeRPO is a functor (a function from modules to modules). It takes your TransferFunctions module and produces a complete analyzer. The framework is literally parameterized over your custom logic at the module level — not strings, not runtime dispatch, but compile-time module composition.

What Infer teaches us
Infer's frontends own all language-specific logic. The exec_instr transfer function dispatches on SIL instructions (language-agnostic), not on Java/C++ AST nodes. KSC's field extraction helpers (hasSymFlag, getTypeString) are the equivalent of Infer's frontend code — they translate from the source representation (TS AST) to the analysis representation (KS tree). They should live with the frontend (spec), not in the analysis framework (machinery).
▶ Astrée & SPARTA: Domain Composition Astrée

Astrée (used in Airbus flight control software verification) and SPARTA (Facebook's domain library, used by ReDex) show how to handle language-specific concerns through domain specialization rather than framework modification.

Astrée: Hierarchical reduced product

Astrée organizes abstract domains into a hierarchical network using OCaml functors:

(* Composing domains via binary product functor *)
module IntOct = BinaryProduct(Intervals)(Octagons)
(* Then composing further — recursively *)
module NumDomain = BinaryProduct(IntOct)(Congruences)
(* Adding C-specific domains alongside generic ones *)
module FullDomain = BinaryProduct(NumDomain)(DigitalFilters)

(* Each domain only defines its own operations.
   The product functor multiplexes communication channels
   so all domains can exchange information. *)

The key insight: rather than putting C-specific idioms (digital filters, floating-point IEEE 754 semantics, bitwise operations) into the analysis framework, Astrée creates specialized domains that plug into the same composition infrastructure as generic domains like intervals and octagons.

SPARTA: Reusable domain building blocks

SPARTA (Semantics, PARTitioning, Abstraction) is a C++ library providing composable domain components with zero language-specific knowledge:

// SPARTA provides generic building blocks:
PatriciaTreeMap<Variable, AbstractValue>   // memory-efficient environments
FiniteAbstractDomain<Sign>                 // finite lattices (⊥, -, 0, +, ⊤)
HashedSetAbstractDomain<Resource>           // power set domains
MonotonicFixpointIterator<CFG>              // generic over any graph type

// The tool developer composes them for their specific analysis:
using ResourceDomain = HashedSetAbstractDomain<FileHandle>;
using EnvDomain = PatriciaTreeMap<Register, ResourceDomain>;

// Language-specific semantics live in the transfer function:
EnvDomain analyze_instruction(const IRInstruction& insn, EnvDomain state) {
  // This is the custom logic — owned by the analysis author
  if (insn.is_open()) return state.set(insn.dest(), ResourceDomain::of(insn.arg()));
  if (insn.is_close()) return state.set(insn.dest(), ResourceDomain::bottom());
  return state;
}
What Astrée/SPARTA teach us
Domain specialization, not framework modification. Language-specific knowledge enters through specialized domain types and transfer functions that compose with generic infrastructure. The framework provides the iteration and composition mechanics — the spec author provides the semantics. This maps to KSC: grammar-specific helpers are semantic knowledge that should compose with the generic compilation framework, not be hardcoded inside it.
▶ Frama-C / Eva: Value vs State Abstraction

Frama-C's Eva plugin (Evolved Value Analysis) adds another dimension: separating value abstractions (what values can an expression take?) from state abstractions (how does a statement change the abstract state?).

(* Eva's two-layer architecture: *)

Value abstraction       (expression semantics)
  │  "x + y in interval [3, 7]"
  │  "p points to {a, b}"
  │
  ▼
State abstraction       (statement semantics)
  │  "after x = e, the env maps x → eval(e)"
  │  "after *p = v, all pointed-to locations may change"
  │
  ▼
Simple_memory functor  (builds state domain from value domain)
  Handles: memory model, variable-to-value mapping, store/load
  Provided by framework — domain author only defines value operations

The Simple_memory functor takes a value abstraction and builds a complete state domain, handling the boilerplate of memory modeling. This enabled five new abstract domains to be implemented in less than a year.

Parallel to KSC
KSC's Functor 2 does something analogous: it takes 8 property declarations and derives 17 additional attributes (context, violation, allViolations). The spec author defines the "value abstraction" (what properties mean), and the functor derives the "state abstraction" (how properties propagate through the tree, where violations occur). This is the same functor-as-boilerplate-eliminator pattern.

How CodeQL / Extractor Systems Handle It

CodeQL and Datalog-based analyzers use a fundamentally different strategy: they separate custom logic through a relational database boundary. Language-specific code runs before analysis, producing a uniform fact store that analysis queries declaratively.

▶ CodeQL: Extract → Database → Query CodeQL

CodeQL (GitHub/Semmle) is a semantic code analysis platform used for security vulnerability detection. Its architecture provides the clearest example of the extractor pattern.

The three-phase pipeline

The extractor IS the custom logic

CodeQL's extractors do exactly what KSC's field extraction helpers do — they read language-specific AST properties and stamp them into a uniform representation:

TRAP: Tuples Representing Abstract Properties

Extractors emit relational tuples — the resolved, flattened semantic data:

// exprs table: (id, kind, parent, index)
exprs(#12345, 3, #67890, 0)    // expression #12345 is kind 3 (VarAccess)

// type_of table: (expr, type)
type_of(#12345, #type_int)     // expression #12345 has type int

// name_binding: (use_site, definition_site)
uses(#12345, #decl_x)          // this var refers to declaration x

// literals: (expr, value, raw)
literals(#99999, "42", "42")   // literal value

Compare to KSC's stamped fields — these are the same data, just in relational form:

// KSC equivalent (on Identifier node):
{
  kind: 'Identifier',
  escapedText: 'x',              // ≈ TRAP name tuple
  resolvesToImport: true,        // ≈ TRAP uses/defs tuple
  symIsVariable: true,           // ≈ TRAP symbol_flags tuple
  typeString: 'number',          // ≈ TRAP type_of tuple
  isDefinitionSite: false,       // ≈ TRAP defs tuple
  resolvedFileName: 'lib.ts',    // ≈ TRAP source_location tuple
}

Extractor code (language-specific, owned by language team)

Each language's extractor contains all the procedural logic for reading that language's AST. Example pattern from Go extractor:

// Go extractor — resolves types and emits tuples
func (e *Extractor) extractExpr(expr ast.Expr) {
  id := e.newLabel()
  kind := exprKindOf(expr)
  parent := e.currentParent()

  // Emit the basic expression tuple
  e.writeTuple("exprs", id, kind, parent, e.childIndex)

  // Custom logic: type resolution (uses Go type checker)
  if typ := e.typeInfo.TypeOf(expr); typ != nil {
    typeId := e.extractType(typ)
    e.writeTuple("type_of", id, typeId)
  }

  // Custom logic: name binding (uses Go identifier resolution)
  if ident, ok := expr.(*ast.Ident); ok {
    if obj := e.typeInfo.Uses[ident]; obj != nil {
      declId := e.extractObject(obj)
      e.writeTuple("uses", id, declId)
    }
  }
}

This is the exact equivalent of KSC's hasSymFlag, getTypeString, etc. The difference: in CodeQL, this code lives in the extractor (language-specific), not in the query engine (generic machinery).

Shared dataflow framework

The generic analysis is parameterized via module signatures:

// shared/dataflow/DataFlowImpl.qll — SHARED across all languages
// Contains: path construction, call-site matching, field R/W pairing

// Each language implements this interface:
module DataFlowImplSpecific {
  module Public {
    class Node { ... }          // language-specific node type
  }
  module Private {
    predicate simpleLocalFlowStep(Node a, Node b) { ... }
    predicate isSource(Node n) { ... }
    predicate isSink(Node n) { ... }
  }
}

// The shared framework imports and uses it:
import DataFlowImplSpecific::Public
import DataFlowImplSpecific::Private

// Consistency checking: validates language-specific code
// DataFlowImplConsistency.qll verifies invariants

This is the same pattern as KSC's AnalysisSpec interface: the framework defines what it needs, each language provides the implementation.

CodeQL's answer
The extractor owns all custom logic. It's a standalone program that reads source code (using the language's own compiler/parser) and writes relational tuples. The analysis engine never touches source AST nodes. Language-specific semantics enter through two channels: (1) the extractor (which produces resolved, flattened data), and (2) QL library packs (which provide high-level abstractions over the database tables).

How Doop / Datalog Analyzers Handle It

▶ Doop: Facts vs Rules Datalog

Doop (the premier points-to analysis framework for Java) makes the cleanest possible separation: facts (data extracted from programs) vs rules (declarative analysis logic).

Java bytecode / source │ ▼ ┌─────────────────────────────────────────┐ │ Soot Fact Generator │ │ │ │ bytecode → Jimple (3-addr code) → TSV │ │ │ │ Language-specific: one standalone binary │ │ Zero knowledge of analysis algorithms │ └───────────────────┬─────────────────────┘ │ TSV fact files ▼ ┌─────────────────────────────────────────┐ │ Fact Schema │ │ │ │ AssignHeapAllocation(insn, idx, │ │ heap, var, meth) │ │ Assign(to, from) │ │ FieldPointsTo(heap, field, value) │ │ Var_DeclaringMethod(var, method) │ │ │ │ Uniform relational interface │ └───────────────────┬─────────────────────┘ │ ▼ ┌─────────────────────────────────────────┐ │ Datalog Analysis Rules │ │ │ │ VarPointsTo(var, heap) :- │ │ AssignHeapAllocation(heap, var, m), │ │ Reachable(m). │ │ │ │ VarPointsTo(to, heap) :- │ │ Assign(to, from), │ │ VarPointsTo(from, heap). │ │ │ │ Pure Datalog — no procedural code │ │ Compiled to C++ by Soufflé engine │ └─────────────────────────────────────────┘

Why this matters

In Doop, the fact generator and the analysis rules are completely independent programs. The fact generator is a Java program that uses Soot to walk bytecode. The analysis rules are Datalog files compiled by Soufflé to C++. They communicate only through the fact schema (TSV files). You can swap fact generators (Soot → WALA → Dex) without touching analysis rules.

Custom logic for handling language-specific complexity (reflection, native methods, threading) is expressed as more Datalog rules, not as procedural code mixed into the framework:

// Java reflection handling — still declarative Datalog

  CallGraphEdge(_, callsite, _, method),
  MethodName(method, "newInstance"),
  ActualParam(callsite, 0, classNameVar),
  VarPointsTo(classNameVar, classNameHeap),
  StringConstant(classNameHeap, className),
  HeapAllocationType(heap, className).

// Context sensitivity — just different rule instantiation
// 1-call-site-sensitive, 2-object-sensitive, etc.
// Same core rules, different context definitions
Doop's answer
Hard boundary between extraction and analysis. The fact generator is the only place where language-specific procedural code lives. Analysis is purely declarative. Custom analysis extensions are also declarative (more Datalog rules). The fact schema is the contract between the two worlds. When ported from LogiQL to Soufflé, the analysis rules transferred almost mechanically — gaining 4x speedup — because declarative specs are engine-independent.

The Universal Pattern

Every system agrees on one thing

Across AG systems (JastAdd, Silver), abstract interpreters (Infer, Astrée, Eva, IKOS), and relational analyzers (CodeQL, Doop), one pattern is universal: language-specific procedural code should be owned by the frontend/spec, not by the analysis framework.

The framework provides composition mechanics (functors, fixpoint iteration, code generation). The spec provides semantics (transfer functions, equations, extractors). The contract between them is a typed interface — TransferFunctions.SIL in Infer, DataFlowImplSpecific in CodeQL, AnalysisSpec/GrammarSpec in KSC.

The common pattern
Every system — AG systems (JastAdd, Silver), abstract interpreters (Infer, Astrée), and relational analyzers (CodeQL, Doop) — agrees: custom logic is owned by the spec/frontend author, not the analysis framework. KSC is the outlier. The frameworks provide composition mechanics (functors, fixpoint iteration, code generation). The spec/frontend provides semantics (transfer functions, equations, extractors, fact generators). The contract between them is a typed interface.

Our Options

There are five viable approaches, ranging from minimal changes to architectural redesign. Each draws on patterns from the systems above. Evaluated against three criteria: pluggability (can a new grammar add helpers without touching machinery?), type safety (are helper references validated?), and simplicity (how much complexity does it add?).

▶ Option A: Preamble Provider Option

Specs provide helper function source code as a string alongside their expression data. The compiler emits whatever preamble the spec provides.

How it works

// GrammarSpec gains a new field:
interface GrammarSpec {
  nodes: ...;
  sumTypes: ...;
  fieldExtractors: Record<string, Record<string, string>>;
  convertPreamble?: string;  // ← NEW: helper function source code
  skipConvert?: Set<string>;
}

// specs/ts-ast/grammar/spec.ts provides it:
export function buildGrammarSpec(builder: GrammarBuilder): GrammarSpec {
  return {
    ...
    convertPreamble: TS_CONVERT_HELPERS,  // ← string of function definitions
  };
}

// specs/ts-ast/grammar/helpers.ts (new file):
export const TS_CONVERT_HELPERS = `
function hasSymFlag(node: ts.Node, flag: ts.SymbolFlags): boolean {
  if (!_checker) return false;
  const sym = _checker.getSymbolAtLocation(node);
  return sym ? !!(sym.flags & flag) : false;
}

function checkIsDefinitionSite(node: ts.Node): boolean {
  // ... full implementation
}
// ... all other helpers
`;

// grammar/compile.ts becomes generic:
function generateConvert(ctx: CompileCtx): string {
  const L: string[] = [];
  L.push(CONVERT_INFRASTRUCTURE);   // Generic: WeakMap, dispatch, etc.
  if (ctx.convertPreamble) {
    L.push(ctx.convertPreamble);     // ← Spec-provided helpers
  }
  // ... generate converters using fieldExtractors as before
}
Pros
  • Minimal change — just move strings from compile.ts to spec
  • Fully pluggable — each grammar provides its own helpers
  • compile.ts becomes truly generic
  • No new abstractions or APIs
Cons
  • Helpers are still strings — no type checking until codegen runs
  • Large string blocks are hard to edit/maintain
  • No reuse of common patterns between grammars
  • IDE support (autocomplete, go-to-definition) doesn't work in strings
▶ Option B: Runtime Helper Module Option Recommended

Instead of emitting helpers inline, specs provide a runtime module that the generated convert.ts imports from. Helpers are real TypeScript functions — not strings.

How it works

// GrammarSpec gains a helper module path:
interface GrammarSpec {
  nodes: ...;
  fieldExtractors: Record<string, Record<string, string>>;
  convertHelpers?: {
    modulePath: string;           // import path for generated convert.ts
    exportedNames: string[];      // names to import
    preamble?: string;            // any inline code still needed
  };
}

// specs/ts-ast/grammar/convert-helpers.ts (NEW — real TypeScript file):
import ts from 'typescript';
import type * as KS from '../../../generated/ts-ast/grammar/node-types.js';

// These are real functions — full IDE support, type checking
export function hasSymFlag(
  checker: ts.TypeChecker | undefined,
  node: ts.Node,
  flag: ts.SymbolFlags,
): boolean {
  if (!checker) return false;
  const sym = checker.getSymbolAtLocation(node);
  return sym ? !!(sym.flags & flag) : false;
}

export function checkIsDefinitionSite(node: ts.Node): boolean {
  if (!ts.isIdentifier(node)) return false;
  const parent = node.parent;
  if (!parent) return false;
  if (ts.isVariableDeclaration(parent) && parent.name === node) return true;
  // ...
}

export const prefixUnaryOperatorMap: Record<number, string> = {
  [ts.SyntaxKind.PlusToken]: '+',
  [ts.SyntaxKind.MinusToken]: '-',
  // ...
};

// ... all other helpers as real exported functions/values

// specs/ts-ast/grammar/spec.ts wires it:
export function buildGrammarSpec(builder: GrammarBuilder): GrammarSpec {
  return {
    ...
    convertHelpers: {
      modulePath: '../../../specs/ts-ast/grammar/convert-helpers.js',
      exportedNames: [
        'hasSymFlag', 'checkIsDefinitionSite', 'isImportReference',
        'getTypeString', 'getResolvedFileName', 'isNodeExported',
        'getLocalCount', 'getResolvedModulePath', 'getImportModuleSpecifier',
        'prefixUnaryOperatorMap', 'postfixUnaryOperatorMap',
        'typeOperatorMap', 'heritageTokenMap', 'metaPropertyKeywordMap',
        'getDeclarationKind',
      ],
    },
  };
}

// generated convert.ts now imports instead of defining:
import {
  hasSymFlag, checkIsDefinitionSite, isImportReference,
  getTypeString, ...
} from '../../../specs/ts-ast/grammar/convert-helpers.js';

What about global state (_checker, _depth)?

The runtime module needs access to the checker and depth. Two sub-options:

B1: Init function pattern

The helpers module exports an init() function. The generated converter calls it once before starting conversion:

// convert-helpers.ts
let _checker: ts.TypeChecker | undefined;
let _depth: AnalysisDepth = 'check';

export function initHelpers(checker: ts.TypeChecker | undefined, depth: AnalysisDepth): void {
  _checker = checker;
  _depth = depth;
}

export function hasSymFlag(node: ts.Node, flag: ts.SymbolFlags): boolean {
  if (!_checker) return false;
  // ...
}

// generated convert.ts calls:
export function buildKSTree(program: ts.Program, depth: AnalysisDepth): KSTree {
  initHelpers(program.getTypeChecker(), depth);  // ← init from external module
  // ... convert source files
}

This is closest to the current design — same mutable state, just in a different file.

B2: Parameter threading

Pass a context object to each helper. Purer, but requires expression strings to include the context parameter:

// convert-helpers.ts
export interface ConvertContext {
  checker?: ts.TypeChecker;
  depth: AnalysisDepth;
}

export function hasSymFlag(ctx: ConvertContext, node: ts.Node, flag: ts.SymbolFlags): boolean {
  if (!ctx.checker) return false;
  const sym = ctx.checker.getSymbolAtLocation(node);
  return sym ? !!(sym.flags & flag) : false;
}

// extractors.ts changes expression strings:
Identifier: {
  symIsVariable: 'hasSymFlag(_ctx, node, ts.SymbolFlags.Variable)',
  //              ^^^^ context parameter
}

// generated convert.ts threads context:
const _ctx: ConvertContext = { checker, depth };
register(ts.SyntaxKind.Identifier, (node, sf, children, pos, end, text) => {
  const n = node as any;
  return {
    symIsVariable: hasSymFlag(_ctx, node, ts.SymbolFlags.Variable),
    // ...
  };
});
Pros
  • Real TypeScript — full IDE support, type checking, refactoring
  • Fully pluggable — each grammar provides its own module
  • Helpers can be tested independently (import and unit test)
  • compile.ts becomes truly generic
  • Clear ownership: helpers live next to the spec that uses them
Cons
  • Expression strings still reference function names by convention
  • Generated code depends on non-generated source file (import path coupling)
  • Need to handle global state init or parameter threading
  • Slightly more complex GrammarSpec interface
Why recommended
This is the Silver pattern adapted to TypeScript: helper functions are standalone, typed, testable, and owned by the spec. The generated code imports them instead of defining them. It's the smallest change that gives the most benefit — expressions in specs still work as before, but the function implementations move from compiler internals to spec-owned modules.
▶ Option C: Converter Plugin Architecture Option

Replace expression strings with a proper plugin API. Specs provide converter functions directly — no string indirection.

How it works

// GrammarSpec uses typed extractors instead of strings:
interface FieldExtractor {
  (node: unknown, checker?: unknown): unknown;
}

interface GrammarSpec {
  nodes: ...;
  fieldExtractors: Record<string, Record<string, FieldExtractor>>;  // ← functions, not strings
}

// specs/ts-ast/grammar/extractors.ts returns actual functions:
import ts from 'typescript';

export const BASE_EXTRACTORS: Record<string, Record<string, FieldExtractor>> = {
  Identifier: {
    resolvesToImport: (node, checker) => {
      const tsNode = node as ts.Node;
      if (!checker || !ts.isIdentifier(tsNode)) return false;
      const sym = (checker as ts.TypeChecker).getSymbolAtLocation(tsNode);
      // ... full logic inline
      return true;
    },
  },
};

// generated convert.ts calls extractors at runtime:
register(ts.SyntaxKind.Identifier, (node, sf, children, pos, end, text) => ({
  kind: 'Identifier',
  resolvesToImport: fieldExtractors.Identifier.resolvesToImport(node, _checker),
  // ...
}));
Pros
  • No string indirection — fully typed end-to-end
  • Extractors are testable without codegen
  • compile.ts is completely generic
  • Like JastAdd's approach — logic lives with the spec data
Cons
  • Major architectural change — extractors go from compile-time strings to runtime functions
  • Generated convert.ts would need to import the spec module at runtime (circular dependency risk)
  • Performance: function call overhead per field per node (vs. inlined code)
  • Loses the "all-in-one generated file" property — convert.ts needs imports
  • Type safety of extractors requires complex generics (node type varies by kind)
▶ Option D: Two-Phase Separation Option

Separate "generic tree conversion" from "semantic stamping." Phase 1 builds a bare AST (children + props from direct access). Phase 2 runs spec-provided enrichment passes.

How it works

// Phase 1: generic, generated by compile.ts
// Only handles: kind, children, direct properties (n.escapedText, n.flags, etc.)
// NO checker, NO symbol resolution, NO type strings
register(ts.SyntaxKind.Identifier, (node, sf, children, pos, end, text) => ({
  kind: 'Identifier',
  escapedText: (node as any).escapedText ?? "",
  pos, end, text, children,
}));

// Phase 2: spec-provided enrichment (runs after tree is built)
interface EnrichmentPass {
  name: string;
  requires: 'parse' | 'bind' | 'check';
  enrich(tree: KSTree, checker?: ts.TypeChecker): void;
}

// specs/ts-ast/grammar/enrichment.ts
export const tsEnrichments: EnrichmentPass[] = [
  {
    name: 'symbolFlags',
    requires: 'bind',
    enrich(tree, checker) {
      walkTree(tree.root, (node) => {
        if (node.kind === 'Identifier' && checker) {
          const tsNode = ...; // need TS→KS mapping
          const sym = checker.getSymbolAtLocation(tsNode);
          (node as any).symIsVariable = !!(sym?.flags & ts.SymbolFlags.Variable);
          // ... stamp all sym* fields
        }
      });
    },
  },
  {
    name: 'typeStrings',
    requires: 'check',
    enrich(tree, checker) {
      // Walk and stamp typeString on all applicable nodes
    },
  },
];
Pros
  • Clean separation: generic tree building vs. semantic enrichment
  • Enrichment passes are self-contained and testable
  • Phase 1 is the same for any grammar target
  • Naturally handles depth-aware execution (skip check-level passes at bind depth)
Cons
  • Largest architectural change — fundamental rethink of conversion
  • Need TS→KS node mapping for enrichment (currently WeakMap is internal to convert.ts)
  • Two tree walks instead of one (performance concern for large codebases)
  • Enrichment must mutate nodes (violates immutability principle) or rebuild them
  • Some fields are entangled (e.g., resolvesToImport + importModuleSpecifier both need the same symbol lookup)
▶ Option E: Extractor Module (Infer/CodeQL-style) Option

Inspired by Infer's frontend pattern and CodeQL's extractor architecture: treat the grammar spec as a complete frontend module that includes not just data declarations but also a self-contained extractor. The spec provides a typed buildTree function that the generated code delegates to.

How it works

// GrammarSpec gains a typed extractor interface:
interface GrammarSpec {
  nodes: ...;
  sumTypes: ...;
  fieldExtractors: Record<string, Record<string, string>>;

  // NEW: extractor module — the spec provides tree building
  extractorModule?: {
    modulePath: string;        // import path
    buildTreeFn: string;       // exported function name
    initFn?: string;           // optional init function
    helperExports?: string[];  // helpers for expression strings
  };
}

// specs/ts-ast/grammar/extractor.ts (NEW — the "frontend"):
import ts from 'typescript';
import type * as KS from '../../../generated/ts-ast/grammar/node-types.js';

// ── State ──
let _checker: ts.TypeChecker | undefined;
let _depth: AnalysisDepth = 'check';

export function init(checker: ts.TypeChecker | undefined, depth: AnalysisDepth) {
  _checker = checker;
  _depth = depth;
}

// ── All language-specific helpers ──
export function hasSymFlag(node: ts.Node, flag: ts.SymbolFlags): boolean { ... }
export function checkIsDefinitionSite(node: ts.Node): boolean { ... }
export function isImportReference(node: ts.Node): boolean { ... }
export function getTypeString(node: ts.Node): string { ... }
// ... all helpers

export const prefixUnaryOperatorMap: Record<number, string> = { ... };
// ... all operator maps

// ── Optional: full buildKSTree if the spec wants to own the entry point ──
export { buildKSTree } from './tree-builder.js';

// grammar/compile.ts generates:
//   import { init, hasSymFlag, ... } from '<extractorModule.modulePath>';
//   + generic dispatch infrastructure
//   + per-kind converters using expression strings (unchanged)

This is essentially Option B but framed as a coherent "extractor module" concept — matching how Infer frontends and CodeQL extractors are self-contained units that handle all language-specific concerns. The spec owns the entire frontend: data declarations, extraction helpers, and optionally the entry point.

Pros
  • Conceptual clarity — the spec IS the frontend/extractor (like Infer/CodeQL)
  • All language-specific code in one place: specs/ts-ast/grammar/
  • Helpers are real TypeScript — full IDE support and type checking
  • compile.ts becomes truly generic — only emits dispatch + converter scaffolding
  • The extractor module is independently testable
  • Clear contract: extractorModule in GrammarSpec defines the interface
Cons
  • Essentially Option B with a name — same mechanics, different framing
  • Same expression-string coupling (helpers referenced by name in strings)
  • If the spec owns buildKSTree, the generated convert.ts becomes more of a library than a standalone module
When to choose E over B

Option E is Option B taken to its logical conclusion. The key difference is scope: Option B moves helpers out of compile.ts. Option E reframes the spec as a complete "frontend module" — the spec owns not just data but the entire language-specific pipeline. Choose E if you want to make the architecture explicit about what the spec is: a frontend that translates from source-language representation to KSC's uniform AST.

Recommendation

Option B/E (Runtime Helper Module / Extractor Module) hits the sweet spot. It makes compile.ts generic without requiring a major architectural change. The key insight from AG systems (JastAdd, Silver), abstract interpreters (Infer, Astrée), and relational analyzers (CodeQL, Doop) is the same: language-specific procedural code should be owned by the frontend/spec, not by the analysis framework.

Option B achieves this while keeping the expression-string mechanism that works well for field extraction mapping. Option E is the same mechanics but reframes the spec as a complete "frontend module" — matching how every other system in this survey structures the separation.

Option A is the quick fix if you want minimal change. Options C and D are worth considering only if you're planning to support many grammar targets with fundamentally different conversion needs.

Quiz

Score: 0 / 10
In KSC, where do the ~200 lines of TypeScript-specific helper functions currently live?
In grammar/compile.ts — the generic grammar compilation machinery. They're hardcoded as template strings inside the generateConvert() function and emitted verbatim into the generated convert.ts file. This makes the "generic" compiler TS-specific.
How does JastAdd let spec authors add custom helper methods to AST node classes?
Via .jadd aspect files (inter-type declarations). These files can add arbitrary Java methods, fields, and constructors to generated AST classes. The methods are woven into the classes at compile time and are available for equation bodies in .jrag files to call.
What mechanism does Silver use for calling non-Silver code (e.g., a Java library)?
Foreign functions and foreign types. Silver can declare a foreign type wrapping a Java class, then define Silver functions that delegate to Java methods via a foreign() call. This is analogous to KSC's helpers calling the TypeScript compiler API.
In KSC's current design, how are spec expression strings and compiler helper functions connected?
By naming convention only. The spec writes 'isImportReference(node)' as a string, and the compiler emits a function named isImportReference into the same file. There's no type checking, no validation — just string matching at code generation time. If the names don't match, you get a runtime error in the generated code.
How does Facebook Infer separate language-specific code from its analysis framework?
Two separations: (1) Language frontends translate to SIL (Smallfoot Intermediate Language), a shared CFG-based IR. All language-specific parsing, type resolution, and import handling happens in the frontend before analysis begins. (2) OCaml functors parameterize the framework — checker authors provide a Domain module (lattice) and TransferFunctions module (exec_instr handler), and AbstractInterpreter.MakeRPO generates the complete analyzer.
What is a TRAP file in CodeQL, and what is its equivalent in KSC?
TRAP = Tuples Representing Abstract Properties. CodeQL extractors emit these relational tuples containing resolved semantic data (types, name bindings, literal values). In KSC, the equivalent is the stamped fields on KS AST nodes — fields like symIsVariable, typeString, and resolvesToImport that are extracted from the TypeScript AST during conversion. Both represent "resolved, flattened semantic data" written by a language-specific extractor into a uniform representation.
How does Doop (Datalog-based analyzer) separate language-specific from analysis-specific code?
Hard boundary via fact schema. A standalone Soot-based fact generator (Java program) reads bytecode and emits TSV fact files. Analysis rules are pure Datalog (compiled to C++ by Soufflé). The two are completely independent programs communicating only through the fact schema. Custom analysis extensions are also declarative (more Datalog rules), not procedural code. When Doop was ported from LogiQL to Soufflé, analysis rules transferred mechanically with 4x speedup.
What does Astrée use instead of modifying the framework to handle C-specific idioms?
Specialized abstract domains that plug into the same hierarchical product as generic domains. Rather than putting C-specific logic (digital filters, IEEE 754 floats, bitwise operations) into the analysis framework, Astrée creates domain modules that compose via binary product functors. The framework provides the composition mechanics; the domains provide the semantics. This is the "domain specialization, not framework modification" pattern.
Why is Option B (Runtime Helper Module) recommended over Option C (Converter Plugin)?
Option B keeps the existing expression-string mechanism (which works well for the declarative mapping from fields to extraction expressions) while moving function implementations into real, typed, testable TypeScript files. Option C would require replacing expression strings with runtime function objects, which is a larger architectural change with potential performance overhead (function calls instead of inlined code) and circular dependency risks (generated code importing spec code that imports generated types).
What is the one insight shared by ALL systems surveyed (JastAdd, Silver, Infer, CodeQL, Doop, Astrée)?
Language-specific procedural code should be owned by the spec/frontend author, not the analysis framework. The framework provides composition mechanics (functors, fixpoint iteration, code generation). The spec/frontend provides semantics (transfer functions, equations, extractors, fact generators). The contract between them is a typed interface. KSC is the only system where custom logic lives in the compiler machinery instead of alongside the spec.
