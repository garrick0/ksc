The Big Picture: How Everything Relates

Map

There are roughly 12 distinct paradigms for static analysis. Some are theories (mathematical frameworks), some are methodologies (engineering approaches), and some are both. Many overlap; several subsume others. Here's the map:

Paradigm Core Question Foundation Soundness Completeness
Abstract Interpretation What values can arise at this point? Lattice theory, Galois connections Sound Incomplete (over-approximate)
Model Checking Does this system satisfy this temporal property? Temporal logic, automata theory Sound (on model) Complete (on model)
Type Systems Is this program well-typed? Lambda calculus, proof theory Sound (if sound type system) Incomplete (conservative)
Hoare / Separation Logic Does this code satisfy its specification? Predicate logic, axiomatic semantics Sound Relatively complete
Dataflow Analysis What facts reach this point along which paths? Lattice fixpoints, graph algorithms Sound Incomplete
Datalog / Query-Based Do these structural/flow patterns exist? Relational algebra, logic programming Depends on query Depends on query
Attribute Grammars What properties can be computed on this tree? Formal grammars, fixpoints on trees Sound (by construction) Depends on attributes
Symbolic Execution What inputs trigger this path/error? Constraint solving (SMT/SAT) Unsound (path explosion) Complete on explored paths
Algebraic Program Analysis What is the path summary over this algebra? Kleene algebra, semirings Sound Depends on algebra
Graph / Reachability Can information flow from A to B? CFL-reachability, graph algorithms Depends on model Depends on model
Incorrectness Logic Can this bug actually happen? Under-approximate logic Unsound (under-approximate) No false positives
ML / Neural Does this code look like a bug? Statistical learning Neither sound nor complete Neither sound nor complete
The Fundamental Tradeoff (Rice's Theorem) No static analysis can be simultaneously sound (no false negatives), complete (no false positives), and terminating for all programs. Every framework makes a different choice about which property to sacrifice. Abstract interpretation sacrifices completeness. Symbolic execution sacrifices termination. ML sacrifices both soundness and completeness in exchange for speed and pattern recognition.

The Subsumption Hierarchy

Cousot showed that abstract interpretation is the most general framework — most other sound analyses can be reformulated as abstract interpretations over particular domains:

  • Type checking = abstract interpretation over a type lattice (Cousot 1997)
  • Dataflow analysis = abstract interpretation over finite lattices (Cousot & Cousot 1977)
  • Model checking = abstract interpretation over trace semantics (Cousot 2000, Clarke & Grumberg)
  • Hoare logic = abstract interpretation with predicate abstraction

This doesn't mean abstract interpretation is always the best engineering choice — just that it's the most general theoretical framework. The specialized frameworks often have better algorithms for their specific problem shapes.

1. Classical Dataflow Analysis

Semantic

The oldest framework. Predates abstract interpretation (Cousot explicitly generalized it). Computes facts that hold at each program point by propagating information along control flow edges.

Theoretical Foundation

Kildall (1973) formulated dataflow analysis as a fixpoint problem over a lattice. Kam & Ullman (1977) defined the monotone framework: given a lattice (L, ⊓) and monotone transfer functions fn for each CFG node, iterate until fixpoint. The result is the MFP (maximal fixpoint) solution.

The Gen/Kill Framework

A special case where transfer functions have the form f(x) = (x − Kill) ∪ Gen. This makes the functions distributive (not just monotone), which means MFP = MOP (meet-over-all-paths), i.e. the iterative solution is as precise as considering every path independently.

// Reaching Definitions: which assignments might reach this point?
// Lattice: powerset of all definitions, ordered by ⊇
// Transfer function per statement s:
function transfer(s, inFacts) {
  return (inFacts - kill(s)) ∪ gen(s);
  // kill(s) = definitions of same variable overwritten by s
  // gen(s) = the definition introduced by s
}
// Iterate to fixpoint over the CFG

Classic Analyses

AnalysisDirectionLatticeWhat It Computes
Reaching DefinitionsForwardPowerset of defsWhich assignments might reach a point
Live VariablesBackwardPowerset of varsWhich variables might be used later
Available ExpressionsForwardPowerset of exprsWhich expressions are already computed
Very Busy ExpressionsBackwardPowerset of exprsWhich expressions will definitely be used
Constant PropagationForwardVars → {⊥, constants, ⊤}Which variables have constant values

Relationship to Abstract Interpretation

Dataflow ⊂ Abstract Interpretation Every dataflow analysis is an abstract interpretation, but not vice versa. Dataflow analyses use finite lattices and distributive transfer functions. Abstract interpretation adds: infinite lattices, non-distributive functions, widening/narrowing, and the Galois connection theory. The moment you need numeric ranges (intervals, polyhedra) or relational properties, you've left classical dataflow and entered abstract interpretation territory.

Tools

Classical dataflow is not a standalone tool category — it's embedded in every compiler (GCC, LLVM, javac, tsc). It powers optimizations (dead code elimination, register allocation, common subexpression elimination) and is the foundation that everything else builds on.

2. Model Checking

Exploration

Where abstract interpretation approximates reachable states, model checking exhaustively explores them. It answers: does every execution of this system satisfy a temporal logic property?

Core Idea

Given a finite-state model M and a temporal logic formula φ, determine whether M ⊨ φ. The model is typically a Kripke structure (states + transitions + labeling), and φ is in LTL (Linear Temporal Logic) or CTL (Computation Tree Logic).

// LTL property: "every request is eventually granted"
G(request → F grant)

// CTL property: "it's always possible to reach a safe state"
AG(EF safe)

// Model checker either:
// (a) proves the property holds on all reachable states, or
// (b) produces a COUNTEREXAMPLE: a concrete execution trace violating it

Variants

Explicit-State (SPIN)

Enumerates reachable states directly. Uses partial-order reduction and state hashing to manage state space. Best for protocol/concurrency verification.

SPIN (Holzmann, 1997): models in Promela, verifies LTL properties. Won 2001 ACM Software System Award.

Symbolic (BDD-based)

Represents state sets as BDDs (Binary Decision Diagrams), computing transitions on entire sets at once. Handles 1020+ state systems.

SMV/NuSMV (McMillan, 1993): the original symbolic model checker. CTL properties on hardware and protocols.

Bounded (BMC)

Unrolls the transition relation k times and checks if a property violation exists within k steps. Encodes as SAT/SMT problem.

CBMC (Clarke et al.): bounded model checker for C/C++. Finds real bugs but can't prove absence. KLEE: symbolic execution engine on LLVM (related, sometimes classified here).

IC3/PDR

Property-Directed Reachability (Bradley, 2011): incremental algorithm that proves safety without exploring the full state space. Finds inductive invariants by analyzing individual counterexample-to-induction traces.

Used by SeaHorn, Spacer, and modern hardware verification tools.

TLA+ (Lamport)

Deserves special mention: TLA+ (Temporal Logic of Actions) by Leslie Lamport is both a specification language and a model-checking framework. Widely used at Amazon (AWS), Microsoft, and other companies for distributed systems design.

The TLC model checker exhaustively checks TLA+ specifications. Apalache adds symbolic (SMT-based) model checking for larger state spaces.

Relationship to Abstract Interpretation

Two Sides of the Same Fixpoint Cousot showed (2000) that model checking can be viewed as abstract interpretation over trace abstractions. Both compute fixpoints — abstract interpretation iterates transfer functions to a post-fixpoint, model checking iterates a transition relation to enumerate reachable states. The key difference: model checking is exact on its model (complete) but requires finite state; abstract interpretation handles infinite state but loses precision. CEGAR bridges them — it's a model checking technique that refines the abstraction (an abstract interpretation concept).

3. Type Systems & Effect Systems

Type Theory

The most ubiquitous static analysis in practice. Every statically-typed language performs a type-based static analysis on every compilation. Cousot (1997) proved type systems are abstract interpretations — but type theory has its own rich tradition independent of Cousot.

The Hierarchy of Type Systems

Type systems themselves form a lattice of precision, mirroring Cousot's abstraction hierarchy:

Key Systems in Detail

Rust's Borrow Checker: Affine Types + Regions

Rust's ownership system is a static analysis for memory safety encoded as a type system. It enforces:

  • Ownership: each value has exactly one owner
  • Borrowing: either one mutable reference (&mut) XOR any number of shared references (&)
  • Lifetimes: references cannot outlive their referent

This is an affine type system (values used at most once, unless explicitly copied) combined with region inference (lifetimes are regions). The borrow checker is, in abstract interpretation terms, a very precise shape/alias analysis encoded directly into the type system.

Result: memory safety without garbage collection — zero-cost at runtime.

Refinement Types: Types + Predicates

Refinement types augment base types with logical predicates: {v: Int | v > 0} is "positive integers." The type checker uses an SMT solver to discharge proof obligations.

  • Liquid Haskell: refinement types for Haskell. Infers refinements via abstract interpretation over liquid types (predicate abstraction). Checks array bounds, termination, totality.
  • F* (Microsoft Research): dependent + refinement types for verification-oriented programming. Generates proof obligations discharged by Z3.
  • RefinedC: refinement types for C, proving memory safety and functional correctness.
Where Refinement Types Meet Abstract Interpretation Liquid types (Rondon, Kawaguchi, Jhala, 2008) are explicitly framed as abstract interpretation: the abstract domain is the set of all conjunctions of predicates from a given template set. The fixpoint solver is Liquid type inference. This is abstract interpretation with predicate abstraction over a type-system-shaped lattice.

Effect Systems: Tracking What Functions Do

Where types describe what a value is, effects describe what a computation does: IO, exceptions, mutation, non-determinism, etc.

  • Koka (Leijen, Microsoft Research): algebraic effects with handlers. Effect types inferred automatically.
  • Eff (Bauer & Pretnar): research language for algebraic effects.
  • Java's checked exceptions: a primitive effect system (tracks which exceptions can be thrown).
  • Scala 3 CanThrow: capability-based effect tracking.

Effect systems are directly relevant to KSC's property checking — "purity" and "no side effects" are effect properties that an effect system would check via types rather than abstract interpretation.

Dependent Types: Types as Proofs

In dependent type theory, types can depend on runtime values: Vec n a is a vector of exactly n elements of type a. The type checker becomes a theorem prover (Curry-Howard correspondence).

  • Idris 2: dependently typed, practically oriented (compiles to Scheme/JS)
  • Agda: proof assistant, total language
  • Lean 4: proof assistant + general-purpose language (used to formalize Mathlib)
  • Coq/Rocq: the original proof assistant, used to verify CompCert C compiler

Dependent types are the most precise point on the type system lattice — expressive enough to prove arbitrary theorems, but type-checking is undecidable in general (the user provides proofs to guide the checker).

4. Program Logics: Hoare, Separation, and Incorrectness

Logic

Program logics give axiomatic specifications of program behavior. The program is correct if its behavior satisfies a logical specification. The three major branches represent three distinct philosophies:

Hoare Logic (1969)

The foundation of program verification. A Hoare triple {P} C {Q} means: if precondition P holds before executing command C, then postcondition Q holds after.

// {x = 5}
x = x + 1;
// {x = 6}

Weakest precondition (Dijkstra, 1975): given Q, compute the weakest P such that {P} C {Q}. Computes backward from postcondition to precondition. Basis for tools like Dafny, Why3, Boogie.

Separation Logic (2002)

O'Hearn, Reynolds, Yang extended Hoare logic with the separating conjunction (P * Q): P and Q hold on disjoint regions of the heap. This enables local reasoning — verify heap-manipulating code by reasoning about only the footprint it touches.

Frame rule: {P} C {Q} implies {P * R} C {Q * R} — if C doesn't touch R's memory, R is preserved. This is what makes compositional analysis possible.

Powers: Infer (Meta), VeriFast, Viper.

Incorrectness Logic (2019)

O'Hearn inverted Hoare logic: instead of proving absence of bugs (over-approximate / sound), prove presence of bugs (under-approximate / no false positives).

A triple [P] C [Q] means: if Q holds at the end, then there exists an execution from a state satisfying P that reaches Q. Every reported bug is a real bug.

Powers: Pulse (Infer's under-approximate engine), RacerD (Meta's race detector).

Over-Approximate vs. Under-Approximate This is the deepest philosophical split in static analysis:
• Sound / over-approximate (Hoare, abstract interpretation): catches all bugs of a given class, but may report false alarms. "If I don't report it, it's safe."
• Complete / under-approximate (incorrectness logic, symbolic execution): every report is a real bug, but may miss some. "If I report it, it's real."
Most industrial tools (Infer, Clang-Tidy, ESLint) are actually in the under-approximate camp or somewhere in between — they trade soundness for usability.

5. Symbolic Execution

Exploration

Instead of running a program with concrete inputs, run it with symbolic values and collect path constraints. At each branch, fork into both paths. Use an SMT solver to check if each path is feasible and to generate concrete test inputs.

function foo(x: int, y: int) {
  if (x > 0) {        // Fork: path A (x>0), path B (x≤0)
    if (y == x + 1) {  // Fork again on path A
      crash();         // Path constraint: x>0 ∧ y==x+1
    }                  // SMT solver: SAT! x=1, y=2 triggers crash
  }
}

Variants

VariantDescriptionTool
Pure symbolicAll inputs symbolic from the startKLEE, SymCC
Concolic (concrete + symbolic)Execute concretely, collect symbolic constraints, negate one to explore new pathSAGE (Microsoft), DART, CUTE
Selective symbolicMix concrete and symbolic values; only symbolize interesting inputsS2E, Angr

The Path Explosion Problem

A program with n branches has 2n paths. Loops with symbolic bounds create infinitely many paths. This is the fundamental limitation of symbolic execution. Mitigations:

  • Bounded exploration: limit loop unrolling depth
  • State merging: merge paths at join points (approaches abstract interpretation!)
  • Concolic execution: use concrete execution to guide path selection
  • Function summaries: analyze functions once, reuse summaries
Symbolic Execution ↔ Abstract Interpretation: Converging State merging in symbolic execution is essentially abstract interpretation (you approximate the union of path states). And abstract interpretation with trace partitioning is essentially symbolic execution (you keep paths separate). The two converge when you tune the precision dial — they're exploring the same tradeoff space from opposite ends.

6. Datalog & Query-Based Analysis

Query

Instead of implementing an analysis as imperative code, declare it as logical rules over a code database. The Datalog engine computes the fixpoint. This separates what to compute from how to compute it.

Datalog as an Analysis Framework

// Points-to analysis in Datalog (simplified)
// Input relations (extracted from program):
.decl Alloc(var: Var, heap: Heap)     // x = new Foo()
.decl Assign(to: Var, from: Var)    // x = y
.decl Load(to: Var, base: Var, fld: Field)  // x = y.f
.decl Store(base: Var, fld: Field, src: Var) // x.f = y

// Derived relation (the analysis result):
.decl VarPointsTo(var: Var, heap: Heap)

// Rules:
VarPointsTo(v, h) :- Alloc(v, h).
VarPointsTo(v, h) :- Assign(v, u), VarPointsTo(u, h).
// ... (load/store rules involve HeapPointsTo)

// The Datalog engine evaluates to fixpoint automatically.

Key Tools

Doop

The state-of-the-art Java/Android points-to analysis, entirely specified in Datalog. Over 40,000 lines of Datalog rules defining context-sensitive, flow-sensitive, object-sensitive analyses. Runs on Soufflé.

Context-sensitivity is a parameter: 1-call-site, 2-object, 1-type — each is a different set of rules. This is why Datalog is powerful: you specify the analysis declaratively and the engine optimizes execution.

Soufflé

High-performance Datalog engine designed specifically for program analysis. Compiles Datalog rules to parallel C++ for speed. Used by Doop, and increasingly in security analysis.

souffle-lang.github.io

CodeQL / Semmle QL

GitHub's query-based analysis platform. Extracts code into a relational database, queries written in QL (a Datalog variant with object-oriented features and aggregates). Finds vulnerabilities via taint-tracking queries.

Not pure Datalog (QL adds recursion control, aggregates, classes), but the same paradigm: analysis as query.

CFL-Reachability

Context-Free Language reachability (Reps, 1998): formulates program analysis as graph reachability where valid paths must form strings in a context-free grammar. Subsumes many interprocedural analyses.

Points-to analysis, taint analysis, and typestate analysis can all be expressed as CFL-reachability problems. Equivalent in power to a specific class of Datalog programs (chain Datalog).

Datalog ↔ Dataflow ↔ Abstract Interpretation Datalog with stratified negation can express any monotone dataflow analysis. The fixpoint semantics of Datalog is exactly the least fixpoint of the monotone operator defined by the rules — the same mathematical object as in Kildall's framework. But Datalog adds relational expressiveness (joins across tables) that goes beyond classical gen/kill.

7. Attribute Grammars

Grammar

Attribute grammars (Knuth, 1968) define analyses as attributes attached to grammar productions. Information flows up the tree (synthesized attributes) and down the tree (inherited attributes). This is the framework KSC uses.

Core Idea

// An attribute grammar for type-checking expressions
// production: Expr → Expr "+" Expr
// synthesized attribute: .type (flows up)
// inherited attribute: .env (flows down)

Expr0.type = if Expr1.type == "int" && Expr2.type == "int"
              then "int"
              else "error";
Expr1.env = Expr0.env;  // pass environment down
Expr2.env = Expr0.env;

Key Implementations

SilverSilver/JavaForwarding, language extension via AGsableC (extensible C compiler)
KiamaScalaEmbedded DSL for AGs in ScalaResearch, teaching

Relationship to Other Frameworks

AGs Are Specialized Abstract Interpretations on Trees An attribute grammar is an abstract interpretation where the abstract domain is the attribute lattice and the program is the AST. Synthesized attributes are bottom-up fixpoints; inherited attributes are top-down fixpoints. Circular attribute grammars (Magnusson & Hedin) explicitly use fixpoint iteration with widening-like convergence — directly paralleling abstract interpretation.

The key advantage of AGs over general abstract interpretation: the tree structure provides a natural evaluation order. You don't need to iterate over arbitrary CFGs — the tree dictates the information flow. This makes AGs simpler to implement and reason about for tree-shaped analyses.

8. Algebraic Program Analysis

Algebraic

A more recent synthesis: express program analysis as computation over an algebraic structure (semiring, Kleene algebra). The program's control flow graph defines a regular expression; the analysis evaluates that expression over a domain-specific algebra.

Core Idea (Tarjan, Reps et al.)

Any intraprocedural analysis on a reducible CFG can be computed in almost-linear time using Tarjan's path-expression algorithm:

  1. Decompose the CFG into a tree of loops (interval graph)
  2. Express the path summary of each region as a regular expression
  3. Evaluate the regular expression over an abstract domain that forms a semiring
// A semiring (S, ⊕, ⊗, 0, 1) where:
//   ⊕ = combine (merge at join points) — must be commutative, associative
//   ⊗ = extend (sequential composition) — must be associative
//   0 = unreachable, 1 = identity (skip)

// Example: shortest-path semiring
//   S = ℕ ∪ {∞}
//   a ⊕ b = min(a, b)
//   a ⊗ b = a + b
//   0 = ∞, 1 = 0

// Example: dataflow (gen/kill) semiring
//   S = (Gen, Kill) pairs
//   (g₁,k₁) ⊕ (g₂,k₂) = (g₁ ∪ g₂, k₁ ∩ k₂)
//   (g₁,k₁) ⊗ (g₂,k₂) = ((g₂ ∪ g₁)\k₂, k₁ ∪ k₂)

Extensions

WPDS (Weighted Pushdown Systems)

Extends algebraic analysis to interprocedural settings. The pushdown system models the call stack; weights (from a semiring) model the analysis. Computes precise meet-over-all-valid-paths (MOVP) solutions respecting call-return matching.

Tools: WALi (Wisconsin), CrocoPat

Kleene Algebra with Tests (KAT)

Kozen (1997): extends regular expressions with boolean tests. Programs are KAT expressions; equivalence checking is decidable. Used for compiler optimization verification and network policy analysis.

NetKAT extends KAT for software-defined networking.

Newtonian Program Analysis (NPA)

Reps, Esparza, et al. (2016): applies Newton's method (from numerical analysis!) to solve dataflow equations. Converges in fewer iterations than Kleene iteration for many analyses. Combines algebraic structure with iterative methods.

Algebraic ↔ Abstract Interpretation Algebraic program analysis and abstract interpretation solve the same fixpoint equations — but using different algorithms. Abstract interpretation uses Kleene iteration (iterate transfer functions until stable). Algebraic analysis uses Tarjan's path-expression algorithm (decompose the CFG, evaluate algebraically). For the same analysis, the algebraic approach can be asymptotically faster on structured (reducible) programs.

9. Graph-Based Analysis & Program Slicing

Graph

Represent the program as a graph, then answer questions via graph algorithms (reachability, dominance, slicing). The graph structure itself encodes semantic relationships.

Program Slicing (Weiser, 1981)

A program slice is the subset of a program that can affect the value of a variable at a given point. Slicing is a graph reachability problem on the PDG/SDG.

  • Backward slice: what statements could have affected variable v at line n?
  • Forward slice: what statements could be affected by variable v at line n?

Applications: debugging (understand what affects a buggy value), testing (identify relevant test cases), security (taint analysis as forward slicing).

Code Property Graphs

Joern (Yamaguchi et al., 2014) merges ASTs, CFGs, and PDGs into a single graph database (a Code Property Graph). Vulnerability patterns are expressed as graph traversals. This is similar to CodeQL but uses graph traversal rather than Datalog.

joern.io — open source, queries in Scala-based DSL over graph database.

Graph Analyses Are Engineering Frameworks Unlike abstract interpretation or type theory, graph-based approaches aren't a unified mathematical theory — they're a collection of useful representations and algorithms. Their power comes from reducing complex semantic questions to well-studied graph problems (reachability, dominance, cut vertices, etc.).

10. Constraint-Based & SAT/SMT Analysis

Constraints

Encode the analysis question as a constraint satisfaction problem and hand it to a solver. The solver does the hard work.

SAT & SMT Solvers as Analysis Engines

Many modern analyses reduce to: "generate constraints from the program, then call Z3/CVC5." The constraints encode the analysis question; the solver's answer is the analysis result.

Set Constraints (Aiken, 1990s)

Express type inference and flow analysis as inclusion constraints over set-valued variables: [[x]] ⊆ [[y]] means "everything that flows to x also flows to y." Solved by iterative closure algorithms.

Subsumes Andersen-style points-to analysis, 0-CFA, and simple type inference.

SMT-Based Verification

Encode verification conditions as SMT formulas. Tools like Dafny, Why3, Frama-C/WP, and CBMC all generate SMT queries internally. The solver (Z3, CVC5, Boolector) does the heavy lifting.

Theories: linear/nonlinear arithmetic, arrays, bitvectors, uninterpreted functions, strings.

Horn Clause Solving (CHC)

Constrained Horn Clauses unify several analysis paradigms: model checking, abstract interpretation, and predicate abstraction all produce problems expressible as CHC systems. Solvers like Z3 (Spacer), Eldarica, and HoIce solve them.

SeaHorn generates CHCs from LLVM; the CHC solver does the verification.

Abstract CDCL

D'Silva, Kroening, et al.: combine SAT solving's conflict-driven clause learning (CDCL) with abstract interpretation. The SAT solver's deduction is abstraction; its backtracking is refinement. Yields a unified algorithm for DPLL(T) + abstract interpretation.

11. Concurrency Analysis

Concurrency

Concurrency introduces interleaving, making the state space exponential. Specialized frameworks handle this:

Process Algebras (CSP, CCS, π-calculus)

Model concurrent systems as algebraic terms with composition operators (parallel, sequential, choice). Equivalences (bisimulation, trace equivalence) allow reasoning about behavior.

CSP (Hoare, 1978): channel-based communication. Checked by FDR4.

π-calculus (Milner, 1992): mobile processes with name-passing. Foundation for session types.

Happens-Before Analysis

Lamport (1978): partial ordering of events in distributed/concurrent systems. A data race exists iff two conflicting accesses are unordered by happens-before.

Tools: ThreadSanitizer (Google), Helgrind (Valgrind), FastTrack (Flanagan & Freund).

Lockset Analysis (Eraser)

Savage et al. (1997): track the set of locks held when each variable is accessed. If the intersection of locksets across accesses is empty, report a potential race. Unsound but fast and effective.

RacerD (Meta/Infer): combines lockset analysis with ownership reasoning in an abstract interpretation framework.

Session Types

Honda (1993): types for communication protocols. A session type prescribes the sequence of messages on a channel. Type-checking ensures deadlock freedom and protocol conformance.

Active research area. Implementations in Scala, Rust, OCaml, and as extensions to mainstream languages.

12. Automata & Language-Theoretic Analysis

Automata

Programs as Languages

A program's set of valid execution traces forms a language. Safety properties are regular languages. The analysis question becomes: is the intersection of the trace language and the bad-behavior language empty?

Pushdown Systems

Model programs with recursion as pushdown automata (the stack models the call stack). Reachability is decidable for pushdown systems. Used for interprocedural analysis with precise call-return matching.

Tools: Moped, jMoped, WALi (weighted pushdown systems — combines with algebraic analysis).

Tree Automata for Shape Analysis

Represent heap shapes as trees; use tree automata to abstract sets of heap configurations. Inclusion checking between tree automata gives a decision procedure for shape properties.

Tools: ARTMC, Forester (Brno). Used for verifying programs manipulating linked data structures.

Visibly Pushdown Languages

Alur & Madhusudan (2004): a subclass of CFLs where the stack operations are determined by the input symbol (calls push, returns pop). Closed under intersection and complementation — unlike general CFLs. Models program executions with visible call/return structure.

13. Machine Learning for Static Analysis

ML

The newest paradigm: train models on code and bug patterns, then use them to predict bugs in new code. Not sound, not complete, but surprisingly effective for some bug classes.

Neural Bug Detection

DeepBugs (Pradel & Sen, 2018): learns from identifier names to detect swapped arguments, wrong operators, wrong operands. Simple embedding + classifier.

CodeBERT / GraphCodeBERT: pre-trained transformer models for code understanding. Fine-tuned for vulnerability detection, clone detection, code repair.

GNN-Based Analysis

Represent code as a graph (AST + CFG + data flow), run a Graph Neural Network to classify patterns. Devign, ReVeal, IVDetect use this approach for vulnerability detection.

Advantage: learns structural patterns that rule-based tools miss. Disadvantage: no soundness guarantees, difficult to explain.

LLM-Based Analysis

Large language models (GPT-4, Claude, etc.) can be prompted or fine-tuned to find bugs, explain code, and suggest fixes. Not a formal analysis framework — it's pattern matching on training data.

Hybrid approaches: use LLMs to generate candidate invariants, then verify them with formal tools. This combines the pattern-recognition strength of ML with the soundness of traditional methods.

The Fundamental Limitation ML-based analysis is statistical, not logical. It can say "this pattern looks like a bug with 87% probability" but cannot say "this property holds for all inputs." For safety-critical systems, this is insufficient. But for finding common bug patterns at scale in non-critical code, it's a useful complement to formal methods.

Check Your Understanding

Rust's borrow checker is an example of which framework?

Correct! Rust's borrow checker is an affine type system combined with region inference (lifetimes). It's enforced at compile time as part of type-checking, not as a separate static analysis pass. You could view it as abstract interpretation over an ownership/borrowing lattice (Cousot 1997), but the engineering reality is type-system-based.
Rust's borrow checker is a type system: specifically an affine type system (values used at most once unless Copy) combined with region/lifetime inference. While you could view it through an abstract interpretation lens, it's engineered and presented as a type system.

Check Your Understanding

What distinguishes Incorrectness Logic from Hoare Logic?

Correct! Hoare Logic's {P} C {Q} says "if P holds before C, Q holds after" — it over-approximates behavior (sound). Incorrectness Logic's [P] C [Q] says "if Q holds after C, there exists an execution from P that reaches it" — it under-approximates (every reported bug is real, but some bugs may be missed).
The key distinction is the direction of approximation. Hoare Logic over-approximates: proves absence of bugs (may have false alarms). Incorrectness Logic under-approximates: proves presence of bugs (no false alarms, but may miss bugs).

Check Your Understanding

A Datalog-based analysis (like Doop) can express which of the following?

Correct! Doop is exactly a context-sensitive points-to analysis expressed in Datalog. Datalog's strength is relational reasoning over finite domains. Numeric analyses (intervals, polyhedra) require infinite lattices and widening, which Datalog doesn't natively support — those are abstract interpretation territory.
Datalog works over finite domains and is excellent for relational analyses like points-to analysis (which is what Doop does). It cannot natively express numeric analyses over infinite lattices — those require abstract interpretation's widening/narrowing machinery.
