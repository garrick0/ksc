Three Hard Questions, Answered Visually

Are inputs really infinite? What are hyperproperties? And what does "decomposition" actually mean?

A "Aren't inputs always finite in a typed language?"

Great instinct — and almost right. Let's look at what TypeScript types actually allow:

// Finite types — you CAN enumerate all possible values boolean // 2 values: true, false "a" | "b" | "c" // 3 values [boolean, boolean]// 4 values: [T,T], [T,F], [F,T], [F,F] enum Color { R, G, B } // 3 values // HUGE but technically finite types number // 2⁶⁴ ≈ 18 quintillion IEEE 754 doubles [number, number] // 2¹²⁸ ≈ 3.4 × 10³⁸ pairs // ACTUALLY INFINITE types — unbounded size string // any length! "a", "aa", "aaa", ... ∞ number[] // any length! [], [1], [1,2], ... ∞ Map<string, any>// any number of entries object // any shape, any depth () => number // functions! uncountably many
The answer: Even in TypeScript, most useful types have infinitely many possible values. string is infinite (strings can be any length). number[] is infinite (arrays can be any length). Any recursive or generic type is infinite. Only literal unions and small enums are truly finite.

But what about number? Aren't there "only" 2⁶⁴ possible doubles? Yes — but that's still way too many to test exhaustively:

2⁶⁴ ≈ 18,446,744,073,709,551,616 possible number values.
At 1 billion tests per second, exhaustive testing would take 585 years.
For f(a: number, b: number), the input space is 2¹²⁸ — 10²² years. Longer than the age of the universe.

So even "finite" types are practically infinite. And here's the kicker:

Types don't save you from undecidability — but they DO narrow the space. A function (x: boolean) => boolean has exactly 4 possible behaviors. A function (x: string) => string has infinitely many. The tighter the type, the more tractable the analysis.

B What are hyperproperties, really?

A regular trace property looks at one run of a program and asks "was this run good?" A hyperproperty looks at two or more runs and asks "is the relationship between these runs good?"

This is easiest to understand by example. Here's a login function — try running it with different inputs and see why you need two runs to spot the problem:

Scenario: A login function that checks a password. We want to know: does it leak information about the password through its timing?

Did you see it? Each individual run looks fine — it returns true or false. No bugs, no crashes. But when you compare two runs with different inputs, you discover the timing differs. That leaks information about the password.

This is why hyperproperties exist: Some bugs are invisible in any single execution. They only appear when you compare multiple executions with different inputs. You literally cannot state "timing doesn't depend on secret input" as a property of one trace — it's inherently a relationship between traces.

C What "decomposition" actually means

Forget the math jargon for a moment. A decomposition theorem says:

"Every property in this space can be split into exactly two simpler components along this axis."

Just like every point on a 2D plane can be split into an x-coordinate and a y-coordinate, every program property can be split along certain fundamental axes. The decomposition is unique and complete — every property splits, and there's only one way to split it.

Why RGB is a perfect analogy (and where it breaks down):

RGB works because human vision has exactly 3 cone types. Three axes perfectly span all visible colors. But the space of program properties doesn't have a fixed number of "cone types." The arithmetic hierarchy keeps going: Σ₁, Π₁, Σ₂, Π₂, Σ₃, … — infinitely many levels. Each decomposition axis (safety/liveness, trace/hyper, etc.) carves off one dimension, but there are always more dimensions remaining.

It's more like: RGB spans all colors you can see. But if you could see infrared, ultraviolet, and X-rays, you'd need more axes. The program-property space is like electromagnetic radiation — it extends beyond any finite number of axes.
