01The Core Question

When analysis rules are always for one grammar, should they depend on it directly?

Consider a system where analysis rules run over an AST. The system allows plugins for new AST grammars (node type definitions) and new analysis rules. But each analysis plugin is always for exactly one grammar. Multiple analyses can target the same grammar, but each analysis is bound to one grammar specifically.

This constraint — the data dimension is fixed per-plugin — fundamentally changes the design calculus. The question becomes: should the analysis directly import and depend on the grammar's concrete types? Or should there be an abstraction layer (a shared interface, a generic base) mediating between them?

The key constraint: An analysis plugin is never reused across multiple grammars. A "kind-checking" analysis for a TypeScript AST will never be repurposed for a Python AST. This eliminates the primary justification for abstraction layers between them.

Two Architectural Options

Option A: Direct Dependency

  • Analysis imports concrete grammar types
  • Domain types reference specific node interfaces
  • Predicates typed with grammar's kind union
  • Full discriminated union narrowing in equations
  • Grammar re-exported from analysis barrel

Option B: Shared Abstraction

  • Both depend on generic interface (ASTNode)
  • Domain types use node: unknown
  • Predicates use Set<string>
  • String-based dispatch, no type narrowing
  • Runtime type assertions throughout

The thesis of this document: Option A is correct, and the benefits are amplified in TypeScript due to its structural type system, discriminated unions, and mapped types.

02The Expression Problem Frame

Why fixing the data dimension changes everything.

Philip Wadler's Expression Problem (1998) describes the tension between two orthogonal extensibility axes: adding new data types vs. adding new operations. Any given approach makes one easy at the cost of the other.

In the standard Expression Problem, you have a 2D matrix:

Existing OperationsNew Operations
Existing Data TypesWorksHard (OOP) / Easy (FP)
New Data TypesEasy (OOP) / Hard (FP)Both need it

When both axes need to be extensible, you need sophisticated solutions like the Visitor pattern, type classes, or multi-dispatch. Abstraction layers exist specifically to mediate this 2D extensibility.

When the data dimension is fixed per-plugin, the problem collapses to 1D:

Existing OperationsNew Operations
Fixed Data TypesWorksEasy — just add operations

You only need to extend along one axis (new analysis rules over a fixed grammar). The abstraction layer that mediates 2D extensibility has no payoff, but still carries its full cost: lost type safety, no narrowing, runtime casts.

Key insight: The abstraction layer between grammar and analysis exists to support cross-grammar portability. When that portability is architecturally impossible (each analysis is for exactly one grammar), the abstraction is pure overhead.

KSC has exactly this structure:

  • Grammar plugins — define AST node kinds (e.g., Grammar<TSNodeKind> with 364 kinds)
  • Analysis plugins — define attribute grammars over a specific grammar (e.g., AnalysisDecl<TSNodeKind>)
  • The K type parameter — links them at the type level (K = TSNodeKind)

Multiple analyses can target Grammar<TSNodeKind> (e.g., kind-checking, complexity, dead-code detection), but each is explicitly and permanently bound to that grammar. A kind-checking analysis for TS nodes will never work on Python nodes.

03Prior Art: How Real Systems Handle This

Lessons from ESLint, Roslyn, rust-analyzer, GHC, and JastAdd.

1 ESLint / typescript-eslint ▼

Architecture: ESLint chose a generic AST interface (ESTree) to decouple rules from any specific language. Rules use string-keyed visitor dispatch: { "Identifier": (node) => ... }.

The cost: typescript-eslint had to build an entirely separate layer. ESTree has no knowledge of TypeScript-specific syntax. The maintainers describe the cost candidly: "we must dedicate time for every TypeScript AST change to update node conversion logic. For lint rule authors, having to convert TSESTree nodes to their TS counterparts before passing to TypeScript APIs is an annoyance." (Josh Goldberg, typescript-eslint maintainer)

The lesson: The abstraction was justified when ESLint aimed to support multiple languages (Flow, TypeScript, vanilla JS). Now that TypeScript is the only popular type-oriented language in web code, "there's no longer a strong need to keep lint rules extensible for other languages." The abstraction layer became technical debt when the cross-language portability assumption proved unnecessary.

ESLint's cautionary tale: The abstraction was the right call for the wrong assumption. When your rules are scoped to one grammar, the entire justification for the abstraction collapses.
2 Roslyn Analyzers (.NET) ▼

Architecture: Roslyn analyzers directly reference language-specific syntax node types. A C# analyzer imports Microsoft.CodeAnalysis.CSharp.Syntax and operates on MethodDeclarationSyntax, InvocationExpressionSyntax, etc.

Direct coupling by design: Analyzers register for specific SyntaxKind enum values and receive strongly-typed nodes. There is no generic "any-language" syntax abstraction — C# analyzers are C# analyzers, VB analyzers are VB analyzers.

The benefit: Full IntelliSense, compile-time safety on node properties, exhaustive pattern matching on syntax kinds. The C# compiler team considered and rejected a cross-language analyzer abstraction because the type safety loss was unacceptable.

3 rust-analyzer ▼

Architecture: Analyses (name resolution, type inference, diagnostics) operate directly on the ra_syntax crate's typed node types. Each analysis function takes specific node types: fn resolve_name(name: &ast::Name) -> Option<Binding>.

No abstraction layer: There is no generic "language-agnostic AST" mediating between analyses and the Rust syntax tree. The typed coupling enables Rust's match exhaustiveness to catch missing node handling at compile time.

4 GHC (Trees That Grow) ▼

Architecture: GHC uses the "Trees That Grow" (TTG) pattern where AST nodes have type family extensions parameterized by a compiler phase. HsExpr GhcPs (parsed), HsExpr GhcRn (renamed), HsExpr GhcTc (typechecked) share the same tree structure but gain phase-specific fields.

Relevant insight: TTG is not about decoupling analyses from the grammar. The grammar is always Haskell's AST. TTG parameterizes the phase (what annotations are available), not the language. Each analysis pass operates on phase-specific concrete types — there's no abstraction layer hiding the Haskell AST from its analyses.

5 JastAdd & Silver (Attribute Grammars) ▼

Architecture: Both JastAdd and Silver define attribute equations directly on grammar productions. An equation like eq Program.numStatements = count(this.statements); names the grammar nonterminal explicitly. There is zero abstraction between the AG specification and the grammar.

Why this matters for KSC: KSC is an attribute grammar system. In every major AG system, equations are explicitly coupled to grammar productions. This is the fundamental nature of attribute grammars — attributes are defined on productions, not independently of them.

AG lineage: In JastAdd, syn int Program.nodeCount() = 1 + sum(getChild().nodeCount()); names Program directly. The equation is for Program. KSC's equation maps (equations: { CompilationUnit: eq_kindDefs }) follow the same pattern.

Pattern across all systems: When an analysis is for one language/grammar, it directly depends on that grammar's concrete types. The only systems that introduce abstraction layers are those aiming for cross-language portability (ESLint) — and even those regret it when the assumption proves wrong.

04Options Analysis

Three coupling strategies, evaluated against the constraint that each analysis is for one grammar.

Option 1: Direct Import (Recommended)

Analysis adapter directly imports from its grammar adapter barrel. All coupling is explicit, visible, and type-safe.

// Analysis adapter directly imports grammar types
import type { TSNodeKind, KSNode, KindToNode,
  KSTypeAliasDeclaration, KSIdentifier
} from '../grammar/grammar/ts-ast/index.js';
import { grammar } from '../grammar/grammar/ts-ast/index.js';

// Domain types reference concrete node types
interface KindDefinition {
  node: KSTypeAliasDeclaration;  // not unknown, not ASTNode
}

// Predicates typed with grammar's kind union
const ASSIGNMENT_OPS: ReadonlySet<TSNodeKind> =
  new Set(grammar.sumTypeMembers['AssignmentOperator']);
Evaluation
Type Safety FULL
Discriminated union narrowing, exhaustiveness checking, property access without casts, refactoring safety when grammar changes.
Portability NONE
Analysis cannot be used with a different grammar. This is acceptable because the architecture guarantees each analysis is for one grammar.
Discoverability HIGH
The analysis barrel re-exports grammar types — consumers see the binding. Import paths make the dependency explicit.

Option 2: Shared Interface (Not Recommended)

Both grammar and analysis depend on a generic ASTNode interface from a shared package. Analysis never sees concrete grammar types.

// Analysis only sees generic ASTNode
import type { ASTNode } from '@kindscript/core-grammar';

// Domain types lose concreteness
interface KindDefinition {
  node: ASTNode;  // which kind? what fields? unknown at compile time
}

// Predicates become untyped string sets
const ASSIGNMENT_OPS: Set<string> = new Set([
  'EqualsToken', 'PlusEqualsToken', ...  // typos not caught
]);
Evaluation
Type Safety MINIMAL
No narrowing, no exhaustiveness, no field access without casts. Every property access requires as KSIdentifier or worse, as any.
Portability THEORETICAL
Could theoretically port analysis across grammars — but the architecture prohibits this. The portability is never exercised.
Discoverability LOW
The grammar binding is implicit (string constants, runtime checks). Nothing in the type system reveals which grammar the analysis targets.

Option 3: Full Generic Parameterization (Overkill)

Analysis types are generic over a grammar parameter G. Every type carries the grammar as a type parameter.

// Every analysis type parameterized on grammar
interface KindDefinition<G extends Grammar> {
  node: NodeOf<G, 'TypeAliasDeclaration'>;
}

// Equation functions generic too
function eq_kindDefs<G extends Grammar>(
  ctx: KindCtx<NodeOf<G, 'CompilationUnit'>>
): KindDefinition<G>[] { ... }
Evaluation
Type Safety FULL (complex)
Same type safety as direct import, but achieved through generic indirection. The generics carry no runtime meaning — G is always instantiated to one grammar.
Portability THEORETICAL
The generic parameter implies portability, but it's never exercised. It misleads readers into thinking the analysis is grammar-agnostic.
Complexity HIGH
Every domain type, equation, predicate, and projection gains a generic parameter that is always the same type. Higher-kinded type tricks needed for NodeOf<G, K>. TypeScript error messages become unreadable.
Watch out: This approach is technically correct but violates YAGNI. The generic parameter communicates "this analysis works with any grammar" — which is a lie. Direct imports are simpler, equally type-safe, and honestly communicate the dependency.

05TypeScript-Specific Benefits of Direct Coupling

How TypeScript's type system makes direct dependency strictly more powerful than abstraction.

Benefit 1: Discriminated Union Exhaustiveness

When the analysis directly imports the grammar's kind union (TSNodeKind), TypeScript's control flow analysis provides compile-time exhaustiveness checking:

function handleNode(kind: TSNodeKind) {
  switch (kind) {
    case 'Identifier': return handleIdentifier();
    case 'VariableDeclaration': return handleVarDecl();
    // TypeScript error: not all members of TSNodeKind handled
    default: assertNever(kind); // compile-time guarantee
  }
}

Through Set<string>? No exhaustiveness. The compiler cannot verify completeness.

Benefit 2: Automatic Type Narrowing

After a kind check, TypeScript automatically narrows the node to its specific interface:

function extractName(ctx: KindCtx<KSVariableDeclaration>): string {
  // ctx.node is narrowed to KSVariableDeclaration
  return ctx.node.name;           // string (type-safe)
  // ctx.node.escapedText;        // ERROR: not on KSVariableDeclaration
}

// Without direct coupling:
function extractNameUntyped(ctx: Ctx): string {
  const node = ctx.node as any;  // cast required
  return node.name;               // no compile-time check
}

Benefit 3: Refactoring Safety

When the grammar changes (rename a field, add a node kind, change a type), TypeScript immediately surfaces every affected analysis file. With string-based dispatch, those breakages are silent until runtime.

Benefit 4: KindToNode Mapped Type

The KindToNode mapped type converts a kind string to its concrete interface at the type level:

type KindToNode = {
  [K in TSNodeKind]: K extends keyof SpecificNodes
    ? SpecificNodes[K]
    : KSNode & { kind: K };
};

// In generated dispatch:
ctx as unknown as KindCtx<KindToNode['Identifier']>
// Resolves to: KindCtx<KSIdentifier>

This enables per-kind type narrowing without runtime cost. The generated dispatch code uses KindToNode to give each equation function its correctly-typed context.

Benefit 5: TypedEquationMap per-kind Context Narrowing

type TypedEquationMap<K, CtxMap, T> = {
  [Kind in K]?: ((ctx: Kind extends keyof CtxMap
    ? CtxMap[Kind] : unknown, ...) => T)
    & { deps?: string[] };
};

With direct coupling, the adapter can instantiate CtxMap with concrete grammar types, giving each equation function the exact context type for its node kind.

The TypeScript advantage: Direct coupling unlocks discriminated union narrowing, KindToNode mapped types, exhaustiveness checking, and refactoring safety. These are not available through a generic abstraction layer — they require the compiler to see the concrete types.

The K Type Parameter

The K type parameter is the primary coupling mechanism. It flows through AnalysisDecl<K> → AttrDecl<K> → EquationMap<K>, ensuring that equation map keys are valid kind strings from the grammar. At composition boundaries (CodegenTarget<K>, EvaluationTarget<K, M, P>), TypeScript verifies that grammar and analysis share the same K.

Critically, K is erased at runtime. The evaluator engine operates on string kind names. K's sole purpose is compile-time verification — making it a pure type-safety mechanism with zero runtime cost.

// Composition boundary: K links grammar and analysis
const target: CodegenTarget<TSNodeKind> = {
  grammar,          // Grammar<TSNodeKind>
  decl: analysisDecl, // AnalysisDecl<TSNodeKind>
  // If you accidentally passed Grammar<MockKind> here,
  // TypeScript would reject it: TSNodeKind ≠ MockKind
};

Domain Type Fields

Domain types express their grammar dependency through concrete node type annotations on fields:

interface KindDefinition {
  node: KSTypeAliasDeclaration;  // not ASTNode, not unknown
  name: string;
  properties: PropertySet;
}

interface Diagnostic {
  node: KSNode;  // union of all 364 grammar node types
  message: string;
  rule: ViolationRule;
}

This gives downstream consumers (projections, application layer, CLI) type-safe access to node fields. When projections extract definition.node.name, TypeScript knows .name is a KSTypeAliasDeclaration field — no cast needed.

KindCtx<N> Narrowing

Each equation function declares which node type it operates on via KindCtx<KSSpecificNode>:

function eq_kindDefs_CompilationUnit(
  ctx: KindCtx<KSCompilationUnit>
): KindDefinition[] {
  // ctx.node is KSCompilationUnit — full type-safe access
  for (const stmt of ctx.node.statements) {
    // TypeScript knows statements exists on KSCompilationUnit
  }
}

The KindEquations<K> type enforces that all equations in a per-kind record accept the correct KindCtx:

type KindEquations<K extends TSNodeKind> = Record<string,
  ((ctx: KindCtx<KindToNode[K]>, ...args: any[]) => any)
  & { deps?: string[] }
>;

const CompilationUnitEquations = {
  kindDefs: eq_kindDefs_CompilationUnit,
} satisfies KindEquations<'CompilationUnit'>;

Predicate Constants

Predicate constants are derived from the grammar instance at module load:

import { grammar } from '../grammar/grammar/ts-ast/index.js';

export const ASSIGNMENT_OPS: ReadonlySet<TSNodeKind> =
  new Set(grammar.sumTypeMembers['AssignmentOperator'] as TSNodeKind[]);

export const SIDE_EFFECT_EXPR_KINDS: ReadonlySet<TSNodeKind> =
  new Set(grammar.sumTypeMembers['SideEffectExpression'] as TSNodeKind[]);

This is a runtime dependency on the grammar instance — the only place analysis imports a grammar value (not just types). The sum type membership data is grammar metadata that would be inaccessible through a generic interface.

KindToNode in Generated Dispatch

The generated dispatch functions use KindToNode for per-kind context narrowing:

// AUTO-GENERATED by compileAnalysis()
function dispatch_kindDefs(ctx: Ctx): KindDefinition[] {
  const _kind = (ctx.node as KSNode).kind;
  switch (_kind) {
    case 'CompilationUnit':
      return eq_kindDefs_CompilationUnit(
        ctx as unknown as KindCtx<KindToNode['CompilationUnit']>
      );
    // ... 363 more cases ...
  }
}

The double cast (as unknown as) is necessary because Ctx (with generic ASTNode) and KindCtx<KSCompilationUnit> share no structural relationship that TypeScript can prove. The evaluator engine ensures the cast is safe at runtime by dispatching on node.kind.

Barrel Re-exports

The analysis adapter barrel explicitly re-exports grammar types, making the binding discoverable to consumers:

// src/adapters/analysis/spec/ts-kind-checking/index.ts
export { grammar } from '../../../grammar/grammar/ts-ast/index.js';
export type { TSNodeKind, KSNode, KindToNode,
  KSProgram, KSCompilationUnit, KSIdentifier,
  KSTypeAliasDeclaration
} from '../../../grammar/grammar/ts-ast/index.js';

Consumers can import grammar types from either the grammar adapter or the analysis adapter barrel. The analysis barrel bundles everything needed: domain types, projections, and the grammar they operate on.

08Two Coupling Mechanisms: K vs. Concrete Types

Understanding the difference between the generic K parameter (core packages) and concrete type imports (adapters).

The system has two distinct coupling mechanisms that serve different purposes:

The K Type Parameter (Core Packages)

K extends string flows through the core package interfaces. It never resolves to a concrete type within the packages — it's a phantom type parameter that exists solely to prevent mismatched pairings at composition boundaries.

// In @kindscript/core-codegen — K is generic
interface AnalysisDecl<K extends string> {
  attrs: AttrDecl<K>[];
}

// In @kindscript/core-evaluator — K is generic
interface EvaluationTarget<K extends string, M, P> {
  grammar: Grammar<K>;
  dispatch: DispatchConfig;  // K not used here
}

K is never instantiated inside core packages. The packages don't know whether K is TSNodeKind, MockKind, or any other string union. K is instantiated only at adapter composition sites.

K's role: It's a compile-time guard rail. It prevents grammar: Grammar<MockKind> from being bundled with decl: AnalysisDecl<TSNodeKind>. At runtime, K is erased — the evaluator operates on plain strings.

Concrete Type Imports (Adapter Layer)

Inside the analysis adapter, code imports concrete grammar types — specific node interfaces, the kind union, and the KindToNode mapped type. These are not mediated by K.

// In analysis adapter — concrete types, not K
import type { KSIdentifier } from '../grammar/ts-ast/index.js';

function eq_defLookup(ctx: KindCtx<KSIdentifier>) {
  // KSIdentifier is a concrete interface, not a generic type
  return ctx.node.escapedText;  // string (known at compile time)
}

This is the direct coupling that ADR-001 mandates. It's where the real type safety comes from — not from K, but from importing KSIdentifier, KSTypeAliasDeclaration, KSVariableDeclaration, etc. directly.

Where Each Mechanism Lives

MechanismLives InPurpose
K extends string Core packages (@kindscript/core-*) Prevent grammar/analysis mismatch at composition boundaries
Concrete type imports Adapter code (src/adapters/) Per-kind narrowing, domain types, refactoring safety
Grammar instance imports Adapter code (predicates) Derive predicate sets from grammar metadata
Barrel re-exports Adapter barrel (index.ts) Make grammar binding discoverable to consumers
The layering: Core packages provide generic K-parameterized contracts. Adapters instantiate K to a concrete kind union and add direct type imports. The direct imports are where the Expression Problem analysis applies — they are the explicit coupling that makes the analysis non-portable (intentionally) and type-safe (beneficially).

09Design Decision Reference

Per-surface decisions, options considered, and recommendations.

A Domain Type Node Fields ▼

Decision: Should KindDefinition.node be typed as KSTypeAliasDeclaration, KSNode, ASTNode, or unknown?

KSTypeAliasDeclaration CHOSEN
Maximum precision. Consumers know exactly which node kind this is. They can access .name, .typeParameters, etc. without casts. Grammar rename in nodes.ts cascades to all consumers — full refactoring safety.
KSNode CONSIDERED
Valid for fields that can hold any node kind (like Diagnostic.node). But for fields that always hold one specific kind, this loses precision unnecessarily.
ASTNode REJECTED
The generic base type from core-grammar. No grammar-specific fields accessible. Requires casts to do anything useful.

Guideline: Use the most specific type that is correct. KindDefinition always holds a TypeAliasDeclaration, so type it as such. Diagnostic can hold any node, so KSNode is correct.

B Predicate Constant Types ▼

Decision: Should predicate sets be typed as ReadonlySet<TSNodeKind> or Set<string>? Should they be hardcoded or derived from grammar metadata?

ReadonlySet<TSNodeKind> from grammar.sumTypeMembers CHOSEN
Derived from grammar sum types at module load. Type-safe: set.has(kind) narrows kind. Single source of truth: adding a new AssignmentOperator in grammar automatically updates the predicate.
Set<string> with hardcoded members REJECTED
Duplicates grammar information. Typos not caught at compile time. Grammar changes don't cascade — predicates silently become stale.
C Equation Function Signatures ▼

Decision: Should equation functions use KindCtx<KSSpecificNode> or generic Ctx?

KindCtx<KSIdentifier> etc. CHOSEN
Full type narrowing inside the function body. Author gets IntelliSense for node-specific fields. Wrong-kind assignments caught by satisfies KindEquations<K>.
Ctx (generic) REJECTED
Every property access requires ctx.node as KSIdentifier. No compile-time verification that the equation is assigned to the correct kind.

Enforcement: The KindEquations<K> type uses TypeScript's function contravariance to verify assignments. A function typed (ctx: KindCtx<KSIdentifier>) => T is assignable to KindEquations<'Identifier'> but not to KindEquations<'CompilationUnit'> — TypeScript catches the mismatch.

D Barrel Re-export Strategy ▼

Decision: Should the analysis barrel re-export grammar types?

Re-export grammar + key types CHOSEN
Makes the grammar binding discoverable. Consumers can import everything from one barrel. The re-export documents "this analysis is for this grammar" in the API surface.
Don't re-export REJECTED
Consumers must import from two separate adapters. The grammar dependency is hidden in internal imports rather than visible in the API.

What to re-export: grammar (instance), TSNodeKind (kind union), KSNode (node union), KindToNode (mapped type), and specific node interfaces referenced by domain types (KSProgram, KSCompilationUnit, KSIdentifier, KSTypeAliasDeclaration).

E Generated Code Grammar Imports ▼

Decision: Should generated dispatch code import grammar types directly or receive them through parameters?

Direct import from grammar adapter CHOSEN
Generated dispatch.ts imports KSNode and KindToNode directly. This enables the as unknown as KindCtx<KindToNode['Kind']> narrowing pattern, which gives type-safe ctx to each equation function.
Untyped dispatch (string only) REJECTED
Without KindToNode, the dispatch would pass unnarrowerd Ctx to equations. Every equation would need internal casts. The type safety moves from one centralized location (dispatch) to N distributed locations (every equation).
F Helper Function Typing ▼

Decision: Should analysis helpers (like diag()) use grammar-specific types or generic ASTNode?

Mixed: generic where possible, concrete where needed CHOSEN
A diag() function that only reads node.pos, node.end, node.text can use ASTNode — those fields exist on the base type. But a function that reads node.escapedText needs KSIdentifier.

Guideline: Use grammar-typed parameters where the function accesses node-specific fields. Use generic types where only base ASTNode fields are used. Don't add grammar coupling where it's not needed, but don't avoid it where it improves safety.

Q

What is the Expression Problem, and why does fixing the data dimension per-plugin change the analysis?

A

The Expression Problem (Wadler, 1998) is the tension between extending data types vs. extending operations. Abstraction layers exist to mediate both axes. When each analysis plugin is for one grammar, the data dimension is fixed — the problem collapses to 1D (new operations only). The abstraction layer has cost but no payoff.

Q

What lesson does ESLint/typescript-eslint teach about grammar abstraction layers?

A

ESLint chose a generic AST (ESTree) for cross-language portability. typescript-eslint had to build a separate conversion layer at significant maintenance cost. The maintainers now acknowledge the abstraction was "the right call for the wrong assumption" — TypeScript is the only target. When cross-grammar portability isn't needed, the abstraction is pure overhead.

Q

What are the three options for coupling analysis to grammar, and which is recommended when analyses are grammar-bound?

A

Direct Import (recommended): analysis imports concrete grammar types. Shared Interface (rejected): both depend on generic ASTNode. Full Generic (overkill): every type carries a grammar parameter G. Direct import is simplest, equally type-safe, and honestly communicates the one-grammar dependency.

Q

What is the difference between the K type parameter (in core packages) and concrete type imports (in adapters)?

A

K extends string is a generic type parameter in core packages that prevents grammar/analysis mismatch at composition boundaries — it's erased at runtime. Concrete type imports (like KSIdentifier) are adapter-layer direct dependencies that enable discriminated union narrowing, field access, and refactoring safety. K guards the boundary; concrete types power the internals.

Q

Name three TypeScript-specific benefits you get from direct grammar coupling that you lose with a shared abstraction layer.

A

1) Discriminated union exhaustiveness: switch (kind) with assertNever catches missing cases. 2) Automatic type narrowing: after a kind check, node properties are available without casts. 3) Refactoring safety: renaming a grammar field/node cascades errors to every affected equation. Also: KindToNode mapped type for per-kind context, TypedEquationMap per-kind narrowing.

Q

Why does the generated dispatch code use a double cast (as unknown as KindCtx<KindToNode[K]>)?

A

Ctx (with generic ASTNode) and KindCtx<KSIdentifier> share no structural subtype relationship that TypeScript can verify statically. The double cast (as unknown as) bridges this gap. The cast is safe because the evaluator engine dispatches on node.kind at runtime, ensuring the cast target matches the actual node type.

Q

How do KSC's predicate constants (ASSIGNMENT_OPS, SIDE_EFFECT_EXPR_KINDS) derive from the grammar, and why is this better than hardcoding?

A

They are constructed at module load from grammar.sumTypeMembers['AssignmentOperator'] and typed as ReadonlySet<TSNodeKind>. This is better than hardcoding because: (1) single source of truth — adding a new member to the sum type in nodes.ts automatically includes it; (2) no string typos possible; (3) .has(kind) provides type narrowing.

Q

What does KindEquations<K extends TSNodeKind> enforce, and how does TypeScript's function contravariance make it work?

A

KindEquations<K> requires all equations in a record to accept KindCtx<KindToNode[K]>. TypeScript's function contravariance means a function (ctx: KindCtx<KSIdentifier>) => T is assignable to KindEquations<'Identifier'> but not to KindEquations<'CompilationUnit'> — because KSCompilationUnit is not assignable to KSIdentifier in the contravariant position.
