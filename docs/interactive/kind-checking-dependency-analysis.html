Kind-Checking Dependency Analysis

How specs/ts-ast/kind-checking/ was decoupled from generated code using declarative const objects with satisfies — enabling independent codegen stages.

Part 1: Before & After

Old Dependency Graph (the problem)

The old problem: spec.ts imported KindToNode (type) and fieldDefs (runtime value) from generated grammar output. equations.ts imported 17 KS* type interfaces from generated code. This created an ordering constraint: grammar codegen had to run before analysis codegen could even load.

Current Dependency Graph (clean)

Clean! kind-checking/spec.ts imports TSNodeKind (type) and NODES (runtime value) directly from ../grammar/nodes.ts — the spec source, not generated code. Both codegen stages are fully independent.

Part 2: The Solution — Declarative Const Objects

Core insight: Replace the imperative GrammarBuilder pattern (b.node(), b.leaf() calls) with declarative const objects using TypeScript's satisfies keyword. This preserves literal key types at the type level while checking structural compatibility — enabling keyof typeof NODES to produce a 364-member string literal union from source code, no codegen needed.

Declarative Const Objects + satisfies

The grammar is now defined as two plain const objects in specs/ts-ast/grammar/nodes.ts:

import type { NodeDef, SumTypeDef } from '../../../grammar/builder.js';
import { child, optChild, list, prop } from '../../../grammar/builder.js';

export const SUM_TYPES = {
  Expression: { fields: { typeString: prop('string') } },
  UnaryExpression: {},
  UpdateExpression: { includes: ['UnaryExpression'] },
  // ... 48 sum types total
} satisfies Record<string, SumTypeDef>;

export const NODES = {
  Program: {
    memberOf: [],
    fields: { compilationUnits: list('CompilationUnit') },
  },
  Identifier: {
    memberOf: ['PrimaryExpression', 'MemberExpression', 'BindingName', /*...*/],
    fields: {
      escapedText: prop('string'),
      resolvesToImport: prop('boolean'),
      importModuleSpecifier: prop('string'),
    },
  },
  // ... 364 nodes total
} satisfies Record<string, NodeDef>;

// The magic line:
export type TSNodeKind = keyof typeof NODES;
// = 'Program' | 'Identifier' | 'CompilationUnit' | ... (364 literals)
Key property: TSNodeKind is a 364-member string literal union derived entirely from source code. If you typo 'Identifer' in an equation key, TypeScript catches it statically. No generated code involved.

Why satisfies Instead of Type Annotation?

▶ The problem with : Record<string, NodeDef>
// With type annotation: LOSES literal keys
const NODES: Record<string, NodeDef> = {
  Identifier: { ... },
  CompilationUnit: { ... },
};

type TSNodeKind = keyof typeof NODES;
// = string  <-- too wide! All literal info lost.

A : Record<string, T> annotation widens the key type to string. TypeScript forgets the specific keys.

▶ The solution with satisfies Record<string, NodeDef>
// With satisfies: PRESERVES literal keys + CHECKS structure
const NODES = {
  Identifier: { ... },
  CompilationUnit: { ... },
} satisfies Record<string, NodeDef>;

type TSNodeKind = keyof typeof NODES;
// = 'Identifier' | 'CompilationUnit'  <-- exact union!

satisfies validates that the object conforms to Record<string, NodeDef> (each value is a valid NodeDef), but does not widen the type. TypeScript retains the literal key information.

▶ The problem with no annotation at all
// No annotation: preserves keys but no structural checking
const NODES = {
  Identifier: { membrrOf: ['Expression'] },  // typo not caught!
};

type TSNodeKind = keyof typeof NODES;
// = 'Identifier'  <-- preserves keys, but no guarantee values are valid NodeDefs

Without satisfies, TypeScript doesn't check that each value conforms to NodeDef. Typos in field names or missing required properties aren't caught.

satisfies gives you both: structural validation (every value is a valid NodeDef) + literal key preservation (every key is individually tracked). It's the TypeScript 4.9+ feature that makes this whole approach possible.

The New buildGrammar() Function

The old GrammarBuilder interface with mutable state was replaced by a pure function:

// grammar/builder.ts

export interface NodeDef {
  memberOf: readonly string[];
  fields: Record<string, FieldDesc>;
}

export interface SumTypeDef {
  includes?: readonly string[];
  fields?: Record<string, FieldDesc>;
}

export function buildGrammar(
  nodeDefs: Record<string, NodeDef>,
  sumTypeDefs: Record<string, SumTypeDef>,
): { nodes: Map<string, NodeEntry>; sumTypes: Map<string, SumTypeEntry> } {
  // 1. Initialize sum type registry
  // 2. Build node registry, populating sum type members
  // 3. Resolve transitive includes (fixed-point)
  // 4. Propagate sum type fields to all members
  return { nodes: nodeRegistry, sumTypes: sumTypeRegistry };
}

Grammar specs now call it directly:

// specs/ts-ast/grammar/spec.ts
import { buildGrammar } from '../../../grammar/builder.js';
import { NODES, SUM_TYPES } from './nodes.js';

export function buildGrammarSpec(): GrammarSpec {
  const { nodes, sumTypes } = buildGrammar(NODES, SUM_TYPES);
  return { nodes, sumTypes, convertGenerator: ... };
}
What changed:
  • Removed: GrammarBuilder interface, createGrammarBuilder() factory
  • Added: NodeDef, SumTypeDef input types, buildGrammar() pure function
  • Unchanged: NodeEntry, SumTypeEntry output types, field helpers (child, optChild, list, prop)
  • buildGrammarSpec() now takes no arguments (was buildGrammarSpec(builder: GrammarBuilder))

Part 3: The Remaining import type in equations.ts

▶ equations.ts still has import type from generated code — is this a problem?
// specs/ts-ast/kind-checking/equations.ts
import type {
  KSNode, KSIdentifier, KSTypeAliasDeclaration,
  KSCompilationUnit, KSVariableStatement,
  // ... 17 types total
} from '../../../generated/ts-ast/grammar/index.js';

No, this is fine. These imports are:

  • import type — erased at runtime by transpilers (esbuild/tsx). Zero JavaScript code emitted.
  • IDE type safety — used for KindCtx<KSIdentifier> type narrowing in equation signatures, enabling autocomplete and type checking on ctx.node.escapedText.
  • Always available — generated files are committed to the repo. They always exist for tsc.
  • Analogous to ts.Node — importing a stable published type for IDE support is standard practice.

The key constraint was: no runtime dependency on generated code in the spec. That's achieved. The import type doesn't create any runtime dependency, codegen ordering constraint, or circular knowledge flow.

Why Independent Codegen Works

▶ The generated evaluator imports from generated grammar — isn't that a dependency?

Yes, the generated evaluator.ts does import { fieldDefs } from '../grammar/index.js'. But this is a runtime dependency between two generated files — both exist after codegen completes. The analysis compiler doesn't evaluate this import; it just emits it as a string literal.

Codegen-time deps

grammar codegen:
  reads: specs/ts-ast/grammar/
  writes: generated grammar files

analysis codegen:
  reads: specs/ts-ast/kind-checking/
         + specs/ts-ast/grammar/nodes.ts
  writes: generated analysis files

→ Independent! No shared state.

Runtime deps

evaluator.ts:
  imports: grammar/index.js
  imports: equations.js
  imports: analysis/ctx.js

→ All files exist after codegen.
   Runtime deps are fine.

Knowledge Check

Q1: Why does satisfies enable keyof typeof NODES to be a 364-member union, but : Record<string, NodeDef> would give just string?
Click to reveal answer
satisfies checks structural compatibility without widening the inferred type. TypeScript preserves the literal key types from the object literal. A type annotation (:) replaces the inferred type with the annotation, widening keys to string.
Q2: How many runtime (non-type) imports does kind-checking/spec.ts now have from generated code?
Click to reveal answer
Zero. The old fieldDefs import was replaced by NODES from ../grammar/nodes.js (source). The old KindToNode type import was replaced by TSNodeKind from the same source file.
Q3: equations.ts still imports types from generated code. Why is this acceptable?
Click to reveal answer
The imports are import type — erased at runtime by transpilers. They provide IDE type safety (KindCtx<KSIdentifier> narrowing) but create zero runtime dependency. Generated files are committed to the repo, so they always exist for tsc. It's analogous to importing ts.Node from the TypeScript package.
Q4: Can grammar codegen and analysis codegen now run in parallel?
Click to reveal answer
Yes! Both compileGrammar() and compileAnalysis() are pure functions from spec data to output strings. Neither reads the other's output. The analysis spec gets allKinds from source NODES, not from generated grammar files. Both stages can run concurrently.
Q5: What replaced the GrammarBuilder interface?
Click to reveal answer
NodeDef and SumTypeDef input types + a pure buildGrammar(nodeDefs, sumTypeDefs) function. Grammar definitions are now declarative const objects (NODES, SUM_TYPES) with satisfies for structural validation. The same resolution logic (transitive includes, field propagation) runs inside buildGrammar().

Summary

What Was Done

Replaced the imperative GrammarBuilder pattern with declarative const objects using TypeScript's satisfies keyword. This enables keyof typeof NODES to produce a 364-member string literal union from source code — eliminating the need to import kind types or runtime data from generated code.

Key Result

specs/ts-ast/kind-checking/spec.ts now has zero imports from generated code. TSNodeKind and the runtime kind set are both derived from specs/ts-ast/grammar/nodes.ts (source). The two codegen stages (grammar + analysis) are fully independent and can run in any order or in parallel.

Verification

314/314 tests pass. TypeScript typecheck clean. Zero regressions.
