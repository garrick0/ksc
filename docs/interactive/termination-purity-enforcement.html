Assuming Termination, Enforcing Purity

What unlocks if you assume programs halt? What tools exist to enforce purity in TS/JS? And how do other languages solve this?

1 What "assuming termination" unlocks

Termination is the single biggest obstacle to program analysis. If you know a program halts, a huge number of previously-undecidable properties become decidable or at least tractable. Here's what changes:

Yes — you can absolutely make this assumption in a product. And many tools already do. The standard framing is:

"The following analysis results are valid assuming your program terminates. We provide guardrails to increase confidence that it does."

This is called partial correctness (vs. total correctness). "If the program halts, it produces the right answer" is far more tractable than "the program halts AND produces the right answer."

Who else makes this assumption? Everyone:

TypeScript compiler: Assumes termination for type narrowing and control flow analysis.
ESLint: All rules assume the code actually executes (terminates through each function).
React: Assumes component render functions terminate. Infinite render → crash.
Rust borrow checker: Assumes termination for lifetime analysis. Infinite loops can actually cause soundness issues.
Every test framework: Tests have timeouts — they assume termination and enforce it with a time limit.
Databases (SQL): Query planners assume queries terminate. Infinite loops in stored procedures → timeout.

2 Practical guardrails for termination

You can build a layered defense — static checks catch the obvious cases, runtime checks catch the rest:

1

Static: Ban dangerous patterns

Lint rules that forbid while(true), unbounded for(;;), and recursion without visible base cases. This catches 80%+ of non-termination risks at write time.

eslint-plugin-functional no-constant-condition rule no-unreachable-loop custom AST rules (buildable)
2

Static: Call graph cycle detection

Build the call graph and flag mutual recursion cycles (f→g→f). If no cycles exist and no unbounded loops exist, termination is guaranteed.

TypeScript compiler (partial) madge (module cycles) custom call-graph analysis (buildable)
3

Static: Bounded loop verification

For loops over finite collections (for..of, .map(), .forEach()), termination is guaranteed if the collection is finite. Flag while loops that don't obviously decrease a counter.

eslint rules (partial) full loop bound analysis (gap)
4

Runtime: Timeout wrappers

Wrap function execution in a timeout. If it doesn't return within N ms, abort and report. This is the ultimate safety net — catches everything static analysis misses.

AbortController / AbortSignal.timeout() Worker thread termination Promise.race with timeout Jest/Vitest test timeouts
5

Runtime: Loop iteration counters

Instrument loops with a counter that throws after N iterations. Simple, effective, and catches runaway recursion too (stack depth limit).

V8 stack depth limit (built-in) babel/ts transform (buildable) infinite-loop-detector (small lib)
The combination of static pattern bans + call graph analysis + runtime timeouts gives you very high confidence in termination. It's not a mathematical proof, but for a practical product it's more than sufficient — and it's exactly what every production system already does.

The fundamental trade-off:

Languages that enforce purity at the language level (Haskell, Elm, Koka) give you the strongest guarantees but the smallest ecosystems. Languages with big ecosystems (TypeScript, Python) give you purity only through opt-in libraries — which means escape hatches are always available.

Effect-TS is the most ambitious attempt to get Haskell-like guarantees in a TypeScript-sized ecosystem. It's the closest thing to "Haskell's IO monad, but for TypeScript" — full effect tracking in the type system, with the entire npm ecosystem still accessible. It's the de facto successor to fp-ts (which is officially merging into it) and has 13k+ stars with rapid growth.

5 What your product could do

Combining everything — the termination assumption, static guardrails, runtime enforcement, and existing tools — here's what a practical system looks like:

The layered approach

Layer 1 — Static primitive scan (KindScript today): Scan AST for which primitives are used. "This function uses fetch and Math.random." Decidable, fast, no assumptions needed.

Layer 2 — Static structure analysis: Build call graph, detect cycles, flag unbounded loops, check for mutation escaping scope. If no dangerous patterns → declare "termination likely" with high confidence.

Layer 3 — Conditional properties (your idea): "Assuming this function terminates: it is pure, deterministic, and its output type is fully determined by its input type." This is the big unlock — partial correctness analysis.

Layer 4 — Runtime validation: Run the function with sampled inputs under a timeout. If it terminates within the timeout for 10,000 random inputs, upgrade "termination likely" to "termination: high confidence."

Layer 5 — Confidence score: Combine all layers into a confidence level: "We are 98% confident this function is pure and terminates. Here's what we checked and what we couldn't verify."

What no one else is doing (the gap):

Most tools pick one layer — ESLint does static linting, Jest does dynamic testing, Effect-TS does type-level effect tracking. No tool combines all five layers into a unified confidence score per function. That's the product opportunity: not "is this function pure?" (binary, often undecidable) but "how confident are we that this function is pure, and here's the evidence." The termination assumption with explicit guardrails is the key enabler.
