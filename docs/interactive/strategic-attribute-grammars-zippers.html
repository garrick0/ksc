1The Big Picture

Two powerful language engineering techniques — unified by one simple data structure.

Attribute Grammars

Context-Dependent Analysis

Declarative specifications that compute properties over trees. Synthesized attributes flow bottom-up (e.g. type of an expression). Inherited attributes flow top-down (e.g. scope environment). Ideal for static analysis, type checking, code generation.

Strategic Programming

Controlled Tree Rewriting

Programmable traversal strategies that apply rewrite rules to trees in controlled ways. Combinators like topdown, bottomup, oncetd define where and how rules are applied. Ideal for transformations, optimizations, refactoring.

Zippers

The Unifying Substrate

A purely functional cursor into a data structure. Invented by Gérard Huet (1997). Enables efficient, localized navigation and updates without mutation. Both AGs and strategies can be expressed as operations on a zipper-navigated tree.

Why Combine Them?

Language engineering tasks often require both analysis (AGs) and transformation (strategies). For example: "Find all variables that are unused (analysis), then remove their declarations (transformation)." Traditionally, these lived in separate frameworks (UUAG for AGs, Strafunski/Stratego for strategies). The zipper-based approach embeds both in the same purely functional setting.

2The Zipper

A purely functional cursor — the tree turned inside-out around a focus point.

The Problem

In pure functional programming, updating a deeply nested node requires reconstructing every ancestor on the path back to the root — an O(log n) operation that creates wasteful intermediate copies. The zipper solves this by splitting the data structure into a focused subtree and its surrounding context.

Core Idea

Think of unzipping a jacket: the zipper head (focus) separates the fabric into two sides (context). Moving the focus through the tree is like sliding the zipper up and down. At any position, you can modify the focused node in O(1) and later reconstruct the whole tree by "zipping back up."

List Zipper

The simplest zipper. A list focused at position i is stored as two lists: elements before the focus (reversed) and elements from the focus onward.

type ListZipper a = ([a], [a])
-- ([elements before, reversed], [focus : elements after])

-- Example: [1,2,3,4,5] focused at 3
-- = ([2,1], [3,4,5])

goRight :: ListZipper a -> ListZipper a
goRight (ls, x:rs) = (x:ls, rs)

goLeft  :: ListZipper a -> ListZipper a
goLeft  (l:ls, rs) = (ls, l:rs)

modify  :: (a -> a) -> ListZipper a -> ListZipper a
modify  f (ls, x:rs) = (ls, f x : rs)

Every operation is O(1). No copying of the elements we don't touch.

Tree Zipper (Huet 1997)

For binary trees, the context records a trail of "breadcrumbs" — at each ancestor, which direction we came from and the sibling subtree we left behind.

data Tree a = Leaf a | Fork (Tree a) (Tree a)

data Cxt a            -- "Breadcrumb" context
  = Top                 -- We're at the root
  | L (Cxt a) (Tree a) -- Came from left; right sibling saved
  | R (Tree a) (Cxt a) -- Came from right; left sibling saved

type Loc a = (Tree a, Cxt a)  -- (focused subtree, context)

left  (Fork l r, c) = (l, L c r)     -- descend left
right (Fork l r, c) = (r, R l c)     -- descend right
up    (t, L c r)    = (Fork t r, c)  -- ascend from left
up    (t, R l c)    = (Fork l t, c)  -- ascend from right

Generic Zipper (Scrap Your Zippers)

Adams (2010) extended the Scrap Your Boilerplate (SYB) framework to define a generic zipper that works over any algebraic data type, not just binary trees. This is what ZipperAG and Ztrategic use.

-- Any type that derives Data gets a zipper for free
data Zipper a = Zipper
  { hole    :: a           -- current focused node
  , context :: Context     -- path back to root
  }

-- Navigation works on ANY Data instance:
down    :: Zipper a -> Maybe (Zipper a)  -- first child
up      :: Zipper a -> Maybe (Zipper a)  -- parent
left    :: Zipper a -> Maybe (Zipper a)  -- left sibling
right   :: Zipper a -> Maybe (Zipper a)  -- right sibling

-- Requirement: types must derive Data (from SYB)
data Expr = Add Expr Expr | Lit Int
  deriving (Data, Typeable)

The Data class (from SYB) provides the reflective machinery to decompose any constructor into its children generically. This eliminates boilerplate navigation code.

Why "inside-out like a glove"?

Huet's original description: "The tree is turned inside-out like a returned glove, pointers from the root to the current position being reversed in a path structure."

In a normal tree, pointers go root → leaves. In a zipper, the pointers from root to focus are reversed: the context stores parent pointers going focus → root. This reversal is what makes navigation to the parent O(1) instead of requiring a full traversal from the root.

3Attribute Grammars

Declarative specifications that decorate syntax trees with computed properties.

Two Directions of Information Flow

SYNTHESIZED

Bottom-Up (children → parent)

Computed from a node's children. Examples: the type of an expression, the minimum value of a subtree, the size of a program. In AG notation: lhs.type = ...

INHERITED

Top-Down (parent → children)

Passed from a node's parent or siblings. Examples: the environment of variable bindings, the expected type in bidirectional checking, indentation level. In AG notation: child.env = ...

Classic Example: Computing Average

-- AG specification (UUAG notation)
DATA Root
  | Root  list : List

DATA List
  | Cons  head : Float   tail : List
  | Nil

-- Synthesized: flow UP
ATTR List [ || sum : Float, length : Float ]
SEM List
  | Nil   lhs.sum    = 0.0
          lhs.length = 0.0
  | Cons  lhs.sum    = @head + @tail.sum
          lhs.length = 1.0 + @tail.length

-- Inherited: flow DOWN
ATTR List [ avg : Float || ]
SEM Root
  | Root  list.avg = @list.sum / @list.length
SEM List
  | Cons  tail.avg = @lhs.avg  -- pass inherited value to tail
The power: You declare what each attribute equals. The AG system figures out when to evaluate them (scheduling) and ensures no circular dependencies (or handles them with fixed-point iteration for circular AGs).

Traditional Implementation Problem

Classical AG systems like UUAG generate code: they compile AG specifications into host language functions. This requires a separate compilation step, custom tooling, and makes it hard to use AGs as a library within a larger program. This motivates the zipper-based embedding approach.

4ZipperAG: AGs as Zipper Functions

The key insight: attribute equations become plain Haskell functions that navigate a zipper.

The Encoding

Instead of a separate AG specification language, attribute rules become regular Haskell functions that take a Zipper position and return the attribute value. Navigation replaces the AG @child.attr syntax:

AG Notation ZipperAG Encoding Meaning
@child.attr attr (child i z) Evaluate attr on the i-th child
@lhs.attr attr (parent z) Evaluate attr on the parent node
@head lexeme z Access terminal value at current node
lhs.attr = expr Pattern match on constructor at z Synthesized attribute equation
child.attr = expr Pattern match on constructor at parent z Inherited attribute equation

Example: Scope Analysis with ZipperAG

-- AST (derives Data for generic zipper)
data Root = Root Its
data Its  = ConsIts It Its | NilIts
data It   = Decl String | Use String | Block Its
  deriving (Data, Typeable)

-- Synthesized attribute: declarations in scope (flows UP)
dclo :: Zipper a -> [(String, Int)]
dclo z = case constructor z of
  "NilIts"  -> dcli z                            -- pass through
  "ConsIts" -> dclo (child 1 z) ++ dclo (child 2 z) -- combine children

-- Inherited attribute: declarations coming in (flows DOWN)
dcli :: Zipper a -> [(String, Int)]
dcli z = case constructor (parent z) of
  "Root"    -> []                                  -- root: empty env
  "ConsIts" -> dcli (parent z)                      -- inherit from parent
  "Block"   -> env (parent z)                       -- block scoping

-- Error checking: use before declaration
errs :: Zipper a -> [String]
errs z = case constructor z of
  "Use" -> if lexeme z `elem` map fst (env z)
           then []
           else ["Undefined: " ++ lexeme z]
  _     -> concatMap errs (children z)
No code generation needed. These are plain Haskell functions. The zipper provides the navigation infrastructure that the AG compiler would normally generate. You get the declarative feel of AGs with the full power of the host language.

5Strategic Programming

Programmable, composable tree traversals — controlling where and how rewrite rules apply.

The Problem Strategic Programming Solves

A naive approach to tree transformation requires writing a recursive function for every AST node type, most of which just propagate to children (boilerplate). Strategic programming separates the what (rewrite rules) from the where (traversal strategy), eliminating boilerplate.

Primitive Combinators

All strategies are built from a small set of primitives:

CombinatorBehaviorType
idAlways succeeds, returns term unchangedTP
failAlways failsTP
s1 ; s2Sequential: apply s1, then s2 to resultTP → TP → TP
s1 <+ s2Left choice: try s1, if it fails try s2TP → TP → TP
all(s)Apply s to all direct children (fail if any fails)TP → TP
one(s)Apply s to one direct child (first that succeeds)TP → TP
some(s)Apply s to as many children as possible (≥1)TP → TP

Derived Traversal Strategies

From just all, one, sequence, and choice, we derive powerful whole-tree traversals:

-- Full traversals (visit every node)
topdown(s)    = s ; all(topdown(s))       -- apply s, then recurse into children
bottomup(s)   = all(bottomup(s)) ; s       -- recurse into children, then apply s

-- Single-hit traversals (stop after first success)
oncetd(s)     = s <+ one(oncetd(s))       -- try s here, else try one child
oncebu(s)     = one(oncebu(s)) <+ s       -- try a child first, else try s here

-- Fixed-point traversals (repeat until no more matches)
innermost(s)  = bottomup(try(s ; innermost(s)))  -- exhaustive reduction
outermost(s)  = repeat(oncetd(s))                  -- top-down fixed point

Type-Preserving (TP) vs Type-Unifying (TU) strategies

Strategies come in two flavors:

  • Type-Preserving (TP): The result has the same type as the input. Used for transformations (e.g., rename variables, simplify expressions).
  • Type-Unifying (TU): The result is a fixed type regardless of input type. Used for queries (e.g., collect all variable names, count nodes).

In Haskell's SYB framework, TP corresponds to mkT ("make transformation") and TU corresponds to mkQ ("make query").

mkT :: (Typeable a, Typeable b) => (b -> b) -> a -> a
-- If a has type b, apply f; otherwise return unchanged

mkQ :: (Typeable a, Typeable b) => r -> (b -> r) -> a -> r
-- If a has type b, query with f; otherwise return default r
Origins: Stratego and Strafunski

Stratego (Visser, 1998) introduced the idea of programmable rewriting strategies as a standalone language for program transformation. Strategies are first-class citizens: you can define, compose, and pass them as arguments.

Strafunski (Lämmel & Visser) brought strategic programming to Haskell as a library, using the SYB generic programming framework. Strategy combinators became higher-order Haskell functions. The Ztrategic library builds on this lineage but replaces SYB traversals with zipper-based navigation.

6Ztrategic: Zipping It All Together

The unified library — strategies and attribute grammars in ~200 lines of Haskell.

Core Idea

Traditional strategic programming (Strafunski) uses SYB's gmapT/gmapQ to traverse children generically. Ztrategic replaces this with zipper navigation: instead of applying a generic map over children, it moves a zipper cursor to each child, applies the strategy there, then moves to the next sibling.

Strategy Combinators in Ztrategic

-- Type-preserving strategy: transforms the focused node
type TP m = Zipper -> m Zipper

-- Type-unifying strategy: queries the focused node
type TU a m = Zipper -> m a

-- Core combinators (zipper-based)
full_tdTP :: Monad m => TP m -> TP m
full_tdTP s z = s z >>= allTP (full_tdTP s)

full_buTP :: Monad m => TP m -> TP m
full_buTP s z = allTP (full_buTP s) z >>= s

once_tdTP :: MonadPlus m => TP m -> TP m
once_tdTP s z = s z `mplus` oneTP (once_tdTP s) z

once_buTP :: MonadPlus m => TP m -> TP m
once_buTP s z = oneTP (once_buTP s) z `mplus` s z

-- allTP: apply strategy to ALL children via zipper navigation
allTP :: Monad m => TP m -> TP m
allTP s z = case down z of
  Nothing -> return z               -- leaf: no children
  Just zc -> applyToSiblings s zc    -- descend, apply to each
            >>= return . fromJust . up  -- ascend back

The Key Advantage: Strategies Can Read AG Attributes

Because strategies now operate on a zipper, and AG attributes are functions on zippers, a strategy can call any attribute function at any node it visits:

-- Strategy: remove declarations of unused variables
removeUnused :: TP Maybe
removeUnused z = case constructor z of
  "Decl" -> if isUsed z   -- isUsed is an AG attribute!
            then Nothing   -- strategy fails (don't transform)
            else Just (removeNode z)
  _      -> Nothing

-- Apply it everywhere, top-down
cleanup = full_tdTP removeUnused
Best of both worlds: Strategies provide the traversal control (where to look), while AGs provide the semantic information (what to check). Neither technique alone can do both as elegantly.

7Memoization in the Zipper

Caching attribute values to avoid redundant recomputation during traversal.

The Problem

AG attributes may be queried multiple times at the same node — especially inherited attributes that siblings share. Without caching, each query recomputes from scratch by navigating to the parent, which navigates to its parent, etc. This leads to exponential blowup.

The Solution: Memoization Table in the Zipper

The zipper's context is extended with a memo table that maps (node position, attribute name) pairs to cached values. When an attribute function is called:

  1. Check the memo table at the current zipper position
  2. If found, return the cached value immediately
  3. If not found, compute the value, store it in the memo table, and return it
-- Extended zipper with memo table
data MemoZipper a = MemoZipper
  { zipper :: Zipper a
  , memo   :: Map (NodeId, AttrName) Dynamic
  }

-- Memoized attribute lookup
memoAttr :: String -> (MemoZipper a -> v) -> MemoZipper a -> v
memoAttr name compute mz =
  case lookup (nodeId mz, name) (memo mz) of
    Just cached -> fromDynamic cached
    Nothing     -> let val = compute mz
                   in store (nodeId mz, name) val mz
                      `seq` val

Critical for strategies: When a strategy traverses a large tree, it may visit thousands of nodes. At each node, it might query AG attributes to decide whether to apply a rewrite. Without memoization, attribute evaluation would dominate runtime. With it, each attribute is computed at most once per node.

8The Repmin Problem

The classic AG benchmark: replace every leaf with the global minimum, in a single pass.

Problem Statement

Given a binary tree with integers at its leaves, produce a new tree with the same shape but where every leaf value is replaced by the minimum of all leaf values in the original tree.

Repmin Example
Input
→
Output (min = 1)

Why It's Interesting

Repmin requires two-directional information flow: the minimum flows up (synthesized), then the replacement value flows back down (inherited). In a strict language this needs two passes. Attribute grammars (and Haskell's laziness) express it as a single logical specification.

ATTR Tree [ globmin : Int || locmin : Int, result : Tree ]

SEM Tree
  | Leaf  lhs.locmin  = @value             -- synthesized: this leaf's value
          lhs.result  = Leaf @lhs.globmin    -- replace with global min

  | Fork  lhs.locmin  = min @left.locmin @right.locmin
          lhs.result  = Fork @left.result @right.result
          left.globmin  = @lhs.globmin      -- inherited: pass down
          right.globmin = @lhs.globmin

SEM Root
  | Root  tree.globmin = @tree.locmin       -- tie the knot!

The last line is the magic: the inherited globmin is defined as the synthesized locmin — creating a circular dependency that the AG system resolves via lazy evaluation or fixed-point iteration.

-- Synthesized: local minimum (flows UP)
locmin :: Zipper a -> Int
locmin z = case constructor z of
  "Leaf" -> lexeme z
  "Fork" -> min (locmin (child 1 z))
                  (locmin (child 2 z))

-- Inherited: global minimum (flows DOWN)
globmin :: Zipper a -> Int
globmin z = case constructor (parent z) of
  "Root" -> locmin z   -- tie the knot at root
  _      -> globmin (parent z) -- propagate down

-- Build result: strategy replaces each leaf
replaceLeaf :: TP Maybe
replaceLeaf z = case constructor z of
  "Leaf" -> Just (setLexeme (globmin z) z)
  _      -> Nothing

repmin = full_tdTP replaceLeaf

Notice how the strategy replaceLeaf calls the AG attribute globmin — this is the strategies + AGs combination in action.

-- Bird's circular programming solution (1984)
repmin :: Tree -> Tree
repmin t = result
  where (result, m) = go t m   -- m used before computed!

go :: Tree -> Int -> (Tree, Int)
go (Leaf n)   m = (Leaf m, n)
go (Fork l r) m = (Fork l' r', min ml mr)
  where (l', ml) = go l m
        (r', mr) = go r m

The where binds m circularly: go receives m as an argument and produces it as a result. Haskell's laziness makes this work. This is essentially what the AG compiler would generate.

Related Concepts

Scrap Your Boilerplate (SYB)

Lämmel & Peyton Jones (2003). A Haskell generic programming framework that lets you write type-generic traversals using the Data and Typeable type classes. Key combinators:

  • mkT :: (Typeable a, Typeable b) => (b -> b) -> a -> a — make a type-specific transformation generic
  • mkQ :: (Typeable a, Typeable b) => r -> (b -> r) -> a -> r — make a type-specific query generic
  • everywhere :: (forall a. Data a => a -> a) -> (forall a. Data a => a -> a) — apply transformation to all nodes
  • everything :: (r -> r -> r) -> (forall a. Data a => a -> r) -> (forall a. Data a => a -> r) — query all nodes and combine

Ztrategic builds on SYB's Data class for the generic zipper but replaces the traversal mechanism with zipper navigation.

Scrap Your Zippers (Adams 2010)

Extended SYB to define a generic zipper for heterogeneous types. Instead of defining a zipper per data type, you get one for free from the Data instance. This is the zipper that ZipperAG and Ztrategic use — it works over any algebraic data type without boilerplate.

Circular Programming (Bird 1984)

A technique exploiting lazy evaluation to express multi-pass algorithms as single-pass functions. The result of a function is fed back as one of its own arguments via Haskell's laziness. This is exactly what happens when an AG's inherited attribute depends on a synthesized attribute from the same tree: the "knot" is tied lazily.

Connection to @ksc/ag

The KSC project's @ksc/ag library takes a different approach to the same problem space:

  • @ksc/ag: Stamps lazy getters onto AST nodes via Object.defineProperty. Demand-driven evaluation. JastAdd-aligned API.
  • ZipperAG: Navigates a zipper; AG rules are functions on zipper positions. Haskell/Python.

Both are embedded library approaches (no code generation), but they differ in host language and mechanism. Ztrategic's contribution of combining strategies with AGs is relevant to @ksc/ag: could strategic combinators be layered on top of the AG library for programmatic tree transformations guided by attribute values?

What is the difference between a synthesized and an inherited attribute?
Synthesized attributes are computed from a node's children (bottom-up) — e.g., the type of an expression. Inherited attributes are passed from a node's parent (top-down) — e.g., the scope environment. Together they enable bidirectional information flow.

How does oncetd(s) differ from topdown(s)?
topdown(s) applies s to every node in the tree (top-to-bottom). oncetd(s) applies s to only the first node where it succeeds (searching top-down, left-to-right), then stops. Formula: oncetd(s) = s <+ one(oncetd(s))

Why is the zipper important for combining strategies with AGs?
Because both strategies and AG attributes are encoded as functions on zipper positions. This means a strategy traversing the tree can call any AG attribute function at the current focus position, and an AG rule can invoke a strategy. The shared zipper representation eliminates the impedance mismatch between analysis (AGs) and transformation (strategies).

What problem does memoization solve in the zipper-based AG embedding?
Without memoization, inherited attributes would be recomputed from scratch each time they're queried — navigating up to the parent, which navigates to its parent, etc., leading to exponential blowup. Memoization caches attribute values in the zipper's memo table, keyed by (nodeId, attributeName), so each attribute is computed at most once per node.

What is the "repmin" problem and why does it demonstrate AG power?
Repmin: given a tree of integers, replace every leaf with the minimum leaf value. It requires two-directional information flow: the minimum is synthesized (bottom-up), then broadcast as inherited (top-down). AGs express this declaratively; Haskell's laziness (or the AG evaluator) handles the scheduling. Bird's circular programming achieves the same in a single pass.

What does all(s) do, and how does Ztrategic implement it?
all(s) applies strategy s to every direct child of the current node; it fails if any child application fails. Strafunski uses SYB's gmapT to deconstruct and reconstruct the term. Ztrategic instead moves the zipper down to the first child, applies s, moves right to the next sibling, repeats, then zips back up. Same semantics, different mechanism.

What role does SYB's Data class play in Ztrategic?
The Data type class (from Scrap Your Boilerplate) provides reflective access to constructors and children of any algebraic data type. The generic zipper uses it to navigate into children without type-specific code. Any type that derives Data automatically gets zipper navigation, strategy application, and AG support.

How does innermost(s) achieve exhaustive normalization?
innermost(s) = bottomup(try(s ; innermost(s))). It works bottom-up: at each node, it tries to apply s. If s succeeds, it recursively re-applies innermost(s) from that point (since the rewrite may create new opportunities). It keeps going until no more rewrites apply anywhere in the tree — a fixed point.
