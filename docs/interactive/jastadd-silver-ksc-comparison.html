The Three Systems at a Glance

Overview

JastAdd Generator (Java)

Approach: Code generator. Takes .ast + .jrag/.jadd files, weaves them into Java classes with demand-driven evaluation, caching, and circular attribute support baked in.

Key innovation: Reference Attributed Grammars (RAGs) — attributes can reference arbitrary nodes in the tree, not just ancestors/descendants.

Math stance: Pragmatic. Not explicitly algebraic. Built from compiler engineering principles with the Magnusson-Hedin fixpoint algorithm bolted on.

Hedin et al., Lund University. Since ~2003.

Silver Extensible AG Language

Approach: A full language for writing AGs. Silver compiles to Java. Self-hosted (Silver is written in Silver).

Key innovation: Forwarding — extension productions delegate to host-language trees for undefined attributes, solving the expression problem. Modular well-definedness guarantees safe composition.

Math stance: More formal. Decorated/undecorated distinction is functorial. Forwarding has a formal semantics (Van Wyk, de Moor, Backhouse 2002). Modular analysis has proof-theoretic foundations.

Van Wyk et al., U. Minnesota MELT group. Since ~2006.

KSC TypeScript Library

Approach: Pure library. Three-object architecture: Grammar (functor), Semantics (product algebra), interpret() (catamorphism). Runtime compilation via compile(decl, eq).

Key innovation: De-fusion of declarations from equations. Runtime dependency analysis. Collection attributes with monoid semantics. All at library level, no code generation.

Math stance: Explicitly algebraic. Grammar = endofunctor, Semantics = product of F-algebras, interpret = catamorphism over Inh→Syn. Zipper construction via stampTree.

libs/ag/ in KSC. TypeScript. 2024–.

Through the Algebraic Lens: F-Algebras, Catamorphisms, and the Three Systems

Category Theory

The algebraic view decomposes every AG system into three concerns: Structure (endofunctor F), Behavior (F-algebra), and Orchestration (catamorphism). How closely does each system match this decomposition?

Structure: The Endofunctor F

JastAdd

The .ast file defines a Java class hierarchy. Each node type has typed, named children. This IS the functor — it defines "one layer of recursion" — but it's not expressed as a functor. It's expressed as OO inheritance.

JastAdd's grammar is multi-sorted by construction: each nonterminal (class) is a separate sort. Children have specific types, not just "child nodes."

// .ast file — the implicit functor
Program ::= CompUnit*;
CompUnit ::= ImportDecl* TypeDecl*;
TypeDecl ::= <Name:String> Member*;

No explicit functor. The Java class hierarchy IS the fixed point μF. JastAdd never separates F from μF.

Silver

Silver has a two-stage model that comes closest to the algebraic view. A nonterminal declaration defines a type. A production defines one layer of structure.

Crucially, Silver distinguishes undecorated trees (pure structure, μF) from decorated trees (structure + computed attributes). This IS the algebra/coalgebra split.

-- Nonterminal = sort in the multi-sorted algebra
nonterminal Expr;
-- Production = one layer of the functor F
abstract production add
top::Expr ::= l::Expr  r::Expr
{ ... }

Closest to F-algebra. Undecorated = μF. Decorated = μF + attribute values. The decorate operator is the catamorphism application.

KSC

The Grammar is an explicit, pure functor: a single function getChildren: (node: N) => N[]. This is deliberately the simplest possible functor — it says "a node has children" without knowing types.

The tree (KSNode) is the fixed point μF. The Grammar is F. They are separate objects.

// Grammar = pure endofunctor F
const grammar = createGrammar(getChildren);
// getChildren: KSNode => KSNode[]

Most algebraically explicit. But the functor is "untyped" — all children are N, no per-production child types. This is a single-sorted functor approximation of a multi-sorted grammar.

Behavior: The F-Algebra (A, α)

JastAdd

Equations in .jrag files are the structure map α. But they're not expressed as a separate object — they're woven into the AST classes as Java methods.

The "carrier" is the attribute value type. The "algebra" is the generated method body. But this is never reified as a first-class algebra object.

// .jrag equations = the structure map α
syn int TypeDecl.memberCount() =
  getNumMember();
inh Env TypeDecl.env();
eq CompUnit.getChild().env() =
  buildEnv();

The algebra exists, but it's fused with the functor (AST class) at compile time. No inspectable algebra object.

Silver

Equations are defined per-production in production bodies or aspect productions. Silver's aspect production mechanism allows adding equations to existing productions from separate files.

The decoration process (supplying inherited attributes, computing synthesized) is the catamorphism application. The equations collectively form the algebra.

-- Synthesized: algebra for this production
abstract production add
top::Expr ::= l::Expr  r::Expr
{
  top.value = l.value + r.value;  -- syn eq
  l.env = top.env;                -- inh eq
  r.env = top.env;
}

Equations are per-production (like F-algebra case analysis). Aspect productions add equations modularly. But no reified "algebra object."

KSC

The SpecInput IS a first-class algebra specification: declarations define what each attribute is (carrier metadata), equations define how it's computed (the structure map α).

The Semantics object is the compiled, validated product algebra — a reified, inspectable object you can pass around.

// SpecInput = algebra specification
{
  declarations: {
    kindDefs: { direction: 'syn' },
    defEnv: { direction: 'inh', root: ... },
  },
  equations: {
    kindDefs: { CompilationUnit: (cu) => ..., _: () => [] },
    defEnv: undefined,  // auto-propagate
  },
}

Most explicit. The algebra is a first-class object with inspectable declarations. compile() fuses decl + eq into AttributeDef.

Orchestration: The Catamorphism

JastAdd

The evaluator is generated Java code. Each attribute becomes a method with a _computed flag and _value field. First access computes; subsequent accesses return cached. Internally, all evaluation state is managed by a thread-local ASTNode$State singleton (cycle detection stacks, caching flags, rewrite state).

The catamorphism is implicit — demand-driven property access triggers the recursive evaluation. There's no explicit "fold" or "interpret" call.

// Generated Java (simplified)
private boolean memberCount_computed = false;
private int memberCount_value;
public int memberCount() {
  if (memberCount_computed) return memberCount_value;
  memberCount_computed = true;
  memberCount_value = getNumMember(); // equation
  return memberCount_value;
}

No reified evaluator. The Java method dispatch IS the catamorphism. Parent references via getParent().

Silver

The catamorphism is the decoration process. decorate expr with { env = ... } takes an undecorated tree (μF) and produces a decorated tree (the algebra applied).

This is the closest to an explicit catamorphism: decorate takes (tree, inherited attrs) and produces (tree + all computed attrs).

-- decorate = apply catamorphism
local decorated :: Decorated Expr =
  decorate rawExpr with {
    env = topLevelEnv;
  };
-- Now decorated.value is available (syn)

decorate ≈ cata. Undecorated → Decorated is the catamorphism. Lazy inside. Multiple decorations possible (same tree, different inherited attrs).

KSC

interpret(semantics, root) is the explicit catamorphism. It takes a sealed algebra and a tree, returning projected results. Internally: stampTree (zipper) → applyAttributes (install getters) → project (trigger evaluation).

// interpret = explicit catamorphism
const results = interpret(semantics, root);
// Returns Map<string, unknown>
// 'ksc-binder' => KindDefinition[]
// 'ksc-checker' => CheckerDiagnostic[]

Most explicit catamorphism. Three-phase: stamp (coalgebra), install (algebra), project (trigger). But lazy inside — actual computation is demand-driven like JastAdd/Silver.

Summary: Algebraic Explicitness JastAdd : Functor fused with μF (Java classes). Algebra fused with functor (woven methods). Cata implicit (method dispatch). Everything is fused. Algebraic structure exists but is not reified. Silver : Functor ≈ productions. μF = undecorated tree. Algebra ≈ equations. Cata ≈ decorate. Partially separated. Undecorated/decorated split mirrors algebra/coalgebra. KSC : Functor = Grammar. μF = KSNode tree. Algebra = Semantics. Cata = interpret(). Fully separated. All three are first-class objects.

Beyond Catamorphisms: Advanced Mathematical Frameworks

The catamorphism view (Backhouse & de Moor 2001) is the dominant lens, but several other categorical frameworks illuminate AG evaluation from different angles:

Comonadic Evaluation (Uustalu & Vene 2005)

AG evaluation can be modeled as an unfold followed by a fold — a hylomorphism. Uustalu and Vene show that the "context" each node needs (its position, parent, siblings) forms a comonad. The attribute equations are then a coalgebra of this comonad.

This directly maps to the three systems' tree models:

  • JastAdd getParent() = imperative comonadic context
  • Silver decorated tree = comonad over undecorated tree
  • KSC stampTree() = explicit comonad construction ($parent, $prev, $next)

KSC's stampTree is the most literal realization of the comonadic construction — it explicitly enriches each node with its navigational context before evaluation begins.

Traced Monoidal Categories (Katsumata 2009)

Katsumata models AGs in traced symmetric monoidal categories. The trace operator captures the "feedback loop" where synthesized attributes flow down as inherited attributes — the Inh↔Syn interaction.

This is the most abstract framework: the trace axioms guarantee that attribute evaluation terminates and is well-defined, regardless of the evaluation strategy. It subsumes both the catamorphism view and the comonadic view.

Relevance to systems: None of the three systems explicitly implement traced monoidal structure, but all three implicitly rely on the trace axioms — their demand-driven evaluation terminates precisely because the dependency graph is acyclic (or, for circular attributes, converges to a fixed point in a lattice).

Tree Models: Parent References, Zippers, and Decoration

Architecture

How each system enables bidirectional tree navigation — the key requirement for inherited attributes.

JastAdd Mutable Parent Pointers

Every ASTNode has a getParent() method. Parent pointers are set when children are added. The tree is mutable — nodes can be replaced (rewrites), parent pointers update accordingly.

This is not a zipper — it's simpler. The tree is a doubly-linked graph. Inherited attributes look up via getParent() chains.

Categorically: The parent pointer makes each node a comonadic context (it knows its surroundings), but this isn't formalized — it's just an imperative back-pointer.

Silver Decorated Trees

Silver has the most sophisticated tree model. An undecorated tree is pure data (like a Haskell ADT). A decorated tree is the tree plus all its computed attributes.

Decoration is explicit: decorate expr with { env = ... }. The same undecorated tree can be decorated multiple times with different inherited attribute sets, producing different decorated trees.

Categorically: Undecorated → Decorated is the closest thing to an explicit catamorphism in any AG system. The InhSet kind system tracks which inherited attributes have been supplied — this is dependent typing over the algebra.

KSC Stamped Zipper

stampTree(root, getChildren) stamps $parent, $children, $index, $root, $prev, $next on every node via BFS. This is the zipper construction (Badouel et al. 2011).

Then applyAttributes() installs lazy getters. The tree becomes both the data and the computation container.

Categorically: stampTree = the coalgebraic half (enriching with navigational context). applyAttributes = the algebraic half (installing equations). Together they form a bialgebra, realized imperatively.

Attribute Types: Feature-by-Feature Comparison

Deep Dive

Synthesized Attributes (Bottom-Up)

Math: F-algebra α : F(A) → A. The most basic recursion scheme — a catamorphism.

JastAdd

syn int Expr.value();
eq Add.value() =
  getLeft().value() + getRight().value();
eq Num.value() = getValue();

Equations are per-production (subclass override). Lazy by default. Generated as Java methods with _computed/_value caching.

Silver

synthesized attribute value :: Integer;
abstract production add
top::Expr ::= l::Expr r::Expr
{ top.value = l.value + r.value; }

Equations in production bodies. Aspect productions can add equations from other files. Lazy evaluation.

KSC

declarations: {
  value: { direction: 'syn' }
},
equations: {
  value: {
    Add: (n) => n.$children[0].value
              + n.$children[1].value,
    Num: (n) => n.rawValue,
  }
}

Declaration separate from equation. Production dispatch via discriminant. Compiled to installLazy getter.

Convergence: All three support synthesized attributes with demand-driven caching. The mechanism is essentially identical — the catamorphism (fold). The only difference is surface syntax and when fusion happens (compile-time for JastAdd/Silver, runtime for KSC).

Inherited Attributes (Top-Down)

Math: Part of the Inh→Syn function carrier (Backhouse-de Moor). Requires bidirectional tree navigation.

JastAdd

inh Env Expr.env();
// Parent provides equation for children
eq CompUnit.getTypeDecl().env() =
  buildEnv();
// Broadcasting: all children inherit same value

Equation location: Defined on the parent, targeting specific child accessors. JastAdd supports broadcasting: a single equation can supply all children.

Silver

inherited attribute env :: Env;
abstract production compUnit
top::Program ::= decls::[Decl]
{
  decls.env = buildEnv();
  -- explicit: parent assigns to child
}

Equation location: In the production body, parent assigns to child's inherited attrs. Part of the decorate step — inherited attrs are "supplied" during decoration.

KSC

declarations: {
  env: { direction: 'inh', root: emptyEnv }
},
equations: {
  env: (parent, child, idx) => {
    if (parent.kind === 'CompUnit')
      return buildEnv();
    return undefined; // auto-propagate
  }
}

Equation location: A function (parent, child, idx) → value. undefined = auto-propagate from parent. Root value required. Compiled to $parent navigation.

Key divergence: JastAdd/Silver define inherited equations on the parent production, targeting specific children. KSC defines them as a function from (parent, child) → value, with auto-propagation default. JastAdd's broadcasting and Silver's explicit assignment are more fine-grained than KSC's approach.

Collection Attributes (Distributed Aggregation)

Math: Catamorphism into a monoid. Each node contributes to a value that aggregates across the tree.

JastAdd

// Declaration: target + type + combiner
coll ArrayList<String> Program.errors()
  [new ArrayList<>()]
  with add;

// Contributions from anywhere in the tree
TypeDecl contributes errorMsg()
  when hasError()
  to Program.errors();

Two-phase evaluation: (1) Survey phase traverses tree finding contributors. (2) Collection phase evaluates contributions and combines. Supports root R scoping and conditional contributions.

Silver

-- Collection via contribution mechanism
synthesized attribute errors ::
  [String];

-- Aspect productions contribute
aspect production typeDecl
top::Decl ::= ...
{
  top.errors <- [errorMsg];
  -- <- is contribution operator
}

Contribution operator <- appends to a collection. Collections use monoid-like semantics. Silver's collections are more tightly integrated with forwarding — forwarded productions automatically contribute.

KSC

declarations: {
  errors: {
    direction: 'collection',
    initial: [],
    combine: (acc, c) => [...acc, ...c],
  }
},
equations: {
  errors: (node) => {
    const v = node.importViolation;
    return v ? [v] : [];
  }
}

Monoid in the declaration: initial + combine define the monoid explicitly. The equation is the per-node contribution function. compile() fuses them into a bottom-up aggregation over $children.

Key divergence: JastAdd uses a two-phase survey+collect. Silver uses a contribution operator <-. KSC makes the monoid structure explicit in the declaration. JastAdd/Silver allow contributions from distant nodes (any descendant targeting any ancestor); KSC aggregates bottom-up through the tree hierarchy.

Circular Attributes (Fixed-Point Iteration)

Math: Beyond recursion schemes. Least fixed point computation via Magnusson-Hedin algorithm (abstract interpretation).

JastAdd

syn boolean Stmt.reachable()
  circular [false] =
    computeReachable();
// Other cycle members: uncache
uncache Stmt.nextStmt();

The original implementation of Magnusson-Hedin (2007). At least one attribute per cycle must be declared circular; others must be uncached. --safeLazy option enables safe caching within cycles.

Silver

Silver does not natively support circular attributes. Circular dependencies are flagged as errors by the circularity checker.

Silver's approach: if you need fixed-point semantics, model it explicitly via iterative tree transformations (higher-order attributes that build new trees until convergence).

This is a significant divergence from JastAdd and KSC.

KSC

declarations: {
  reachable: {
    direction: 'circular',
    bottom: false,
    equals: (a, b) => a === b,
  }
},
equations: {
  reachable: (node) => computeReachable(node),
}

Implements Magnusson-Hedin directly. Module-level cycle tracking. The first accessor becomes the "cycle driver." Bottom value + equality check in declaration. All cycle participants iterate until convergence.

Key divergence: Silver deliberately omits circular attributes. JastAdd originated the algorithm. KSC reimplements it at library level. The math is the same (least fixed point over a lattice), but Silver's philosophy is "make circularity explicit" while JastAdd/KSC allow implicit cycles.

Higher-Order Attributes (Attributes that are Trees)

Math: Breaks the catamorphism model. The carrier includes new μF values — the fold can create fresh trees.

JastAdd

// NTA: non-terminal attribute
syn nta Expr Stmt.desugared() =
  new DesugaredExpr(...);

JastAdd's Non-Terminal Attributes (NTAs) are full AST subtrees. The NTA tree participates in the attribute evaluation — you can access attributes on it. NTA children are included in collection attribute surveys.

Silver

Silver has first-class support for higher-order attributes. An attribute can return an undecorated tree which can then be decorated (possibly with different inherited attributes).

Silver's forwarding is itself a higher-order mechanism — the forwarded-to tree is an HOA.

KSC

KSC does not currently support higher-order attributes. An equation can return any value, but there's no mechanism to stamp/evaluate a synthesized tree as part of the AG evaluation.

This is a gap compared to JastAdd and Silver.

Reference Attributes (Cross-Tree Pointers)

Math: Extends the algebra with morphisms between different nodes in the tree. Not a standard recursion scheme.

JastAdd

// Reference to another node
syn Decl Use.decl() =
  lookup(getName());
// Can access attributes on the target
Use.type() = decl().type();

JastAdd's key innovation: Reference Attributed Grammars (RAGs). Attributes can point to arbitrary nodes. You can then access attributes on the referenced node. This enables name resolution, type checking, etc.

Silver

-- Reference: decorated type
synthesized attribute decl ::
  Decorated Decl;
-- Access attributes on referenced node
top.type = top.decl.type;

Silver models references as attributes of Decorated type. A reference to a decorated node gives you access to all its computed attributes. The InhSet kind tracks which inherited attributes the reference carries.

KSC

// Implicit via stamped tree navigation
defLookup: (node) => {
  const env = node.defEnv; // inh attr
  return (name) => env.get(name);
}
// Returns the KindDefinition directly
// (not a node reference, but a value)

KSC doesn't have explicit reference attributes, but achieves similar results via inherited environments + parameterized synthesized attributes. The defLookup paramSyn returns definition objects, not node references.

Forwarding: Silver's Unique Algebraic Innovation

Key Divergence

Forwarding is Silver's most distinctive feature and has no direct equivalent in JastAdd or KSC. It solves the expression problem for attribute grammars.

How Forwarding Works

When an extension adds a new production (e.g., forEach loop), it can forwards to a semantically equivalent host-language tree (e.g., a while loop). Any attribute not explicitly defined on forEach is automatically computed on the forwarded-to while tree.

-- Extension production: forEach forwards to while
abstract production forEach
top::Stmt ::= var::Name list::Expr body::Stmt
{
  -- This production FORWARDS to a desugared while loop
  forwards to while(hasNext(list), block([
    assign(var, next(list)),
    body
  ]));

  -- Can still define some attributes explicitly
  top.prettyPrint = "for(" ++ var.pp ++ " in " ++ list.pp ++ ")...";

  -- All OTHER attributes (type checking, errors, code gen, etc.)
  -- are automatically evaluated on the forwarded-to while loop.
  -- This is the magic: new analyses work on new syntax automatically.
}

The Expression Problem

The classic dilemma: you want to add both new data types (productions) and new operations (attributes) independently.

  • Extension A adds forEach (new production)
  • Extension B adds typeCheck (new attribute)
  • Problem: Who defines typeCheck for forEach?
  • Forwarding solution: forEach forwards to while, so typeCheck evaluates on the while tree. No one needs to define the missing equation.

Algebraic Interpretation

Forwarding can be seen as a default morphism: when the algebra doesn't have an equation for a particular constructor, it falls back to evaluating on an equivalent constructor that does.

Formally (Van Wyk, de Moor, Backhouse 2002): forwarding is a higher-order extension of standard AGs. The forwarded-to tree is a non-terminal attribute (HOA) that provides default semantics. The circularity check extends naturally; the closure check is relaxed (some attributes need not be defined if forwarding provides them).

This is closely related to the concept of coercion in type theory — a default embedding from extension constructs into host constructs.

How Do JastAdd and KSC Handle This?

JastAdd — No Forwarding

JastAdd solves extensibility via aspect weaving (multiple .jrag files) and OO inheritance (subclass equations override superclass). But there's no mechanism for new productions to delegate to existing ones for undefined attributes.

JastAdd's rewrites partially address this: a node can be rewritten to a different node type, but this changes the tree permanently rather than providing fallback semantics.

KSC — No Forwarding

KSC's production dispatch ({ CompilationUnit: ..., _: () => default }) provides a default equation via the _ handler, which covers new productions. But this is much weaker than forwarding — the default is a fixed value, not a delegation to a semantically equivalent tree.

KSC's deps mechanism (spec ordering) enables modular extension, but doesn't solve the expression problem the way forwarding does.

Modular Composition: Aspects, Extensions, and Specs

Architecture

All three systems support defining attributes across multiple files/modules. The mechanisms and guarantees differ significantly.

JastAdd Aspect Weaving

Multiple .jrag files are woven together at code generation time. Each file is an "aspect" that contributes intertype declarations (attributes, equations, methods) to AST classes.

Validation: Compile-time. The generated Java code won't compile if equations are missing. The refine keyword allows one aspect to override another's equations.

Guarantee: If it compiles (Java), it's well-formed. But this is a whole-program check — not modular.

Silver Modular Well-Definedness

Silver's MWDA (Modular Well-Definedness Analysis) is unique. Each grammar module is verified independently with only information from its dependencies. The key rules:

  • Non-forwarding productions must define all attributes on their nonterminal
  • Extension productions must forward
  • Equations must be "exported by" either the attribute occurrence or the production

Guarantee: If each module passes MWDA, any composition of verified modules is well-defined. This is the strongest guarantee of any AG system.

KSC Runtime Validation

createSemantics(grammar, specs) performs runtime validation:

  • No duplicate attribute names across specs
  • No circular spec dependencies (topo-sort)
  • No unknown deps

The deps field orders spec evaluation. analyzeDeps() discovers attribute-level dependencies dynamically.

Guarantee: Runtime. Validates when Semantics is constructed, before any tree is evaluated.

Composability Guarantees (Strongest to Weakest) Silver : Modular proof. Each module verified independently. Any composition of verified modules is guaranteed well-defined. Forwarding fills gaps. Proof-theoretic: if Module_A ok and Module_B ok, then Module_A + Module_B ok. JastAdd : Whole-program check. Java compilation catches missing equations. Aspects weave freely, but correctness is only checked after full composition. Pragmatic: if it compiles, it works. KSC : Runtime validation. createSemantics() checks at construction time. No compile-time guarantees (TypeScript types help but don't enforce completeness). Dynamic: fast feedback, but errors found at runtime, not build time.

Convergence and Divergence: The Mathematical View

Synthesis

Where All Three Converge

The Shared Core

Despite their surface differences, all three systems implement the same mathematical core:

  • Demand-driven evaluation: Attributes are computed lazily on first access, then cached. This is universal.
  • The Inh→Syn catamorphism: All three use parent references + lazy getters to implement the Backhouse-de Moor result (AGs as catamorphisms over function types). JastAdd uses getParent(), Silver uses decoration, KSC uses $parent stamps.
  • Multi-sorted product algebra: All three compute multiple attributes of different types simultaneously. The "carrier" is always an implicit product of attribute value types.
  • Per-production dispatch: All three let equations vary by node/production type. This is the "case analysis" in the F-algebra.

Where They Diverge

Dimension Divergence Mathematical Significance
Algebra reification KSC reifies the algebra as a Semantics object. JastAdd and Silver don't — the algebra is the generated code itself. KSC can inspect, compose, and validate algebras as first-class values. The others can only do this at compile/generation time.
Functor/data separation Silver separates undecorated (functor) from decorated (algebra applied). KSC separates Grammar from tree. JastAdd fuses them into Java classes. Silver's model is closest to the categorical ideal. KSC achieves it differently (simpler functor). JastAdd sacrifices separation for Java integration.
Forwarding (expression problem) Silver has forwarding. JastAdd has rewrites. KSC has default handlers. Forwarding is a categorical "default morphism" — a principled way to extend algebras. Rewrites are coalgebraic (tree transformation). Default handlers are just wildcard patterns.
Circular attributes JastAdd and KSC support them. Silver doesn't. Circular attrs go beyond recursion schemes into abstract interpretation. Silver's stance: "make it explicit" (model as iterative HOA transforms). JastAdd/KSC: "make it transparent" (declare circular, iterate automatically).
Modular safety Silver has proof-theoretic modular guarantees. JastAdd has whole-program checking. KSC has runtime validation. Silver's MWDA is the strongest formal result. It guarantees algebra composability without seeing the full composition. This is related to the algebraic notion of "conservative extension."
Collection semantics JastAdd surveys the tree (two-phase). Silver uses contribution operators. KSC uses explicit monoid declarations. All three implement monoid catamorphisms. The difference is how contributions are gathered: global survey (JastAdd), scoped contribution (Silver), or hierarchical bottom-up (KSC).

What Each System Could Learn From the Others

Design

What JastAdd Has That Others Don't

  • Rewrites: Iterative tree rewriting with conditional rules. Powerful for desugaring. Neither Silver nor KSC has this.
  • RAGs: First-class reference attributes to arbitrary nodes. Silver has decorated references; KSC has only value-level references.
  • NTAs: Non-terminal attributes that participate in evaluation. Full higher-order AG support.
  • Mature tooling: DrAST inspector, IDE integration, decades of compiler-building experience.

What Silver Has That Others Don't

  • Forwarding: The expression problem solution. Neither JastAdd nor KSC can safely compose independent extensions.
  • MWDA: Modular well-definedness proof. The strongest composability guarantee.
  • Decorated/undecorated distinction: Closest to the categorical model. Multiple decorations of the same tree.
  • InhSet kinds: Type-level tracking of which inherited attributes are available. Dependent typing for AGs.
  • Implicit monads: Attribute computations can be lifted into monadic contexts (MonadFail, IO) — Kleisli composition over AG equations.
  • Strategy attributes: Stratego-style rewriting strategies operating on decorated trees. Enables programmable traversals as attributes.

What KSC Has That Others Don't

  • Reified algebra: The Semantics object is a first-class, inspectable, passable value. Neither JastAdd nor Silver can do this.
  • De-fused declarations: Declarations separate from equations, enabling introspection before compilation.
  • Explicit monoid collections: The monoid (initial + combine) is in the declaration, not hidden in the runtime.
  • No build step: Pure library. No code generation, no special language. Works in any TypeScript project.
  • Runtime dep analysis: Discovers attribute dependencies dynamically (analyzeDeps).

Flashcard Quiz

14 Cards
Q: Which system has the most explicit separation of functor, algebra, and catamorphism?
Click to reveal
KSC. Grammar = functor, Semantics = product algebra (first-class object), interpret() = catamorphism. All three are separate, inspectable objects. Silver partially separates via undecorated/decorated distinction. JastAdd fuses everything into Java classes.
Q: What is Silver's "forwarding" and what problem does it solve?
Click to reveal
Forwarding lets an extension production delegate to a semantically equivalent host-language tree for any attributes not explicitly defined. It solves the expression problem: independently developed extensions (one adding new productions, another adding new attributes) compose safely because forwarding fills in the missing equations. Categorically, it's a "default morphism."
Q: How do the three systems handle parent references / bidirectional navigation?
Click to reveal
JastAdd: mutable getParent() back-pointers on Java objects. Silver: implicit in decorated trees (inherited attrs are supplied during decoration, no explicit parent pointer). KSC: stampTree() installs $parent, $children, $prev, $next — the zipper construction (Badouel et al. 2011). All three enable the Inh→Syn catamorphism, just implemented differently.
Q: Which system(s) support circular attributes, and which doesn't? Why?
Click to reveal
JastAdd (original Magnusson-Hedin) and KSC (reimplemented) support circular attributes. Silver deliberately omits them. Silver's philosophy: circularity should be modeled explicitly via iterative HOA transformations, not hidden in implicit fixed-point iteration. This is a design choice, not a limitation.
Q: What is Silver's "modular well-definedness" and why can't JastAdd/KSC replicate it?
Click to reveal
MWDA verifies each grammar module independently: non-forwarding productions must define all attributes; extension productions must forward; equations must be "exported by" their occurrence or production. Any composition of verified modules is guaranteed well-defined. JastAdd does whole-program checks (weave then compile). KSC validates at runtime. Neither can guarantee composability without seeing the full composition, because they lack forwarding (which fills the gaps that modular checking can't see).
Q: How does collection attribute evaluation differ across the three systems?
Click to reveal
JastAdd: Two-phase (survey tree for contributors, then collect). Global — any node can contribute to any ancestor's collection. Silver: Contribution operator (<-) in aspect productions. KSC: Explicit monoid (initial + combine) in the CollectionDecl, per-node contribution function, bottom-up aggregation over $children. All implement monoid catamorphisms; the gathering strategy differs.
Q: What is Silver's "decorated vs undecorated" distinction, and what's the categorical interpretation?
Click to reveal
An undecorated tree is pure structure (μF). A decorated tree is the tree plus all computed attributes (the algebra applied). Decoration takes an undecorated tree + inherited attributes and produces a decorated tree. This is the closest any AG system gets to an explicit catamorphism: decorate(tree, inh) = cata(algebra)(tree)(inh). The InhSet kind system tracks which inherited attributes are available — essentially dependent typing over the algebra.
Q: What shared mathematical core do all three systems implement?
Click to reveal
All three implement: (1) demand-driven lazy evaluation with caching, (2) the Backhouse-de Moor result (AGs as catamorphisms over Inh→Syn function types), (3) multi-sorted product algebras (multiple attributes with different types), (4) per-production case analysis in equations. The surface syntax differs vastly, but the computational model is the same.
Q: What does KSC have that neither JastAdd nor Silver has?
Click to reveal
A reified, first-class algebra object (Semantics). You can inspect declarations, compilation order, and the dependency graph without evaluating any tree. Also: de-fused declarations (what an attribute is) from equations (how it's computed), explicit monoid semantics in collection declarations, and runtime dependency analysis via analyzeDeps(). No build step required.
Q: How do JastAdd's rewrites relate to Silver's forwarding?
Click to reveal
Both provide alternative tree semantics, but differ fundamentally. Rewrites (JastAdd) permanently replace nodes in the mutable tree — they're coalgebraic (tree transformation). Forwarding (Silver) provides fallback semantics without changing the tree — the original production remains, and undefined attributes are looked up on the forwarded-to tree. Forwarding is non-destructive and enables the expression problem solution; rewrites are destructive and don't.
Q: Why doesn't KSC support higher-order attributes (NTAs)?
Click to reveal
KSC's three-phase interpret() (stamp → install → project) operates on a fixed tree. HOAs require creating new subtrees during evaluation and evaluating attributes on them. This would need: (1) a mechanism to stamp/evaluate synthesized subtrees, (2) integration with the demand-driven cache. It's feasible (add an 'hoa' direction to AttrDecl where compile() stamps and evaluates the result tree) but not yet implemented.
Q: Where does each system sit on the pragmatic-to-algebraic spectrum?
Click to reveal
JastAdd = most pragmatic (OO + Java, implicit algebra, whole-program check). Silver = middle (formal AG language, partial separation via decorated/undecorated, proof-theoretic MWDA). KSC = most algebraically explicit (Grammar=functor, Semantics=product algebra, interpret=catamorphism, all as first-class objects). But all three implement the same mathematical core — they just expose different amounts of the algebraic structure.
Q: What is the comonadic view of AG evaluation (Uustalu & Vene), and how does it map to each system's tree model?
Click to reveal
Uustalu & Vene show that the navigational context each node needs (parent, siblings, position) forms a comonad. Attribute equations are a coalgebra of this comonad. JastAdd's getParent() is an imperative comonadic context. Silver's decoration enriches an undecorated tree with its full context. KSC's stampTree() is the most literal realization — it explicitly constructs the comonad by stamping $parent, $children, $prev, $next on every node before evaluation begins.
Q: What are Silver's "implicit monads" and "strategy attributes"?
Click to reveal
Implicit monads allow attribute computations to be lifted into monadic contexts (MonadFail, IO, etc.) — essentially Kleisli composition over AG equations. This lets error handling and effects compose through the attribute grammar without manual plumbing. Strategy attributes bring Stratego-style programmable rewriting strategies into Silver, operating on decorated trees. These are Silver-unique features that neither JastAdd nor KSC currently offers.
