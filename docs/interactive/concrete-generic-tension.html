01The Fundamental Tension

Every typed generic system faces this: concrete code knows things that generic code cannot. The question is how much knowledge survives the crossing.

CONCRETE GENERIC
Record<'If'|'While', H> satisfies Record<K, H> factory<K>(grammar) Record<K, H> Record<string, H>

What Concrete Gives You

  • Exhaustiveness — miss a case, get a compile error
  • Literal narrowing — TS knows the exact string values
  • Autocomplete — IDE suggests valid keys
  • Refactor safety — rename propagates everywhere
  • Return type checking — wrong type = red squiggle

What Generic Gives You

  • Reusability — one evaluator for any grammar
  • Decoupling — engine doesn't import specs
  • Composability — swap grammar/spec pairs freely
  • Testability — mock grammars with 5 kinds
  • Extensibility — new grammar = zero engine changes

The Dispatch Table Example

Consider a dispatch table that must have a handler for every AST node kind:

// CONCRETE — full exhaustiveness checking
type TSNodeKind = 'SourceFile' | 'ClassDecl' | 'IfStmt' | 'Block' | /* ...360 more */;

const dispatch: Record<TSNodeKind, Handler> = {
  SourceFile: handleSourceFile,
  ClassDecl:  handleClassDecl,
  // IfStmt: ???   ← COMPILE ERROR: missing property 'IfStmt'
};
// GENERIC — reusable but unverified
function createEvaluator<K extends string>(
  dispatch: Record<K, Handler>
): Evaluator { ... }

// K is inferred from what you pass in — but the engine
// internally only knows "K extends string". No exhaustiveness.
// No autocomplete. No literal narrowing.
Key insight: You don't have to choose. The trick is to write concrete cases separately (with full safety), then bridge them into generic interfaces. TypeScript has at least 10 techniques for this — the question is which bridge to use at each boundary.

Stage 1: Concrete Equation Functions

Full type safety. Each equation function has a precise signature: it accepts KindCtx<KSCompilationUnit> and returns KindDefinition[]. TypeScript verifies the return type, parameter access, and all attribute reads within the function body.

export const eq_kindDefs_CompilationUnit = withDeps([],
  function eq_kindDefs_CompilationUnit(
    ctx: KindCtx<KSCompilationUnit>
  ): KindDefinition[] {
    // Full narrowing: ctx.node is KSCompilationUnit
    // Return type checked: must be KindDefinition[]
  }
);

What you have: return type checking, parameter narrowing, attr() type safety (if using TypedAGNode), IDE autocomplete on node fields.

Stage 2: AnalysisSpec Packaging

Return types erased. When equations enter AnalysisSpec<K, P>, they become Partial<Record<K, Function>>. The Function type is TypeScript's top type for callables — no signature information survives.

interface SynAttr<K> {
  direction: 'syn';
  equations: Partial<Record<K, Function>>;  // ← No signature info!
  type: string;                                 // ← Type as string, not TS type
}

What's lost: return types become string descriptions, function signatures become Function, exhaustiveness is only Partial<Record> (allows missing keys). What survives: K still links to concrete kinds — you can't put a MockKind equation into a TSNodeKind spec.

Stage 3: Codegen Compilation

Types become strings. compileAnalysis() reads fn.name from equation functions and emits them as string code. The generated dispatch uses ctx as unknown as KindCtx<...> — a double cast through unknown.

// Generated code (string output from compileAnalysis):
case 'CompilationUnit':
  return eq_kindDefs_CompilationUnit(
    ctx as unknown as KindCtx<KindToNode['CompilationUnit']>
  );

What's lost: direct function references become string-assembled code, casts bypass type checking. What's gained: the generated .ts file is type-checked by tsc, so the never exhaustive default catches missing cases in the generated output.

Stage 4: DispatchConfig

Attribute names become strings. DispatchConfig is Record<string, DispatchEntry>. No type-level knowledge of which attributes exist, their types, or their directions. The evaluator receives this as an opaque bag of handlers.

type DispatchConfig = Record<string, DispatchEntry>;

// No way to ask: "does this have a 'kindDefs' entry?"
// No way to ask: "what does 'kindDefs' return?"
// Only runtime: dispatch['kindDefs'] ?? throw

What's lost: attribute names, attribute types, attribute completeness. What's recovered: wireEvaluator() does a runtime validation check — missing attributes throw before evaluation begins.

Stage 5: AGNode.attr()

Complete type erasure. The Ctx interface exposes attr(name: string, ...args: unknown[]): any. Callers pass string names and get any back. No compile-time verification that the attribute exists or that the return type is correct.

interface Ctx {
  attr(name: string, ...args: unknown[]): any;  // ← Full erasure
}

// Consumer code:
const defs = ctx.attr('kindDefs');  // type: any
// Typo? ctx.attr('kindDef') — no compile error, runtime crash

Partial recovery: TypedAGNode<M> wraps this with a typed overlay: attr<A extends keyof M>(name: A): M[A]. But only at the public API boundary, not inside equations.

The paradox: The places with the most type erasure (Ctx.attr) are the most frequently called code paths. Every equation function calls attr() multiple times. This is where incorrect types cause the most bugs — but where TypeScript provides the least help.

1. satisfies — Write Concrete, Present Generic

The satisfies operator (TS 4.9+) validates that an expression conforms to a type without widening it. The object retains its literal type for downstream use, while the constraint ensures completeness.

const dispatch = {
  SourceFile:  { compute: eq_SourceFile },
  ClassDecl:   { compute: eq_ClassDecl },
  IfStatement: { compute: eq_IfStatement },
  // Missing 'Block'? COMPILE ERROR here!
} satisfies Record<TSNodeKind, DispatchEntry>;

// dispatch's type is STILL the literal object type
// dispatch.SourceFile is known — not just DispatchEntry

// But it can be passed to generic functions:
createEvaluator(dispatch); // accepts Record<string, DispatchEntry>
✓ Exhaustiveness ✓ Literal types retained ✓ Autocomplete ✓ Refactor-safe
Best for: Adapter-level objects that you define once and pass into generic machinery. This is the simplest bridge technique — try it first.

2. Factory Capture — K Inferred from Runtime

A generic factory function accepts a value whose type carries K (e.g., Grammar<K>). Once K is captured, any subsequent calls on the factory have K concretely resolved. The call site gets full exhaustiveness checking.

function createDispatchFactory<K extends string>(
  grammar: Grammar<K>
) {
  return {
    define(entries: Record<K, DispatchEntry>): DispatchConfig {
      return entries;
    }
  };
}

// Usage — K = TSNodeKind inferred from tsGrammar
const factory = createDispatchFactory(tsGrammar);
factory.define({
  SourceFile: ...,
  // Missing kind? COMPILE ERROR — K is TSNodeKind here!
});
✓ Exhaustiveness at call site ✓ Generic factory is reusable ~ Returns generic type (K erased in output)
This is what KSC's K-linking already does. CodegenTarget<K, P> captures K from grammar and ensures the spec matches. The pattern could be extended further into dispatch.

3. Accumulating Builder — Incremental Exhaustiveness

A builder whose type parameter grows with each call. build() is only callable when the accumulated type equals the required type. TypeScript enforces this at compile time.

class DispatchBuilder<K extends string, Built extends string = never> {
  private entries = {} as Record<string, DispatchEntry>;

  add<N extends Exclude<K, Built>>(
    kind: N,
    entry: DispatchEntry
  ): DispatchBuilder<K, Built | N> {
    this.entries[kind] = entry;
    return this as any;
  }

  // Only callable when Built = K (all kinds covered)
  build(this: DispatchBuilder<K, K>): DispatchConfig {
    return this.entries;
  }
}

new DispatchBuilder<'If' | 'While' | 'For'>()
  .add('If', handler1)
  .add('While', handler2)
  // .build()  ← ERROR! 'For' not yet added
  .add('For', handler3)
  .build() // ✓ OK, all kinds covered
✓ Incremental exhaustiveness ✓ Prevents duplicates ✗ Verbose for 364 kinds ~ Can slow IDE type-checking

4. Branded / Phantom Types — Tag and Recover

Attach an invisible type-level "brand" to a generic container. Later, generic code can extract K from the brand. The brand has zero runtime cost — it exists only in the type system.

// Brand a generic container with its concrete type info
type BrandedDispatch<K extends string> =
  DispatchConfig & { readonly __kinds: K };

function brandDispatch<K extends string>(
  grammar: Grammar<K>,
  config: DispatchConfig
): BrandedDispatch<K> {
  validateDispatch(config, grammar); // Runtime check!
  return config as BrandedDispatch<K>;
}

// Generic code can now recover K:
function useDispatch<K extends string>(
  d: BrandedDispatch<K>
) {
  // K is available for type-level operations
}
✓ K recoverable downstream ✓ Zero runtime cost ✗ Convention-based (can be bypassed with cast)

5. Mapped Type Validators — Compile-Time Error Messages

Use conditional types to produce custom error messages when keys are missing. The error appears as a type mismatch with a descriptive string.

type ValidateExhaustive<
  Required extends string,
  Provided extends Record<string, unknown>
> = Exclude<Required, keyof Provided> extends never
  ? Provided
  : { readonly __error: `Missing: ${Exclude<Required, keyof Provided>}` };

function register<K extends string, D extends Record<K, Handler>>(
  grammar: Grammar<K>,
  dispatch: D & ValidateExhaustive<K, D>
): void;

// Missing 'Block' → error: { __error: "Missing: Block" }
✓ Custom error messages ✓ Works in generic context ✗ Can be cryptic with large unions

6. Concrete Subclass of Generic Base

The classic OOP bridge: an abstract generic base class defines the machinery, a concrete subclass fills in the specifics with full type checking.

abstract class Evaluator<K extends string, P> {
  abstract dispatch: Record<K, DispatchEntry>;
  abstract projections: { [Key in keyof P]: (root: Ctx) => P[Key] };

  evaluate(node: ASTNode): EvaluationResult<P> {
    // Generic machinery — reusable for any K, P
  }
}

class TSKindCheckEvaluator
  extends Evaluator<TSNodeKind, KSCProjections> {
  dispatch = {
    SourceFile: ...,
    ClassDecl: ...,
    // Missing kind? COMPILE ERROR!
  };
  projections = { ... };
}
✓ Full exhaustiveness in subclass ✓ Generic reuse in base ✗ Inheritance coupling

7. TypedAGNode<M> — Typed Overlay

Wrap the untyped attr(name: string): any with a typed interface that maps attribute names to their return types. The runtime is unchanged — only the type-level view is narrowed.

interface KSCAttrMap {
  kindDefs: KindDefinition[];
  defEnv: Map<string, KindDefinition>;
  nodeCount: number;
}

interface TypedAGNode<M extends Record<string, unknown>> {
  attr<A extends keyof M & string>(name: A): M[A];
}

// Usage — fully typed attribute access!
function equation(ctx: TypedAGNode<KSCAttrMap>) {
  const defs = ctx.attr('kindDefs');     // KindDefinition[] ✓
  ctx.attr('nonExistent');               // COMPILE ERROR ✓
}
✓ Attribute names checked ✓ Return types inferred ~ Requires parallel type map ✗ Runtime still untyped
KSC already has this! TypedAGNode<M> exists in evaluator/types.ts. The opportunity is to use it more pervasively — inside equations, not just at the consumer API.

8. Injected Type Guards — Runtime Safety in Generic Code

Pass narrowing/assertion functions into generic code as dependencies. The generic engine calls them, and TypeScript narrows the types based on the guard signatures.

interface TypeGuards<K extends string> {
  isKind(
    node: ASTNode, kind: K
  ): node is ASTNode & { kind: K };
  assertKind(
    node: ASTNode, kind: K
  ): asserts node is ASTNode & { kind: K };
}

function createEvaluator<K extends string>(
  guards: TypeGuards<K>,
  dispatch: Record<K, DispatchEntry>
) {
  return (node: ASTNode) => {
    guards.assertKind(node, node.kind as K);
    // node is now narrowed — type guard works!
  };
}
✓ Runtime narrowing in generic code ✓ Catches invalid kinds at runtime ✗ Performance cost

9. Module Augmentation / Registry Pattern

A central empty interface that concrete modules augment via declaration merging. The generic code sees the merged type. Each adapter adds its own types without coupling to others.

// evaluator/registry.ts — generic, starts empty
export interface AttrRegistry {}

// specs/ts-ast/kind-checking/register.ts — concrete augmentation
declare module '../evaluator/registry' {
  interface AttrRegistry {
    'kind-checking': {
      kindDefs: KindDefinition[];
      defEnv: Map<string, KindDefinition>;
    };
  }
}

// Generic code uses the merged type:
type AllAttrs = AttrRegistry[keyof AttrRegistry];
✓ Distributed type definitions ✓ No adapter coupling ~ Declaration merging can be fragile

10. const Type Parameters (TS 5.0+)

The const modifier on type parameters tells TypeScript to infer literal types from arguments, without the caller needing as const.

function defineGrammar<const K extends readonly string[]>(
  kinds: K
): Grammar<K[number]> { ... }

// K inferred as readonly ['If', 'While', 'For'] (literal!)
const g = defineGrammar(['If', 'While', 'For']);
// g: Grammar<'If' | 'While' | 'For'>  (not Grammar<string>!)

// Without const: would infer Grammar<string>
// With const: infers the literal union
✓ Automatic literal inference ✓ No as const at call site ✗ Only for definition-site values

04The Meta-Pattern: Concrete Shell, Generic Core

The recommended architecture places concrete code at the outside (where you write it) and generic code at the inside (where it runs). The bridge layer transfers type information between them.

CONCRETE SHELL — Where You Write

Specs, equations, adapters. Written with full concrete types: KindCtx<KSCompilationUnit>, Record<TSNodeKind, DispatchEntry>. Get exhaustiveness, narrowing, autocomplete, return type checking. This is where type safety matters most — it's where humans write code.

Files: specs/ts-ast/kind-checking/equations.ts, specs/ts-ast/grammar/nodes.ts

BRIDGE — Type Information Transfer

Port interfaces with K-linking. Grammar<K>, AnalysisSpec<K, P>, CodegenTarget<K, P>. K is captured from the concrete shell and carried inward. Techniques: satisfies, factory capture, branded types.

Files: grammar/derive.ts, analysis/types.ts, evaluator/types.ts, ports.ts

GENERIC CORE — Where It Runs

Evaluator engine, compile machinery, schema utils. Works with any K extends string. Has no concrete knowledge — receives everything through port interfaces. Reusable across grammars, testable with mocks.

Files: evaluator/engine.ts, analysis/compile.ts, grammar/schema-utils.ts

Key insight: Type safety is a property of where code is written, not where it runs. The human writes concrete cases in the shell (with all of TypeScript's help). The machine executes generic code in the core (where flexibility matters more than IDE assistance). The bridge determines how much type information crosses from one to the other.

05KSC Thinning Analysis

Where type safety thins in the KSC codebase, from strongest to weakest. Click each stage to see what's lost and how it could be recovered.

K-Linking: Grammar + Spec Pairing

TypeScript prevents mismatched grammar/spec pairs via the K type parameter.

CodegenTarget<TSNodeKind, KSCProjections> requires both grammar: Grammar<TSNodeKind> and spec: AnalysisSpec<TSNodeKind, ...>. If you try to pass a Grammar<MockKind> with an AnalysisSpec<TSNodeKind, ...>, TypeScript catches it at compile time.

Status: This is the strongest type safety in the system. No changes needed.

Generated Dispatch: Exhaustive Switch

Generated dispatch.ts has default: { const _exhaustive: never = _kind; } — tsc checks it.

The generated code is checked by TypeScript when you run tsc. If a kind is missing from the switch, the never assignment fails. This is exhaustiveness checking via codegen — the generated concrete code gets the same safety as hand-written concrete code.

Status: Effective. The safety exists in the generated output, not the generic machinery.

Equation Functions: Typed Signatures, Untyped Container

Individual equations have precise types, but AnalysisSpec stores them as Function.

Each equation like eq_kindDefs_CompilationUnit(ctx: KindCtx<KSCompilationUnit>): KindDefinition[] is fully typed internally. But when packaged into SynAttr<K>.equations, they become Partial<Record<K, Function>>. The return type information is lost.

Could recover with: A typed equation map like Record<K, (ctx: Ctx) => T> where T is the attr's declared type. Or use satisfies on the equations object before it enters the spec.

withDeps(): Annotation Disconnected from Reality

Declared deps (withDeps(['kindDefs'], fn)) may not match actual attr() calls in the function.

The dependency graph is manually annotated. Nothing prevents withDeps([], fn) where fn actually calls ctx.attr('kindDefs'). This can cause evaluation order bugs that only manifest at runtime with specific tree shapes.

Could recover with: A typed wrapper that tracks deps at the type level: withDeps<['kindDefs']>(fn) where fn's parameter type is constrained to only access those attrs. Or a lint rule that compares annotations to actual calls.

DispatchConfig: String-Keyed Attribute Names

Record<string, DispatchEntry> — no type-level knowledge of which attributes exist.

After codegen, the dispatch config is Record<string, DispatchEntry>. The evaluator engine does a runtime lookup: this._config.dispatch[name]. Typos in attribute names cause runtime errors, not compile errors. wireEvaluator() validates at construction time, but not at every call site.

Could recover with: Branded dispatch (BrandedDispatch<K>) or a typed dispatch interface generated alongside dispatch.ts, mapping attr names to their types.

Ctx.attr(): Complete Type Erasure

attr(name: string, ...args: unknown[]): any — the most-called function has the least safety.

Every equation function calls ctx.attr() to read other attributes. The signature is attr(name: string, ...args: unknown[]): any. A typo in the name, wrong parameter, or incorrect type assumption won't be caught until runtime. This is called hundreds of times per evaluation.

Could recover with: TypedAGNode<KSCAttrMap> used inside equation functions (not just at the public API). This would require equations to accept TypedCtx<KSCAttrMap> instead of bare Ctx.

Kind-to-Equation Cast: ctx as unknown as KindCtx<T>

Generated dispatch casts through unknown — if the wrong equation runs, the cast silently succeeds.

The generated switch/case dispatches to per-kind equations. Each case casts ctx as unknown as KindCtx<KindToNode['CompilationUnit']>. If there's a mismatch (equation for kind X called with node of kind Y), the cast succeeds silently. The function then tries to access fields that don't exist, causing subtle runtime errors.

Could recover with: Injected type guards that validate the node kind before the cast. Or a runtime assertion: assert(ctx.node.kind === 'CompilationUnit') inside each case.

06Advanced Patterns

Three sophisticated combinations that push the boundary of what TypeScript can enforce across the concrete/generic divide.

Pattern A: The Typed Equation Contract

Instead of Function, define a typed equation interface that preserves the return type:

// Port: typed equation — preserves return type T
interface TypedEquation<T> {
  (ctx: Ctx): T;
  deps?: string[];
}

// Port: typed attribute declaration
interface TypedSynAttr<K extends string, T> {
  direction: 'syn';
  equations: Partial<Record<K, TypedEquation<T>>>;
  default?: TypedEquation<T> | T;
}

// Adapter: TypeScript verifies return type at definition
const kindDefsAttr: TypedSynAttr<TSNodeKind, KindDefinition[]> = {
  direction: 'syn',
  equations: {
    CompilationUnit: eq_kindDefs_CompilationUnit,
    // If eq returns wrong type → COMPILE ERROR
  },
  default: () => [],
};

Pattern B: Phantom Dep Tracking

Use the type system to enforce that declared deps match actual usage:

// A Ctx that only allows reading declared deps
interface RestrictedCtx<Allowed extends string> {
  attr<A extends Allowed>(name: A): any;
  readonly node: ASTNode;
  readonly children: readonly RestrictedCtx<Allowed>[];
}

// withDeps now constrains the ctx parameter type
function withDeps<const D extends readonly string[]>(
  deps: D,
  fn: (ctx: RestrictedCtx<D[number]>) => any
) { ... }

// Now this is a COMPILE ERROR:
withDeps([], (ctx) => {
  ctx.attr('kindDefs');  // ERROR: 'kindDefs' not in never
});

Pattern C: Generic + satisfies + Factory (Triple Bridge)

Combine all three techniques for maximum safety at each layer:

// 1. Write concrete with satisfies (exhaustiveness)
const equations = {
  CompilationUnit: eq_kindDefs_CompilationUnit,
  Identifier: eq_kindDefs_Identifier,
  // Missing kind? COMPILE ERROR
} satisfies Record<TSNodeKind, TypedEquation<KindDefinition[]>>;

// 2. Pass through factory capture (K-linking)
const spec = createSpec(tsGrammar, {  // K captured here
  attrs: [{ name: 'kindDefs', equations }],
});

// 3. Brand the output (K recoverable downstream)
const evaluator = wireEvaluator({
  grammar: tsGrammar,  // K = TSNodeKind
  spec,
  dispatch: brandedDispatch,
});
The layers compose: satisfies gives exhaustiveness at the definition site. Factory capture links K at the composition boundary. Branded types let downstream generic code recover K when needed. Each technique covers a different gap.

Q

Why can't Record<K extends string, V> give exhaustiveness checking inside a generic function?

A

Inside a generic function, K is an unresolved type variable. TypeScript doesn't know the specific members of K, so it can't verify that all members are present. Exhaustiveness checking requires the compiler to enumerate all members of a union — which only works when the union is concretely known at the checking site.

Q

How does satisfies differ from a type annotation for preserving type information?

A

A type annotation (const x: Type = ...) widens the value to the declared type, losing literal information. satisfies validates the constraint but lets the value retain its narrower inferred type. So const x = { a: 1 } satisfies Record<string, number> keeps the type as { a: number } (with known key "a"), not Record<string, number>.

Q

What type information is lost when an equation function enters AnalysisSpec<K, P> as Partial<Record<K, Function>>?

A

The function signature is lost entirely: parameter types (KindCtx<SpecificNode>), return type (KindDefinition[]), and arity. Function is TypeScript's top callable type — it accepts any args and returns any. K-membership of the equation keys survives (can't put invalid kinds), but completeness is weakened to Partial.

Q

What is the "Concrete Shell, Generic Core" meta-pattern?

A

Write concrete cases at the outside (adapters/specs) with full type safety: exhaustiveness, narrowing, return type checking. Use generic machinery at the inside (engine/compiler) for reusability. Connect them via a bridge layer (port interfaces with K-linking, satisfies, branded types) that transfers as much type information as possible across the boundary.

Q

How does the accumulating builder pattern enforce exhaustiveness incrementally?

A

The builder has two type params: K (all required kinds) and Built (kinds added so far, starts as never). Each .add(kind) call accepts Exclude<K, Built> and returns Builder<K, Built | kind>. The .build() method has a this: Builder<K, K> constraint — only callable when Built = K, meaning all kinds have been added.

Q

In KSC, the most-called function (Ctx.attr()) has complete type erasure. What technique could recover safety there?

A

TypedAGNode<M> — a typed overlay that maps attr<A extends keyof M>(name: A): M[A]. KSC already has this interface in evaluator/types.ts. The opportunity is to use it inside equation functions (not just at the public API), so that ctx.attr('kindDefs') returns KindDefinition[] instead of any.

Q

Why is factory capture (K-linking) effective even though the factory function itself is generic?

A

Because TypeScript resolves K at the call site, not inside the factory. When you call createFactory(tsGrammar), K is inferred as TSNodeKind from the argument type. All subsequent operations on the returned object have K concretely resolved. The factory is generic in definition but concrete in use — safety lives where the human writes code.
