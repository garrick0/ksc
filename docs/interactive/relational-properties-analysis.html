Relational Properties in KindScript

Why noDependency and noCycles can't be AG attributes, what architectural changes they need, and the options for implementing them.

▶

Two Fundamentally Different Kinds of Properties

Concepts

KindScript's checker design doc describes two categories of properties. They look similar in config but require completely different analysis architectures.

Intrinsic Properties

Properties of a single scope's AST. "Does this code do X?"

  • noImports — no import references
  • noConsole — no console.* usage
  • immutable — no let/var bindings
  • noMutation — no =, ++, delete
  • noSideEffects — no expression statements
  • noIO — no IO module usage
  • static — no dynamic import()

Scope: single node / single subtree

Relational Properties

Constraints between members of a composite. "Does member A import from member B?"

  • noDependency([A, B]) — A must not import B
  • noCycles([A, B, C]) — no circular deps
  • noTransitiveDependency — no reachable path
  • noSiblingDependency — no inter-member imports at all

Scope: import graph across files

The key distinction: Intrinsic properties can be checked by looking at one node and its subtree. Relational properties require looking at the relationships between entire files/directories — a fundamentally different data structure than the AST.

What a composite looks like in config

export default defineConfig({
  app: {
    members: {
      domain:         { path: './src/domain',         rules: { pure: true, noIO: true } },
      infrastructure: { path: './src/infrastructure' },
      application:    { path: './src/application',    rules: { noConsole: true } },
    },
    rules: {
      // Relational: domain must not depend on infrastructure or application
      noDependency: [['domain', 'infrastructure'], ['domain', 'application']],
      // Relational: no circular dependencies among any members
      noCycles: ['domain', 'infrastructure', 'application'],
    },
  },
});

The rules on each member (pure, noIO, noConsole) are intrinsic — they're checked per-node in the AST. The top-level rules (noDependency, noCycles) are relational — they need a graph built from all members' imports.

What KSC Supports Today

Current State

KSC uses an Attribute Grammar (AG) compiled into a single-pass tree evaluator. Here's the full pipeline:

TS Source Files | v ts.createProgram() --> TypeScript Program (parser + binder + checker) | v buildKSTree() --> KSC AST (stamps: sym* flags, typeString, resolvedFileName, ...) | v evaluate() --> AG Evaluation (22 attributes, single bottom-up pass) | kindDefs (syn) --> defEnv (inh) --> defLookup (syn) | kindAnnotations (syn) --> 8x context (inh) --> 8x violation (syn) | all violations --> allViolations (collection, bottom-up) v { definitions, diagnostics } (returned to program.ts)

The 22 AG attributes

PhaseAttributeDirectionWhat it does
BinderkindDefssynExtracts Kind<...> type aliases from CompilationUnit
BinderdefEnvinhPropagates all definitions down from root
BinderdefLookupsynClosure for name-to-definition lookup
CheckerkindAnnotationssynFinds kind annotations on VariableDeclaration
Checker8 context attrsinhEach property's context inherits from annotated parent
Checker8 violation attrssynEach property checks its node-local violation
CheckerallViolationscollectionAggregates all per-node violations bottom-up
CheckernodeCountcollectionCounts nodes bottom-up

Stamped fields available for cross-file analysis

During AST conversion, the TS checker stamps these fields onto KSC AST nodes:

NodeFieldValue
IdentifierresolvesToImportWhether this identifier resolves to an imported binding
IdentifierresolvedFileNameAbsolute path of the file the import was declared in
IdentifiersymIsAliasWhether the symbol is an import alias
ImportDeclarationresolvedModulePathAbsolute file path this import resolves to

What works today

All 8 intrinsic properties are implemented as AG attributes with per-node checking. The stamped fields give us the raw data for cross-file analysis. But there is no infrastructure to build or query an import graph between composite members.

Why the AG Model Can't Express Relational Properties

Architectural Gap

This is the core issue. AG attributes operate on a tree. Relational properties operate on a graph. These are fundamentally different structures with different query patterns.

How AG attributes flow through the tree

Program direction | +- CompilationUnit (math.ts) kindDefs = [NoImports] ↑ syn | | | +- VariableDeclaration kindAnnotations = [NoImports] | | | +- ArrowFunction noImportsContext = NoImports ↓ inh | | | +- Identifier "helper" importViolation = VIOLATION ↑ syn | +- CompilationUnit (helpers.ts) | +- FunctionDeclaration "helper" (no annotations, no checking)

Each node can only see:

  • Its parent chain (via inherited attributes flowing down)
  • Its subtree (via synthesized attributes flowing up)
  • Nothing else — no sibling CUs, no files outside the subtree

What relational properties need: a member import graph

Member Import Graph domain/ ---imports---> infrastructure/ ---imports---> application/ ^ | | | +--------------------imports------------------------+ noDependency(['domain', 'infrastructure']) => Check: does domain/ import infrastructure/? noCycles(['domain', 'infra', 'app']) => Check: is there a cycle in this graph?

This requires:

  • Knowing which files belong to which member (path-based mapping)
  • Resolving every import to a target file (module resolution)
  • Classifying target files by member (which member does this import land in?)
  • Graph algorithms (DFS for cycle detection, edge filtering for dependency checks)

Five reasons AG can't do this

1 AG attributes are vertical, import graphs are horizontal

AG attributes propagate up and down the tree spine (parent-child relationships). But import dependencies are sibling relationships between CompilationUnits. A node in math.ts has no AG mechanism to query nodes in helpers.ts.

2 No concept of "members" in the tree

The AST has CompilationUnits (files) as children of Program. But there's no tree-level concept of "domain member = files under ./src/domain/". This is a config-level mapping that doesn't exist in the AST structure.

3 Collection attributes aggregate vertically, not across files

allViolations is a collection attribute that aggregates bottom-up: each node contributes, and the root collects everything. But you can't have a collection that "combines import edges from file A with member mappings from config" — that's not tree aggregation, it's graph construction.

4 Cycle detection requires global state

Checking noCycles requires DFS or Tarjan's algorithm over the complete member graph. AG evaluation visits each node exactly once in topological attribute order. It doesn't support "visit multiple files, build adjacency lists, then DFS for back-edges" within a single attribute computation.

5 The input data isn't in the tree

Relational property config (noDependency: [['domain', 'infrastructure']]) lives in a config file, not in the AST. The evaluator's Ctx interface has no way to access config data. Even if we added it, the member-to-path mapping is external to the tree structure.

Bottom line: You cannot express "does member A import from member B across all files in a project" as a tree attribute. It's not a limitation of our AG implementation — it's a limitation of tree-based attribute grammars in general. Relational properties need a different analysis phase.

What a Relational Checker Needs

Requirements

Regardless of where in the pipeline we put it, a relational checker needs these components:

1 Member resolution: config path → set of files

Given { path: './src/domain' }, resolve to all .ts files under that directory. For file members, resolve to a single file. This maps config entries to concrete CompilationUnits.

// Input: config member definitions
{ domain: { path: './src/domain' } }

// Output: member → file set
Map<string, Set<string>>
domain → { 'src/domain/handler.ts', 'src/domain/service.ts', 'src/domain/models/user.ts' }
2 Import edge extraction: for each file, find all imports

Walk each file's ImportDeclaration nodes. Use the resolvedModulePath stamped field (already on every ImportDeclaration) to get the absolute path each import points to.

// For src/domain/handler.ts:
import { readConfig } from '../infrastructure/config';
//  resolvedModulePath = '/abs/path/src/infrastructure/config.ts'

// Edge: domain → infrastructure (via handler.ts → config.ts)
3 Edge classification: resolved path → which member?

For each resolved import path, determine which member (if any) it belongs to. If the resolved path is in the domain file set, the edge targets domain. If it's in infrastructure's file set, the edge targets infrastructure. If it's outside all members (e.g., node_modules), ignore it.

4 Graph construction: member-level adjacency 
interface MemberDependencyGraph {
  members: Map<string, Set<string>>;  // member name → file paths
  edges: Array<{
    from: string;       // source member
    to: string;         // target member
    sourceFile: string; // which file has the import
    importPath: string; // which file it resolves to
  }>;
}
5 Constraint checking: graph algorithms
ConstraintAlgorithmComplexity
noDependency([A, B])Filter edges for from=A, to=BO(E)
noCycles(members)DFS with back-edge detection / Tarjan's SCCO(V+E)
noTransitiveDependency([A, B])BFS/DFS reachability from A to BO(V+E)
noSiblingDependencyCheck if any cross-member edges existO(E)

Implementation Options

Design Decisions

OPTION A Post-AG analysis phase in program.ts

After evaluate() returns with all AG results, run a separate relational analysis function. Completely outside the AG system.

// src/program.ts
export function createProgramFromTSProgram(tsProgram, config) {
  const ksTree = buildKSTree(tsProgram, depth);
  const { definitions, diagnostics } = evaluate(ksTree.root);

  // NEW: relational checking as a post-AG step
  const relationalDiags = checkRelationalProperties(config, ksTree, tsProgram);

  return {
    getDiagnostics: () => [...diagnostics, ...relationalDiags],
    // ...
  };
}

Advantages

  • Zero changes to AG evaluator or codegen
  • Clean separation: tree analysis vs graph analysis
  • Can use resolvedModulePath directly from KSC AST nodes
  • Testable in isolation
  • Easy to add more graph algorithms later

Disadvantages

  • Two diagnostic sources to merge
  • Relational diagnostics don't participate in the dep graph
  • Config must be threaded through (currently only used for analysisDepth)

Advantages

  • Unified diagnostic pipeline
  • Shows up in the attribute dependency graph
  • Participates in evaluation ordering

Disadvantages

  • Config access problem: AG equations don't have access to KindScriptConfig. The Ctx interface only exposes tree navigation.
  • Stuffing graph algorithms into a syn equation is unnatural
  • Couples the evaluator to config format
  • The equation would need to do deep DFS over children — violating AG's "local computation" principle
Fundamental problem: AG equations receive (ctx: Ctx, raw: KSNode). There's no way to pass config data (member definitions, path mappings) into the equation without changing the Ctx interface, which would break the clean AG abstraction.

OPTION C Completely separate analysis pass

Build a dedicated RelationalChecker module that operates independently from the AG evaluator. It takes the TS program + config and produces its own diagnostics.

// src/relational-checker.ts

  // 5. Check noDependency, noCycles, etc.
  // 6. Return diagnostics
}

Advantages

  • Maximum isolation — doesn't touch AG at all
  • Can use ts.resolveModuleName() directly for import resolution
  • No dependency on KSC AST or stamped fields
  • Could be shipped as a standalone tool

Disadvantages

  • Duplicates module resolution work (TS does it, KSC stamps it, this does it again)
  • Doesn't use the resolvedModulePath field we already stamp
  • Two completely separate codepaths with different diagnostic formats
  • No shared caching

OPTION D Hybrid: AG stamps data, post-pass analyzes graph

Use the AG evaluation to collect per-file import data into a collection attribute, then run graph analysis on that collected data in a post-pass.

// AG phase: collect import edges per CompilationUnit
/** @kind collection */
export function eq_importEdges_contribute(ctx: Ctx): ImportEdge[] {
  if (ctx.node.kind !== 'ImportDeclaration') return [];
  const imp = ctx.node as KSImportDeclaration;
  if (!imp.resolvedModulePath) return [];
  return [{ sourceFile: findFileName(ctx), targetFile: imp.resolvedModulePath }];
}

// Post-AG: use collected edges + config for graph analysis
const { importEdges } = evaluate(ksTree.root);
const graph = buildMemberGraph(config, importEdges);
const relDiags = checkConstraints(config, graph);

Advantages

  • AG does what it's good at: per-node data extraction + collection
  • Graph analysis gets clean, pre-extracted data
  • Import edges appear in the dep graph and are cached
  • Uses stamped fields naturally

Disadvantages

  • Adds a collection attribute just to pass data to a post-pass
  • Still needs post-pass for the graph algorithms
  • Config access still needed in the post-pass
  • Slightly more complex pipeline

Recommended for v1: Option A

Post-AG phase in program.ts. Simplest, no AG changes, uses existing stamped fields, clean separation of concerns. Get it working, then optimize.

Long-term consideration: Option D

If we find that walking the KSC AST for import edges is slow or we want import data visible in the dep graph, migrate to the hybrid approach. The post-pass graph analysis stays the same; only the data source changes.

Config Changes Required

Design

The current KindScriptConfig only supports include, exclude, strict, and analysisDepth. It has no concept of composites, members, or relational rules.

Current config

interface KindScriptConfig {
  readonly include?: string[];
  readonly exclude?: string[];
  readonly strict?: boolean;
  readonly analysisDepth?: 'parse' | 'bind' | 'check';
}

What it would need to add

interface KindScriptConfig {
  // ... existing fields ...

  /** Named composites with members and relational rules. */
  readonly composites?: Record<string, CompositeEntry>;
}

interface CompositeEntry {
  readonly members: Record<string, { path: string }>;
  readonly rules?: {
    readonly noDependency?: [string, string][];
    readonly noCycles?: string[];
    readonly noTransitiveDependency?: [string, string][];
    readonly noSiblingDependency?: boolean;
  };
}

This is a straightforward extension. The composites field is optional, so existing configs continue to work unchanged.

Prior Art and Analogies

Context

The intrinsic-vs-relational split isn't unique to KindScript. Many tools face this exact architectural divide:

ESLint (intrinsic) vs eslint-plugin-import (relational)

ESLint rules operate on single-file ASTs. But import/no-cycle and import/no-restricted-paths need cross-file module resolution — they use a completely separate dependency graph built from the TS/JS module resolver.

Same pattern: AST rules + separate graph pass.

TypeScript (type checking) vs project references (relational)

TS type checking is per-file (with imports resolved). But project references (composite: true, references: [...]) enforce build-order dependencies at the project level — a separate graph-based system.

Same pattern: per-file checker + project-level graph.

Dependency Cruiser

A dedicated tool that only does relational analysis: builds the full module dependency graph, then checks rules like "no cycles", "not reachable from X", "only from folder Y". No AST analysis at all.

Validates: relational analysis is a distinct concern.

Nx / Turborepo (workspace boundaries)

Workspace-level build tools enforce dependency constraints between packages. The @nx/enforce-module-boundaries rule checks that imports between Nx projects follow declared dependency edges. Again — graph-based, not AST-based.

Validates: module boundaries = graph analysis.

Pattern: Every tool that checks both "what's in the code" and "how files relate to each other" ends up with two separate analysis systems. The AST is the wrong data structure for relationship queries, and the dependency graph is the wrong data structure for code pattern matching.

Review: Test Your Understanding

Quiz
Why can't noCycles be an AG attribute?
Click to reveal
AG attributes flow vertically through the tree (parent-child). Cycle detection requires building a graph from cross-file import edges and running DFS — a horizontal, global analysis that AG's tree-based evaluation model cannot express.
What stamped field on ImportDeclaration gives us the absolute path an import resolves to?
Click to reveal
resolvedModulePath — stamped during conversion by calling checker.getSymbolAtLocation() on the module specifier. Only populated when analysisDepth === 'check'.
What's the difference between noDependency and noTransitiveDependency?
Click to reveal
noDependency([A, B]) checks that A has no direct import edge to B. noTransitiveDependency([A, B]) checks that there is no path from A to B through any intermediate members. The first is an edge filter; the second is a reachability query.
Why is Option B (AG root attribute) problematic?
Click to reveal
Two reasons: (1) AG equations receive (ctx: Ctx, raw: KSNode) — there's no way to pass config data (member definitions, path mappings) into the equation without breaking the Ctx abstraction. (2) Graph algorithms like DFS cycle detection require global mutable state, which violates AG's "local computation" principle.
What existing tool is analogous to what the KSC relational checker would do?
Click to reveal
Dependency Cruiser, eslint-plugin-import (import/no-cycle, import/no-restricted-paths), or Nx's @nx/enforce-module-boundaries. All build a module dependency graph and check relational constraints on it — separate from AST-level analysis.
In the recommended Option A approach, where does the relational checker run?
Click to reveal
In src/program.ts, after evaluate() returns the AG results. It takes the KindScriptConfig (for composite member definitions), the KSTree (for stamped resolvedModulePath fields), and produces additional CheckerDiagnostic[] that are merged with the AG diagnostics.

Summary

Takeaways
KSC Pipeline with Relational Properties TS Source → buildKSTree → evaluate() → checkRelational() → unified diagnostics | | | stamps fields AG attributes graph algorithms (resolvedModule (22 intrinsic (noDependency, Path, sym*, ...) attributes) noCycles, ...) | | | per-node data tree analysis graph analysis

What we have

  • 8 intrinsic property checks via AG
  • Stamped fields for import resolution
  • resolvedModulePath on every ImportDeclaration

What we need

  • Config types for composites
  • Member-to-file resolution
  • Graph builder + constraint checker

What we can't do in AG

  • Cross-file import graph queries
  • Cycle detection algorithms
  • Config-driven member mapping

Decision Framework

Intrinsic properties = AG attributes (per-node, tree-based, already working).

Relational properties = post-AG graph analysis (cross-file, config-driven, new module needed).

These are separate concerns that happen to produce the same output type (CheckerDiagnostic[]). The unification point is program.ts, which merges both diagnostic sources.
