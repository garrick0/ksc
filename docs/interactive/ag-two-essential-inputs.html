The Two Essential Inputs to an Attribute Grammar

Domain + Rules. Everything else is either inside them, derived from them, or about composing multiple AGs rather than defining one.

1

Domain

Attribute declarations — the what. Names and types of values computed at each node.

2

Semantic Rules

Attribute equations — the how. Local functions computing each attribute from its neighborhood.

KSC AGSpecInput + Grammar: The Implemented Design

Implemented

After implementing Strategies 1, 3, 5, and 7 from the library-vs-generator prior art, the spec is now AGSpecInput<N, R> — four fields, with tree structure owned by a shared Grammar object:

// The spec: pure domain + rules + minimal infrastructure
interface AGSpecInput<N extends object, R = unknown> {
  name:       string;                // For dependency resolution
  attributes: AttributeMap<N>;      // THE AG (domain + rules — currently fused)
  deps?:      string[];              // Cross-AG dependencies
  project?:   (root: N) => R;      // Result extraction (convenience)
}

// The grammar: shared tree substrate (Strategy 7)
interface Grammar<N extends object> {
  readonly getChildren: (node: N) => N[];
  evaluate<R>(spec: AGSpecInput, root: N): R | undefined;
  evaluateAll(specs: AGSpecInput[], root: N): Map<string, unknown>;
}
Four fields — how many are essential?

The children accessor moved to the Grammar object (Strategy 7). The domain sub-object was flattened to a simple project function. Let's categorize what remains:

Field Category Why
attributes Essential #2 The semantic rules. Currently fuses domain and rules inside AttributeDef closures — planned to be separated into explicit declarations + equations.
name Infrastructure For multi-AG orchestration (dep resolution). Not about any single AG's definition.
deps Composition About composing multiple AGs. Not a property of any single AG's definition.
project Convenience Result extraction. Could be a standalone function — the AG doesn't need it to be an AG.

And in the Grammar object (shared, not per-spec):

Field Category Why
getChildren Shared substrate The tree structure. Shared across all AGs over the same AST — the .ast file equivalent. Now lives on Grammar, not in each spec.

The binder spec: 3 attributes + minimal infrastructure. No children or domain — the Grammar owns tree structure:

return {
  name: 'ksc-binder',                          // infrastructure
  attributes: { kindDefs, defEnv, defLookup },   // THE AG
  project: (root) => [...allDefs],             // convenience
};

The actual AG content is just the attributes object. name is for orchestration; project is for result extraction. The tree structure lives on createGrammar(getChildren).

The checker: 9 attributes, with deps for cross-AG composition:

return {
  name: 'ksc-checker',                          // infrastructure
  attributes: {                                    // THE AG
    valueImports, fileImports, localBindings, enclosingLocals,
    isReference, kindAnnotations, noImportsContext,
    importViolation, allViolations
  },
  deps: ['ksc-binder'],                          // composition
  project: (root) => root.allViolations,       // convenience
};

The Grammar object evaluates specs. It owns the tree and orchestrates:

// Grammar owns tree structure (Strategy 7)
const grammar = createGrammar<KSNode>(getChildren);

// Single spec evaluation:
grammar.evaluate(binderSpec, root);

// Multi-spec with validation + topo sort (Strategy 1):
const results = grammar.evaluateAll(
  [checkerSpec, binderSpec], // any order — auto-sorted
  root,
);
// Validates: no duplicate attrs, no circular deps, no unknown deps

evaluateAll stamps the tree once, validates all specs, topologically sorts them, then applies attributes in order. The input is just spec.attributes per spec.

The core claim — and its implication

An attribute grammar is fully defined by what gets computed (domain) and how each node computes it (rules). Everything else is either inside these two, derivable from them, or about composing/orchestrating multiple AGs. The current AGSpecInput acknowledges this by having a single attributes field — but it fuses domain and rules inside AttributeDef closures. The next step is to separate them: explicit declarations (domain) + equations (rules), matching JastAdd's architecture and enabling collection attributes, extensibility, and completeness checking.

What's "inside" the two inputs

Several concepts that feel like separate architectural concerns are actually embedded in the domain and rules:

Initial State = root's equation

There's no separate "init" concept. The root's transfer function is the initial inherited context.

Feels like a separate concept

// "Initial context" for the binder
const initialEnv = new Map();
// somehow passed to root...

Actually just the root's equation

const defEnv = inh<KSNode, Map>(
  // rootValue IS the initial state
  (root) => {
    const map = new Map();
    for (const cu of root.$children)
      for (const def of cu.kindDefs)
        map.set(def.name, def);
    return map;
  },
);

The first argument to inh() produces the value at the root. That's the "initial state." It's not separate — it's what the root node's equation computes.

Direction = emergent from equations

You don't declare "this is synthesized" or "this is inherited." The equation is the direction.

Synthesized: equation reads children

// kindDefs reads child statements
const kindDefs = match('kind', {
  CompilationUnit: (cu) => {
    for (const stmt of cu.children)
      // reads DOWN the tree
  }
});

Inherited: equation reads parent

// fileImports reads parent's value
const fileImports = inh(
  new Set(),
  (parent) => {
    // reads UP the tree
    return parent.valueImports;
  }
);

The factory name (syn vs inh) is an implementation hint, not a declaration. The equation's data flow determines the direction.

Scope = inherited attribute + domain component

"The scope chain" sounds architectural, but it's just an inherited attribute whose value type is a map/set.

// Scope is just an inherited attr of type Map
const defEnv = inh<KSNode, Map<string, KindDefinition>>(
  (root) => buildGlobalMap(root),  // root computes initial scope
  // auto-propagate: all children see same scope
);

// A richer scope system would have Block nodes extending it:
// eq: (parent, child) => parent.kind === 'Block'
//   ? new Map([...parent.defEnv, ...parent.localDefs])
//   : undefined  // auto-propagate

The scope chain is what certain transfer functions do with a certain part of the domain. It's not a separate concept.

Join/Lattice = what branch equations do

Different analyses handle control-flow merges differently. That's just what the equation says at branch points:

AnalysisIfStatement equationLattice operation
KSC Binder Identity (passthrough) None needed — defs don't branch
Type checker union(then.type, else.type) Join (widening)
CFA intersect(then.facts, else.facts) Meet (narrowing)
Definite assignment both ? assigned : maybe Must-analysis meet

There's no separate "lattice" concept. The lattice operation is encoded in the equation for that specific production.

Dependencies = multi-AG composition (now auto-discovered)

The checker needs the binder's attributes. This is about composing two AGs, not defining one. The deps field declares spec-level ordering, while analyzeDeps() (Strategy 5) discovers attribute-level dependencies automatically:

// Spec-level: declared (for topo sort of evaluateAll)
deps: ['ksc-binder']

// Attribute-level: discovered at runtime (Strategy 5)
const result = analyzeDeps(getChildren, [binder, checker], root);
// result.deps: importViolation → {noImportsContext, isReference, ...}
// result.order: [kindDefs, defEnv, ..., allViolations]
// result.cycles: [] (none in KSC)

The grammar = shared substrate (now a first-class object)

The tree structure is shared across all AGs. After Strategy 7, it's a first-class Grammar object:

// Grammar owns the tree structure — equivalent to JastAdd's .ast files
const grammar = createGrammar<KSNode>(getChildren);

// Specs are pure { attributes, deps? } — no children accessor
grammar.evaluateAll([binderSpec, checkerSpec], root);

In JastAdd, the grammar is in .ast files, completely separate from .jrag specs. KSC now mirrors this: createGrammar(getChildren) is the .ast equivalent. Each spec adds attributes to the grammar's nodes — the grammar itself is shared.

The demand-driven insight

After evaluate(), the tree just sits there with lazy getters. Nothing has been computed. JastAdd: The Canonical Two-Input System

JastAdd

JastAdd is the reference implementation that most cleanly separates the two inputs. Its file structure is the separation:

.ast files — The Grammar (shared substrate)

// Grammar = tree structure
// NOT part of any AG spec
Program ::= CompUnit*;
CompUnit ::= Stmt*;
VarDecl: Stmt ::= Name:IdDecl
                   Init:Expr;
IdUse: Expr ::= <ID:String>;

.jrag files — Domain + Rules (the AG)

// Domain: attribute declarations
syn Set<Def> CompUnit.kindDefs();
inh Map<String,Def> ASTNode.defEnv();

// Rules: equations per production
eq CompUnit.kindDefs() {
  return getStmts().stream()
    .filter(s -> s instanceof TypeAlias)
    .map(s -> extractDef(s))
    .collect(toSet());
}

eq Program.getCompUnit().defEnv() {
  return allKindDefs();
}
JastAdd's radical simplicity

A .jrag file contains only two things: attribute declarations (domain) and equations (rules). No name, no children accessor, no deps, no project function. The grammar is separate. The evaluator is generated. Composition is automatic via the JastAdd compiler reading multiple .jrag files.

JastAdd's handling of "everything else"

Concept JastAdd's approach Mapping to two inputs

Scoping inh Env Block.env() + equations An inherited attribute of type Env
Circular syn Set<X> Node.reaching() circular [emptySet()] Domain (with bottom) + rule (equation)
Caching Default: cached. Opt out: uncache Evaluator concern, not spec

JastAdd's inherited attribute protocol

JastAdd's approach to inherited attributes is notably different from ours. The equation is written on the parent, specifying what value it provides to a named child accessor:

// JastAdd: parent defines what child receives
inh Env Stmt.env();                     // declaration

eq Block.getStmt(int i).env() {         // equation on parent
  if (i == 0) return env();              // first child: inherit from block
  return getStmt(i-1).envAfter();         // others: chained from prev sibling
}

// KSC: child looks up via parent reference
const fileImports = inh(
  new Set(),
  (parent, child, idx) => {            // dispatched at child
    if (parent.kind === 'CompilationUnit')
      return parent.valueImports;
    return undefined;                     // auto-propagate
  }
);

Both approaches encode the same information. JastAdd's is "parent pushes to named child." KSC's is "child queries parent via $parent." The equation is the same — just written from different perspectives.

Silver (MELT): Forwarding and Higher-Order

Silver

Silver (from the MELT group at UMN) extends the AG formalism with forwarding, higher-order attributes, and collection attributes. Even with these extensions, the core remains: domain + rules.

Silver's two inputs

-- Nonterminal (grammar = shared substrate)
nonterminal Expr;

-- Domain: attribute declarations
synthesized attribute value :: Integer;
inherited attribute env :: Env;

-- Rules: equations per production
aspect production addExpr
top::Expr ::= l::Expr r::Expr
{
  top.value = l.value + r.value;
  l.env = top.env;
  r.env = top.env;
}

Note the symmetry with JastAdd: declarations (domain) + per-production equations (rules). Silver makes inherited attribute flow explicit: l.env = top.env rather than auto-propagation.

Forwarding: extensibility via the two inputs

Silver's unique contribution. A production can "forward to" another production, inheriting its attributes by default:

abstract production forEach
top::Stmt ::= v::Name lst::Expr body::Stmt
{
  -- "forEach" doesn't exist in the base language
  -- Forward to a while loop desugaring
  forwards to whileLoop(iterCond, seqStmt(body, iterStep));

  -- Only define attributes that DIFFER from the forwarded-to tree
  top.pp = "for (" ++ v.pp ++ " in " ++ lst.pp ++ ") " ++ body.pp;
}

Forwarding is still just rules. The forwarding equation says "for attributes I don't define, delegate to this other tree." It's a default equation strategy, not a new concept.

Collection attributes

Silver's collection attributes aggregate contributions from anywhere in the tree:

-- Domain: collection attribute declaration
collection attribute errors :: [Error]
  with ++;   -- combine via list append

-- Rules: contributions from any production
aspect production varRef
top::Expr ::= n::Name
{
  top.errors <-   -- contribute to (not assign)
    if lookup(n, top.env) == nothing()
    then [err("undefined: " ++ n)]
    else [];
}

This is domain (a collection-typed attribute with a combining operator) plus rules (contribution equations). The "collection" is a kind of domain component with a specific combiner — still inside the two inputs.

Propagate: auto-copy equations

Silver has an explicit propagate mechanism for inherited attributes that should just pass through:

-- Instead of writing env equations on every production:
propagate env on Expr, Stmt, Decl;

-- Equivalent to writing on every production of those types:
-- child.env = top.env;

Compare with KSC's inh() returning undefined for auto-propagation:

// KSC equivalent: return undefined = auto-propagate
const defEnv = inh(rootValue, (parent) => {
  return undefined; // use parent's own value
});

Silver's propagate is syntactic sugar for "write identity equations on these productions." It's a shorthand for rules, not a new concept.

Key structural difference — and how KSC bridges it

JastAdd and Silver are code generators — they read .jrag/.sv files and produce Java/C code with the caching machinery baked in. After implementing Strategies 1, 3, 5, and 7, KSC now matches the structural separation of generators: tree structure lives in a shared Grammar object (like .ast files), specs are pure { attributes, deps? } (like .jrag files), and evaluateAll() validates + sorts (like the JastAdd compiler). The Implemented Design: AGSpecInput + Grammar

Implemented

The "two essential inputs" thesis was used to guide the redesign. Here's what was implemented vs. the theoretical minimum:

Theoretical minimum: just attributes

// The AG is JUST the attributes object.
// Everything else is provided externally.

const binderAttrs: AttributeMap<KSNode> = {
  kindDefs, defEnv, defLookup
};

const checkerAttrs: AttributeMap<KSNode> = {
  valueImports, fileImports, localBindings, enclosingLocals,
  isReference, kindAnnotations, noImportsContext,
  importViolation, allViolations
};

// Usage: framework handles everything else
const framework = createAGFramework(getChildren);
framework.apply(root, binderAttrs);
framework.apply(root, checkerAttrs);
const violations = root.allViolations;
The domain is implicit — and that's the problem

In this view, the domain declaration is inside the attributes — hidden in type signatures and closure bodies. You can't inspect, extend, or validate without executing. The planned AGSpec2 makes domain declarations explicit: declarations are inspectable data objects, while equations are the extensible compute functions.

Implemented: AGSpecInput + Grammar

// AGSpecInput: two inputs + minimal infrastructure
interface AGSpecInput<N extends object, R = unknown> {
  name: string;                  // for dep resolution
  attributes: AttributeMap<N>;    // THE AG (domain + rules fused — to be separated)
  deps?: string[];                // multi-AG ordering
  project?: (root: N) => R;      // result extraction
}

// Grammar owns tree structure (Strategy 7):
const grammar = createGrammar<KSNode>(getChildren);

// evaluateAll validates + topo-sorts (Strategy 1):
const results = grammar.evaluateAll(
  [checkerSpec, binderSpec], root
);

// project() extracts results:
const violations = results.get('ksc-checker');

// analyzeDeps discovers attribute-level deps (Strategy 5):
const analysis = analyzeDeps(getChildren, specs, root);
// → { deps, order, cycles }

This is the current KSC implementation. 100 tests passing.

Old design (AGSpec): five fields with mixed concerns

// HISTORICAL — the old API before Strategy 7
return {
  name: 'ksc-checker',              // orchestration
  domain: {                           // metadata + convenience
    name: 'Kind Violations',
    project: (root) => root.allViolations,
  },
  children: getChildren,              // shared substrate (now on Grammar)
  attributes: { ... },                // THE ACTUAL AG
  deps: ['ksc-binder'],              // orchestration
};

This old API is still supported via grammar.toAGSpec() for backward compatibility.

Resolution: pragmatic first, then principled

The first redesign moved children to a shared Grammar object and eliminated the domain nesting. But it left domain and rules fused inside AttributeDef closures. The next step (see Implementation Plan below) separates them into explicit declarations + equations, matching JastAdd's architecture while enabling collection attributes, extensibility, and completeness checking that the fused design cannot support.

The Fusion Spectrum: Where Domain and Rules Are (and Aren't) Separated

Analysis

The current AttributeDef appears to fuse domain and rules into one object. But looking at the actual factories, the fusion is a spectrum, not a binary choice — and the factories already sit at different points on it:

syn: maximally fused

Direction, caching, and equation in a single call. Domain is entirely implicit in the type signature.

// Everything fused: direction + caching + equation
const enclosingLocals = syn<KSNode, Set<string>>((node) => {
  const locals = new Set<string>();
  let current = node.$parent;
  while (current) { /* walk up, accumulate */ }
  return locals;
});

There's no way to inspect what this attribute is without executing it. No way to extend it from another module. No way to check if all productions are covered.

inh: partially separated

rootValue is a domain concern (initial state / lattice top). eq is a rule (propagation equation). They're already separate parameters.

const defEnv = inh<KSNode, Map<string, KindDef>>(
  (root) => buildMap(root),  // ← DOMAIN: initial state
  (parent) => { ... }        // ← RULE: propagation equation
);

The first argument declares the lattice structure (what the root starts with). The second provides the transfer function. These serve different purposes — inh already partially un-fuses them.

circular: three distinct concerns

circular separates three things that the two-inputs thesis says should be separate:

const reaching = circular<N, Set<string>>(
  new Set(),             // ← DOMAIN: bottom value (lattice structure)
  (node) => compute(node),  // ← RULE: equation
  { equals: setEquals }    // ← DOMAIN: convergence criterion
);

The bottom value and equality function are domain (they define the lattice). The compute function is the rule. circular already separates these — it has to, because you can't infer a lattice structure from an equation alone.

match: structural separation, but sealed

match separates the dispatch mechanism from per-production equations. But the equations are sealed inside the object literal — you can't extend them from another module.

const kindDefs = match<KSNode, KindDefinition[]>('kind', {
  CompilationUnit: (cu) => { ... },  // ← per-production RULE
  _: () => [],                        // ← default RULE
});
// Can't add: kindDefs.addEquation('NewProd', (n) => ...)
// Must replace the entire match() call to change equations

In JastAdd, eq Mul.eval() = ... in File C extends the attribute without touching File B. In KSC, the match() object is a sealed, self-contained unit.

Where separation matters

1. Collection attributes (Silver-style)

-- Silver: domain declared ONCE
collection attribute errors :: [Error]
  with ++;  -- combiner

-- Rules come from ANYWHERE:
aspect varRef  { top.errors <- [err(...)] }
aspect typeErr { top.errors <- [typeErr(...)] }

KSC can't do this. allViolations manually does a DFS in syn() to aggregate. A real collection attribute needs the domain (combiner, initial value) separate from distributed contribution rules.

2. Extensibility / aspect weaving

// JastAdd: File A — declaration
syn int Expr.eval();

// File B — base equations
eq Add.eval() = l.eval() + r.eval();

// File C (extension) — new production
eq Mul.eval() = l.eval() * r.eval();

File C extends without touching File B. In KSC, match('kind', {...}) is sealed — you can't add a production equation from another module without replacing the entire match.

3. Equation completeness checking

When domain and rules are separate, you can check: "does every production have an equation for this attribute?" JastAdd does this statically.

With fusion, _ in match() is an escape hatch — "I don't care about most productions." MatchEquations validates keys you write, but can't detect keys you forgot.

4. Introspection / tooling

With separated declarations, tools can enumerate "what attributes does a CompilationUnit carry?" without executing anything.

With fusion, you need analyzeDeps() (runtime execution) to discover attribute structure. Domain metadata is lost inside closure bodies.

The honest framing: fusion is a trade-off, not a principle

The earlier claim that "the compute function implicitly declares the domain" is only true for syn. Even within the current codebase, circular and inh already partially un-fuse things. The fusion trades extensibility, introspectability, and composability for simplicity. It works for KSC specifically (fixed grammar, 12 attributes, single author) but breaks down for collection attributes, extensible tooling, static completeness guarantees, and third-party aspect weaving. The solution: separate the two inputs at the API level.

Bridging the Library-Generator Gap: Four Implemented Strategies

Implemented

Four strategies from the library-vs-generator prior art have been implemented, closing ~80% of the gap with JastAdd:

Strategy 7: Shared Grammar Object

Extracted children from specs into a shared Grammar object. This is the .ast file equivalent.

// Before: children repeated in every spec
const binder = { children: getChildren, attributes: {...} };
const checker = { children: getChildren, attributes: {...} };

// After: Grammar owns tree structure, specs are pure
const grammar = createGrammar<KSNode>(getChildren);
const binder: AGSpecInput = { name: 'ksc-binder', attributes: {...} };
const checker: AGSpecInput = { name: 'ksc-checker', attributes: {...} };

Strategy 1: Registration + Validation

grammar.evaluateAll() validates all specs before first evaluation — the library's "compile step":

// evaluateAll validates before applying:
grammar.evaluateAll([binder, checker], root);
// Throws if:
//   - Duplicate attribute 'x' in specs 'a' and 'b'
//   - Circular dependency: a → b → a
//   - Unknown dependency: 'nonexistent'

// Also topo-sorts: pass [checker, binder] and
// binder still evaluates first (based on deps)

Like Django's metaclass validation or attrs' @define, the validation runs once before any real work happens.

Strategy 3: Type-Level Equation Validation

Enhanced match() with MatchEquations<N, V> — a mapped type that validates equation keys against the node union's kind values at compile time. Zero runtime cost.

// KindValues extracts all 'kind' literals from a discriminated union
type KindValues<N> = N extends { kind: infer K extends string } ? K : never;

// MatchEquations validates keys and narrows callback types
type MatchEquations<N, V> =
  & { [K in KindValues<N>]?: (node: Extract<N, { kind: K }>) => V }
  & { _?: (node: N) => V };

// Typos caught at compile time:
const bad = match<KSNode, boolean>('kind', {
  ComipilationUnit: ... // TS error: typo!
});

This is the TS equivalent of JastAdd's static equation validation. The TypeScript compiler is the "JastAdd compiler."

Strategy 5: Runtime Dependency Analysis

analyzeDeps() discovers cross-attribute dependencies at runtime, producing the same dependency graph that JastAdd computes statically:

const result = analyzeDeps(getChildren, [binder, checker], root);

result.deps;    // Map: importViolation → {noImportsContext, isReference, ...}
result.order;   // [kindDefs, valueImports, ..., allViolations]
result.cycles;  // [] (empty = no circular dependencies)
How it works: prototype-chained clones

For each target attribute, analyzeDeps creates ephemeral clones (Object.create(node)) with tracking getters on non-target attributes. It installs the real AttributeDef for the target, triggers it, and records which attributes were accessed. The prototype chain preserves kind, $children, $parent reads, while all Object.defineProperty calls land on the clone. No cleanup needed — clones are garbage-collected.

This approach was necessary because installLazy creates configurable: false data properties on first access, making Proxy-based interception and save/restore approaches fail. Clones side-step the problem entirely.

Result: library with generator-class capabilities

With these four strategies, KSC's runtime AG library now provides: grammar separation (Strategy 7), validation at registration time (Strategy 1), compile-time equation validation (Strategy 3), and runtime dependency analysis (Strategy 5). This matches or exceeds SQLAlchemy's strategy count (3) and puts KSC among the most sophisticated library-based AG systems. The remaining gap (optimized caching via optional compiler — Strategy 6) is a stretch goal.

Implementation Plan: Separating Domain from Rules

Planned

Shift from a fused AttributeDef to a true two-input spec where domain declarations and semantic rules are separate first-class objects.

The two-input AGSpec

// ── Input 1: Domain declarations ──
// What gets computed. Inspectable, extensible, no closures.

interface SynDecl<V> {
  direction: 'syn';
  cached?: boolean;            // default true
}

interface InhDecl<V> {
  direction: 'inh';
  initial: V | ((root: any) => V);  // lattice top / root value
}

interface CircularDecl<V> {
  direction: 'circular';
  bottom: V;                  // lattice bottom
  equals?: (a: V, b: V) => boolean;
}

interface CollectionDecl<V> {
  direction: 'collection';
  initial: V;                 // identity element
  combine: (a: V, b: V) => V;  // monoid operation
}

type AttrDecl = SynDecl | InhDecl | CircularDecl | CollectionDecl;

// ── Input 2: Semantic rules ──
// How each production computes it. Extensible, composable.

type Equation<N, V> = (node: N) => V;
type InhEquation<N, V> = (parent: N, child: N, idx: number) => V | undefined;

// Per-production dispatch (open, unlike match())
type ProductionRules<N, V> = {
  [production: string]: Equation<N, V>;
} & { _?: Equation<N, V> };

// ── The two-input spec ──
interface AGSpec2<N extends object, R = unknown> {
  name: string;
  declarations: Record<string, AttrDecl>;     // Input 1: domain
  equations: Record<string,                  // Input 2: rules
    Equation | InhEquation | ProductionRules>;
  deps?: string[];
  project?: (root: N) => R;
}

Current: fused AttributeDef

Domain and rules are entangled inside factory closures:

// The binder: domain is implicit, rules are sealed
const kindDefs = match<KSNode, KindDefinition[]>('kind', {
  CompilationUnit: (cu) => { /* extract defs */ },
  _: () => [],
});

const defEnv = inh<KSNode, Map>(
  (root) => buildMap(root),
);

const defLookup = syn<KSNode, Function>(
  (node) => (name) => node.defEnv.get(name),
);

return {
  name: 'ksc-binder',
  attributes: { kindDefs, defEnv, defLookup }, // fused blob
};

Problems: Can't inspect declarations without running code. Can't extend equations. Can't check completeness. Can't add collection attributes.

Proposed: separated domain + rules

const binderSpec: AGSpec2<KSNode> = {
  name: 'ksc-binder',

  // Input 1: DOMAIN — what gets computed
  declarations: {
    kindDefs:  { direction: 'syn' },
    defEnv:    { direction: 'inh', initial: (root) => buildMap(root) },
    defLookup: { direction: 'syn' },
  },

  // Input 2: RULES — how each production computes it
  equations: {
    kindDefs: {
      CompilationUnit: (cu) => { /* extract defs */ },
      _: () => [],
    },
    defEnv: undefined,  // auto-propagate (no eq needed)
    defLookup: (node) => (name) => node.defEnv.get(name),
  },
};

// Extensions from another module — impossible with fused design:
binderSpec.equations.kindDefs.ModuleDeclaration = (mod) => {
  /* extract defs from module syntax */
};

Declarations are inspectable data. Equations are extensible. Collection attributes become possible. Completeness is checkable.

Implementation phases

Phase What Details Unlocks
1 AttrDecl types Define SynDecl, InhDecl, CircularDecl interfaces. Add declarations + equations to AGSpec2. Keep old AGSpecInput working via adapter. Static introspection of attribute structure without running code
2 Compile step grammar.compile(spec) reads declarations + equations, validates completeness (every declared attr has equations for required productions), and produces an internal AttributeMap. Replace evaluateAll's ad-hoc validation. Equation completeness checking at "compile time"
3 Open equations Production rules are a Map that can be extended after initial spec creation. spec.addEquation('kindDefs', 'NewProd', fn). Compile step merges all contributions. Extensibility / aspect weaving without replacing entire match()
4 Collection attributes Add CollectionDecl with initial + combine. Contribution rules use contribute() instead of eq(). Compile step wires the combiner. Replace allViolations DFS with a proper collection attribute. Silver-style distributed aggregation from any production
5 Migrate KSC specs Convert createBinderSpec() and createCheckerSpec() to AGSpec2. Remove old fused factories or keep as convenience wrappers that produce { declaration, equation } pairs. Real-world validation of the new API
Migration strategy: adapter, not big bang

Each phase should keep existing tests passing. Phase 1 adds new types alongside AGSpecInput. Phase 2 adds a compile step that can accept either format. Only Phase 5 migrates the actual KSC specs. The old syn()/inh()/match() factories can be kept as convenience wrappers that produce a { decl, eq } pair — no need to delete them.

What de-fusion enables

Collection attributes

// Replace manual DFS allViolations:
declarations: {
  allViolations: {
    direction: 'collection',
    initial: [],
    combine: (a, b) => [...a, ...b],
  },
},
// Contributions from anywhere:
contribute('allViolations', 'Identifier',
  (n) => n.importViolation ? [n.importViolation] : []
);

No more manual tree walks. The framework combines contributions using the declared combiner.

Extensibility

// Base spec:
kindDefs: {
  CompilationUnit: extractFromCU,
  _: () => [],
}

// Plugin adds equation for new syntax:
plugin.addEquation('kindDefs',
  'ModuleDeclaration', extractFromModule
);

Like JastAdd aspects: extend without modifying the original spec file.

Completeness checking

// grammar.compile() can detect:
// "Attribute 'kindDefs' has no equation
//  for production 'ModuleDeclaration'"
// (instead of silently falling to _ default)

Opt-in: mark attributes as exhaustive: true to require equations for all productions.

Static introspection

// No execution needed:
for (const [name, decl] of
  Object.entries(spec.declarations)) {
  console.log(name, decl.direction);
}
// "kindDefs syn"
// "defEnv inh"
// "defLookup syn"

Dashboard/tooling can enumerate the attribute structure without running any equations.

The two-input spec is the JastAdd model in TypeScript

declarations = .jrag attribute declarations (syn Type Node.attr()). equations = .jrag equation blocks (eq Prod.attr() { ... }). Grammar = .ast files. grammar.compile() = the JastAdd compiler. The separation KSC should have had from the start — now achievable without a code generator, using TypeScript's type system for the compile-time checks JastAdd's DSL provides.
