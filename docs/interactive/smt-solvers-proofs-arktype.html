Part 16 — KindScript Research Series

SMT Solvers, Proof Assistants & ArkType

What SMT solvers actually are and how they work, how dependent types and proof assistants are the same thing, and how ArkType brings TypeScript syntax to runtime validation.

Part 1: What Is an SMT Solver?

Short answer: An SMT solver is a program that automatically proves mathematical statements. You give it a formula like "is there an x where x > 5 and x < 3?" and it says "no" (unsatisfiable) or gives you a value. Refinement type systems like LiquidHaskell use this to automatically verify program properties — they extract logical formulas from your code and ask Z3 to prove them.

1.1 From SAT to SMT

SAT Boolean Satisfiability — the foundation

Given a Boolean formula, is there an assignment that makes it true?

// SAT problem:
(A ∨ ¬B) ∧ (¬A ∨ C) ∧ (B ∨ ¬C)

// Question: Can we assign true/false to A, B, C
// so the whole formula is true?

// Answer: A=true, B=true, C=true → YES (satisfiable)

SAT was the first problem proven NP-complete (Cook-Levin, 1971). Despite worst-case exponential time, modern SAT solvers handle formulas with millions of variables in practice. They form the backbone of all SMT solvers.

CDCL Conflict-Driven Clause Learning — modern SAT solving

The breakthrough that made SAT practical. Three key ideas:

// 1. UNIT PROPAGATION
// If a clause has one unassigned literal, force it true.
(A ∨ ¬B) and B=true → must set A=true

// 2. CONFLICT ANALYSIS
// When we hit a dead end, figure out WHY.
// "Setting A=false and C=true always fails"

// 3. CLAUSE LEARNING
// Add a new clause to prevent this conflict pattern.
// (¬A ∨ ¬C) — "never set both A=false and C=true"

// 4. NON-CHRONOLOGICAL BACKJUMP
// Don't undo decisions one by one.
// Jump back to the earliest relevant decision.

CDCL lets the solver learn from mistakes. Each conflict teaches it something about the problem structure, dramatically pruning the search space.

SMT Satisfiability Modulo Theories — adding meaning

SAT only understands true/false. SMT adds theories — mathematical domains with real meaning:

// SAT sees: (P₁ ∧ P₂) ∨ P₃
// (just Boolean variables)

// SMT sees: (x > 5 ∧ y < 10) ∨ (z = x + y)
// (actual arithmetic with meaning!)

// The CDCL solver handles the Boolean structure.
// Theory solvers handle the arithmetic.
// They communicate via DPLL(T) framework:

// CDCL: "I'm trying x > 5 = true, y < 10 = true"
// Arithmetic solver: "That's consistent, carry on"

// CDCL: "Now also x + y = 3"
// Arithmetic solver: "CONFLICT! If x > 5 and x+y = 3
//                    then y < -2, but you said y < 10
//                    — wait, that's fine. But let me
//                    check if other constraints break..."

Step 1: Annotate your code with predicates

-- LiquidHaskell example:
{-@ type Pos = { v:Int | v > 0 } @-}
{-@ type NonZero = { v:Int | v /= 0 } @-}

{-@ safeDiv :: Int -> NonZero -> Int @-}
safeDiv x y = x `div` y

{-@ abs :: Int -> Pos @-}
abs x = if x >= 0 then x else (-x)

You add logical predicates to types using special annotation comments. The base Haskell code compiles normally — these annotations are only for verification.

Step 2: Extract Verification Conditions (VCs)

-- For abs :: Int → Pos, LiquidHaskell generates:

-- Branch 1: x >= 0
-- Return value: x
-- Must prove: x > 0   ← but wait, x could be 0!

-- Actually the VC is:
-- ∀x. (x ≥ 0 → x > 0) ∧ (x < 0 → -x > 0)

-- Hmm, x ≥ 0 → x > 0 fails when x = 0!
-- LiquidHaskell would flag this.

-- Fixed version:
{-@ type Nat = { v:Int | v >= 0 } @-}
{-@ abs :: Int -> Nat @-}
-- VC: ∀x. (x ≥ 0 → x ≥ 0) ∧ (x < 0 → -x ≥ 0)  ✓

The system walks the GHC Core IL (Haskell's intermediate language) and generates a logical formula for each function. Each branch produces a constraint that must hold.

Step 3: Encode as SMT-LIB

; The VC becomes this SMT-LIB query:
(declare-const x Int)

; Check branch 1: if x >= 0, result is x, and x >= 0
(assert (not (=> (>= x 0) (>= x 0))))
(check-sat)
; → unsat (good! the negation is impossible, so the VC holds)

; Check branch 2: if x < 0, result is -x, and -x >= 0
(assert (not (=> (< x 0) (>= (- x) 0))))
(check-sat)
; → unsat (good! this VC holds too)

VCs are negated before sending to Z3. If the negation is unsatisfiable, the original formula is valid (always true). This is the standard technique — you prove P by showing ¬P is impossible.

Step 4: Z3 Solves

// Z3 applies DPLL(T):
// 1. CDCL assigns Boolean structure
// 2. Linear arithmetic solver checks: can x ≥ 0 and x < 0?
//    No — contradiction. Learn clause.
// 3. Try: x < 0 and -x < 0?
//    No — if x < 0 then -x > 0. Contradiction.
// 4. All branches lead to contradiction.
// → UNSAT (the negated VC is impossible)
// → Original VC is VALID ✓

Z3 typically solves these in milliseconds. The decidable fragment (quantifier-free linear arithmetic) guarantees termination — Z3 will always give you an answer.

Step 5: Result

-- If Z3 says UNSAT → your code is VERIFIED ✓
-- If Z3 says SAT   → your code has a BUG ✗
--                     Z3 gives a counterexample:
--                     "x = 0 violates v > 0"

-- LiquidHaskell reports:
Error: Liquid Type Mismatch
  The inferred type: { v:Int | v >= 0 }
  is not a subtype of: { v:Int | v > 0 }
  Counterexample: v = 0

The error traces back to your source code location with a concrete counterexample. You know exactly what input violates the refinement.

1.4 The Refinement Type Tools

LiquidHaskell

-- Refine types with predicates over values
{-@ type Btwn Lo Hi = { v:Int | Lo <= v && v < Hi } @-}

{-@ safeIndex :: ls:[a] -> Btwn 0 (len ls) -> a @-}
safeIndex ls i = ls !! i
-- Z3 verifies: at every call site, 0 ≤ i < length(ls)

-- Measures (functions usable in refinements):
{-@ measure len :: [a] -> Nat @-}
{-@ len [] = 0 @-}
{-@ len (x:xs) = 1 + len xs @-}

Hooks into GHC's compilation pipeline. Predicates restricted to decidable theories. Z3 as backend. Automated — no manual proofs needed for decidable fragments.

F* — SMT-Backed Dependent Types

// F* combines dependent types WITH SMT automation:
val factorial : n:nat -> Tot (x:nat{x >= 1})
let rec factorial n =
  if n = 0 then 1
  else n * factorial (n - 1)

// F* generates: n ≥ 0 ∧ n = 0 → 1 ≥ 1   (trivial)
//              n ≥ 0 ∧ n ≠ 0 ∧ f(n-1) ≥ 1 → n*f(n-1) ≥ 1
// Z3 solves automatically ✓

// When Z3 can't solve, use tactics:
let hard_lemma () : Lemma (forall x. x * x >= 0) =
  Classical.forall_intro (fun x ->
    assert (x * x >= 0) by (smt())  // hint Z3
  )

F* sits between full dependent types (Idris/Lean) and refinement types (LiquidHaskell). It can express richer properties than LiquidHaskell but automates more than Lean/Coq. Used to verify real cryptographic code (Project Everest, HACL*).

Flux — Refinement Types for Rust

// Integrates with Rust's ownership system:
#[flux::sig(fn(v: &mut Vec<i32>, i: {usize | i < len(v)}) )]
fn increment(v: &mut Vec<i32>, i: usize) {
    v[i] += 1;  // proven safe: i < len(v)
}

// Rust's ownership guarantees no aliasing → Flux can
// do STRONG UPDATES: after mutation, the refined type
// tracks the new value precisely.

Key innovation: Rust's borrow checker already prevents aliasing. Flux leverages this to verify mutable code without full separation logic. ~10x faster than alternatives, near-zero annotation overhead.

Stainless — Verification for Scala

// Scala with formal specifications:
def factorial(n: BigInt): BigInt = {
  require(n >= 0)            // precondition
  if (n == 0) BigInt(1)
  else n * factorial(n - 1)
} ensuring(res => res > 0)   // postcondition

// Stainless sends to Z3 and cvc5 in parallel (portfolio mode).
// If EITHER solver proves it, verification succeeds.

Stainless extracts a pure functional subset of Scala and verifies it. Uses portfolio solving — multiple SMT solvers in parallel, first one to finish wins.

The key insight for refinement types: By restricting predicates to decidable logic fragments (quantifier-free linear arithmetic, uninterpreted functions), verification is fully automatic. You write annotations; Z3 does the proving. No manual proofs, no tactics, no proof scripts. The trade-off: you can only express properties that fit in these fragments.

Part 2: Dependent Types & Proof Assistants

They're the same thing. This surprises people because "proof assistant" sounds like a math tool and "dependently typed language" sounds like a programming tool. But the Curry-Howard correspondence tells us: types ARE propositions, programs ARE proofs. A "proof assistant" and a "dependently typed language" are just different marketing for the same underlying mechanism.

2.1 The Curry-Howard Correspondence

This is the fundamental bridge. Click each row to see the connection:

→ Function type A → B = Implication A ⊃ B 
// A function from A to B is a PROOF that A implies B.
// If you can write a total function f : A → B,
// you've proven: "given A, I can construct B."

// In logic: A ⊃ B ("if A then B")
// In types: A → B ("function from A to B")
// SAME THING.

modus_ponens : (A → B) → A → B
modus_ponens f a = f a
-- This program IS the proof of modus ponens!
× Product type (A, B) = Conjunction A ∧ B 
// A pair (A, B) is proof that BOTH A and B hold.
// To construct a pair, you need evidence for both.

and_intro : A → B → (A, B)
and_intro a b = (a, b)
-- "If I have proof of A and proof of B,
//  I have proof of A ∧ B"

and_elim_left : (A, B) → A
and_elim_left (a, _) = a
-- "From proof of A ∧ B, I can extract proof of A"
+ Sum type Either A B = Disjunction A ∨ B 
// Either A B is proof that at LEAST ONE of A, B holds.

or_intro_left : A → Either A B
or_intro_left a = Left a
-- "Proof of A gives proof of A ∨ B"

or_elim : Either A B → (A → C) → (B → C) → C
or_elim (Left a)  f _ = f a
or_elim (Right b) _ g = g b
-- "If A→C and B→C, and I know A∨B, I can get C"
⊥ Empty type Void = Falsehood ⊥ 
// A type with NO constructors represents falsity.
// You can't build a value of type Void,
// just like you can't prove False.

data Void  -- no constructors

absurd : Void → A
absurd v = case v of {}   -- no cases needed!
-- "From falsehood, anything follows" (ex falso quodlibet)

// Negation: ¬A = A → Void
// "A is false" means "assuming A leads to contradiction"
∀ Dependent function (x : A) → B(x) = Universal quantification ∀x:A. B(x) 
// THIS is where dependent types enter.
// A function where the RETURN TYPE depends on the INPUT VALUE.

// In logic: ∀n:Nat. n+0 = n
// "For ALL natural numbers n, n+0 equals n"

// In types:
plusZero : (n : Nat) → n + 0 = n
-- A function that takes a Nat and returns a PROOF
-- that n+0 = n (for that specific n).

// If you can write this total function,
// you've proven the universal statement!
∃ Dependent pair (x : A, B(x)) = Existential quantification ∃x:A. B(x) 
// A pair where the second component's TYPE depends on
// the first component's VALUE.

// In logic: ∃n:Nat. n > 5
// "There EXISTS a natural number greater than 5"

// In types:
example : (n : Nat ** n > 5)
example = (6 ** Refl)
-- The witness is 6, and the proof is that 6 > 5 = Refl

// To prove ∃x.P(x), you give a SPECIFIC x and prove P(x).
The punchline: Type checking IS proof checking. When the compiler accepts your program, it has verified your proofs. When it rejects your program, your proof is wrong. "Proof assistant" = "type checker for dependent types" = "compiler that verifies mathematical theorems."

2.3 How Each System Works

Coq/Rocq — Tactics-Based Proving

(* Two languages: Gallina (terms) and Ltac (tactics) *)

(* Gallina: define data and functions *)
Fixpoint plus (n m : nat) : nat :=
  match n with
  | O => m
  | S k => S (plus k m)
  end.

(* Ltac: prove theorems interactively *)
Theorem plus_0_r : forall n, plus n 0 = n.
Proof.
  intro n.              (* assume n *)
  induction n.          (* split into base + inductive *)
  - reflexivity.        (* base: plus 0 0 = 0 ✓ *)
  - simpl. rewrite IHn. reflexivity.  (* inductive step *)
Qed.

(* Extract to OCaml for execution: *)
Extraction plus.
(* → let rec plus n m = match n with O → m | S k → S (plus k m) *)

You write proofs as scripts — sequences of tactic commands that transform a goal step by step. Under the hood, tactics build Gallina proof terms. The separation means you think procedurally ("first do induction, then simplify") rather than constructively.

Real-world: CompCert (verified C compiler), seL4 helpers, CertiKOS (verified OS kernel).

Lean 4 — Unified Prover + Language

-- Lean 4 compilation: Source → Macros → Elaboration → C

-- Define and prove in the same language:
def append : List α → List α → List α
  | [],    ys => ys
  | x::xs, ys => x :: append xs ys

-- Prove properties with tactics (by keyword):
theorem append_nil : ∀ (xs : List α), append xs [] = xs := by
  intro xs
  induction xs with
  | nil => rfl
  | cons x xs ih => simp [append, ih]

-- Or prove with terms (no tactics):
theorem append_nil' : ∀ (xs : List α), append xs [] = xs
  | [] => rfl
  | x::xs => congrArg (x :: ·) (append_nil' xs)

-- Compiles directly to C (no extraction step)

Lean 4 deliberately includes classical axioms (propositional extensionality, function extensionality, choice). This makes it easier to formalize classical mathematics but breaks some type-theoretic properties. The trade-off: more practical math, slightly less pure theory.

Real-world: mathlib (1.9M lines of formalized mathematics), the largest coherent mathematical library ever created.

Agda — Everything Is a Term

-- No tactics. Proofs are written directly as programs.
-- Interactive refinement in Emacs: type ? for holes.

data _≡_ {A : Set} (x : A) : A → Set where
  refl : x ≡ x

-- Prove: n + 0 ≡ n (by pattern matching and recursion)
+-identity : (n : ℕ) → n + 0 ≡ n
+-identity zero    = refl
+-identity (suc n) = cong suc (+-identity n)

-- The proof IS the program.
-- cong suc wraps the recursive proof in a successor.
-- This is a complete, verified proof of n+0=n.

-- Workflow in Emacs:
-- 1. Write: +-identity (suc n) = ?
-- 2. Agda shows: Goal: suc (n + 0) ≡ suc n
-- 3. You refine: cong suc (?)
-- 4. Agda shows: Goal: n + 0 ≡ n
-- 5. You write: +-identity n

Agda's philosophy: the proof is the program, literally. No separate tactic language. Interactive development via holes (?). The cleanest expression of Curry-Howard, but requires you to think constructively about every proof step.

Key Differences

Tactics vs Terms

Coq/Lean: Proof scripts — "intro n. induction n. simpl."
Procedural thinking: "what steps do I take?"

Agda: Direct construction — pattern match and recurse.
Constructive thinking: "what term has this type?"

Both produce the same thing (a proof term). Tactics are a UI for building terms.

Extraction vs Compilation

Coq: "Extract" proof terms to OCaml/Haskell. Separate compiler runs that code.

Lean 4: Compiles directly to C. No extraction step.

Idris 2: Compiles to Chez Scheme/JS/C. Full erasure of proofs via QTT.

Agda: Primarily an interpreter. Can compile to Haskell.

What They Share

All four are built on the same foundational idea: the Calculus of Inductive Constructions (or close variant). All use dependent types where types can contain values. All exploit Curry-Howard to verify mathematical properties. The differences are in user interface (tactics vs terms), practical features (IO, compilation targets), and design philosophy (math-first vs programming-first).

2.4 Real-World Impact

1 CompCert — Verified C Compiler (Coq)

A C compiler with a machine-checked proof that the generated assembly is semantically equivalent to the source C. Every optimization pass is proven correct. Eliminates entire classes of compiler bugs. Used in aerospace and nuclear industries. ~100,000 lines of Coq proofs.

2 mathlib — Formalized Mathematics (Lean 4)

1.9 million lines of formalized mathematics. Covers algebra, analysis, topology, number theory, category theory. The largest coherent math library ever created. Monolithic design allows large-scale refactoring of fundamental definitions.

3 seL4 — Verified OS Microkernel (Isabelle/HOL)

8,700 lines of C with a complete formal verification proof. Every execution path is guaranteed to match the formal specification. The first general-purpose OS kernel with end-to-end formal verification. Eliminates buffer overflows, null pointer dereferences, etc. by mathematical proof.

4 Project Everest — Verified Cryptography (F*)

Formally verified implementations of TLS 1.3 and cryptographic algorithms. The HACL* library provides verified C code for crypto primitives. Used in Firefox (NSS) and Linux kernel (WireGuard). Proves both functional correctness and side-channel resistance.

Part 3: ArkType

ArkType is a TypeScript runtime validation library that uses TypeScript-like syntax for type definitions. It's not a type system or static analyzer — it's a runtime validator that generates optimized checking functions from string-based type expressions. Think Zod but with native TypeScript syntax and 20-100x better performance.

3.1 How It Works

Core API — TypeScript Syntax as Strings

import { type } from "arktype"

// Looks like TypeScript, but it's a runtime validator:
const user = type({
  name: "string",
  age: "number",
  email: "string.email",        // built-in email validation
  role: "'admin' | 'user'",     // literal union
  scores: "number[]",            // array type
})

// Validate at runtime:
const result = user({ name: "Alice", age: 30, ... })

// TypeScript infers the static type automatically:
// typeof user.infer = {
//   name: string;
//   age: number;
//   email: string;
//   role: "admin" | "user";
//   scores: number[];
// }

One definition serves double duty: runtime validation AND compile-time TypeScript type. No separate schema + type definition.

How It Parses Type Expressions

// ArkType parses string type expressions at RUNTIME:

"string | number"
// → Parser tokenizes: ["string", "|", "number"]
// → Builds AST: Union(StringType, NumberType)

"string.email"
// → Parser tokenizes: ["string", ".", "email"]
// → Builds AST: Constrained(StringType, EmailValidator)

"number > 0"
// → Parser tokenizes: ["number", ">", "0"]
// → Builds AST: Refined(NumberType, GreaterThan(0))

"string.numeric.parse |> number % 2"
// → Pipe: first parse string to number, then check even
// → Builds AST: Pipe(Parse(String→Number), Modulo(2))

// The parser is a shift-reduce parser optimized
// for this specific grammar. NOT using TypeScript's
// template literal types (those would be too slow).

The type strings are parsed by ArkType's own runtime parser, not by TypeScript. TypeScript's type system is used separately to infer static types from the same definitions (via complex generic types and overloads).

Dynamic Code Generation

// ArkType's performance secret: it GENERATES validation code

// When you write:
const check = type({ name: "string", age: "number > 0" })

// ArkType generates something like:
function validate(data) {
  if (typeof data !== "object" || data === null) return err;
  if (typeof data.name !== "string") return err;
  if (typeof data.age !== "number" || data.age <= 0) return err;
  return data;
}

// This is a FLAT, optimized function.
// No recursion. No method dispatch. No schema traversal.
// Just straight-line checks.
// This is why it's 20-100x faster than Zod.

Zod builds a tree of schema objects and traverses them at runtime. ArkType pre-compiles schemas into flat validation functions using new Function(). The validation itself is as fast as hand-written code.

Part 4: The Verification Landscape

4.1 How Everything Connects

All these tools address the same question: "how do I know my program is correct?" They just operate at different levels:

Level Tool What It Proves How Automation
Runtime ArkType, Zod Input matches expected shape Generated validation code Full
Lint eslint-plugin-functional Code follows syntactic patterns AST pattern matching Full
Types TypeScript, Effect-TS Types align across operations Bidirectional type checking Full
Static Analysis KindScript Semantic properties hold (purity, immutability) AST walking + property checks Full
Refinement LiquidHaskell, Flux, F* Logical predicates hold (x > 0, len < n) VC extraction + SMT solver Full
Dependent Idris, Lean, Coq, Agda Arbitrary mathematical theorems Dependent type checking + proofs Partial
The trade-off pattern: As you go down this table, you can prove stronger properties but require more effort. Runtime validation is effortless but only checks shapes. Dependent types can prove anything but may require manual proofs. Refinement types hit a sweet spot: strong properties with full automation, restricted to decidable fragments.

4.2 Where Does KindScript Go From Here?

KindScript currently sits at the "static analysis" level. The research in this series suggests several evolution paths:

Short term: Add transitive property analysis (call graph walking) to catch impurity propagation. This closes the biggest gap vs Effect-TS.

Medium term: Add refinement-type-style predicates for numeric bounds, array lengths, non-null guarantees. Use Z3 as a backend solver.

Long term: Move toward dependent-type-style property proofs for complex invariants. Though full dependent types may be overkill for a TypeScript analysis tool — the refinement type level (SMT-backed) is likely the sweet spot.
