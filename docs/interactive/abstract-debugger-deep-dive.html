1. The Problem: Why Abstract Analysis Needs a Debugger

Traditional abstract interpretation tools operate as black boxes. You feed in source code, wait for the analysis to complete (seconds to hours), then examine a report of warnings. When results are surprising — a false alarm, a missed bug, or unexpected imprecision — the analyst has almost no way to understand why.

Traditional Workflow

1 Write/modify C code
2 Run full analysis (batch)
3 Read warnings report
4 Confused about false alarm?
5 Add pragmas/annotations blindly, re-run entire analysis

Average feedback loop: minutes to hours

Abstract Debugger Workflow

1 Write/modify C code
2 Set breakpoint at suspicious location
3 Step through analysis, inspect abstract states
4 See exactly where precision is lost
5 Fix the root cause, continue from that point

Average feedback loop: seconds

The Core Insight

An abstract debugger doesn't debug the program — it debugs the analysis of the program. Just as GDB lets you step through concrete execution to understand runtime behavior, an abstract debugger lets you step through fixpoint computation to understand why the analyzer reaches certain conclusions.

Who Needs This?

  • Analysis developers — debugging new abstract domains or transfer functions
  • Verification engineers — understanding why an alarm is raised (or missed)
  • Researchers — observing convergence behavior, comparing iteration strategies
  • Tool users — learning what the analyzer actually computes

What Makes It Hard?

  • Non-sequential execution — analysis doesn't follow program order; it follows the iteration strategy
  • Infinite-height lattices — widening jumps create non-monotone state changes
  • Multiple iterations per loop — the same code point is visited many times with different abstract states
  • Branch merging — joins lose information; you need to see which branch caused imprecision

2. What the Frama-C Abstract Debugger Provides

The Abstract Debugger is a feature of Frama-C's EVA (Evolved Value Analysis) plugin. It was developed by Jules Massart during a 6-month internship (March–August 2025) at CEA List, supervised by Michele Alberti, David Bühler, and Virgile Prevosto. It is planned for the next Frama-C release after Germanium (v32.0, December 2025).

Critically, the Eva Abstract Debugger operates on-the-fly during analysis execution, not post-hoc on pre-computed results (like the GobPie debugger for Goblint, from University of Tartu). Users guide the analysis as it runs — choosing branch exploration order and adding ACSL annotations that affect subsequent computation. The concept was formalized by Karoliine Holter et al. in their Onward! 2024 paper "Abstract Debuggers: Exploring Program Behaviors Using Static Analysis Results".

State Inspection

At any point, you can inspect:

  • The abstract state — variable ranges (e.g., x ∈ [0, 42]), pointer targets, memory regions
  • The call stack — which function analysis context you're in
  • The iteration count — which pass through a loop you're on
  • The domain contributions — which abstract domain(s) provide which constraints (EVA uses a reduced product of multiple domains)
  • The pending worklist — which program points still need to be (re-)analyzed

Technical Capabilities

  • Deterministic replay — abstract interpretation is deterministic, so you can replay any analysis session exactly
  • Branch exploration — at an if statement, step into the true branch or the false branch independently before joining
  • Widening observation — see the pre-widening state, the widening operator's output, and the post-widening state side-by-side
  • Narrowing passes — step through the descending iteration (narrowing) to see precision recovery after fixpoint
  • Domain decomposition — EVA uses ~12-24 domains simultaneously; the debugger can show each domain's contribution individually
  • History tracking — the full sequence of states at any program point across all iterations is recorded and browsable

What WPO Enables

The traditional iteration strategy (WTO — Weak Topological Ordering) processes the entire control-flow graph in a fixed order. The newer Weak Partial Order (WPO) structure decomposes iteration into independently schedulable units. This is what makes interactive stepping possible: the debugger can pause between any two schedulable units without corrupting the analysis state.

Interface Modes

VS Code via DAP

Communicates via the Debug Adapter Protocol (DAP) — Microsoft's standard protocol for IDE/debugger integration. Standard DAP operations (launch, setBreakpoints, next, stepIn, continue, variables, stackTrace) are mapped to abstract interpretation operations. This means any DAP-compatible IDE works out of the box.

Ivette GUI

Frama-C's Ivette graphical interface provides a visual debugger panel: source code with highlighted current location, variable state panels, iteration graphs, and control flow visualization with abstract state annotations.

ACSL-Integrated Breakpoints

Breakpoints can be set using ACSL annotations (Frama-C's specification language) — including conditional breakpoints based on ACSL predicates. Users can also add ACSL annotations on-the-fly during debugging to refine the analysis in real-time.

Comparison: GobPie

The University of Tartu's GobPie abstract debugger for Goblint uses MagpieBridge (a framework for integrating static analyzers into IDEs) and also speaks DAP. Unlike Frama-C's on-the-fly approach, GobPie explores pre-computed abstract states — it shows multiple ARG nodes at breakpoints as "threads" for synchronized stepping.

3. Under the Hood: Fixpoint Iteration

To understand how an abstract debugger works, you need to understand what it's debugging: fixpoint computation.

The Abstract Interpretation Pipeline

Widening: x ∈ [0,+∞) ← widening accelerates convergence Narrowing: x ∈ [0,10] ← narrowing recovers precision

Why Stepping Is Interesting

The critical insight: each iteration step transforms an abstract state using a transfer function, and these transformations are individually inspectable. At each step, you can see:

The Transfer Function

What semantic operation is being applied?

// Assignment: x = x + 1
// Transfer: [a,b] → [a+1, b+1]
pre:  x ∈ [0, 9]
post: x ∈ [1, 10]

The Join (Merge)

When branches reconverge, information is lost:

// if (flag) x = 5; else x = 10;
true branch:  x ∈ [5, 5]
false branch: x ∈ [10, 10]
join:         x ∈ [5, 10]  ← precision lost

The Widening

To guarantee termination on loops:

// Iteration 2: x ∈ [0, 2]
// Iteration 3: x ∈ [0, 3]
// Widening applied:
// [0,2] ▽ [0,3] = [0, +∞)
// "Growing upper bound → extrapolate to ∞"

The Narrowing

After fixpoint, recover precision:

// Fixpoint: x ∈ [0, +∞)
// But loop condition: x < 10
// Narrowing:
// [0,+∞) △ [0,9] = [0, 10]
// "Condition constrains the upper bound"

4. Weak Topological Ordering (WTO) — Bourdoncle's Foundation

François Bourdoncle's 1993 paper "Efficient Chaotic Iteration Strategies with Widenings" introduced the Weak Topological Ordering (WTO), which became the standard iteration strategy for abstract interpretation for three decades.

The Idea

A WTO decomposes a control-flow graph into a hierarchical ordering of nodes, where:

  • Singleton elements represent straight-line code — visited once per pass
  • Components (delimited by parentheses) represent loops — iterated until stabilization
  • The head of a component is the loop header — where widening is applied
CFG: WTO: ┌─────────┐ 1 2 (3 4 5) 6 │ 1 → 2 → 3 → 4 → 5 ──┐ │ │ │ │ │ ↑ │ │ │ └─loop────┘ │ └────────┘ │ │ │ 3 → 6 └─sequential─────┘ └─────────┘ Read: "Process 1, then 2, then iterate {3,4,5} until stable, then 6" Nested loops produce nested components: 1 (2 (3 4) 5) 6 = "Process 1, then iterate {2, iterate {3,4}, 5}, then 6"

How WTO Drives Iteration

Recursive Iteration Strategy

function iterateWTO(wto):
  for each element in wto:
    if element is singleton node n:
      state[n] = transferFunction(n, predecessorStates(n))
    if element is component (head, body...):
      // Iterate the component until fixpoint
      loop:
        old = state[head]
        state[head] = join(predecessorStates(head))
        if iterations > threshold:
          state[head] = widen(old, state[head])  // ← widening point
        iterateWTO(body)  // recursively process body
        if state[head] == old:  // fixpoint reached
          break
      // Optional: narrowing passes
      for i in 1..narrowingPasses:
        iterateWTO([head, ...body])

The WTO determines exactly where widening is applied (at component heads) and in what order nodes are processed. This is the "iteration strategy."

Example: Simple Loop

// Source:
int x = 0;           // node 1
while (x < 100) {    // node 2 (loop head)
  x = x + 1;         // node 3
}
return x;             // node 4

// WTO: 1 (2 3) 4
// Iteration trace:

Pass 1:
  node 1: x ∈ [0,0]
  node 2: x ∈ [0,0]  (from node 1)
  node 3: x ∈ [1,1]  (x+1 on [0,0])

Pass 2:
  node 2: join([0,0], [1,1]) = [0,1]
  node 3: x ∈ [1,2]

Pass 3:
  node 2: join([0,0], [1,2]) = [0,2]
  → WIDEN: [0,1] ▽ [0,2] = [0, +∞)  ← 🔍 debugger shows this!
  node 3: x ∈ [1, +∞)

Pass 4 (check fixpoint):
  node 2: join([0,0], [1,+∞)) = [0,+∞) = old → STABLE ✓

Narrowing:
  node 2: [0,+∞) △ [0,99] = [0, 99]  (from condition x<100)
  node 3: x ∈ [1, 100]
  node 2: narrowed to [0, 100]

  node 4: x ∈ [100, 100]  (from condition ¬(x<100))

Why WTO Alone Isn't Enough for Debugging

WTO defines a total order on the iteration sequence. This means:

  • No parallelism — nodes in the same component must be processed sequentially in WTO order
  • Coarse stepping granularity — you can only pause between complete WTO elements, not at arbitrary points within a component iteration
  • No branch exploration — at a conditional, WTO forces you to process both branches in order; you can't choose to explore one branch first
  • Rigid schedule — the iteration order is fixed at WTO construction time; the debugger can't adapt the schedule based on user interest

This is why the Weak Partial Order (WPO) was developed — it generalizes WTO to a partial order, enabling flexible, interactive scheduling.

5. Weak Partial Orders (WPO) — The Key Enabler

The WPO was introduced by Sung Kook Kim, Arnaud J. Venet, and Aditya V. Thakur (UC Davis / Facebook) at POPL 2020 in their paper "Deterministic Parallel Fixpoint Computation". It generalizes Bourdoncle's WTO from a total ordering to a partial ordering, unlocking both parallelism and interactivity. Their implementation PIKOS demonstrated deterministic parallel abstract interpretation based on the NASA IKOS tool. A follow-up paper at SAS 2020 ("Memory-Efficient Fixpoint Computation") introduced MIKOS with 24.57x peak-memory reduction.

A WPO uses two types of edges: scheduling edges (encoding "must happen before" dependencies) and stabilization edges (encoding "loop must reach fixpoint before exit"). The fundamental theorem: a WTO is a linear extension of a WPO — parallel/interactive fixpoint computation based on a WPO produces the same fixpoint as sequential computation based on a WTO.

WTO vs WPO: The Key Difference

WTO (Total Order)

1 → 2 → (3 → 4 → 5) → 6 One valid execution order. Must process 3 before 4 before 5. No flexibility. Like a recipe: "first this, then that, then the other"

WPO (Partial Order)

1 │ 2 ╱ ╲ 3 4 ← can run in any order! ╲ ╱ 5 │ (L) ← loop scheduling node │ 6 Like a dependency graph: "3 and 4 are independent; 5 needs both"

How WPO Enables the Debugger

Flexible Scheduling

WPO only specifies dependencies, not a total order. The debugger acts as a scheduler: it picks which ready node to process next based on user commands (step, continue, explore branch).

Branch Selection

At a conditional, both branches become independently schedulable. The user can choose to explore the true branch first, inspect its final state, then explore the false branch — before the join.

Pause Anywhere

Since the WPO captures all necessary ordering constraints, pausing between any two ready nodes is always safe. The analysis state is consistent at every pause point.

Deterministic Replay

Recording the sequence of scheduling decisions (which ready node was picked) allows exact replay. The same WPO + same schedule = same analysis trace.

WPO Construction

Technical Details: WPO Algorithm

A WPO for a CFG G = (V, E) with entry s is a DAG W = (N, ≤) where:

  • Each CFG node v ∈ V has a corresponding WPO node
  • Each loop (strongly-connected component) gets an additional scheduling node that enforces iteration
  • The partial order ≤ captures all data-flow dependencies
// WPO construction (simplified)
function buildWPO(cfg, entry):
  // 1. Compute dominator tree
  domTree = computeDominators(cfg, entry)

  // 2. Identify back edges (loops)
  backEdges = findBackEdges(cfg, domTree)

  // 3. For each natural loop:
  for each (tail → head) in backEdges:
    loopBody = nodesInLoop(head, tail)
    // Create scheduling node for the loop
    schedNode = createSchedulingNode(head)
    // Add dependency: all loop-exit edges depend on schedNode
    for each exit in loopExits(loopBody):
      addEdge(schedNode, exit)
    // schedNode depends on loop body completion
    addEdge(tail, schedNode)

  // 4. Non-loop edges become direct WPO edges
  for each (u → v) in cfg.edges - backEdges:
    addEdge(u, v)

  return WPO

// Runs in almost-linear time: O(E · α(E, V))
// where α is the inverse Ackermann function

The algorithm is described in detail in Kim et al., 2019. Their implementation PIKOS (Parallel IKOS) demonstrated that WPO-based scheduling achieves near-linear speedup on multi-core machines — the same property that enables interactive debugging.

7. Abstract Domains: What You See at Each Step

The abstract debugger shows you abstract values at each program point. Different abstract domains give different views of the same concrete state:

Interval Domain

Each variable independently bounded: x ∈ [lo, hi]

After: int x = 0; int y = 10; while(x < 5) { x++; y--; } Interval analysis result: x ∈ [0, 5] ← knows x range y ∈ [5, 10] ← knows y range x + y ∈ [5, 15] ← imprecise! (actual: always 10) Why imprecise? Intervals can't express "x + y = 10"

Cost: O(n) per variable. Precision: No relational information between variables.

Octagon Domain (Antoine Miné, 2001)

Constraints of the form ±x ± y ≤ c

After the same program: Octagon constraints: x ≥ 0, x ≤ 5 y ≥ 5, y ≤ 10 x + y ≤ 10 ← relational! x + y ≥ 10 ← relational! → x + y = 10 ← PRECISE ✓ The octagon captures the invariant x + y = 10 because it's expressible as ±x ± y ≤ c

Cost: O(n²) per program point. Precision: Captures difference constraints and sums between pairs of variables.

Polyhedra Domain (Cousot & Halbwachs, 1978)

Arbitrary linear constraints: a₁x₁ + a₂x₂ + ... + aₙxₙ ≤ c

After the same program: Polyhedra constraints: x ≥ 0 ∧ x ≤ 5 ∧ y ≥ 5 ∧ y ≤ 10 ∧ x + y = 10 Plus any combination: 2x + 3y ≤ 40 (derivable from above) Most precise numeric domain, but...

Cost: Exponential in the worst case (vertex/constraint conversion). Precision: Maximum for linear properties, but too expensive for large programs.

Reduced Product (What Astrée/EVA Actually Use)

Production analyzers combine multiple domains, reducing them against each other:

EVA's domain stack (simplified): ┌─────────────────────────────────┐ │ Interval Domain x∈[0,5] │──┐ ├─────────────────────────────────┤ │ │ Congruence Domain x ≡ 0 (1) │ │ reduce ├─────────────────────────────────┤ │ ↕ │ Gauge Domain x = 0+1·i │──┘ ├─────────────────────────────────┤ │ Octagon Domain x+y ≤ 10 │──── relational ├─────────────────────────────────┤ │ Symbolic Const Prop y=10-x │──── symbolic ├─────────────────────────────────┤ │ Pointer Domain p → {&a, &b} │──── memory └─────────────────────────────────┘ "Reduction" = domains share information: If intervals say x ∈ [0, 5] and gauge says x = 0 + 1·i (loop counter) then octagons can infer x + y = 10 The debugger shows EACH domain's contribution!

The abstract debugger's power: it can show which domain contributed which constraint, and how reduction improved precision. This is invaluable for domain developers.

9. Prior Art: AG Debuggers (DrAST, Aki, Silver)

Before we design our own abstract debugger, let's learn from existing attribute grammar debugging tools:

DrAST (Lund University)

A visual debugger for JastAdd attribute grammars (Java-based RAG system).

  • Visualizes the decorated AST with attribute values
  • Filters nodes by type — "clustering" uninteresting subtrees
  • Draws reference edges between nodes (for reference attributes)
  • On-demand attribute evaluation — click an attribute to compute it
  • Built as a separate tool that connects to JastAdd's runtime

Key paper: Nilsson-Nyman et al., "DrAST — An Attribute Debugger for JastAdd"

Aki (Systematic AG Debugging)

Uses algorithmic debugging (Shapiro's method) adapted for attribute grammars.

  • Asks yes/no questions: "Is this attribute value correct?"
  • Uses answers to narrow down the buggy attribution rule
  • Also supports slice-based debugging — computes which rules influenced a wrong attribute
  • Combines both approaches for faster bug localization

Key paper: Chitil & Luo, "Generalized Systematic Debugging for Attribute Grammars"

Silver (University of Minnesota)

An extensible AG system with IDE integration.

  • Eclipse-based editor with AG-aware error reporting
  • Attribute value inspection in IDE tooltips
  • Dependency graph visualization
  • Attribute profiling — which attributes are most expensive
  • Forwarding visualization for higher-order attributes

Key system: Van Wyk et al., "Silver: An Extensible Attribute Grammar System"

JastAdd Tracing Framework

JastAdd has built-in tracing activated via --tracing flag:

  • Event categories: compute, cache, rewrite, circular, copy — each independently toggleable
  • API: ASTState.Trace.setReceiver(receiver) with events like COMPUTE_BEGIN, COMPUTE_END on (node, attribute, params, value)
  • Used for profiling, cache optimization, and the "Automated Selective Caching" paper (Söderberg & Hedin, SLE 2010)

This is the closest existing model to what KSC's evaluator instrumentation would look like.

Lessons for Our Design

Common themes from all tools:

  • Demand-driven evaluation is more debuggable than batch — evaluate attributes only when inspected
  • Tree visualization is essential — attributes only make sense in tree context
  • Dependency edges between attributes are the key debugging aid
  • Filtering/clustering is necessary — real ASTs are too large to show fully
  • DAP protocol for IDE integration is the standard (both Frama-C and GobPie use it)

Other Abstract Analysis Debuggers

MOPSA Interactive Debugger

MOPSA already has a GDB-like abstract debugger with commands:

step / stepi   — enter function / interpretation subtree
next / nexti   — skip function / subtree
continue       — run to next breakpoint
finish         — complete current function
backward       — time-travel to calling site
break <line>   — location breakpoint
break #alarm   — break at next analysis warning
print <var>    — show abstract value projection
env            — show current abstract environment
backtrace      — abstract call stack

Works because MOPSA "interprets the AST" rather than solving generic equations — execution flow mirrors concrete execution. Supports configuration-comparative debugging via tmux: run two domain configurations side-by-side with synchronized breakpoints.

VisuFlow (2018)

Eclipse plugin for debugging Soot-based data-flow analyses.

  • Graph visualization of CFG annotated with analysis states
  • Special breakpoints for stepping through analysis and analyzed code simultaneously
  • User study: 25% more errors identified, 50% more errors fixed vs standard Eclipse debugging

Kruger et al., "VISUFLOW: A Debugging Environment for Static Analyses" (2018)

Cross-Level Debugging (SLE 2023)

Introduced meta-predicates — breakpoint conditions that express properties about the analyzer's state, not just the analyzed program.

  • Debug at the base-layer (analyzed program) with meta-layer (analysis) accessible via conditional breakpoints
  • Implemented in MAF (modular abstract interpretation framework)

Van Molle, Vandenbogaerde, De Roover, SLE 2023

10. Building Our Own: The KSC Abstract Debugger

KSC's attribute grammar evaluator is structurally analogous to an abstract interpreter. Here's the mapping:

Abstract Interpretation ConceptKSC AG EquivalentExample in KSC
Abstract state at program point Attribute values at AST node kindDefs(FunctionDeclaration) = [{ name: "greet", ... }]
Transfer function Attribution equation defEnv(child) = defEnv(parent) ∪ kindDefs(siblings)
Fixpoint iteration Attribute evaluation schedule Bottom-up (syn), then top-down (inh), then collect
Control-flow graph AST + dependency graph 22 attributes with 27 dependency edges
Widening N/A (finite lattice — all KSC domains are finite) No widening needed: finite property sets, finite kind enums
Join/merge Collection attribute aggregation allViolations = ∪ violations from all property attrs
Abstract domain Attribute type KindDefinition[], PropertySet, CheckerDiagnostic[]
Branch exploration Child subtree exploration Step into function body vs. step over to next sibling

The Deep Connection: AG Evaluation IS Abstract Interpretation

This mapping isn't just an analogy — the connection is established in the literature (Roskilde University, PLILP 1990; Midkiff & Padua, ScienceDirect). The decorated tree IS the abstract fixpoint. KSC's demand-driven evaluation (where each attribute method checks if (this._c_X !== undefined) return this._c_X) is exactly lazy fixpoint iteration with memoization. An abstract debugger lets users observe this fixpoint being computed, one attribute at a time.

Key Advantage: No Widening Needed

Unlike Frama-C which must handle infinite-height lattices (integer intervals → widening), KSC's attribute domains are all finite:

  • kindDefs: bounded by number of declarations in program
  • defEnv: bounded by scope size
  • Property violations: bounded by number of AST nodes

This means KSC's evaluation always terminates in one pass (no fixpoint iteration needed for basic AGs without circular attributes). This makes the debugger simpler — no iteration tracking, no widening visualization.

11. Design: AG Evaluation as Abstract Interpretation

Two-Phase Evaluation with Instrumentation

KSC evaluates attributes in a dependency-driven order:

// Phase 1: Synthesized attributes (bottom-up)
//   Leaf nodes first, then parents
//   kindDefs, kindAnnotations, nodeCount, violations

// Phase 2: Inherited attributes (top-down)
//   Root first, then children
//   defEnv, defLookup, context attrs

// Phase 3: Collection attributes (full tree)
//   allViolations = aggregate all violation attrs

The debugger instruments this by wrapping each equation evaluation:

// Instrumented evaluation (conceptual)
function evaluateAttr(node, attrName, equation) {
  const deps = getDependencies(attrName);
  const depValues = deps.map(d => getAttr(d.node, d.attr));

  // 🔍 DEBUGGER HOOK: before evaluation
  debugger.beforeEval(node, attrName, depValues);
  if (debugger.hasBreakpoint(node, attrName)) {
    debugger.pause(); // yield to user
  }

  const result = equation(ctx, node);

  // 🔍 DEBUGGER HOOK: after evaluation
  debugger.afterEval(node, attrName, result);

  return result;
}

step-into If current attr depends on child attrs, step into the child node's evaluation Understanding how a synthesized attribute is built from children
step-over Evaluate all attributes of the current subtree, then pause at the next sibling Skipping uninteresting subtrees

continue Run until next breakpoint or evaluation complete Jump to the interesting part

// Evaluator becomes a generator
function* evaluateTree(root: KSCDNode): Generator<EvalStep> {
  // Phase 1: synthesized (bottom-up)
  for (const node of postOrder(root)) {
    for (const attr of synAttrs) {
      const step = evaluateAttr(node, attr);
      yield step;  // ← pause point for debugger
    }
  }
  // Phase 2: inherited (top-down)
  for (const node of preOrder(root)) {
    for (const attr of inhAttrs) {
      const step = evaluateAttr(node, attr);
      yield step;
    }
  }
}

2. Generator variant for live stepping

// New codegen option: generateSteppingEvaluator
function* evaluateStepping(root, options = {}) {
  const trace = options.trace ?? [];

  // Bottom-up pass
  function* visitPostOrder(node) {
    for (const child of getChildFields(node)) {
      yield* visitPostOrder(child);
    }
    // Evaluate synthesized attributes
    for (const [name, eq] of synEquations(node.kind)) {
      const val = eq(makeCtx(node));
      node[name] = val;
      yield { nodeId: node.id, attr: name, value: val, phase: 'syn' };
    }
  }
  yield* visitPostOrder(root);

  // Top-down pass
  function* visitPreOrder(node) {
    for (const [name, eq] of inhEquations(node.kind)) {
      const val = eq(makeParentCtx(node));
      node[name] = val;
      yield { nodeId: node.id, attr: name, value: val, phase: 'inh' };
    }
    for (const child of getChildFields(node)) {
      yield* visitPreOrder(child);
    }
  }
  yield* visitPreOrder(root);
}

13. Comparison: Frama-C vs KSC Approach

AspectFrama-C Abstract DebuggerKSC AG Debugger (Proposed)
Target C programs (safety-critical embedded) TypeScript/JavaScript programs
Analysis type Abstract interpretation (numeric domains, pointer analysis, memory) Attribute grammar evaluation (name binding, kind checking, property violations)
Iteration Fixpoint iteration with widening/narrowing (may iterate loops many times) Single-pass tree traversal (no fixpoint needed for non-circular AGs)
Ordering structure WPO over CFG (Weak Partial Order) Dependency graph over attributes (DAG of 27 edges)
State space Infinite-height lattices (intervals, polyhedra) — needs widening Finite domains (kind enums, property sets) — always terminates
Stepping granularity Per CFG node (statement-level) Per attribute evaluation (finer: multiple attributes per AST node)
Branch exploration True/false branches at conditionals Child subtrees at AST nodes
Undo mechanism Immutable abstract states in history Immutable attribute values + trace replay
Implementation language OCaml TypeScript (same language as analysis target)
Protocol DAP (Debug Adapter Protocol) via VS Code extension WebSocket or postMessage to dashboard
UI VS Code (DAP) + Ivette GUI Existing React/Remix dashboard + CLI
Key advantage Production-proven on safety-critical industrial code Same-language tooling (TS analyzing TS), web-native UI, simpler evaluation model
Key challenge Widening/narrowing makes traces non-monotone and hard to follow Large ASTs (thousands of nodes × 22 attributes = many steps to navigate)

Unique Opportunities for KSC

Same-Language Advantage

KSC analyzes TypeScript using TypeScript. The debugger, evaluator, and target program all run in the same runtime. This enables features like: evaluating arbitrary TS expressions against the abstract state, using Chrome DevTools Protocol for integration, and running everything in the browser.

Web-Native UI

The dashboard is already a web app. No separate desktop tool needed. The debugger can be a URL you open — no installation. Share debugging sessions by sharing a trace file. Even embed it in VS Code's webview panel.

Simpler Model

No widening, no narrowing, no fixpoint iteration. The evaluation is a single tree traversal. This makes the debugger much simpler to implement and much easier to understand. Every attribute is computed exactly once.

Generator-Based Stepping

JavaScript generators (function*) provide pause/resume natively. No coroutine library needed. Combined with async generators, can even support WebSocket-based remote debugging with minimal complexity.
