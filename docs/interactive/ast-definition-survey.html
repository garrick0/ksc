How Languages Define Their ASTs

A survey of how real compiler systems define tree structure, node schemas, child relationships, and properties — and what KSC should learn from them.

▶

1. The Fundamental Question

Foundation

Every compiler needs to define its AST — the tree of nodes that represents a program's structure. But how you define that tree varies enormously across systems. The central question is:

Where does the grammar live?

Every system must answer: "For each node kind, what children can it have, in what order, and with what types?" Some systems answer this explicitly in a schema. Others encode it implicitly across their parser, factory, and traversal code. The answer shapes everything downstream: type safety, tooling, extensibility, and maintenance burden.

We can categorize systems along two axes:

Schema is Explicit Schema is Implicit
Code Generated Roslyn (Syntax.xml), rust-analyzer (ungrammar), JastAdd (.ast files), tree-sitter (grammar.js) ANTLR (grammar defines parse tree, not AST)
Hand-Written KSC defineNode() (our new approach) TypeScript (4 parallel tables), Go, GCC, most compilers

Let's examine each major system in detail.

▶

2. Roslyn — The Gold Standard of Schema-First

Generated

Roslyn (the C# and VB compiler) is the most sophisticated example of schema-first AST definition. Everything is generated from a single XML schema.

The Pipeline

Syntax.xml → GenerateSyntax tool → SyntaxNode classes + SyntaxFactory + Visitor + Rewriter

The schema defines every node kind, its children, and their types:

<!-- From dotnet/roslyn: src/Compilers/CSharp/Portable/Syntax/Syntax.xml -->

<Node Name="IfStatementSyntax" Base="StatementSyntax">
  <Kind Name="IfStatement"/>
  <Field Name="IfKeyword" Type="SyntaxToken">
    <Kind Name="IfKeyword"/>
  </Field>
  <Field Name="OpenParenToken" Type="SyntaxToken">
    <Kind Name="OpenParenToken"/>
  </Field>
  <Field Name="Condition" Type="ExpressionSyntax"/>
  <Field Name="CloseParenToken" Type="SyntaxToken">
    <Kind Name="CloseParenToken"/>
  </Field>
  <Field Name="Statement" Type="StatementSyntax"/>
  <Field Name="Else" Type="ElseClauseSyntax" Optional="true"/>
</Node>

<Node Name="ClassDeclarationSyntax" Base="TypeDeclarationSyntax">
  <Kind Name="ClassDeclaration"/>
  <Field Name="Keyword" Type="SyntaxToken">
    <Kind Name="ClassKeyword"/>
  </Field>
</Node>

<!-- Base class defines shared children -->
<AbstractNode Name="TypeDeclarationSyntax" Base="BaseTypeDeclarationSyntax">
  <Field Name="TypeParameterList" Type="TypeParameterListSyntax" Optional="true"/>
  <Field Name="ConstraintClauses" Type="SyntaxList<TypeParameterConstraintClauseSyntax>"/>
  <Field Name="Members" Type="SyntaxList<MemberDeclarationSyntax>"/>
</AbstractNode>

Key features: Each field has a name, type, and optionality. Inheritance (Base=) avoids repeating shared fields. Token kinds are constrained via nested <Kind> elements. Lists use SyntaxList<T> or SeparatedSyntaxList<T>. Roslyn also supports <Choice> and <Sequence> elements for expressing alternatives (e.g., a method can have a block body OR an expression body + semicolon) — these are flattened into optional fields during codegen.

From this XML, Roslyn generates ~30,000 lines of C# code:

// GENERATED — do not edit by hand
public sealed partial class IfStatementSyntax : StatementSyntax
{
    internal IfStatementSyntax(
        SyntaxToken ifKeyword,
        SyntaxToken openParenToken,
        ExpressionSyntax condition,
        SyntaxToken closeParenToken,
        StatementSyntax statement,
        ElseClauseSyntax? @else)
    {
        this.ifKeyword = ifKeyword;
        this.condition = condition;
        this.statement = statement;
        this.@else = @else;
        // ...
    }

    public SyntaxToken IfKeyword => this.ifKeyword;
    public ExpressionSyntax Condition => this.condition;
    public StatementSyntax Statement => this.statement;
    public ElseClauseSyntax? Else => this.@else;

    // Also generates: Accept(visitor), Update(...), WithXxx(...)
}

Also generates: SyntaxFactory.IfStatement(...), SyntaxVisitor.VisitIfStatement(), SyntaxRewriter dispatch, serialization, etc.

Red/Green Tree Architecture

Roslyn uses an immutable "green tree" (position-free, shared) wrapped by a "red tree" (position-aware, parent-linked). Both are generated from the same schema.

  • Green nodes: Immutable, position-free, structurally shared. This is the persistent data structure.
  • Red nodes: Lazy wrappers that add parent pointers, absolute positions. Created on demand as you navigate.

The XML schema generates both layers. Every <Field> becomes a property on both the green and red node.

What Roslyn generates from the schema

  • Green node classes with child slots
  • Red node classes with typed accessors
  • SyntaxFactory: SyntaxFactory.IfStatement(condition, body, elseClause)
  • Visitor base class with one method per node kind
  • Rewriter with Update/With methods for immutable transforms
  • Serialization and debugging support
  • Internal slot arrays for child enumeration

Lesson for KSC

Roslyn proves that schema-first works at massive scale (800+ node kinds). The XML schema is the single source of truth — nothing is hand-written. Our defineNode() approach is the TypeScript equivalent of Roslyn's Syntax.xml, but using runtime builders instead of XML + codegen.

3. rust-analyzer — Ungrammar + Code Generation

Generated

rust-analyzer uses ungrammar, a purpose-built DSL that describes the concrete syntax tree structure. It's designed to be the single source of truth for the AST, generating typed Rust node wrappers.

The Pipeline

rust.ungram → sourcegen_ast → generated/nodes.rs + generated/tokens.rs

The ungrammar format is concise and readable:

// From rust-analyzer: crates/syntax/rust.ungram

IfExpr =
  Attr* 'if' condition:Expr
  then_branch:BlockExpr
  ('else' else_branch:(IfExpr | BlockExpr))?

FnDef =
  Attr* Visibility?
  'default'? 'const'? 'async'? 'unsafe'? Abi?
  'fn' Name GenericParamList? ParamList RetType?
  WhereClause?
  (body:BlockExpr | ';')

StructDef =
  Attr* Visibility?
  'struct' Name GenericParamList? WhereClause?
  (RecordFieldList | TupleFieldList | ';')

// Enum nodes: alternatives separated by |
Item =
  Const
| Enum
| ExternBlock
| ExternCrate
| Fn
| Impl
| MacroCall
| Module
| Static
| Struct
| TypeAlias
| Union
| Use

Key features: Named fields use name:Type syntax. Optionality is ?. Repetition is *. Alternatives are |. Tokens are quoted literals. The format is deliberately not a parsing grammar — it describes tree structure only (hence "un-grammar").

The codegen tool produces typed wrapper structs:

// GENERATED by sourcegen_ast

#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct IfExpr {
    pub(crate) syntax: SyntaxNode,
}

impl IfExpr {
    pub fn if_token(&self) -> Option<SyntaxToken> { ... }
    pub fn condition(&self) -> Option<Expr> { ... }
    pub fn then_branch(&self) -> Option<BlockExpr> { ... }
    pub fn else_branch(&self) -> Option<ElseBranch> { ... }
}

impl AstNode for IfExpr {
    fn can_cast(kind: SyntaxKind) -> bool {
        kind == SyntaxKind::IF_EXPR
    }
    fn cast(syntax: SyntaxNode) -> Option<Self> { ... }
    fn syntax(&self) -> &SyntaxNode { &self.syntax }
}

Rowan: The Underlying Tree Library

rust-analyzer's syntax tree is built on rowan, a library that provides the generic tree structure. Rowan is analogous to our ag library:

  • SyntaxKind: A numeric tag (like our kind: string)
  • GreenNode: Immutable, position-free tree nodes (like Roslyn's green tree)
  • SyntaxNode: Position-aware wrapper with parent pointers (like our stamped tree)
  • Language trait: Parametrizes rowan for a specific language — provides kind_from_raw() and kind_to_raw()

The generated AST types (IfExpr, FnDef, etc.) are thin wrappers over SyntaxNode — they add typed accessors but the tree structure is owned by rowan.

Two-layer architecture (like ours!)

  • rowan = generic tree library (our libs/ag/)
  • ungrammar + generated code = language-specific schema (our libs/ag-schema/ + node-defs.ts)

The parallel is almost exact: rowan doesn't know about Rust syntax, just like our AG library doesn't know about TypeScript syntax. The schema layer adds language-specific knowledge on top.

Lesson for KSC

Ungrammar is the closest prior art to what we want. It's a schema DSL that generates typed accessors from a single source of truth. The key difference: ungrammar uses a custom DSL + codegen, while we use TypeScript builders (defineNode()) that provide types via inference. Our approach avoids a build step at the cost of more verbose definitions.

4. JastAdd — The AG Pioneer

Generated

JastAdd is the system our AG library is most directly inspired by. It uses .ast grammar files to define tree structure, then .jrag/.jadd files for attribute equations. The grammar is explicitly separate from the attributes.

The Pipeline

.ast grammar + . jrag attributes → JastAdd compiler → Java AST classes

JastAdd's grammar is BNF-like, defining the tree structure:

// .ast file — tree structure definition

Program ::= CompilationUnit*;

CompilationUnit ::= ImportDecl* TypeDecl*
    /<fileName:String>/;     // token attribute (non-child property)

// Children use names and types
IfStmt : Stmt ::=
    Condition:Expr
    Then:Stmt
    [Else:Stmt];              // [...] means optional child

ClassDecl : TypeDecl ::=
    <Name:String>             // <...> is a token (scalar property)
    [SuperClass:Access]
    Implements:Access*        // * means list
    BodyDecl*;

MethodDecl : BodyDecl ::=
    Modifiers
    TypeAccess:Access
    <Name:String>
    Parameter:ParameterDecl*
    Exception:Access*
    [Block];

// Abstract productions
abstract Stmt;
abstract Expr : Stmt;
abstract TypeDecl : BodyDecl ::= <Name:String>;

Syntax summary:

  • Child:Type — named child
  • [Child:Type] — optional child (Opt node)
  • Child:Type* — list of children (List node)
  • <Name:Type> — token attribute (scalar property, not a child)
  • /Child/ — NTA (non-terminal attribute): a child computed by an equation, not the parser
  • abstract — abstract node (cannot be instantiated)
  • A : B — A extends B

Attributes are defined separately in .jrag files:

// .jrag file — attribute equations (separate from grammar!)

// Synthesized attribute: computed bottom-up
syn String MethodDecl.signature() =
    getName() + "(" + paramTypes() + ")";

// Inherited attribute: passed top-down
inh TypeDecl Expr.hostType();
eq ClassDecl.getBodyDecl(int i).hostType() = this;

// Collection attribute: aggregated across the tree
coll Set<String> CompilationUnit.allMethodNames()
    [new HashSet<String>()]   // initial value
    with add;                  // combiner operation
MethodDecl contributes getName()
    to CompilationUnit.allMethodNames();

// Circular attribute: fixed-point iteration
syn boolean Stmt.isReachable() circular [true];
eq IfStmt.getThen().isReachable() =
    getCondition().isTrue() || isReachable();

// NTA (non-terminal attribute): computed subtree
syn nta TypeDecl ClassDecl.unknownType() =
    new UnknownType();

// Aspect: add equations to existing productions from another file
aspect CodeGeneration {
    syn byte TypeDecl.arrayLoad() {
        throw new Error("No array load");
    }
    eq IntType.arrayLoad() = Bytecode.IALOAD;
    eq ReferenceType.arrayLoad() = Bytecode.AALOAD;
}

The key design: Grammar (structure) and attributes (behavior) are completely separate files. The grammar defines WHAT the tree looks like. The attributes define HOW to compute over it. This is the de-fusion principle we adopted in KSC.

JastAdd generates Java classes with both structure and attributes woven in:

// GENERATED by JastAdd

public class IfStmt extends Stmt {
    // From grammar: children
    private Expr tokenExpr_Condition;
    private Stmt tokenStmt_Then;
    private Opt<Stmt> tokenOpt_Else;  // optional child

    // Accessors
    public Expr getCondition() { return tokenExpr_Condition; }
    public Stmt getThen() { return tokenStmt_Then; }
    public boolean hasElse() { return tokenOpt_Else.getNumChild() != 0; }
    public Stmt getElse() { return tokenOpt_Else.getChild(0); }

    // From attributes: woven in
    private boolean isReachable_computed = false;
    private boolean isReachable_value;
    public boolean isReachable() {
        if (isReachable_computed) return isReachable_value;
        // ... Magnusson-Hedin circular evaluation
    }

    // Tree navigation (built-in)
    public ASTNode getParent() { return parent; }
    public int getNumChild() { return numChildren; }
    public ASTNode getChild(int i) { return children[i]; }
}

Lesson for KSC

JastAdd's clean separation of grammar (.ast) and attributes (.jrag) is what inspired our de-fusion architecture. Our defineNode() schema is the equivalent of .ast files. Our SpecInput declarations + equations are the equivalent of .jrag files. The difference: JastAdd uses code generation (Java classes), while we use runtime installation (lazy getters on JavaScript objects).

5. TypeScript Compiler — Implicit Schema, Four Parallel Tables

Hand-Written

TypeScript's compiler is the system we mirror most directly — our AST is a projection of tsc's AST. But tsc has no explicit schema. Instead, the same structural knowledge is encoded redundantly across four independent dispatch tables.

TypeScript encodes "what children does IfStatement have?" in four separate places, each maintained independently:

Table Location Purpose Encodes
1. Interfaces typescript.d.ts Type definitions Field names, types, optionality
2. forEachChildTable typescript.js:32485 Read-only traversal Child fields, visitation order, single vs list
3. visitEachChildTable typescript.js:95327 Transform/clone Child fields, factory method calls, type guards
4. NodeFactory typescript.js:27125 Node creation Constructor parameters, child assignment, flags

If you add a child to a node kind, you must update all four places. They can (and occasionally do) get out of sync. This is the exact problem a schema-first approach solves.

The forEachChildTable is a dispatcher object mapping SyntaxKind numbers to handler functions. This is tsc's closest equivalent to a grammar:

// typescript.js:32485+ (simplified)

const forEachChildTable = {
  [246 /* IfStatement */]: function(node, cbNode, _cbNodes) {
    return visitNode2(cbNode, node.expression)    // single child
        || visitNode2(cbNode, node.thenStatement)  // single child
        || visitNode2(cbNode, node.elseStatement); // optional child
  },

  [263 /* FunctionDeclaration */]: function(node, cbNode, cbNodes) {
    return visitNodes(cbNode, cbNodes, node.modifiers)     // list
        || visitNode2(cbNode, node.asteriskToken)           // opt token
        || visitNode2(cbNode, node.name)                    // opt child
        || visitNodes(cbNode, cbNodes, node.typeParameters) // list
        || visitNodes(cbNode, cbNodes, node.parameters)     // list
        || visitNode2(cbNode, node.type)                    // opt child
        || visitNode2(cbNode, node.body);                   // opt child
  },

  [264 /* ClassDeclaration */]:
    forEachChildInClassDeclarationOrExpression,  // shared handler
  [232 /* ClassExpression */]:
    forEachChildInClassDeclarationOrExpression,  // same handler

  // ... 150+ more entries
};

Notice that visitNode2 = single child, visitNodes = child list. This is the implicit encoding of child/list/optChild that our defineNode() makes explicit.

The factory must re-encode the same children as parameters:

// typescript.d.ts — Factory interface

createIfStatement(
  expression: Expression,
  thenStatement: Statement,
  elseStatement?: Statement
): IfStatement;

createFunctionDeclaration(
  modifiers: readonly ModifierLike[] | undefined,
  asteriskToken: AsteriskToken | undefined,
  name: string | Identifier | undefined,
  typeParameters: readonly TypeParameterDeclaration[] | undefined,
  parameters: readonly ParameterDeclaration[],
  type: TypeNode | undefined,
  body: Block | undefined
): FunctionDeclaration;

// Also: updateIfStatement(...), updateFunctionDeclaration(...)
// which take the same parameters + the original node

The type definitions encode the same info yet again:

// typescript.d.ts

interface IfStatement extends Statement, FlowContainer {
  readonly kind: SyntaxKind.IfStatement;
  readonly expression: Expression;          // child
  readonly thenStatement: Statement;        // child
  readonly elseStatement?: Statement;       // optional child
}

// But children are spread across inheritance:
interface FunctionDeclaration
  extends FunctionLikeDeclarationBase, DeclarationStatement {
  readonly kind: SyntaxKind.FunctionDeclaration;
  readonly modifiers?: NodeArray<ModifierLike>;
  readonly name?: Identifier;
  readonly body?: FunctionBody;
}

// Must look at FunctionLikeDeclarationBase for:
//   asteriskToken, questionToken, exclamationToken, body
// And SignatureDeclarationBase for:
//   typeParameters, parameters, type
// And NamedDeclaration for:
//   name

The deep inheritance chains mean you need to traverse 4-5 base classes to find all children of a node. A flat schema avoids this complexity.

Lesson for KSC

TypeScript's approach is what happens when you grow a compiler organically without a schema. The knowledge is there, but it's scattered and redundant. Since our AST mirrors tsc's, we could theoretically extract our schema from tsc's forEachChildTable — it's the most complete encoding of "what children does each node have?" in the entire compiler. But our defineNode() approach is better because it's explicit, introspectable, and type-safe.

6. tree-sitter — JavaScript Grammar DSL

Generated

tree-sitter uses a JavaScript-based grammar DSL that defines both the parsing rules and the tree structure. The grammar is executable JavaScript that generates a C parser.

The Pipeline

grammar.js → tree-sitter generate → parser.c + node-types.json 
// tree-sitter-javascript/grammar.js (simplified)

module.exports = grammar({
  name: 'javascript',

  rules: {
    if_statement: $ => seq(
      'if',
      field('condition', $.parenthesized_expression),
      field('consequence', $.statement),
      optional(seq('else', field('alternative', $.statement)))
    ),

    function_declaration: $ => seq(
      optional('async'),
      'function',
      optional(field('name', $.identifier)),
      field('parameters', $.formal_parameters),
      field('body', $.statement_block)
    ),

    class_declaration: $ => seq(
      repeat(field('decorator', $.decorator)),
      'class',
      field('name', $.identifier),
      optional(field('type_parameters', $.type_parameters)),
      optional($.class_heritage),
      field('body', $.class_body)
    ),
  }
});

Key features: field('name', ...) creates named children — accessible as node.childForFieldName('name'). optional(...) marks optional children. repeat(...) marks lists. The grammar serves double duty: it defines both parsing rules AND tree structure.

tree-sitter also generates a node-types.json schema:

// Generated node-types.json (schema as data!)
[
  {
    "type": "if_statement",
    "named": true,
    "fields": {
      "condition": {
        "multiple": false,
        "required": true,
        "types": [{ "type": "parenthesized_expression", "named": true }]
      },
      "consequence": {
        "multiple": false,
        "required": true,
        "types": [{ "type": "statement", "named": true }]
      },
      "alternative": {
        "multiple": false,
        "required": false,
        "types": [{ "type": "statement", "named": true }]
      }
    }
  }
]

This JSON schema is exactly the kind of introspectable data our NodeSchema provides at runtime. tree-sitter uses it for syntax highlighting queries, code navigation, etc.

Lesson for KSC

tree-sitter's node-types.json is compelling: a machine-readable schema derived from the grammar. Our ksNodeSchema provides the same capability but at runtime in TypeScript. tree-sitter also shows the value of named fields — field('condition', ...) is the parsing-grammar equivalent of our condition: child().

7. Silver — Extensible AG with Forwarding

Generated

Silver (from the University of Minnesota) is an attribute grammar system designed for language extension. Its grammar definitions are similar to JastAdd's but with a focus on composability and forwarding.

-- Silver grammar definition

nonterminal Expr with pp, errors, env, type;
nonterminal Stmt with pp, errors, env;
nonterminal Decl with pp, errors, defs;

-- Productions define tree structure
abstract production ifStmt
top::Stmt ::= c::Expr  th::Stmt  el::Stmt
{
  top.pp = "if (" ++ c.pp ++ ") " ++ th.pp ++ " else " ++ el.pp;
  top.errors = c.errors ++ th.errors ++ el.errors;
  c.env = top.env;
  th.env = top.env;
  el.env = top.env;
}

abstract production classDecl
top::Decl ::= name::String  super::Maybe<TypeExpr>  body::[Decl]
{
  top.defs = [classDef(name, ...)];
  top.pp = "class " ++ name ++ ...;
}

-- Forwarding: extension productions translate to core
abstract production forLoop
top::Stmt ::= init::Stmt  cond::Expr  step::Stmt  body::Stmt
{
  forwards to
    blockStmt([init, whileStmt(cond, blockStmt([body, step]))]);
}

Key features: Silver combines grammar and attributes in the same production. Children are declared in the production signature (c::Expr th::Stmt el::Stmt). nonterminal declarations list which attributes exist. Forwarding enables language extensions.

Silver's Attribute System

  • Synthesized: synthesized attribute pp :: String occurs on Expr, Stmt;
  • Inherited: inherited attribute env :: [Def] occurs on Expr, Stmt;
  • Collection: collection attribute errors :: [Error] with ++ occurs on Expr;
  • Propagate: propagate env; — automatically copies inherited attrs to all children
  • Forwarding: Productions can delegate to other productions, enabling modular extension
  • Aspects: aspect production add ... { } — add equations to existing productions from another module

Silver's key innovation over JastAdd is modular well-definedness analysis — you can prove that independently developed language extensions won't conflict at compile time.

Lesson for KSC

Silver's approach of bundling grammar and attributes in the same production is the opposite of our de-fusion. But Silver's nonterminal declarations (listing which attributes exist on which nodes) are interesting — they're a form of declaration that could inform our completeness checking. Silver also shows that collection attributes with explicit monoid structure (with ++) are standard practice, validating our CollectionDecl design.

8. Other Notable Systems

Reference

ANTLR — Grammar Defines Parse Tree, Not AST

// ANTLR grammar (defines parse tree structure)
ifStatement
    : IF '(' expression ')' statement (ELSE statement)?
    ;

classDeclaration
    : CLASS IDENTIFIER typeParameters?
      (EXTENDS typeType)?
      (IMPLEMENTS typeList)?
      classBody
    ;

ANTLR generates a parse tree (CST), not an AST. Each grammar rule becomes a Context class with typed accessors for children. ANTLR also generates Visitor and Listener base classes. The parse tree is typically lowered to an AST by hand. This means ANTLR doesn't actually define the AST schema — just the parsing structure.

Kiama (Scala) — Case Classes as Schema

// Kiama uses Scala case classes directly
sealed trait Stmt
case class IfStmt(
  cond: Expr,
  thenBranch: Stmt,
  elseBranch: Option[Stmt]
) extends Stmt

case class ClassDecl(
  name: String,
  members: List[Member],
  parent: Option[TypeRef]
) extends Stmt

// Attribution via Kiama's library
val lookup: Expr => Option[Decl] =
  attr { case IdnUse(n) => env(n) }

val env: Node => Map[String, Decl] =
  attrWithDefault(Map.empty) {
    case tree.parent(Block(stmts, _)) => /* ... */
  }

Kiama's approach is the library approach — no code generation, no DSL. Scala case classes serve as the schema (they define children via constructor parameters). Pattern matching serves as production equations. This is the closest existing system to what we're building in TypeScript.

Spoofax/SDF3 — Language Workbench

// SDF3 syntax definition (Spoofax)
context-free syntax
  Stmt.If = [if ([Expr]) [Stmt] else [Stmt]]
  Stmt.While = [while ([Expr]) [Stmt]]
  Decl.Class = [class [ID] { [Member*] }]

// Generates: Java term classes, pretty-printer,
// parser, type signatures for Stratego rewriting

Spoofax is a full language workbench: the syntax definition generates everything including the parser, AST term classes, editor services, and Stratego rewriting rules. The grammar IS the schema. Most ambitious but also most heavyweight.

10. Patterns and Insights

Analysis

Pattern 1: Successful compilers converge on schema-first

Every compiler that has reached large scale (100+ node kinds) and is actively maintained eventually moves toward an explicit schema. Roslyn started hand-written and migrated to Syntax.xml. rust-analyzer created ungrammar specifically because hand-writing was unsustainable. JastAdd and Silver were schema-first from day one. TypeScript is the notable holdout — and it pays the cost in four parallel tables.

Pattern 2: The schema must define exactly four things

Every system that has an explicit schema defines the same four things per node kind:

  1. Kind discriminant — what kind of node this is
  2. Children — named sub-nodes, with cardinality (single, optional, list)
  3. Child types — what types of nodes can appear in each child slot
  4. Properties — non-child data (strings, numbers, booleans, flags)

Our defineNode('IfStatement', { expression: child(), thenStatement: child(), elseStatement: optChild() }) captures exactly these four things.

Pattern 3: Schema and attributes should be separate

JastAdd separates .ast (grammar) from .jrag (attributes). Roslyn separates Syntax.xml (structure) from binding/analysis (semantics). rust-analyzer separates ungrammar (syntax) from IDE analysis. Silver is the exception — it bundles them, but this makes extension harder. Clean separation enables:

  • Multiple independent analyses over the same tree structure
  • Schema validation independent of specific analyses
  • Tooling that works with structure alone (formatters, linters, visualizers)

Our architecture follows this: defineNode() (structure) is separate from SpecInput (attributes/analysis).

Pattern 4: Two layers — generic tree library + language-specific schema

Multiple systems use a two-layer architecture:

  • rust-analyzer: rowan (generic tree) + ungrammar (Rust-specific schema)
  • Kiama: Kiama core (generic attribution) + case classes (language-specific schema)
  • KSC: libs/ag/ (generic AG library) + libs/ag-schema/ + node-defs.ts (TypeScript-specific schema)

This is a validated architectural pattern. The generic library doesn't need to know about specific node kinds — that knowledge lives in the schema layer.

Pattern 5: Codegen vs Runtime — a genuine tradeoff

Systems split between code generation (Roslyn, JastAdd, rust-analyzer, tree-sitter) and runtime/library approaches (Kiama, KSC). The tradeoffs:

AspectCode GenerationRuntime Library
PerformanceOptimal (no dispatch)Good (lazy getters)
Type safetyFull static typesInferred types (phantom)
Build complexityNeeds build stepZero build step
DebuggingGenerated code is opaqueTransparent source
FlexibilityRigid (re-run generator)Dynamic (runtime schema)
IntrospectionIf schema is retainedSchema IS the runtime

For KSC, the runtime approach wins because: (a) our tree is already built by tsc's parser, so we don't need codegen for parsing; (b) TypeScript's type inference gives us static types without codegen; (c) runtime introspection enables completeness checking and tooling.

  • Cons: Verbose (2147 lines for 360 nodes). Type inference is phantom-based (less intuitive than plain interfaces). No auto-sync with tsc upgrades.
  • Best for: Iterating quickly, keeping complexity low.

Option B: Custom DSL + Codegen (Roslyn/ungrammar style)

Write our own DSL (like ungrammar) and generate ast.ts, node-defs.ts, and converter scaffolding from it.

// Hypothetical ksc.grammar
IfStatement : Statement =
  expression: Expression
  thenStatement: Statement
  elseStatement?: Statement

ClassDeclaration : Declaration =
  <name>: Identifier?
  typeParameters: TypeParameter*
  members: ClassElement*

  modifiers: Modifier*
  • Pros: Concise. Could generate both interfaces AND defineNode calls. Clean diff when AST changes. Could auto-generate converters too.
  • Cons: Needs a parser for the DSL. Build step required. Custom tooling to maintain. Overkill for 360 mostly-stable definitions.
  • Best for: If we expect frequent AST changes or want to target multiple languages.

Option C: Derive from tsc's Source

Use TypeScript's compiler API to parse typescript.d.ts and extract node interfaces automatically. Or parse the forEachChildTable from typescript.js.

  • Pros: Auto-sync with TypeScript upgrades. Zero manual maintenance. Always matches tsc's actual behavior.
  • Cons: Fragile (depends on tsc internals). Our AST isn't a pure mirror (KSProgram, KSCompilationUnit, string kinds, extra props). Complex extraction logic. Deep inheritance chains in tsc's types make extraction non-trivial.
  • Best for: If TypeScript upgrades frequently break our AST.

Option D: Hybrid — defineNode() + Verification Against tsc

Keep defineNode() as the source of truth, but add a verification test that compares our schema against tsc's forEachChild behavior at runtime. Catches drift on TypeScript upgrades.

  • Pros: Best of both worlds. Our schema is the source of truth (explicit, introspectable). tsc's behavior is the validation oracle. Fails fast if they diverge.
  • Cons: Need to construct sample nodes for each SyntaxKind to test. Some fields (like our extra props) can't be verified against tsc.
  • Best for: Production-quality confidence with minimal complexity.

Option E: tree-sitter Style — Export node-types.json

Add a build step that exports our ksNodeSchema to a node-types.json file, enabling tooling that doesn't need to import our TypeScript code.

  • Pros: Machine-readable schema for external tools. Could power editor extensions, syntax highlighting, code navigation. Standard format.
  • Cons: Another artifact to keep in sync. Minor — easy to generate from existing schema.
  • Best for: When we need external tooling support.

Recommended: A + D (current approach + tsc verification)

Option A is already implemented and working. Adding Option D (verification against tsc) would give us confidence that our schema stays in sync with TypeScript. This is the pragmatic path: we get schema-first benefits now, with a safety net for the future. Options B, C, and E can be added later if needed.

12. Quiz: Test Your Understanding

Quiz

Click each question to reveal the answer.

1. What is Roslyn's Syntax.xml?
An XML schema file that defines every syntax node kind in C#/VB. It specifies each node's name, base class, child fields (with names, types, and optionality), and token constraints. A code generator produces ~30,000 lines of C# from it: node classes, factory methods, visitor dispatch, and the red/green tree layers.
2. How does rust-analyzer's ungrammar differ from a parsing grammar?
Ungrammar describes tree structure, not parsing rules. It defines what children each node kind has (with names, types, optionality, repetition) but says nothing about how to parse text into that tree. The parser is written separately. That's why it's called "un-grammar" — it's the complement of a parsing grammar.
3. How does JastAdd separate structure from behavior?
.ast files define tree structure (children, their types, cardinality, token attributes). .jrag/.jadd files define attribute equations (syn, inh, collection, circular, NTA). The JastAdd compiler weaves them together into Java classes. This separation is the "de-fusion" principle our architecture follows.
4. Why does TypeScript's compiler have "four parallel tables"?
Because tsc has no explicit schema. The knowledge of "what children does IfStatement have?" is encoded independently in: (1) the interface definition in .d.ts, (2) the forEachChildTable for traversal, (3) the visitEachChildTable for transformation, (4) the NodeFactory for creation. These four must be kept in sync manually.
5. What is tree-sitter's node-types.json?
A machine-readable JSON schema generated from the grammar. For each node kind, it lists all named fields with their types, cardinality (multiple: true/false), and whether they're required. This enables tooling (syntax highlighting, code navigation) to understand the tree structure without parsing the grammar DSL.
6. What do Kiama and KSC have in common?
Both are library-based approaches that use the host language's own type system as the schema. Kiama uses Scala case classes; KSC uses TypeScript's defineNode() with phantom types. Neither requires a build step or code generation. Both provide runtime attribution (lazy computed attributes on tree nodes).
7. What is rowan's relationship to rust-analyzer?
Rowan is the generic tree library that rust-analyzer builds on. It provides SyntaxNode, SyntaxToken, and the green/red tree architecture. It doesn't know about Rust syntax — that knowledge lives in the ungrammar-generated typed wrappers. This two-layer architecture (generic tree + language-specific schema) mirrors our libs/ag/ + libs/ag-schema/ separation.
8. What are the four things every AST schema must define per node kind?
(1) Kind discriminant — what type of node this is. (2) Children — named sub-nodes with cardinality (single, optional, list). (3) Child types — what types of nodes can appear in each child slot. (4) Properties — non-child scalar data (strings, numbers, flags). Every system with an explicit schema encodes exactly these four things.
9. Why does KSC use runtime schema instead of code generation?
Three reasons: (a) Our tree is already built by tsc's parser, so we don't need codegen for construction. (b) TypeScript's type inference (via phantom types) gives us static types without codegen. (c) Runtime introspection enables completeness checking, schema-derived getChildren, and tooling — the schema IS the runtime, not a separate artifact that generates code.
10. What is Silver's "forwarding" and why does it matter?
Forwarding lets an extension production delegate to a host-language production. A forLoop production can forward to a whileLoop, meaning any attribute defined on whileLoop automatically works on forLoop without additional equations. This enables modular language extensions that compose safely. KSC doesn't need forwarding (we analyze a fixed language) but the concept validates modular attribute definition.
11. What is the recommended option for KSC going forward?
Option A + D: Keep defineNode() as the source of truth (already implemented, 360 definitions, 107 tests passing) and add a verification test that compares our schema against tsc's forEachChild behavior at runtime. This gives schema-first benefits now with a safety net that catches drift on TypeScript upgrades. Other options (DSL, codegen, tsc extraction) can be added later if needed.
