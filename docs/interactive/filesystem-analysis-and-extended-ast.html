1 Current State

What ast-schema/ gives us today — and where the tree stops.

What We Have

364 node kinds, 25 sum types mirroring TypeScript's full AST. The schema is the single source of truth — it generates typed interfaces, runtime schema, type guards, and getChildren.

Top of the tree:

Program
  └─ compilationUnits: CompilationUnit[]
       ├─ fileName: string
       ├─ isDeclarationFile: boolean
       ├─ sourceText: string
       └─ lineStarts: readonly number[]
         └─ (per-file AST nodes below)

What We Don't Have

  • No directory or folder nodes
  • No package/workspace structure
  • No repository root concept
  • No cross-file edges (import resolution)
  • No module dependency graph
  • No configuration nodes (tsconfig, package.json)
  • No control flow or data flow graphs
  • No call graph or type hierarchy
Key constraint
Program is the root. CompilationUnit is a flat list of files. There is no structural representation of how files relate to each other, where they live in the filesystem, or what packages they belong to.

2 The Gap

What questions can't we answer with today's schema?

Filesystem Questions

Which files are in src/utils/? What's the directory structure? Do any files outside src/ import from internal modules?

Module Questions

What does import { x } from './foo' resolve to? What's the full dependency graph? Are there circular imports?

Package Questions

Which workspace package does this file belong to? What are the cross-package dependencies? Do package boundaries match kind boundaries?

Architectural Questions

Does the core package depend on ui? Do all public API exports satisfy their kind contracts? Is the dependency graph acyclic?

Cross-File Properties

If module A is Kind<{pure}>, do all its transitive imports also satisfy purity? Can we verify kind contracts across module boundaries?

Configuration Questions

What compiler options affect this file? What path aliases are configured? What's the project's target environment?

The core insight
Per-file ASTs answer "what's in this file?" We also need to answer "how do files relate?", "what structures contain files?", and "what properties hold across boundaries?"

3 Prior Art

How existing tools model code above the per-file AST level.

Code Property Graph (Joern / ShiftLeft)

The CPG merges AST + CFG + PDG + Call Graph into a single queryable graph. It was introduced by Fabian Yamaguchi et al. (2014) and popularized by Joern and ShiftLeft Ocular.

Above-AST Node Types

Node TypeDescriptionProperties
META_DATARoot: language, version, overlays appliedlanguage, version, overlays[]
FILESource filename (absolute path), hash, order
NAMESPACE_BLOCKPackage/module containername, fullName, filename, order
TYPE_DECLType/class declarationname, fullName, inheritsFrom[]
METHODFunction/methodname, fullName, signature, lineNumber
DEPENDENCYExternal dependency (from build files)name, version, dependencyGroupId

Key Edge Types

METHOD
→ AST
BLOCK
→ CFG
CALL
→ REACHING_DEF
IDENTIFIER

AST (tree containment), CFG (control flow successor), CDG (control dependence), REACHING_DEF (data flow), CALL (call site → callee), REF (identifier → declaration), EVAL_TYPE (node → type), INHERITS_FROM (type hierarchy), SOURCE_FILE (node → file).

CPG's key idea
The AST is a subgraph of the CPG. Non-tree edges (CFG, data flow, calls) are overlaid onto the same node set. This means you can query "find all data flows from user input to SQL queries" as a single graph traversal.

CodeQL (GitHub / Semmle)

CodeQL uses language-specific relational databases extracted from source. Each language has a dedicated schema, but the filesystem model is common.

Filesystem Modeling

ConceptCodeQL RepresentationQuery Example
FileFile class with getAbsolutePath(), getFolder()from File f where f.getExtension() = "ts"
FolderFolder class with getParentContainer()from Folder f where f.getBaseName() = "src"
ContainerAbstract base of File and FolderEnables mixed queries
ModuleModule with getAnImport()from Import i select i.getImportedModule()

CodeQL can answer questions like "find all files in the src/ folder that import from node_modules" because the filesystem hierarchy is part of the database. Cross-file references are first-class — imports resolve to their target modules.

Roslyn (.NET Compiler Platform)

Roslyn introduces a full workspace hierarchy above the per-file syntax tree:

Roslyn's Workspace Model
Workspace
 ├─ Solution  (.sln file)
 │  ├─ Project  (.csproj file)
 │  │  ├─ Document  (.cs source file)
 │  │  │  └─ SyntaxTree  (per-file AST)
 │  │  ├─ Document
 │  │  ├─ ProjectReference  (cross-project)
 │  │  └─ MetadataReference  (NuGet, framework)
 │  └─ Project

Each Document owns a SyntaxTree (the per-file AST). The Compilation object sits alongside each Project and provides the semantic model (resolved types, symbols, diagnostics) — it's the bridge between syntax and semantics across files. Projects reference each other via ProjectReference edges, forming a project dependency graph.

SCIP & Kythe

These are source code indexing formats designed for cross-repository code navigation.

SCIP (Sourcegraph)

Index contains metadata + Document[] (one per file). Each Document has Occurrence[] and Symbol[] with cross-file references. Symbols use a structured naming scheme: scheme manager package descriptor. The graph is flattened — files are listed at the top level with no directory hierarchy.

Kythe (Google)

Everything is a (VName, Fact/Edge) triple. VNames encode corpus/root/path/language. Node kinds include file, package, record, function, variable. Edge kinds: /kythe/edge/childof, /kythe/edge/ref, /kythe/edge/defines/binding. The corpus+root fields implicitly encode repository structure.

Module Graph Tools

dependency-cruiser

Builds a module dependency graph from import/require statements. Each node is a file path. Edges carry metadata: resolved (target path), coreModule (Node built-in?), dependencyTypes (import, require, dynamic), valid (passes configured rules). Rules can express architectural boundaries: "nothing in src/ may import from test/".

Nx / Turborepo

Models a workspace graph at the package level. Each node is a project (workspace package) with metadata from project.json / package.json. Edges represent inter-package dependencies. Nx also models a task graph (build, test, lint) layered on top. The file-level module graph is computed within each project.

Two levels of graph
Module-graph tools operate at two levels: (1) the package graph (which workspace packages depend on which), and (2) the file module graph (which files import which files). KSC could benefit from modeling both.

TypeScript Compiler Internals

TypeScript's own compiler models multiple levels above the per-file AST:

LevelObjectWhat It Provides
SolutionSolutionBuilderMulti-project builds, project references, build ordering
Projectts.ProgramRoot files, compiler options, resolved modules, type checker

Notably, TypeScript's ts.Program has getSourceFiles() (flat list, like our CompilationUnit[]) but also getResolvedModuleWithFailedLookupLocations() which gives us the resolved import graph. The module resolution information exists at runtime but is not represented in the AST — it's a separate API.

Repository → Package[] → Directory[] → SourceFile[] → CompilationUnit → AST

// The edge overlay is a side structure:
interface EdgeOverlay {

D Full CPG: Unified Property Graph with AST as Subgraph
High effort Major change

Follow Joern's approach: build a Code Property Graph where the AST is one layer, and CFG, PDG, call graph, type hierarchy, and filesystem structure are all edges in a single unified graph. Every node has a unique ID, and edges connect any two nodes regardless of tree structure.

// Everything is a node with properties:
interface CPGNode {
  id:         number;
  nodeType:   'REPOSITORY' | 'PACKAGE' | 'FILE' | 'METHOD' | 'CALL' | ...;
  properties: Record<string, any>;
}

// Everything is connected by typed edges:
interface CPGEdge {
  src:      number;
  dst:      number;

// The existing AST is recovered by filtering to AST edges only
// AG evaluation can run on the AST subgraph

E Layered Schema: Container Tree + AG-Computed Semantic Edges
Recommended Best fit

A pragmatic synthesis of Options A and B. Extend the schema with filesystem container nodes (Option A). Use AG attributes — specifically reference attributes and collection attributes — to compute cross-file relationships declaratively (Option B). The result is a system where:

  • The tree represents containment: Repository → Package → Directory → SourceFile → AST
  • Inherited attributes flow context down (which package am I in? what config applies?)
  • Synthesized attributes aggregate up (what are this directory's total imports? what kind violations exist in this package?)
  • Reference attributes create cross-tree links (this import resolves to that SourceFile)
  • Collection attributes gather contributions (all files that export a particular symbol)

6 Cross-File Edge Types

The relationships that can't be expressed by tree containment — and how to represent them.

In a pure tree, every node has exactly one parent. But real code has many cross-cutting relationships. Here are the key edge types we'd want, ordered by implementation priority:

Priority 1   Import Resolution (IMPORTS) ▶

ImportDeclaration → SourceFile — "this import statement resolves to that file."

Representation: AG reference attribute resolvedModule on ImportDeclaration. The equation delegates to TypeScript's module resolution (already available via ts.Program.getResolvedModuleWithFailedLookupLocations). The result is a pointer to the target SourceFile node in the tree.

Enables: Module dependency graph, transitive import analysis, cross-file kind property verification, circular import detection, boundary enforcement ("no imports from outside this package").

Priority 1   Export Resolution (EXPORTS) ▶

SourceFile → exported symbols — "this file exports these declarations."

Representation: AG collection attribute exportedSymbols on SourceFile. Contributions come from ExportDeclaration, ExportAssignment, and exported variable/function/class declarations.

Enables: Public API surface analysis, unused export detection, kind contract checking at module boundaries.

Priority 2   Symbol Reference (REF) ▶

Identifier → Declaration — "this identifier refers to that declaration."

Representation: AG reference attribute resolvedDecl on Identifier. For within-file references, uses scope chain (already partially implemented in the binder). For cross-file references, follows the import resolution chain.

Enables: Find-all-references, rename analysis, dead code detection, usage tracking for kind-annotated symbols.

Priority 2   Package Dependency (DEPENDS_ON) ▶

Package → Package — "this package depends on that package."

Representation: AG synthesized attribute on Package that computes the actual package dependency graph from (1) declared dependencies in PackageDependency[] and (2) actual import resolution from the files within the package.

Enables: Package-level architectural rules, layer violation detection, dependency cycle detection across workspace packages.

Priority 3   Call Graph (CALLS) ▶

CallExpression → FunctionDeclaration | MethodDeclaration — "this call invokes that function."

Representation: AG reference attribute on CallExpression. Requires symbol resolution (Priority 2). Could initially be limited to static calls with known targets.

Enables: Purity analysis across function boundaries, side-effect tracking, dead function detection. Critical for deep kind property verification (e.g., "is this function truly pure given what it calls?").

Priority 4   Type Hierarchy (INHERITS_FROM) ▶

ClassDeclaration → ClassDeclaration | InterfaceDeclaration — "this class extends/implements that type."

Representation: AG reference attribute on HeritageClause entries. Resolves the type reference in extends/implements to the target declaration.

Enables: Type hierarchy queries, interface compliance checking, kind inheritance (if class A is Kind<{pure}>, are all subclasses also pure?).

Priority 4   Data Flow (REACHING_DEF) ▶

Definition site → use site — "the value assigned here flows to that usage."

Representation: This is a more fundamental change. Data flow analysis typically requires a CFG first, then computing reaching definitions or SSA form. Could be implemented as a separate AG overlay pass that produces a Map<Node, Node[]>.

Enables: Taint analysis, mutation tracking, value flow analysis for kind properties. The "nuclear option" for deep property verification.

7 AG Integration

How attribute grammars naturally extend from per-file analysis to whole-repository analysis.

Why Trees Are the Right Foundation

Attribute grammars evaluate on trees. The filesystem is a tree. The insight: a repository's directory structure is already a tree, and the per-file ASTs hang off its leaves. By extending the schema with container nodes, the AG library gains the ability to compute attributes at every level — from individual expressions up through files, directories, packages, and the repository root.

Concrete AG Spec Examples

Directory Metrics (trivial synthesized attributes)

// syn fileCount: number — how many .ts files in this directory (recursive)
// On Directory:
equation: (node) => {
  let count = node.files.length;
  for (const sub of node.subdirectories) {
    count += sub.attr('fileCount');
  }
  return count;
}

// syn totalLines: number — total lines of code
// On SourceFile:
equation: (node) => node.compilationUnit.lineStarts.length;
// On Directory: sum of children

Kind Property Propagation (up through containers)

// syn allViolations: Diagnostic[] — aggregate kind violations

// On SourceFile:
equation: (node) => node.compilationUnit.attr('allViolations');

// On Directory:
equation: (node) => [
  ...node.files.flatMap(f => f.attr('allViolations')),
  ...node.subdirectories.flatMap(d => d.attr('allViolations')),
];

// On Package:
equation: (node) => node.rootDir.attr('allViolations');

// On Repository:
equation: (node) =>
  node.packages.flatMap(p => p.attr('allViolations'));

// Result: repository.attr('allViolations') gives ALL violations
// across every file in every package — computed lazily!

Package Boundary Enforcement

// inh enclosingPackage: Package — inherited from Package ancestor
// On Package children: equation: (node, parent) => parent is Package ? parent : parent.attr('enclosingPackage')

// syn boundaryViolations: Diagnostic[] — on ImportDeclaration
equation: (node) => {
  const myPkg = node.attr('enclosingPackage');
  const target = node.attr('resolvedModule');
  if (!target) return [];

  const targetPkg = target.attr('enclosingPackage');
  if (targetPkg !== myPkg) {
    // Cross-package import! Check if it's declared in package.json
    const declared = myPkg.dependencies
      .some(d => d.name === targetPkg.name);
    if (!declared) {
      return [{ message: `Undeclared cross-package import: ${targetPkg.name}` }];
    }
  }
  return [];
}

Circular Import Detection (circular attribute)

// syn transitiveImports: Set<SourceFile> — on SourceFile
// This is a CIRCULAR attribute (mutual dependencies)
// Initial value: empty set. Iterates until stable.

circular({
  initial: () => new Set(),
  equation: (node) => {
    const result = new Set(node.attr('directImports'));
    for (const dep of node.attr('directImports')) {
      for (const transitive of dep.attr('transitiveImports')) {
        result.add(transitive);
      }
    }
    return result;
  },
  equals: (a, b) => setsEqual(a, b),
});

// syn hasCircularImport: boolean — on SourceFile
equation: (node) => node.attr('transitiveImports').has(node);

// This naturally handles mutual recursion via the circular attribute
// Fixed-point iteration (Magnusson-Hedin algorithm)
The AG advantage
Because AG evaluation is demand-driven and cached, adding container nodes and cross-file attributes doesn't force computation of anything that isn't needed. If you only query repository.attr('allViolations'), only the violation-related attributes are computed. The file-count and import-graph attributes are never evaluated unless someone asks for them.

8 Recommendation

A phased approach starting with Option E (Layered Schema + AG-Computed Edges).

Why Option E

Plays to KSC's strengths

The AG library is KSC's core differentiator. Expressing cross-file relationships as AG attributes means they're declarative, cached, demand-driven, and composable with existing binder/checker specs. A CPG (Option D) would require abandoning this for a generic graph query model.

Minimal disruption

Adding 8 new node kinds to a schema with 364 existing kinds is a 2% schema expansion. The existing per-file AST, binder, and checker continue to work unchanged. The new container nodes are purely additive.

Incremental path

Phase 1 (container nodes) is valuable on its own for directory-level metrics. Phase 2 (import resolution) enables the module graph. Phase 3 (cross-file kind checking) is the payoff. Each phase is independently useful.

Upgradable to CPG later

If you later need full CPG capabilities (CFG, data flow, taint analysis), the container nodes and cross-file attributes serve as a foundation. The tree + reference attribute model doesn't preclude adding an edge overlay (Option C) or graph store (Option D) later.

Implementation Phases

Phase 1: Container Nodes

Add Repository, Package, Directory, SourceFile, PackageDependency to ast-schema/schema.ts. Update convert.ts to build the directory tree from ts.Program.getSourceFiles() and filesystem structure. Add ContainerNode sum type. Run codegen. All existing tests still pass.

Phase 2: Cross-File Resolution

Create a new AG spec (module-resolver) that computes resolvedModule on ImportDeclaration nodes by delegating to TypeScript's module resolution. This requires passing the ts.Program or a resolution map to the AG spec as context. Compute directImports on SourceFile.

Phase 3: Cross-File Kind Checking

Extend the checker spec to verify kind contracts across module boundaries. If a function is annotated Kind<{pure}> and it calls an imported function, follow the import resolution to check whether the callee is also pure. This is the payoff — architectural enforcement at the module/package level.

Phase 4: Advanced Queries

Add transitiveImports (circular attribute), package dependency graph, boundary violation checking, and directory-level kind enforcement (e.g., "everything in src/core/ must be Kind<{noSideEffects}>").

What We Explicitly Defer

  • CFG / data flow / PDG — Not needed for kind property checking. Add later if we need taint analysis or deep purity verification.
  • Call graph — Useful for deep purity checking but complex. Can be approximated via import + export resolution.
  • Full CPG — Overkill for current use case. The layered approach gets us 80% of the value at 20% of the cost.
  • Git metadata — Blame, history, commit info. Valuable for tooling but not for semantic analysis.

9 Check Your Understanding

Test your understanding of the design space.

Q1: Why can't cross-file import resolution be represented as a normal child/parent relationship in the AST tree?

Correct! In a tree, every node has exactly one parent. If FileA imports FileB and FileC also imports FileB, FileB would need two parents — which violates the tree invariant. That's why cross-file references need a different mechanism: reference attributes, an edge overlay, or a general graph.

Q2: What CPG edge type represents "the value assigned at line 5 is used at line 12"?

Correct! REACHING_DEF edges connect definition sites to use sites, representing data flow. CFG edges represent control flow order (which statement executes after which), while AST edges represent syntactic containment.

Q3: In Roslyn's workspace model, what sits between Solution and Document?

Correct! Roslyn's hierarchy is Workspace → Solution → Project → Document → SyntaxTree. The Project corresponds to a .csproj file and is the compilation unit. It's analogous to the Package concept in our proposed schema.

Q4: Why is Option E preferred over a full CPG (Option D) for KSC?

Correct! KSC's core architecture is built on attribute grammars, which evaluate on trees. A full CPG is a general graph that would require a different evaluation model. Option E keeps the tree structure (so AG continues to work) while using reference attributes for cross-tree edges — getting most of the CPG's analysis power without abandoning the AG foundation.

Q5: What AG attribute kind is needed to compute transitive import closures when files can have mutual imports?

Correct! Mutual imports create cycles: A imports B, B imports A. Computing transitive closures across cycles requires circular attributes with fixed-point iteration (the Magnusson-Hedin algorithm). The attribute starts with an initial value (empty set) and iterates until stable.

Q6: How many new node kinds does Option E add to the existing 364-node schema?

Correct! Option E proposes 8 new node kinds. The 4 container nodes (Repository, Package, Directory, SourceFile) plus 4 supporting nodes (PackageDependency, RepositoryConfig, PackageConfig, ExternalModule). This is a ~2% expansion of the existing 364-node schema.
