Unifying Static & Dynamic Analysis

Static analysis tells you what might happen. Dynamic analysis tells you what did happen. The magic is in connecting them formally — so each makes the other stronger.

Part 1: The Bridge — Abstract Interpretation

Cousot & Cousot's framework (1977) gives us the mathematical bridge between static and dynamic analysis. The core idea: a Galois connection — a pair of functions that formally relate the world of concrete runtime values to the world of abstract static approximations.

Concrete Domain
All actual runtime values
{-7, 0, 3, 42, ...}
→ α (abstract)
← γ (concretize)
Abstract Domain
Simplified static model
{−, 0, +, ⊤}

α (abstraction) maps concrete values to abstract values: α({3, 42, 7}) = +. γ (concretization) maps back: γ(+) = {all positive numbers}. The key property: α(γ(a)) ≤ a and c ≤ γ(α(c)) — the abstract domain always over-approximates. If it says "positive," the real value is guaranteed positive. It might say "unknown" when the real value is positive (imprecise), but it never says "positive" when the real value is negative (unsound).

Runtime Verification: Monitors Against Formal Specs

Attach a monitor to a running program that checks every event against a formal specification (usually in temporal logic).

// Temporal logic specification (LTL):
// □(request → ◇response)
// "Every request is EVENTUALLY followed by a response"

// The monitor watches the execution trace:
// request, process, response, request, process, response  ← ✓ satisfies
// request, process, request, process, ...                 ← ✗ violates (no response!)

// JavaScript conceptual equivalent:
monitor("every .open() is followed by .close()", {
  on: "open",   goto: "waiting",
  on: "close",  goto: "satisfied",
  timeout: 5000, violation: "resource leak detected!"
});

JavaMOP / RV-Monitor

Monitor 150+ specifications simultaneously. Generate efficient monitoring code from temporal logic specs. Used in real JDK analysis.

Production-tested

TeSSLa / LOLA

Stream-based runtime verification. Specs process streams of events in real time. Used for embedded systems and IoT.

Real-time capable

For TypeScript: No equivalent exists today. A KindScript runtime monitor could watch for property violations (effect escapes, purity breaches, timing anomalies) using a lightweight event-based architecture.

Design by Contract: Preconditions, Postconditions, Invariants

Every function declares what it requires (precondition), what it guarantees (postcondition), and what must always be true (invariant). The contract is checked at runtime.

// Eiffel-style contracts (the original, Bertrand Meyer 1986):
class BankAccount {
  // Invariant: balance >= 0

  withdraw(amount: number) {
    // Precondition:  amount > 0 && amount <= this.balance
    // Postcondition: this.balance === old(this.balance) - amount
  }
}

// In TypeScript with Zod (approximation):
const WithdrawInput = z.object({
  amount: z.number().positive().max(balance)  // precondition
});

// Racket has the MOST sophisticated contracts:
// - Higher-order: contracts on functions that return functions
// - Blame tracking: when a contract fails, who's at fault?
// - Temporal contracts: sequences of interactions over time

Key insight — blame tracking (Findler & Felleisen, 2002): When a higher-order contract fails, the system identifies which party broke the contract. If f is given to g with a contract, and g calls f with a bad argument, the blame falls on g, not f. This is essential for modular reasoning.

Concolic Execution: Concrete + Symbolic

Run the program concretely but simultaneously track symbolic constraints on inputs. At each branch, solve constraints to generate inputs that take the other path.

// Concolic execution of:
function f(x: number, y: number) {
  if (x > 0) {              // branch 1
    if (y === x * 2) {       // branch 2
      throw new Error("bug"); // hard to reach!
    }
  }
}

// Step 1: Run concretely with x=0, y=0
//   Branch 1: 0 > 0 → false. Constraint: x ≤ 0

// Step 2: Negate constraint → x > 0. Solve → x=1, y=0
//   Branch 1: true. Branch 2: 0 === 2 → false. Constraint: y ≠ x*2

// Step 3: Negate → y = x*2. Solve → x=1, y=2
//   Branch 1: true. Branch 2: true → BUG FOUND!

// Three runs to find a bug that random testing might never hit.

SAGE (Microsoft) — Found ~1/3 of all Windows 7 security bugs

Systematic concolic testing of file parsers. Started from well-formed files, mutated symbolically to find crashes. Found bugs that blackbox fuzzing and static analysis both missed. Arguably the most impactful automated testing tool ever deployed.

Fuzzing & Property-Based Testing

Generate random (or guided) inputs, run the program, check if properties hold.

// Property-based testing with fast-check (JS/TS):
import fc from 'fast-check';

// "reverse of reverse is identity"
fc.assert(
  fc.property(fc.array(fc.integer()), (arr) => {
    const result = arr.slice().reverse().reverse();
    return JSON.stringify(result) === JSON.stringify(arr);
  })
);

// "sort is idempotent"
fc.assert(
  fc.property(fc.array(fc.integer()), (arr) => {
    const once = arr.slice().sort((a,b) => a-b);
    const twice = once.slice().sort((a,b) => a-b);
    return JSON.stringify(once) === JSON.stringify(twice);
  })
);

// If fast-check finds a counterexample, it SHRINKS it
// to the minimal failing case — invaluable for debugging.

Coverage-guided fuzzing (AFL++) instruments the program to track which code paths are reached. It then mutates inputs to maximize coverage, systematically exploring deeper and deeper branches. Property-based testing and fuzzing are essentially the same thing at different abstraction levels — PBT generates structured data, fuzzing generates raw bytes.

Dynamic Taint Analysis: Tracking Information Flow

Mark certain inputs as "tainted" (e.g., user input) and track how taint propagates through the program. Alert if tainted data reaches a sensitive sink (e.g., SQL query) without sanitization.

// Conceptual taint tracking:
let userInput = taint(req.query.name);  // mark as tainted
let greeting = "Hello, " + userInput;    // taint propagates
db.query(`SELECT * WHERE name='${greeting}'`); // ✗ TAINTED DATA IN SQL!

// After sanitization:
let safe = sanitize(userInput);          // taint removed
db.query(`SELECT * WHERE name='${safe}'`); // ✓ safe

Tools: TaintDroid (Android), Phosphor (JVM — 3 orders of magnitude faster than predecessors), NodeProf (JavaScript). For TypeScript: No production taint tracker exists, but taint analysis is exactly what a KindScript runtime component could provide — track effect propagation through function calls.

Dynamic Invariant Detection (Daikon)

Run the program on many inputs, observe the values at each program point, and propose likely invariants.

// Run sort() on 1000 random arrays. Daikon observes:
// At function entry:  arr.length >= 0
// At function exit:   result.length === arr.length
//                     result is sorted (∀i. result[i] ≤ result[i+1])
//                     result is a permutation of arr

// These are HYPOTHESES, not proofs!
// But they can be fed to a static verifier:
// "Hey LiquidHaskell, can you prove result.length === arr.length?"
// If yes → dynamic observation promoted to static proof.
// If no → maybe the invariant is wrong, or the prover isn't strong enough.

Daikon supports 40+ invariant templates (bounds, equality, ordering, containment, linear relationships) across C, C++, Java, and more. The invariants it discovers bridge the dynamic→static gap: observations become proof obligations.

Gradual Typing: Contracts at the Boundary

In Typed Racket, when typed code calls untyped code (or vice versa), the system automatically inserts contracts at the boundary:

;; Typed module:
(: add-positive (-> Positive-Integer Positive-Integer Positive-Integer))
(define (add-positive x y) (+ x y))

;; Untyped module calls add-positive:
(add-positive 5 -3)  ; ← RUNTIME CONTRACT ERROR!
; blame: the untyped module (it provided -3 for Positive-Integer)

;; The type annotation compiles to a runtime contract:
;; (-> positive? positive? positive?)
;; Checked at every cross-boundary call.

This is the closest existing system to a true static+dynamic bridge: the type system generates the runtime checks, and blame tracking identifies the faulty module. TypeScript erases types at runtime and provides no such bridge — this is a major gap that KindScript could fill.

Part 4: Designing a Unified System — KindScript as Example

Here's what a system that unifies static and dynamic analysis could look like, concretely, for a TypeScript analyzer:

The Architecture

📝
Layer 1: Static Analysis Engine
Parse → Type-check → Abstract interpret → Report
▼

The static engine analyzes TypeScript source without executing it:

// KindScript static analysis pipeline:

// 1. Parse TS to AST (use TypeScript compiler API)
const program = ts.createProgram(files, options);
const checker = program.getTypeChecker();

// 2. Type-level analysis (what TS already does)
//    - Type narrowing in branches
//    - Generic inference
//    - Strict null checks

// 3. Abstract interpretation (what KindScript adds)
//    - Effect tracking: does this function do I/O?
//    - Purity analysis: does it mutate anything?
//    - Termination estimation: likely to halt?
//    - Refinement inference: what value ranges flow here?

// 4. Output: property map per function
interface StaticReport {
  effects: Set<'io' | 'mutation' | 'exception' | 'async'>;
  purity: 'pure' | 'readonly' | 'impure';
  termination: 'proven' | 'likely' | 'unknown';
  refinements: Map<string, Refinement>;  // variable → value range
  confidence: number;  // 0.0 to 1.0
}

Over-approximates — never says "pure" when it isn't. May say "impure" when function is actually pure (false positive). This is sound.

▶️
Layer 2: Dynamic Analysis Engine
Instrument → Execute → Observe → Record
▼

The dynamic engine instruments and runs the code:

// KindScript dynamic analysis modes:

// MODE A: Contract insertion
// Generate runtime checks from static analysis results
function withdraw(amount: number) {
  // KindScript inserts: (from static refinement inference)
  __ksc_assert(amount > 0, "precondition: amount > 0");
  __ksc_assert(amount <= this.balance, "precondition: amount <= balance");
  // ... original code ...
  __ksc_assert(this.balance >= 0, "invariant: balance non-negative");
}

// MODE B: Property-based testing
// Generate tests that target unproven properties
fc.assert(fc.property(
  fc.integer({ min: 0, max: 1000 }),
  fc.integer({ min: 0, max: 1000 }),
  (balance, amount) => {
    // Test the property static analysis couldn't prove
    const result = withdraw.call({ balance }, amount);
    return result.balance >= 0;
  }
));

// MODE C: Runtime monitoring
// Watch for effect violations in running code
__ksc_monitor({
  target: myPureFunction,
  disallowed: ['console.log', 'fetch', 'fs.*', 'Math.random'],
  onViolation: (effect) => report(`Purity violation: ${effect}`)
});

// MODE D: Invariant detection (Daikon-style)
// Run on many inputs, detect patterns
__ksc_observe('sort', {
  precondition: (arr) => ({ length: arr.length }),
  postcondition: (arr, result) => ({
    length: result.length,
    sorted: isSorted(result),
    permutation: isPermutation(arr, result)
  })
});
// After 10,000 calls: "Likely invariant: result.length === arr.length"

Under-approximates — only observes executions that actually happen. Can't find bugs in untested paths. But what it finds is ground truth.

🔗
Layer 3: The Bridge — Unification Engine
Combine static over-approximation + dynamic under-approximation
▼

This is where the magic happens — the bridge combines both sides:

// The unification engine has FIVE strategies:

// STRATEGY 1: Dynamic confirms static
// Static says: "function f is pure" (confidence: high)
// Dynamic says: "ran f 10,000 times, no effects observed"
// Unified: "f is pure" (confidence: very high)

// STRATEGY 2: Dynamic refutes static
// Static says: "function g might be impure" (conservative)
// Dynamic says: "ran g 10,000 times, no effects observed"
// Unified: "g is LIKELY pure" — promote from 'unknown' to 'likely pure'

// STRATEGY 3: Dynamic finds what static missed
// Static says: "h is pure" (missed a code path)
// Dynamic says: "h called console.log on input X!"
// Unified: "h is IMPURE" — dynamic caught unsoundness in static

// STRATEGY 4: CEGAR refinement
// Static abstraction too coarse → false positive
// Dynamic counterexample shows it's spurious
// Refine static abstraction → re-analyze → try again

// STRATEGY 5: Gradual promotion
// Start: all properties are "unverified" (runtime checked)
// Phase 1: Dynamic testing builds evidence
// Phase 2: Static analysis proves some properties
// Phase 3: Proven properties drop runtime checks
// Phase 4: Remaining properties keep runtime guards

interface UnifiedReport {
  property: string;
  staticResult: 'proven' | 'likely' | 'unknown' | 'disproven';
  dynamicResult: 'confirmed' | 'no_violation' | 'violated';
  unified: 'verified' | 'high_confidence' | 'needs_review' | 'violated';
  runtimeCheckNeeded: boolean;  // false if statically proven
}

Phase 0: Runtime Only

// Just add runtime contracts (Zod-style validation)
// No static analysis yet — build the contract vocabulary

/** @ksc pure */
function add(a: number, b: number): number {
  return a + b;
}
// KindScript runtime: wraps `add` with a monitor
// that checks it doesn't perform effects.
// Violation? → report & log.

Start here. No complex infrastructure. Just runtime monitoring and contract checking. Build user trust and collect data about how programs actually behave.

Phase 1: Runtime + Observe (Daikon-style)

// Run the test suite with instrumentation ON
// Observe patterns, propose invariants

// After 10,000 test runs, KindScript observes:
// add: ∀ inputs → no effects, always returns, output = a + b
// sort: ∀ inputs → output.length === input.length, output is sorted
// fetchUser: calls fetch(), throws on 404

// Generate report:
// "Likely invariants discovered: 47
//  Functions likely pure: 108 of 127
//  Functions that always terminate: 125 of 127"

Now you have evidence. Not proofs, but strong empirical evidence. Users see which functions are probably pure, probably terminate, etc.

Phase 2: Infer + Runtime Fallback

// Static analysis infers properties from code structure
// Where it can't prove → keep runtime checks

function processOrder(order: Order): Receipt {
  // Static: infers no I/O, no mutation → PURE ✓ (proven)
  // Static: infers terminates (no loops/recursion) → HALTS ✓ (proven)
  // → No runtime checks needed!

  const tax = order.total * 0.08;
  return { ...order, tax, finalTotal: order.total + tax };
}

function complexCalc(data: Data[]): Result {
  // Static: can't prove termination (has while loop)
  // Dynamic: never seen to loop > 1000 iterations
  // → Runtime guard inserted:
  __ksc_guard_iterations(1000, "complexCalc loop");
  // → Property: "likely terminates" (confidence: 82%)
}

The hybrid: static proves what it can, dynamic guards the rest. Gradually, as the static engine improves, more properties get promoted from "runtime-checked" to "statically proven."

Phase 3: Prove + Fallback

// CEGAR-style refinement loop active
// Static analysis tries to prove properties
// When it produces false positives:
//   1. Dynamic testing shows the alarm is spurious
//   2. Abstraction is refined (more precise domain)
//   3. Re-analyze with refined abstraction

// Result: most properties are now statically proven
// Remaining unknowns have runtime guards

// Dashboard: Purity 95% proven, Termination 78% proven
// The 5% and 22% still have runtime monitors

Phase 4: Mostly Static

// With refinement types + effect types + termination analysis:
// Most properties are statically verified
// Runtime checks only at system boundaries (external input, FFI)

// The dream:
// "127 functions analyzed
//  121 fully verified statically (95%)
//    4 verified with runtime evidence (3%)
//    2 require runtime guards (2%)
//
//  All array accesses: proven safe
//  All divisions: proven safe (non-zero divisors)
//  All functions: proven pure or effects documented
//  Termination: 125/127 proven (2 require timeout guards)"

Part 5: Concrete Implementation Options for KindScript

What can you actually build today?

Static: TypeScript Compiler API

Use ts.createProgram() to parse and type-check. Walk the AST, track effects through the call graph, infer refinements from guards and narrowing. This is your abstract interpreter.

Available now Core of KindScript

Static: z3-solver (npm)

Z3 compiled to WASM, usable from Node.js. Send verification conditions from your abstract interpreter to Z3 for SMT solving. Enables refinement type checking.

Available now Complex integration

Dynamic: fast-check

Property-based testing for JS/TS. Generate random inputs, test invariants. Use KindScript's static analysis to generate the properties to test.

Production-ready 13k+ downloads/week

Dynamic: Proxy-based effect monitoring

Wrap functions in ES Proxies that intercept all property access, function calls, and mutations. Detect impurity, effect leakage, and contract violations at runtime.

Available now Performance overhead

Bridge: Zod as contract language

Use Zod schemas as the shared specification language. Static side: infer Zod schemas from code. Dynamic side: enforce Zod schemas at boundaries. Same spec, dual enforcement — like CodeContracts for TypeScript.

Available now Ecosystem fit

Bridge: Effect-TS as effect system

Effect-TS already tracks effects in the type system (Effect<A, E, R>). KindScript could analyze Effect-TS code with much higher precision — the effect types ARE the specification.

Available now Natural fit

Check Your Understanding

Q1: What does the Galois connection in abstract interpretation guarantee?

That the static analysis finds all bugs
That if the abstraction says "safe," every concrete execution is safe (soundness)
That the concrete and abstract domains have the same number of elements
That the analysis runs in polynomial time

Q2: What does the CEGAR loop do when it finds a spurious counterexample?

Reports it as a bug anyway (conservative)
Gives up and says "unknown"
Refines the abstraction to be more precise, then re-verifies
Switches to dynamic analysis exclusively

Q3: Why is "gradual verification" a good model for KindScript?

Because TypeScript is already gradually typed
Because it's faster than static analysis
Because you can start with runtime checks everywhere and progressively replace them with static proofs, without requiring everything upfront
Because it doesn't require any static analysis
