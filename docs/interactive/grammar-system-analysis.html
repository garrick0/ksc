1. KSC Grammar Architecture

KSC Grammar Pipeline (Functor 1) ┌─────────────────────────┐ ┌──────────────────────────────────────┐ │ specs/ts-ast/grammar/ │ │ grammar/ (fully generic machinery) │ │ │ │ │ │ nodes.ts ────────┐ │ │ builder.ts types.ts │ │ extractors.ts │ │ │ │ │ │ │ convert-helpers.ts│ │ │ ▼ ▼ │ │ convert-generator │ │ │ createGrammarBuilder() │ │ verify.ts │ │ │ │ │ │ │ │ │ ▼ │ │ spec.ts ─────────┤ │ │ GrammarSpec │ │ (factory fn) │ │ │ { nodes, sumTypes, │ │ │ │ │ convertGenerator? } │ └────────────────────┼────┘ │ │ │ │ │ ▼ │ └───────────▶│ compileGrammar() │ │ │ │ │ ▼ │ │ CompiledGrammar │ │ { files[], kinds, hasKind } │ └──────────────────────────────────────┘ │ ▼ ┌──────────────────────────┐ │ generated/ts-ast/grammar/ │ │ node-types.ts │ │ schema.ts │ │ builders.ts │ │ serialize.ts │ │ kind-map.ts │ │ index.ts │ │ convert.ts (from spec) │ └──────────────────────────┘

The grammar system is the first functor in KSC's two-functor architecture. It takes a GrammarSpec (nodes + sum types + optional convert generator) and produces generated TypeScript files that define the AST infrastructure.

▶ Key Design Principle: Spec/Machinery Separation Strength

The grammar/ directory is fully generic — it contains zero TypeScript-specific code. All language-specific data lives in specs/. This means adding a new grammar target (e.g., Python AST) requires zero changes to the machinery.

  • Machinery (grammar/): Builder DSL, compilation, shared codegen helpers, field extractor assembly, extractor DSL
  • Spec data (specs/ts-ast/grammar/): Node definitions, extractors, convert helpers, convert generator, verification

The convert.ts generation is elegantly handled via the closure pattern: the spec's convertGenerator is a () => string function that closes over all convert-specific data. compileGrammar() just calls it without knowing what's inside.

3. System Comparison

3.1 Grammar Specification Models

KSC: Builder DSL (imperative, TypeScript)

// specs/ts-ast/grammar/nodes.ts
const Expression = b.sumType('Expression');
const Statement  = b.sumType('Statement');

b.node('IfStatement', [Statement], {
  expression: child('Expression'),
  thenStatement: child('Statement'),
  elseStatement: optChild('Statement'),
});

b.node('Identifier', [Expression], {
  escapedText: prop('string'),
  symIsVariable: prop('boolean'),
});

b.leaf('TrueKeyword');

Character: Imperative builder pattern. Sum types are explicit, not via inheritance. Fields use tagged descriptors (child, optChild, list, prop). TypeScript provides compile-time type safety.

JastAdd: .ast file (declarative, custom DSL)

// Grammar.ast
abstract Stmt;
abstract Expr;

IfStmt : Stmt ::= Condition:Expr
                   Then:Stmt
                   [Else:Stmt];

IdUse : Expr ::= <ID:String>;

Character: True class hierarchy with single inheritance. Children declared inline with ::=. Optional children use [], lists use *, tokens use <>. Generates Java classes with getters/setters.

Silver: Nonterminals + Productions (declarative, Silver language)

nonterminal Stmt;
nonterminal Expr;

abstract production ifStmt
top::Stmt ::= cond::Expr  th::Stmt  el::MaybeStmt
{
  top.pp = "if (" ++ cond.pp ++ ") " ++ th.pp
        ++ " else " ++ el.pp;
}

abstract production idExpr
top::Expr ::= name::String
{ ... }

Character: Nonterminals are declared separately from their productions. Each production is a distinct entity with a name. Attribute equations are written inside productions, co-located with the grammar rule. Silver is a full language with its own type system.

Facebook Infer: OCaml algebraic types (no metaprogramming)

(* infer/src/IR/Sil.ml *)
type instr =
  | Load of { id: Ident.t; e: Exp.t; typ: Typ.t }
  | Store of { e1: Exp.t; typ: Typ.t; e2: Exp.t }
  | Prune of Exp.t * Location.t * bool * if_kind
  | Call of Ident.t * Exp.t * (Exp.t * Typ.t) list * Location.t
  | Metadata of instr_metadata

Character: No code generation at all for the IR. The AST/IR is defined directly as OCaml algebraic data types. Frontends compile source languages into this IR. The IR is deliberately minimal (5 instruction types) and CFG-based, not tree-based.

3.3 Architectural Paradigm Comparison

KSC — Two-Functor Specializer

KSC is a code generator that produces specialized evaluators from declarative specs. The grammar functor generates AST infrastructure; the analysis functor generates the attribute evaluator. Both are pure functions: Spec → GeneratedFiles.

Closest to: a staged metaprogramming system or partial evaluator

JastAdd — Metacompiler

JastAdd is a metacompiler that weaves attribute declarations into Java AST classes. Input: .ast + .jrag files. Output: Java classes with lazy attribute methods. The paradigm is aspect-oriented — attributes from different concerns compose via static weaving.

Closest to: AspectJ but for AGs

Silver — AG Language

Silver is a full programming language for specifying attribute grammars. It has its own type system, module system, and composition verification. The key innovation is forwarding for modular language extensibility. Self-hosted.

Closest to: a domain-specific language for compilers

Infer — Abstract Interpretation Framework

Infer is fundamentally different: it's not an AG system but an abstract interpretation framework. It defines analyses as abstract domains + transfer functions over a flat CFG-based IR, not tree-structured attribute equations.

Closest to: a pluggable static analysis framework

4. Deep Dives — Key Design Patterns

▶ 4.1 The Closure Pattern for Frontend Ownership Elegant

KSC's approach to frontend code generation is novel and well-designed. The convertGenerator field on GrammarSpec is a () => string closure that the spec creates in spec.ts:

// specs/ts-ast/grammar/spec.ts
return {
  nodes, sumTypes,
  convertGenerator: () => generateTsConvert({
    nodes, sumTypes, fieldExtractors,
    skipConvert, syntaxKindOverrides, jsDocMembers,
  }),
};

This elegantly keeps compileGrammar() ignorant of convert-specific data while letting the spec own all frontend logic. Compare with the alternatives:

  • JastAdd: User writes parser separately — no integration point
  • Silver: Concrete productions + Copper — tightly coupled
  • Infer: Separate frontend executables per language

KSC's closure pattern sits in a sweet spot: the machinery provides shared infrastructure (convert-skeleton.ts, convert-codegen.ts), while the spec customizes everything language-specific.

▶ 4.2 Sum Types vs Inheritance vs Nonterminals Design Tradeoff

These three systems model type hierarchies very differently:

SystemMechanismField InheritanceOpen/Closed
KSC sumType() + sumTypeIncludes() addFieldToSumTypeMembers() (manual) Closed (resolved at build time)
JastAdd Single inheritance (B : A ::= ...) Automatic (Java class inheritance) Closed + open via aspects
Silver Nonterminal = union of its productions Nonterminal carries attributes, productions define equations Open (new productions via extensions)

KSC's sum types are essentially tagged unions — they group node kinds without providing inheritance. The addFieldToSumTypeMembers() method is a workaround for the lack of field inheritance, adding the same field to every member of a sum type.

JastAdd advantage: If Expr has a type() attribute, all subclasses automatically inherit it. In KSC, you'd need to call addFieldToSumTypeMembers('Expression', 'typeString', prop('string')).

▶ 4.3 Field Extraction — String Expressions vs Type-Safe Code Tension

KSC's converter generation uses string-based expressions for field extraction. The extractor DSL provides compile-time validation of function names:

// Type-safe at the reference level...
const sfn = statefulCallBuilders(STATEFUL_FN_NAMES);
sfn.hasSymFlag('node', 'ts.SymbolFlags.Variable')
// ...produces a string: "hasSymFlag(_ctx, node, ts.SymbolFlags.Variable)"

This is a layered validation approach:

  1. Compile time: satisfies readonly (keyof typeof Helpers)[] ensures function names match exports
  2. Codegen time: validateFieldExpressions() checks that all function-call references resolve
  3. Runtime: The generated TypeScript compiler catches remaining type errors

Compare with JastAdd where equations are Java code compiled directly, or Silver where equations are type-checked Silver code. KSC's approach trades some safety for flexibility in expression construction.

▶ 4.4 Evaluation: Batch vs Demand-Driven Fundamental Difference

This is perhaps the deepest architectural difference between KSC and classical AG systems:

KSC: Batch Evaluator

The generated evaluator walks the entire tree and evaluates all attributes in a single pass (or multiple fixed passes). Every node gets every attribute computed, whether needed or not.

Advantage: Simple, predictable, easy to generate

Disadvantage: May compute unnecessary attributes; harder to support circular deps

JastAdd/Silver: Demand-Driven

Attributes are evaluated only when accessed. node.attr() triggers computation, which may recursively evaluate other attributes. Results are cached. No scheduling needed.

Advantage: Zero cost for unused attributes; natural support for circular evaluation

Disadvantage: Harder to reason about performance; cache memory overhead

KSC's batch approach works well for its current use case (kind-checking analysis) but would struggle with features like circular attributes or incremental re-evaluation that demand-driven systems handle naturally.

▶ 4.5 Infer's Two-Phase Architecture vs KSC's Two Functors Structural Parallel

Despite being fundamentally different systems, there's a structural parallel:

KSCInfer
Phase 1 Functor 1: Grammar → AST infrastructure Capture phase: Source → SIL IR
Phase 2 Functor 2: Analysis → Evaluator Analysis phase: Checkers over SIL
Spec interface GrammarSpec / AnalysisSpec AbstractDomain.S / TransferFunctions
Pluggability New specs in specs/ New checkers in absint/

Both achieve analysis-language independence through a common intermediate representation. But KSC generates code at build time while Infer runs analyses at analysis time over a shared runtime IR.

5. Issues & Improvement Opportunities

Issue 1: grammar/export.ts breaks genericity

export.ts imports directly from generated/ts-ast/grammar/, making it TypeScript-specific. This violates the core principle that grammar/ is fully generic.

// grammar/export.ts — line 8-10
import type { KSTree } from '../generated/ts-ast/grammar/convert.js';
import type { KSNode, ... } from '../generated/ts-ast/grammar/node-types.js';
import { getChildFields, fieldDefs, ... } from '../generated/ts-ast/grammar/schema.js';

Options:

  • Move to specs/ts-ast/grammar/ — it's TS-specific data extraction, belongs with the spec
  • Parameterize it — accept schema/node types as arguments so it stays generic
  • Move to app/ — it's a composition-level concern, not machinery
Issue 2: grammar/convert-skeleton.ts is TS-frontend specific despite living in generic machinery

The skeleton generates code that references ts.Node, ts.SourceFile, ts.SyntaxKind, ts.forEachChild() — these are all TypeScript compiler API types. The file's own comment says "target the TypeScript compiler API." Despite not importing the TS module, it generates TS-specific string templates.

Options:

  • Move to specs/ts-ast/grammar/ — acknowledge it's TS-specific infrastructure
  • Abstract the skeleton — introduce a FrontendSkeleton interface that specs implement (more work, may be premature)
  • Leave it if the mock grammar also uses it — but document the TS assumption clearly
Issue 3: No field inheritance — addFieldToSumTypeMembers is a workaround

In JastAdd, if Expr declares type(), all concrete expression nodes inherit it automatically. In KSC, you must manually call b.addFieldToSumTypeMembers('Expression', 'typeString', prop('string')) after all nodes are registered. This is:

  • Error-prone (easy to forget a new sum type member)
  • Order-dependent (must be called after all nodes that should receive the field)
  • Not checked at the type level

JastAdd-aligned option: Add fields to SumTypeEntry — fields declared on a sum type would be automatically applied to all members during resolveIncludes().

Issue 4: sumTypeIncludes resolution is shallow (single level)

resolveIncludes() only resolves one level of includes. If A includes B which includes C, members of C won't propagate to A unless B's includes have already been resolved. The current implementation iterates sum types in map order, which may not reflect the dependency order.

// grammar/builder.ts — resolveIncludes()
for (const [, st] of sumTypeRegistry) {
  for (const incName of st.includes) {
    const incSt = sumTypeRegistry.get(incName);
    // Only copies current members — transitive includes may be incomplete
    for (const member of incSt.members) { ... }
  }
}

Fix: Topological sort the sum type dependency graph, or iterate to fixed point.

Issue 5: CompilationUnit special-casing in generated code

Several generated files have hardcoded CompilationUnit references:

  • generateSchema(): if (kind === 'CompilationUnit') lines.push("'CompilationUnit': ['children']")
  • generateSerializer(): treeFromJSON special-cases Program

This couples the generic machinery to the TS AST grammar's specific root node names. A truly generic system would derive root/file handling from the spec (e.g., via grammarConfig.rootKind).

Issue 6: String-based code generation without AST

All generated code is built via string concatenation (lines.push()). This makes the generators fragile — indentation errors, missing semicolons, and syntax mistakes are only caught when the generated file is compiled. Compare:

  • JastAdd: Generates Java AST, then pretty-prints
  • Silver: Self-hosted, uses its own tree representation
  • Better approach: A lightweight TypeScript AST builder (or template literal system) that guarantees syntactic correctness

However, for KSC's current scale (7 files, ~2000 generated lines total), string concatenation is adequate.

Issue 7: No demand-driven / lazy evaluation option

Both JastAdd and Silver achieve significant performance benefits from demand-driven evaluation — attributes are only computed when accessed. This also naturally enables:

  • Circular attribute evaluation (Magnusson-Hedin)
  • Incremental re-evaluation when the tree changes
  • Zero cost for unused attributes

KSC's batch evaluator is simpler but limits future capability. This is an architectural boundary that would be hard to change later.

Not necessarily a problem now — batch evaluation is fine for the current kind-checking analysis. But worth noting for future roadmap.

Issue 8: No aspect-oriented attribute composition

In JastAdd, attributes from different concerns (name analysis, type checking, code generation) live in separate .jrag files that are woven together. In Silver, aspect production allows adding equations to existing productions without modifying them.

KSC requires all attributes to be in a single AnalysisSpec. For large analyses with many attributes, this could become unwieldy. The current workaround is to have separate analysis specs per concern, but these can't share attribute definitions.

Issue 9: No NTA (Non-Terminal Attribute) support

JastAdd's NTAs allow computed subtrees — attributes whose values are new AST nodes that can themselves have attributes. This is powerful for desugaring, inserting implicit nodes, or creating canonical representations.

KSC's attribute types are limited to syn, inh, and collection with primitive/aggregate values. Computed subtrees would require a different evaluation model.

Issue 10: Builder API is imperative where other systems are declarative

KSC's builder is an imperative API (b.node(), b.leaf(), b.sumType()) called in sequence. JastAdd and Silver use declarative specifications that can be statically analyzed. The imperative approach means:

  • Order matters (e.g., resolveIncludes() must be called after all nodes)
  • No static analysis of the grammar specification itself
  • But: full TypeScript expressiveness for programmatic grammar construction
Strength: Type-safe extractor DSL

The statefulCallBuilders/pureCallBuilders/mapLookupBuilders pattern with satisfies validation is a creative solution for type-safe string template construction. This gives near-compile-time safety for extractor expressions — a problem that other systems don't even attempt to solve because they use their own type systems.

Strength: Comprehensive generated serialization

KSC automatically generates schema-aware JSON serialization for all 364 node kinds. Neither JastAdd nor Silver provide built-in serialization — users must implement it manually. Infer uses a separate Textual format. KSC's approach is production-ready out of the box.

7. Knowledge Check

Q: What makes KSC's grammar/ directory "fully generic"?
Click to reveal answer
It contains zero imports from specs/, generated/, or any TypeScript compiler API. All language-specific data comes through the GrammarSpec interface. Exception: export.ts breaks this rule by importing from generated/ts-ast/, and convert-skeleton.ts generates TS-specific string templates.
Q: How does JastAdd handle inherited attributes differently from KSC?
Click to reveal answer
In JastAdd, inherited attribute equations are written in the parent node's .jrag file: eq Parent.getChild().attr() = expr;. The equation specifies which child it applies to. Broadcasting propagates the value to all descendants. In KSC, inherited attributes use parentEquations (a Record mapping parent kind → function) and copy-down is the default (returning undefined).
Q: What is Silver's "forwarding" mechanism and why doesn't KSC need it?
Click to reveal answer
Forwarding lets a production delegate to a semantically equivalent tree for attribute values it doesn't explicitly define. It enables modular language extensions — new syntax automatically gets semantics via forwarding. KSC doesn't need forwarding because it doesn't support extensible grammars — its grammar is fixed at build time for a specific target (e.g., TypeScript AST), not designed for third-party extension.
Q: How does Infer's abstract domain approach differ from KSC's attribute grammar approach?
Click to reveal answer
Infer defines analyses as abstract domains (leq, join, widen) with transfer functions (exec_instr) over a flat CFG. It computes procedure summaries independently. KSC defines analyses as attribute equations on a tree structure. Infer works on control flow; KSC works on syntax trees. Infer is inherently interprocedural (via summaries); KSC is intraprocedural (per-file tree evaluation).
Q: Why does KSC use string-based code generation instead of a proper AST?
Click to reveal answer
Pragmatic simplicity. The generated output is straightforward TypeScript — interfaces, switch statements, factory functions. A template string approach (lines.push()) is easy to understand, debug, and modify. At the current scale (~2000 lines of generated code across 7 files), the benefits of an intermediate AST representation don't justify the complexity. JastAdd and Silver both generate much more complex code (woven Java classes with caching, circularity detection, etc.) where a proper IR is more justified.
Q: What's the key difference between KSC's two functors and Infer's two phases?
Click to reveal answer
KSC's functors operate at build time — they generate source code that is then compiled. Infer's phases operate at analysis time — capture translates source to SIL on each run, then analysis processes SIL. KSC's approach means the generated evaluator is specialized and fast; Infer's approach means new analyses can be added without regenerating anything, and summaries enable incremental analysis across runs.
