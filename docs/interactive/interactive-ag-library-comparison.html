Attribute Grammar Systems Compared

A deep comparison of @ksc/ag (our TypeScript AG library, JastAdd-aligned) with JastAdd, Silver, and Kiama — nomenclature, semantics, implementation, and design tradeoffs.

1System Overview

Four AG systems, four approaches to the same core idea: decorating trees with computed properties.

@ksc/ag

Embedded TS Library (JastAdd-Aligned)

Runtime library using Object.defineProperty to stamp lazy getters directly onto AST nodes. Demand-driven. No code generation. API surface strictly aligned to JastAdd primitives: syn, inh, circular, match. Implements the Magnusson-Hedin algorithm for circular attributes. Formalized AGSpec type for composable analysis specifications.

JastAdd

Java Code Generator

Specifications written in .jrag/.jadd files compiled into Java classes. Attributes become methods on AST node classes. Inter-type declarations weave code into generated ASTs. Used in production compilers (ExtendJ, JModelica).

Silver

Standalone AG Language

Full language with its own type system, compiled to Java. Emphasizes modular, extensible grammars via forwarding and aspect productions. Developed by the MELT group (U. of Minnesota). Used for ableC, ableJ.

Kiama

Embedded Scala Library

Pure Scala embedding using pattern matching and higher-order functions. Attributes are Scala functions wrapped in caching combinators. Also provides strategy-based term rewriting. Kiama inspired @ksc/ag's early design, but @ksc/ag has since aligned with JastAdd semantics.

Design philosophy: @ksc/ag accepts the "library ceiling" — the subset of AG features expressible without code generation. Within that ceiling, it aligns strictly to JastAdd nomenclature and semantics. Non-JastAdd concepts from Kiama (like chain, coll, lookup) have been removed in favor of composing core primitives: syn, inh, circular, match.
What Changed: The JastAdd Alignment

Earlier versions of @ksc/ag included Kiama-inspired convenience combinators:

  • chain(init, update) — threaded inh+syn pair (Kiama/Silver's chain/threaded). Removed: encode manually with paired syn+inh attributes.
  • coll(init, combine, contrib) — subtree aggregation. Removed: JastAdd's coll has different directed-contribution semantics; subtree aggregation is achieved via syn with DFS.
  • lookup(collect) — global name resolution shortcut. Removed: name resolution is a composition pattern using syn + inh + match, exactly how ExtendJ handles it.

This aligns the API surface strictly with JastAdd's core primitives, avoiding semantic confusion between JastAdd's coll (directed contributions to remote nodes) and the old Kiama-style subtree fold.

2Synthesized Attributes bottom-up

Computed from a node and its children, propagating information upward. The most fundamental attribute type.

Uses syn() factory — returns an AttributeDef that installs a lazy getter via Object.defineProperty. First access computes and replaces the getter with a data property (zero overhead thereafter).

// Cached synthesized attribute
const attrs = {
  locmin: syn((node: Tree) => {
    if (node.type === 'Leaf') return node.value;
    return Math.min(node.left.locmin, node.right.locmin);
  }),
};

// Uncached variant (recomputed every access)
  timestamp: uncached((node) => Date.now()),

// Parameterized (memoized by param)
  lookup: paramSyn((node, key: string) => env.get(key)),

Caching: syn = cached (getter → data property). uncached = no cache (persistent getter). paramSyn = cached by (node, param) pair via internal Map.

Declared with syn keyword in .jrag files. Equations can be provided inline or per-production. Subclasses can override equations.

// Declaration with inline equation
syn int Expr.value() = 0;

// Per-production equations (subclass overrides)
eq Add.value() = getLeft().value() + getRight().value();
eq Literal.value() = getNumber();

// Caching control
syn lazy Type Expr.type();    // cached (default for syn)
uncache Expr.value();          // force recompute

Caching: All syn attributes are cached by default (lazy is the default). Use uncache to disable. Under the hood: _computed boolean + _value field per attribute per node.

Declared with synthesized attribute. Must also declare which nonterminals the attribute occurs on. Equations given in productions.

-- Attribute declaration
synthesized attribute pp :: String;
attribute pp occurs on Expr;

-- Equation in a production
concrete production add_c
sum::Expr ::= e::Expr '+' t::Term
{
  sum.pp = e.pp ++ " + " ++ t.pp;
}

-- Shorthand: declare + occurs on together
synthesized attribute pp :: String occurs on Expr, Stmt;

Caching: Silver compiles to Java with demand-driven caching. Silver also supports collection attributes via the with clause: synthesized attribute errors :: [Message] with ++;

Attributes are Scala Attr[T] values — partial functions wrapped in a caching combinator. Access via node->attr syntax.

// Cached attribute (default)
val locmin: RepTree => Int = attr {
  case Leaf(v) => v
  case Fork(l, r) => math.min(l->locmin, r->locmin)
}

// Uncached (dynamic attribute)
val timestamp: Tree => Long = dynAttr {
  case _ => System.currentTimeMillis()
}

// Parameterized attribute
val lookup: Tree => String => Option[Def] = paramAttr {
  key => { case n => n.env.get(key) }
}

Caching: attr = cached (memoized). dynAttr = uncached. paramAttr = parameterized+cached. Uses internal memo tables (not on-node).

Nomenclature alignment: All four systems use essentially the same term — "synthesized" — though @ksc/ag abbreviates to syn() matching JastAdd's syn keyword. Kiama uses attr as the general cached-attribute combinator.

3Inherited Attributes top-down

Pass context information downward from parents to children. The dual of synthesized attributes.

Uses inh(rootValue, eq?) — parent-defined, matching JastAdd's inh keyword and Silver's inherited semantics. The parent's equation decides what value each child receives.

// Parent provides depth+1 to each child
depth: inh(
  0,                                // root value
  (parent) => parent.depth + 1,    // parent equation
),

// Scope nodes provide env to children; others auto-propagate
env: inh(
  'global',                         // root value
  (parent) => {                     // parent equation
    if (parent.kind === 'scope') return parent.localEnv;
    return undefined;               // auto-propagate (like Silver's propagate)
  }
),

// Broadcast from root (no eq = propagate everywhere)
globalMin: inh((root) => root.locmin),

inh(rootValue, eq?): Root gets rootValue (or rootValue(root) if a function). For non-root nodes, eq(parent, child, childIndex) provides the value. Return undefined to auto-propagate (child inherits parent's own value — equivalent to Silver's propagate). Omit eq entirely to broadcast root value to all descendants.

Declared with inh keyword. Equations are written on the parent, defining what value each child receives.

// Declaration
inh Env Expr.env();

// Equation on PARENT (defines child's inherited attr)
eq Block.getStmt(int index).env() = localEnv();

// Can differentiate by child index
eq IfStmt.getThen().env() = extendEnv(condition());
eq IfStmt.getElse().env() = env();  // pass through

// Broadcast to all children
eq Program.getChild().env() = globalEnv();

Same pattern: Both JastAdd and @ksc/ag use parent-defined equations. JastAdd writes eq Block.getStmt().env() = ...; @ksc/ag writes inh(rootVal, (parent) => ...). The parent decides what value each child receives.

Declared with inherited attribute. Equations assign to children within productions. Silver also has autocopy for automatic propagation.

-- Declaration
inherited attribute env :: Env;
attribute env occurs on Expr, Stmt;

-- Equation in production (assigns to children)
production block
b::Block ::= s::Stmt
{
  s.env = b.localEnv;
}

-- Autocopy: automatically pass through unchanged
propagate env on Expr, Stmt;

Autocopy / propagate: Silver's propagate automatically copies an inherited attribute to all children that need it. @ksc/ag's inh() achieves the same: returning undefined from the equation auto-propagates the parent's own value.

Kiama provides inh, down, and atRoot combinators. These use Scala pattern matching and walk upward from the child to find the inherited value.

// inh combinator (child walks up to parent)
val env: Tree => Env = inh {
  case tree.parent(p: Block) => p->localEnv
}

// down: like inh but with default
val depth: Tree => Int = down(0) {
  case tree.parent(p) => (p->depth) + 1
}

// atRoot: compute once at root
val globalMin: Tree => Int = atRoot {
  case r => r->locmin
}

@ksc/ag's inh() is parent-defined (closer to JastAdd/Silver) rather than child-walking (Kiama-style). Both achieve the same results, but parent-defined is the classical AG approach.

Deep Dive: Parent-Defined Inherited Attributes Across Systems

All four systems now follow the parent-defined model (though Kiama also offers child-walking variants):

@ksc/ag — inh(rootValue, eq?)

Parent equation returns a value or undefined (auto-propagate). Root gets rootValue. The equation receives (parent, child, childIndex), matching JastAdd's indexed child access pattern. Auto-propagation (return undefined) is equivalent to Silver's propagate.

JastAdd — eq Parent.getChild().attr()

Equations on parent AST classes. Each parent class defines what value its children get. Missing equations are compile-time errors. Can differentiate by child index (getStmt(int i)).

Silver — child.attr = expr in productions

Equations assign to children within production bodies. propagate auto-copies unchanged. The MWDA statically verifies completeness across modular grammars.

Kiama — inh { case ... }

Child-walking: pattern matches on tree.parent(p) to find inherited value. Also provides down(default) and atRoot combinators. More dynamic than the parent-defined approach.

4Circular Attributes fixed-point

Handle mutually recursive attribute definitions via iterative fixed-point computation.

Uses circular(init, compute, options?). Implements the Magnusson-Hedin algorithm — all circular attributes encountered during a cycle are tracked and re-evaluated together until global convergence.

// Fixed-point: starts at empty set, iterates until stable
liveVars: circular(
  new Set(),
  (node) => computeLiveVars(node),
  { equals: (a, b) => setsEqual(a, b) }
),

// Simple counter: starts at 0, converges at 5
counter: circular(0, (node) => {
  const current = node.counter;
  return current < 5 ? current + 1 : current;
}),

Algorithm (Magnusson-Hedin): When a circular attribute is first accessed, it becomes the "cycle driver" and initializes a cycleEntries array. During evaluation, other circular attributes accessed from equations register themselves in this array. All cycle members are iterated together in a do...while(changed) loop until no value changes across the entire cycle. After convergence, all members are stamped as plain data properties (zero-overhead subsequent access). Uses module-level inCycle/changed flags and per-node Symbol('ag:circular') state.

Uses circular modifier with a bottom value. The original Magnusson-Hedin fixed-point algorithm.

// Circular attribute with bottom value
syn Set<Variable> Stmt.liveAfter()
  circular [new HashSet<>()] = computeLive();

// The bottom value (empty set) is provided in brackets
// The equation can reference itself circularly
eq Assign.liveAfter() {
  Set<Variable> s = new HashSet<>(successor().liveAfter());
  s.remove(getDef());
  s.addAll(getUses());
  return s;
}

Algorithm: Magnusson-Hedin algorithm. Maintains change tracking across the entire AG, iterating until global convergence. @ksc/ag now implements this same algorithm — the cycle driver tracks all circular attributes encountered during evaluation and iterates them together.

Silver supports circular attributes via fixed-point iteration. Used for classic analyses like computing nullable, FIRST, and FOLLOW sets.

-- Silver supports circular evaluation
-- The evaluator iterates from a bottom value until convergence
-- Classic example: nullable/FIRST/FOLLOW sets for grammars

-- Silver compiles to Java with lazy evaluation semantics
-- and can leverage the same fixed-point iteration
-- algorithm used by JastAdd

Silver treats circular attributes as "changing the fixed-point operation used in the attribute evaluator." The evaluation uses successive approximation on strongly connected components of the dependency graph.

Kiama uses circular combinator with initial value. Uses a simpler per-attribute iteration loop.

// Circular attribute with initial value
val liveIn: Stmt => Set[Var] = circular(Set.empty) {
  case s: Assign =>
    (s.succ->liveIn) -- s.defs ++ s.uses
  case s =>
    s.succs.flatMap(_->liveIn).toSet
}

Algorithm: Module-level inCircle/changed flags. Simpler per-attribute loop. @ksc/ag's implementation goes beyond Kiama by tracking all cycle members together (Magnusson-Hedin style), correctly handling inter-attribute circular dependencies.

Correctness requirement: Circular attributes require the compute function to be monotone over a lattice of finite height. All four systems assume this — none verify it statically. Non-monotone functions may cause infinite loops.

5Per-Production Dispatch

Dispatch attribute computation based on node type/kind. The core of grammar-driven programming.

match(discriminant, equations) dispatches on a field value. _ is the default fallback. Built on top of syn() — fully cached.

label: match('type', {
  root:   () => 'ROOT',
  branch: () => 'BRANCH',
  leaf:   (n) => `LEAF(${n.value})`,
  _:      () => 'UNKNOWN',       // default fallback
}),

Dispatches on node[discriminant] (typically 'type' or 'kind'). Throws if no equation matches and no _ default is provided. This is a runtime dispatch — no type narrowing at compile time (though TypeScript discriminated unions could be leveraged).

In JastAdd, per-production dispatch is the fundamental mechanism. Equations are written directly on AST subclasses.

// Equations are per-production by design
syn String Expr.label();

// Each subclass provides its own equation
eq Add.label() = "ADD";
eq Literal.label() = "LIT(" + getValue() + ")";
eq VarRef.label() = "VAR(" + getName() + ")";

// Default via abstract class
syn String Expr.label() = "UNKNOWN";
// Subclasses override as needed

JastAdd's equations are inter-type declarations woven into generated Java classes. There's no need for a separate match combinator — Java's virtual dispatch handles it. @ksc/ag's match() simulates this in a non-OOP setting.

Silver's productions are the natural dispatch mechanism. Each production provides its own equations.

-- Each production defines its own equations
abstract production add
e::Expr ::= l::Expr r::Expr
{ e.label = "ADD"; }

abstract production literal
e::Expr ::= v::Integer
{ e.label = "LIT(" ++ toString(v) ++ ")"; }

-- Aspect productions add equations to existing productions
aspect production add
e::Expr ::= l::Expr r::Expr
{ e.extraInfo = "binary op"; }

Aspect productions are Silver's key extensibility feature — add equations to existing productions from separate modules without modifying the original. @ksc/ag achieves something similar by merging AttributeMap objects.

Kiama uses Scala's pattern matching as the dispatch mechanism. This is what @ksc/ag's match() approximates.

val label: Expr => String = attr {
  case Add(l, r)    => "ADD"
  case Literal(v)   => s"LIT($v)"
  case VarRef(name) => s"VAR($name)"
  case _            => "UNKNOWN"
}

Scala's pattern matching is more powerful than @ksc/ag's match() — it can destructure, guard, and nest patterns. @ksc/ag trades expressiveness for simplicity.

6Name Resolution Pattern composition

Connecting distant nodes — name resolution, scope lookup, cross-references. In @ksc/ag, this is a composition pattern using core primitives, not a dedicated combinator.

Name resolution is composed from syn + inh + match — exactly how ExtendJ (JastAdd's Java compiler) handles it. No dedicated lookup primitive needed.

// 1. Collect local definitions per scope node (syn via match)
const localDefs = match<Node, Map<string, Decl>>('kind', {
  Block:    (node) => collectDeclsFromBlock(node),
  FuncDecl: (node) => collectParamsFromFunc(node),
  _:        () => new Map(),
});

// 2. Propagate environment downward (inh)
const env = inh<Node, Map<string, Decl>>(
  (root) => (root as any).localDefs,
  (parent, child, idx) => {
    if (isScopeNode(parent)) {
      return new Map([...(parent as any).env, ...(parent as any).localDefs]);
    }
    return undefined;  // auto-propagate
  }
);

// 3. Reference attribute: resolve use to declaration (syn)
const decl = syn<Node, Decl | undefined>((node) => {
  if (node.kind !== 'VarUse') return undefined;
  return (node as any).env.get(node.name);
});

Pattern: (1) match collects local definitions per scope. (2) inh propagates the accumulated environment downward, extending at scope boundaries. (3) syn resolves a use to its declaration via the inherited environment. This composes with circular for class hierarchies and with nested inh for scope chains.

JastAdd uses reference attributes (a first-class concept) combined with coll/contributes and inh for name resolution.

// Reference attribute: returns a node reference
syn Decl Use.decl();
eq Use.decl() = lookup(getName());

// Environment via inherited + collection
coll Map<String, Decl> Program.allDefs()
  [new HashMap<>()] with putAll;

Decl contributes singletonMap(getName(), this)
  to Program.allDefs();

// Inherited env from root
inh Map<String, Decl> Expr.env();
eq Program.getChild().env() = allDefs();

// Parameterized lookup
syn Decl Scope.lookup(String name) = env().get(name);

JastAdd's coll/contributes enables directed contributions: any node can contribute a binding to any target (e.g., a specific scope). @ksc/ag's composition pattern achieves the same result for scope-chain resolution without directed contributions.

Silver uses decorated references — references to nodes that carry their attribute values. Part of Silver's type system.

-- Reference via decorated nonterminal
synthesized attribute resolved :: Decorated Decl;

production varRef
e::Expr ::= name::String
{
  e.resolved = lookupInEnv(name, e.env);
}

-- Through the reference, access the target's attributes
e.resolvedType = e.resolved.declType;

Decorated vs Undecorated: Silver distinguishes between bare tree nodes (undecorated) and fully-attributed nodes (decorated). The type system tracks which inherited attributes a decorated reference carries: Decorated Expr with {env, frame}. This enables type-safe cross-references.

Kiama handles references through regular attribute values — no special mechanism.

// Reference is just an attribute returning a node
val decl: Use => Option[Decl] = attr {
  case u: Use => u->env get u.name
}

// Env passed down via inherited attribute
val env: Tree => Map[String, Decl] = inh {
  case tree.parent(p: Block) => p->localEnv
}
Design choice: @ksc/ag deliberately avoids a dedicated lookup combinator. Name resolution is a pattern composed from core primitives, matching how production compilers (ExtendJ) actually implement it. This keeps the primitive set minimal and avoids semantic mismatch with JastAdd's directed coll/contributes.

7Tree Navigation & Stamping

How each system provides parent/child/sibling access for attribute equations.

stampTree(root, getChildren) walks the tree via BFS and stamps navigation properties directly on each node as non-enumerable properties.

stampTree(root, (node) => node.kids);

// After stamping, each node has:
node.$parent    // parent node (undefined for root)
node.$children  // array of child nodes
node.$index     // child index in parent (-1 for root)
node.$root      // true only for root node
node.$prev      // previous sibling (undefined if first)
node.$next      // next sibling (undefined if last)

Properties are stamped as non-enumerable data properties via Object.defineProperty. This means JSON.stringify and Object.keys ignore them, avoiding circular reference issues. $parent/$prev/$next create circular references in the object graph but not in serialization.

JastAdd generates Java classes with built-in parent/child methods. No manual stamping needed.

// Generated API on every AST node
node.getParent()           // parent node
node.getChild(index)       // child by index
node.getNumChild()         // number of children
node.getLeft()             // named child accessor
node.getRight()
node.getStmt(int index)    // list child by index
node.getStmtList()         // full list child

JastAdd generates typed accessor methods from the AST grammar (.ast files). Each child gets a named getter (e.g., getLeft(), getRight()). Much more type-safe than @ksc/ag's $children array.

Silver's productions define tree structure. Children are named in production signatures.

-- Children are named in production signature
production add
e::Expr ::= l::Expr r::Expr
{
  -- Access children by name: l, r
  e.value = l.value + r.value;
}

Silver productions define the tree structure declaratively. Parent access is not directly exposed — information flows down via inherited attributes.

Kiama uses a Tree wrapper that builds parent/sibling indices from the original tree.

// Build tree relations
val tree = new Tree[Node, Root](root)

// Navigation via tree object
tree.parent(node)      // Option[Node]
tree.children(node)    // Seq[Node]
tree.index(node)       // child index
tree.prev(node)        // previous sibling
tree.next(node)        // next sibling

Kiama stores relations externally (in the Tree object via WeakHashMaps). @ksc/ag stamps them directly on nodes. Both provide the same information; @ksc/ag's approach avoids an extra indirection object.

8Evaluation Strategy AGSpec

How attribute values are computed, scheduled, and cached. @ksc/ag formalizes this into AGSpec.

Demand-driven via lazy getters. AGSpec formalizes the specification; evaluate(spec, root) orchestrates the pipeline.

// A complete attribute grammar specification (AGSpec)
const myAnalysis: AGSpec<Node, Result> = {
  name: 'myAnalysis',
  domain: {
    name: 'My Analysis Domain',
    project: (root) => extractResults(root),
  },
  children: (node) => node.kids,
  attributes: { localDefs, env, decl, errors },
  deps: ['otherSpec'],  // dependency ordering
};

// Single-spec evaluation
const result = evaluate(myAnalysis, root);

// Multi-spec: topologically sorted by deps, tree stamped once
const results = evaluateAll([specB, specA], root);
// Returns Map<string, unknown> of spec name → projection

Pipeline: (1) stampTree stamps $parent/$children/etc. (2) applyAttributes installs lazy getters. (3) domain.project() triggers computation by accessing attributes. No values are computed during evaluate() — only lazy getters are installed. evaluateAll topologically sorts specs by deps and stamps the tree only once.

Demand-driven with per-attribute caching. Attributes are Java methods that memoize their results.

// Generated Java code (conceptual):
private boolean value_computed = false;
private int value_value;
public int value() {
  if (value_computed) return value_value;
  value_computed = true;
  value_value = /* equation */;
  return value_value;
}

JastAdd's _computed/_value pattern is exactly what @ksc/ag emulates with Object.defineProperty getter replacement.

Demand-driven. Silver compiles to Java with lazy evaluation semantics.

Silver adds compile-time completeness checking: the compiler verifies that every attribute has an equation on every production where it occurs. This prevents runtime "missing equation" errors.

Demand-driven via Scala lazy evaluation and internal memo tables.

Kiama stores cached values in Map[(Node, Attr), Value] structures. @ksc/ag's on-node stamping avoids this indirection, achieving faster subsequent access (direct property read vs. map lookup).

AGSpec Type System

@ksc/ag formalizes the analysis specification into typed interfaces:

// The analysis domain (what we compute)
interface AGDomain<N, R> {
  name: string;
  project?: (root: N) => R;
}

// The complete specification (Cousot Phase 1)
interface AGSpec<N, R> {
  name: string;
  domain: AGDomain<N, R>;
  children: (node: N) => N[];
  attributes: AttributeMap<N>;
  deps?: string[];
}

This is analogous to a .jrag specification in JastAdd or a grammar in Silver. The deps field enables evaluateAll to topologically sort multiple specs, ensuring attributes from one analysis are available to another.

9Serialization

Persisting attributed trees for debugging, visualization, or transfer.

Built-in serializeTree and deserializeTree with configurable attribute inclusion.

// Serialize: strips circular nav props, optionally includes attributes
const json = serializeTree(root, getChildren, {
  includeAttributes: true,
  attributeFilter: ['errors', 'type'],
  excludeKeys: ['tsNode'],
  childrenKey: 'kids',
});

// Deserialize: reconstructs tree and stamps navigation
const root = deserializeTree(json, (n) => n.children ?? []);
// Ready for applyAttributes()

Smart detection: Distinguishes AG-computed attributes from original node properties by checking property descriptors (configurable: false for cached values, get for pending lazy getters). Handles Set, Map, nested objects. Strips $parent/$prev/$next (circular refs) and functions automatically.

JastAdd provides no built-in serialization. ASTs are Java object graphs; serialization is typically done via custom tree walkers or the Beaver parser's token stream.

JastAdd's generated code uses Java object identity for caching, which complicates serialization. The _computed/_value fields are not designed to be serialized.

Silver distinguishes decorated (with attribute values) and undecorated (bare tree) nonterminals at the type level. Only undecorated trees can be easily serialized.

Silver's compilation to Java means serialization requires writing Java-level code.

Kiama stores attributes in external memo tables, so the original tree structure is serializable without special handling. However, attribute values must be re-computed after deserialization.

Unique advantage: @ksc/ag's serializeTree can capture a snapshot of the attributed tree including computed values — useful for compiler dashboards, debugging tools, and test fixtures. The property descriptor detection means you don't need to manually track which properties are AG attributes.

10Advanced Features

Features available in JastAdd and Silver that are beyond @ksc/ag's "library ceiling."

JastAdd

Non-Terminal Attributes (NTAs)

Attributes whose values are new AST nodes. These NTA nodes can themselves have attributes and participate in rewrites. Used for virtual "ghost" nodes like implicit type conversions.

syn nta TypeDecl Expr.implicitCast()
  = new CastExpr(targetType(), this);
JastAdd

AST Rewrites

Declarative tree transformations triggered automatically during traversal. The tree is rewritten in-place, transparently to attribute equations.

rewrite SimpleName {
  when (isQualified())
  to QualifiedName {
    return new QualifiedName(qualifier(), name());
  }
}
JastAdd

Collection Attributes (coll/contributes)

Two-phase system: survey walks tree to find contributors, collect aggregates contributions. Contributors can target any node via the for clause — much more powerful than subtree aggregation.

coll Set<String> Program.errors()
  [new HashSet<>()] with add root Program;

Expr contributes errorMsg()
  when hasError()
  to Program.errors();
JastAdd

Inter-Type Declarations (ITDs)

Add methods and fields to generated AST classes from .jrag files. Like aspect-oriented programming's inter-type declarations.

// Add a method to Expr class
public boolean Expr.isLValue() {
  return false;
}
public boolean VarRef.isLValue() {
  return true;
}
Silver

Forwarding

A production can forward to another tree, providing default attribute values. The forwarded-to tree acts as a "semantic prototype." Key to Silver's modular extensibility.

production forEach
s::Stmt ::= v::Name l::Expr b::Stmt
{
  forwards to desugar(v, l, b);
  -- Attributes not explicitly defined
  -- are inherited from the forwarded tree
}
Silver

Threaded / Monoid / Functor / Strategy

Silver has additional built-in attribute kinds beyond the basics:

-- Threaded: built-in inh+syn pair
threaded attribute cIn, cOut :: Int;

-- Monoid: modern collection attr
monoid attribute errs :: [Msg]
  with [], ++;

-- Functor: auto copy-with-modification
functor attribute host;

-- Strategy: composable term rewriting
strategy attribute simplify =
  innermost(step);
What Could @ksc/ag Add Within the Library Ceiling?

Features that could be added without code generation:

  • NTA (Non-Terminal Attributes): A syn() that returns a fresh stamped+attributed subtree. Feasible — call stampTree/applyAttributes on the returned nodes.
  • Rewrites: A post-evaluation pass that transforms nodes matching conditions, then re-stamps/re-applies. Feasible but complex to make transparent.
  • Threaded (chain): Previously supported, intentionally removed for JastAdd alignment. Can be encoded as paired syn+inh attributes.
  • Subtree aggregation: Previously coll(), removed to avoid confusion with JastAdd's directed coll. Encode as syn with DFS walk.

Features that require code generation (beyond the ceiling):

  • Directed contributions (JastAdd's contributes...to...for)
  • Compile-time equation completeness checking (Silver's MWDA)
  • Incremental evaluation (JastAdd's dependency tracking after AST edits)
  • Forwarding (Silver's modular extension mechanism)

What is the JastAdd equivalent of @ksc/ag's syn()?
The syn keyword: syn int Expr.value() = ...;. Both create a cached, bottom-up computed attribute. JastAdd's version is declared in .jrag files and compiled into a Java method with _computed/_value caching. @ksc/ag's version installs a JS getter that self-replaces with a data property.

How does @ksc/ag's inh() relate to JastAdd's inh?
Same pattern: Both are parent-defined. JastAdd writes eq Block.getStmt().env() = ...; @ksc/ag writes inh(rootVal, (parent, child, idx) => ...). The parent decides what value each child gets. The key difference is that JastAdd can statically verify every child gets an equation (compile-time), while @ksc/ag falls back to auto-propagation at runtime when eq returns undefined.

What algorithm does @ksc/ag use for circular attributes? How does it compare to Kiama?
@ksc/ag implements the Magnusson-Hedin algorithm (same as JastAdd): when a circular attribute is first accessed, it becomes the "cycle driver" and tracks all other circular attributes encountered during evaluation in a cycleEntries array. All cycle members are re-evaluated together each iteration until global convergence. Kiama uses a simpler per-attribute inCircle/changed loop that doesn't track inter-attribute dependencies.

Why doesn't @ksc/ag have a coll() or lookup() combinator?
JastAdd alignment. JastAdd's coll has specific semantics — directed contributions to remote nodes via contributes...to...for. The old Kiama-style coll (subtree fold) was semantically different and could cause confusion. Subtree aggregation is now done via syn with a DFS walk. Name resolution is a composition pattern using syn + inh + match, matching how ExtendJ handles it.

What is Silver's forwarding and why doesn't @ksc/ag need it?
Forwarding lets a production delegate to another tree as a "semantic prototype" — any attribute not explicitly defined uses the forwarded-to tree's value. This is critical for Silver's modular extensibility (the "expression problem"). @ksc/ag doesn't need it because it's an embedded library — you can extend behavior by merging AttributeMap objects or composing specs via evaluateAll().

What is the "library ceiling" and what lies beyond it?
The library ceiling is the subset of AG features expressible as a runtime library without code generation. Within the ceiling: syn, inh, circular, match, paramSyn, uncached, on-node caching, demand-driven evaluation. Beyond the ceiling: directed contributions (coll/contributes), compile-time equation completeness checking, incremental evaluation, AST rewrites, and forwarding — all of which require compile-time code generation.
Click to reveal
What is an AGSpec and how does evaluateAll use it?
An AGSpec is a complete attribute grammar specification: a name, a domain (with optional project function), a children accessor, an AttributeMap of equations, and optional deps (names of other specs). evaluateAll topologically sorts specs by deps, stamps the tree once, and applies each spec's attributes in dependency order. It returns a Map from spec name to projection result. This enables composable, multi-phase analyses.
Click to reveal
How does @ksc/ag's caching compare to Kiama's caching?
@ksc/ag stamps values directly on AST nodes via Object.defineProperty (getter that self-replaces with a data property). Subsequent access is a direct property read with zero overhead. Kiama uses external memo tables (Map[(Node, Attr), Value]), which adds a map lookup on each access. @ksc/ag's approach is faster for reads but "pollutes" the AST nodes with extra properties (mitigated by making AG navigation properties non-enumerable).
Click to reveal
How does @ksc/ag handle name resolution without a dedicated lookup combinator?
Composition pattern: (1) match on node kind collects local definitions per scope. (2) inh propagates the environment downward, extending at scope boundaries (return undefined to auto-propagate). (3) syn resolves a use to its declaration by looking up in the inherited environment. This is exactly how ExtendJ (JastAdd's Java compiler) handles name resolution — the pattern composes naturally with circular for class hierarchies.
