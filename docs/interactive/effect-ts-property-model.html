Effect-TS: What Properties Does It Actually Compute?

How a library encodes behavioral properties using only TypeScript's type system — no compiler plugins, no AST transforms, no magic

1. The Three Properties

TypeScript tracks one thing per expression: its type (shape of the value). Effect-TS tracks three additional behavioral properties on every Effect expression:

A — Success
What it produces
E — Error
How it can fail
R — Requirements
What it needs

A (Success type) — Same as a function's return type. If the computation succeeds, you get a value of type A. This is the property TypeScript would already track.

E (Error type) — The typed error channel. Not unknown like thrown exceptions — a precise union of all the ways this computation can fail. E = never means it cannot fail. E = NotFound | Timeout means exactly those two failure modes.

R (Requirements) — The dependency/context channel. What services or capabilities this computation needs to run. R = never means it's self-contained. R = Database | Logger means it needs both services.

Click each tab to see what TypeScript's type system cannot express about the same computation:

TypeScript

// TS knows the return type
function getUser(id: string): Promise<User>
// Return type: User ✓

Effect-TS

// Effect knows the same
const getUser = (id: string):
  Effect<User, never, never>
// A = User (same as TS) ✓

For success types, Effect-TS and TypeScript agree. This property isn't new.

TypeScript

// TS has NO IDEA what errors can occur
async function getUser(id: string): Promise<User> {
  const res = await fetch(`/api/${id}`)
  if (!res.ok) throw new HttpError(res.status)
  return parseUser(await res.json())
}
// Errors: ??? (untyped, invisible)

Effect-TS

// Effect tracks EVERY error in the type
const getUser = (id: string):
  Effect<User, HttpError | ParseError, never>
//        E = HttpError | ParseError
// Compiler FORCES you to handle both

This is the first genuinely new property. TypeScript has no way to express "this function can throw HttpError or ParseError." It all collapses to unknown in catch blocks.

TypeScript

// TS has no concept of "requirements"
async function getUser(id: string): Promise<User> {
  // Where does db come from? Global? Closure?
  const user = await db.query(`SELECT...`)
  logger.info("fetched user")
  return user
}
// Dependencies: invisible, implicit

Effect-TS

// Effect tracks EVERY dependency in the type
const getUser = (id: string):
  Effect<User, HttpError, Database | Logger>
//        R = Database | Logger
// Cannot run without providing both

This is the second genuinely new property. TypeScript doesn't track implicit dependencies. Effect-TS makes every dependency explicit — the compiler won't let you run the effect until all requirements are satisfied.

2. How It's Implemented: Zero Compiler Changes

Effect-TS achieves all of this using only standard TypeScript features. No compiler plugins, no babel transforms, no custom syntax.

The Core Trick: Variance Markers as Phantom Types

// From Effect-TS source: packages/effect/src/Types.ts

// A appears in output position only → covariant
type Covariant<A> = (_: never) => A

// A appears in input position only → contravariant
type Contravariant<A> = (_: A) => void

// A appears in both positions → invariant
type Invariant<A> = (_: A) => A

These are phantom types — never called at runtime. They exist only as instructions to TypeScript's variance checker. When the type checker sees Covariant<A>, it understands: "A can be widened — if Effect<Cat, ...> is needed, Effect<Animal, ...> works."

The Variance Struct: Where Properties Live

// packages/effect/src/Effect.ts — the actual interface

interface Effect<out A, out E = never, out R = never>
  extends Effect.Variance<A, E, R>, Pipeable {
  readonly [Unify.typeSymbol]?: unknown
  readonly [Unify.unifySymbol]?: EffectUnify<this>
  [Symbol.iterator](): EffectGenerator<Effect<A, E, R>>
}

// The variance struct tells TS how A, E, R behave
interface VarianceStruct<out A, out E, out R> {
  readonly _V: string           // version marker
  readonly _A: Covariant<A>   // success is covariant
  readonly _E: Covariant<E>   // error is covariant
  readonly _R: Covariant<R>   // requirement is covariant
}

At runtime, every Effect object carries a single shared variance marker object — the same object for all effects:

// packages/effect/src/internal/effectable.ts
export const effectVariance = {
  _R: (_: never) => _,    // actual function, never called
  _E: (_: never) => _,
  _A: (_: never) => _,
  _V: currentVersion
}

The runtime cost is one shared object. All property tracking happens in TypeScript's type checker at compile time and is completely erased in the JavaScript output.

4. The Runtime Instruction Set

Effect-TS doesn't modify TypeScript's AST. But it creates its own runtime instruction tree — an internal "AST" the fiber executor interprets. Click an instruction to see what it does and how properties propagate:

Two Tree Structures, Two Levels

1. TypeScript's AST (compile time) — The standard syntax tree that tsc parses. Effect-TS does NOT touch this. Every Effect program is valid TypeScript. KindScript walks this tree.

2. Effect's instruction tree (runtime) — Built when Effect values are composed. pipe(a, flatMap(f), map(g)) builds a chain of OnSuccess nodes. The fiber executor traverses this tree.

The two trees have similar structure but exist at different times. Properties (A, E, R) are tracked on the TS AST level via types, and encoded in the instruction tree via the node structure at runtime.

7. What Effect-TS Does NOT Introduce

No new AST nodes

Every Effect program parses as standard TypeScript. Running ts.createSourceFile() on Effect code gives the same node types as any TS code: CallExpression, ArrowFunction, PropertyAccessExpression, etc.

No compiler plugins

No babel plugins, no SWC plugins, no TypeScript transformer plugins. Standard tsc handles everything.

No decorators

Services are created via Context.Tag() function calls, not @Injectable()-style annotations.

What it DOES use (standard TS features only)

// Generic type parameters carry properties
interface Effect<out A, out E = never, out R = never>

// Conditional types extract properties
type Success<T> = T extends Effect<infer A, any, any> ? A : never

// Mapped types transform properties
type MergeErrors<T extends Effect<any,any,any>[]> =
  T[number] extends Effect<any, infer E, any> ? E : never

Generic type parameters are the primary vehicle. TypeScript's standard inference propagates A, E, R through function calls.

// pipe() has ~20 overloads for type propagation
interface Pipeable {
  pipe<A>(this: A): A
  pipe<A, B>(this: A, ab: (_: A) => B): B
  pipe<A, B, C>(this: A, ab: (_: A) => B, bc: (_: B) => C): C
  // ... up to 20 arguments
}

Overloads tell TypeScript exactly how types flow through chains. Each overload is a specific arity — TS picks the matching one and threads parameters through.

// Effect.gen uses generators as do-notation sugar
const program = Effect.gen(function*() {
  const db = yield* Database      // R += Database
  const rows = yield* db.query(q) // E += SqlError
  return rows[0]                  // A = Row
})

// gen() extracts E and R from yielded types via
// conditional type extraction on the generator's
// yield type parameter

Generators are standard TS syntax. yield* returns A. Effect.gen's return type uses conditional types to extract and union all E and R from yielded effects.

// Unique symbols for type-level identity
export const EffectTypeId: unique symbol = Symbol.for("effect/Effect")

// Symbol-keyed properties carry type metadata
interface Effect<A, E, R> {
  readonly [EffectTypeId]: VarianceStruct<A, E, R>
  readonly [Unify.typeSymbol]?: unknown
  readonly [Unify.unifySymbol]?: EffectUnify<this>
}

Unique symbols create nominal identity within TypeScript's structural type system. Two different effect-like types won't accidentally unify.

8. Beyond Effect: Types With More Properties

// Channel: the most property-rich type in Effect-TS
interface Channel<
  out OutElem,   // elements produced (covariant)
  in  InElem,    // elements consumed (contravariant)
  out OutErr,    // errors emitted (covariant)
  in  InErr,     // errors received (contravariant)
  out OutDone,   // completion value (covariant)
  in  InDone,    // upstream done signal (contravariant)
  out Env        // requirements (covariant)
>

Seven type parameters with mixed variance. Channels model bidirectional communication. The variance tells TypeScript exactly which parameters can be widened and which must match exactly.

// Stream: unbounded sequence of values
interface Stream<
  out A,  // element type
  out E,  // error type
  out R   // requirements
>

Same three properties as Effect but A is each element, not the final result. Built on Channel internally.

// Layer: dependency provider (inverted variance!)
interface Layer<
  in  ROut,  // services PROVIDED (contravariant!)
  out E,     // construction errors
  out RIn    // services REQUIRED
>

Layer inverts R's variance. Effect's R is covariant (what you need). Layer's ROut is contravariant (what you provide). This makes Effect.provide(layer) work: it subtracts the layer's contravariant output from the effect's covariant requirement.

// STM: Software Transactional Memory
interface STM<
  out A,  // result on commit
  out E,  // error type
  out R   // requirements
>

Same type parameters as Effect, but transactions are retried automatically on conflict. The behavioral difference is in the executor, not the types.

9. Implications for KindScript

The architecture comparison

Effect-TS encodes three behavioral properties as generic type parameters on expressions. TypeScript's inference propagates them. The programmer opts in by using wrappers. Untracked code is invisible.

KindScript computes behavioral properties by walking the AST. No opt-in needed — existing code is analyzed. But properties are currently per-function, not per-expression.

Where they converge

Verify Effect-TS claims. When Effect.sync(() => thunk) claims E = never, R = never, KindScript could walk into the thunk and verify it truly has no errors and no dependencies.

Infer Effect-style signatures for plain code. Given a normal TS function, produce an A/E/R triple: what it returns, what it can throw, what globals it accesses.

Go beyond A/E/R. Properties like "terminates," "is deterministic," "doesn't access the network" are beyond what generic type parameters can encode — but a static analyzer can check them.
