What Are Types Applied To?

The three layers of a programming language, what lives in each, and how TypeScript and KindScript attach predicates to the same underlying entities.

The Three Layers

Every programming language has three distinct layers. TypeScript and KindScript both operate on Layer 1 to make claims about Layer 2. Layer 3 is what actually happens when code runs.

Layer 1: Syntax (the AST)

This is literally text. The expression 2 + 3 is a sequence of characters that the parser converts into an AST node:

{
  "kind": "BinaryExpression",
  "operator": "+",
  "left":  { "kind": "NumericLiteral", "value": "2" },
  "right": { "kind": "NumericLiteral", "value": "3" }
}

This is what TypeScript and KindScript actually look at. They walk these nodes. Every "type check" or "property check" is an algorithm processing AST nodes.

Key point: 2 + 3 and 4 + 1 are different AST nodes. At this layer, they are not the same thing. They are different pieces of syntax that happen to evaluate to the same value.

Layer 2: Denotation (the mathematical meaning)

The denotation function ⟦·⟧ maps syntax to math:

⟦2 + 3⟧ = 5

⟦"hello"⟧ = the string hello

⟦(x) => x + 1⟧ = λx. x + 1

These mathematical objects exist independently of any computer. The number 5 is not "computed" — it simply is. The function λx. x+1 is a mathematical mapping, not a piece of code.

Key point: ⟦2 + 3⟧ = ⟦4 + 1⟧ = 5. Different syntax, same denotation. Types and properties are predicates on these objects.

No tool ever computes denotations. TypeScript and KindScript analyze syntax (Layer 1) to make predictions about denotations (Layer 2). The predictions are the types/properties.

Layer 3: Runtime (what actually happens)

When V8 executes 2 + 3, it produces a machine-level representation of the number 5 (a 64-bit IEEE 754 double, or a V8 Smi). When it executes (x) => x + 1, it allocates a closure object on the heap with a pointer to the compiled code and captured variables.

Runtime values are implementations of denotations in a specific machine. They have properties that denotations don't: memory addresses, garbage collector metadata, hidden class pointers, JIT compilation status.

Key point: Neither TypeScript nor KindScript ever sees runtime values. They are purely static tools. They predict what will happen at Layer 3 by analyzing Layer 1, using Layer 2 as the conceptual framework for those predictions.

The Core Mechanism

Both TypeScript and KindScript do the same fundamental thing, just asking different questions:

Analyze syntax  →  make predicates  about  denotations
TypeScript asks:

"What shape does this denotation have?"

→ is it a number? an object with key name? a function from A to B?

KindScript asks:

"What behavior does this denotation exhibit?"

→ does it mutate? perform IO? always terminate? depend only on inputs?

Same entities, different questions. TypeScript and KindScript both make claims about the denotations of expressions. TypeScript's predicate language talks about structure: "what members does this have?" KindScript's predicate language talks about behavior: "what effects does this perform?" Both analyze syntax. Both target denotations. They're orthogonal dimensions of the same underlying objects.

What This Means for KindScript

Your property primitives are atomic predicates on function denotations

Just as TypeScript's primitive types (number, string, boolean) are the atomic predicates on value denotations, KindScript's property primitives should be the atomic predicates on behavioral denotations of functions.

The criteria for a good primitive:

// 1. COMPOSITIONAL — composes at boundaries without looking inside
//    If f is noIO and g is noIO, then g∘f is noIO ✓

// 2. EXTENSIONAL — part of the interface, not implementation detail
//    Callers can rely on it without reading the source ✓

// 3. APPROXIMABLE — can be verified by syntactic analysis
//    Walking the AST gives a sound (maybe conservative) answer ✓

// 4. INDEPENDENT — not derivable from other primitives
//    noIO and noMutation say different things ✓

// 5. USEFUL — programmers actually care about this distinction
//    "Can I call this in a React render?" = noSideEffects ✓
