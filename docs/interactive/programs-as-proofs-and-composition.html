Programs as Proofs & The Compositional Structure

Three views of computation that all converge: programs are proofs, programs are morphisms, and halting makes everything simpler.

Part 1: Programs ARE Proofs

The Curry-Howard correspondence isn't a metaphor — it's a mathematical isomorphism. Every program you write is literally a proof of a theorem, and the theorem is the program's type.

function id<A>(x: A): A { return x; }

Theorem: For any proposition A, given a proof of A, we can produce a proof of A.

In logic: ∀A. A → A — the law of identity. Trivially true, but the program IS the proof.

function compose<A,B,C>(f: (a:A)=>B, g: (b:B)=>C): (a:A)=>C {
  return (a) => g(f(a));
}

Theorem: If A implies B, and B implies C, then A implies C.

In logic: (A → B) → (B → C) → (A → C) — transitivity of implication. This is hypothetical syllogism, a rule of classical logic. The function body is the proof: apply f to get B, then apply g to get C.

function first<A,B>(pair: [A, B]): A { return pair[0]; }

Theorem: If A AND B are both true, then A is true.

In logic: A ∧ B → A — conjunction elimination (left projection). The tuple is a proof of both; the function extracts one.

function injectLeft<A,B>(a: A): A | B { return a; }

Theorem: If A is true, then A OR B is true.

In logic: A → A ∨ B — disjunction introduction. Having proof of A automatically gives proof of "A or B".

function absurd<A>(x: never): A { return x; }

Theorem: From falsehood, anything follows.

In logic: ⊥ → A — ex falso quodlibet. If you have a value of type never (a proof of falsity), you can produce anything. The function is valid because it can never actually be called — there are no values of type never.

Where TypeScript Breaks the Correspondence

TypeScript is an unsound type system — it deliberately breaks the isomorphism in several places:

any — the lying witness

const lie: string = (42 as any);
// Claims to have a proof of "string"
// but actually holds a number

any is a "proof" of every proposition simultaneously — which in a consistent logic would mean everything is true and false. It breaks the correspondence entirely.

as — proof by assertion

const fake = (5 as unknown as string);
// "Trust me, 5 is a string"

Type assertions bypass proof checking. In logic, this is like writing "Proof: obvious" — the checker just takes your word for it.

No termination guarantee

function loop(): number {
  return loop(); // infinite loop
}
// Type says: produces a number
// Reality: produces nothing

In constructive logic, a proof of A must actually construct an A. A non-terminating function claims to produce a value but never does — it's a false proof that slips through.

Languages like Agda, Lean, and Coq take the correspondence seriously: no any, no type assertions, and all programs must terminate. In those languages, if your program type-checks, it's a valid mathematical proof.

Part 2: The Compositional Structure

Category theory gives us the most abstract view of programs: programs are arrows between types, and the only thing that matters is how they compose.

Why Composition is Everything

The reason category theory matters for programming isn't abstraction for its own sake — it's that composition is the only way to build complex programs from simple ones. And the main obstacle to composition is effects.

The Problem: Effects Break Composition

// Pure functions compose perfectly:
const length = (s: string): number => s.length;
const isEven = (n: number): boolean => n % 2 === 0;
const isEvenLength = (s: string): boolean => isEven(length(s)); // ✓ trivial

// But effectful functions DON'T compose:
const getUser = (id: number): Promise<User> => fetch(`/users/${id}`);
const getName = (user: User): string => user.name;

// This doesn't work:
// getName(getUser(42))  ← TYPE ERROR: getName wants User, got Promise<User>

// The effect (Promise) is stuck in the middle, blocking composition.

This is the central problem that category theory solves for programming.

Functors: Lifting Functions Into Containers

A functor is a mapping between categories that preserves structure. In programming: if you can transform A → B, a functor lets you transform F<A> → F<B>.

// Array is a functor

// Promise is a functor

// The LAWS a functor must obey:
// 1. Identity:    F.map(id)    = id
//    [1,2].map(x => x) deepEquals [1,2] ✓

//    [1,2].map(x => String(x+1)) deepEquals [1,2].map(x => x+1).map(String) ✓

Functors give you map — the ability to reach inside a container and transform its contents without changing the container's structure.

Natural transformations let you change the container type (Array → Maybe, Promise → Array) while guaranteeing the contents are treated consistently.

Monads: Composing Effectful Functions

A monad is a functor with two extra operations that let you sequence computations that produce wrapped values.

of / return — wrap a value

// Put a plain value into the effect
Promise.resolve(42)    // number → Promise<number>
[42]                   // number → number[]
{ kind:'just', v:42 }  // number → Maybe<number>

flatMap / bind — chain effects

// Feed a value into the next effectful function
// WITHOUT nesting: Promise<Promise<A>> → Promise<A>
promise.then(x => fetchMore(x))
arr.flatMap(x => [x, x*2])
maybe.flatMap(x => safeDivide(x, 2))
// The THREE MONAD LAWS:

// 1. Left identity: of(a).flatMap(f)  ===  f(a)
Promise.resolve(5).then(x => fetch(`/api/${x}`))
// same as: fetch('/api/5')

// 2. Right identity: m.flatMap(of)  ===  m
Promise.resolve(5).then(x => Promise.resolve(x))
// same as: Promise.resolve(5)

// 3. Associativity: m.flatMap(f).flatMap(g) === m.flatMap(x => f(x).flatMap(g))
// Chaining order doesn't matter — just like function composition!

The punchline: Law 3 is exactly associativity of composition. Monads recover composition for effectful functions. That's their entire purpose.

The Kleisli Category: Where Effects Live

Here's the categorical view that ties it all together. For any monad M, there's a category — the Kleisli category — where:

•Objects: the same types (string, number, User, etc.)
•Morphisms: functions A → M<B> (effectful functions)
•Composition: Kleisli composition via flatMap
•Identity: return / of
// In the "normal" category:
//   f: A → B,   g: B → C,   compose: g(f(x))

// In the Kleisli category for Promise:
//   f: A → Promise<B>,   g: B → Promise<C>
//   compose: f(x).then(g)      ← THIS is composition in Kleisli

const getUser   = (id: number): Promise<User>    => fetch(`/users/${id}`);
const getOrders = (user: User):  Promise<Order[]> => fetch(`/orders/${user.id}`);
const getTotal  = (orders: Order[]): Promise<number> => calculateTotal(orders);

// Kleisli composition (>=> in Haskell):
const pipeline = (id: number): Promise<number> =>
  getUser(id)
    .then(getOrders)     // compose in Kleisli
    .then(getTotal);     // compose again

// This IS composition. It IS associative. It HAS identity.
// The Kleisli category is a real category, with all the laws.

Every monad gives rise to a Kleisli category. Promise.then IS composition in the Kleisli category of async effects. Array.flatMap IS composition in the Kleisli category of nondeterminism. This is why monads feel compositional — they literally create a category where effectful functions compose.

When you assume all computations terminate, the mathematical landscape simplifies dramatically. Several complex models collapse into simpler ones:

The entire apparatus of domain theory — partial orders, Scott continuity, least fixed points, chain-complete posets — exists only to handle ⊥. Remove ⊥ (by guaranteeing termination), and types are just sets, functions are just functions. You recover classical mathematics.

Programs Form Proper Algebraic Structures

With termination, programs obey algebraic laws that break down when functions might not return:

Monoid (closed, associative, has identity)

Strong Normalization: Every Path Leads Home

A term rewriting system is strongly normalizing if every sequence of reductions terminates — no matter what order you reduce in.

Example: reduce ((λx. x+1) 5) * ((λy. y*2) 3)

→Path 1: reduce left first → (5+1) * ((λy. y*2) 3) → 6 * (3*2) → 6 * 6 → 36
→Path 2: reduce right first → ((λx. x+1) 5) * (3*2) → (5+1) * 6 → 6 * 6 → 36
→Path 3: reduce inner first → ((λx. x+1) 5) * ((λy. y*2) 3) → 6 * 6 → 36

All paths converge to 36. This is the Church-Rosser property + strong normalization.

Without termination, some paths could diverge forever. You'd need to choose the right evaluation strategy (lazy vs strict) or risk getting stuck.

Consequence: In a strongly normalizing system, every expression has a unique, canonical "answer" — its normal form. The compiler can evaluate any sub-expression at compile time, reorder computations freely, and optimize aggressively, because all paths lead to the same result.

System T and System F: Total Languages That Work

System T (Gödel, 1958)

Simply typed lambda calculus + natural numbers + primitive recursion.

-- Everything in System T terminates.
-- Can express all primitive recursive functions:
add : Nat → Nat → Nat
add 0     n = n
add (S m) n = S (add m n)

mul : Nat → Nat → Nat
mul 0     n = 0
mul (S m) n = add n (mul m n)

All programs halt Can't express Ackermann

System T is the theoretical foundation for total functional programming. It proves you can compute a huge class of functions while guaranteeing termination.

System F (Girard, 1972)

Polymorphic lambda calculus. Adds type abstraction: functions can take types as arguments.

-- System F adds ∀ (forall) quantification
id : ∀α. α → α
id = Λα. λx:α. x

-- Polymorphic pairs, church encodings, etc.
-- ALL still strongly normalizing!
-- Even with parametric polymorphism,
-- every program terminates.

All programs halt Parametric polymorphism

System F is the theoretical basis for Haskell's type system (minus general recursion) and ML-family languages. Remarkably, adding polymorphism doesn't break termination.

Cartesian Closed Categories Without Lifting

The simply typed lambda calculus corresponds to Cartesian Closed Categories (CCCs):

•Objects: types
•Morphisms: total functions
•Products: tuple types A × B
•Exponentials: function types A → B
•Terminal object: unit type ()
// Without termination (in Haskell's Hask):
// Hask is NOT actually a category! Because ⊥ breaks the laws.
// seq ⊥ () = ⊥  but  seq () ⊥ = ⊥
// This means f ∘ id ≠ f in some cases (when f = seq)!

// With termination (in a total language like Agda):
// You get a GENUINE Cartesian Closed Category.
// All laws hold. All equational reasoning is valid.
// β-reduction:  (λx. e) a = e[a/x]     always
// η-reduction:  λx. f x   = f           always
// These are TRUE equalities, not approximations.

The punchline: Non-termination makes Hask not actually a category (in the strict sense). Termination gives you a real CCC where all the pretty diagrams actually commute. This is why proof assistants like Agda and Lean require totality — they need the categorical structure to be sound for proofs to work.

What This Means for Analysis

Curry-Howard for analysis

If your analyzer can infer that a function has type (x: {v:int | v>0}) → {v:int | v>0}, it has simultaneously proved the theorem "positive integers are closed under this operation." Type inference IS theorem proving.

Composition for analysis

If you can verify that f is pure and g is pure, composition guarantees g∘f is pure. Properties that are preserved by composition are compositional properties — and those are the ones a static analyzer can track module-by-module, without whole-program analysis.

Totality for analysis

If you can verify (or assume) termination, every function becomes a morphism in a real CCC. Equational reasoning is sound. Refactoring preserves semantics. Optimizations are valid. The entire algebraic toolkit opens up.
