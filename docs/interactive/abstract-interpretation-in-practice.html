01What "Absence of Runtime Errors" Actually Means

It does not mean "absence of all bugs." It means something far more specific and mathematically precise.

Common misconception: "Formally verified = bug-free." No. It means a specific, bounded class of errors has been proven impossible. The software could still compute the wrong answer, implement the wrong formula, or violate a business requirement. What it cannot do is exhibit undefined behavior.

The closed set of C undefined behaviors

The C language standard defines specific situations where program behavior becomes mathematically undefined — the compiler can generate any code it wants, including code that crashes, corrupts memory, or silently produces wrong results. Astrée proves none of these can ever occur:

⚠

Signed Integer Overflow

INT_MAX + 1 is undefined. The compiler may assume it never happens and optimize based on that assumption, silently breaking your code.

÷

Division by Zero

x / 0 and x % 0. Not just a crash risk — it's undefined, so the compiler can do anything, including "succeed" with a garbage value.

▮

Array Out-of-Bounds

a[n] where n >= length. Reads/writes arbitrary memory. The #1 source of security vulnerabilities in C code.

∅

Null Pointer Dereference

*p where p == NULL. May crash, may silently read page zero, may be optimized away entirely by the compiler.

?

Uninitialized Variable Reads

Reading a variable before it's assigned. The value isn't "zero" or "random" — it's undefined, and can change between runs or compiler versions.

⇄

Invalid Pointer Arithmetic

Pointer math that goes before the start or past one-past-the-end of an allocation. Undefined even if you never dereference the result.

∞

Floating-Point Exceptions

NaN and infinity in contexts that don't expect them. In flight control: a NaN in an actuator command is catastrophic.

‖

Data Races

Concurrent unsynchronized access to shared memory. Results are undefined — not just "one thread wins" but literally anything can happen.

What it proves vs. what it doesn't

✓ Proven

  • Program always executes per C semantics
  • No undefined behavior on any execution path
  • No undefined behavior for any possible input
  • Compiled code will behave as source code says
  • Entire class of UB eliminated mathematically

✗ Not Proven

  • Software does what the pilot intended
  • Requirements are correct
  • Aerodynamic model is accurate
  • Sensors provide correct readings
  • Logical bugs (wrong formula, swapped variable)
  • Timing deadlines are met (that's WCET analysis)
Why this matters for fly-by-wire: The A340's flight control reads sensor data, computes actuator commands, and writes them to control surfaces. If the C program has undefined behavior, the compiler's optimizer may have generated machine code that does something entirely different from what the source code appears to say. Proving absence of UB means: whatever the code computes, it's at least computing what the C says, not something random an optimizer decided.

The entire engineering challenge of abstract interpretation is navigating this spectrum: precise enough that most checks are proven safe (avoiding false alarms), but abstract enough that the analysis terminates in reasonable time. Abstract domains are the mathematical structures that define where you sit on this spectrum.

Intervals track each variable independently as a range [min, max]. For example, knowing x ∈ [3, 9] and y ∈ [5, 12]. Fast (O(n) per program point) but loses all relationships between variables. If y = x + 2, intervals can only say y ∈ [5, 11] — it doesn't know that when x=3, y must be 5. This means y - x is estimated as [-6, 9] instead of the exact value 2. That lost precision can cause false alarms on expressions like a[y - x] where the array has size 3.

Key idea: Astrée doesn't pick one domain — it runs all of them simultaneously, combining their knowledge at every program point. The interval domain handles simple range checks. Octagons catch x - y relationships. The filter domain understands digital signal processing stability. Each domain contributes what it's good at. This is called a reduced product of abstract domains.

04The 6 Hard Problems

Understanding abstract interpretation theory takes a semester. Making it work on 132,000 lines of real flight control code with zero false alarms takes years. Here's why.

1 Choosing & Tuning Abstract Domains ▼

The analyzer needs to be precise enough for every single check in the program. One imprecise abstraction anywhere = one false alarm = failure.

For the Airbus code, the Astrée team had to invent new abstract domains:

  • Digital filter domain: Understands mathematical stability properties of IIR/FIR filters used in flight control. Without this domain, the analyzer cannot prove that a feedback loop's output stays bounded — it just sees "multiply by coefficient, add to accumulator" and the accumulator's abstract value grows unboundedly.
  • Floating-point domain: Tracks rounding errors precisely. Not just "some float" but "a float in [3.14159265, 3.14159266]." IEEE 754 rounding is deterministic, so the analysis can track it exactly, but only if it models the specific rounding mode the hardware uses.
  • Relational domains: When one variable bounds another (e.g., an index computed from an array length), the analysis needs to track that relationship. Intervals alone lose this information completely.

Getting zero false alarms on 132K lines required the abstract domains to be precise enough for every single one of the thousands of array accesses, divisions, pointer dereferences, and arithmetic operations in the program.

2 Loop Invariants & Widening ▼

Loops are THE hard part. If a loop runs 10,000 iterations, you can't analyze each one separately — you need to find a loop invariant: a property that holds at the start of every iteration.

Abstract interpretation uses widening to force convergence: after a few iterations, it jumps to a safe over-approximation. But widening is inherently imprecise:

Loop: for (int i = 0; i < 1024; i++) Iter 0: i ∈ [0, 0] Iter 1: i ∈ [0, 1] Iter 2: i ∈ [0, 2] ... analyzing 1024 iterations is too slow ... Widen: i ∈ [0, +∞) ← sound but imprecise! a[i] with i ∈ [0, +∞) → ALARM (potential out-of-bounds) Fix: Add widening threshold {i: 1024} Widen: i ∈ [0, 1023] ← precise, and a[i] is safe

Each loop in the program potentially needs its own widening strategy. Techniques include:

  • Widening with thresholds: Provide hints about bounds ("this counter is bounded by 1024")
  • Loop unrolling: Analyze the first N iterations precisely before widening
  • Delayed widening: Wait longer before giving up on precise iteration
  • Narrowing: After widening, iterate backwards to tighten the approximation
3 Pointer & Memory Analysis ▼

C has unconstrained pointers. To prove an array access is safe, you need to know both what the pointer points to and what the index value is.

For Airbus code with complex data structures (linked sensor data, flight state records), the pointer analysis needs to track aliasing relationships precisely. If two pointers might alias the same memory, a write through one pointer invalidates everything the analysis knew about the other. Imprecise aliasing = massive precision loss = false alarms everywhere.

The analyzer must model the memory layout: struct field offsets, array strides, stack vs. heap allocation, and alignment. For embedded systems, this includes memory-mapped I/O regions with specific address ranges.

4 Concurrency & Scheduling ▼

The A340 flight control runs multiple tasks on a real-time scheduler. Proving absence of data races requires modeling the specific RTOS scheduler semantics.

The analyzer doesn't just check each task in isolation — it must consider all possible interleavings of task executions that the scheduler permits. For a priority-based preemptive scheduler, this means modeling which tasks can preempt which, and proving that shared data is always accessed under the correct synchronization.

This is where soundness matters most: an analysis that ignores concurrency is unsound (could miss real data races), but an analysis that considers all possible interleavings is exponentially expensive. The abstract scheduler model must be precise enough to eliminate impossible interleavings while remaining tractable.

5 Environment Modeling ▼

The code reads from sensors and writes to actuators. The analysis needs a model of what values sensors can produce. Get this wrong and everything falls apart:

ModelConstraintProblem
Too wide altitude: [-∞, +∞] False alarms everywhere — analysis thinks altitude can overflow any computation
Too narrow altitude: [0, 45000] Unsound if sensor can report negative values during nose-down pitch — missed real bugs
Just right altitude: [-1000, 60000] Matches sensor hardware spec + all flight regimes. Sound and precise.

Getting these ranges right requires collaboration with the systems engineering team, reading hardware datasheets, understanding sensor behavior in all flight regimes (normal, abnormal, failure modes), and sometimes physical testing.

6 Manual Annotation & Configuration ▼

This is where the real labor is. To achieve zero false alarms, the Astrée team had to:

  • Specify entry points & initial conditions: What functions are called, in what order, with what initial state
  • Provide environment constraints: Sensor value ranges, timer frequencies, memory-mapped I/O behavior
  • Add widening hints: Thresholds for every difficult loop in the program
  • Configure domain precision: Which abstract domains to enable for which code regions
  • Tune partitioning: Where to split the analysis into separate cases (e.g., analyze "flaps up" and "flaps down" paths separately)
  • Iterate on false alarms: For each alarm, determine if it's a real bug or a precision issue, then refine the configuration
The real timeline: For the Airbus work, this was a multi-year research collaboration between the Astrée team (ENS Paris, CNRS) and Airbus engineers. It wasn't "install tool, press run, get results." It was "run analysis, get 500 false alarms, understand why each one occurred, design new abstract domains or tuning strategies, repeat for months until zero false alarms." Ongoing use on incremental changes is faster, but initial setup for a new codebase is where 80% of the cost sits.

The LLM is doing something genuinely difficult here: combining natural language understanding of code intent with knowledge of abstract interpretation techniques. This cross-domain reasoning is exactly where LLMs shine.

This is what makes the formal verification + LLM combination so powerful: the LLM has a mathematical spellchecker that catches all its mistakes. In every other domain where LLMs are applied, their errors propagate to the output. Here, errors are caught at the boundary and only correct suggestions pass through.

Q

What does "absence of all runtime errors" actually prove, and what does it not prove?

A

It proves that no execution path can trigger C undefined behavior (overflow, null deref, out-of-bounds, etc.) — a closed, finite set defined by the C standard. It does not prove the software computes the right answer, that requirements are correct, or that there are no logical bugs. It guarantees the program executes as C semantics define, never entering undefined territory.

Q

Why is "perfectly precise" abstract interpretation impossible?

A

Perfectly precise reasoning about all possible executions of an arbitrary C program is equivalent to the halting problem — it's mathematically undecidable. That's why abstract interpretation uses over-approximation: it may say "this might overflow" when it actually can't (false alarm), but it will never say "this is safe" when it isn't (no false negatives). The engineering challenge is minimizing false alarms while maintaining soundness.

Q

What is "widening" and why is it necessary for loops?

A

Widening is a technique to force the abstract analysis to converge when analyzing loops. Without it, the analyzer would need to simulate every iteration (impractical for 10,000+ iterations). Widening jumps to a safe over-approximation after a few iterations — e.g., after seeing i=0, i=1, i=2, it may widen to i∈[0, +∞). This is sound but imprecise. Widening thresholds refine this: telling the analyzer "i is bounded by 1024" so it widens to [0, 1023] instead.

Q

What is a "reduced product" of abstract domains and why does Astrée use it?

A

A reduced product runs multiple abstract domains simultaneously, combining their knowledge at every program point. Intervals handle simple range checks. Octagons track pairwise variable relationships (x-y ≤ 5). The filter domain understands digital signal processing stability. Each domain contributes what it's good at, and their combined precision exceeds any single domain. Astrée needed this to achieve zero false alarms — no single domain was precise enough for all 132K lines.

Q

Why is environment modeling both critical and error-prone?

A

The analysis needs to know what values sensors can produce. Constraints that are too wide (altitude: [-∞, +∞]) cause false alarms everywhere. Constraints that are too narrow (altitude: [0, 45000]) are unsound — if the sensor can report -1000 in a dive, the analysis misses real bugs. Getting it right requires reading hardware datasheets, understanding all flight regimes including failure modes, and collaboration with systems engineers. This manual process is expensive and is one of the best targets for LLM automation.

Q

Why is using LLMs for verification configuration fundamentally safer than using LLMs for code generation?

A

In code generation, LLM errors become runtime bugs — there's no safety net. In verification configuration, every LLM suggestion is checked by the mathematical verification engine. Wrong suggestions are harmlessly rejected (the proof simply fails to go through). The LLM proposes, the math disposes. The soundness guarantee depends on the verification engine, never on the LLM's accuracy. It's like having a perfect oracle that catches all the LLM's mistakes before they matter.
