1The Landscape

A comprehensive survey of existing open-source abstract interpretation frameworks, libraries, and analyzers.

The abstract interpretation ecosystem spans from full-scale industrial analyzers (IKOS, Infer, Frama-C) to reusable domain libraries (Apron, ELINA, SPARTA) to language-specific tools (TAJS, MIRAI). Critically, no production-quality AI framework exists in TypeScript — this is a greenfield space.

Produces a custom AR (Abstract Representation) from LLVM bitcode. Numerical domains (intervals, octagons via Apron), gauge domain. Deterministic parallel fixpoint (POPL 2020). Widening with thresholds. Active — v3.5 Dec 2024.

OCaml • C • LGPL • 185 stars
Plugin in the Frama-C verification platform. Rich domain product: intervals, congruences, gauge, symbolic locations, bitwise domain. Iterative widening with descending (narrowing) iterations. Strong ACSL annotation integration. Biannual releases.
MOPSA
OCaml • C + Python • LGPL-3.0 • GitLab (5k commits)
Modular Open Platform for Static Analysis. Unique AST-based approach (no CFG lowering). Small domains collaborate via expression rewriting. Analyzes C, Python, and mixed C/Python. Highly modular — easy to add new domains.
Goblint
OCaml • Concurrent C • MIT • 235 stars
Specializes in thread-aware analysis. Side-effecting constraint system solver with multiple strategies (TD, SLR). Value, address, region, mutex, and thread domains. Highly modular analysis composition via products. 19k commits.

OCaml • C/C++ • Commercial (AbsInt) • Airbus-certified
Sound analyzer proving absence of runtime errors. Intervals, octagons, polyhedra, filter, trace partitioning, gauge, and many specialized domains. Used for Airbus A380/A340/A350 flight control software certification.
Apron
C + OCaml/Java bindings • LGPL • 144 stars
The standard numerical abstract domain library. Unified API across intervals (boxes), octagons, polyhedra (newpolka), zonotopes. "One-line change" to switch domains. Used as backend by IKOS, Crab, Sparrow, Goblint.
ELINA (ETH Zurich)
C++ • MIT • 137 stars
Optimized Apron-compatible implementation. Online decomposition, Intel AVX vectorization, GPU acceleration (CUDA). Significantly faster than Apron. Specialized neural network domains: DeepZ, DeepPoly.
SPARTA (Meta)
C++ (header-only) + Rust • MIT • 666 stars
Composable toolkit for building analyzers. AbstractDomain CRTP base, monotonic fixpoint iterator (WPO), set/map/product/lifted domains. Three pillars: Semantics, Partitioning, Abstraction. Powers ReDex (Android optimizer).
Crab (SeaHorn)
C++ • Apache-2.0 • 250 stars
Abstract interpretation framework for LLVM. CrabIR (three-address code). Intervals, congruences, zones, octagons, disjunctive intervals (LDDs). Bourdoncle WTO fixpoint. Integrates Apron/ELINA/PPLite. Forward + backward analysis.
PPL (Parma Polyhedra Library)
C++ • GPL-3.0 • Established
Exact-arithmetic convex polyhedra, grids, finite powersets, products. Simplex-based. C/Java/OCaml/Prolog interfaces. TUV SUD-certified commercial variant (BPL). The gold standard for polyhedral operations.
LiSA
Java • MIT • 74 stars
Modular multi-language static analysis framework. Language-agnostic core with generic extensible CFG. Users write parsers to translate to LiSA's CFG. Powers JLiSA (Java, SV-COMP 2026 bronze) and EVMLiSA (EVM/Solidity).
TAJS (Aarhus University)
Java • JavaScript ES3/5 • Apache-2.0 • 200 stars (archived)
Type Analysis for JavaScript. Custom flow graph IR (~20 instruction types). Product lattice Value = Undef × Null × Bool × Num × String × P(L). Allocation-site based memory, recency abstraction, on-the-fly call graph. Successor: Jelly.
MIRAI (Meta → Endorlabs)
Rust • Rust via MIR • MIT • 1k stars
Path-sensitive abstract interpreter for Rust MIR. Panic detection, correctness verification, taint analysis, constant-time analysis. Originally Facebook, now maintained by Endorlabs (v1.1.12, March 2025).
PhASAR (Univ. Paderborn)
C++ • Any LLVM language • MIT • 1k stars
IFDS/IDE/monotone data-flow solvers for LLVM IR. Users specify data-flow problems; framework solves automatically. Points-to analysis, call graph construction.
Tai-e (Nanjing University)
Java • Java • LGPL/GPL • 1.8k stars
Modern Java analysis framework. Pointer analysis, taint analysis, constant propagation, live variables, dead code, reflection, lambda support. Plugin system. ISSTA'23 paper. Very active.
Pyre/Pysa (Meta)
OCaml • Python • MIT • 7.2k stars
Pyre = type checker (PEP 484). Pysa = taint analysis for security. Data flow tracking across function boundaries. Transitioning to successor Pyrefly.
JsShaker
Rust • JavaScript • Open source • 413 stars
Abstract interpretation for JS code size optimization. Constant propagation, dead code/branch elimination, object property tracking. "Soundy" — pragmatic unsoundness for eval, prototypes. Up to 25% size reduction.
Sturdy
Haskell → Scala • BSD-3 • 63 stars
Generic interpreter parameterized by language semantics. Each analysis component is independently provably sound. Supports PCF, While, Scheme, Stratego, LambdaJS, Jimple. Reimplemented in Scala.
JIPDA
JavaScript • JS (ES3) • Academic • 22 stars
Abstract CESK-style interpreter using Abstracting Abstract Machines (AAM). Flow-sensitive store, split stack, abstract counting, integrated GC. Research prototype.
flowR
TypeScript • R language • Open source • 86 stars
The only significant TypeScript-based analyzer. Static dataflow analysis, program slicing, dependency analysis, call graphs for R. VSCode extension, REPL, Docker support. 4.7k commits.
Sparrow (Seoul Nat'l Univ.)
OCaml • C • Academic • 127 stars
Sound C analyzer for buffer overflows. Uses Apron for numerical domains. ML-based alarm filtering for precision. Partially flow-sensitive with configurable precision. Commercial variant exists.
CodeHawk
OCaml • Binary/C/Java • MIT • 74 stars
Language-independent AI engine with separate frontends. Analyzes x86/MIPS binaries, C source, Java bytecode. Based directly on Cousot's framework.
YASA (Ant Group)
TypeScript • Multi-language • Apache-2.0 • 254 stars
Taint/dataflow analysis engine. Unified AST representation. Field/context/object/path/flow-sensitive. Not purely abstract interpretation but uses related techniques. Active.

Key finding: OCaml dominates (IKOS aside) due to its algebraic data types and module system. C++ is used for performance-critical domain libraries. Zero general-purpose abstract interpretation frameworks exist in TypeScript or JavaScript. The closest are flowR (TypeScript, analyzes R) and YASA (TypeScript, taint analysis). This is wide open territory.

3The Design Decision Space

Eight key architectural decisions, the spectrum of options for each, and what existing implementations have chosen.

1Program Representation (IR)
▼

The choice of intermediate representation is the first major architectural fork. It determines what the analysis sees and what precision is possible.

OptionUsed ByTradeoffs
AST (no lowering) MOPSA Preserves all source-level information. Requires domains to handle high-level constructs. Avoids precision loss from lowering. Unique among major frameworks.
Custom flow graph TAJS ~20 instruction types tailored to analysis. Call/return edges added dynamically during fixpoint. Decouples from source syntax while preserving semantic intent.
Three-address code Crab (CrabIR), IKOS (AR) Simple, strongly typed. Easy to analyze. Decouples from source language. Requires lowering pass.
Compiler IR (LLVM/MIR) IKOS, Clam, MIRAI Leverages compiler infrastructure. Ties to specific compiler ecosystem. May lose source-level info.
Decorated tree (AG-style) KSC (current) Tree-structured with parent/child edges. Information flows syn/inh. Natural for syntax-directed analysis. Limited to tree topology.

Our choice: A hybrid AST + flow graph approach. Keep the AST as the primary representation (like MOPSA) to preserve source-level information and compatibility with KSC's tree-structured AG, but build a lightweight CFG overlay for control flow analysis. This gives us both tree-structured (AG) and graph-structured (AI) information flow.

2Abstract Domain Interface
▼

How domains are defined and composed. The contract every domain must satisfy.

PatternUsed ByHow
CRTP (Curiously Recurring Template) SPARTA, IKOS class D extends AbstractDomain<D> — compile-time polymorphism, zero vtable overhead. SPARTA provides scaffolding that handles Top/Bottom boilerplate.
Manager/Uniform API Apron Manager object dispatches to concrete backends. "One-line change" to switch domains. Runtime polymorphism.
OCaml modules/functors Infer, Frama-C, MOPSA Module types as contracts, functors for composition. Powerful but requires OCaml's module system.
Java interfaces LiSA, Tai-e Standard OOP hierarchy. Familiar, less type-safe than modules.
TypeScript interfaces (none yet) Structural typing, generics, discriminated unions. Natural fit for lattice modeling. No CRTP but similar safety via generic constraints.

Our choice: TypeScript interface + generic constraints. Immutable value semantics (return new instances from join/widen). Discriminated unions for lattice elements. Map-based abstract environments with implicit-Top semantics (following SPARTA's AbstractEnvironment pattern).

3Domain Composition Strategy
▼

How to combine multiple domains into a richer analysis. This is one of the most critical decisions.

StrategyUsed ByTradeoffs
Direct product SPARTA, most frameworks Componentwise operations. No cross-domain info flow. Simple, always terminates. The starting point.
Reduced product SPARTA, Crab, EVA Adds reduce_product() for cross-domain propagation. Must be triggered explicitly (not automatic — can break termination). SPARTA: smash-bottom semantics (any bottom → all bottom).
Cofibered domains MOPSA Stack-based composition via expression rewriting. Each domain handles its constructs. Low coupling, high extensibility. Novel but unproven outside MOPSA.
Attribute product (AG-style) KSC, JastAdd Each attribute is an independent "domain." Composition is the set of all attributes. Inter-attribute dependencies are explicit (equation refs). Natural for tree-structured analysis.

Our choice: Direct product as the default, with reduced product via explicit reduce() for cross-domain propagation. The attribute product (AG-style) is a special case we preserve for backward compatibility with KSC. Long-term, explore cofibered composition for multi-language analysis.

4Fixpoint Computation Strategy
▼

How to iterate to a fixpoint. The scheduling strategy determines convergence speed and precision.

StrategyUsed ByTradeoffs
Chaotic iteration Cousot & Cousot (1977) Process nodes in arbitrary order until convergence. Simple but slow. Widening everywhere.
Worklist TAJS, many tools Dirty nodes in a queue. More efficient than chaotic. Order matters for performance.
WTO (Weak Topological Ordering) IKOS, Crab, Astrée, EVA State of the art. Bourdoncle's 1993 algorithm. Hierarchical cycle/vertex structure. Widening only at loop headers. Nested loops handled by nested iteration.
WPO (Weak Partial Ordering) SPARTA Similar to WTO but non-recursive. Avoids stack overflow for deeply nested structures. Supports parallel computation.
Demand-driven (AG-style) KSC, JastAdd Attribute values computed on demand and cached. No explicit iteration — cache + recursion handle fixpoint. Natural for tree-structured problems.
Constraint system (side-effecting) Goblint Local/global unknowns with side effects. Multiple solver strategies. Handles concurrent analysis naturally.

Our choice: WTO/Bourdoncle for graph-based (CFG) analysis + demand-driven for tree-based (AG) analysis. The framework supports both modes: when the program model is a tree, use demand-driven evaluation (preserving KSC's behavior); when it's a CFG, use WTO iteration. Interleaved ascending (widening) and descending (narrowing) phases.

5Widening & Convergence
▼

How to ensure termination for infinite-height domains while preserving as much precision as possible.

StrategyDescriptionPrevalence
Standard widening If bounds diverge, widen to ∞. Fastest convergence, lowest precision. Universal baseline
Delayed widening Apply join for first N iterations, then switch to widening. Captures initial loop behavior precisely. IKOS, Astrée, EVA
Widening with thresholds Instead of ∞, widen to nearest constant from the program (loop bounds, array sizes, etc.). Nearly universal (Crab, IKOS, Astrée, EVA)
Lookahead widening Considers future iterations when choosing widening targets. Crab
Localized widening Apply widening only at cycle entries (loop headers via WTO), not everywhere. IKOS, standard practice
Narrowing (descending) iterations After ascending fixpoint stabilizes, apply narrowing to recover precision. IKOS, EVA, Crab, Astrée
No widening needed (finite domains) For finite-height lattices, Kleene iteration terminates naturally. KSC (AG), type systems

Our choice: Delayed widening (join for first N iterations) + widening with thresholds (extract constants from the program) + localized widening (only at WTO cycle entries) + narrowing iterations after stabilization. For tree-based AG analysis, finite-domain Kleene iteration (no widening needed, as in current KSC).

6Interprocedural Analysis
▼

How to handle function calls, closures, and modules. Critical for real-world codebases.

StrategyUsed ByTradeoffs
Compositional / summaries Infer (bi-abduction) Analyze procedures independently. Compute pre/post summaries. Scales to huge codebases. Enables incrementality. Complex to implement.
Top-down + memoization Crab, IKOS Analyze callees in context, cache results. Handles recursion via memo tables. Good precision, moderate scalability.
Inlining IKOS, Astrée, EVA Conceptually inline callees. Most precise but expensive. Context-sensitive by construction.
On-the-fly call graph TAJS Add call/return edges during fixpoint as data flow reveals callable targets. Essential for dynamic languages (JS/TS).
Whole-tree (AG-style) KSC, JastAdd The entire program tree is available. No inter-procedural boundary — attributes flow through the whole tree. Limited to single-file or merged trees.

Our choice: Top-down with memoization as the initial strategy, with on-the-fly call graph construction for dynamic dispatch resolution (essential for TS/JS). Long-term goal: Infer-style compositional analysis for scalability.

7Memory / Heap Model
▼

How to abstract heap-allocated objects, closures, and reference semantics.

ModelUsed ByTradeoffs
Allocation-site based TAJS, IKOS Objects abstracted by where they were created. P(L) tracks which sites a reference may point to. Good for OO/prototype languages.
Recency abstraction TAJS Distinguish "most recent" allocation from "summary" of all older allocations at a site. More precise for loops that create objects.
Separation logic Infer Heap as separating conjunction of predicates. Enables compositional reasoning (frame rule). Complex but very powerful.
Cell-based IKOS Memory divided into cells (variable, offset, size). Good for C-style pointer arithmetic.
Flat / variable-based Simple analyses, KSC Each variable maps to an abstract value. No heap modeling. Sufficient for tree-structured AG analysis.

Our choice: Allocation-site based with recency abstraction, following TAJS's proven approach for JavaScript. Objects modeled as abstract property maps with default_index (array-like properties) and default_other (all other string properties) for dynamic access.

8Extensibility & Modularity
▼

How users extend the framework with new domains, analyses, and language frontends.

PatternUsed ByTradeoffs
Plugin architecture Frama-C, LiSA, Tai-e Formal extension points. Well-defined lifecycle. Can feel heavyweight for simple extensions.
Composable toolkit SPARTA, Crab Users build analyzers from composable primitives. Maximum flexibility. Requires understanding of the primitives.
Small collaborative domains MOPSA Each domain is small, handles one concern. Domains collaborate via expression rewriting. Easy to add new domains.
Checker-based Infer Each analysis is an independent checker. Separate abstractions per analysis kind.
Parameterized semantics Sturdy Generic interpreter parameterized by semantics interfaces. Each analysis provides its own implementation. Compositionally sound.
Spec-driven (AG-style) KSC Users define AnalysisSpec (attributes + equations). Framework generates dispatch. Declarative, high-level.

Our choice: Composable toolkit (SPARTA-style) as the core library, with a spec-driven (KSC-style) declarative layer on top. Users can work at either level: low-level domain composition for custom analyses, or high-level AnalysisSpec declarations for tree-structured analyses. The spec-driven layer compiles down to the toolkit primitives.

SPARTA — Three Pillars Architecture

SPARTA (Meta) is the closest to what we want: a composable toolkit for building analyzers, not a monolithic tool.

Semantics Layer
What properties to track — abstract domain definitions

SPARTA provides a library of abstract domains via CRTP:

  • ConstantAbstractDomain — singleton constants or top/bottom
  • IntervalDomain — integer intervals with widening
  • SetAbstractDomain (Hashed, SmallSorted, Sparse, OverUnder variants)
  • AbstractEnvironment — variable-to-value map with implicit Top for absent keys
  • DirectProductAbstractDomain — tuple of domains, componentwise
  • ReducedProductAbstractDomain — with explicit reduce_product()
  • LiftedDomain — adds new bottom below existing domain

Key pattern: AbstractDomainScaffolding<D> handles Top/Bottom boilerplate. Derived class returns AbstractValueKind enum (Bottom, Value, Top) from operations.

↓
Partitioning Layer
At what granularity to track properties — per-block, per-instruction, per-context

Partitioning determines the analysis state shape. An AbstractPartition maps partition keys to abstract values. Context sensitivity is achieved by partitioning the state space by calling context. SPARTA lets you choose partitioning independently from domains.

↓
Abstraction Layer
How to represent abstract values efficiently — data structures and algorithms

Implementation-level concerns: Patricia trees for environments, hash maps for sets, bitwise representations for small domains. SPARTA's PatriciaTreeMapAbstractEnvironment provides efficient persistent maps.

+ Composable + Reusable domains + MIT license - C++ only - No built-in fixpoint (WPO is separate)

MOPSA — Cofibered Domain Composition

MOPSA's innovation: operate directly on ASTs (no CFG lowering) with a stack of small, collaborative domains that communicate through expression rewriting.

Language Frontend
Parse source to AST. Currently: C (via Clang), Python (custom parser)

MOPSA processes undesugared ASTs. Complex constructs like for loops or method calls are present in the AST and handled by specific domains, not lowered away.

↓
Domain Stack
Multiple small domains, each handling specific constructs

Each domain handles AST nodes it understands and rewrites others into simpler forms. Example: a "for-loop domain" rewrites for into while; a "while domain" rewrites to conditional jumps; a "conditional domain" forks the abstract state. The expression/statement domains form a collaboration stack.

This is the cofibered domain approach — each domain is small, focused, and independently testable. New language constructs are handled by adding a new domain, not modifying existing ones.

↓
Numerical Core
Base numerical domains that handle arithmetic expressions

At the bottom of the stack, standard numerical domains (intervals, octagons) handle the simple arithmetic that everything has been rewritten into. The rewriting chain eventually reaches these ground-level domains.

+ AST-based (preserves source info) + Highly modular + Easy to extend + Multi-language - OCaml only - Novel (less proven than CFG approach)

TAJS — JS-Specific Abstract Interpretation

TAJS is the most relevant precedent for analyzing JavaScript. Its design choices directly inform our approach.

Custom Flow Graph IR
~20 instruction types designed specifically for JS analysis

TAJS decomposes JavaScript into primitives: DeclareVariable, ReadProperty, WriteProperty, Call, Construct, AfterCall, Return, BinaryOp, UnaryOp, ConstNode, NewObject, Phi, BeginFor/EndFor, etc. Call/return edges are added during fixpoint iteration as analysis discovers callable targets.

↓
JS Value Lattice
Value = Undef × Null × Bool × Num × String × P(L)

Each JS value is a product of type-specific sub-lattices:

  • Bool: {⊥, true, false, Bool}
  • Num: singleton constants + UInt + NotUInt + Num + NaN + Inf
  • String: singleton constants + IdentifierParts + Other
  • P(L): powerset of allocation-site labels (objects, functions, arrays)

Key insight: The lattice "evolved in a number of steps driven by an observed lack of precision on small test cases" — iterative design is essential. We should expect the same.

↓
Abstract Heap
Allocation-site + recency abstraction + default properties

Objects: Obj = (P → Value × Absent × Attributes × Modified) × P(ScopeChain). Four special properties: [[Prototype]], [[Value]], default_index (array-like), default_other (all other strings). Recency abstraction: "most recent" allocation from a site is tracked separately from "summary" of all older allocations.

+ JS-specific precision + Proven for JS + Recency abstraction - Java only - Monolithic (hard to reuse) - Archived (succeeded by Jelly)

Infer — Compositional Modular Analysis

Infer's key innovation: compositional analysis where each procedure is analyzed independently, producing summaries that are applied at call sites.

Bi-Abduction Engine
Automatically discovers pre/post specs via separation logic

Given A * ?antiframe |- B * ?frame, bi-abduction solves for both the antiframe (what the function needs but doesn't have) and the frame (what it doesn't touch). This enables:

  • Procedures analyzed independently — no whole-program analysis needed
  • Only modified procedures need re-analysis on code changes
  • Summaries are inferred automatically, not manually specified
↓
Independent Checkers
Pulse, RacerD, InferBO, Cost, Starvation — each is a separate engine

Each checker has its own abstract domain and analysis strategy. They share infrastructure (SIL IR, procedure summaries) but are architecturally independent. This enables adding new analyses without modifying existing ones.

↓
Incremental Pipeline
Only re-analyze changed procedures + dependents

Because analysis is compositional, Infer can incrementally re-analyze only the procedures that changed and those that depend on changed summaries. This makes it practical for CI/CD on massive codebases (Meta scale).

+ Scales to enormous codebases + Incremental + Compositional - Complex (separation logic) - OCaml only

Sturdy — Parameterized Generic Interpreter

Sturdy takes the Abstracting Abstract Machines (AAM) approach: write a concrete interpreter, then systematically parameterize it to produce abstract interpreters.

Language Semantics Interfaces
Abstract interfaces for language operations (eval, apply, store, env)

The key idea: define interfaces for eval, apply, store, env operations. A concrete implementation gives you a standard interpreter. An abstract implementation gives you a static analyzer. The same interpreter skeleton works for both.

Supports: PCF, While, Scheme, Stratego, LambdaJS, Jimple. Each language provides its own semantics implementation.

↓
Analysis Components
Independently composable + provably sound analysis modules

Each component (store abstraction, environment abstraction, control flow, etc.) is independently provably sound. They compose freely, and the composition is sound by construction. This is the most principled approach to building abstract interpreters.

↓
Generic Interpreter
Parameterized by semantics → instantiate for concrete or abstract execution

The interpreter is generic over its semantics interfaces. Plug in concrete semantics → you get a standard interpreter. Plug in abstract semantics → you get an analyzer. This is exactly the "parameterize an AI to get an AG" idea from our earlier research.

+ Compositional soundness + Concrete ↔ abstract duality + Multi-language - Haskell/Scala (niche) - Academic (not production-tested)

5TypeScript Considerations

Advantages, challenges, and recommended patterns for implementing abstract interpretation in TypeScript.

ADVANTAGES

Structural typing — naturally models domain interfaces without CRTP
Discriminated unions — ideal for lattice elements (kind: 'top' | 'bottom' | 'value')
Readonly types / as const — enforce immutability for abstract values
Map/Set built-ins — natural for abstract environments and stores
Generic constraints — enforce domain contracts at compile time
Same ecosystem as target — analyzing TS/JS with TS means shared AST types

vs

CHALLENGES

No value types — every domain element is heap-allocated (perf implications for inner loops)
Generic erasure — no runtime domain dispatch via generics; need registry or class hierarchy
IEEE 754 only — no directed rounding for sound float analysis; need BigInt for integer precision
GC pauses — can affect fixpoint iteration timing; persistent data structures help
Single-threaded — parallel fixpoint requires Workers (complex orchestration)
No operator overloading — a.join(b) not a | b

Recommended Patterns

// ── Abstract Domain Interface (TypeScript) ──────────────────

interface AbstractDomain<D extends AbstractDomain<D>> {
  join(other: D): D;       // least upper bound
  meet(other: D): D;       // greatest lower bound
  widen(other: D): D;      // convergence acceleration
  narrow(other: D): D;     // precision recovery
  leq(other: D): boolean;   // partial order check
  isBottom(): boolean;
  isTop(): boolean;
}

// ── Interval Domain (discriminated union pattern) ───────────

type Interval =
  | { readonly kind: 'bottom' }
  | { readonly kind: 'top' }
  | { readonly kind: 'range'; readonly lo: number; readonly hi: number };

// ── Abstract Environment (Map-based, implicit Top) ──────────

class AbstractEnv<V extends AbstractDomain<V>> {
  private readonly map: ReadonlyMap<string, V>;
  // Absent key → Top (variable not yet constrained)
  get(key: string): V { return this.map.get(key) ?? this.topValue; }
  set(key: string, val: V): AbstractEnv<V> {
    return new AbstractEnv(new Map([...this.map, [key, val]]), this.topValue);
  }
}

Immutability is key: All domain operations return new instances rather than mutating. This is more natural in TypeScript than C++-style mutation, enables structural sharing, and makes the framework compatible with persistent data structures and caching strategies.

6Proposed Architecture

A layered TypeScript architecture that unifies graph-based (AI) and tree-based (AG) analysis in a single framework.

Entry Points / CLI
Thin shells: CLI, programmatic API, IDE plugin interface

Thin entry points that delegate to the application layer. CLI for running analyses, programmatic API for IDE integrations and custom tooling. Same pattern as KSC's apps/cli/ and packages/core/.

↓
Application Layer
Use cases: analyze-program, check-file, extract-types, run-codegen

Orchestrates analysis pipelines. Wires together frontends, domain configurations, and fixpoint engines. Manages configuration, output formatting, and reporting. Follows KSC's application/ layer pattern.

↓
Analysis Adapters
Concrete analysis specs: type-checking, taint, complexity, null-safety

Each analysis is an adapter that provides domain configuration, transfer functions, and result projections. Follows KSC's adapters/analysis/spec/ pattern. Adapters can define tree-structured (AG) or graph-structured (AI) analyses — or both.

↓
Language Adapters
Frontends: TS/JS parser, Python parser, custom language frontends

Core Engine (Ports)
Generic machinery: domain library, fixpoint solvers, program models

The heart of the framework. Five sub-modules:

  • domains/ — abstract domain interfaces + built-in domains (intervals, sets, environments, products)
  • fixpoint/ — WTO computation, fixpoint iterators (ascending/descending), worklist engine
  • program-model/ — AST nodes, CFG builder, flow graph, program points
  • transfer/ — transfer function interfaces, abstract semantics for built-in operations
  • tree-evaluator/ — demand-driven AG evaluator (evolved from KSC's libs/evaluator/)

All of these are ports — generic machinery with no adapter imports. They define contracts that adapters implement.

The unifying insight: KSC's tree evaluator becomes one mode of the framework. When the program model is a tree (AST), use demand-driven evaluation (preserving KSC's current behavior exactly). When the program model is a CFG, use WTO fixpoint iteration. Both modes share the same domain library, same abstract values, same composition machinery. An analysis can use both modes simultaneously — tree-structured attributes for syntax-directed properties, CFG-based fixpoint for control flow-sensitive properties.

Domain Port Interfaces

// ── Core lattice contract ──────────────────────────────────

interface Lattice<D> {
  readonly bottom: D;
  readonly top: D;
  join(a: D, b: D): D;
  meet(a: D, b: D): D;
  leq(a: D, b: D): boolean;
  eq(a: D, b: D): boolean;
}

// ── Abstract domain = lattice + widening/narrowing ────────

interface AbstractDomain<D> extends Lattice<D> {
  widen(prev: D, next: D): D;
  narrow(prev: D, next: D): D;
}

// ── Domain combinators (following SPARTA) ─────────────────

function directProduct<A, B>(
  domA: AbstractDomain<A>,
  domB: AbstractDomain<B>
): AbstractDomain<[A, B]>;

function reducedProduct<A, B>(
  domA: AbstractDomain<A>,
  domB: AbstractDomain<B>,
  reduce: (a: A, b: B) => [A, B]
): AbstractDomain<[A, B]>;

function liftedDomain<D>(
  inner: AbstractDomain<D>
): AbstractDomain<D | null>; // null = new bottom

function powerset<E>(
  maxSize: number  // widen to top when |set| exceeds this
): AbstractDomain<ReadonlySet<E> | 'top'>;

Design choice: Domains are externalized (operations live on a separate AbstractDomain<D> object, not on the value itself). This follows the Apron/SPARTA pattern and avoids bloating every abstract value with method implementations. The domain object is a type class — a bundle of operations parameterized by the value type.

Fixpoint Engine

// ── Flow graph contract ────────────────────────────────────

interface FlowGraph<N> {
  readonly entry: N;
  readonly exit: N;
  predecessors(node: N): ReadonlyArray<N>;
  successors(node: N): ReadonlyArray<N>;
}

// ── WTO node (Bourdoncle hierarchy) ───────────────────────

type WtoElement<N> =
  | { kind: 'vertex'; node: N }
  | { kind: 'cycle';  head: N; body: WtoElement<N>[] };

function computeWTO<N>(graph: FlowGraph<N>): WtoElement<N>[];

// ── Fixpoint iterator ─────────────────────────────────────

interface FixpointConfig<N, D> {
  graph: FlowGraph<N>;
  domain: AbstractDomain<D>;
  transfer: (node: N, state: D) => D;
  // Widening config
  widenDelay?: number;           // join for first N iters, then widen
  thresholds?: ReadonlySet<number>; // constants for widening targets
  narrowIterations?: number;    // max descending iterations
}

interface FixpointResult<N, D> {
  stateAt(node: N): D;
  readonly iterations: number;
  readonly wideningPoints: ReadonlySet<N>;
}

function solveFixpoint<N, D>(
  config: FixpointConfig<N, D>
): FixpointResult<N, D>;

Key design: The WTO is a discriminated union tree — WtoElement is either a plain vertex or a cycle containing nested elements. The fixpoint iterator walks this tree: vertices are processed in a single pass; cycles trigger iterative widening + narrowing loops. This is exactly Bourdoncle's 1993 algorithm translated into TypeScript's type system.

Program Model

// ── Program point (node in the flow graph) ────────────────

interface ProgramPoint {
  readonly id: number;
  readonly kind: InstructionKind;
  readonly astNode?: ASTNode;    // back-reference to source
}

// ── Instruction set (following TAJS's ~20-type approach) ──

type InstructionKind =
  | 'declare-variable'
  | 'read-variable'
  | 'write-variable'
  | 'read-property'
  | 'write-property'
  | 'delete-property'
  | 'call'
  | 'construct'
  | 'return'
  | 'throw'
  | 'binary-op'
  | 'unary-op'
  | 'constant'
  | 'new-object'
  | 'new-array'
  | 'new-function'
  | 'typeof'
  | 'phi'        // join point
  | 'nop'        // no-op (entry/exit)
  | 'assume';    // branch condition guard

// ── CFG builder ───────────────────────────────────────────

interface CFGBuilder<AST> {
  buildCFG(ast: AST): FlowGraph<ProgramPoint>;
}

// ── Dual model: tree + graph ───────────────────────────────

interface ProgramModel<AST> {
  readonly tree: AST;                           // for AG-style analysis
  readonly cfg: FlowGraph<ProgramPoint>;       // for AI-style analysis
  pointToNode(pt: ProgramPoint): AST | undefined;  // cross-reference
  nodeToPoints(node: AST): ProgramPoint[];       // reverse map
}

Dual representation: The ProgramModel holds both the tree (AST) and the graph (CFG), with cross-references between them. This is the key to unifying AG and AI analysis — tree-structured attributes reference the AST, while control-flow analysis operates on the CFG, and results from either side can inform the other.

Transfer Function Interface

// ── Transfer function (graph-based analysis) ──────────────

interface TransferFunction<D> {
  /** Apply transfer function at a program point */
  transfer(point: ProgramPoint, state: D): D;
}

// ── Abstract semantics (language-specific operations) ──────

interface AbstractSemantics<V> {
  // Arithmetic
  binaryOp(op: BinaryOp, left: V, right: V): V;
  unaryOp(op: UnaryOp, operand: V): V;

  // Object model
  readProperty(obj: V, prop: V): V;
  writeProperty(obj: V, prop: V, val: V): V;

  // Control flow
  assume(cond: V, trueBranch: boolean, state: V): V;

  // Coercion (JS-specific)
  toBoolean(val: V): V;
  toNumber(val: V): V;
  toString(val: V): V;
}

// ── Tree equations (AG-style, compatible with KSC) ────────

interface TreeTransfer<Ctx> {
  /** Synthesized: computed from children */
  syn(ctx: Ctx): unknown;
  /** Inherited: computed from parent */
  inh(ctx: Ctx): unknown | undefined;
  /** Collection: fold over children */
  collect(acc: unknown, child: unknown): unknown;
}

Two transfer modes: TransferFunction<D> for graph-based (AI) analysis operates on program points and abstract states. TreeTransfer<Ctx> for tree-based (AG) analysis uses KSC's familiar syn/inh/collection pattern. An analysis can provide either or both, and the framework dispatches accordingly based on the program model mode.

Tree Evaluator (Evolved from KSC)

// ── This is KSC's evaluator, evolved into the AI framework ──
// Key change: AGNode now has access to CFG-derived info too

interface DualModeNode extends AGNodeInterface {
  /** Tree-based attribute access (KSC-compatible) */
  attr(name: string, ...args: unknown[]): any;

  /** CFG-based abstract state at this node's program point */
  cfgState<D>(analysisId: string): D;

  /** Back-reference to program model */
  readonly programModel: ProgramModel<ASTNode>;
}

// ── Unified evaluator ─────────────────────────────────────

interface UnifiedEvaluator<M, P> {
  /** Full analysis: tree eval + CFG fixpoint + projections */
  analyze(root: ASTNode): P;

  /** Tree-only (backward compat with KSC's evaluate()) */
  evaluate(root: ASTNode): P;

  /** Build dual-mode tree for inspection */
  buildTree(root: ASTNode): TypedAGNode<M>;
}

Backward compatible: The UnifiedEvaluator preserves KSC's existing evaluate() and buildTree() APIs. Existing KSC code continues to work unchanged. The new analyze() method adds CFG-based analysis on top. DualModeNode extends AGNodeInterface so tree equations can optionally access CFG-derived information (e.g., flow-sensitive types) via cfgState().

9KSC Integration Path

How the new framework relates to and evolves from KSC's existing attribute grammar system.

KSC's current AG system is not being replaced — it becomes a specialized mode within the broader framework. The migration is incremental and backward-compatible.

Phase 1: Core Domain Library FOUNDATION

Implement the domain library (AbstractDomain, Lattice, combinators) and the fixpoint engine (WTO, interleaved iterator) as standalone modules in ai/core/. These have zero dependency on KSC's existing code. They can be used independently for any analysis.

What stays unchanged: All of KSC's current code. The AI framework is additive only.

+ Zero KSC changes + Independently testable + Usable for non-KSC analysis
Phase 2: Shared Program Model BRIDGING

Create the ProgramModel interface that holds both tree (AST) and graph (CFG) views. Build a CFG constructor from KSC's existing AST nodes. Implement cross-references between tree nodes and program points.

What changes: KSC's AST nodes gain optional programPoint back-references. The existing ASTNode type gets a small extension. No breaking changes.

+ Tree + graph views unified + Backward compatible ~ Small ASTNode extension
Phase 3: Dual-Mode Evaluator UNIFICATION

Evolve KSC's AGNode into DualModeNode that can access both tree attributes (existing) and CFG-derived states (new). The UnifiedEvaluator wraps the existing evaluator and adds CFG fixpoint solving as a preprocessing step.

What changes: AGNode gains a cfgState() method. Existing attr() continues to work identically. The evaluate() method is preserved.

+ AG equations can access CFG results + Full backward compat + Incremental adoption
Phase 4: New Analyses EXPANSION

Build new analyses that use the full AI framework: null-safety (requires control-flow sensitivity), taint analysis (requires interprocedural data flow), complexity analysis (requires loop analysis). These analyses can combine AG-style tree attributes with CFG-based fixpoint results in a single analysis spec.

What changes: New adapter directories in ai/adapters/analysis/. KSC's existing kind-checking analysis is unaffected.

+ Null safety, taint, complexity analyses + Mixed AG+AI analysis specs + No KSC regressions

Q

Why is Bourdoncle's Weak Topological Ordering (WTO) preferred over naive chaotic iteration for fixpoint computation?

A

WTO organizes the CFG into a hierarchical cycle/vertex structure. Widening is applied only at cycle entry points (loop headers), not everywhere. This maximizes precision for acyclic code (single-pass, no widening) while ensuring termination for loops. Nested loops are handled by nested iteration. Most production frameworks (IKOS, Crab, Astrée) use WTO.

Q

What is a reduced product domain, and how does it differ from a direct product?

A

Both combine multiple domains into a tuple with componentwise operations. A direct product has no cross-domain information flow. A reduced product adds a reduce() function that propagates information between components (e.g., if one component proves a variable is zero, reduce the interval in the other component). Reduction must be triggered explicitly to avoid breaking termination.

Q

Why are there no production-quality abstract interpretation frameworks in TypeScript? What are the main challenges?

A

Key challenges: (1) No value types — every domain element is heap-allocated, affecting inner-loop performance. (2) Generic erasure — no runtime dispatch on generic types. (3) IEEE 754 only — no directed rounding for sound float analysis. (4) Single-threaded — parallel fixpoint requires Workers. (5) GC pauses affect iteration timing. However, TS advantages (structural typing, discriminated unions, readonly types) partially compensate.

Q

How does MOPSA's "cofibered domain" composition differ from traditional reduced products?

A

In MOPSA, domains form a collaborative stack where each domain handles AST constructs it understands and rewrites others into simpler forms for lower domains. A for-loop domain rewrites to while; a while domain rewrites to conditionals; conditionals fork the abstract state. This is fundamentally different from products (which combine values) — cofibered domains collaborate on control flow and expression evaluation, not just value abstraction.

Q

What is TAJS's approach to modeling JavaScript values, and why is it a product lattice rather than a single numerical domain?

A

TAJS models JS values as Value = Undef × Null × Bool × Num × String × P(L) because JavaScript is dynamically typed — a variable can hold any type. Each component has its own sub-lattice (Bool: {⊥, true, false, Bool}; Num: singletons + UInt + NotUInt + Num + special values). The product captures "this variable might be a number OR a string" precisely, which a single numerical domain cannot express.

Q

How does the proposed framework unify KSC's demand-driven AG evaluation with graph-based AI fixpoint computation?

A

The framework uses a ProgramModel that holds both a tree (AST) and a graph (CFG) with cross-references. The DualModeNode extends AGNode with a cfgState() method. Tree-structured attributes use demand-driven evaluation (existing KSC behavior). CFG-based analyses use WTO fixpoint iteration. Both results are available at each node — tree equations can read CFG states and vice versa. evaluate() is preserved for backward compatibility.

Q

What is widening with thresholds, and why is it nearly universal in production abstract interpreters?

A

Standard widening jumps to ±∞ when interval bounds diverge, which loses all precision. Widening with thresholds instead widens to the nearest constant found in the program (loop bounds, array sizes, comparison constants). For example, if a loop counts i from 0 and compares i < 10, the threshold set includes 10, so widening produces [0, 10] instead of [0, +∞]. This recovers most precision at almost no cost, which is why nearly every production framework uses it.

Q

What makes Infer's compositional analysis approach unique, and what is bi-abduction?

A

Infer analyzes each procedure independently using separation logic. Bi-abduction solves A * ?antiframe |- B * ?frame to automatically discover both what a function needs (antiframe/precondition) and what it doesn't touch (frame). This enables: (1) no whole-program analysis needed, (2) only modified procedures re-analyzed on changes, (3) specs are inferred, not manual. This is why Infer scales to Meta's enormous codebases.

Q

Why does the proposed design use externalized domains (operations on a separate object) rather than methods on values?

A

Externalized domains follow the type class pattern: a separate AbstractDomain<D> object holds operations (join, widen, leq) parameterized by value type D. This avoids bloating every abstract value with method implementations, enables values to be simple immutable data (discriminated unions, tuples), and makes domain composition (products, lifts) work on types that don't know about each other. It follows Apron's manager pattern and SPARTA's CRTP scaffolding pattern.

Q

What is recency abstraction in heap modeling, and which framework pioneered it?

A

Recency abstraction (from TAJS) distinguishes between the most recently allocated object at an allocation site and a summary of all older allocations from that site. This is crucial for loops that create objects: without recency, all loop-created objects are merged into one imprecise summary. With recency, the current iteration's object is tracked precisely while older ones are summarized. This dramatically improves precision for OO/prototype-based languages.

Q

Which implementation language dominates in the abstract interpretation ecosystem, and why?

A

OCaml dominates: Infer, Frama-C, MOPSA, Goblint, Sparrow, Astrée, CodeHawk are all OCaml. Reasons: (1) algebraic data types are natural for AST/lattice modeling, (2) the module system (functors) enables domain composition elegantly, (3) pattern matching simplifies transfer function dispatch, (4) strong typing catches lattice errors at compile time, (5) historical momentum from Cousot's group at ENS (Paris). C++ is used for domain libraries (Apron, ELINA, PPL, SPARTA) where raw performance matters.

Q

What is the key difference between Sturdy's "parameterized generic interpreter" approach and traditional abstract interpretation?

A

Traditional AI starts from a concrete semantics and constructs an abstraction via Galois connections. Sturdy instead writes a single generic interpreter parameterized by interfaces (eval, apply, store, env). A concrete implementation of these interfaces gives a standard interpreter; an abstract implementation gives an analyzer. The key advantage: each analysis component is independently provably sound, and components compose freely with compositional soundness. This is the AAM (Abstracting Abstract Machines) approach.
