01The Three-File Problem

Adding a new attribute to KSC currently requires updating three separate files that must stay in sync:

STEP 1
attr-types.ts
Add the TypeScript type to the KSCAttrMap interface
→
STEP 2
binder.ts / checker.ts
Write equation functions following naming conventions
→
STEP 3
gen-ksc-evaluator.ts
Add a hand-written entry to the ATTRS config array

The ATTRS array is the bottleneck. It has 6 fields per attribute, all maintained by hand:

{
  name: 'kindDefs',           // attribute identity
  spec: 'binder',            // which file owns it
  direction: 'syn',          // syn | inh | collection
  type: typeMap.get('kindDefs')!, // (auto-derived from attr-types.ts)
  cases: ['CompilationUnit'], // which node kinds get specific equations
  extraArgs: { CompilationUnit: ['this._counter'] },
}
The irony: Steps 1 and 2 already encode most of this information. The equation naming conventions (eq_kindDefs_CompilationUnit, eq_defEnv_root, etc.) tell you the direction and production cases. The ATTRS array is largely restating what the code already says.

03Naming Conventions Already Encode Metadata

The equation functions follow strict naming patterns. Each pattern encodes the attribute direction and production dispatch:

Synthesized Attributes

Flow upward from children to parent. Two sub-patterns:

// Production-specific: different equation per node kind
export function eq_kindDefs_CompilationUnit(ctx, counter) { ... }
export function eq_kindDefs_default() { ... }
                  ^^^^^^^^ ^^^^^^^^^^^^^^^ 
                  attr     production case

// Universal: same equation for all node kinds
export function eq_defLookup(ctx) { ... }
                  ^^^^^^^^^ 
                  attr (no suffix = universal)

Detection rule: No _root/_rootValue/_contribute/_combine → synthesized. Non-reserved suffixes = production cases.

Inherited Attributes

Flow downward from parent to children. Identified by root value exports:

// Root value as function (computed at root node)
export function eq_defEnv_root(rootCtx) { ... }
                  ^^^^^^ ^^^^
                  attr   "root" = rootFn: true

// Root value as constant
export const eq_fileImports_rootValue: Set<string> = new Set();
              ^^^^^^^^^^^ ^^^^^^^^^
              attr         "rootValue" = rootFn: false

// Parent equation (overrides propagation at specific parents)
export function eq_fileImports(parentCtx) { ... }
                  ^^^^^^^^^^^
                  bare name = hasParentEq: true

Detection rule: Has _root or _rootValue export → inherited.

Collection Attributes

Aggregated bottom-up across the entire tree:

// Contribute: what each node contributes to the collection
export function eq_allViolations_contribute(ctx) { ... }

// Combine: how to merge contributions
export function eq_allViolations_combine(acc, contrib) { ... }
                  ^^^^^^^^^^^^^ ^^^^^^^
                  attr           reserved suffix

Detection rule: Has both _contribute and _combine → collection.

Decision Tree: From Function Names to Direction

Has eq_X_contribute AND eq_X_combine?
direction = collection
↓ no
Has eq_X_root OR eq_X_rootValue export?
direction = inh
↓ no
Otherwise (has eq_X_default, eq_X_Kind, or bare eq_X)
direction = syn
All 12 attributes are unambiguous. Every current attribute's direction can be determined from its function names alone. There are no conflicts or ambiguous cases.

Silver: Three-Layer Attribute Declaration

-- Layer 1: Declare direction + type
synthesized attribute pp :: String;
inherited attribute env :: Env;
synthesized attribute errors :: [Message] with ++;  -- collection

-- Layer 2: Link to nonterminals
attribute pp occurs on Expr, Stmt;

-- Layer 3: Equations inside productions
abstract production add
sum::Expr ::= l::Expr r::Expr
{
  sum.pp = l.pp ++ " + " ++ r.pp;
  l.env = sum.env;   -- parent provides inherited attr
}

Silver infers which productions have equations by scanning production bodies. The propagate env on Expr, Stmt; shorthand auto-generates copy equations for inherited attributes.

JastAdd: Keywords + Class Hierarchy Dispatch

// Declaration in aspect files (.jrag)
syn Type Exp.actualType();
inh Type Exp.expectedType();
coll LinkedList<Problem> CompilationUnit.problems() [new LinkedList<>()] with add;

// Syn equation: on the owning class
eq LogicalExp.actualType() = Type.boolean();

// Inh equation: on PARENT, specifying child
eq WhileStmt.getExp().expectedType() = Type.boolean();

// Collection contribution: from any node
PostfixExpr contributes error("cannot apply to final")
    to CompilationUnit.problems();

JastAdd generates Java source with lazy cache fields and Define_ methods for inherited attribute propagation.

Kiama: Zero Declarations, Full Inference

// Synthesized: reads from children (inferred!)
val locmin: Tree => Int = attr {
    case Fork(l, r) => (locmin(l)) min (locmin(r))
    case Leaf(v)    => v
}

// Inherited: reads from parent (inferred!)
val globmin: Tree => Int = attr {
    case tree.parent(p) => globmin(p)  // parent access = inherited
    case t              => locmin(t)   // root fallback
}

// Circular: different factory method
val dataIn: Stm => Set[Var] = circular(Set()) {
    s => uses(s) ++ (out(s) -- defines(s))
}

Kiama proves direction can be fully inferred from equation bodies. No codegen — pure runtime embedding with memoization.

Eli/LIDO: Classical AG with Two-Level Equations

-- Attribute declaration with explicit direction
ATTR code: PTGNode SYNT;

-- RULE-level equation (specific production)
RULE p: Stmt ::= 'while' Expr 'do' Stmt
COMPUTE
  Expr.postType = boolType;
END;

-- SYMBOL-level equation (default for all productions)
SYMBOL Expr
COMPUTE
  SYNT.coercion = coerce(THIS.preType, THIS.postType);
END;

Eli's SYMBOL/RULE two-level system maps directly to KSC's eq_X_default() / eq_X_SpecificKind() pattern.

Q

How many files must you currently update to add a new attribute to KSC?

A

Three files: attr-types.ts (the type), binder.ts or checker.ts (the equation functions), and the ATTRS array in gen-ksc-evaluator.ts (hand-maintained metadata). Plus running the codegen script makes it effectively 4 steps.

Q

How does the naming convention distinguish a collection attribute from a synthesized one?

A

Collection attributes have both eq_X_contribute and eq_X_combine functions exported. Synthesized attributes have neither of these reserved suffixes — they use eq_X_Kind for production-specific equations and eq_X_default for the fallback.

Q

What distinguishes eq_X_root from eq_X_rootValue?

A

eq_X_root is an exported function that computes the root value dynamically (e.g., eq_defEnv_root builds a Map from child attributes). eq_X_rootValue is an exported constant (e.g., eq_fileImports_rootValue = new Set()). Both indicate an inherited attribute, but they set rootFn to true vs false.

Q

Which AG system infers attribute direction entirely from the equation body, with no keywords or annotations?

A

Kiama (Scala, UNSW). Direction is inferred from whether the equation pattern-matches on tree.parent(p) (inherited) or destructures children (synthesized). The only explicit distinction is circular() vs attr(). Kiama has no codegen — it's pure runtime embedding.

Q

Why can't extraArgs be inferred from naming conventions? What's the proposed solution?

A

The extraArgs field specifies arguments beyond the standard (ctx, raw) pattern — like this._counter for kindDefs. There's no naming convention that can encode "pass this specific expression as an argument." The proposed solution is a @extraArg this._counter JSDoc tag on the equation function. Currently only 1 of 12 attributes needs it.

Q

How does Silver handle inherited attribute propagation boilerplate?

A

Silver provides the propagate statement: propagate env on Expr, Stmt; auto-generates copy equations that pass inherited attributes from parent to all children for every production of those nonterminals. This is analogous to KSC's inherited attribute auto-propagation where a child inherits its parent's value by default.

Q

What principle from Rust's serde and Next.js file routing does Option D follow?

A

"Infer the common case, annotate the exception." Serde infers JSON field names from struct field names but supports #[serde(rename = "...")] for overrides. Next.js derives routes from file names but supports rewrites in config. Option D infers all metadata from naming conventions but supports @extraArg JSDoc for the one edge case.
