The Big Picture: AG Ideas Escape the Compiler

Foundation

Attribute grammars were invented by Knuth (1968) to specify the semantics of programming languages. But the core idea — decorating tree nodes with computed values that flow up (synthesized) and down (inherited) — applies to any hierarchical structure.

Beyond ASTs

Directory trees, XML documents, build dependency graphs — any tree-shaped data can carry attributes.

Directory Attribute Grammars

Beyond ASTs

Key insight: A file-system directory tree has exactly the same structure as a parse tree — interior nodes (directories) are non-terminals and leaf nodes (files) are terminals. Attribute grammar evaluation works directly on this structure.

DAG = AG + Makefiles

Simões & Almeida's Directory Attribute Grammars combine two ideas:

  • Grammar rules describe the expected directory structure (which subdirectories, which file types)
  • Attribute equations compute derived content (thumbnails, index pages, metadata) as file-system attributes
  • Makefile-like laziness — attributes are only re-evaluated if their dependencies have changed (modification timestamps)

How It Works

Concrete Example: Photo Gallery Generator

// DAG specification (conceptual translation to AG notation)

Grammar:
  Gallery  →  Album*
  Album    →  Image*

Synthesized attributes:
  Image._thumbnail  = resize(Image.path, 150, 150)
  Album._index      = htmlGallery(Album.children.map(c → c._thumbnail))
  Gallery._index    = htmlPortal(Gallery.children.map(a → a._index))

Inherited attributes:
  Album._style      = Gallery._style      // CSS flows down from root
  Album._basePath   = Gallery._basePath + "/" + Album.name

Evaluation rule:
  // Like make: only recompute if source file is newer than attribute file
  if (mtime(Image.path) > mtime(Image._thumbnail))
    evaluate(Image._thumbnail)

Key Properties

  • Attributes are files: Each computed attribute is a real file on disk (e.g., _thumbnail.jpg, _index.html)
  • Attribute names start with _ to distinguish them from terminal files
  • Equations are Perl functions that receive filenames and return file content
  • Incremental by timestamps — exactly like make, attributes are recomputed only when dependencies are newer

Attribute Grammars for Structured Documents

Beyond ASTs

XML and HTML documents are unranked trees — each node can have an arbitrary number of children, unlike ASTs where the arity of each node type is fixed by the grammar. This requires extended attribute grammars.

Standard AGs (Ranked Trees)

Each production has a fixed number of children:

BinExpr → Expr "+" Expr   // always 2 children
IfStmt  → Expr Stmt Stmt  // always 3 children

Attribute equations can name each child: $1, $2, etc.

Extended AGs (Unranked Trees)

Productions use regular expressions on the right-hand side:

ul → li+              // 1 or more children
div → (p | ul | div)*  // any number, any order

Attribute equations must handle variable-length child lists using folds.

Query Language Power

Neven, Schwentick & Vianu (2003) showed that extended AGs can serve as a query language for XML, equivalent in power to monadic second-order logic over unranked trees. Key results:

Property Result
Non-circularity Decidable in EXPTIME (same as ranked case)
Equivalence to MSO Extended AGs capture exactly the MSO-definable queries
Region Algebra Can be efficiently translated into extended AGs
Ordered children AGs naturally respect sibling order — crucial for documents

Scene Graph AGs: 3D Rendering

Steinlechner et al. (TU Wien, GRAPP 2019) applied AGs to scene graphs for real-time 3D rendering — a striking non-compiler use case:

Inherited (Down the Scene Graph)

  • Transform matrix — accumulated model-view transformation
  • Shader/material — surface properties propagated to child geometry
// Shader flows down from parent to children
Group.child0.shader = this.shader
Shader.child.shader = this.surface

Synthesized (Up the Scene Graph)

  • RenderObjects — draw commands collected from leaves
  • BoundingBox — spatial extent from children
// Render objects collected bottom-up
Group.renderObjs = child0.renderObjs
                 ∪ child1.renderObjs
Geometry.renderObjs = {createRO(this)}

Because the computation is purely functional (no imperative draw calls), incremental evaluation recomputes only affected render objects when a subtree changes — O(delta) instead of O(n). Nominated for Best Paper at GRAPP 2019.

Example: Computing a Table of Contents from HTML

// Extended AG over an HTML document tree
syn headings: List<{level, text, id}>

// Leaves: heading elements synthesize themselves
h1.headings = [{level: 1, text: h1.textContent, id: h1.id}]
h2.headings = [{level: 2, text: h2.textContent, id: h2.id}]

// Containers: fold over variable-length children
div.headings = div.children.flatMap(c → c.headings)
body.headings = body.children.flatMap(c → c.headings)

// Root: build the TOC from collected headings
html.toc = buildTocHtml(body.headings)

Build Systems as Attribute Evaluation

Beyond ASTs

The landmark paper "Build Systems à la Carte" (Mokhov, Mitchell & Peyton Jones, 2018) showed that build systems are essentially dependency evaluation engines — and share deep structural parallels with attribute grammar evaluation.

The Parallel

Attribute Grammar

  • Tree structure defines nodes
  • Attribute equations define values
  • Dependencies between attributes
  • Evaluator computes fixed point
  • Incrementality: only recompute changed

Build System

  • File/target graph defines nodes
  • Build rules define outputs
  • Dependencies between targets
  • Scheduler executes rules
  • Incrementality: only rebuild changed

The Two Fundamental Questions

Mokhov et al. decompose every build system along two orthogonal axes:

The AG Connection

Each build system quadrant corresponds to an AG evaluation strategy:

  • Make = static dependency analysis + topological evaluation. This is exactly how classical ordered attribute grammars work.
  • Shake = dynamic dependency discovery during evaluation. This is demand-driven AG evaluation — you don't know all dependencies upfront.
  • Bazel = content-addressable caching with dynamic deps. This is memoized AG evaluation with content hashing instead of timestamps.
  • Excel = restart-based re-evaluation with dirty flags. This resembles circular AG evaluation with fixed-point iteration.

Incremental Attribute Evaluation

Evaluation

The problem: When a user edits one token in a 10,000-node AST, re-evaluating all attributes is wasteful. Incremental evaluation recomputes only the attributes affected by the change.

The Reps-Demers-Teitelbaum Algorithm (1981)

Core Idea: Change Propagation on the Dependency Graph

  1. 1 Edit the tree — insert, delete, or replace a subtree
  2. 2 Mark affected attributes — starting from the edit point, mark directly-dependent attributes as "possibly changed"
  3. 3 Propagate — follow the dependency graph outward, marking transitively-dependent attributes
  4. 4 Re-evaluate only marked attributes — in topological order of dependencies
  5. 5 Early termination — if a re-evaluated attribute produces the same value as before, stop propagating along that path

Interactive: Incremental Re-evaluation

Click buttons above to step through incremental evaluation.

Why Step 5 Matters

Early Termination: The Key Optimization

If we change a variable name from x to y, the errors attribute of the enclosing function might still be 0 (the rename was valid). In that case, we stop propagating — the root's errors attribute doesn't need re-evaluation.

This makes incremental evaluation proportional to the size of the semantic change, not the syntactic change.

The Synthesizer Generator

Reps & Teitelbaum's Synthesizer Generator (1984) used this algorithm to build interactive programming environments: given an AG specification, it generated a complete editor with live type checking, error highlighting, and pretty-printing — all incrementally updated on every keystroke.

Modern Descendants

  • Tree-sitter — incremental parsing (structure only, not full AG)
  • Roslyn / rust-analyzer — demand-driven incremental semantic analysis
  • Salsa (Rust) — generic incremental computation framework inspired by AG incrementality
  • Adapton — self-adjusting computations framework

Memoized & Demand-Driven Evaluation

Evaluation

Instead of evaluating all attributes in a predetermined order, demand-driven evaluation computes an attribute only when someone asks for its value — and memoization ensures each attribute is computed at most once.

Eager / Exhaustive Evaluation

The traditional approach: analyze the dependency graph, find a valid topological order, evaluate every attribute in that order.

// Pass 1: all inherited attributes (top-down)
for (node of tree.topDown()) {
  node.env = node.parent.env.extend(node.declarations)
}

// Pass 2: all synthesized attributes (bottom-up)
for (node of tree.bottomUp()) {
  node.type = inferType(node, node.env)
  node.errors = node.children.flatMap(c => c.errors)
}

Problem: evaluates attributes that may never be needed.

Demand-Driven Evaluation

Attributes are lazy thunks. When you access node.type, the system evaluates it (and any dependencies) on the fly. Memoization caches the result.

// Each attribute is a memoized getter
get type() {
  if (this._type_cached) return this._type_value
  this._type_value = inferType(this, this.env)  // triggers env evaluation
  this._type_cached = true
  return this._type_value
}

// env is also demand-driven — evaluated when first accessed
get env() {
  if (this._env_cached) return this._env_value
  this._env_value = this.parent.env.extend(this.declarations)
  this._env_cached = true
  return this._env_value
}

Advantage: only computes what's actually needed. No pass ordering required.

OOLAG: Object-Oriented Lazy Attribute Grammars

Used in Apache Daffodil (DFDL parser). The tree is a standard object graph with lazy, memoized properties.

// OOLAG pattern (Scala-like pseudocode)
class Element extends SchemaComponent {
  lazy val typeDef = findTypeDef(this.typeName)
  lazy val minOccurs = this.attr("minOccurs").toInt
  lazy val isOptional = minOccurs == 0

  // Computations done exactly once, cached internally
  // Circular dependencies detected at runtime
}

Key properties:

  • Self-organizing: no need for pass structure
  • Computed exactly once and cached
  • Circular dependency detection at runtime

Memoized Zipper-Based AGs

Uses a zipper (Huet, 1997) to navigate the tree — a cursor that maintains the "context" of the current node. Attributes are memoized on the zipper positions.

// Zipper = (focused subtree, context with path back to root)
type Zipper = {
  focus: Tree,
  context: Context  // parent, left siblings, right siblings
}

// Attribute lookup on zipper position
attr(z: Zipper, name: string): Value {
  const key = zipperKey(z) + "." + name
  if (memo.has(key)) return memo.get(key)
  const val = evaluate(z, name)
  memo.set(key, val)
  return val
}

Advantage: functional and persistent — previous tree versions retain their cached attributes, enabling efficient undo and incremental updates.

Strategic & Decorated Attribute Grammars

Formalism

Strategic programming (from the Stratego world) separates what to do at each node from how to traverse the tree. When combined with AGs, you get decorated attribute grammars — where traversal strategies and attribute equations work together.

Traditional AG Evaluation

Traversal order is determined by the evaluator (fixed passes, topological order, or demand). The AG author writes equations only.

// AG: just equations, evaluator decides order
syn type = inferType(this)
inh env = parent.env.extend(decls)

Strategic AG Evaluation

The AG author also controls traversal strategy — which nodes to visit, in what order, whether to continue or stop.

// Strategic: equations + traversal control
strategy typeCheck =
  topDown(try(
    match VarDecl → checkType
    match FuncDecl → checkSignature
  ))

Key Systems

Aster: AGs in Stratego

Kats, Sloane & Visser (2009) built Aster as an extension of the Stratego term-rewriting language. It uses Stratego's pattern matching and concrete syntax to write attribute equations.

  • Attribute decorators describe high-level attribution patterns (e.g., "collect all errors bottom-up")
  • Uses standard Stratego facilities: pattern matching, concrete syntax, generic traversals
  • Equations are associated with AST nodes via pattern matching
// Aster (Stratego syntax)
decorator collect-all     // built-in: bottom-up collection
def errors:
  VarDecl(name, type, init) →
    if <not(compatible)> (type, init.type)
    then [TypeError(name, type, init.type)]
    else []

Silver: Strategy Attributes

Silver (MELT group, University of Minnesota) introduced strategy attributes as an alternative to reflection-based term rewriting. Rewrite rules and strategies are compiled into higher-order attributes and equations.

  • Strategies operate on decorated trees (trees that already have attribute values), not raw terms
  • This means rewriting can be semantics-aware — a strategy can inspect attribute values to decide whether to apply a rule
  • Avoids the "phase ordering" problem of separating analysis from transformation
-- Silver strategy attribute
strategy attribute simplify =
  innermost(
    rule addZero: add(e, intLit(0)) → e
    rule mulOne:  mul(e, intLit(1)) → e
    -- Can check attributes during rewriting!
    rule constFold: add(a, b)
      when a.isConstant && b.isConstant
      → intLit(a.value + b.value)
  )

Decorated Attribute Grammars

The general concept: decorators are parameterized attribution patterns inspired by strategic programming. They separate:

The "what"

  • Individual attribute equations
  • What value to compute at each node type

The "how"

  • Traversal strategy (top-down, bottom-up, innermost)
  • Combination operator (collect, fold, first-match)

Common decorators:

DecoratorDirectionSemantics
collect-allBottom-upGather values from all descendants into a list
broadcastTop-downPass a value down to all descendants
chainLeft-to-rightThread a value through siblings in order
rewriteConfigurableApply transformation rules with a strategy

Higher-Order & Reference Attribute Grammars

Formalism

Higher-Order Attribute Grammars

In a standard AG, attribute values are simple data (numbers, strings, types). In a higher-order AG, an attribute's value can be an entire tree — which then gets its own attributes evaluated.

Key Ideas (Vogt, Swierstra & Kuiper, 1989)

  • An attribute value can be a tree, blurring the line between parse trees and computed values
  • These "local trees" are grafted onto the main tree and decorated with their own attributes
  • Use case: desugaring (if → match), generating intermediate representations, multi-phase compilation
  • A well-defined higher-order AG must be complete, non-circular, and create only finitely many new trees

Reference Attribute Grammars (RAGs)

In standard AGs, information flows strictly along tree edges (parent↔child). Reference attributes break this limitation — an attribute can point to any node in the tree.

JastAdd: The Reference AG System

JastAdd (Hedin, Lund University) is the most prominent RAG system. Key features:

  • Reference attributes return pointers to other tree nodes, enabling name resolution, type lookup, and cross-references
  • Demand-driven evaluation with memoization — attributes computed on first access
  • Circular attribute support for fixed-point computations (e.g., data-flow analysis)
  • Aspect-oriented — attribute definitions can be spread across multiple modules
  • Used to build ExtendJ (a full Java compiler) and JModelica

Circular Attribute Grammars

Standard AGs require the dependency graph to be acyclic. Circular AGs allow cycles in dependencies, resolved by fixed-point iteration.

How Fixed-Point Evaluation Works

  1. 1 Initialize all circular attributes to a bottom value (e.g., empty set, false)
  2. 2 Evaluate all equations — some values change
  3. 3 Re-evaluate until no values change (fixed point reached)
  4. 4 Guaranteed to terminate if the attribute domain forms a lattice of finite height

Example: Liveness Analysis

// Live variables: a classic circular attribute
// live_out depends on successors' live_in,
// live_in depends on live_out — a cycle!

circular live_in: Set<Var> = new Set()  // bottom: empty set
circular live_out: Set<Var> = new Set()

stmt.live_in  = (stmt.live_out − stmt.defs) ∪ stmt.uses
stmt.live_out = stmt.successors.flatMap(s → s.live_in)

// Iterate until fixed point:
// Round 1: live_in = uses (since live_out starts empty)
// Round 2: live_out picks up successor live_in values
// Round 3: live_in grows as live_out grows
// ... until stable

Flashcard Quiz

Review

Click each question to reveal the answer.

Q: What does "DAG = AG + Makefiles" mean?
Think about how attribute evaluation relates to file timestamps...
A: Directory Attribute Grammars treat the file system as a tree, with attribute equations generating derived files (thumbnails, index pages). Like make, attributes are only recomputed when their source files are newer than the output — combining AG semantics with Makefile-style incremental builds.
Q: What is "early termination" in incremental AG evaluation?
What happens when a re-evaluated attribute produces the same value as before?
A: When an attribute is re-evaluated and produces the same value as before, the change propagation stops along that dependency path. This means the cost of re-evaluation is proportional to the semantic change, not the syntactic edit.
Q: How do extended AGs differ from standard AGs for document processing?
Think about the number of children each node can have in XML vs an AST...
A: Standard AGs work on ranked trees where each production has a fixed number of children. Extended AGs handle unranked trees (like XML/HTML) where nodes can have arbitrarily many children, using regular expressions on the right-hand side of productions and fold-based attribute equations.
Q: How does Build Systems à la Carte relate AG evaluation to build tools?
Two orthogonal questions: how to find dependencies, and how to schedule...
A: The paper decomposes build systems along two axes: dependency discovery (static vs dynamic) and scheduling (topological vs restarting). Make ≈ ordered AG (static deps, topo sort). Shake ≈ demand-driven AG (dynamic deps). Bazel ≈ memoized AG (content-addressed caching).
Q: What is a higher-order attribute?
What if an attribute's value isn't a number or string, but a tree?
A: A higher-order attribute has a value that is itself a tree. This new tree gets grafted into the AG and decorated with its own attributes. Use cases include desugaring (translating surface syntax to core syntax) and multi-phase compilation. The AG must create finitely many trees to guarantee termination.
Q: What problem do Reference Attribute Grammars (RAGs) solve?
Standard AGs only allow information flow along tree edges...
A: RAGs allow attributes to reference arbitrary nodes in the tree, not just parents/children. This enables name resolution and type lookup without threading information through the entire tree. A VarRef node's decl attribute can directly point to the VarDecl node it refers to, wherever it is in the tree.
Q: How do circular attribute grammars handle cycles in the dependency graph?
Think about data-flow analysis algorithms...
A: Circular attributes are initialized to a bottom value (e.g., empty set), then all equations are repeatedly evaluated until no values change (fixed-point iteration). This terminates if the attribute domains form lattices of finite height. JastAdd supports this natively for analyses like liveness and reaching definitions.
Q: What distinguishes strategic AGs from traditional AGs?
Who controls the traversal order?
A: In traditional AGs, the evaluator controls traversal order. In strategic AGs, the AG author specifies both equations and traversal strategies (top-down, bottom-up, innermost). This separation of "what to do" from "how to traverse" comes from strategic programming (Stratego). Silver's strategy attributes even allow rewriting on decorated (attribute-bearing) trees.
Q: What is OOLAG and where is it used?
Think about lazy vals in Scala...
A: Object-Oriented Lazy Attribute Grammars (OOLAG) represent the AST as a standard object graph with lazy val properties. Attributes are evaluated on first access and cached. Used in Apache Daffodil (a DFDL data format parser). Self-organizing with no pass structure needed, but no true incrementality — tree changes require rebuilding.
