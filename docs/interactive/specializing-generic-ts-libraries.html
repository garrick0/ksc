Specializing a Generic TypeScript Library

When a library is generic and you need project-specific types back, what are the options? A survey of 8 patterns, the codegen-vs-inference debate, and what the industry has settled on.

01The Problem

You have a generic library. Your users want project-specific types back. How do you bridge the gap?

Consider a generic attribute grammar library. It works on any tree shape — you pass getChildren and it handles the rest. But when a user defines 360 AST node types and 12 attributes, they want the library to know about their specific types: autocomplete on node fields, type-narrowing in production equations, completeness checking over their exact set of kinds.

This isn't unique to AG libraries. Every TypeScript library faces the same tension:

Option A
Pure Inference
Users define their schema in TypeScript. The library infers all types from runtime definitions. No build step needed.
No build stepTS perf at scale
Option B
Code Generation
A build step reads a schema and generates typed TypeScript files. Users import the generated code.
Build stepFast type checking
Option C
Type Augmentation
The library provides open interfaces. Users extend them via declaration merging. The library reads the merged types.
No build stepSpooky action
The central question: When should types be resolved? At edit-time (inference), at build-time (codegen), or at augmentation-time (declaration merging)? All the good options keep configuration in TypeScript — no DSL needed.

Pattern 1: Generic Type Parameter (Kysely)

The user defines an interface describing their domain, then passes it as a type parameter to the library's constructor. The library's entire API becomes specialized to that interface.

// User defines their database structure
interface Database {
  users: { id: number; name: string; email: string };
  posts: { id: number; title: string; author_id: number };
}

// Library becomes specialized via type parameter
const db = new Kysely<Database>(config);
db.selectFrom('users')  // TS knows columns: id, name, email
  .select('name')         // autocomplete on column names
  .where('id', '=', 1);  // type-checked value

Who uses it: Kysely, Effect-TS Context.Tag

Pros: Simple mental model, no build step, no magic. Cons: User must maintain a separate type definition. Library can't introspect the type at runtime.

Pattern 2: Builder + Inference Chain (tRPC, Hono)

The user builds their API through a fluent builder. Each method call accumulates type information. The final typeof export carries the complete inferred type.

// Server: fluent builder accumulates types
const appRouter = router({
  getUser: publicProcedure
    .input(z.object({ id: z.string() }))
    .query(({ input }) => fetchUser(input.id)),
  createPost: publicProcedure
    .input(z.object({ title: z.string() }))
    .mutation(({ input }) => createPost(input)),
});

// Export the inferred type (no codegen!)
export type AppRouter = typeof appRouter;

// Client: uses inferred type, full autocomplete
const client = createTRPCClient<AppRouter>({ ... });
client.getUser.query({ id: "1" }); // typed!

Who uses it: tRPC, Hono RPC

Pros: Zero codegen, excellent DX, define once → types flow everywhere. Cons: Breaks with Higher-Kinded Types (tRPC hit this limit). Chaining must be preserved (Hono loses types if you break the chain).

Pattern 3: Schema-as-Code (Drizzle, Zod, our defineNode)

The user defines their schema using TypeScript function calls. The library infers types from the runtime definitions. No separate type definition, no codegen.

// Drizzle: define table in TS, infer type
const users = pgTable('users', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
});
type User = typeof users.$inferSelect;

// Zod: define schema, infer type
const UserSchema = z.object({ name: z.string(), age: z.number() });
type User = z.infer<typeof UserSchema>;

// Our AG library: defineNode + InferNode
const IfStatementDef = defineNode('IfStatement', {
  expression: child(), thenStatement: child(), elseStatement: optChild(),
});
type KSIfStatement = InferNode<typeof IfStatementDef>;

Who uses it: Drizzle ORM, Zod, Effect Schema, our ag-schema

Pros: Single source of truth (runtime + types), no build step, no separate definitions. Cons: Complex type-level gymnastics. At scale, TS compiler does 95x more work than codegen (Prisma's benchmarks).

Pattern 4: defineConfig + as const satisfies (Vite, Vitest, Nuxt)

The user defines a config object, freezes its literal types with as const, and validates its shape with satisfies. The library reads the config's narrow type.

// Vite: defineConfig provides intellisense
export default defineConfig({
  plugins: [react()],
  server: { port: 3000 },
});

// Custom: as const satisfies preserves literals
const config = {
  nodes: {
    Program: { children: ['compilationUnits'] },
    Identifier: { props: ['escapedText'] },
  }
} as const satisfies AGConfig;
// typeof config retains literal types

Who uses it: Vite, Vitest, Nuxt, TanStack Router

Pros: Very declarative, single object, great IDE support. Cons: Can't carry phantom types (no child<KSIdentifier>()), massive config objects strain the TS language server.

Pattern 5: Type Registry / Declaration Merging (Express, Fastify)

The library provides an open interface. Users augment it via declaration merging. The library reads the merged type globally.

// Library provides an empty open interface
declare module '@ksc/ag' {
  interface NodeRegistry {}
}

// User augments it in their project
declare module '@ksc/ag' {
  interface NodeRegistry {
    Program: { compilationUnits: CompilationUnit[] };
    Identifier: { escapedText: string };
  }
}

// Library reads the merged interface
type AllNodes = NodeRegistry[keyof NodeRegistry];

Who uses it: Express Request augmentation, Fastify Type Providers, Frontend Masters Type Registry

Pros: No build step, very TypeScript-idiomatic, plugins can extend independently. Cons: Spooky action at a distance (types change globally), can't augment default exports, easy to get wrong.

Pattern 6: Codegen Script (Prisma, GraphQL Codegen, our gen-node-defs.ts)

A script reads input (schema file, database, existing code) and generates TypeScript source files. Generated files are committed and imported normally.

// Prisma: schema.prisma → prisma generate → @prisma/client
// GraphQL: schema.graphql → graphql-codegen → types.ts
// KSC: ast.ts → gen-node-defs.ts → node-defs.ts

// Configuration is TypeScript (codegen.ts or gen-node-defs.ts)
import { CodegenConfig } from '@graphql-codegen/cli';
const config: CodegenConfig = {
  schema: './schema.graphql',
  generates: { './src/types.ts': { plugins: ['typescript'] } },
};

Who uses it: Prisma, GraphQL Codegen, TanStack Router routeTree.gen.ts, OpenAPI codegen

Pros: Generated code is plain TypeScript (fast type checking), can produce any structure, easy to debug. Cons: Extra build step, generated files can drift from source, noisy diffs.

Pattern 7: ts-morph / TypeScript Compiler API (Custom generators)

A TypeScript program uses the Compiler API (or its wrapper ts-morph) to programmatically construct and emit .ts files.

import { Project } from 'ts-morph';

const project = new Project();
const file = project.createSourceFile('generated.ts', '');
file.addInterface({
  name: 'KSIdentifier',
  properties: [
    { name: 'kind', type: "'Identifier'" },
    { name: 'escapedText', type: 'string' },
  ],
});
file.saveSync();

Who uses it: ts-morph, custom generators, contentful-typescript-codegen

Pros: Type-safe code generation (no string bugs), can read/modify existing files. Cons: Heavy dependency (pulls full TS compiler), slower, overkill for uniform patterns.

Pattern 8: TSC Transform Plugin (ts-patch, ttypescript)

A custom transformer runs during tsc compilation, rewriting the AST before emit.

// tsconfig.json
{
  "compilerOptions": {
    "plugins": [{ "transform": "./my-transform.ts" }]
  }
}

// my-transform.ts — rewrites AST during tsc
export default function(program: ts.Program) {
  return (ctx: ts.TransformationContext) =>
    (sourceFile: ts.SourceFile) => {
      // rewrite AST nodes, inject code, etc.
    };
}

Who uses it: ts-patch, ttypescript, vite-plugin-typescript-transform

Pros: No separate build step (runs during normal tsc). Cons: Fragile (breaks on TS version updates), non-standard, cannot generate type declarations (transforms operate on emit, not type checking), awful DX for debugging.

03Deep Dives: How the Types Flow

Expand each pattern to see how type information propagates from user definition to library internals.

1 Kysely: User Type → Generic Parameter → Autocomplete ▼

The user writes interface Database { users: {...} } and passes Database as new Kysely<Database>(). Every method in the Kysely API is generic over this type. When you call .selectFrom('users'), TypeScript indexes into Database['users'] to narrow the column types. The library never sees the type at runtime — it's purely a compile-time flow.

Kysely also offers kysely-codegen to generate the Database interface from an actual database, combining patterns 1 + 6.

2 tRPC/Hono: Builder Chain → typeof export → Client inference ▼

Each builder call (e.g., .input(z.object({...}))) returns a new type that includes the previous type plus the new information. The final typeof appRouter is a massive nested type describing every procedure. The client uses this type parameter to infer request/response types.

Limitation: tRPC's typing is lossy — it cannot represent generic/dynamic return types because TypeScript lacks Higher-Kinded Types. The ZenStack team had to fall back to codegen for this case.

3 Drizzle/Zod: Runtime Definition → Phantom Type → z.infer ▼

Each builder function (text('name'), z.string(), our child<T>()) returns a runtime object that also carries a phantom type parameter. The object's typeof encodes all type information. Then z.infer<typeof schema> (or our InferNode<typeof def>) uses conditional types to extract the TypeScript type from the definition.

This is the most powerful pattern: the runtime schema and the types are the same object. Change the schema, types update immediately. But the compiler must re-derive everything on every check.

4 Prisma/GraphQL Codegen: Schema File → Script → Generated .ts ▼

A schema file (schema.prisma, schema.graphql, or our ast.ts) is read by a codegen script. The script generates plain .d.ts or .ts files with explicit interfaces, types, and client APIs. TypeScript loads these as normal files — no inference needed.

This front-loads the type derivation work into the build step. Prisma benchmarks show 2.9x faster type checking and 95x fewer type instantiations than inference-based approaches. The tradeoff: you must re-run the generator when the schema changes.

5 Fastify Type Provider: Open Interface → Module Augmentation → Merged Type ▼

Fastify defines open interfaces (like FastifyRequest). Users augment them via declare module 'fastify' to add their specific schema types. Type Providers enable different schema libraries (Zod, TypeBox, JSON Schema) to plug into the same type system.

Express uses the same pattern for Request.user etc. The Slash Financial team scaled this to 1M lines using file-extension-based registries.

04The Great Debate: Inference vs Codegen

The TypeScript community has been arguing about this since Prisma and Drizzle took opposite approaches. Here's what the data says.

Prisma: "Codegen is 2.9x faster"

In late 2025, Prisma published benchmarks showing their codegen approach requires dramatically fewer type instantiations:

MetricPrisma (codegen)Drizzle (inference)Ratio
Schema type instantiations42841,15095x
Check time (schema)2.9x fasterbaseline
Query type instantiations48% fewerbaseline
Check time (queries)~2.1x fasterbaseline

Key argument: "Drizzle asks the compiler to re-derive everything from first principles on each build. Prisma front-loads that work during prisma generate." Generated types avoid deep recursion and prefer interfaces for structural reuse.

Prisma 7 (late 2025) dropped its Rust engine entirely — the client is now pure TypeScript, and type checking improved by 70%.

Drizzle: "No generate step, instant feedback"

Drizzle's philosophy is radical simplicity: "No generate models step. No background daemon. You just import your schema and query your database."

The instant-feedback DX advantage is real. When you change a column type, every query using that column immediately shows a TypeScript error. No prisma generate step to forget. The schema is the types.

For most real-world schemas (under ~100 tables), the performance difference is imperceptible. The 95x instantiation gap only manifests at significant scale, and even then, modern TypeScript versions keep improving inference performance.

The Drizzle insight: For the vast majority of projects, the DX improvement of "change schema, see type error immediately" outweighs a few extra seconds of type checking.

tRPC: "Inference has a ceiling"

tRPC proved that pure inference hits fundamental limits. When return types need to adapt dynamically based on input parameters (like Prisma's include parameter), TypeScript's lack of Higher-Kinded Types makes inference impossible:

// This SHOULD have different types for p1 and p2
const p1 = await client.loadPost({ id: "1" });
const p2 = await client.loadPost({ id: "1", include: { author: true } });
// But tRPC types both as just `Post` — dynamicity is lost

The ZenStack solution: generate a custom client that casts the types correctly. This is a direct admission that when inference hits its ceiling, codegen becomes necessary.

TanStack Router: "Codegen helps the language service"

TanStack Router takes a pragmatic hybrid approach. Route definitions use inference (Pattern 3), but the framework also generates a routeTree.gen.ts file that declares necessary types where they help the language service.

The generated file is recommended as readonly in VS Code, excluded from search results and file watchers. It's an implementation detail that accelerates the IDE.

The emerging consensus: Use inference for the primary API surface (Pattern 3). Use codegen as an optimization layer when the TS language service needs help (Pattern 6). This is exactly what we built with defineNode() + gen-node-defs.ts.

05What We Built (KSC)

Our attribute grammar library uses the same hybrid architecture as Prisma and TanStack Router.

libs/ag-schema/src/define.ts — Schema-as-Code

Builder functions with phantom types. Each defineNode() call produces both a runtime schema entry AND a TypeScript type via InferNode<>.

// Builder functions carry phantom types
export function child<T>(): ChildSpec<T> {
  return { __tag: 'child' } as ChildSpec<T>;
}

// Define a node: runtime schema + phantom type info
const TypeAliasDef = defineNode('TypeAliasDeclaration', {
  name: child<KSIdentifier>(),     // required typed child
  typeParameters: list(),          // KSNode[]
  type: child(),                   // required generic child
  modifiers: list(),               // KSNode[]
});

// Infer TypeScript type from the definition
type KSTypeAliasDeclaration = InferNode<typeof TypeAliasDef>;
// = { kind: 'TypeAliasDeclaration'; name: KSIdentifier; type: any; ... }

scripts/gen-node-defs.ts — Migration Script

Parses the existing ast.ts (2252 lines, 360 interfaces) using regex and generates node-defs.ts with all defineNode() calls.

// Field classification rules:
//   KSNode          → child()
//   KSIdentifier    → child<KSIdentifier>()
//   KSNode | undef  → optChild()
//   KSNode[]        → list()
//   string, boolean → prop<T>()

// Output: 2147 lines, 156 complex + 204 leaf defs
// npx tsx scripts/gen-node-defs.ts
// Parsed 360 interfaces
//   Complex: 156
//   Leaf:    204

src/pipeline/node-defs.ts — Generated Source of Truth

// AUTO-GENERATED by scripts/gen-node-defs.ts
import { defineNode, defineLeaf, child, optChild, list, prop,
  createNodeSchema } from '../../libs/ag-schema/src/index.js';

export const ProgramDef = defineNode('Program', {
  compilationUnits: list<KSCompilationUnit>(),
});

export const IfStatementDef = defineNode('IfStatement', {
  expression: child(), thenStatement: child(), elseStatement: optChild(),
});

export const OpenBraceTokenDef = defineLeaf('OpenBraceToken');
// ... 360 total definitions

// Schema creation — runtime introspection + derived getChildren
export const ksNodeSchema = createNodeSchema<KSNode>(allNodeDefs);
export const getChildren = ksNodeSchema.getChildren;

Wiring: program.ts uses schema-derived getChildren

// src/program.ts — the entire pipeline uses schema-derived traversal
import { getChildren } from './pipeline/node-defs.js';

const grammar = createGrammar(getChildren);
const semantics = createSemantics(grammar, [
  createBinderSpec(),
  createCheckerSpec(),
]);
const results = interpret(semantics, ksTree.root);

// Schema also provides introspection:
ksNodeSchema.kinds.size;               // 360
ksNodeSchema.getChildFields('IfStatement'); // ['expression', 'thenStatement', 'elseStatement']
ksNodeSchema.checkCompleteness(eqs);     // missing kinds

107/107 tests pass. Schema-derived getChildren verified equivalent to the manual version on every node in real ASTs.

07What AG Systems Do

How traditional attribute grammar systems handle this problem.

JastAdd: .ast + .jrag + .jadd files

JastAdd uses a dedicated grammar format (.ast) plus two types of aspect files:

  • .jrag — Declarative aspects: attribute declarations, equations, rewrites (an extended Java dialect)
  • .jadd — Imperative aspects: ordinary Java fields and methods added to AST classes

JastAdd's approach is Pattern 6 (codegen): the .ast grammar and .jrag/.jadd aspects are processed by the JastAdd tool, which generates Java classes. The generated classes have all the attribute accessors, caching, and lazy evaluation baked in.

The specification language is an extended Java — not a completely separate DSL. This was a deliberate design choice to keep the learning curve low for Java programmers.

Silver: Dedicated specification language

Silver takes a different approach: a fully dedicated specification language for attribute grammars. It's not an extension of an existing language but a purpose-built notation.

Silver provides dedicated syntax for control-and data-flow analysis, which translates to control-flow graphs and temporal logic formulae. This is closer to a DSL approach — more expressive for AG patterns but with a steeper learning curve.

KSC: TypeScript IS the specification language

Our approach: no DSL at all. TypeScript itself is the specification language. Attribute declarations, equations, and grammar structure are all plain TypeScript function calls:

const spec: SpecInput<KSNode> = {
  name: 'ksc-binder',
  declarations: {
    kindDefs: { direction: 'syn' },
    defEnv:   { direction: 'inh', root: [] },
  },
  equations: {
    kindDefs: { TypeAliasDeclaration: (n) => extractKind(n), _: () => [] },
    defEnv:   (parent, child) => parent.kindDefs,
  },
};

This is Pattern 3 (Schema-as-Code) for the AG library itself, plus Pattern 6 (Codegen) for the 360 node type definitions. TypeScript's type system handles what JastAdd's code generator does — ensuring equations match declarations, kinds are covered, etc.

08Recommendations

When to use which pattern.

A Small domain (<50 types): Use Pattern 3 (Schema-as-Code) ▼

If your domain is small enough that TS inference won't struggle, Schema-as-Code (Drizzle/Zod pattern) gives the best DX. Define once, types flow everywhere, no build step. This is what we do for the AG library itself (SpecInput, AttrDecl, etc.).

B Large domain (>100 types): Use Pattern 3 + 6 (Hybrid) ▼

When you have hundreds of node types (like our 360), combine inference with codegen. The library API uses inference (Pattern 3: defineNode + InferNode), but the instance (all 360 definitions) is generated by a script (Pattern 6). This is the Prisma architecture and the TanStack Router architecture.

C Plugin system: Use Pattern 5 (Type Registry) ▼

If third-party plugins need to extend your types independently, declaration merging is the TypeScript-idiomatic solution. Express and Fastify prove it works at scale. But use sparingly — the "spooky action at a distance" problem is real.

D API boundary: Use Pattern 2 (Builder Chain) ▼

For server/client type sharing (API routers, RPC), the tRPC/Hono builder pattern is unbeatable. Export the router's typeof and let the client infer everything. Just be aware of the HKT limitation.

The industry trend: The hybrid approach (inference for the API, codegen for the instance) is emerging as the standard. Prisma, TanStack Router, and KSC all independently converged on this pattern. Pure inference scales badly. Pure codegen has poor DX. The hybrid gets the best of both.

Q

What is the fundamental difference between how Drizzle and Prisma provide type safety?

A

Drizzle uses type inference — you define your schema in TypeScript and types are derived automatically at edit-time. Prisma uses code generation — you run prisma generate which produces .d.ts files with pre-computed types. Prisma front-loads the type derivation work; Drizzle re-derives on every check.

Q

Why did tRPC's pure inference approach hit a ceiling? What TypeScript feature is missing?

A

tRPC's inference breaks down with dynamic generic return types (e.g., return type depends on an include parameter). TypeScript lacks Higher-Kinded Types (HKTs), so it can't hold "uninstantiated" generic function types and instantiate them in a different context. The ZenStack team had to fall back to codegen to fix this.

Q

In the Kysely pattern (Pattern 1), how does the library become specialized to the user's database?

A

The user defines a Database interface describing all tables and their column types, then passes it as a generic type parameter: new Kysely<Database>(config). The entire Kysely API becomes specialized to that interface. The library never sees the type at runtime — it's purely compile-time.

Q

How do phantom types work in a schema-as-code system like Zod or our defineNode()?

A

Builder functions like child<KSIdentifier>() return a runtime object ({ __tag: 'child' }) that also carries a generic type parameter T via a phantom field (__phantom?: T). At runtime, T is erased. At the type level, InferNode uses conditional types to extract T from the definition and reconstruct the full interface.

Q

What is the "type registry" pattern (Pattern 5), and which libraries use it?

A

The library provides an open interface (like Express's Request). Users extend it via declare module and TypeScript's declaration merging. The library reads the merged type globally. Used by Express (augment Request.user), Fastify (Type Providers), and the Slash Financial registry system (1M lines of TypeScript).

Q

Prisma found 95x more type instantiations in Drizzle's approach. What causes this difference?

A

Drizzle's inference forces the TypeScript compiler to re-derive types from first principles on every build — walking through conditional types, mapped types, and generic instantiations. Prisma's codegen precomputes these types during the generate step, writing explicit .d.ts files that use interfaces (which TypeScript can structurally reuse) instead of complex type-level computations.

Q

What hybrid pattern is emerging as the industry standard, and who uses it?

A

Inference for the API surface + codegen for the instance. The library API uses Pattern 3 (Schema-as-Code) for good DX, while the large generated file uses Pattern 6 (Codegen) for fast type checking. Prisma, TanStack Router (routeTree.gen.ts), and KSC (defineNode + gen-node-defs.ts) all independently converged on this architecture.

Q

How does JastAdd's specification approach compare to KSC's? What is each one closest to?

A

JastAdd uses Pattern 6 (Codegen): .ast grammar and .jrag/.jadd aspects are processed by the JastAdd tool, which generates Java classes. KSC uses TypeScript as the specification language (Pattern 3): attribute declarations and equations are plain TypeScript objects and functions. KSC adds Pattern 6 only for the 360 node definitions, not for the AG spec itself.
