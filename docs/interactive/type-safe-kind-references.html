01The Problem

Equation keys are completely untyped. Every kind reference is a silent potential bug.

In the current KindScript analysis spec, equation keys are Record<string, Function>. This means:

Current — No Safety
const allAttrs: AttrDecl[] = [
  {
    name: 'kindDefs',
    direction: 'syn',
    type: 'KindDefinition[]',
    default: eq_kindDefs_default,
    equations: {
      Compilaton: eq_kindDefs_CU,
      // ^^^ Typo! Silently ignored.
    },
  },
];
→
Goal — Full Safety
const overrides: KindOverrides = {
  Compilaton: {
    // ^^^ Type error! Not in TSKind
    kindDefs: eq_kindDefs_CU,
  },
  CompilationUnit: {
    kindDefs: eq_violationFor_Id,
    // ^^^ Type error! Wrong return type
  },
};

Three categories of bugs go undetected today:

Bug ClassExampleSeverity
Kind name typos{ Compilaton: fn } — equation is silently dead codeHigh
Wrong functionPassing eq_violationFor_Id where eq_kindDefs_CU was neededHigh
Grammar driftA kind is renamed in the grammar, equations still reference old nameMedium
Key insight: Even JastAdd — the weakest of the three major AG systems — catches invalid kind names during its weaving phase. Our current system is weaker than all three established AG systems.

Approach 1: Partial<Record<K, Function>>

interface SynAttr<K extends string = string> extends AttrBase {
  direction: 'syn';
  default: AttrExpr;
  equations?: Partial<Record<K, Function>>;
}

// Usage:
const attrs: AttrDecl<TSKind>[] = [
  { equations: { CompilationUnit: fn } }, // OK
  { equations: { Compilaton: fn } },       // Type error!
];

Catches: Kind name typos (validity).

Misses: Wrong function passed (return type not checked). Function erases all signature info. This is JastAdd-level safety.

Effort: Minimal — add <K> generic param, default to string for backward compat.

Approach 2: Typed Equation Objects

interface KindEquation<K extends string, R> {
  readonly kind: K;
  readonly fn: (ctx: Ctx) => R;
}

function forKind<K, R>(kind: K, fn: (ctx: Ctx) => R): KindEquation<K, R>;

interface SynAttr<K, R> {
  default: (ctx: Ctx) => R;
  equations?: KindEquation<K, R>[];
}

// Catches BOTH invalid kind AND wrong return type:
forKind('Compilaton', eq_kindDefs_CU)        // Error: not in TSKind
forKind('CompilationUnit', eq_violationFor) // Error: returns Diagnostic, not KindDef[]

Catches: Kind name typos + wrong return type. The R flows from default to every equation.

Misses: Completeness (no check that all needed kinds have equations).

Cost: equations: { K: fn } becomes equations: [forKind('K', fn)]. Codegen reads .kind instead of record keys.

Approach 3: withKind on Functions

// In equations.ts — kind binding at definition site:
export const eq_kindDefs_CU = withKind(
  'CompilationUnit' satisfies TSKind,
  withDeps(['defEnv'], function eq_kindDefs_CU(ctx) { ... })
);

// In spec.ts — just list the functions:
equations: [eq_kindDefs_CU] // kind binding is ON the function

Catches: Kind name typos (via satisfies TSKind).

Tradeoff: Kind binding is co-located with the equation (good for authors), but invisible in the spec file (bad for reviewers). You can't see which kinds an attribute covers by reading spec.ts.

Coupling: equations.ts must import TSKind, creating a dependency from equation module to grammar definition.

Approach 4: Production-Centric (Silver-Style)

// Flip the axis: organize by kind, not by attribute
type KindOverrides = Partial<Record<TSKind, {
  kindDefs?: (ctx: Ctx) => KindDefinition[];
  violationFor?: (ctx: Ctx, p: string) => Diagnostic | null;
  // ... all syn/inh attrs with their exact signatures
}>>;

const overrides: KindOverrides = {
  CompilationUnit: { kindDefs: eq_kindDefs_CU },
  Identifier:      { violationFor: eq_violationFor_Id },
};

Catches: Invalid kind names (outer keys) + invalid attribute names (inner keys) + wrong function signatures (per-attr typing). Two-dimensional safety.

This is the Silver model: equations live at the intersection of production × attribute.

Cost: Inverts the current architecture. Requires pivotToAttrCentric() to reshape for codegen. See the deep dive in Section 5.

Approach 5: Exhaustive Generated Switches

// Generated evaluator — explicit case for ALL 364 kinds:
get kindDefs(): KindDefinition[] {
  switch (this.node.kind) {
    case 'CompilationUnit': return eq_kindDefs_CU(this);
    case 'Identifier': return eq_kindDefs_default(this);
    case 'BinaryExpression': return eq_kindDefs_default(this);
    // ... all 364 kinds
  }
  const _: never = this.node.kind; // exhaustiveness check
}

Catches: Grammar/evaluator drift. If a kind is added to the grammar but codegen isn't rerun, tsc fails on the generated code. Complementary to approaches 1–4.

Cost: Generated switch is larger (every kind listed), but it's machine-generated so readability isn't a concern.

Approach 6: Codegen-Time Validation

// In compile.ts, during codegen:
for (const kind of Object.keys(attr.equations)) {
  if (!grammarKinds.has(kind)) {
    throw new Error(
      `Attr '${attr.name}': equation references unknown kind '${kind}'`
    );
  }
}

Catches: Invalid kind names at npm run codegen time. Zero type system changes needed.

No IDE support: Errors don't appear as red squiggles while editing. They surface when codegen runs (before runtime, but after writing).

This is the JastAdd weaving model: JastAdd validates equation targets during compilation, not during IDE editing.

03How AG Systems Handle This

JastAdd, Silver, and Infer each solve type safety for kind references differently.

JastAdd — Weaving-Phase Validation

JastAdd validates class names during its weaving phase. If you write eq Foo.x() and Foo isn't in any .ast file, weaving fails. Beyond that, it delegates to javac — generated Java code that references nonexistent classes/methods fails to compile.

// JastAdd .jrag file:
syn KindDefinition[] CompilationUnit.kindDefs() { ... }
syn KindDefinition[] Compilaton.kindDefs() { ... }
//                   ^^^^^^^^^^^ Weaving error: no such AST class

Safety level: Catches invalid kind names. Does NOT check equation completeness. Errors surface late (during weaving/Java compilation, not in IDE). Our Approach 6 matches this.

Silver — Full Static Type System + MWDA

Silver has its own static type checker. Nonterminal names are types. Attribute access is checked. The MWDA (Modular Well-Definedness Analysis) goes further — it checks that every non-forwarding production provides equations for every attribute occurring on its nonterminal.

-- Silver: production-centric equation binding
abstract production add
top::Expr ::= l::Expr  r::Expr
{
  top.pp = l.pp ++ " + " ++ r.pp;
  -- Equations are bound AT the production, not the attribute
}

-- Silver: propagate = auto-generate default equations
propagate errors on Expr, Stmt;
-- ≈ our SynAttr.default field

Safety level: The gold standard. Validates kind names, attribute names, signatures, AND completeness (MWDA). Our Approach 4 is closest to Silver's production-centric model.

Silver insight: Silver's propagate mechanism (auto-generating default equations for all productions) maps directly to our default field on SynAttr. The pattern "default for all + override for specific kinds" is identical in both systems.

Infer — OCaml Exhaustive Pattern Matching

Infer's SIL IR is a closed OCaml algebraic data type. Pattern matching is exhaustive — misspelling a variant constructor is a compile error. Adding a new variant causes warnings in every incomplete pattern match.

(* OCaml: closed algebraic type *)
type instruction =
  | Load of ...
  | Store of ...
  | Call of ...

match instr with
| Laod _ -> ...  (* Compile error: no such constructor *)
(* Missing a variant → exhaustiveness warning *)

Safety level: Strongest possible. TypeScript can't match OCaml's exhaustiveness on records, but our Approach 5 (exhaustive generated switches with never) gets close for the generated code.

KSC Current — No Safety

// analysis/types.ts:
export interface SynAttr extends AttrBase {
  equations?: Record<string, Function>;
  //          ^^^^^^^^^^^^^^^^^^^^^^^^
  //          No kind validation.
  //          No return type checking.
  //          No completeness.
}

Safety level: Weaker than all three established AG systems. Any string is accepted as a kind name. Any function is accepted as an equation. No drift detection.

05Production-Centric Deep Dive

The recommended approach: Silver-style inversion with pivotToAttrCentric().

The Core Inversion

Currently, equations are organized per-attribute (attr-centric): each attribute declaration lists which kinds override the default. The production-centric model inverts this: equations are organized per-kind, listing which attributes that kind overrides.

Current: Attr-Centric
// Per-attribute, list overriding kinds:
{
  name: 'kindDefs',
  equations: {
    CompilationUnit: eq_kindDefs_CU,
  }
}
{
  name: 'violationFor',
  equations: {
    Identifier: eq_viol_Id,
    CallExpression: eq_viol_Call,
  }
}
→
New: Production-Centric
// Per-kind, list overriding attrs:
{
  CompilationUnit: {
    kindDefs: eq_kindDefs_CU,
  },
  Identifier: {
    violationFor: eq_viol_Id,
  },
  CallExpression: {
    violationFor: eq_viol_Call,
  },
}

Step 1: The AttrOverrideMap Type

First, define a type that maps each attribute name to its equation function signature. This is the inner dimension of the 2D safety:

// Generated or hand-written type that knows every attribute's signature:
interface AttrEquationSignatures {
  // Non-parameterized syn attrs:
  kindDefs: (ctx: Ctx) => KindDefinition[];
  defLookup: (ctx: Ctx) => (name: string) => KindDefinition | undefined;
  kindAnnotations: (ctx: Ctx) => KindDefinition[];
  allViolations: (ctx: Ctx) => Diagnostic[];

  // Parameterized syn attrs:
  violationFor: (ctx: Ctx, property: string) => Diagnostic | null;

  // Inherited attrs (parentEquation signatures):
  defEnv: (ctx: Ctx) => Map<string, KindDefinition>;
  contextFor: (ctx: Ctx, property: string) => KindDefinition | null | undefined;
}

// The 2D type: outer = kind, inner = attr with typed signatures
type KindOverrides = Partial<Record<TSKind, Partial<AttrEquationSignatures>>>;
Key insight: Partial<Record<TSKind, Partial<AttrEquationSignatures>>> gives two-dimensional safety from a single type. The outer Partial<Record<TSKind, ...>> validates kind names. The inner Partial<AttrEquationSignatures> validates attribute names AND function signatures simultaneously.

Step 2: The Spec's Override Declaration

In the spec file, the overrides replace scattered equation records:

Step 3: The pivotToAttrCentric() Function

The codegen expects attr-centric data (the current format). A pure function reshapes production-centric overrides into the format codegen already understands:

/**
 * Pivot production-centric overrides into attr-centric format.
 *
 * Input:  { CompilationUnit: { kindDefs: fn }, Identifier: { violationFor: fn } }
 * Output: { kindDefs: { CompilationUnit: fn }, violationFor: { Identifier: fn } }
 *
 * withDeps metadata on functions is preserved through the reshape.
 */
function pivotToAttrCentric<K extends string>(
  overrides: Partial<Record<K, Record<string, Function>>>
): Record<string, Partial<Record<K, Function>>> {
  const result: Record<string, Partial<Record<K, Function>>> = {};

  for (const [kind, attrMap] of Object.entries(overrides)) {
    for (const [attr, fn] of Object.entries(attrMap as Record<string, Function>)) {
      if (!result[attr]) result[attr] = {};
      (result[attr] as any)[kind] = fn;
      // fn still carries .deps and .name from withDeps() — preserved
    }
  }

  return result;
}

Step 4: Building the AnalysisSpec

The pivoted data feeds into AttrDecl construction. The attr declarations keep the same shape the codegen expects:

// In spec.ts — assemble the AnalysisSpec using pivoted overrides:

const pivotedSyn = pivotToAttrCentric(synOverrides);
const pivotedInh = pivotToAttrCentric(inhOverrides);

const allAttrs: AttrDecl[] = [
  {
    name: 'kindDefs',
    direction: 'syn',
    type: 'KindDefinition[]',
    default: eq_kindDefs_default,
    equations: pivotedSyn['kindDefs'], // { CompilationUnit: fn }
  },
  {
    name: 'violationFor',
    direction: 'syn',
    type: 'Diagnostic | null',
    parameter: { name: 'property', type: 'string' },
    default: null,
    equations: pivotedSyn['violationFor'], // { Identifier: fn, Call: fn, ... }
  },
  {
    name: 'contextFor',
    direction: 'inh',
    type: 'KindDefinition | null',
    parameter: { name: 'property', type: 'string' },
    rootValue: null,
    parentEquations: pivotedInh['contextFor'], // { VariableDeclaration: fn }
  },
  // ... etc
];
Key insight: compile.ts doesn't change at all. It still receives attr-centric AttrDecl[]. The inversion is entirely within the spec — the typed KindOverrides catches errors at the spec definition site, and pivotToAttrCentric() reshapes the data for codegen consumption.

Step 5: How withDeps() Integrates

Dependencies are attached to the equation functions themselves, not to the override records. pivotToAttrCentric() moves function references without cloning them, so .deps and .name metadata are preserved:

// equations.ts — unchanged:
export const eq_kindDefs_CU = withDeps(
  ['defEnv'],
  function eq_kindDefs_CompilationUnit(ctx: Ctx): KindDefinition[] { ... }
);
// eq_kindDefs_CU.deps === ['defEnv']
// eq_kindDefs_CU.name === 'eq_kindDefs_CompilationUnit'

// After pivot, the same function object is in:
// pivotedSyn['kindDefs']['CompilationUnit'] === eq_kindDefs_CU
// ∴ .deps and .name still accessible for compile.ts auto-import

Step 6: Where the AttrEquationSignatures Type Comes From

Two options for defining the inner type:

Option A: Hand-Written (Recommended Initially)

The spec author writes the signature type alongside the attr declarations. For 8 attributes, this is trivial:

// specs/ts-ast/kind-checking/overrides.ts

interface SynEquationSignatures {
  kindDefs: (ctx: Ctx) => KindDefinition[];
  defLookup: (ctx: Ctx) => (name: string) => KindDefinition | undefined;
  kindAnnotations: (ctx: Ctx) => KindDefinition[];
  violationFor: (ctx: Ctx, property: string) => Diagnostic | null;
  allViolations: (ctx: Ctx) => Diagnostic[];
}

interface InhEquationSignatures {
  defEnv: (ctx: Ctx) => Map<string, KindDefinition> | undefined;
  contextFor: (ctx: Ctx, p: string) => KindDefinition | null | undefined;
}

Separate types for syn and inh because inherited parentEquations return T | undefined (undefined = copy-down).

Option B: Derived from AttrDecl (Future)

With TypeScript mapped types, signatures could be derived from the attr declarations themselves. This avoids duplication but is more complex:

// Hypothetical: derive signatures from attrs array
type AttrSignature<A extends AttrDecl> =
  A extends { parameter: ParamDef }
    ? (ctx: Ctx, param: string) => unknown
    : (ctx: Ctx) => unknown;

// This loses the concrete return type (KindDefinition[], Diagnostic, etc.)
// because AttrDecl.type is a string, not a TypeScript type.
// Hand-written is more precise until we add type-level return types.

The derived approach can't recover concrete return types from the type: string field in AttrDecl. Hand-written signatures are more precise. This is fine for our scale (8 attributes).

06Extractor DSL Simplification

Option B: kill the DSL layer, flatten the assembly, let validateFieldExpressions() be the single validation path.

The Problem: Two Overlapping Validation Systems

Today, field extractor expressions are built two ways. The DSL layer (grammar/frontend/dsl.ts, 51 lines) provides type-safe builder functions that construct expression strings. But validateFieldExpressions() in grammar/frontend/codegen.ts also validates all expression strings via regex scanning for unknown function references. The DSL provides compile-time safety for DSL-constructed expressions; the validator catches everything else. Two overlapping systems, redundant for the DSL-constructed case.

Before / After

Before — DSL Builders
// specs/ts-ast/frontend/extractors.ts
import { statefulCallBuilders }
  from '../../../grammar/frontend/dsl.js';

const sfn = statefulCallBuilders([
  'hasSymFlag',
  'getJsDoc',
  'resolveType',
  // ... 12 more helper names
] as const);

// Usage in extractor config:
Identifier: {
  isVariable:
    sfn.hasSymFlag('node',
      'ts.SymbolFlags.Variable'),
}
→
After — Raw Strings
// specs/ts-ast/frontend/extractors.ts
// No DSL import needed.

// Usage in extractor config:
Identifier: {
  isVariable:
    'hasSymFlag(_ctx, node, '
    + 'ts.SymbolFlags.Variable)',
}

// validateFieldExpressions() scans
// 'hasSymFlag' against knownHelpers
// — same safety, single path.

What Gets Deleted

grammar/frontend/dsl.ts (51 lines) — the three builder factories:

  • statefulCallBuilders(names) — auto-prepends _ctx as first arg
  • pureCallBuilders(names) — no context variable
  • mapLookupBuilders(names) — produces bracket-access expressions

All three produce the same thing: a string like 'hasSymFlag(_ctx, node, ts.SymbolFlags.Variable)'. The builders add a layer of indirection for something that's already a string literal.

What Gets Simplified

specs/ts-ast/frontend/extractors.ts — drops the DSL imports and writes expression strings directly. The extractor config becomes plain data with zero runtime dependencies beyond the string values themselves:

// Before: DSL-constructed expression
sfn.hasSymFlag('node', 'ts.SymbolFlags.Variable')
// → 'hasSymFlag(_ctx, node, ts.SymbolFlags.Variable)'

// After: just write the string
'hasSymFlag(_ctx, node, ts.SymbolFlags.Variable)'
Key insight: The DSL builders are pure string concatenation factories. They don't carry types from the expression domain — the arguments are all string. The "type safety" they provide is limited to validating the helper function name at the call site (you can't call sfn.hasSymFlg because it doesn't exist on the builder object). But validateFieldExpressions() already catches unknown function names via regex scan of the final strings. Deleting the DSL removes a redundant validation path.

No more "the DSL validates helper names at build time, and the validator re-validates them at codegen time." One path, no overlap.

Integration with the Broader Plan

Once TSKind exists from as const grammar definitions (Section 05's production-centric model), the extractor config keys also become type-checked. The extractor config type tightens from an open record to a constrained one:

// Before: open record, any string key
type ExtractorConfig =
  Record<string, Record<string, string>>;

// After: kind names type-checked via TSKind
type ExtractorConfig =
  Partial<Record<TSKind, Record<string, string>>>;

The outer keys get TSKind validation (catch kind name typos in extractor config), and validateFieldExpressions() validates the inner expression strings (catch unknown helper references). Two complementary checks, each with a clear, non-overlapping responsibility.

Net result: Delete 51 lines of DSL machinery. Simplify extractors.ts to plain string data. Consolidate to a single validation path via validateFieldExpressions(). Get better coverage once TSKind arrives, because the outer keys gain type checking that the DSL never provided.

08Exhaustive Generated Switches (Layer 2)

Approach 5 complements the production-centric model by catching grammar/evaluator drift.

Current Generated Switch

Today, the generated evaluator uses a default case to handle all kinds without custom equations:

// CURRENT generated evaluator (default fallthrough):
get kindDefs(): KindDefinition[] {
  switch (this.node.kind) {
    case 'CompilationUnit': return eq_kindDefs_CU(this);
    default: return eq_kindDefs_default(this);
  }
}

Exhaustive Generated Switch

Instead, generate an explicit case for every kind — custom equation or default — with a never check at the end:

// NEW generated evaluator (exhaustive):
get kindDefs(): KindDefinition[] {
  switch (this.node.kind) {
    case 'CompilationUnit': return eq_kindDefs_CU(this);
    case 'Identifier': return eq_kindDefs_default(this);
    case 'BinaryExpression': return eq_kindDefs_default(this);
    // ... all 364 kinds listed explicitly ...
    case 'TypeReference': return eq_kindDefs_default(this);
  }
  const _exhaustive: never = this.node.kind;
  throw new Error(`Unhandled kind: ${_exhaustive}`);
}

What This Catches

If someone adds NewKind to the grammar but doesn't regenerate the evaluator:

// tsc error on the generated code:
// Type 'NewKind' is not assignable to type 'never'.
const _exhaustive: never = this.node.kind;
//                         ^^^^^^^^^^^^^^^^^

Implementation in compile.ts

The change is in the generateMethod function. Instead of generating a switch with a default case, iterate over all kinds from the grammar and generate a case for each:

// In compile.ts — generateMethod for SynAttr:
function generateSynSwitch(attr: SynAttr, allKinds: Set<string>): string[] {
  const lines: string[] = [];
  lines.push(`switch (this.node.kind) {`);

  for (const kind of allKinds) {
    const eq = attr.equations?.[kind];
    if (eq) {
      lines.push(`  case '${kind}': return ${eq.name}(this);`);
    } else {
      lines.push(`  case '${kind}': return ${emitExpr(attr.default)};`);
    }
  }

  lines.push(`}`);
  lines.push(`const _: never = this.node.kind;`);
  return lines;
}
Watch out: The exhaustive switch requires compile.ts to know all valid kinds. Currently it doesn't have this info. The grammar's kind set must be passed via GrammarConfig or a new field on AnalysisSpec. One approach: add allKinds?: ReadonlySet<string> to GrammarConfig.

09Codegen-Time Validation (Layer 3)

Approach 6: belt-and-suspenders validation that runs during npm run codegen.

Where It Runs

Validation runs inside compileAnalysis() in analysis/compile.ts, before code generation begins. If validation fails, no files are written.

What It Checks

// In compile.ts — validation phase before codegen:

function validateSpec(
  spec: AnalysisSpec,
  grammarKinds: ReadonlySet<string>
): void {
  const errors: string[] = [];

  // 1. Validate rootKind exists
  if (!grammarKinds.has(spec.grammarConfig.rootKind)) {
    errors.push(
      `rootKind '${spec.grammarConfig.rootKind}' is not a valid kind`
    );
  }

  // 2. Validate all equation kind references
  for (const attr of spec.attrs) {
    const eqs =
      attr.direction === 'syn' ? attr.equations :
      attr.direction === 'inh' ? attr.parentEquations :
      undefined;

    if (eqs) {
      for (const kind of Object.keys(eqs)) {
        if (!grammarKinds.has(kind)) {
          errors.push(
            `Attr '${attr.name}': equation references unknown kind '${kind}'`
          );
        }
      }
    }
  }

  // 3. Validate equation function names exist (for import generation)
  for (const attr of spec.attrs) {
    const eqs =
      attr.direction === 'syn' ? attr.equations :
      attr.direction === 'inh' ? attr.parentEquations :
      undefined;

    if (eqs) {
      for (const [kind, fn] of Object.entries(eqs)) {
        if (!fn.name) {
          errors.push(
            `Attr '${attr.name}', kind '${kind}': equation function has no name (anonymous)`
          );
        }
      }
    }
  }

  if (errors.length > 0) {
    throw new Error(
      `Analysis spec validation failed:\n  - ${errors.join('\n  - ')}`
    );
  }
}

Error Output

$ npm run codegen:analysis

Error: Analysis spec validation failed:
  - Attr 'violationFor': equation references unknown kind 'Compilaton'
  - Attr 'kindDefs': equation function has no name (anonymous)
Key insight: This is the JastAdd weaving model implemented in our system. JastAdd catches these errors during weaving (its compilation phase). We catch them during npm run codegen. The advantage over pure type checking: this works even if the spec doesn't use typed overrides (e.g., mock specs that use AttrDecl without generic params).

Q

Why can't Partial<Record<K, Function>> (Approach 1) catch wrong equation functions being passed?

A

Because Function erases all signature information. Any function matches Function, so passing eq_violationFor_Id (returns Diagnostic | null) where eq_kindDefs_CU (returns KindDefinition[]) was expected is perfectly valid to TypeScript. You need a return type parameter R (Approach 2) or typed attribute signatures (Approach 4) to catch this.

Q

What does pivotToAttrCentric() do, and why is it needed?

A

It reshapes production-centric data (keyed by kind, then by attribute) into attr-centric data (keyed by attribute, then by kind). It's needed because compile.ts expects attr-centric AttrDecl with equations: Record<kind, fn>. The pivot lets us define overrides in the type-safe production-centric format while feeding codegen the format it already understands.

Q

What is "two-dimensional type safety" and which approach achieves it?

A

Two-dimensional type safety validates both axes simultaneously: the outer key (kind name, validated against TSKind) and the inner key (attribute name + function signature, validated against AttrEquationSignatures). Approach 4 (production-centric) achieves this with Partial<Record<TSKind, Partial<AttrEquationSignatures>>>.

Q

How does Silver's propagate mechanism relate to our SynAttr.default field?

A

They're the same pattern: "auto-generate a default equation for all productions, allow overrides for specific ones." Silver's propagate errors on Expr, Stmt auto-generates collection/copy equations. Our default: eq_kindDefs_default provides the equation used for all kinds that don't have a specific override in equations. Both use "default for all + override for specific kinds."

Q

What does the const _: never = this.node.kind trick catch in Approach 5, and when does the error surface?

A

It catches grammar/evaluator drift. If a new kind is added to the grammar but npm run codegen isn't rerun, the generated switch won't have a case for the new kind. TypeScript will error because the kind value can't be assigned to never. The error surfaces during tsc (type checking of the generated code), not at runtime.

Q

Why does compile.ts require no changes in the production-centric approach?

A

Because pivotToAttrCentric() reshapes the data back into the attr-centric format that compile.ts already expects. The AttrDecl shape is unchanged — it still has equations: Record<string, Function>. The type safety lives in the spec definition site (where KindOverrides is checked), not in the codegen consumer. The generic default K = string means compile.ts sees plain AttrDecl without type parameters.

Q

Why are SynSigs and InhSigs defined separately instead of as a single AttrEquationSignatures type?

A

Because inherited attribute parentEquations have a different return type contract: they return T | undefined where undefined means "copy-down" (use parent's value). Synthesized equations return T directly. Separating the types prevents accidentally mixing syn and inh equation signatures, and makes the override records clearer about which equations are syn overrides vs inh overrides.

Q

Why is validateFieldExpressions() kept even though the DSL that validated helper names is removed?

A

Because validateFieldExpressions() validates ALL expression strings (including hand-written ones and auto-detect defaults), not just DSL-constructed ones. With the DSL gone, it becomes the single, clear validation mechanism — checking for unknown function references via regex scan of the expression strings. One validation path is cleaner than two overlapping systems.
