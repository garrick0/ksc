What Is a Monad

A visual, interactive explanation — from the problem it solves, through the mechanics, to real examples in Haskell and TypeScript.

1. The Problem

You have functions that return wrapped values. A function doesn't just return a number — it returns a number that might be missing, or a number plus some log output, or a number that required IO. The wrapper carries extra context.

Without monads, you'd manually unwrap at every step:

case lookupUser uid of
  Nothing  -> Nothing                   -- propagate failure
  Just user -> case getAddress user of
    Nothing   -> Nothing                -- propagate failure again
    Just addr -> getPostcode addr       -- finally

Every step: check for Nothing, unwrap, pass to next function, re-wrap. This is tedious and repetitive. A monad automates this plumbing.

2. The Solution: Bind

A monad gives you one operation that solves the problem: bind (written >>= in Haskell).

Maybe
User
>>=
getAddress
=
Maybe
Address
Bind takes a wrapped value and a function that expects an unwrapped value.
It handles the unwrapping, applies the function, and gives back a wrapped result.

Now you can chain all three:

lookupUser uid >>= getAddress >>= getPostcode

Each >>= handles the plumbing. If any step returns Nothing, the whole chain short-circuits. If every step succeeds, the values flow through.

4. The Two Operations

A monad is any type m that implements exactly two operations:

return (aka pure / unit)

return :: a -> m a

Takes a plain value and wraps it. Put a value into the container. For Maybe: return 5 gives Just 5. For List: return 5 gives [5].

bind (>>= )

(>>=) :: m a -> (a -> m b) -> m b

Takes a wrapped value and a function that produces a wrapped value. Unwraps, applies, returns the new wrapped result. The how of unwrapping depends on the specific monad.

That's the entire interface. Everything else — do notation, monad transformers, all the patterns — is built on these two operations.

5. What Bind Does for Each Monad

The magic is that different monads define different plumbing in their bind. Same interface, different behavior between the steps:

Maybe
List
Either
IO
State
Writer

Maybe — short-circuit on failure

Bind checks if the left side is Nothing. If so, skip the function and return Nothing. If it's Just x, unwrap x and pass it to the function.

-- Bind implementation for Maybe:
Nothing  >>= f = Nothing        -- short-circuit
(Just x) >>= f = f x            -- unwrap, apply

Plumbing: "If anything fails, stop immediately and propagate the failure."

List — try all combinations

Bind applies the function to every element and concatenates all the results. This is just flatMap.

-- Bind implementation for List:
xs >>= f = concat (map f xs)
-- or equivalently: concatMap f xs

-- Example:
[1,2,3] >>= \x -> [x, x*10]
-- = concat [[1,10], [2,20], [3,30]]
-- = [1, 10, 2, 20, 3, 30]

Plumbing: "Explore every possibility. Nondeterminism."

Either — failure with a reason

Like Maybe, but Left carries an error value instead of just being empty. Bind propagates the error without running the next function.

-- Bind implementation for Either:
(Left err) >>= f = Left err     -- propagate error
(Right x)  >>= f = f x          -- unwrap, apply

-- Example:
parseAge "abc" >>= validateRange
-- parseAge returns Left "not a number"
-- bind short-circuits: Left "not a number"
-- validateRange never runs
Left "not a number"
>>=
validateRange
=
Left "not a number"

Plumbing: "If anything fails, stop and carry the error message through."

IO — sequence side effects

An IO a is a description of an action that produces an a. Bind sequences these descriptions: "do this action, take the result, feed it to the next action."

-- IO bind sequences real-world actions:
getLine >>= \name -> putStrLn ("Hello, " ++ name)

-- Step 1: perform getLine (read from stdin)
-- Step 2: take the string, pass to the lambda
-- Step 3: perform putStrLn (print to stdout)
IO "Sam"
>>=
greet
=
IO "Hello, Sam"

Plumbing: "Execute this effect, take the result, feed it to the next effect. Enforce sequencing."

This is Haskell's entire approach to side effects. All IO lives in the IO monad. You can't extract values from it — you can only compose IO actions together. The runtime executes the final composed action.

State — thread a value through

A State s a is a function s → (a, s) — takes a state in, returns a result and a new state. Bind threads the state from one step to the next.

-- State s a  is really just:  s -> (a, s)
-- Bind implementation:
m >>= f = \s0 ->
  let (a, s1) = m s0       -- run first, get result + new state
      (b, s2) = (f a) s1   -- run second with new state
  in (b, s2)                -- return final result + final state

Plumbing: "Carry a state value forward through each step, updating it along the way."

Writer — accumulate a log

A Writer w a is a pair (a, w) — a value plus accumulated output. Bind runs the function and concatenates the logs.

-- Writer w a  is really just:  (a, w)
-- Bind implementation:
(a, w1) >>= f =
  let (b, w2) = f a       -- apply function to value
  in (b, w1 <> w2)          -- combine logs with (<>)

-- Example:
addTax price   = (price * 1.2, ["Added 20% tax"])
roundUp amount = (ceiling amount, ["Rounded up"])

(100, ["Base price"]) >>= addTax >>= roundUp
-- = (120, ["Base price", "Added 20% tax", "Rounded up"])

Plumbing: "Run the computation, combine the log outputs from each step."

6. Do Notation — Syntactic Sugar

Haskell provides do notation which makes monadic code look imperative. It's purely syntactic sugar over >>=:

With bind (>>=)

lookupUser uid >>= \user ->
getAddress user >>= \addr ->
getPostcode addr >>= \code ->
return (user, code)
=

With do notation

do
  user <- lookupUser uid
  addr <- getAddress user
  code <- getPostcode addr
  return (user, code)

Every x <- desugars into a >>= \x ->. The do block looks like sequential statements with variable assignment, but it's actually a chain of bind calls. The monad's bind decides what happens between lines — short-circuit for Maybe, explore all paths for List, sequence effects for IO.

7. The Three Laws

For return and >>= to form a valid monad, three laws must hold. These aren't checked by the compiler — they're a contract the implementor promises:

Left Identity

return a >>= f  ≡  f a

Wrapping a value and immediately binding it into a function is the same as just calling the function. return doesn't add any extra context — it's a neutral wrapper.

Just 5
>>= f ≡ f 5

Right Identity

m >>= return  ≡  m

Binding a wrapped value into return gives you back the same wrapped value. Unwrapping and re-wrapping is a no-op.

Just 5
>>= return ≡
Just 5

Associativity

(m >>= f) >>= g  ≡  m >>= (\x -> f x >>= g)

Chaining binds in different groupings gives the same result. It doesn't matter whether you group left or right — the chain is the chain.

These correspond exactly to the identity and associativity laws for morphism composition in a Kleisli category — the category theory structure underlying monads.

9. In TypeScript Terms

You've already used monads in TypeScript — you just didn't call them that:

Promise

Promise<A> is a monad. .then() is bind. It unwraps the resolved value, passes it to your callback, and returns a new Promise.

fetch("/api")
  .then(res => res.json())
  .then(data => process(data))

Plumbing: sequence async operations, propagate rejections.

Array

Array<A> is a monad. .flatMap() is bind. It applies the function to every element and flattens the result.

[1,2,3].flatMap(x => [x, x*10])
// [1, 10, 2, 20, 3, 30]

Plumbing: try every element, collect all results.

Optional chaining

user?.address?.postcode is Maybe-monad bind in disguise. Each ?. checks for null/undefined and short-circuits if missing.

const code = user?.address?.postcode
// If user is null → undefined
// If address is null → undefined
// Otherwise → postcode value

Plumbing: short-circuit on null/undefined.

Effect-TS

Effect<R,E,A> is explicitly a monad. Effect.flatMap() is bind. It threads requirements, accumulates errors, and sequences effects.

pipe(
  getUser(id),
  Effect.flatMap(getAddress),
  Effect.flatMap(getPostcode),
)

Plumbing: track deps (R), propagate errors (E), sequence effects.

10. The Big Picture

A monad is a pattern for composing functions that produce wrapped values. return wraps. bind unwraps-and-rewraps with custom plumbing in between. Different monads, different plumbing, same interface. That's it.

▶ The category theory view (optional)

A monad is an endofunctor T : C → C equipped with two natural transformations: η : Id → T (unit, aka return) and μ : T∘T → T (join, aka flatten). These satisfy the associativity and identity laws. In programming terms: T is the type constructor (Maybe, List, IO), η is return, and μ is flattening nested wrappers (Maybe (Maybe a) → Maybe a). Bind is derived: m >>= f = join (fmap f m). The Kleisli category has the same objects as C but morphisms a → T b, with bind as composition.
