The File / Sub-File Boundary Problem

KindScript needs to assign properties at both the module level (files, folders, architecture layers) and the sub-file level (expressions, declarations, statements). These two worlds have different mechanics — how do we unify them?

Part 22 — KindScript Research Series

1. The Problem: Two Languages, One Property System

TypeScript has what we've called the "core language" (expressions, statements, declarations within a file) and the "module language" (imports, exports, file identity, dependency graph). These operate at fundamentally different levels with different mechanics:

Sub-File Level

The "core language." Works on AST nodes. TypeScript's type checker understands this deeply.

// TypeScript assigns types here:
const x = 5;              // symbol + type
Math.sin(x);              // expression type
if (x > 0) { ... }        // well-formedness
class Foo {
  private id: string;   // member visibility
  public name: string;   // member visibility
}
// TS knows EVERYTHING about what's inside

File Level

The "module language." Works on file paths and the import graph. TypeScript's type checker sees this through a narrow lens.

// TypeScript sees modules as export signatures:
import { sort } from './utils';
export function process() { ... }

// What TS knows about './utils':
//   - Its export signature (typeof import)
//   - Nothing about its internals
//   - Nothing about its file path
//   - Nothing about its "layer"
//   - Nothing about its role in architecture

1.1 The Containment Isomorphism

Here's your key insight: containment with visibility is the same pattern at every level. Click each level to see:

📁 Folder: src/domain/
Contains files. Some are "public API" (re-exported via index.ts), others are internal implementation. exported via index not re-exported
📄 Module: user.ts
Contains declarations. Some are exported, others are internal to the file. export no export
🔷 Class: User
Contains members. Some are public, others are private. public private
🔹 Method: validate()
Contains statements and expressions. Local bindings are scoped. local scope
{ } Block: if branch
Contains statements. Block-scoped declarations. block scope
The pattern is identical at every level: a container holds named things, some visible outside, some hidden. Folder → file → class → function → block. The mechanism differs (export keyword vs public/private vs let/const scoping), but the structure is the same. If we could unify the mechanism, there'd be no friction between "file-level properties" and "sub-file properties."

1.2 Where the Friction Comes From

① Modules aren't values in JS/TS

In Rust, a module is a path you can reference: use crate::domain::user. In OCaml, modules are first-class values with types (signatures). In JS/TS, a module is a compilation unit — it's a file on disk, not a runtime entity. You can't write const m = myModule. You can only reference specific exports.

This means TypeScript's type system has no native representation for "the module as a whole." typeof import('./user') gives you the export signature, but that's like typing a class by only its public methods — you lose all the internal structure.

② Architecture lives outside the type system

When you say "the domain layer can only import from the domain layer," that's a constraint on file paths and folder structure. TypeScript's type system doesn't know about file paths. It doesn't know that src/domain/user.ts is a "domain" file. That's metadata that lives in your head, in folder naming conventions, or in external tool configuration.

This is why tools like Nx (tags on projects), eslint-plugin-boundaries (file pattern → element type mapping), and dependency-cruiser (regex path rules) all had to invent their own configuration systems outside TypeScript.

③ Different visibility models at different levels

Within a class, visibility is public/private/protected — TypeScript enforces these and includes private members in the type's internal representation. At the module level, visibility is export or nothing — but TypeScript only exposes exports in the module's type. At the folder level, there's no visibility mechanism at all — it's purely convention (barrel files, index.ts re-exports).

So when you want to say "this folder only exports User and createUser," you're using an index.ts convention, not a language feature. There's no TypeScript syntax for folder-level export control.

④ The "what is the complete object?" question

For a class, the "complete object" is clear — all members (public + private). TypeScript has a full type for it. For a module, the "complete object" is ambiguous. Is it just the exports? All declarations? The import graph? The file's role in the architecture? There's no single "module type" that captures everything KindScript might want to know.

Your idea of a "custom file object" comes from this — if TypeScript doesn't model modules completely enough, maybe KindScript should build its own richer model.

2. How Existing Tools Handle This

Every TypeScript architecture enforcement tool has hit this exact friction and invented its own solution. Here's what they do:

eslint-plugin-boundaries — Pattern-Based Element Types

The most sophisticated ESLint-based approach. It solves the friction by creating an explicit mapping layer between file paths and semantic "element types":

// .eslintrc — bridge layer between files and semantics
{
  "settings": {
    "boundaries/elements": [
      { "type": "domain",    "pattern": "src/domain/*" },
      { "type": "service",   "pattern": "src/services/*" },
      { "type": "component", "pattern": "src/components/*" },
      { "type": "infra",     "pattern": "src/infrastructure/*" }
    ],
    "boundaries/dependency-nodes": ["import"]
  },
  "rules": {
    "boundaries/element-types": ["error", {
      "default": "disallow",
      "rules": [
        { "from": "domain",    "allow": ["domain"] },
        { "from": "service",   "allow": ["domain", "service"] },
        { "from": "component", "allow": ["domain", "service"] },
        { "from": "infra",     "allow": ["domain", "infra"] }
      ]
    }]
  }
}

Key insight: The "element type" concept is a bridge between the file world and the rule world. Files get classified by glob patterns, then rules operate on the classifications, not the files directly. This is essentially a type system for files defined in configuration.

✓ Works within ESLint. Rules fire on import AST nodes (sub-file). Classification is by file path (file-level).

✗ The classification is separate from the code. The code itself doesn't know it's a "domain" file.

Nx — Tag-Based Project Constraints

Nx operates at the project level (one level above files). Each project gets tags, and rules constrain tag relationships:

// project.json for the "domain" library
{
  "name": "domain",
  "tags": ["layer:domain", "scope:user"]
}

// nx.json — global constraint rules
{
  "depConstraints": [
    {
      "sourceTag": "layer:domain",
      "onlyDependOnLibsWithTags": ["layer:domain", "layer:shared"]
    },
    {
      "sourceTag": "layer:feature",
      "onlyDependOnLibsWithTags": ["layer:domain", "layer:feature", "layer:shared"]
    }
  ]
}

Key insight: Tags are multi-dimensional — a project can have both layer:domain and scope:user. Constraints can operate on any tag dimension. This is like an ad-hoc type system for the project graph.

✓ Clean, declarative. Multi-dimensional tags are powerful. Battle-tested at scale.

✗ Project-level only — can't say "this FILE within a project is special." Also completely divorced from the TypeScript type system.

dependency-cruiser — Regex Path Rules

Uses a configuration DSL where rules match against file path patterns:

// .dependency-cruiser.js
module.exports = {
  forbidden: [
    {
      name: 'domain-no-infra',
      from: { path: '^src/domain' },
      to: { path: '^src/infrastructure' }
    },
    {
      name: 'no-circular',
      from: {},
      to: { circular: true }
    },
    {
      name: 'no-orphans',
      from: { orphan: true },
      to: {}
    }
  ]
};

Key insight: dependency-cruiser operates on the import graph — it builds a graph of file → file dependencies, then checks it against rules. It also detects structural properties like circular dependencies and orphan files that only make sense at the graph level.

✓ Powerful graph analysis. Detects structural problems (cycles, orphans). Visualizes the dependency graph.

✗ Completely separate from the type system. Rules are about paths, not about what's inside the files.

ArchUnitTS — Architecture as Tests

Treats architectural constraints as unit tests:

// architecture.test.ts
import { filesOfProject } from 'ts-arch';

describe('Architecture', () => {
  it('domain should not depend on infrastructure', () => {
    const rule = filesOfProject()
      .inFolder('domain')
      .shouldNot()
      .dependOnFiles()
      .inFolder('infrastructure');
    expect(rule).toPass();
  });

  it('all domain files should export a function named *Decider', () => {
    const rule = filesOfProject()
      .inFolder('domain')
      .should()
      .matchPattern('*Decider*');
    expect(rule).toPass();
  });
});

Key insight: Architecture rules become executable assertions. The test framework provides a fluent API for querying the file/dependency graph. It bridges file-level concerns (which folder?) with sub-file concerns (what's exported?) through the API.

✓ Integrated with test runners. Can mix file-level and sub-file queries. Familiar assertion syntax.

✗ Only runs during testing, not during editing. Errors show in test output, not in the IDE.

3. How Other Languages Solved This

Rust — Unified Visibility at Every Level

Rust arguably has the cleanest solution. There is no "two-language problem" because visibility modifiers work identically at every scope level:

// Module (file) level
pub fn public_api() {}           // visible outside this module
fn internal() {}                   // private to this module
pub(crate) fn crate_only() {}    // visible within the crate only
pub(super) fn parent_only() {}   // visible to parent module only
pub(in path::to::mod) fn targeted() {} // visible to specific module

// Struct (class) level — SAME modifiers
pub struct User {
    pub name: String,                // public field
    pub(crate) email: String,       // crate-internal field
    password_hash: String,          // private field (default)
}

// Nested module — same modifiers again
mod internal {
    pub(super) fn helper() {}     // visible to parent
}
Why this works: Rust has one visibility system that applies uniformly from the crate level down to individual struct fields. There's no friction between "module-level" and "item-level" because they use the exact same pub(...) mechanism. A module is a scope, a struct is a scope, a function is a scope — and the same visibility rules apply everywhere.

OCaml — Modules as First-Class Typed Entities

OCaml goes further: modules are values with types. Module types are called "signatures," and you can abstract over modules using "functors" (functions from modules to modules):

(* Module SIGNATURE — the "type" of a module *)
module type REPOSITORY = sig
  type t                          (* abstract type — hidden impl *)
  val find : string -> t option   (* exported function *)
  val save : t -> unit           (* exported function *)
end

(* Module IMPLEMENTATION — satisfies the signature *)
module PostgresRepo : REPOSITORY = struct
  type t = { id: int; data: string }   (* concrete type *)
  let find key = ...
  let save item = ...
  let internal_helper = ...            (* NOT in signature → hidden *)
end

(* FUNCTOR — a function from modules to modules *)
module MakeService (R: REPOSITORY) = struct
  let process key =
    match R.find key with
    | Some item -> R.save item
    | None -> ()
end
Why this matters: In OCaml, a module's "type" (its signature) is as rich as a class's type. You can say "this module satisfies this signature" just like you say "this value has this type." The containment isomorphism is fully realized — modules, classes, and values all participate in the same type system. If KindScript could give modules signatures that include behavioral properties, the file/sub-file boundary would dissolve.

Java JPMS — Explicit Module Descriptors

Java 9 introduced a dedicated module-info.java file at the package root — an explicit "module language" file:

// module-info.java — declares module properties
module com.example.domain {
    // What this module needs
    requires com.example.shared;

    // What this module exposes (and to whom)
    exports com.example.domain.api;
    exports com.example.domain.spi to com.example.infra;

    // Service provider relationships
    uses com.example.domain.spi.Repository;
    provides com.example.domain.spi.Repository
        with com.example.domain.internal.InMemoryRepo;
}
Key insight: Java acknowledged the two-language problem and solved it by creating an explicit module descriptor language. The module-info.java file is effectively a "configuration file written in a subset of Java syntax." It lives in the code, uses Java-like syntax, but has completely different semantics from regular Java code.

Haskell — Explicit Export Lists

Haskell modules have a simple but powerful mechanism: the export list controls exactly what's visible:

-- Module with explicit exports
module Domain.User
  ( User           -- export the type
  , createUser     -- export the constructor function
  , userName       -- export a field accessor
  -- NOT exporting: userPassword, internalValidate
  ) where

data User = MkUser    -- constructor NOT exported → abstract type
  { userName :: String
  , userPassword :: String  -- field not in export list → inaccessible
  }

createUser :: String -> String -> User
createUser = MkUser

internalValidate :: User -> Bool   -- not exported
internalValidate u = length (userPassword u) > 8
Interesting detail: Haskell can export a type without its constructors. Outside modules can use User values but can't pattern match on them or construct them directly — they must use createUser. This gives the module author control over invariants. The export list is the module's "interface" and it can be more restrictive than the module's actual contents.

4. Options for KindScript

Here are seven approaches, from simplest to most ambitious. Click each to explore in detail:

A Unified Scope Model (Rust-inspired) ★★★★★

Treat every containment level uniformly. A file is just another scope. A folder is just another scope. Properties apply to any scope. No separate "module language."

// KindScript sees a uniform hierarchy of scopes:
//
// Scope: src/domain/          [layer: "domain", pure: true]
//   Scope: user.ts             [inherits: pure]
//     Scope: class User        [inherits: pure]
//       Scope: validate()      [inherits: pure]
//         Expression: x > 0    [pure: ✓]
//
// Properties at any level apply to everything contained.
// "This folder is pure" means every file, class, function,
// and expression within must be pure.

// In code, via file-level annotation:
// @ksc scope: pure, layer: "domain"
export function createUser(name: string): User { ... }

// Or in a folder-level config file (ksc.config.ts):
// scope("src/domain/**") { pure: true, layer: "domain" }

How it handles your architecture example:

// "Domain layer must contain functions named *Decider"
// In KindScript's scope model:
scope("src/domain/**") {
  layer: "domain",
  requires: {
    containsExport: /.*Decider$/  // at least one export matches
  }
}

// "Domain can only import from domain or shared"
scope("src/domain/**") {
  canImportFrom: ["src/domain/**", "src/shared/**"]
}

✓ Elegant: one concept (scope) at every level. Properties compose naturally — inner scopes inherit from outer. The containment isomorphism is fully exploited.

✓ File naming rules ("must have *Decider") and export rules are just scope constraints, same as "this function must be pure."

✗ Requires KindScript to model folder structure, which TypeScript doesn't do. Need to build folder-level scope resolution.

✗ Inheritance semantics need care: does a "pure" folder annotation mean "every expression must be pure" or "every export must be pure"?

B Module Virtual Types ★★★★

Create an internal "virtual type" for each file that represents everything in it (not just exports). Apply properties to this virtual type using the same machinery as expression/declaration properties.

// KindScript internally constructs:
type __Module_user_ts = {
  // ALL declarations, not just exports
  export createUser: (name: string) => User;
  export class User { name: string; private id: string; };
  // Internal declarations too:
  internalValidate: (u: User) => boolean;
  SALT: string;
  // Metadata:
  __path: "src/domain/user.ts";
  __imports: ["src/shared/types.ts"];
  __hasTopLevelSideEffects: false;
};

// Then properties work on this type:
// @ksc pure, layer: "domain"
// ...checker verifies the virtual type satisfies these

✓ Unifies with sub-file property system — modules become "just another typed entity."

✓ Can include internal declarations, so properties can reason about everything in the file, not just exports.

✗ The virtual type is a KindScript invention — it doesn't correspond to anything in TypeScript.

✗ Metadata properties (path, imports) are stringly-typed and feel bolted-on.

C Extend typeof import() ★★★

Use TypeScript's existing typeof import('./mod') as the module type, and extend it with KindScript properties.

// TypeScript already gives you:
type Utils = typeof import('./utils');
// = { sort: (arr: T[]) => T[]; filter: ... }

// KindScript would annotate this:
// @ksc moduleProperties("./utils") { pure: true, layer: "shared" }

// When importing, KindScript checks:
import { sort } from './utils';
// → look up moduleProperties for './utils'
// → sort inherits the module's purity property
// → if current file is @ksc pure, verify sort is compatible

✓ Builds on existing TypeScript concepts. Minimal new machinery.

✗ typeof import() only captures exports. Can't reason about internal structure.

✗ No folder-level grouping. Each file needs individual annotation.

D Separate Architecture DSL ★★★

Accept the two-language reality. Use KindScript's property system for sub-file concerns and a separate configuration language for file/folder/architecture concerns.

// ksc.architecture.ts — the "module language" for KindScript
export default {
  layers: {
    domain:         { path: "src/domain/**",         pure: true },
    application:    { path: "src/application/**",    pure: true },
    infrastructure: { path: "src/infrastructure/**", pure: false },
    ui:             { path: "src/ui/**",             pure: false },
  },
  rules: [
    { from: "domain",         canImport: ["domain"] },
    { from: "application",    canImport: ["domain", "application"] },
    { from: "infrastructure", canImport: ["domain", "infrastructure"] },
    { from: "ui",             canImport: ["domain", "application", "ui"] },
  ],
  naming: [
    { layer: "domain", requiresExport: /.*Decider$/ },
  ]
};

// In-file annotations still work for sub-file properties:
// @ksc pure, noIO
function processOrder(order: Order): Decision { ... }

✓ Proven pattern (Nx, eslint-plugin-boundaries, dependency-cruiser all do this). Familiar to developers.

✓ Architecture config is centralized and readable. Easy to review in PRs.

✗ Two systems to learn. Properties in config vs properties in code. Harder to see the connection.

✗ Configuration can drift from reality if not enforced.

E Co-located Folder Config (.ksc files) ★★★

Each folder can have a .ksc config file that declares properties for that folder and its contents. Properties cascade down — inner folders inherit from outer folders.

// src/domain/.ksc
{
  "layer": "domain",
  "pure": true,
  "canImportFrom": ["domain", "shared"],
  "exportNaming": "*Decider"
}

// src/domain/orders/.ksc — inherits from parent, adds specifics
{
  "subdomain": "orders",
  "maxFanOut": 5
}

// src/infrastructure/.ksc
{
  "layer": "infrastructure",
  "pure": false,
  "canImportFrom": ["domain", "infrastructure", "shared"]
}

✓ Properties live next to the code they describe. Cascading inheritance is natural.

✓ Feels like .eslintrc or tsconfig.json — familiar convention.

✗ Configuration scattered across the codebase. Hard to see the full architecture at a glance.

✗ Merge conflicts when multiple developers modify the same .ksc files.

F Module Signatures (OCaml-inspired) ★★★★

Create explicit "module signatures" that describe what a module should look like — its exports, their properties, and its architectural role. The module must conform to its signature.

// src/domain/user.ksc.ts — the module signature
import type { ModuleSignature } from '@ksc/types';

export const signature: ModuleSignature = {
  layer: "domain",
  pure: true,
  exports: {
    User: { kind: "class", properties: ["immutable", "serializable"] },
    createUser: { kind: "function", properties: ["pure", "total"] },
    validateUser: { kind: "function", properties: ["pure"] },
  },
  canImportFrom: ["src/domain/**", "src/shared/**"],
};

// KindScript verifies: does user.ts actually satisfy this signature?
// - Does it export User, createUser, validateUser?
// - Are they pure/total/immutable/serializable as declared?
// - Does it only import from allowed paths?

✓ Modules get first-class "types" (signatures). Directly parallels how TypeScript gives expressions types.

✓ Signatures can specify both structural requirements (what's exported) and behavioral properties (purity, totality).

✓ Conformance checking is analogous to type checking — "does this module satisfy its signature?"

✗ Requires maintaining signature files alongside implementation files.

✗ Could feel redundant — TypeScript already generates .d.ts files for structural signatures.

G Hybrid: Annotations + Glob Selectors ★★★★★

Combine the best of both: in-file annotations for sub-file properties (where you already are) + glob-based selectors in a central config for file/folder properties. But use the same property language for both.

// ksc.config.ts — central config with glob selectors
export default {
  scopes: [
    {
      match: "src/domain/**",
      properties: { layer: "domain", pure: true },
      canImportFrom: ["src/domain/**", "src/shared/**"],
      requires: { exportMatching: /Decider$/ }
    },
    {
      match: "src/infrastructure/**",
      properties: { layer: "infra" },
      canImportFrom: ["src/domain/**", "src/infra/**", "src/shared/**"]
    },
  ]
};

// In-file annotation (sub-file level) — same property names!
// @ksc pure, total
export function orderDecider(event: OrderEvent): Decision {
  // checker verifies purity and totality at expression level
  ...
}

// The file inherits { layer: "domain", pure: true } from config.
// The function gets { pure, total } from its annotation.
// KindScript sees one unified property set for each scope level.

✓ One property language for everything — "pure" means the same thing whether applied to a folder, file, or function.

✓ Central config handles cross-cutting concerns (layers, import rules). In-file annotations handle per-function specifics.

✓ Config is optional — you can start with just in-file annotations and add config later.

✓ Glob selectors are the natural "bridge" between file paths and property assignments.

✗ Two places to look for properties (config file + source annotations). Need clear precedence rules.

✗ Glob patterns can be fragile — renaming a folder breaks config rules.

6. The Recommended Path

Start with G (Hybrid), evolve toward A (Unified Scope)

The hybrid approach gives you the most practical value immediately while leaving the door open for a more principled unification later:

Phase 1 In-file annotations (what KindScript already has)

Keep @ksc pure on functions. This is the sub-file level. Already working. No changes needed.

Phase 2 Add file-level annotations

Allow a @ksc comment at the top of a file to set properties for the entire file. A @ksc pure at the file level means "every export from this file must be pure." This bridges file-level and sub-file by using the same annotation syntax.

// @ksc pure, noIO
// ↑ file-level: applies to all declarations in this file

export function add(a: number, b: number) { return a + b; } // ✓
export function log(msg: string) { console.log(msg); } // ✗ violates noIO
Phase 3 Add glob-based config for folder/architecture rules

A ksc.config.ts that maps file patterns to properties and import constraints. Same property names as in-file annotations. The config is the "architecture language" — it says which files are in which layer and who can import whom.

Phase 4 Unify into scope model

Once you have both in-file and config-based properties working, the unification is straightforward: everything is a scope, properties cascade from outer scopes to inner scopes. The config defines outer scopes (folders → files). The annotations define inner scopes (files → declarations → expressions). The cascade rules are: inner overrides outer, more specific overrides less specific.

The key principle: Use the same property names and semantics at every level. "pure" means the same thing whether it appears in a folder config, a file-level annotation, or a function-level annotation. The only difference is the scope it applies to. This makes the system feel like one thing, even if the implementation has different mechanisms for different levels.

Test Your Understanding

1. Why can't KindScript simply use TypeScript's typeof import('./mod') to represent module properties?

typeof import('./mod') returns the module's export signature — essentially its public interface. But KindScript needs to reason about: internal declarations (are ALL functions pure, not just exported ones?), the file's path (to determine its architectural layer), its import graph (who does it depend on?), and top-level side effects (does importing it do anything?). The export signature captures none of this.

2. How does Rust avoid the file/sub-file friction entirely?

Rust's key insight: visibility is just "who can see this name." The pub(in path) modifier works identically whether applied to a function in a module, a field in a struct, a method in an impl block, or a nested module. There's no "module language" vs "core language" — it's one language with one visibility system. This eliminates the friction because there's no boundary to cross.

3. In the "Hybrid" approach (Option G), what bridges the file-level and sub-file-level worlds?

The bridge is semantic consistency. When "pure" in a glob config means exactly the same thing as "pure" in a function annotation, developers think in one concept even though the mechanism differs. The config says "files in src/domain/ are pure" and the annotation says "this function is pure" — same word, same verification, different scope. This makes the two-level system feel like one system.

4. What is eslint-plugin-boundaries' "element type" concept an example of?

Element types are the explicit bridge between the file world and the rule world. The configuration maps file glob patterns (a file-level concept) to named "element types" like "domain", "service", "component" (a semantic concept). Then rules operate on these semantic types, not on raw file paths. The element type IS the connection point between "where is this file?" and "what rules apply to it?"

5. Why is the containment isomorphism (folder → file → class → function → block) relevant to KindScript's design?

The isomorphism reveals that folder, file, class, function, and block are all instances of the same abstract concept: "a container of named things with a visibility boundary." If KindScript can model this abstract concept once and apply it at every level, there's no friction between "file properties" and "function properties" — they're the same thing at different nesting depths. The Rust model proves this is achievable: pub(in path) works uniformly from crate level to field level.
