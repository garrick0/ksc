1. What Is Partial Evaluation?

A partial evaluator (also called a specializer or mix) is a program transformer. Given a program P that takes inputs (s, d), where s is known now (static) and d will only be known later (dynamic), a partial evaluator produces a residual program P_s that only needs d:

The Specialization Equation
mix(P, s) = Ps    such that    P(s, d) = Ps(d)   for all d

The residual program P_s is specialized to s. All computations that depended only on s have been pre-computed and "baked in". What remains is only the work that truly requires d.

▶ Concrete Example: The Power Function

Consider a generic power function:

function power(base, exp) {
  if (exp === 0) return 1;
  if (exp === 1) return base;
  return base * power(base, exp - 1);
}

If we know that exp = 3 at specialization time, a partial evaluator produces:

// mix(power, {exp: 3}) produces:
function power_3(base) {
  return base * base * base * 1;
}

The recursion, conditionals, and counter are all gone — they've been evaluated away because they only depended on the static input exp. What remains is a straight-line multiplication that only needs the dynamic input base.

The Three Key Concepts

1. Binding Times

Every sub-expression in a program is classified as either static (computable from known inputs) or dynamic (depends on unknown inputs). This classification is called Binding-Time Analysis (BTA).

2. The Specializer (mix)

The specializer processes the annotated program: it reduces (evaluates) static expressions and rebuilds (generates code for) dynamic expressions. The output is a residual program that is equivalent to the original but faster, because all static work has been pre-done.

3. Residual Programs

The output of specialization. A residual program contains only the dynamic parts of the original. It is a new, specialized program that runs faster because some of its inputs have been "compiled in".

2. The Futamura Projections (Including the Fourth)

Yoshihiko Futamura (1971) discovered that applying mix to interpreters and to itself yields profound results. Robert Glück (2009) showed that the abstraction scheme continues beyond three, identifying the Fourth Projection and proving that projections 4–6 form a closed system. The classical setup starts with:

Interpreter
interp(source, input) = output

An interpreter takes a program (source) and that program's input and produces the program's output. Now apply mix:

First Futamura Projection: Compilation

Specialize the interpreter on a specific program
target = mix(interp, source)

By specializing the interpreter for a specific source program, we get a target program (compiled code). The interpreter's dispatch loops, parse logic, and generic tree-walking that depend on source are all pre-computed. What's left is a residual program that only needs input.

This target is equivalent to a compiled version of the source — it runs directly without needing the interpreter at runtime.

Key insight: Compilation is just partial evaluation of an interpreter with respect to a source program. A compiler is nothing more than a specialized interpreter.

Second Futamura Projection: Compiler Generation

Specialize mix on the interpreter itself
compiler = mix(mix, interp)

By specializing the specializer with respect to the interpreter, we get a compiler. This compiler is a program that takes any source program and produces a target program — without needing mix at compile time.

The resulting compiler has the interpreter's semantics "baked in" and can translate any program from the source language to the target language.

Key insight: We never wrote a compiler. We wrote an interpreter and a specializer, and the compiler emerged from their interaction.

Third Futamura Projection: Compiler Generator

Specialize mix on itself
compiler_generator = mix(mix, mix)

By specializing the specializer with respect to itself, we get a compiler generator (cogen). This is a program that takes any interpreter and produces a compiler for that interpreter's language.

Feed it a Python interpreter → get a Python compiler. Feed it a SQL interpreter → get a SQL compiler. The mix machinery has been "compiled away" into a general-purpose compiler factory.

Fourth Futamura Projection: Self-Generating Compiler Generator

Apply the compiler generator to the specializer itself
cogen' = cogen(mix)    where    cogen = mix(mix, mix)

The Third Projection produces a compiler generator cogen. Glück observed that cogen is itself a program that takes one input (an interpreter). If we feed cogen the specializer mix as its "interpreter", we get cogen' — a self-generating compiler generator.

The remarkable result: cogen' = cogen. The compiler generator reproduces itself. It is a fixpoint of the abstraction process. Glück proved that projections 4, 5, and 6 all collapse to this same identity — the system is closed.

The fixpoint chain (Glück 2009)
cogen(mix) = cogen     [4th ≡ 3rd]

cogen(cogen) = cogen     [self-generating]
Key insight: The abstraction process has a ceiling. After three projections, you've extracted all the generality there is. The fourth projection proves this by showing that the compiler generator is a fixpoint — specializing it further just reproduces itself. This is analogous to how the derivative of ex is itself: further differentiation produces no new information.
Practical meaning: If you have a working compiler generator (3rd projection), you can bootstrap it: feed it its own specializer and it will reproduce itself. This is the PE equivalent of a self-hosting compiler. Glück experimentally validated this with the Unmix partial evaluator.
Projection Apply mix to... Result What it does
1st mix(interp, source) Target program Compiles one specific program
2nd mix(mix, interp) Compiler Compiles any program in the language
3rd mix(mix, mix) Compiler generator (cogen) Produces a compiler from any interpreter
4th cogen(mix) cogen itself (fixpoint) Proves cogen is self-generating; the system is closed

Each projection eliminates one more layer of interpretation. The first removes the interpreter loop; the second removes the specializer loop; the third removes the meta-specializer loop. The fourth proves that no more layers exist — the compiler generator is a fixpoint under self-application.

Reference: Glück, R. “Is There a Fourth Futamura Projection?” PEPM 2009. Also: Glück, R. “An Experiment with the Fourth Futamura Projection.” Springer LNCS 2010.

3. Multi-Stage Specialization: Beyond Two Inputs

The classical Futamura setup has a program with two inputs: (source, input). But what if your program has three or more inputs that become known at different times?

A 3-input program with progressive binding
P(a, b, c) = result

If a is known first, then b, then c, we can specialize in stages:

  P(a, b, c) = result

  Stage 1:  Pa     = mix(P, a)         // specialize on a
  Stage 2:  Pa,b   = mix(Pa, b)        // further specialize on b
  Stage 3:  result = Pa,b(c)            // execute with c

  Equivalence:  P(a, b, c) = Pa,b(c)  for all c

This is called multi-level partial evaluation or multi-stage programming. Each stage:

  1. Takes in one more piece of "static" information
  2. Produces a new residual program that's more specialized than the previous
  3. Eliminates all computation that depended only on the now-known inputs
This is exactly what KSC does. KSC has a 3-input function: ag_eval(grammar, attributes, program) → results — and it specializes in three stages, each producing generated code that's more specific than the last.

Multi-Stage Programming (MSP) Systems

This idea has been formalized in systems like:

  • MetaOCaml — uses quasi-quotation brackets .<...>. to mark code generation boundaries
  • Lightweight Modular Staging (LMS) — uses types (not syntax) to distinguish binding times: Rep[Int] vs Int
  • Braid — extends staging to heterogeneous targets (GPU, CPU)
  • weval — applies PE at the WebAssembly IR level for interpreter optimization

KSC's approach is closest to offline multi-level PE: binding times are determined at design time (not automatically), and each stage is a separate code generator script.

4. KSC's Architecture as Multi-Stage PE

Let's formally define KSC's complete function and then show how each stage is a specialization.

The Generic AG Evaluator (Stage 0 — the "interpreter")
ag_eval(grammar, attributes, program) = decorated_tree

A completely generic attribute grammar evaluator takes three things:

  1. Grammar — the AST node kinds, their fields, child structure, sum types
  2. Attributes — the equation functions, their types, their dependency graph, their directions (syn/inh/coll)
  3. Program — the actual user source code (a specific TypeScript file)

KSC never runs this generic function directly. Instead, it applies mix twice at build time, producing a highly specialized evaluator that only needs the user's program at runtime:

  Stage 0:  ag_eval(grammar, attributes, program) = results
           The "God function" — fully generic, never actually exists as code

  Stage 1:  spec_grammar = mix(ag_eval, grammar)
           ast-schema/codegen.ts reads schema.ts, generates:
           • node-types.ts (364 typed interfaces)
           • schema.ts (getChildFields, exhaustive child map)
           • convert.ts (TS → KS converters per kind)
           • builders.ts, serialize.ts

  Stage 2:  spec_attrs = mix(spec_grammar, attributes)
           scripts/gen-ksc-evaluator.ts reads behavior/*.ts, generates:
           • evaluator.ts (KSCDNode with typed caches, switch dispatch,
           •   direct eq calls, static dep graph, buildKSCTree)

  Stage 3:  results = spec_attrs(program)
           At runtime: buildKSTree(tsProgram) then evaluate(root)
           Returns decorated tree, definitions, diagnostics

Stage 1   Grammar Specialization

Static input: The TypeScript AST grammar (364 node kinds, their fields, sum type hierarchy)

Specializer: ast-schema/codegen.ts reading ast-schema/schema.ts

Residual program: ast-schema/generated/*

What gets "baked in" at this stage:

  • Every node kind gets its own TypeScript interface with typed fields — no generic Record<string, any>
  • The child field map C is an exhaustive static lookup table — no runtime introspection
  • Each ts.SyntaxKind gets a specific converter function registered at module load — no generic visitor pattern
  • Operator maps, field extractors, symbol flag checks are all specialized per-kind

What's still dynamic after this stage:

  • Which attributes exist, what equations to run
  • The actual user source code

Stage 2   Attribute Specialization

Static input: The 22 AG attributes, their equation functions, their types, their dependency graph

Specializer: scripts/gen-ksc-evaluator.ts reading ksc-behavior/*.ts

Residual program: ksc-generated/evaluator.ts

What gets "baked in" at this stage:

  • Each attribute gets a dedicated typed cache field (_c_kindDefs, _c_defEnv, ...) — no Map<string, any>
  • The attr() method is a switch statement over known attribute names — no hash lookup
  • Each attribute method calls its specific equation function directly (eq_kindDefs_CompilationUnit(this, ...)) — no generic dispatch table
  • Syn/inh/collection evaluation patterns are inlined per attribute — no runtime direction check
  • The dependency graph is pre-computed and embedded as a static constant
  • The topological evaluation order is pre-sorted

What's still dynamic after this stage:

  • The actual user source code (which nodes exist, their field values)

Stage 3   Program Execution

Dynamic input: The user's TypeScript source files

Executor: evaluate(root) in ksc-generated/evaluator.ts

Output: Decorated tree with all 22 attributes computed, plus KindDefinition[] and CheckerDiagnostic[]

What happens at this stage:

  • buildKSTree(tsProgram, depth) uses Stage 1's generated converters to produce a KS tree
  • buildKSCTree(root, counter) wraps each KS node in a KSCDNode with Stage 2's generated caches and methods
  • project_binder(root) and project_checker(root) trigger lazy attribute evaluation using Stage 2's specialized equations
  • Each attribute is computed at most once per node (memoized in typed cache fields)

What Each Stage Eliminates

Stage Generic Operation Eliminated Replaced With
1 "What children does kind X have?" Static lookup table C['BinaryExpression'] = ['left','operatorToken','right']
1 "How do I convert TS node to KS node?" 364 registered converter functions, one per SyntaxKind
1 "What type is this node's field?" TypeScript interfaces with typed fields
2 "What attributes exist?" 22 typed cache fields + 22 dedicated methods
2 "Which equation handles kind X for attr Y?" Switch cases: case 'CompilationUnit': return eq_kindDefs_CompilationUnit(this, ...)
2 "Is attr Y synthesized, inherited, or collection?" Direction-specific inlined evaluation patterns per method
2 "What depends on what?" Static KSC_STATIC_DEP_GRAPH constant + pre-sorted order
2 Map.get(attrName) cache lookup Direct field access: this._c_kindDefs

5. KSC's mix Functions — The Concrete Code

In PE theory, mix(P, s) = P_s is an abstract transformer. In KSC, mix is two concrete TypeScript scripts that read static data and emit specialized code. Here's exactly where they live, what they do line-by-line, and what "applying mix to itself" would concretely mean.

The Two Specializers

mix₁   ast-schema/codegen.ts — Grammar Specializer (1246 lines)

What it is: A script that reads the grammar registry (populated by schema.ts) and generates 5 TypeScript source files, each specialized to the 364 AST node kinds.

Static input being specialized over:

// schema.ts populates these registries via node(), leaf(), sumType() calls:

const sumTypes: Map<string, SumTypeEntry> // 12 sum types → member kinds

// A NodeEntry looks like:
{ kind: 'BinaryExpression', fields: {

    operatorToken: { tag: 'child', typeRef: 'Token' },
    right:         { tag: 'child', typeRef: 'Expression' },

function generate(): string {
  // Per-attribute typed cache field:
  for (const a of ATTRS) {
    L.push(`  private _c_${a.name}: ${a.type} | undefined = undefined;`);
  }

  // Per-attribute switch case in attr() dispatch:
  for (const a of ATTRS) {
    L.push(`      case '${a.name}': return this.${a.name}();`);

  // Per-attribute typed method with direction-specific logic:
  for (const a of ATTRS) {
    generateMethod(L, a);
    // dispatches on a.direction:
    // syn  → switch(node.kind) { case X: return eq_X(this) }
    // inh  → if(isRoot) rootValue; else check parent; else inherit
    // coll → contribute(self) + fold over children
  }

  // Static dep graph embedded as constant:
  for (const [from, to] of edges) {
    L.push(`    ['${from}', '${to}'],`);
  }
}

The Shared Pattern: Read → Analyze → Iterate → Emit

Both mix functions follow the exact same template:

// The universal structure of a KSC mix function:
function mix(staticInput) {
  // 1. READ — load the static data
  const registry = parse(staticInput);

  // 2. ANALYZE — extract metadata
  const metadata = analyze(registry);  // types, deps, patterns

  // 3. GENERATE — iterate over metadata, emit specialized code per item
  const output = [];
  for (const item of metadata) {
    output.push(emitSpecialized(item));
  }

  // 4. WRITE — save the residual program
  writeFile(output.join('\n'));
}

1. Read import './schema.js' populates nodes + sumTypes registries fs.readFileSync loads attr-types.ts, binder.ts, checker.ts as text
2. Analyze Build convertFieldOverrides, resolve sum type refs, detect patterns (isExported, localCount, typeString) scanExports → inferAttrDef → parseDeps → topoSort
3. Generate 5 generate*() functions, each looping for (const [kind, entry] of nodes) 1 generate() function, looping for (const a of ATTRS)
4. Write 5 files → ast-schema/generated/ 1 file → ksc-generated/evaluator.ts
Static data type Grammar: node kinds + field structure Attributes: names + types + equations + deps
Per-item output Interface + child field entry + converter + builder + serializer Cache field + method + switch case + import
Key insight: Both mix functions are template metaprograms. They don't transform code abstractly — they iterate over a finite registry and stamp out specialized code per item. The "specialization" is the loop: for (const item of registry) emit(template(item)). This is the simplest and most robust form of partial evaluation: unrolling a data-dependent loop into statically-specialized cases.

Self-Application: What Would mix(mix, mix) Concretely Mean?

The Futamura projections ask: what happens when you apply mix to itself? In KSC's concrete terms, this becomes surprisingly tangible.

Step 1: What does mix₂ "know" at generation time?

The generator has three kinds of hardcoded knowledge — its own "static data":

// BAKED INTO gen-ksc-evaluator.ts (the generator's own static data):

1. Evaluation semantics (how each direction works):
   syn  → "compute from self + children, cache, return"
   inh  → "if root: use rootValue, else: check parent eq, else: inherit"
   coll → "contribute(self) + combine over children"

2. Source analysis conventions (how to discover attributes):
   eq_X_contribute + eq_X_combine → collection attribute
   eq_X_root / eq_X_rootValue    → inherited attribute
   otherwise                     → synthesized attribute
   .attr('Y') in function body   → dependency edge X → Y

3. Code templates (what to emit per attribute):
   "private _c_${name}: ${type} | undefined"      → cache field
   "case '${name}': return this.${name}()"         → switch dispatch
   "if (this._c_${name} !== undefined) return ..." → memoization
   generateSynBody / generateInhBody / generateCollectionBody

Step 2: The Second Projection — mix₂ is already a "compiler"

gen-ksc-evaluator.ts already is the Second Projection (approximately). It's a compiler from attribute specs to evaluators:

// gen-ksc-evaluator.ts as a compiler — works for ANY set of eq_* functions:
gen-ksc-evaluator(binder.ts + checker.ts)  → evaluator.ts (22 attrs)
gen-ksc-evaluator(hypothetical-other.ts)   → evaluator.ts (N attrs)
gen-ksc-evaluator(just-binder.ts)          → evaluator.ts (3 attrs)

// The AG evaluation semantics are baked into the generator's templates.
// You supply different equation functions; it produces different evaluators.

Step 3: The Third Projection — Generating the Generator

To achieve mix(mix, mix), we'd describe mix₂'s own structure as data that a meta-generator could consume:

metaGen(queryCompilerSpec)      → gen-sql-compiler.ts     // SQL query compiler

Step 4: The Fourth Projection — The Fixpoint

Now the key question: what is the meta-generator's OWN structure?

// The meta-generator itself follows the SAME pattern:
function metaGen(spec) {
  // 1. READ  — parse the spec
  const { itemDiscovery, metadataInference, codeTemplates, globals } = spec;

  // 2. ANALYZE — understand what the spec describes
  const templateList = codeTemplates.map(t => parseTemplate(t));

  // 3. GENERATE — for each template in the spec, emit code that
  //    iterates over items and applies that template
  for (const template of templateList) {
    emit(`for (const item of metadata) {`);
    emit(`  ${expandTemplate(template)}`);
    emit(`}`);
  }

  // 4. WRITE — save the generated generator
}

// So we can describe metaGen ITSELF as a spec:
const metaGenSpec = {
  itemDiscovery:     { method: 'read spec.codeTemplates array' },
  metadataInference: { method: 'parseTemplate per template' },
  codeTemplates: [
    { target: 'read-phase',     pattern: 'parse spec object' },
    { target: 'analyze-phase',  pattern: 'map templates' },
    { target: 'generate-phase', pattern: 'for each template: emit iteration code' },
    { target: 'write-phase',    pattern: 'writeFile' },
  ],
};

// And then:
metaGen(metaGenSpec) = metaGen   ← FIXPOINT!

// The generator-of-generators, applied to its own spec,
// reproduces itself. No new artifact emerges.
Why is this a fixpoint? Because all generators in KSC have the same shape: read registry → analyze → iterate → emit. The meta-generator captures this shape. When you describe the meta-generator's own shape to itself, it re-instantiates that same shape. There's no deeper level of "generator structure" to extract — the pattern for (item of registry) emit(template(item)) is the atomic unit of code generation.
Practical parallel: This is why template engines are fixpoints. A Mustache renderer, described as a Mustache template, renders itself. Handlebars compiled through Handlebars produces Handlebars. KSC's generators are elaborate template engines — and template engines are fixpoints of the "apply template to data" operation. The 4th projection isn't exotic; it's the tautology that underlies every self-hosting system.

Stage 0: The Generic AG Evaluator

This stage is conceptual — it never exists as actual code in KSC. It represents the fully generic algorithm:

function ag_eval(grammar, attributes, program) {
  const tree = parseToAST(grammar, program);  // generic parse
  const dnodes = wrapInDecorators(tree);      // generic wrap
  for (const attr of attributes) {            // generic attr loop
    for (const node of traverseInOrder(dnodes, attr.deps)) {
      const eq = findEquation(attr, node.kind); // generic dispatch
      node.cache.set(attr.name, eq(node));      // generic cache
    }
  }
  return project(dnodes, attributes);
}

Everything is generic: the grammar is data, the attributes are data, the equations are looked up at runtime. This is how systems like Kiama or a naive AG library work — fully interpreted.

Performance cost of genericity: Hash lookups for every attribute access, dynamic dispatch for every equation call, runtime introspection for tree structure. Each of these adds overhead that specialization can eliminate.

Stage 1: Grammar Specialization — ast-schema/codegen.ts

Specializer input:

  • ast-schema/schema.ts — 364 node definitions via node(), leaf(), sumType()
  • ast-schema/builder.ts — registry of nodes + sum types + field additions

Specializer process: codegen.ts iterates over the registry and generates TypeScript source code. For each node kind, it:

  1. Emits a typed interface (KSBinaryExpression { left: KSExpression; operatorToken: KSToken; ... })
  2. Emits a child field entry ('BinaryExpression': ['left', 'operatorToken', 'right'])
  3. Emits a converter function registered by ts.SyntaxKind
  4. Emits a builder function (createBinaryExpression({left, operatorToken, right}))
  5. Emits serialization metadata

Residual programs (5 generated files):

ast-schema/generated/
  node-types.ts    — 364 interfaces + union types + type guards
  schema.ts        — getChildFields(), getChildren(), allKinds
  convert.ts       — TS → KS converters + buildKSTree()
  builders.ts      — createXxx() factory functions
  serialize.ts     — nodeToJSON / nodeFromJSON

What's specialized away: The generic "what fields does this node have?" question is now a static lookup. No runtime schema introspection.

Stage 2: Attribute Specialization — scripts/gen-ksc-evaluator.ts

Specializer input:

  • ksc-behavior/attr-types.ts — the KSCAttrMap type registry (22 attributes with types)
  • ksc-behavior/binder.ts — equation functions for binder attributes
  • ksc-behavior/checker.ts — equation functions for checker attributes

Specializer process: gen-ksc-evaluator.ts does five things:

  1. Parses the type map — extracts attribute names and their TypeScript types from KSCAttrMap
  2. Scans equation exports — finds all eq_* functions, their suffixes, their JSDoc annotations
  3. Infers attribute metadata — determines direction (syn/inh/coll), production cases, root functions from naming conventions
  4. Parses dependency graph — scans equation bodies for .attr('xxx') calls to build edges
  5. Generates evaluator — emits KSCDNode class with typed caches, switch dispatch, inlined evaluation patterns

Residual program:

ksc-generated/
  evaluator.ts  — KSCDNode class + buildKSCTree() + evaluate()
                   22 typed cache fields
                   22 direct methods with switch dispatch
                   Static imports of all equation functions
                   Pre-computed dependency graph
                   Pre-sorted evaluation order

Stage 3: Program Execution — Runtime

Dynamic input: The user's TypeScript source files (via ts.Program)

Execution flow:

// src/program.ts

  • Walking the specific user's AST
  • Evaluating specific equation results for specific nodes
  • Collecting the results

9. Formal Mapping: KSC and the Futamura Framework

KSC as a Two-Level First Futamura Projection

In the classical Futamura framework, the First Projection compiles by specializing an interpreter on a source program. KSC applies this principle twice, at two different levels of the input space:

Classical First Projection
target = mix(interp, source)
KSC's Two-Level Analog
grammar_eval = mix1(ag_eval, grammar)      [codegen.ts]

compiled_eval = mix2(grammar_eval, attributes)      [gen-ksc-evaluator.ts]

Each mix application is a First Projection at its level. The first says "compile the grammar into the evaluator infrastructure." The second says "compile the attributes into the evaluation logic."

The Analogy Table

What About the 2nd and 3rd Projections?

▶ Could KSC achieve the Second Projection?

The Second Projection would produce a compiler: a tool that takes any attribute specification and automatically generates a specialized evaluator. Formally:

Second Projection for KSC
attr_compiler = mix(mix2, grammar_eval)

This is essentially what gen-ksc-evaluator.ts already is — but only partially. Currently it reads equation source and generates an evaluator, but it's a hand-written specializer, not one derived from mix. A "true" Second Projection would mean:

  • The evaluator generator itself would be generated from a meta-description
  • Changing the grammar would automatically update the generator
  • The generator would be provably correct by construction

In practice, KSC's generator is close but relies on conventions (naming, .attr() call scanning) rather than formal specialization.

▶ Could KSC achieve the Third Projection?

The Third Projection would produce a compiler generator generator: a tool that takes any grammar and produces a tool that takes any attributes for that grammar and produces a specialized evaluator. Formally:

Third Projection for KSC
cogen = mix(mix, mix)

This would mean: give it the TypeScript grammar → get a tool that generates evaluators for any TypeScript analysis. Give it the Python grammar → get a tool that generates evaluators for any Python analysis.

This is the domain of systems like JastAdd (a compiler-compiler for reference attribute grammars) and Silver (an AG-based language workbench). KSC could evolve toward this if:

  • The schema format became a standard DSL (not TypeScript-specific)
  • The equation format became grammar-parametric
  • The generator became itself generated from the meta-level

10. Architectural Implications & Future Directions

The Stage Boundary Principle

Each stage boundary should be a clean interface where all information from the current stage is fully resolved before passing to the next. Crossing stage boundaries with "dynamic" data that could have been "static" is a staging error — it means the specializer is leaving performance on the table.

Current Boundaries and Their Quality

Boundary: Grammar → Attributes

Interface: Stage 2 reads the behavior source files and imports the generated types from Stage 1.

Quality: Good. The boundary is clean:

  • Stage 2's generator imports nothing from Stage 1's runtime — it only reads attr-types.ts and equation source
  • The equation functions reference generated types (KSIdentifier, KSVariableDeclaration) via normal TypeScript imports
  • The Ctx interface is the stable contract between stages

Improvement opportunity: The gen-ksc-evaluator.ts generator could import and use the schema registry from Stage 1 to validate that equation production cases match actual node kinds. Currently, a typo in a case name (e.g., eq_kindDefs_CompliationUnit) would produce unreachable generated code silently.

Boundary: Attributes → Runtime

Interface: evaluate(root: KSNode): EvaluationResult

Quality: Excellent. The boundary is minimal and clean:

  • A single function taking a KS tree root and returning structured results
  • No configuration parameters needed — everything is baked in
  • The KSCDNode class is an implementation detail hidden behind evaluate()

The buildTree() export exists for testing but doesn't break the boundary — it's a convenience for direct attribute inspection.

Potential Staging Errors in Current Architecture

1. AnalysisDepth as Runtime Configuration

AnalysisDepth = 'parse' | 'bind' | 'check' is a runtime parameter to buildKSTree, but it controls which TS checker fields get populated. This is information that could be a Stage 1 parameter, producing different generated converters for different depths. Currently, the depth check happens at runtime via if (_depth !== 'check') return '' guards.

Staging optimization: Generate three variants of convert.ts for each depth level, eliminating runtime depth checks.

2. String-Based Attribute Names at Runtime

The attr(name: string) method uses a switch-case on string names. While this is already specialized from a generic Map, a fully staged system would generate direct method calls at the equation level too — replacing ctx.attr('kindAnnotations') with ctx.kindAnnotations() in the equation source itself.

Staging optimization: Transform equation source to replace .attr('X') calls with direct .X() method calls in the generated evaluator.

3. Runtime Kind Switch in Syn Equations

For synthesized attributes with production cases, the generated code does switch (this.node.kind) at runtime. If the tree builder could tag each KSCDNode with a numeric kind index, the switch could be replaced with a jump table or even virtual dispatch.

Possible Future Stages

Stage 1.5   Stamped-Field Specialization

Currently, which stamped fields exist on which nodes (e.g., isExported on declaration nodes, symIsAlias on Identifiers) is determined by convertFieldOverrides in codegen.ts. This could be formalized as a sub-stage that takes a "field stamping specification" and generates optimized converters.

Stage 2.5   Per-Property Module Specialization

Currently, all 22 attributes are compiled into a single KSCDNode. A further specialization could generate separate modules per property (noImports, immutable, etc.) that could be tree-shaken or loaded on demand. The dep graph already knows which attributes are independent.

Stage 3.5   Incremental Re-evaluation

Given a previously decorated tree and a delta (changed nodes), a specialized incremental evaluator could re-evaluate only affected attributes. The static dep graph from Stage 2 already provides the information needed to generate this — it would be another specialization of the evaluator for a specific change pattern.

12. Flashcard Quiz

Click each question to reveal the answer.

What is a partial evaluator (specializer)?
A program transformer that takes a program P(s, d) and a known static input s, and produces a residual program P_s(d) that behaves identically to P but only needs the remaining dynamic input d. All computations depending only on s are pre-evaluated.
What does the First Futamura Projection produce?
A compiled target program. By specializing an interpreter on a specific source program (target = mix(interp, source)), the interpreter's dispatch and parsing logic is eliminated, leaving only the operations specific to that program. This is compilation by specialization.
What is "Binding-Time Analysis" (BTA)?
The process of classifying every sub-expression in a program as either static (computable from known inputs) or dynamic (requires unknown inputs). BTA determines what the specializer can pre-compute and what must remain in the residual program.
How many specialization stages does KSC have, and what are they?
Three stages:
Stage 1 (Grammar): codegen.ts specializes on the AST schema → generates typed interfaces, child field maps, converters
Stage 2 (Attributes): gen-ksc-evaluator.ts specializes on the 22 AG attributes → generates typed caches, switch dispatch, direct eq calls
Stage 3 (Runtime): evaluate(root) processes the specific user program using all the specialized code
What does codegen.ts specialize away (what generic operations does it eliminate)?
It eliminates: (1) runtime schema introspection ("what fields does this kind have?"), replaced by a static lookup table C; (2) generic node construction, replaced by 364 specific converter functions; (3) untyped field access, replaced by typed TypeScript interfaces; (4) runtime child enumeration, replaced by per-kind field name arrays.
What does gen-ksc-evaluator.ts specialize away?
It eliminates: (1) generic Map<string, any> attribute caching, replaced by 22 typed cache fields; (2) runtime attribute lookup, replaced by a switch statement; (3) generic direction-based evaluation, replaced by inlined syn/inh/coll patterns per attribute; (4) dynamic equation dispatch tables, replaced by direct function imports and calls; (5) runtime dep graph construction, replaced by a static constant.
Why can't KSC's equation functions use factory patterns like makeContextEquation()?
Because the Stage 2 specializer (gen-ksc-evaluator.ts) determines dependencies by scanning equation function bodies for .attr('xxx') calls using regex. Factory-produced functions hide the .attr() calls inside the factory, making them invisible to the scanner. This is a binding-time constraint: the dependency information must be statically visible in the equation source for the specializer to extract it.
How does KSC relate to the Second Futamura Projection?
gen-ksc-evaluator.ts is an approximation of the Second Projection: it takes an "interpreter" (the attribute equations) and produces a "compiler" (the specialized evaluator). However, it's hand-written rather than derived by specializing a specializer. A true Second Projection would generate the generator itself from a meta-description of AG evaluation.
What is a "staging error" in this context?
A staging error occurs when information that could be resolved at an earlier stage is instead left as runtime computation. For example, KSC's AnalysisDepth is a runtime parameter that controls which converter fields get populated, but it could be a Stage 1 parameter that produces different generated code for each depth level, eliminating runtime if (_depth !== 'check') guards entirely.
What is the relationship between Kiama and KSC in terms of partial evaluation?
Kiama operates at Stage 0: it's a fully generic AG library where grammar, attributes, and programs are all runtime data. KSC is the result of applying two stages of partial evaluation to the same abstract algorithm. KSC's generated code is the residual program you'd get if you could partially evaluate a Kiama-like library with respect to a specific grammar and specific attributes.
Name three concrete optimizations that result from KSC's multi-stage specialization.
(1) Typed cache fields instead of Map: this._c_kindDefs vs cache.get('kindDefs') — eliminates hash computation and type casting.
(2) Direct function imports instead of dispatch table: eq_kindDefs_CompilationUnit(this, ...) vs equations['kindDefs']['CompilationUnit'](this, ...) — enables JS engine inlining.
(3) Static child field arrays instead of runtime introspection: C['BinaryExpression'] returns a pre-built array vs computing it by reflecting on node structure.
What would a "Stage 2.5" specialization look like?
Per-property module specialization. Using the static dep graph from Stage 2, generate separate evaluator modules for independent property groups (e.g., noImports vs. immutable). Since these don't share dependencies, they could be loaded on demand or tree-shaken. The dep graph already contains the information to do this; it's "free" static knowledge that isn't yet exploited.
What is the shared structural pattern of KSC's two mix functions?
Both follow: Read → Analyze → Iterate → Emit. They load a static data source (grammar registry or behavior source), analyze it to extract metadata (field definitions or attribute definitions), iterate over the metadata emitting specialized code per item (for (const [kind, entry] of nodes) or for (const a of ATTRS)), and write the generated residual files. The "specialization" is the loop that unrolls generic data-driven behavior into static per-item code.
What concrete code constitutes mix₂, and what are its five phases?
mix₂ is scripts/gen-ksc-evaluator.ts (745 lines). Its five phases: (1) Parse type map from attr-types.ts via regex; (2) Scan exports from binder.ts/checker.ts for eq_* functions; (3) Infer direction (syn/inh/coll) from naming conventions; (4) Parse deps by regex-scanning function bodies for .attr('xxx') calls; (5) Generate KSCDNode class with typed caches, switch dispatch, and inlined evaluation patterns per attribute.
What would "applying mix to itself" mean concretely in KSC? Why is it a fixpoint?
You'd describe gen-ksc-evaluator.ts's own structure as a "meta-spec" — what items it discovers, what metadata it infers, what templates it emits. A meta-generator reading this spec would produce gen-ksc-evaluator.ts. Applying the meta-generator to its own spec reproduces itself, because all KSC generators share the same atomic pattern: for (item of registry) emit(template(item)). This pattern can't be further decomposed — it's the fixpoint. Template engines are fixpoints of "apply template to data."
