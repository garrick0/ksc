Language Primitives & the Properties They Create

Can we analyze TypeScript by analyzing its building blocks? And do other languages have the same blocks?

1 Yes — every language is built from primitives

A programming language is defined by a small, finite set of primitives — the atomic operations the language lets you perform. Every program is a composition of these primitives. If you know the full set of primitives, you know what the language can possibly do.

This isn't a metaphor. It's literally how programming languages are defined — via a formal semantics that gives meaning to each primitive construct, then rules for how they compose.

The key idea: Properties of programs emerge from which primitives you combine. If a language doesn't have a mutation primitive, no program in it can mutate state. If it doesn't have a network I/O primitive, no program can make HTTP calls. The primitives determine the envelope of all possible properties.

4 Languages group into "capability buckets"

Now the big question: do other languages have the same primitives? Yes — mostly. Languages differ in syntax and details, but the underlying capabilities cluster into well-known tiers. This is a real field of study — Felleisen (1991) formalized it as the expressiveness hierarchy.

Languages in the same bucket can simulate each other locally — you can translate any construct from one to the other without restructuring the whole program. Moving between buckets requires a global transformation (e.g., converting all mutation into a state monad, or CPS-transforming all async code).

5 But wait — aren't they all Turing-complete?

Yes! Almost every practical language is Turing-complete, meaning they can all compute the same functions. So why bother with buckets?

Turing-completeness is about what you can compute. It says nothing about how you compute it — which effects you use, what guarantees you get, what properties your programs have. A Haskell program and a C program can both sort a list, but only the C program might corrupt memory while doing it.

The primitive-bucket classification captures something Turing-completeness ignores: the operational character of computation. Two languages in different buckets produce programs with fundamentally different property profiles, even when computing the same function.

Analogy: All cars can drive from A to B (Turing-complete). But an electric car, a diesel truck, and a horse-drawn carriage have very different properties while doing so — emissions, speed, noise, fuel type. The "destination" is the same; the observable characteristics differ entirely. Language primitives are like the engine type.

6 What this means for analyzing TypeScript

Coming back to your original question:

Can we analyze TS by analyzing its primitives? Yes — and this is exactly what static analysis tools do. TypeScript has a finite set of primitive operations. Every TS program is a composition of these primitives. You can check which primitives a program uses (decidable, syntactic) and infer properties from that.

But there's a ceiling. You can determine which primitives are used (easy). You can determine how they're combined in the syntax tree (harder but still decidable). But determining what the combination ultimately does at runtime — that's where Rice's theorem kicks in and things become undecidable. The primitives tell you the ingredients; they don't tell you the recipe's outcome.

So the hierarchy from your property-space doc maps directly onto this:

What you're checking Difficulty Example
Which primitives are used Decidable (syntax scan) Uses fetch? Uses Math.random?
How they're structured Decidable (AST analysis) Is await inside a loop? Does mutation escape its scope?
What data flows where Approximable (abstract interpretation) Can user input reach eval?
What the program ultimately does Undecidable (Rice's theorem) Does it always terminate? Are these two programs equivalent?
