01What the Bind Stage Does

The binder transforms a user configuration into KindSymbol objects — the compiler's internal representation of architectural constraint targets.

The binder (src/binder.ts, ~90 lines) is deliberately lightweight. Unlike a traditional language binder that resolves names and builds scope chains, the KSC binder is a config-to-symbol converter. It reads the user's kindscript.config.ts and produces two collections of symbols for the checker to validate against actual code.

1Input: KindScriptConfig ▼

The user's configuration object mapping entry names to targets. Each entry is either a simple target (file or directory with rules) or a composite target (group of members with relational rules).

{
  domain: { path: './src/domain', rules: { pure: true, noIO: true } },
  infra:  { path: './src/infra',  rules: { noConsole: true } },
  app: {
    members: { domain: { path: './src/domain' }, infra: { path: './src/infra' } },
    rules: { noDependency: [['domain', 'infra']] }
  }
}
2Processing: ksBind() ▼

For each config entry, the binder:

  • Assigns a sequential ID (sym-0, sym-1, ...)
  • Detects valueKind via regex: paths with extensions = file, otherwise directory
  • Preserves declaredProperties verbatim from config rules
  • For composites: creates member symbols first, then parent with members Map
3Output: BinderResult ▼ 
interface BinderResult {
  symbols: KindSymbol[];  // ALL symbols (including composite members)
  targets: KindSymbol[];  // Top-level entries only (for checking)
}

interface KindSymbol {
  id: string;                      // "sym-0", "sym-1", ...
  name: string;                    // Config entry name
  declaredProperties: PropertySpec; // The rules
  members?: Map<string, KindSymbol>; // Composite only
  path?: string;                   // Filesystem target
  valueKind: 'file' | 'directory' | 'composite';
}

Key distinction: symbols has everything for export/introspection. targets has only top-level entries for the checker — composite members are excluded to avoid duplicate checking.

Key insight: This binder is NOT a traditional compiler binder. It doesn't resolve names, build scope chains, or link declarations to references. It's a config transformer that assigns IDs and preserves rules. All the interesting semantic work (file resolution, AST walking, violation detection) happens in the checker.

Import / Dependency Graph

The crown jewel. A force-directed or hierarchical graph showing import dependencies between symbol-governed regions. This is the most information-dense visualization and also the prerequisite for implementing noDependency, noTransitiveDependency, and noCycles checks.

┌──────────┐ ┌──────────┐ │ domain │───X───→│ infra │ ← noDependency violated! │ pure noIO │ │noConsole │ └────┬─────┘ └────┬─────┘ │ │ ▼ ▼ ┌──────────┐ ┌──────────┐ │ utils │ │ config │ ← uncovered (no symbol) │immutable │ │ │ └──────────┘ └──────────┘ Nodes: symbols | Edges: import dependencies | Red edges: rule violations Edge weight = number of file-level import edges between regions

Effort: Major compiler work. The checker must extract import specifiers from every file (it already does this in checkMaxFanOut but discards the data), resolve them to files, map files to symbols, and build an aggregated graph. Then export.ts serializes it and the viz renders a D3 force graph.

High strategic value: This isn't just a visualization — building the import graph also enables implementing the noDependency, noTransitiveDependency, and noCycles rules that are declared in PropertySpec but have no checker implementation yet.

09The Architectural Problem

ksBind() doesn't use TypeScript at all. The TS compiler's rich bind output sits right there, untouched. What should change?

What Actually Happens Today

Here's the exact call sequence in program.ts:

export function createProgram(rootNames, config, options) {
  // Step 1: TS does scan → parse → bind → type-check internally
  const tsProgram = ts.createProgram(rootNames, options ?? {});

  // Step 2: KSC binder runs — receives ONLY the config, not the TS program
  return createProgramFromTSProgram(tsProgram, config);
}

export function createProgramFromTSProgram(tsProgram, config) {
  const binderResult = ksBind(config);  // ← config only, no TS program!
  // ... wraps both into KSProgram
}

The critical observation: ksBind(config) takes a KindScriptConfig and nothing else. It has no access to the TS program, no access to source files, no access to the type checker. It's a pure data transformation: iterate config entries, assign IDs, detect file vs directory from path strings.

The disconnect: By the time ksBind() runs, TypeScript has already completed its full bind pass — symbol tables, scope chains, module resolution, flow analysis are all built and available on tsProgram. But ksBind() doesn't receive tsProgram, so it can't use any of it. The rich TS binding output sits right next to the KSC binder and is completely ignored.

Where TS Data Finally Gets Used (Barely)

The connection between KSC symbols and actual code happens much later, in the checker, through crude string matching:

// checker.ts:368 — resolveValueNodes()
// This is where KSC symbols finally meet TS source files
// It's just substring matching on file paths:

case 'directory': {
  const dirSuffix = sym.path.replace(/^\.\//, '');
  return tsProgram.getSourceFiles().filter(sf =>
    sf.fileName.includes('/' + dirSuffix + '/')  // ← string matching!
  );
}

And even the check functions that receive the ts.TypeChecker never call it:

// checker.ts:23 — the type signature accepts it
type IntrinsicCheckFn = (
  node: ts.Node,
  checker: ts.TypeChecker,    // ← accepted...
) => { ok: boolean; violations: PropertyViolation[] };

// But every check function is pure AST pattern matching:
function checkNoConsole(node: ts.Node): ... {  // ← doesn't even take checker param
  // walks AST looking for "console" identifiers by name
}

The Four Options

Option A: Pass tsProgram to ksBind()

Make the binder the place where TS's world meets KSC's world. The binder becomes responsible for file resolution, import graph extraction, and TS symbol mapping.

// binder.ts — enriched signature
export function ksBind(
  config: KindScriptConfig,
  tsProgram: ts.Program,        // ← NEW: access to TS binding output
): BinderResult {
  // Phase 1: what it does today (config → KindSymbols)
  // Phase 2: NEW — resolve symbols to actual files
  // Phase 3: NEW — extract import graph per file
  // Phase 4: NEW — build file→symbol reverse mapping
  // Phase 5: NEW — compute cross-symbol dependency edges
}
Pros • Single phase does all binding work — clean mental model • KindSymbols come out fully resolved (files, imports, dependencies attached) • Checker becomes simpler — just validates, doesn't resolve • All bind-stage visualization data produced in one place Cons • Binder grows from 90 lines to potentially 300+ • Mixes two concerns: config transformation + TS introspection • Breaks createProgramFromTSProgram ordering (binder already runs on line 44) • Actually wait — tsProgram IS available at that point. No ordering issue.

Option B: Add a Link / Resolve Phase

Keep ksBind() as the lightweight config transformer. Add a new explicit phase that takes binder output + TS program and produces the enriched data.

// NEW: linker.ts
export interface LinkResult {
  resolvedSymbols: ResolvedKindSymbol[];  // symbols with files attached
  importGraph: ImportGraph;              // file-level and symbol-level edges
  fileCoverage: FileCoverage;            // coverage, overlaps, gaps
}

export function ksLink(
  binderResult: BinderResult,
  tsProgram: ts.Program,
): LinkResult { ... }

// program.ts — updated pipeline
const binderResult = ksBind(config);          // Phase 1: config → symbols
const linkResult = ksLink(binderResult, tsProgram);  // Phase 2: symbols + TS → resolved
// Phase 3: checker uses linkResult
Pros • Clean separation: binder = config transform, linker = TS integration • Mirrors traditional compiler phases (bind → resolve → check) • ksBind() stays simple and testable without TS • Link phase can be incrementally built • Makes the TS/KSC boundary explicit in the architecture Cons • New file, new types, new phase in the pipeline • Data flows through more stages (binder → linker → checker) • KindSymbol might need to become ResolvedKindSymbol or carry optional fields

Option C: Minimal — Move resolveValueNodes to Binder

The smallest useful change. Just move the file resolution from the checker to the binder, so symbols know which files they govern immediately after binding.

// binder.ts — minimal enrichment
export function ksBind(
  config: KindScriptConfig,
  tsProgram: ts.Program,
): BinderResult {
  // ... existing config → symbol logic ...

  // NEW: resolve each symbol to actual files
  for (const sym of symbols) {
    if (sym.path) {
      sym.resolvedFiles = resolveFiles(sym, tsProgram);
    }
  }

  return { symbols, targets };
}
Pros • Smallest change (~30 lines moved from checker to binder) • Symbols carry resolved files — immediately useful for visualization • Checker's resolveValueNodes() becomes a simple lookup • Low risk, easy to review Cons • Doesn't address import graph, TS symbol mapping, or other gaps • Still no bridge between TS symbols and KSC symbols • "Minimal" might not be worth the churn if you plan to do more soon

Option D: Keep Binder As-Is, Enrich Checker Output

Don't change the binder. Instead, make the checker persist and export more of what it already computes internally (import specifiers, file mappings, violation details).

// checker.ts — expose internal data
export interface KSChecker {
  checkProgram(): KSDiagnostic[];
  checkSourceFile(sf: ts.SourceFile): KSDiagnostic[];

  // NEW: expose internal data for visualization
  getImportGraph(): ImportGraph;
  getFileCoverage(): FileCoverage;
  getResolvedSymbols(): ResolvedSymbol[];
}
Pros • Zero changes to binder or pipeline structure • Checker already walks ASTs and extracts imports — just persist them • Incrementally addable (one new getter at a time) • Export.ts can pull from checker instead of rebuilding Cons • Architecturally muddled: resolution is a bind-time concern, not check-time • Checker must run before data is available (lazy creation means data appears late) • Binder output remains impoverished — KindSymbol stays a flat config mirror • Doesn't fix the conceptual problem: bind phase should bind

Recommendation

Option B (Link Phase) is the cleanest path forward. It preserves ksBind() as the simple config transformer it is, adds a well-defined phase that bridges TS and KSC, and creates a natural home for all the data the visualization needs. The pipeline becomes: Config → Bind → Link → Check, where each phase has a clear single responsibility. Option A is also viable if you prefer fewer files, but mixes two distinct concerns in one function.

Here's what the pipeline would look like with Option B:

1ts.createProgram() — TS does scan/parse/bind/typecheck

Unchanged. Produces ts.Program with full TS binding output (symbol tables, scopes, types, module resolution).

2ksBind(config) — Config to KindSymbols

Unchanged. Pure config transformation. Produces KindSymbol[] with IDs, paths, declared rules, composite members.

3ksLink(binderResult, tsProgram) — NEW

The bridge phase. Takes KindSymbols + TS program and produces:

  • Resolved symbols — each KindSymbol with its actual source files attached
  • File coverage — which files are covered, uncovered, or overlapping
  • Import graph — file-level edges extracted from TS source files, mapped to symbol-level
  • TS symbol data — exports per file, types at import sites, scope info

This is where ts.TypeChecker APIs get used for the first time: getSymbolAtLocation(), getTypeAtLocation(), getExportsOfModule().

4createKSChecker(linkResult) — Validate rules

Simplified. No longer does file resolution (it's already done). Receives pre-resolved symbols and import graph. Checks intrinsic rules against ASTs, relational rules against import graph.

10TS Binding vs KSC Binding

Two completely different binders run during the bind phase. Understanding what each produces — and where they don't connect — is essential for knowing what's available to visualize.

When createProgram() runs, two bind stages execute in sequence. First, TypeScript's internal binder runs as part of ts.createProgram(), building a rich semantic model. Then, KSC's ksBind() runs, converting the user's config into KindSymbol objects. These two worlds currently have zero crossover.

TypeScript's Binder (runs inside ts.createProgram)

Runs automatically. KSC never calls it directly — it's a side effect of creating the TS program.

How They Connect (Barely)

The only bridge between TS binding output and KSC's world is path-string matching in the checker:

1KSC symbol says: path = "./src/domain" ▼

The KindSymbol carries a path string from the user's config. This is the only "address" KSC has for locating code.

{ id: 'sym-0', name: 'domain', path: './src/domain', valueKind: 'directory' }
2resolveValueNodes() matches against TS source files ▼

The checker iterates tsProgram.getSourceFiles() and does a substring match on filenames. This is pure string comparison — no TS symbol resolution involved.

// For directories: match "/src/domain/" in file paths
tsProgram.getSourceFiles().filter(sf =>
  sf.fileName.includes('/' + dirSuffix + '/')
);
3Check functions walk the matched AST nodes ▼

Once files are matched, checks walk the AST using ts.forEachChild() looking for syntactic patterns (import declarations, console accesses, assignments). They never query "what does this symbol resolve to?" or "what is the type of this expression?"

// checkNoConsole: pure syntax pattern matching
if (ts.isPropertyAccessExpression(n)) {
  if (ts.isIdentifier(n.expression) && n.expression.text === 'console') {
    violations.push({ ... });
  }
}

Bridge Data: What Needs to Be Built

To enable the combined view, we need explicit bridge data structures connecting TS and KSC binding output. Currently none of these exist.

// Bridge 1: TS import resolution → KSC symbol mapping
interface ResolvedImport {
  fromFile: string;           // source file path
  specifier: string;          // import string (TS knows this)
  resolvedFile?: string;      // where it resolves (TS module resolution)
  fromSymbol?: string;        // KSC symbol governing fromFile
  toSymbol?: string;          // KSC symbol governing resolvedFile
  crossesBoundary: boolean;   // fromSymbol !== toSymbol
  violatesRule?: string;      // if this crosses a noDependency boundary
}

// Bridge 2: TS symbol → KSC governance
interface GovernedSymbol {
  tsSymbolName: string;       // the identifier in code
  tsType: string;             // TS type as string
  declarationFile: string;    // where it's declared
  governingSymbol?: string;   // which KindSymbol owns that file
  relevantRules: string[];    // rules that apply to this usage
}

// Bridge 3: File → full governance context
interface FileGovernance {
  fileName: string;
  governingSymbols: string[];  // which KindSymbols claim this file
  effectiveRules: string[];   // union of all governing symbols' rules
  tsExports: string[];        // what this file exports (from TS)
  tsImports: ResolvedImport[]; // resolved imports (from TS)
}
Implementation note: Building these bridges requires using the ts.TypeChecker instance that already sits unused in checker.ts:481. The IntrinsicCheckFn signature already accepts it as a parameter — the hooks are in place, just not connected.

Where bridge data would be built:

compiler
checker.ts Use tsChecker.getSymbolAtLocation() to resolve import identifiers to their declaration files. Use tsChecker.getTypeAtLocation() for type-aware checks.
compiler
export.ts Build ResolvedImport[] and FileGovernance by combining TS module resolution with KSC symbol file mapping.
both
types.ts + dashboard Define bridge interfaces in types.ts, serialize in export.ts, render in dashboard with dual-layer coloring.

Q

What two collections does the binder produce, and what's the difference between them?

A

symbols (all symbols including composite members, for export/introspection) and targets (top-level config entries only, for checking). Composite members are in symbols but excluded from targets to avoid duplicate checking.

Q

What does the current bind visualization show, and what critical information does it not show?

A

It shows a symbol inventory: D3 tree of symbols grouped by kind, property chips, resolved files, and cross-stage diagnostics. It does NOT show: import/dependency graphs, file coverage gaps, symbol overlaps, AST-level rule annotations, scope information, or cross-symbol relationships.

Q

Which three visualization ideas require NO compiler changes and can be built with existing exported data?

A

Property Heatmap and Symbol Table Explorer need zero compiler changes — all data is in DashboardExportData.bind.symbols. Scope & Containment (basic version) can also use existing data. The File Coverage Map needs a minor export.ts addition.

Q

Why is the Import/Dependency Graph the highest-effort enhancement, and why does it have strategic value beyond visualization?

A

It requires extracting import specifiers from every file, resolving them to target files, mapping files to symbols, and building an aggregated graph — none of which exists in the export pipeline today. Its strategic value: the same import graph infrastructure is the prerequisite for implementing noDependency, noTransitiveDependency, and noCycles checks.

Q

How does the proposed Annotated AST View differ from the existing parse-stage AST view?

A

The parse AST shows raw syntax structure (node kinds, names, positions). The annotated bind AST overlays binding context: which nodes trigger violations (e.g., [noConsole ✗]), which imports count toward fan-out, which symbol's rules govern the file, and color-coding by rule compliance. It bridges two compiler stages in one view.

Q

What is ComputedPropertySpec and why is it relevant to the bind visualization roadmap?

A

Defined in types.ts with fields like pure: boolean, fanOut: number, and violations: PropertyViolation[]. It represents the computed state after checking (vs. the declared state from config). It's never fully populated or exported, but if it were, it would enable showing computed vs. declared property states — e.g., "symbol declares pure but computed purity is false."

Q

What data does the checker's checkMaxFanOut() compute internally but discard, and what visualization would capturing it enable?

A

It collects all import specifiers (both static imports and dynamic import()) into a Set<string> per file, counts them, and discards the set after comparison. Capturing these specifiers would enable the Import/Dependency Graph: mapping which files import from which others, aggregating to the symbol level, and detecting cross-symbol dependency violations.

Q

What does TypeScript's bind stage produce that KSC currently ignores entirely? Name at least four things.

A

TS binding produces: (1) symbol tables mapping identifiers to declarations, (2) scope chains with lexical nesting, (3) module resolution (import specifier → file), (4) flow analysis / control flow graphs, (5) type relationships and compatibility, (6) declaration-to-reference links. KSC accesses none of these — it only uses the AST structure and file list from the TS program.

Q

The ts.TypeChecker instance is acquired in checker.ts and passed to every check function. Why doesn't any check actually use it?

A

All current checks are syntactic pattern matchers: they walk AST nodes looking for structural patterns (import declarations, console identifiers, assignment operators). They don't need type information. The ts.TypeChecker parameter is infrastructure for future expansion — a hook point for semantic checks like type-aware purity analysis or deep IO detection that would need getTypeAtLocation().

Q

Give an example where the combined TS + KSC view reveals something neither layer can show alone.

A

Consider import { Database } from '../infra/db' in a domain file. TS alone knows this resolves to src/infra/db.ts but has no concept of "infra" as a constrained zone. KSC alone knows domain shouldn't depend on infra, but can only see the import specifier string, not prove it resolves to an infra file. Combined: TS resolves the import to a concrete file, KSC maps that file to a governing symbol, and the noDependency violation is proven with full traceability.
