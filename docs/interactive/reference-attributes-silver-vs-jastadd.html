Reference Attributes & Lookup

How do AG systems connect distant nodes — name resolution, scope lookup, cross-references? Comparing Silver's decorated references, JastAdd's composable coll+inh+ref pattern, and @ksc/ag's convenience lookup().

1The Problem: Connecting Distant Nodes

Why trees aren't enough — references create a graph overlay

Attribute grammars work on trees. But real programs have cross-references — a variable use at one leaf must find its declaration potentially many levels away. Name resolution creates a graph overlay on the tree: reference edges that skip the tree structure entirely.

Name Resolution

Variable x in function body must find the parameter x in the function signature. Tree distance: arbitrarily far.

Type Lookup

MyType in a type annotation must find type MyType = ... in the same module or an import.

Import Resolution

import { foo } from './bar' must resolve across compilation units, finding the exported binding.

The three AG systems solve this with fundamentally different designs:

System Approach Key Mechanism
JastAdd Composable attribute patterns coll + inh + reference attributes (any syn can return a node reference)
Silver Decorated references in the type system Decorated Expr with {env} — references carry their inherited context
@ksc/ag Convenience shortcut lookup(collect) — walk tree, build Map, return lookup function

2Current @ksc/ag: lookup()

A single convenience function that combines coll + inh into one call

Our lookup() is a shortcut: it walks the entire tree from the root, collects key-value pairs, builds a shared Map, and installs a lookup function on every node. It's implemented as a syn() wrapper — the map is lazily computed on first access from any node.

// libs/ag/src/lookup.ts — current implementation
export function lookup<N, K, V>(
  collect: (node: N) => Iterable<[K, V]>,
): AttributeDef<N, (key: K) => V | undefined> {
  let map: Map<K, V> | undefined;

  function getMap(startNode: N): Map<K, V> {
    if (map) return map;
    map = new Map();
    // Walk up to root via $parent
    let root: any = startNode;
    while (root.$parent) root = root.$parent;
    // DFS walk entire tree, collect entries
    const stack = [root];
    while (stack.length > 0) {
      const node = stack.pop()!;
      for (const [k, v] of collect(node)) map.set(k, v);
      // push children...
    }
    return map;
  }

  return syn((_node) => {
    const m = getMap(_node);
    return (key) => m.get(key);
  });
}

How it's used in the KSC binder

// src/pipeline/binder.ts
const defLookup = lookup<KSNode, string, KindDefinition>((node) => {
  if (node.kind !== 'CompilationUnit') return [];
  const defs: KindDefinition[] = (node as any).kindDefs;
  return defs.map(d => [d.name, d]);
});
What works well ▶
  • One-liner for simple cases — flat global namespace lookup in a single call
  • Shared map — computed once, all nodes share the same lookup function
  • Lazy — only computed if any node actually accesses the attribute
  • Composable with match/syn — the collect function can reference other attributes
What doesn't work ▶
  • No scoping — lookup is always global (whole tree). Can't model lexical scope, block scope, module scope.
  • No shadowing — last-write-wins for duplicate keys (DFS order). Can't model inner scope shadowing outer scope.
  • No context sensitivity — every node sees the same map. Can't model "visible declarations at this point in the file."
  • No reference attribute — returns a lookup function, not a reference to a node. Can't chain: use.decl.type.
  • No decorated references — even if you get a node reference, it doesn't carry inherited attribute context.
  • No contribution targeting — can't direct different contributions to different collection nodes (everything goes to the global map).

3JastAdd: Composable Attribute Patterns

No special "reference" feature — just the right combination of coll, inh, and syn

JastAdd doesn't have a dedicated "reference attribute" or "lookup" construct. Instead, name resolution emerges from composing three basic attribute kinds:

1. coll — Collect Definitions

Aggregate all definitions in a scope into a map. Contributors from anywhere in the subtree add entries.

2. inh — Propagate Environment

Pass the collected map down as an inherited attribute. Each scope level can extend/shadow the parent's map.

3. syn (reference) — Resolve

A synthesized attribute that returns a node reference by looking up in the inherited environment.

JastAdd's coll with contributes is a two-phase system: first survey which nodes contribute, then aggregate. Each declaration node contributes itself to the nearest scope's map.

// JastAdd .jrag — collect all declarations into a scope map
coll Map<String, Decl> Scope.localDefs()
  [new HashMap<>()] with putAll;

// Every VarDecl contributes to its enclosing scope
VarDecl contributes singletonMap(getName(), this)
  to Scope.localDefs()
  for each enclosingScope();

Key insight: contributes...to...for is a directed contribution — a declaration targets a specific collection attribute on a specific node. This is not a global bucket; it's targeted.

The collected local definitions are then propagated downward via inherited attributes, building a chain of scope environments.

// JastAdd .jrag — inherited environment = parent env + local defs
inh Map<String, Decl> ASTNode.env();

// Program root: env is its own collected defs
eq Program.getChild().env() = localDefs();

// Block scope: shadow parent env with local defs
eq Block.getChild().env() {
  Map<String, Decl> merged = new HashMap<>(env());
  merged.putAll(localDefs());
  return merged;
}

Each scope level extends the parent environment and shadows names that are re-declared. Children see the combined environment. This is lexical scoping as attribute propagation.

Resolution is just a synthesized attribute that looks up in the inherited environment:

// JastAdd .jrag — reference attribute
syn Decl VarUse.decl() = env().get(getName());

// Through the reference, access the target's attributes
syn Type VarUse.type() = decl().declaredType();

The result is a node reference — a direct pointer to the declaration node. You can chain through it to access the target's own attributes: use.decl.type. This creates a reference edge in the AG graph — a non-tree connection between nodes.

JastAdd references are just node values. You can chain through them freely:

// Follow reference chains
VarUse use = ...;
Decl d = use.decl();           // reference attribute
Type t = d.declaredType();     // syn on the target
Scope s = d.enclosingScope(); // navigate target's context

// Even circular: a type can reference itself
syn ClassDecl ClassDecl.superClass() =
  (ClassDecl) env().get(getSuperName());
syn Set<String> ClassDecl.allMethods()
  circular [new HashSet<>()] =
    localMethods().union(superClass().allMethods());

The key power: reference attributes compose with ALL other attribute kinds. Including circular — you can have circular dependencies that follow reference edges.

Why this design is powerful

JastAdd has no "special" reference system. It has three orthogonal primitives (coll, inh, syn) that compose to express arbitrarily complex name resolution. The pattern generalizes to:

  • Nested scopes — each Block extends its parent's env
  • Module imports — a Module's env merges its own defs with imported defs from other modules
  • Class hierarchies — env includes inherited members from superclasses (via reference to superclass node)
  • Overloading — env maps names to sets of declarations, resolve picks the right one based on context

4Silver: Decorated References in the Type System

References that carry their inherited context — type-safe cross-references

Silver's innovation is decorated references — when you hold a reference to a node, the type system tracks which inherited attributes it carries. This makes cross-references type-safe and enables forwarding.

In Silver, every nonterminal has two forms: undecorated (bare tree structure) and decorated (tree + evaluated inherited attributes). The type system distinguishes them:

-- Silver: undecorated = just structure
nonterminal Expr;
synthesized attribute type :: Type;
inherited attribute env :: Env;

-- An undecorated Expr is just { production, children }
-- A Decorated Expr has env provided, type computable

production varRef
e::Expr ::= name::String
{
  -- e.env is available because Expr has inh env
  e.type = lookupType(name, e.env);
}

The key insight: a Decorated Expr with {env} is a node that has its env inherited attribute already supplied. You can safely access its synthesized attributes.

-- Silver: reference type carries context info
synthesized attribute resolved :: Decorated Decl with {env};

production varRef
e::Expr ::= name::String
{
  -- Returns a decorated reference
  e.resolved = lookupInEnv(name, e.env);

  -- Through the decorated reference, access attributes
  e.type = e.resolved.declType;
}

-- Type error if you try to access .declType on
-- an undecorated reference that lacks {env}!

Silver's type system statically prevents accessing attributes on references that haven't been properly decorated. JastAdd does this at runtime (null pointer if attribute not yet computed).

Silver's name resolution follows the same pattern as JastAdd but with explicit type safety:

-- Silver: environment as inherited attribute
inherited attribute env :: [Pair<String, Decorated Decl>];

-- Collection attribute for gathering declarations
synthesized attribute defs :: [Pair<String, Decorated Decl>];

-- Propagate env: Silver has autocopy for inherited attrs
propagate env on Expr, Stmt;
-- (equivalent to: every child inherits parent's env unchanged)

-- Block extends env with local defs
production block
s::Stmt ::= decls::Decls body::Stmt
{
  body.env = decls.defs ++ s.env;
}

The propagate keyword is Silver's autocopy — it auto-generates the equation "child.env = parent.env" for every production of the listed nonterminals. This is what @ksc/ag's inh(rootValue) with no eq function does.

Silver's unique feature: forwarding. A production can delegate to another production, enabling modular language extensions.

-- Silver: forwarding = delegation to synthesized tree
production forLoop
s::Stmt ::= init::Expr cond::Expr step::Expr body::Stmt
{
  -- Desugar: for(init;cond;step) body => { init; while(cond) { body; step; } }
  forwards to block(
    seq(init, whileLoop(cond, seq(body, step)))
  );

  -- Any attribute not explicitly defined here is
  -- automatically delegated to the forwarded tree.
  -- Including env propagation, type checking, etc.
}

Forwarding means a new production only needs to define how to desugar itself. All other attributes are automatically inherited from the target production. This is modular: a language extension can add new syntax without modifying existing attribute equations.

This has no equivalent in JastAdd or @ksc/ag. It's Silver's key contribution to the AG field.

Three approaches to the same tree

Global flat map. One DFS walk, one Map, all nodes see the same thing.

// @ksc/ag: everyone shares the same global map
lookup((node) => {
  if (node.kind === 'VarDecl') return [[node.name, node]];
  if (node.kind === 'FuncDecl') return [[node.name, node]];
  return [];
})

// Result: Map { "x" => VarDecl, "foo" => FuncDecl }
// Every node sees this same map
// No scoping. "y" param not visible unless collected too.

Problem: VarUse(foo) inside the function body can see "x" from the outer block — but also, VarUse(x) can see "foo" even if it shouldn't. No scope boundaries.

Layered environments. Each scope builds on its parent's, with shadowing.

// JastAdd: each scope has its own env = parent.env + localDefs
Program.env = { "foo": FuncDecl }
  Block.env = Program.env + { "x": VarDecl }
    VarUse(x).env = Block.env   // sees {foo, x}
  FuncDecl(foo).env = Program.env
    Params.env = FuncDecl.env + { "y": ParamDecl }
      VarUse(y).env = Params.env // sees {foo, y} (not x!)
      VarUse(foo).env = Params.env // sees {foo, y}

Correct: VarUse(y) sees {foo, y} but not x. VarUse(x) sees {foo, x} but not y. Each scope inherits from parent and adds its own defs.

Same as JastAdd + type safety. References carry decoration context.

-- Silver: same scoping, but references are typed
production varUse
e::Expr ::= name::String
{
  -- Type: Decorated Decl with {env}
  -- This means "a Decl that has env available"
  e.resolved = lookup(name, e.env);

  -- Safe: resolved has env, so .declType is computable
  e.type = e.resolved.declType;
}

-- Compile error if resolved doesn't carry {env}:
-- "Cannot access .declType on Undecorated Decl"

Silver adds static type checking on top of JastAdd's approach. The scoping logic is identical; what's different is the type system prevents you from accessing attributes through undecorated references.

7Verdict: Which is Better for @ksc/ag?

Both are better than what we have. But they solve different problems.

The Core Tradeoff

JastAdd's approach is more practical for a library because it requires no type system changes — it's just a design pattern using existing primitives (coll + inh + syn).

Silver's approach is more principled but requires Decorated types in the type system, which would need branded types or phantom types in TypeScript — significant complexity for uncertain benefit.

For @ksc/ag: adopt the JastAdd pattern (scoped environments via coll+inh), don't try to replicate Silver's decorated references. Consider Silver's propagate (autocopy) as a DX improvement for inh().

Capability Current @ksc/ag + JastAdd pattern + Silver pattern
Flat global lookup Yes — lookup() Yes Yes
Lexical scoping No Yes — inh env chain Yes
Shadowing No Yes — map merge Yes
Reference attributes No (returns function) Yes — syn returns node ref Yes — typed
Reference chaining No Yes — use.decl.type Yes — type-safe
Type-safe references No No (runtime) Yes (compile-time)
Forwarding No No Yes
Implementation effort Done Low — already have coll+inh+syn High — needs branded types, wrappers

8Uplift Options for @ksc/ag

Three concrete paths forward, from minimal to ambitious

Option A: JastAdd Pattern — Scoped Env via coll+inh

Don't add new primitives. Instead, provide a pattern and a helper for building scoped environments using the existing coll(), inh(), and syn() primitives. Keep lookup() for the flat global case.

  • Zero new API surface — it's a usage pattern, not a new primitive
  • Already have all the building blocks (coll, inh, syn all work)
  • Proven approach — this is exactly how ExtendJ (full Java compiler) handles scoping
  • Reference attributes work automatically (syn that returns a node is already a reference)
  • Composable with circular for class hierarchies
  • More boilerplate than lookup() for the simple flat case
  • User must manually wire coll -> inh -> syn for each scope layer
  • No type safety on references (same as JastAdd)

Concrete API

// Option A: use existing primitives in the JastAdd pattern
import { coll, inh, syn, match } from '@ksc/ag';

// 1. Collect local definitions per scope
const localDefs = match('kind', {
  Block: (node) => collectDeclsFromBlock(node),
  FuncDecl: (node) => collectParamsFromFunc(node),
  _: () => new Map(),
});

// 2. Propagate env downward, extending at scope boundaries
const env = inh<Node, Map<string, Node>>(
  (root) => (root as any).localDefs,  // root value
  (parent, child, idx) => {
    if (isScopeNode(parent)) {
      // Merge parent env + local defs (shadowing)
      return new Map([...(parent as any).env, ...(parent as any).localDefs]);
    }
    return undefined;  // auto-propagate
  }
);

// 3. Reference attribute: resolve a use to its declaration node
const decl = syn((node) => {
  if (node.kind !== 'VarUse') return undefined;
  return (node as any).env.get(node.name);  // returns a node reference!
});

// 4. Chain through the reference
const resolvedType = syn((node) => {
  const d = (node as any).decl;
  return d ? d.declaredType : undefined;
});

Option B: scopedLookup() Helper — Convenience on Top of Option A

Add a scopedLookup() convenience function that encapsulates the coll+inh+syn pattern into a single call. This is like what lookup() is today, but scope-aware.

  • One-liner for the common scoped-lookup case
  • Encapsulates the JastAdd pattern without users needing to understand it
  • Backwards compatible — keep lookup() for flat case, add scopedLookup() for scoped
  • New API surface — another function to learn and maintain
  • Harder to customize (what if your scoping rules don't fit the default pattern?)
  • Hides the composable nature of the underlying primitives

Possible API

// Option B: scopedLookup() convenience
import { scopedLookup } from '@ksc/ag';

const { env, resolve } = scopedLookup<Node, string, Node>({
  // Which nodes introduce a new scope?
  isScope: (node) => ['Block', 'FuncDecl', 'Program'].includes(node.kind),
  // What does each node contribute to its scope?
  contribute: (node) => {
    if (node.kind === 'VarDecl') return [[node.name, node]];
    return [];
  },
});

// Returns two AttributeDefs:
//   env: inh attribute (Map per scope level with shadowing)
//   resolve: syn attribute ((name) => Node | undefined)

applyAttributes(root, { env, resolve, /* ... */ });

Option C: Silver-style Decorated References (Branded Types)

Implement Decorated<N, Attrs> branded types that track which inherited attributes a reference carries. Use TypeScript's structural type system and branded intersections.

  • Type-safe references — compile-time errors for accessing attributes on undecorated nodes
  • Closer to Silver's semantics
  • Better developer experience for complex compilers with many attribute dependencies
  • Significant implementation complexity (branded types, wrapper functions, type inference)
  • TypeScript's type system may not express this cleanly (no HKTs)
  • Overhead: wrapping/unwrapping decorated references at runtime
  • No ecosystem precedent — untested in TypeScript AG context
  • Forwarding (Silver's main benefit) can't be replicated without codegen

Sketch

// Option C: branded decorated references (sketch)
type Decorated<N, Attrs extends string> = N & { __decorated: Attrs };

// A reference attribute that returns a decorated node
const decl = refAttr<VarUse, Decl, 'env' | 'type'>(
  (node) => (node as any).env.get(node.name)
);

// TypeScript enforces: can only access .type if 'type' is in Attrs
// Problem: this is fragile and doesn't compose well with existing code

9Detailed API Designs

What each option looks like in real @ksc/ag code

Real-world example: KSC binder with scoped lookup

Currently the KSC binder uses flat lookup(). Here's how it would look with each option:

// Current: flat global lookup (binder.ts)
const defLookup = lookup<KSNode, string, KindDefinition>((node) => {
  if (node.kind !== 'CompilationUnit') return [];
  const defs = (node as any).kindDefs;
  return defs.map(d => [d.name, d]);
});

return {
  attributes: { kindDefs, defLookup },
};

For KSC's current flat Kind-definition namespace, this works perfectly. Scoped lookup would matter if KSC adds module scope or nested kind definitions.

// Option A: JastAdd pattern with existing primitives
// (only needed if we add scoped kind definitions)

const localKindDefs = match('kind', {
  CompilationUnit: (cu) => {
    const defs = extractKindDefs(cu);
    return new Map(defs.map(d => [d.name, d]));
  },
  Module: (mod) => {  // hypothetical future scope node
    const defs = extractKindDefs(mod);
    return new Map(defs.map(d => [d.name, d]));
  },
  _: () => new Map(),
});

const kindEnv = inh<KSNode, Map<string, KindDefinition>>(
  (root) => (root as any).localKindDefs,
  (parent, child, idx) => {
    if (parent.kind === 'Module') {
      return new Map([
        ...(parent as any).kindEnv,
        ...(parent as any).localKindDefs,
      ]);
    }
    return undefined;  // auto-propagate
  }
);

const defLookup = syn((node) => {
  const env = (node as any).kindEnv;
  return (name: string) => env.get(name);
});

return {
  attributes: { localKindDefs, kindEnv, defLookup },
};
// Option B: scopedLookup() convenience
const { kindEnv, defLookup } = scopedLookup<KSNode, string, KindDefinition>({
  isScope: (n) => ['CompilationUnit', 'Module'].includes(n.kind),
  contribute: (node) => {
    if (node.kind !== 'TypeAliasDeclaration') return [];
    const def = tryExtractKindDef(node);
    return def ? [[def.name, def]] : [];
  },
  envKey: 'kindEnv',
  resolveKey: 'defLookup',
});

return {
  attributes: { kindEnv, defLookup },
};

What lookup() should become

Regardless of which option we pick, lookup() should stay for the flat global case. It's clean and sufficient for KSC's current needs. The scoped approach is additive.

Should we deprecate lookup()? ▶

No. lookup() is the right tool for flat namespaces. Not everything needs scoping. KSC's kind definitions are a flat global namespace — lookup() is the correct choice there.

Think of it as: lookup() is for program-global lookups (like JastAdd's Program.allDefs()). The scoped approach is for lexically-scoped lookups (like JastAdd's env() chain).

Both patterns coexist in real compilers — ExtendJ uses both.
