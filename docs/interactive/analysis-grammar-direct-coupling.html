Analysis & Grammar

01The Expression Problem Dissolves

Philip Wadler's 1998 formulation defines the fundamental extensibility tension. Your constraint renders half of it moot.

The Expression Problem states that you have two orthogonal dimensions of extensibility: new data types (AST node variants) and new operations (analysis rules). Any approach makes one easy at the cost of the other. This is why systems like ESLint build string-keyed visitor dispatch — to allow new operations without modifying the data type definition.

But in KindScript, each analysis adapter is bound to exactly one grammar. The ts-kind-checking analysis is always for Grammar<TSNodeKind>. The mock analysis is always for Grammar<MockKind>. This constraint fixes the data type dimension per-plugin, which means:

Fixed Data Dimension
EXPRESSION PROBLEM
The grammar is a known, closed set of node kinds for each analysis. No need for open extension.
Free Operation Dimension
EXPRESSION PROBLEM
Multiple analyses can exist for the same grammar. New analyses add new operations without modifying nodes.
Abstraction Tax = Wasted
CONCLUSION
A generic AST interface adds indirection that only pays off when rules must be portable across grammars. Here they can't be.

When a rule set is scoped to one grammar, the union of node kinds is statically known. TypeScript can represent it as a discriminated union (TSNodeKind = 364 literal types), enabling exhaustiveness checking, narrowing, and refactoring safety. This is impossible through a generic { type: string } interface.

You might have ts-kind-checking, ts-complexity, and ts-security — all analyzing the same TS AST grammar. Each is a separate AnalysisDecl<TSNodeKind>. The grammar doesn't change per analysis; analyses compose independently. This is the Visitor pattern's strength, and it works perfectly with direct coupling.

Generic AST interfaces (like ESTree) pay a concrete cost: you lose type narrowing, discriminated union exhaustiveness, typed field access, and compile-time refactoring detection. These costs are justified only when rules must be portable across multiple grammars. Since each KindScript analysis adapter is explicitly for one grammar, the cost has no payoff.

Key insight: The generic abstraction layer between grammar and analysis is the wrong trade-off when each analysis is scoped to one grammar. Direct coupling is not just acceptable — it's the architecturally correct choice that maximizes what TypeScript's type system can verify.

02Current Architecture: The Status Quo

KindScript's analysis adapters already depend on their grammar adapter in practice. But the coupling is split across layers with inconsistent patterns.

How coupling currently works

The current architecture has a mixed coupling strategy. Some layers directly import concrete grammar types; others go through generic interfaces. The table below maps every coupling surface:

The paradox: The equation functions (the actual analysis logic) are already directly coupled to grammar types via KindCtx<KSIdentifier>. But the surrounding infrastructure — predicates, projections, domain types — uses generic interfaces, creating a mixed coupling pattern where some code has full type safety and some doesn't.

The K type parameter

Currently, the K type parameter serves as the only formal coupling between grammar and analysis at the port level. AnalysisDecl<TSNodeKind> says "this declaration is for a grammar whose kinds are TSNodeKind". But K only constrains the keys of equation records — it doesn't give the analysis access to node field types, sum type membership, or any structural information about the grammar.

// K gives you this:
equations: Partial<Record<K, Function>>
//         ^^^^^ you know the keys, but Function is untyped

// K does NOT give you:
// - What fields an Identifier has (escapedText, symIsVariable, ...)
// - Which kinds are members of Expression
// - The discriminated union for narrowing

04TypeScript Benefits of Direct Dependency

The concrete static analysis benefits that direct coupling unlocks — none of which are possible through a shared abstraction.

1 Discriminated Union Exhaustiveness ▼

When an equation function uses KindCtx<KSNode> where KSNode has kind: TSNodeKind (a union of 364 literal types), TypeScript's control flow analysis enables exhaustive switch statements. A default: assertNever(node) pattern will cause a compile error if a new node type is added to the grammar and the equation doesn't handle it.

This is already working in KindScript's generated dispatch:

// generated/dispatch.ts — exhaustive switch
function dispatch_kindDefs(ctx: Ctx): KindDefinition[] {
  const _kind = (ctx.node as KSNode).kind;
  switch (_kind) {
    case 'CompilationUnit': return eq_kindDefs_CompilationUnit(...);
    case 'Identifier':
    case 'BinaryExpression':
    // ... 361 more cases
    default: return eq_kindDefs_default(ctx);
  }
}

Through a generic { type: string } interface, this would be impossible — you'd get zero compile-time feedback when the grammar evolves.

2 Type Narrowing Without Casts ▼

Per-kind equation functions use KindCtx<KSIdentifier> to get type-safe access to node fields without any manual casting:

function eq_violationFor_Identifier(
  ctx: KindCtx<KSIdentifier>, property: string
): Diagnostic | null {
  // ctx.node is KSIdentifier — all fields typed:
  if (ctx.node.resolvesToImport) { ... }  // boolean
  ctx.node.escapedText                   // string
  ctx.node.importModuleSpecifier           // string
}

Without direct coupling, the equation function would receive Ctx where ctx.node is ASTNode (the minimal contract: kind, pos, end, text, children). Accessing escapedText would require (ctx.node as any).escapedText — unsafe, unrefactorable, and invisible to the compiler.

3 Refactoring Safety ▼

When the grammar adapter renames a field (e.g., KSIdentifier.escapedText → KSIdentifier.name), TypeScript immediately surfaces every affected equation as a compile error. The developer sees exactly which rules need updating.

Without direct coupling, the rename propagates silently. Every (ctx.node as any).escapedText continues to compile but returns undefined at runtime. The test suite might catch it — or it might not, if the test fixtures don't exercise that path.

Current state: The equation functions in equations/attributes.ts and equations/definitions.ts already have this safety. But equations/predicates.ts uses string constants like 'EqualsToken' and 'CallExpression' — these would not be caught by a rename.

4 Sum Type Membership Verification ▼

The grammar adapter exports sumTypeMembers and sumTypeMembership tables. With direct coupling, analysis code can use the grammar's type guards:

import { isSumTypeMember } from 'grammar/grammar/ts-ast/index.js';

// Type-safe check: is this node an Expression?
if (isSumTypeMember(ctx.node, 'Expression')) {
  // ctx.node.typeString is available (propagated from Expression sum type)
}

Currently, predicates like SIDE_EFFECT_EXPR_KINDS are hand-maintained string sets. With direct coupling, they could be derived from the grammar's sum type membership tables, ensuring they stay in sync when the grammar changes.

The compound effect: Each of these benefits reinforces the others. Exhaustiveness catches new node kinds; narrowing makes rule code readable; refactoring safety ensures changes propagate; sum type membership verification prevents stale predicates. Together they form a closed feedback loop that a generic interface cannot provide.

05ESLint Case Study: What Abstraction Costs

ESLint chose generic AST interfaces for multi-language support. The architectural cost is well-documented by its own maintainers.

String-keyed visitor dispatch

ESLint rules use a string-keyed visitor pattern: { "Identifier": (node) => ... }. The rule never imports node types — it matches on string names from the ESTree spec. This enables language-agnosticism: the same rule infrastructure works for JavaScript, TypeScript (via conversion), and potentially other languages.

// ESLint rule — string-keyed, no type imports
module.exports = {
  create(context) {
    return {
      "Identifier"(node) {
        // node is weakly typed — field access is unchecked
        if (node.name === "eval") { ... }
      }
    };
  }
};

The node parameter has a loose type. ESLint does not know what fields Identifier has at the type level.

The dual-tree tax (typescript-eslint)

To support TypeScript-specific syntax, typescript-eslint had to build an entire conversion layer from TypeScript's AST to ESTree. Josh Goldberg (core maintainer) describes the cost candidly:

From the maintainer: "The main downside of this dual-tree format is the complication for linter teams and lint rule authors working with TypeScript APIs. We must dedicate time for every TypeScript AST change to update node conversion logic."

And critically, Goldberg notes the original justification has evaporated:

From the maintainer: "TypeScript is now the only popular type-oriented language in web code — there's no longer a strong need to keep lint rules extensible for other languages such as Flow."

The abstraction layer was the right call when it was made (for language generality). But when your rules are scoped to one grammar, the entire justification collapses — and you're left paying the cost with no payoff.

What this means for KindScript

KindScript is in a stronger position than typescript-eslint. Because each analysis adapter declares its grammar binding up front (AnalysisDecl<TSNodeKind>), there's no pretense of cross-grammar portability. The architecture already acknowledges the coupling — it just doesn't fully exploit the benefits of that coupling.

The lesson: pay the abstraction tax only when you need the abstraction. ESLint needed it (originally). KindScript doesn't. Making the coupling explicit and typed is not adding coupling — it's acknowledging coupling that already exists and extracting maximum value from it.

06Design Decisions & Options

Six concrete decision points for making the analysis-grammar coupling explicit. Each has multiple options with a recommended approach.

1 Domain Types: node: unknown vs Typed Node References ▼

Current: KindDefinition.node and Diagnostic.node are typed unknown, erasing grammar information at the domain boundary.

OPTION A: Keep unknown
Status quo — domain types remain grammar-agnostic
Consumers can't access node fields without casting. Forces runtime type assertions. The domain types pretend they don't know what grammar they came from.
OPTION B: Make domain types generic — KindDefinition<N>
Parameterize domain types by their grammar's node type
KindDefinition<KSNode> with node: KSNode. Projections become KSCProjections<KSNode>. Downstream consumers get typed node access without casting. The generic parameter is always supplied at the composition root, so usage sites don't need to specify it.
OPTION C: Use the specific node type directly
node: KSTypeAliasDeclaration on KindDefinition
Maximum specificity, but locks domain types to one grammar. If you later have ts-complexity analysis with its own ComplexityDefinition, it can't reuse the same type shape. Option B is more flexible without losing type safety.
2 Predicate Constants: String Sets vs Grammar-Derived ▼

Current: ASSIGNMENT_OPS, IO_MODULES, SIDE_EFFECT_EXPR_KINDS are hand-maintained Set<string> values. They reference grammar kind names as raw strings.

OPTION A: Keep raw string sets
Status quo — easy to write, invisible to refactoring
If the grammar renames 'EqualsToken' or 'CallExpression', these sets silently become stale. No compile error.
OPTION B: Type the sets as Set<TSNodeKind>
Use the grammar's kind union to validate set contents at compile time
import type { TSNodeKind } from 'grammar/grammar/ts-ast/nodes.js';

export const SIDE_EFFECT_EXPR_KINDS: ReadonlySet<TSNodeKind> =
  new Set(['CallExpression', 'AwaitExpression', 'YieldExpression']);
// Compile error if any string is not a valid TSNodeKind
Minimal change, maximum safety. Every string in the set is validated against the grammar's kind union at compile time.
OPTION C: Derive from grammar sum type tables
Compute at runtime from sumTypeMembers
E.g., SIDE_EFFECT_EXPR_KINDS = new Set(grammar.sumTypeMembers['SideEffectExpression']). Maximum DRY, but requires defining sum types in the grammar for every predicate set. Over-engineers for simple cases.
3 Projections: Generic Ctx vs Typed KindCtx ▼

Current: Projection functions use (root: Ctx) and string-based attr() calls: root.attr('kindDefs') returns any.

OPTION A: Keep generic Ctx
Projections remain grammar-agnostic
Returns any from attr() calls. No type checking on projection logic. But projections are trivial (2 lines each) and rarely change, so the risk is low.
OPTION B: Use TypedAGNode<KSCAttrMap>
Type-safe attr() calls with known return types
root.attr('kindDefs') returns KindDefinition[] instead of any. But projections are not equation functions — they receive Ctx, not TypedAGNode. This would require changing the AnalysisProjections port interface in core-evaluator.
OPTION C: Use KindCtx<KSProgram> for root projection
Narrow the root context to the grammar's root node type
The root node is always KSProgram (or MockProgram). The projection function can use KindCtx<KSProgram> to get typed field access on the root. This is a narrow, targeted change that doesn't require modifying the port interface — just add the import and cast at the adapter level.
4 Equation Object Typing: Function vs Typed Signatures ▼

Current: AttrDecl<K>.equations is typed Partial<Record<K, Function>>. The Function type erases the equation's signature — any function is accepted regardless of parameter or return types.

OPTION A: Keep Function
Status quo — any callable accepted
You could pass (x: number) => string as a kindDefs equation and TypeScript wouldn't complain. The error only surfaces at runtime (wrong return type) or in generated code (wrong parameter type).
OPTION B: Typed equation signature on AttrDecl
Generic equation type: (ctx: Ctx, ...args) => T
interface SynAttr<K, T = unknown> {
  direction: 'syn';
  name: string;
  type: string;
  default?: (ctx: Ctx, ...args: unknown[]) => T;
  equations?: Partial<Record<K,
    (ctx: Ctx, ...args: unknown[]) => T>>;
}
This ensures equations at least conform to the (ctx, ...args) => T shape. The T parameter links default and per-kind equations to the same return type.
OPTION C: Full per-kind typed equations with KindToNode
Each equation key constrains the KindCtx parameter type
Maximum safety, but requires a grammar-specific KindToNode mapping at the port level. This would make AttrDecl dependent on the grammar adapter's type map — exactly the kind of explicit coupling this analysis argues for, but it's a more invasive change to the port interface.
5 Helper Functions: Generic Ctx vs Concrete Node Params ▼

Current: getKindCtx(ctx: Ctx, property: string) and diag(ctx: Ctx, ...) in predicates.ts use generic Ctx. They don't know what grammar they're operating on.

OPTION A: Keep generic
Helpers remain grammar-agnostic
These helpers don't actually access node fields — they only call ctx.attr() and read ctx.node.pos/end (which are on ASTNode). For helpers that genuinely only need the Ctx contract, staying generic is correct.
OPTION B: Keep generic where correct, type where beneficial
Audit each helper individually
getKindCtx and diag genuinely only need Ctx — leave them generic. But if a helper accesses node-specific fields (e.g., a future isModuleImport helper that checks ctx.node.resolvedModulePath), it should take KindCtx<KSImportDeclaration>. The rule: use the narrowest correct type for each function.
6 Composition Root: Implicit vs Explicit Grammar Dependency ▼

Current: The analysis adapter's index.ts barrel does not re-export or reference the grammar it depends on. The coupling is implicit — you have to read spec.ts imports to discover it.

OPTION A: Keep implicit
Grammar dependency remains hidden in internal imports
Consumers of the analysis adapter don't know which grammar it targets without reading source. The composition root (codegen-targets.ts) pairs them manually.
OPTION B: Export a typed grammar reference from the adapter barrel
Make the grammar dependency part of the adapter's public API
// src/adapters/analysis/spec/ts-kind-checking/index.ts
export { grammar } from '../../../grammar/grammar/ts-ast/index.js';
export type { TSNodeKind, KSNode, KindToNode }
  from '../../../grammar/grammar/ts-ast/index.js';
Makes the dependency explicit and discoverable. The composition root can import both grammar and analysis from the same barrel. Consumers see immediately which grammar this analysis targets.
OPTION C: Bundle grammar + analysis into a combined adapter object
Single object exports grammar, decl, and projections together
Creates a TSKindCheckingAdapter that bundles everything. Redundant with CodegenTarget and EvaluationTarget which already serve this role at the application layer.

07Proposed Architecture

Applying the recommended option from each design decision, here's what the refactored analysis adapter looks like.

Summary of changes

Decision Before After
Domain types node: unknown KindDefinition<N> with node: N
Predicate constants Set<string> ReadonlySet<TSNodeKind>
Projections root (root: Ctx) (root: KindCtx<KSProgram>)
Equation type Function (ctx: Ctx, ...args) => T
Helpers All generic Ctx Generic where correct, typed where beneficial
Adapter barrel Grammar not referenced Re-exports grammar + grammar types

Key principle: explicit coupling at every layer

The guiding principle is: if the analysis adapter is bound to one grammar, every layer should know and exploit that fact. The current architecture has a gradient — equations are tightly coupled, but the surrounding infrastructure pretends to be generic. The proposed architecture makes every layer consistently aware of its grammar.

// BEFORE: types.ts
export interface KindDefinition {
  node: unknown;  // who is this for? nobody knows
  ...
}

// AFTER: types.ts
import type { ASTNode } from '@kindscript/core-grammar';

export interface KindDefinition<N extends ASTNode = ASTNode> {
  node: N;  // defaulted for backward compat, but narrowed at composition
  ...
}
// BEFORE: predicates.ts
export const SIDE_EFFECT_EXPR_KINDS = new Set([
  'CallExpression', 'AwaitExpression', 'YieldExpression',
]);
// No compile-time validation of kind strings

// AFTER: predicates.ts
import type { TSNodeKind } from 'grammar/grammar/ts-ast/nodes.js';

export const SIDE_EFFECT_EXPR_KINDS: ReadonlySet<TSNodeKind> = new Set([
  'CallExpression', 'AwaitExpression', 'YieldExpression',
]);
// 'Calllexpression' → compile error (typo caught!)
// BEFORE: index.ts (adapter barrel)
export type { KindDefinition, Diagnostic } from './types.js';
export { analysisProjections } from './projections.js';
// No grammar reference — consumers don't know which grammar this targets

// AFTER: index.ts (adapter barrel)
// Grammar dependency — explicit and discoverable
export { grammar } from '../../../grammar/grammar/ts-ast/index.js';
export type { TSNodeKind, KSNode, KindToNode }
  from '../../../grammar/grammar/ts-ast/index.js';

// Analysis exports
export type { KindDefinition, Diagnostic } from './types.js';
export { analysisProjections } from './projections.js';
export { dispatchConfig } from './generated/dispatch.js';
export { depGraph } from './generated/dep-graph.js';
export type { KSCAttrMap } from './generated/attr-types.js';
Key insight: None of these changes modify port interfaces in workspace packages. They're all adapter-level changes — making the concrete adapter more explicit about its grammar dependency. The generic AnalysisDecl<K>, AnalysisProjections<P>, and Ctx ports remain unchanged.

Q

Why does fixing the "data type dimension" of the Expression Problem change the coupling calculus?

A

When each analysis is bound to one grammar, the node type union is a closed, known set. The generic abstraction layer (which exists to enable cross-grammar portability) has no payoff. Direct coupling gives you exhaustiveness checking, type narrowing, and refactoring safety that a generic interface cannot provide.

Q

What does KindCtx<KSIdentifier> give an equation function that Ctx alone does not?

A

KindCtx<KSIdentifier> narrows ctx.node from generic ASTNode (only kind, pos, end, text, children) to KSIdentifier — giving typed access to fields like escapedText, resolvesToImport, importModuleSpecifier, and all symbol flags. No casts needed.

Q

Why is SIDE_EFFECT_EXPR_KINDS: Set<string> a refactoring hazard, and what's the fix?

A

If the grammar renames a kind (e.g., 'CallExpression' → 'CallExpr'), the string set silently becomes stale — no compile error. Fix: type it as ReadonlySet<TSNodeKind>. TypeScript validates every string in the set against the grammar's 364-member kind union at compile time.

Q

What problem did typescript-eslint's dual-tree architecture solve, and why doesn't that problem apply to KindScript?

A

It solved cross-language portability — ESLint rules needed to work on both JavaScript (ESTree) and TypeScript (TSESTree). KindScript's analysis adapters are explicitly scoped to one grammar (AnalysisDecl<TSNodeKind>), so there's no cross-grammar portability requirement to justify the abstraction layer.

Q

In the current architecture, which modules of the analysis adapter are directly coupled to grammar types and which are fully generic?

A

Directly coupled: equations/attributes.ts (12 concrete types via KindCtx<N>), equations/definitions.ts (7 concrete types), generated/dispatch.ts (KSNode, KindToNode). Fully generic: projections.ts (generic Ctx), types.ts (node: unknown), predicates.ts (string constants).

Q

Why does the proposed architecture default-parameterize domain types (KindDefinition<N = ASTNode>) rather than hardcoding KSNode?

A

The default parameter preserves backward compatibility: existing unparameterized usage (KindDefinition) continues to work with ASTNode. Composition roots can narrow it (KindDefinition<KSNode>), and if a future analysis targets a different grammar (e.g., Python AST), it can use KindDefinition<PyNode> with the same type.

Q

What role does the K type parameter in AnalysisDecl<K> play, and what can't it do?

A

K constrains the keys of equation records (Partial<Record<K, Function>>), ensuring equations are only defined for valid node kinds. But K is just a string union — it doesn't provide node field types, sum type membership, or discriminated union narrowing. The structural grammar information requires direct imports from the grammar adapter.
