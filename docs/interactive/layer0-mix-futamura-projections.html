01The Core Equation

Partial evaluation is the idea that if a program takes multiple inputs, and some are known early, you can specialize the program on the known inputs to produce a faster residual program that only needs the remaining inputs.

The Specialization Equation
mix(P, s) = Ps   such that   Ps(d) = P(s, d)

P is the original program. s is the static input (known at specialization time). mix is the partial evaluator. Ps is the specialized residual program. d is the dynamic input (known only at runtime).

In KSC, this maps precisely to:

PE conceptKSC instanceConcrete artifact
P (program)Generic AG interpreterLayer 0 (reconstructed below)
s (static input)Grammar spec + Analysis specGrammarSpec + AnalysisSpec
mix (specializer)The two compilation functorscompileGrammar + compileAnalysis
Ps (residual)Compiled evaluatorgenerated/evaluator.ts
d (dynamic input)User's TypeScript sourceAST at runtime
Key insight: KSC doesn't just use partial evaluation as a metaphor. The two functors literally perform binding-time-guided code generation: they examine the spec (static input), unfold all dynamic dispatch into switch statements, and emit a residual program (evaluator.ts) that contains zero interpretation overhead.

02Layer 0: The Generic Interpreter

Layer 0 is the program P in the specialization equation — the generic AG interpreter that takes any grammar and any analysis spec and evaluates attributes at runtime. KSC removed this layer, but understanding it is essential for seeing what mix eliminates.

What Layer 0 does at runtime

1 Dynamic Schema Lookup ▼

Given a node of kind K, the interpreter looks up schema.getChildFields(K) at runtime to discover which fields exist. This is a Map lookup on every tree traversal step.

// Layer 0: runtime schema lookup on every node
const fields = schemaMap.get(node.kind);  // Map<string, FieldDesc[]>
for (const field of fields) {
  const child = node[field.name];
  if (child) visit(child);
}

In the specialized evaluator: getChildFields returns a precomputed array — the schema is a static lookup table, not a runtime discovery.

2 Dynamic Attribute Dispatch ▼

When attribute A is requested on a node, the interpreter looks up the equation in a Map<attrName, Map<kind, EquationFn>>. Two Map lookups per attribute access.

// Layer 0: double Map lookup per attribute
function evalAttr(node, attrName) {
  const cache = cacheMap.get(node);
  if (cache?.has(attrName)) return cache.get(attrName);

  const equations = attrRegistry.get(attrName);  // Map lookup 1
  const eq = equations.get(node.kind)            // Map lookup 2
            ?? equations.get('_default');
  const result = eq(ctx, node);
  cache.set(attrName, result);
  return result;
}
3 Generic Memoization ▼

Caching uses Map<Node, Map<string, unknown>> — a two-level generic Map. Every cache check is a hash probe plus string comparison. No type safety: values are unknown.

// Layer 0: generic Map cache
const globalCache = new WeakMap<Node, Map<string, unknown>>();

// Every attribute access: 2 hash probes
const nodeCache = globalCache.get(node) ?? new Map();
if (nodeCache.has('kindDefs')) return nodeCache.get('kindDefs');

In the specialized evaluator: each attribute has a dedicated typed field (_c_kindDefs: KindDefinition[] | undefined). Cache check is a single property access: if (this._c_kindDefs !== undefined).

4 Runtime Dep Graph ▼

The dependency graph between attributes is computed at module load by scanning equations and building edges. Topological sort runs at startup. Cycle detection is a runtime check during evaluation.

// Layer 0: build dep graph at startup
const graph = buildDepGraph(allAttrs);  // scan equations, find deps
const order = topologicalSort(graph);   // sort each time

// During evaluation: track cycle with Set
const evaluating = new Set<string>();
if (evaluating.has(attrName)) throw 'cycle!';

In the specialized evaluator: the dep graph is a compile-time constant (KSC_STATIC_DEP_GRAPH). 22 attributes, 27 edges, topological order — all embedded as a static object literal.

Layer 0 pseudocode (complete)

// ── Layer 0: Generic AG Interpreter ──
// ag_eval(grammar, analysisSpec, program) → results

function ag_eval(
  grammar: GrammarSpec,
  spec: AnalysisSpec,
  program: KSNode
): EvaluationResult {
  // 1. Build schema from grammar (static, but done at runtime)
  const schema = buildSchema(grammar.nodes);

  // 2. Register all equations from spec (static, but done at runtime)
  const attrRegistry = new Map<string, Map<string, EquationFn>>();
  for (const attr of deriveAllAttrs(spec)) {
    attrRegistry.set(attr.name, buildEquationMap(attr));
  }

  // 3. Build dep graph and sort
  const depGraph = buildDepGraphFromAttrs([...attrRegistry.keys()]);

  // 4. Build decorated tree with generic cache
  const root = buildTree(program, schema);

  // 5. Evaluate: demand-driven with Map-based cache
  const evalCtx = { schema, attrRegistry, cache: new WeakMap() };
  const defs = project_binder(root, evalCtx);
  const diags = project_checker(root, evalCtx);

  return { definitions: defs, diagnostics: diags, getDepGraph: () => depGraph };
}
This is what mix eliminates. Every one of these runtime operations — schema lookup, equation dispatch, generic caching, dep graph construction — is computed statically by the functors and baked into the generated evaluator. The residual program has zero interpretation overhead.

03Mix: The Partial Evaluator

In KSC, "mix" is split into two stages that progressively specialize the generic interpreter:

Mix1: compileGrammar(GrammarSpec)
// Static input: 364 node kinds, 48 sum types,
// 50+ field extractors
//
// Eliminates:
//   "What children does kind X have?"
//   "How do I convert TS node to KS node?"
//   "What fields does this kind have?"
//
// Residual (7 files):
//   node-types.ts  → typed interfaces
//   schema.ts      → static C map
//   convert.ts     → 364 converter fns
//   builders.ts    → factory functions
//   serialize.ts   → schema-aware JSON
//   kind-map.ts    → type-level map
//   index.ts       → barrel
Mix2: compileAnalysis(AnalysisSpec)
// Static input: 5 structural attrs,
// 8 properties, 27 dep edges
//
// Eliminates:
//   "What attributes exist?"
//   "Which equation for kind/attr?"
//   "Is attr syn/inh/collection?"
//   "What depends on what?"
//
// Residual (2 files):
//   evaluator.ts   → KSCDNode class
//                    22 typed methods
//                    22 typed cache fields
//                    switch dispatch
//                    static dep graph
//   attr-types.ts  → KSCAttrMap interface

What mix does, line by line

The function compileAnalysis in compiler/analysis.ts (716 lines) is the core "mix" operation. It performs these steps:

1 Derive all attributes from spec ▼

deriveAllAttrs(spec) reads the 5 structural + 8 property declarations and mechanically derives 22 total attributes:

  • 5 structural (as-is): kindDefs, defEnv, defLookup, kindAnnotations, nodeCount
  • 8 context attrs (derived): one {name}Context per property — inherited, depends on kindAnnotations
  • 8 violation attrs (derived): one {violationAttr} per property — synthesized, depends on its context
  • 1 collection: allViolations — depends on all 8 violation attrs

This is static computation: the number and names of attributes are fully determined by the spec.

2 Build dependency graph + topological sort ▼

buildDepGraphFromAttrs(attrs) computes 27 dependency edges from the derived attributes and topologically sorts them. This produces the evaluation order that gets embedded as a constant in the generated code.

The edges come from spec structure, not regex scanning:

  • defEnv → kindDefs (explicit dep in structural attr)
  • noImportsContext → kindAnnotations (mechanically derived: all contexts depend on kindAnnotations)
  • importViolation → noImportsContext (each violation depends on its context)
  • allViolations → all 8 violation attrs (collection gathers all)
3 Generate KSCDNode class with typed methods ▼

generateEvaluator() emits TypeScript source code string. For each of the 22 attributes, it generates:

  • A typed cache field: private _c_kindDefs: KindDefinition[] | undefined
  • A direct method: kindDefs(): KindDefinition[]
  • A case in the attr() switch dispatch
  • A case in hasAttr() and _setCache()

Each method body is generated based on the attribute's category:

  • Synthesized: switch(this.node.kind) dispatching to case-specific equations
  • Inherited: root check → parent propagation with VariableDeclaration override
  • Collection: contribute + combine across children
  • Violation: context check → kind switch → predicate call → diagnostic construction
4 Emit static dep graph + evaluate() ▼

The topologically sorted order and all 27 edges are emitted as a constant object literal:

const KSC_STATIC_DEP_GRAPH: AttributeDepGraph = {
  attributes: ['kindDefs', 'defEnv', ...],  // 22 names
  edges: [['defEnv', 'kindDefs'], ...],    // 27 edges
  order: ['kindDefs', 'defEnv', ...],      // topo sort
};

And the entry point evaluate(root) calls buildKSCTree then the two projection functions.

What specialization eliminates: Every comparison above shows the same semantic operation. The left column does runtime dispatch (Map lookups, direction checks, rule iteration). The right column has zero interpretation overhead — all dispatch is statically resolved into switch statements, direct property access, and inlined method calls.

05The Four Futamura Projections

Yoshihiko Futamura (1971) showed that applying a partial evaluator in different configurations produces increasingly powerful results. Here is how all four projections map to KSC:

Projection 1: Compilation

First Futamura Projection
mix(interpreter, source) = target program

Specializing the interpreter on a particular source program produces an efficient target program that directly computes results without interpretation.

In KSC

// mix₁(ag_eval, grammarSpec) → node types + schema + convert
const grammar = compileGrammar(grammarSpec);  // 7 files

// mix₂(ag_eval_grammar, analysisSpec) → evaluator
const analyzer = compileAnalysis(analysisSpec);  // evaluator.ts

// The residual program runs without interpretation:
const tree = buildKSCTree(ast);        // no schema lookup
const result = evaluate(tree);          // no dispatch overhead

Validation: The generated evaluator.ts contains no Map lookups, no attrRegistry, no generic visitor. All 22 attribute methods are hard-coded. This is Projection 1.

Projection 2: Compiler Generation

Second Futamura Projection
mix(mix, interpreter) = compiler

Specializing mix on a particular interpreter produces a compiler — a program that takes any source program (in the interpreter's language) and generates efficient code.

In KSC

// The codegen scripts ARE the compiler:
// mix(mix, ag_eval) = scripts/codegen-grammar.ts
//                   + scripts/codegen-analysis.ts

// These scripts take ANY spec as input:
const spec = importSpec();          // any AnalysisSpec
const result = compileAnalysis(spec);  // → evaluator for THAT spec
writeFile('evaluator.ts', result);   // compiler output

Validation: The scripts codegen-grammar.ts and codegen-analysis.ts are parameterized by the spec. Change the spec (add a property, add a structural attribute), rerun the script, get a new specialized evaluator. The scripts are the compiler; the spec is the source language.

Projection 3: Compiler Generator (Cogen)

Third Futamura Projection
mix(mix, mix) = cogen

Specializing mix on itself produces a compiler generator (cogen): give it any interpreter and it produces a compiler for that interpreter's language.

In KSC

This would be a meta-codegen that takes any codegen template and produces a specialized codegen script. In practice:

// Cogen would be a function that takes:
//   - A description of what "StructuralAttr" means
//   - A description of what "PropertyDecl" means
//   - Template rules for generating methods
// And produces compileAnalysis() itself.

// Currently, compileAnalysis is hand-written (716 lines)
// Projection 3 would auto-generate it from a meta-spec
function cogen(interpreterSpec: InterpreterDescription): CompilerFunction {
  // Analyze the interpreter's binding times
  // Generate a compiler that specializes on static inputs
  // Return: (spec) => GeneratedFile[]
}

Status: Not implemented. KSC has hand-written compilers (the functors). Projection 3 would auto-generate the functors from a higher-level description of the AG evaluation strategy.

Projection 4: Fixpoint (Self-Generation)

Fourth Futamura Projection
cogen(mix) = cogen

Applying cogen to mix produces cogen again. This is the fixpoint — the system is closed. Cogen can generate itself.

In KSC

This would mean: the meta-codegen (Projection 3) can regenerate itself. Concretely, KSC's compiler would be written in terms that KSC itself can analyze and regenerate.

Status: Theoretical. Would require the compiler to be self-hosting — analyzing its own code with its own AG framework, then generating its own codegen scripts.

Real-world precedent: GCC bootstrapping (C compiler compiles itself), PyPy (Python interpreter written in RPython, then specialized by the RPython toolchain).

08Validating the Projections

How do we prove that KSC actually implements Projections 1 and 2, and what would it take for 3 and 4?

Projection 1
Compilation
Show that the generated evaluator has no interpretation overhead
Projection 2
Compiler Generation
Show that the codegen scripts work for any spec
Projection 3
Cogen
Design a meta-codegen that generates compileAnalysis
Projection 4
Fixpoint
Self-hosting: the compiler analyzes and regenerates itself

Projection 1 Validation Checklist

  1. No Map-based dispatch in evaluator.ts — grep for new Map or .get( in the generated evaluator. Only _structural.get() at module load (one-time setup), zero during evaluation.
  2. 22 typed cache fields — _c_kindDefs through _c_nodeCount, each with a concrete TypeScript type. No Map<string, unknown>.
  3. Switch dispatch replaces Map dispatch — the attr() method uses a 22-case switch, not a registry lookup.
  4. Direction-specific methods — syn attributes use switch(this.node.kind), inh attributes use if (this.isRoot) ... else this.parent!.X(). Direction is baked into the method body.
  5. Functional equivalence — run both Layer 0 and the specialized evaluator on the same input, compare results. The 208+ existing tests validate this.

Projection 2 Validation Checklist

  1. Add a new property to the spec — add a PropertyDecl to analysis/spec.ts, rerun npx tsx scripts/codegen-analysis.ts. A new evaluator is generated with 24 attributes (was 22). The codegen script didn't change.
  2. Add a structural attribute — add a new StructuralAttr to the spec. Rerun codegen. New cache field, new method, new switch case appear in evaluator.ts.
  3. Remove a property — remove one from the spec. Evaluator shrinks correspondingly.
  4. The scripts are spec-agnostic — codegen-analysis.ts imports the spec but treats it as data. The script is the compiler; the spec is the source program.

Projection 3: What it would take

Currently, compileAnalysis is a hand-written 716-line function. To achieve Projection 3, you would need:

  1. A meta-description of the AG evaluation strategy — formalize what "syn with switch dispatch" means, what "inh with root/parent propagation" means, what "collection with contribute+combine" means.
  2. A meta-codegen (cogen) that reads this description and generates compileAnalysis itself.
  3. The result: cogen(agEvalDescription) = compileAnalysis. Then compileAnalysis(spec) = evaluator.ts.

This is feasible because the method generation patterns in compiler/analysis.ts are highly regular — the 5 method generators (generateStructuralBody, generateContextBody, generateViolationBody, generateAllViolationsBody, generateCollectionBody) follow clear templates.

Projection 4: The fixpoint

Self-generation requires that cogen can analyze its own code. In KSC terms:

  1. Write cogen using TypeScript that KSC can analyze
  2. KSC analyzes cogen's code with the AG framework
  3. KSC generates a specialized version of cogen from its own spec
  4. The specialized cogen produces the same output as the original

This is the self-bootstrapping property. Real-world examples: GCC compiling itself, PyPy translating its own interpreter. In KSC, this would mean the codegen scripts are themselves analyzed and optimized by the KSC AG framework.

09Three Approaches to Building Layer 0 + Mix

Option A: Reconstruct Layer 0 as a reference interpreter

Idea: Build a working generic AG interpreter that takes GrammarSpec + AnalysisSpec + AST and produces the same results as the compiled evaluator. Then show that the compiled evaluator is the specialization of this interpreter.

Pros

  • Makes the Futamura story concrete and testable
  • Can run both Layer 0 and specialized side-by-side, compare outputs
  • Useful for debugging: run generic interpreter when generated code is suspect
  • Enables benchmarking: measure exact speedup from specialization

Cons

  • ~300-500 LOC of new code that has no production use
  • Must be kept in sync with the specialization output
  • Risk of the reference interpreter diverging from the real system

Effort: Medium (~2 days)

Build interpreter/generic-ag-eval.ts that implements the Layer 0 pseudocode from Section 02. Test it against the same fixtures as the compiled evaluator.

Option B: Build mix as a standalone partial evaluator

Idea: Factor out the specialization logic from compileGrammar and compileAnalysis into a general-purpose mix function that takes a program description + binding-time annotations and produces specialized code.

Pros

  • Clean separation of concerns: mix is a reusable tool
  • Could specialize other interpreters, not just AG eval
  • Directly validates Projection 2 (mix is the compiler generator)
  • Opens path to Projection 3 (mix specializing itself)

Cons

  • High complexity: general PE is hard (termination, polyvariance)
  • Over-engineering for current needs
  • The existing functors already work and are well-tested

Effort: High (~1-2 weeks)

Would require implementing binding-time analysis, an unfolding/folding strategy, and a code generator that works on arbitrary TypeScript ASTs.

Option C: Formalize the existing functors as mix instances

Idea: Don't build new code. Instead, annotate the existing compileGrammar and compileAnalysis to show which parts correspond to which PE operations. Add documentation and tests that validate the projection properties.

Pros

  • Zero new production code
  • Forces precise understanding of the existing architecture
  • Tests validate the projection properties against real outputs
  • Documentation value: makes the PE story explicit in the codebase

Cons

  • No runnable Layer 0 for comparison
  • Projection validation is indirect (structural, not behavioral)
  • Harder to benchmark without a reference interpreter

Effort: Low (~1 day)

Add tests in test/futamura.test.ts that verify projection properties: (1) evaluator.ts has no Map dispatch, (2) adding a property to spec → new evaluator, (3) dep graph is a static constant, (4) cache fields are typed.

Recommended path: Start with Option C (formalize existing), then optionally build Layer 0 (Option A) for benchmarking and comparison. Option B is interesting but premature.

Q

What is "Layer 0" in KSC's architecture, and why doesn't it exist in the codebase?

A

Layer 0 is the generic AG interpreter that takes any grammar + analysis spec + AST and evaluates attributes using dynamic dispatch (Map lookups, generic caching). It doesn't exist because the two compilation functors (compileGrammar and compileAnalysis) specialize it away entirely — the generated evaluator.ts is the residual program.

Q

What is "mix" in the context of partial evaluation, and what are the two mix functions in KSC?

A

Mix is the partial evaluator (specializer): it takes a program P and its static input s, and produces a residual program Ps. In KSC, mix1 is compileGrammar(GrammarSpec) (generates node types + schema) and mix2 is compileAnalysis(AnalysisSpec) (generates the typed evaluator with switch dispatch).

Q

How does the generated evaluator's cache differ from Layer 0's cache?

A

Layer 0 uses Map<string, unknown> per node (2 hash probes per access, no type safety). The generated evaluator uses dedicated typed fields like _c_kindDefs: KindDefinition[] | undefined — a single property check (!== undefined) with full TypeScript type safety. This is one of the key eliminations mix performs.

Q

What is the First Futamura Projection and how does KSC implement it?

A

mix(interpreter, source) = target program. Specializing the interpreter on a source program produces efficient code. In KSC: compileGrammar(grammarSpec) + compileAnalysis(analysisSpec) produce evaluator.ts — a target program with zero interpretation overhead that directly computes results via switch dispatch and typed cache fields.

Q

What is the Second Futamura Projection and what validates it in KSC?

A

mix(mix, interpreter) = compiler. Specializing mix on an interpreter produces a compiler. In KSC: the codegen scripts (codegen-grammar.ts, codegen-analysis.ts) are the "compiler" — they take any spec as input and produce a specialized evaluator. Validation: change the spec (add a property), rerun the script, get a new evaluator. The script is spec-agnostic.

Q

After both mix stages, which computations remain dynamic (can't be resolved at specialization time)?

A

Only three things remain dynamic: (1) the AST structure (the actual tree shape of the user's program), (2) violation predicate outcomes (whether a specific node actually violates a rule), and (3) actual node values (field contents). Everything else — node kinds, field names, attribute names, equation dispatch, dep graph, cache structure — is resolved to static constants.

Q

How many attributes, edges, and properties does KSC's compiled evaluator have, and how are the 17 derived attributes generated?

A

22 attributes, 27 edges, 8 properties. From 5 structural + 8 properties, mix mechanically derives: 8 context attrs (inh, one per property), 8 violation attrs (syn, one per property), and 1 allViolations collection = 17 derived. Total: 5 + 17 = 22. All 27 edges come from spec structure (no manual wiring).

Q

What would Projection 3 (cogen) look like in KSC, and why hasn't it been implemented?

A

Projection 3 would be a meta-codegen that takes a description of the AG evaluation strategy (what "syn with switch dispatch" means, what "inh with parent propagation" means) and auto-generates compileAnalysis itself. It hasn't been built because compileAnalysis is a 716-line hand-written function that works well, and the meta-description formalism would be a significant research effort for limited practical benefit at this scale.
