Practical Analysis & the Mathematics of Programs

What you can actually learn — by reading code, running code, and combining both.

1 Can you just… check for the obvious non-termination?

Yes! And this is exactly what real tools do. The undecidability of termination means you can't build a perfect checker — but you can build an extremely good one.

The pragmatist's approach: Instead of asking "does this program halt?" (undecidable), ask "does this program contain any of the known patterns that cause non-termination?" That question is decidable — it's a syntactic check.

Here are the patterns that cause non-termination in TypeScript. If a function avoids all of them, it almost certainly terminates:

PatternWhy it's dangerousDecidable to detect?
while (true) Infinite loop with no guaranteed exit Yes — literal pattern match
while (condition) where condition isn't provably shrinking Loop may never exit Partially — heuristics work well
Unbounded recursion: f() { f(); } Call stack grows forever Partially — call graph + no base case detection
Mutual recursion: f→g→f→g→… Cycle in call graph Yes — cycle detection on call graph
for (;;) or do {} while (true) Same as while(true) Yes — syntax
Generator that yields forever function* () { while(true) yield; } Yes — yield inside unbounded loop
Event loop hangs: unresolved Promise new Promise(() => {}) Partially — specific pattern matching

The key insight: Rice's theorem says you can't be perfect. But "not perfect" ≠ "useless." In practice, you can catch 95%+ of non-termination bugs with pattern matching and heuristics. The remaining edge cases are pathological — programs that terminate for some deeply number-theoretic reason no one can prove.

This is the difference between theory (impossible to be perfect) and practice (easy to be good enough). Most static analysis lives in this gap.

The practical strategy: assume termination, flag the known dangerous patterns, and verify with testing. This is sound engineering even if it's not mathematically complete.

2 What running a program tells you

When you execute a program, you get a trace — a concrete record of everything that happened. Each trace is a finite window into the computation. The question is: what can you infer from these windows?

3 Two views, one program

Here's the deep point: static analysis (reading the code) and dynamic analysis (running the code) give you complementary information. Each sees things the other misses.

Click a stage to see what it reveals.
How they feed each other:

Static analysis tells you what might happen — all possible paths, all reachable code. But it over-approximates: it warns about paths that may never actually execute.

Dynamic analysis tells you what does happen — concrete, actual behavior. But it under-approximates: it only sees the inputs you tested.

Together: Static narrows the search space, dynamic confirms or refutes within that space. Static says "this function might throw." Dynamic says "in 10,000 runs it never threw." Neither is proof, but together they give high confidence.

5 Computations as mathematical objects

A computation is what happens when you run a program. While a program is a static, finite object, a computation can be infinite. Click to explore:

Click a representation to learn more.
The program → computation relationship:

A program is a finite description. A computation is the (possibly infinite) behavior it generates. The relationship between them is:

program + input → computation

Or mathematically: a program is a function from inputs to computations. This is why program properties (finite, syntactic, decidable) and computation properties (possibly infinite, semantic, often undecidable) are so different — you're analyzing the description vs. analyzing the behavior it produces.

6 How knowing one tells you about the other

This is the payoff. The static (program) and dynamic (computation) views aren't separate — they're two projections of the same thing, and each constrains the other.

The practical upshot:

You don't have to choose between static and dynamic. The most powerful analysis tools combine both:

1. Static pass: Scan the AST for primitives used, build the call graph, flag dangerous patterns. This gives you the upper bound on what the program can do.
2. Dynamic pass: Run the program on many inputs, observe actual traces, measure actual timing. This gives you concrete evidence about what it actually does.
3. Cross-reference: If static says "might throw" but 10,000 runs never throw → high confidence it's safe (though not proof). If dynamic reveals a crash that static missed → improve the static model.
