Frontend → Analysis Boundary

How KSC and Facebook Infer handle the critical seam between AST generation (frontends) and analysis (checkers) — and what design options exist

1. The Core Problem

Every analysis framework that supports multiple source languages (or wants to evolve its AST independently of its analysis) faces a fundamental architectural question:

The Boundary Question

How does the analysis stage know about the node types defined by the frontend/grammar stage? How do you get exhaustiveness guarantees that every AST node kind is handled — and what happens when the grammar changes?

This is really three sub-problems:

  1. Type flow — how do AST node type definitions propagate from the grammar layer to the analysis layer?
  2. Exhaustiveness — how do you statically verify that the analysis handles all possible node kinds?
  3. Cross-validation — how do you detect when the analysis references node kinds that don't exist in the grammar?

Both KSC and Infer have elegant but very different answers to these questions, driven by their different host languages (TypeScript vs OCaml) and different goals (attribute grammar evaluation vs abstract interpretation).

2. Facebook Infer's Architecture

Infer is a static analyzer for Java, C/C++, Objective-C, Hack, and Erlang, written in OCaml. Its architecture has a clean capture → analyze pipeline with the Smallfoot Intermediate Language (SIL) as the boundary.

▶ How Infer's Frontends Generate Type-Safe IR

The Clang Frontend Pipeline

Infer's most sophisticated frontend is for C/C++/Objective-C. It works through a multi-stage codegen pipeline:

1
Clang AST export — A custom Clang plugin (ASTExporter.h) serializes the Clang AST to JSON/Biniou format. ATD annotations embedded in C++ preprocessor macros enumerate every Clang node kind.
2
ATDGen codegen — The atdgen tool reads ATD definitions extracted from the C++ source and generates OCaml variant types + parsers. This produces clang_ast_t.ml with a variant constructor for every Clang AST node type.
3
Translation to SIL — cTrans.ml pattern-matches over the clang_ast_t variant types, translating each AST construct to SIL instructions. OCaml's exhaustive pattern matching guarantees at compile time that every Clang AST node kind is handled.
4
SIL validation — A SilValidation checker verifies that all emitted SIL instructions conform to frontend-specific invariants. This provides a runtime cross-check alongside the compile-time guarantees.

The SIL Type System

SIL is defined as OCaml variant types in infer/src/IR/Sil.mli:

type instr =
  | Load of { id: Ident.t; e: Exp.t; typ: Typ.t; loc: Location.t }
  | Store of { e1: Exp.t; typ: Typ.t; e2: Exp.t; loc: Location.t }
  | Prune of Exp.t * Location.t * bool * if_kind
  | Call of (Ident.t * Typ.t) * Exp.t * (Exp.t * Typ.t) list * Location.t * CallFlags.t
  | Metadata of instr_metadata

Because this is a closed OCaml variant type with only 5 constructors, every checker that processes SIL instructions must handle all 5 cases. The compiler enforces this.

▶ How Checkers Consume the IR

The Absint Framework

Infer's abstract interpretation framework (Absint) is the standard way to write analyses. A checker provides:

module type TransferFunctions = sig
  type analysis_data
  module CFG : ProcCfg.S

  (* The core: how does each SIL instruction transform the abstract state? *)
  val exec_instr :
    Domain.t           (* current abstract state *)
    -> analysis_data
    -> CFG.Node.t      (* CFG node containing the instruction *)
    -> int             (* instruction index *)
    -> Sil.instr       (* THE SIL INSTRUCTION *)
    -> Domain.t        (* new abstract state *)
end

The key parameter is Sil.instr — the checker operates on the shared intermediate language, not on source-language AST nodes. This is what makes checkers language-independent.

Exhaustiveness in Checkers

When a checker pattern-matches on Sil.instr:

let exec_instr astate _proc_data _node _idx instr =
  match instr with
  | Load {id; e; _} ->  (* handle load *)
  | Store {e1; e2; _} -> (* handle store *)
  | Call (ret, fun_exp, args, _, _) -> (* handle call *)
  | Prune (cond, _, _, _) -> (* handle branch *)
  | Metadata _ -> astate  (* skip metadata *)

OCaml's type system ensures that if a new SIL instruction variant is added, every checker fails to compile until it handles the new case. This is the gold standard of exhaustiveness enforcement.

▶ Infer's Multiple Input Formats (Evolution of the Boundary)

Over time, Infer has evolved three distinct ways for frontends to emit SIL:

FormatUsed byHow it works
Native OCaml Clang, Java frontends Frontend directly constructs OCaml SIL data structures. Maximum type safety — OCaml compiler catches all errors.
Textual Hack frontend Text-based SIL format parsed by Infer. Looser coupling — frontend can be in any language. Validated at parse time.
JSON SIL C# (Infer#) Language-agnostic JSON serialization. Deserialized into OCaml SIL types. Runtime validation during deserialization.
Key Insight

The evolution from native OCaml → Textual → JSON shows a general tension: tighter coupling gives stronger type safety, while looser coupling enables more frontends. Infer chose to support all three, using SilValidation as a safety net for the loosely-coupled paths.

3. KSC's Architecture

KSC uses a two-functor specialization architecture where both the grammar and analysis stages are code generators, producing TypeScript source files that are then compiled together by tsc.

▶ Functor 1: Grammar → AST Types (the "Frontend")

KSC's Functor 1 is analogous to Infer's frontend: it takes a declarative grammar spec and generates typed AST node definitions.

GrammarSpec — the input

interface GrammarSpec {
  nodes: ReadonlyMap<string, NodeEntry>;     // 364 node kinds
  sumTypes: ReadonlyMap<string, SumTypeEntry>; // union types
  convertGenerator?: ConvertGenerator;          // spec-owned
}

Generated output — what it produces

The key files for the boundary question are:

  • node-types.ts — One TypeScript interface per node kind (KSIdentifier, KSCallExpression, etc.) plus the KSNode discriminated union of all 364 kinds
  • kind-map.ts — KindToNode type mapping from kind string literals to their interface types (enables type-safe violation predicates)
  • schema.ts — Exhaustive child-field metadata table (every kind has an entry) + allKinds set
// Generated KSNode union type (364 members)
export type KSNode =
  | KSIdentifier
  | KSCallExpression
  | KSVariableDeclaration
  | KSFunctionDeclaration
  | ... // 360 more
  ;

// Generated KindToNode type map
export type KindToNode = {
  'Identifier': KSIdentifier;
  'CallExpression': KSCallExpression;
  'VariableDeclaration': KSVariableDeclaration;
  ... // 361 more
};
▶ Functor 2: Analysis → Evaluator (the "Backend")

KSC's Functor 2 generates an evaluator that processes the AST produced by Functor 1. The generated evaluator imports KSNode from the grammar output and dispatches on node.kind strings.

How the evaluator uses grammar types

// Generated evaluator.ts — imports grammar types
import type { KSNode } from '../grammar/index.js';
import { getChildFields } from '../grammar/index.js';

// Switch dispatch on node.kind (string discriminant)
kindDefs(): KindDefinition[] {
  switch (this.node.kind) {
    case 'CompilationUnit': return this._c_kindDefs = eq_kindDefs_CompilationUnit(this, this._counter);
    default: return this._c_kindDefs = eq_kindDefs_default();
  }
}

Key difference from Infer

The evaluator does not need to exhaustively match all 364 kinds. Instead, it uses default cases in switches. Analysis specs declare which specific kinds they care about, and the rest fall through to defaults. The getChildFields() schema function handles the exhaustive tree traversal.

▶ Cross-Validation: The Safety Net

Since TypeScript doesn't enforce exhaustive string matching the way OCaml enforces variant matching, KSC uses a codegen-time validation step:

// analysis/validate.ts
function validateSpec(spec: AnalysisSpec, grammar: CompiledGrammar) {
  // Check: every violation triggerKind exists in grammar
  for (const rule of spec.violationRules) {
    if (!grammar.hasKind(rule.triggerKind)) {
      diagnostics.push({ level: 'error', message: `...unknown kind '${rule.triggerKind}'...` });
    }
  }

  // Check: every equation case key exists in grammar
  for (const attr of spec.attrs) {
    for (const caseName of Object.keys(attr.equations.cases)) {
      if (!grammar.hasKind(caseName)) {
        diagnostics.push({ level: 'error', message: `...unknown kind...` });
      }
    }
  }
}

This catches typos and stale references at codegen time, before any generated code is compiled or run. It's weaker than OCaml's compile-time exhaustiveness, but it precisely targets the specific problem: "does the analysis reference valid grammar kinds?"

Fundamental Difference

Infer lowers source ASTs to a shared IR (SIL), erasing language-specific details. KSC preserves the full AST — analysis operates directly on grammar-defined node types. This means KSC's analysis can inspect any grammar field (e.g., node.declarationKind), while Infer's checkers only see SIL's 5 instruction types.

Exhaustiveness Guarantees

Mechanism Infer KSC
Frontend → IR Compile-time: OCaml pattern match on clang_ast_t variant. Missing cases = compiler error Codegen-time: Grammar spec exhaustively defines all kinds. schema.ts has an entry for every kind. Expression validation checks extractor references

Why KSC Doesn't Need Full Exhaustiveness

In Infer, checkers must handle every SIL instruction because missing a case means losing information in the abstract state. In KSC's attribute grammar model, the tree traversal is always exhaustive (every node is visited via schema-driven getChildFields). Analysis specs only need kind-specific cases for nodes where the default behavior isn't sufficient. An unhandled kind just gets the default equation, which is semantically correct for the AG evaluation model.

How Analysis Code Dispatches on Node Types

(* Infer: checker matches on closed SIL.instr variant *)
let exec_instr astate _ _ _ instr =
  match instr with
  | Load {id; e; typ; _} ->
      (* update abstract state for load *)
      Domain.add_live id astate
  | Store {e1; _; e2; _} ->
      (* update abstract state for store *)
      Domain.kill_live e1 astate
  | Call (ret, _, args, _, _) ->
      (* handle function call *)
      ...
  | Prune _ -> astate
  | Metadata _ -> astate
  (* Exhaustive — adding a 6th variant would break this *)

Model: Closed variant, exhaustive match required, language-independent. Analysis never sees source-language constructs.

// KSC: generated evaluator switches on node.kind string
// (from analysis spec's equation cases)

Because KSC analysis specs use default cases, adding new AST node kinds is non-breaking for existing analyses. This is a significant advantage over Infer's model where a new SIL variant is a global breaking change.

5. The Deep Structural Differences

Source AST vs Shared IR: Two Fundamentally Different Boundaries

The most important architectural difference between KSC and Infer isn't a detail — it's a category distinction in where the frontend/analysis boundary sits:

Infer: Source AST ──→ [TRANSLATION] ──→ Shared IR (SIL) ──→ Analysis │ │ Information loss: language-specific structure erased │ Information gain: uniform 5-instruction model KSC: Source AST ──→ [CODEGEN] ──→ Typed AST (KSNode) ──→ Analysis │ │ No information loss: all fields preserved │ Schema metadata drives exhaustive traversal
▶ Why This Difference Matters for Exhaustiveness

In Infer, exhaustiveness is mandatory because the analysis model (abstract interpretation over CFGs) requires processing every instruction. Missing an instruction means the abstract state could be unsound — you might miss a null pointer dereference because a Store wasn't tracked.

In KSC, exhaustiveness is structural rather than behavioral. The attribute grammar model separates two concerns:

  1. Tree traversal (always exhaustive) — driven by the generated schema's getChildFields(), which has an entry for every grammar kind. Every node is visited, every child field is followed.
  2. Per-kind behavior (selective by design) — analysis specs declare equations for specific kinds they care about. The AG evaluation model provides correct defaults: inherited attributes propagate down, collections aggregate up, synthesized attributes use the default equation.

This means KSC gets a different kind of exhaustiveness guarantee: not "every node kind has explicit handling" but "every node participates correctly in the attribute evaluation regardless of whether it has explicit handling."

▶ Infer's ATDGen ≈ KSC's compileGrammar: The Codegen Analogy

Both systems use code generation to bridge the gap between language-specific AST definitions and their analysis-consumable representations:

Step Infer KSC
Input format ATD definitions embedded in C++ macros (ASTExporter.h) TypeScript GrammarSpec with builder DSL (specs/.../spec.ts)
Codegen tool atdgen (external OCaml tool) compileGrammar() (built-in TypeScript function)
Output OCaml variant types + JSON parsers (clang_ast_t.ml) TypeScript interfaces + schema + builders (node-types.ts, etc.)
What it enables Exhaustive pattern matching on source AST for translation to SIL Typed AST access + schema-driven traversal for attribute evaluation
Update process Update Clang plugin → re-run atdgen → fix compile errors Update grammar spec → re-run compileGrammar → fix validation errors
The Key Parallel

Both systems solve the same problem: "given a description of AST node types, generate type-safe code for processing those nodes." The difference is what happens after the codegen: Infer translates to a simpler IR, while KSC preserves the full AST and instead generates a schema that makes traversal safe.

▶ String Discriminants vs Closed Variants: TypeScript's Limitation

A fundamental difference comes from the host language's type system:

OCaml: Closed Algebraic Data Types

(* The compiler KNOWS all possible variants *)
type sil_instr = Load of ... | Store of ... | Prune of ... | Call of ... | Metadata of ...

(* Incomplete match = compile error *)
match instr with
| Load _ -> ...
| Store _ -> ...
(* Error: this pattern-matching is not exhaustive.
   Here is an example of a case that is not matched: Prune _ *)

TypeScript: Discriminated Unions with String Literals

// TypeScript CAN check exhaustiveness on discriminated unions...
type KSNode = KSIdentifier | KSCallExpression | ...;

// ...but only with explicit exhaustive switch + never check:
switch (node.kind) {
  case 'Identifier': ...
  case 'CallExpression': ...
  default: {
    const _exhaustive: never = node; // compile error if cases missing
  }
}

// However, with 364 kinds this is impractical!
// KSC's approach: use `default:` for unhandled cases,
// validate kind references at codegen time instead

With 364 node kinds, TypeScript's exhaustive switch pattern is unwieldy. KSC's design acknowledges this by making exhaustiveness a schema-level property (the generated child-field table is exhaustive) rather than a switch-level property (individual attribute methods don't need to be exhaustive).

6. Design Options for KSC

Given these two reference architectures, what options exist for designing the frontend/analysis boundary? Here are three approaches, ranging from "more like Infer" to "more like current KSC" to "hybrid."

Option A: Shared IR (Infer-style)

Define a small, fixed IR that all grammars lower to. Analysis specs operate on the shared IR, not on grammar-specific node types.

+ Checkers work across grammars

+ Small IR = easy exhaustive matching

− Loses language-specific AST details

− Translation is complex, error-prone

− Doesn't suit AG model (attrs need tree structure)

Option B: Current KSC (Grammar-Preserving)

Keep the current design: analysis operates on the full grammar AST. Codegen-time validation catches invalid kind references. Schema-driven traversal provides structural exhaustiveness.

+ Full AST access in analysis

+ AG model works naturally

+ Schema evolution is non-breaking

− No cross-grammar checker reuse

− String-based dispatch lacks compile-time safety

Option C: Hybrid (Typed Kind Groups)

Generate TypeScript type-level groupings from grammar sum types. Analysis specs declare which sum types they operate on. Codegen generates type-narrowed dispatchers with partial exhaustiveness.

+ Better type safety than string dispatch

+ Preserves full AST access

+ Sum-type-level exhaustiveness feasible

− More complex codegen

− Requires sum types to be well-designed

▶ Option A Deep Dive: Shared IR

This would mean adding a "lowering" phase between Functor 1 and Functor 2:

GrammarSpec ──→ compileGrammar ──→ KSNode types │ ▼ lowerToIR(KSNode) ──→ KscIR │ ▼ compileAnalysis(AnalysisSpec) ──→ evaluator over KscIR

You would define something like:

type KscIR =
  | { type: 'declaration'; name: string; initializer?: KscIR }
  | { type: 'expression'; operator?: string; operands: KscIR[] }
  | { type: 'statement'; body: KscIR[] }
  | { type: 'type-annotation'; typeKind: string }
  | { type: 'literal'; value: unknown }
  | { type: 'container'; children: KscIR[] }
Why This Is Problematic for KSC

KSC's kind-checking analysis specifically needs to inspect language-level constructs (e.g., "is this a VariableDeclaration with const?", "does this Identifier resolve to an import?"). These questions are unanswerable from a lowered IR. The attribute grammar model also relies on the tree structure matching the source — lowering to a different tree shape would break inherited/synthesized attribute propagation.

▶ Option B Deep Dive: Current KSC (Strengthened)

The current design is already well-suited to KSC's goals. Possible enhancements:

Enhancement 1: Typed Equation Case Keys

Currently, equation case keys are plain strings validated at codegen time. With KindToNode, we could make them type-safe:

// Instead of: cases: { 'VariableDeclaration': fn }
// Type-safe version using KindToNode:
type TypedEquationSet<K extends keyof KindToNode> = {
  cases: { [P in K]?: (ctx: Ctx, node: KindToNode[P]) => unknown };
  default?: (ctx: Ctx) => unknown;
};

// Compile error if you reference a non-existent kind:
cases: {
  'VariableDeclaration': (ctx, node: KSVariableDeclaration) => ..., // ok
  'FakeNode': (ctx, node) => ...  // TypeScript error!
}

Enhancement 2: Schema-Validated Runtime Dispatch

The generated evaluator could include a development-mode check:

// In dev mode, verify at tree-build time that every node.kind is in allKinds
if (process.env.NODE_ENV === 'development') {
  if (!allKinds.has(raw.kind)) {
    throw new Error(`Unknown kind '${raw.kind}' not in grammar schema`);
  }
}

Enhancement 3: Coverage Reports

Codegen could generate a coverage report showing which kinds have explicit equation cases vs. falling through to defaults. This helps spec authors verify they haven't forgotten important cases without requiring exhaustiveness.

▶ Option C Deep Dive: Hybrid with Sum Type Groups

This approach leverages KSC's existing sum types to create type-safe groups that analysis specs can declare exhaustive matching over:

// Grammar already defines sum types:
sumType('Expression', ['CallExpression', 'Identifier', 'BinaryExpression', ...]);
sumType('Statement', ['ExpressionStatement', 'ReturnStatement', ...]);

// Analysis spec could declare sum-type-scoped equations:
{
  name: 'expressionPurity',
  direction: 'syn',
  scope: 'Expression',  // restrict to this sum type
  equations: {
    cases: {
      'CallExpression': (ctx, node) => ...,
      'Identifier': (ctx, node) => ...,
      // codegen warns about unhandled Expression members
    },
    default: (ctx) => true,
  }
}

The codegen could then:

  1. Validate that all case keys are members of the declared sum type
  2. Generate a coverage warning for unhandled members
  3. Use the sum type's type guard for narrowing: if (isExpression(this.node)) { ... }
Why This Is Attractive

Sum types are small enough for practical exhaustive matching (typically 5-30 members vs. 364 total kinds). They represent meaningful semantic groupings. And KSC's grammar already generates type guards for them. The main cost is additional codegen complexity and requiring sum types to be thoughtfully designed.

7. Architecture Summary

Dimension Infer KSC
Host language OCaml (algebraic data types, pattern matching) TypeScript (structural types, string discriminants)

Boundary shape Source AST → shared IR (SIL) → checkers Grammar spec → typed AST (KSNode) → evaluator
Node type count at boundary 5 SIL instruction types (fixed) 364 grammar-specific node kinds (extensible)

Invalid reference detection OCaml type error (compile time) validateSpec() (codegen time) + TS compilation

Cross-language reuse Checkers reuse across all languages (via shared IR) Analysis specs are grammar-specific (by design)

The Architectural Insight

Infer and KSC represent two legitimate poles of the frontend/analysis boundary spectrum. Infer prioritizes universal analysis (write once, analyze any language) at the cost of losing source-level detail. KSC prioritizes source-level precision (full AST access for language-specific analysis) at the cost of per-grammar analysis specs. Neither is "better" — they serve different goals. The right choice depends on whether your analyses need language-specific detail (KSC-style) or language-independent reasoning (Infer-style).

Q3: What is the closest analog to Infer's ATDGen codegen in KSC?

ATDGen takes Clang header definitions and generates OCaml types for AST nodes. compileGrammar() takes a GrammarSpec and generates TypeScript interfaces for AST nodes. Both solve the same problem: "given a description of node types, generate type-safe code for processing them." convertGenerator is the closest to Infer's frontend translation (AST → IR), not the type generation step.
Q4: An Infer checker works on all supported languages because:

All languages are lowered to the same SIL, so checkers only need to handle SIL types
Infer uses dynamic dispatch to route to the right checker
Each language frontend includes analysis hooks
This is the fundamental value proposition of a shared IR: write a checker once against SIL's 5 instruction types, and it automatically works for any language that has a frontend emitting SIL. The cost is that language-specific constructs are erased — a checker can't ask "is this a Python list comprehension?" because that concept doesn't exist in SIL.
Q5: When a new node kind is added to KSC's grammar, what happens to existing analysis specs?
They fail to compile — every equation switch must be updated
They fail validation — validateSpec() reports the uncovered kind
They continue working — the new kind falls through to default equations and participates via schema-driven traversal
They require a manual migration to add a case for the new kind
This is one of KSC's key design advantages. New node kinds automatically participate in tree traversal (via the regenerated schema), inherit attributes from parents, contribute to collections, and use default synthesized values. Only analysis specs that need kind-specific behavior for the new kind need updates. Compare this to Infer where a new SIL variant is a global breaking change requiring every checker to update.
