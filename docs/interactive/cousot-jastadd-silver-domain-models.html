The Three Frameworks at a Glance

Overview

Cousot Abstract Interpretation

What it is: A mathematical framework for sound approximation of program semantics. The "weak interpreter" is the abstract interpreter — a systematically weakened version of the concrete interpreter that trades precision for computability.

Core idea: Start with a concrete semantics (exact but undecidable). Derive abstract semantics via Galois connections. The abstract interpreter computes fixpoints over abstract domains, guaranteed to be sound (never misses real behaviors) but potentially incomplete (may report false alarms).

Applied to grammars: Cousot & Cousot (2007/2011) showed that classical grammar analyses — Knuth's problem, flow analysis, parsing — are all abstract interpretations of a fixpoint "protoderivation semantics."

P. Cousot & R. Cousot. POPL 1977, TCS 2011. ENS Paris / NYU.

JastAdd Metacompiler

What it is: A code generator for Reference Attribute Grammars (RAGs). Takes .ast + .jrag/.jadd aspect files and weaves them into Java classes with demand-driven evaluation, caching, and circular attribute support.

Core idea: Define tree structure in .ast files. Define attribute equations in aspect files. JastAdd generates Java code where each attribute access triggers lazy evaluation. Circular attributes use Magnusson-Hedin fixpoint iteration.

Key innovation: Reference attributes (pointers to arbitrary tree nodes), NTAs (computed subtrees), collection attributes (gather contributions from entire tree).

G. Hedin et al. Lund University. Since ~2003.

Silver AG Language

What it is: A full programming language for writing attribute grammars, compiling to Java. Self-hosted (Silver is written in Silver).

Core idea: Nonterminals are types, productions are constructors, attributes are declared to "occur on" nonterminals, equations live inside productions. Undecorated trees (data) become decorated trees (attributed) by supplying inherited attributes.

Key innovation: Forwarding (extension productions delegate to host-language trees), Modular Well-Definedness Analysis (guarantees safe composition of independently-developed extensions).

E. Van Wyk et al. U. Minnesota MELT group. Since ~2006.

Cousot: The Abstract Interpretation Domain Model

Core Objects

A Note on the Term "Weak Interpreter"

The specific phrase "weak interpreter" is not a named concept in Cousot's published work. What Cousot's framework describes is an abstract interpreter — a systematically weakened version of the concrete interpreter. The "weakness" is the defining feature: it trades precision for computability. We use "weak interpreter" here as an intuitive label for this central idea.

Cousot's grammar-related work (TCS 2011) applies abstract interpretation to context-free grammar derivation semantics — showing that classical grammar analyses (FIRST/FOLLOW, parsing) are progressively weaker interpreters of the same protoderivation semantics. Separately, Martin Jourdan (INRIA, 1984) developed the recursive evaluator for attribute grammars — a demand-driven/lazy AG evaluator from the same French research milieu. These are distinct contributions that address different problems.

Cousot's abstract interpretation is a theory of sound approximation. The core idea: systematically weaken (abstract) a concrete interpreter to make it computable, while guaranteeing it never misses real behaviors. Every component of the concrete interpreter has a corresponding abstract counterpart.

The Core Objects

The Seven Pillars of Abstract Interpretation

Concrete Domain (C, ≤) A complete lattice of concrete properties (e.g., sets of program states, sets of execution traces). The "ground truth" — what actually happens when a program runs. Abstract Domain (A, &sqle;) A complete lattice of abstract properties (e.g., intervals [lo,hi], sign {+,-,0}, types). Machine-representable approximations of concrete properties. Abstraction (α: C → A) Maps concrete properties to their "best" abstract approximation. Monotone: more concrete detail → at least as precise an abstraction. Concretization (γ: A → C) Maps abstract properties back to the set of concrete properties they represent. Every abstract value "means" a set of possible concrete values. Galois Connection (α, γ) The fundamental bridge: α and γ form an adjunction where α(c) &sqle; a ⇔ c ≤ γ(a). Ensures the abstraction is "best possible" — no unnecessary loss of information. Transfer Functions (f#) Abstract counterparts of concrete operations. If concrete operation f transforms states, then f# transforms abstract states soundly: α(f(c)) &sqle; f#(α(c)). Fixpoint Computation Programs with loops/recursion need fixpoints. The concrete fixpoint (lfp f) is approximated by computing lfp f# in the abstract domain, possibly accelerated by widening (∇) and refined by narrowing (Δ).

The Concrete ↔ Abstract Symmetry

CONCRETE WORLD ABSTRACT WORLD ("weak interpreter") Concrete Domain (C, ≤) ←— γ — (A, &sqle;) Abstract Domain exact states — α —→ approximate states {x=3, y=7} x ∈ [1,10], y ∈ [0,∞) Concrete Semantics — abstracted to —→ Abstract Semantics all possible traces sound over-approximation (undecidable in general) (computable) Concrete Transfer f — abstracted to —→ f# Abstract Transfer x := x + 1 [lo,hi] → [lo+1, hi+1] Concrete Fixpoint lfp f — approximated by —→ lfp f# Abstract Fixpoint (may not terminate) (terminates via widening)

The Hierarchy of Semantics

A key Cousot contribution: program semantics form a hierarchy of abstractions, each level an abstract interpretation of the one below it:

Cousot's Semantic Hierarchy (by successive abstraction)

  1. Maximal Trace Semantics — all possible execution traces (most concrete)
  2. Big-Step Semantics — input→output pairs (abstracting away intermediate steps)
  3. Relational Semantics — natural, demoniac, angelic variants
  4. Denotational Semantics — Scott/Strachey-style function denotations
  5. Collecting Semantics — sets of reachable states at each program point
  6. Predicate Transformer Semantics — wp / wlp (Dijkstra)
  7. Axiomatic Semantics — Hoare triples {P}S{Q} (most abstract)

Each level is connected to the one above by a Galois connection. The abstract interpreter at each level is a "weaker" interpreter than the one below — it sees less detail but can answer coarser questions more efficiently.

Applied to Grammars

Cousot & Cousot (2007/2011) showed that grammar analysis follows the same pattern. Starting from a concrete protoderivation semantics (all possible derivation steps of a context-free grammar, modeled as a transition system akin to a pushdown automaton), successive abstractions yield:

  • Derivational semantics — sets of derivation sequences
  • Equational (language) semantics — Knuth's context-free grammar problem
  • Grammar flow analysis — classical FIRST/FOLLOW/nullable analyses
  • Parsing algorithms — Earley's algorithm is an abstract interpretation of the derivation semantics

Each of these is a progressively weaker interpreter of the grammar — losing detail about derivations but gaining computability and efficiency.

JastAdd: The Reference Attribute Grammar Domain Model

Core Objects

JastAdd is an engineering tool: it takes a grammar specification and weaves it into Java classes. Its domain model is organized around three file types (.ast, .jrag, .jadd) and a demand-driven evaluation engine.

Core Objects

The JastAdd Object Inventory

AST Class A node type in the abstract syntax tree. Defined in .ast files. Generates a Java class extending ASTNode. Has typed, named children, optional children ([C]), list children (C*), and tokens (<Name:String>). Child A structural sub-node of an AST class. Accessed via generated getX() methods. Defines the tree shape — the "functor" in algebraic terms. Token A leaf-level value (string, integer) stored directly on a node. Not a tree child — primitive data. Syntax: <Name:Type>. Aspect A named module in .jrag (declarative) or .jadd (imperative) files grouping related attribute declarations, equations, and inter-type declarations. JastAdd weaves aspects into generated classes. Synthesized Attribute Value computed bottom-up from a node's own children. Declared with syn T A.x();. May have a default equation and be overridden in subclasses. Inherited Attribute Value passed top-down from a parent. Declared with inh T A.y();. Equation defined in the parent: eq C.getA().y() = expr;. Collection Attribute Value assembled from contributions scattered across the tree. Declared with coll T A.c() [init] with combiner;. Contributions via N contributes v to M.c();. Circular Attribute Attribute whose definition is self-referential. Evaluated by iterating from a bottom value until a fixpoint is reached. Syntax: syn T A.x() circular [bottom] = expr;. Parameterized Attribute Attribute that takes parameters, behaving like a cached method. syn T A.x(int a);. Cached per parameter combination. NTA (Nonterminal Attribute) An attribute whose value is an entire AST subtree, computed on demand. A "higher-order attribute." Syntax: syn nta C A.anNTA() = new C(...); or declared in .ast as /C/. Equation Defines how to compute an attribute's value. For syn: eq A.x() = expr;. For inh: eq Parent.getChild().y() = expr;. Must be side-effect-free. Rewrite A rule that replaces an AST node with a different node on first access. Conditional (when) or unconditional. Iterates until stable. Unique to JastAdd. Inter-type Declaration Methods, fields, or interface implementations injected into AST classes from aspect files. Java-like syntax: void A.method() { ... }. Refinement One aspect overriding a method from another aspect: refine AspectName void A.m() { ... refined(); }.

The Three File Types

1 .ast — Tree Structure

// Defines the shape of the AST
abstract Expr;
Add : Expr ::= Left:Expr Right:Expr;
Num : Expr ::= <Value:int>;
Stmt ::= Expr*;
IfStmt : Stmt ::= Cond:Expr Then:Stmt [Else:Stmt];

2 .jrag — Declarative Aspects

aspect Eval {
  syn int Expr.eval();
  eq Add.eval() = getLeft().eval()
                  + getRight().eval();
  eq Num.eval() = getValue();
}

aspect TypeCheck {
  inh Env Expr.env();
  eq Stmt.getExpr(int i).env() = localEnv();
}

3 .jadd — Imperative Aspects

aspect PrettyPrint {
  // inter-type method
  void Expr.prettyPrint(PrintStream out) {
    out.print("<expr>");
  }

  // refine method from another aspect
  refine Eval void Add.prettyPrint(
    PrintStream out) {
    getLeft().prettyPrint(out);
    out.print(" + ");
    getRight().prettyPrint(out);
  }
}

Evaluation Model

JastAdd Demand-Driven Evaluation User code calls: node.someAttribute() | [is cached?] / \ yes no | | return cached [compute equation] | may trigger other attribute accesses (recursive demand) | [is circular?] / \ no yes | | cache & return [iterate from bottom] | evaluate equation compare with previous | [changed?] / \ yes no | | re-evaluate fixpoint reached (loop) cache & return

Silver: The Extensible AG Language Domain Model

Core Objects

Silver is a language for writing attribute grammars. Its domain model centers on the distinction between undecorated trees (pure structure) and decorated trees (structure + attribute values), connected by a decoration operation that supplies inherited attributes.

Core Objects

The Silver Object Inventory

Nonterminal A type representing a syntactic category. Like a Haskell data type or abstract class. Declared with nonterminal Expr with pp, env, errors;. The with clause declares which attributes occur on it. Production A constructor for a nonterminal. Contains equations for attributes. Four kinds: abstract (data constructors), concrete (parsing rules), aspect (add equations to existing productions), default (fallback equations for a nonterminal). Attribute A named, typed property. Two core kinds: synthesized (bottom-up) and inherited (top-down). Declared separately from where they occur. Occurs-on Declaration that an attribute exists on a nonterminal: attribute pp :: String occurs on Expr;. Separates attribute definition from the types it applies to. Equation Defines an attribute value inside a production: e.pp = "(" ++ l.pp ++ " + " ++ r.pp ++ ")";. Syn equations define the LHS nonterminal's attributes; inh equations define children's attributes. Undecorated Tree The raw tree structure — a production applied to its children. Space-efficient. No attribute values computed yet. Type: Expr. Decorated Tree An undecorated tree + inherited attributes → all attribute values available. Type: Decorated Expr with {env}. A tree can be decorated multiple times with different inherited attributes, yielding different decorated trees. Decoration The operation of supplying inherited attributes to an undecorated tree to produce a decorated tree. Happens implicitly for production children or explicitly via decorate expr with { ... }. Forwarding A production specifies a semantically equivalent tree to provide default attribute values: forwards to not(lt(l,r));. Undefined attributes are looked up on the forwarded-to tree. Essential for extensibility. Collection Attribute An attribute assembled from contributions. Declared with a combining operator: synthesized attribute errors :: [Message] with ++;. Contributions accumulate via the operator. Autocopy Attribute An inherited attribute automatically propagated from parent to children via propagate env;. Eliminates boilerplate copy-down equations. Higher-Order Attribute An attribute whose value is an undecorated tree (a transformation). Used for AST transformations, translation to lower-level IRs. Reference Attribute An attribute whose value is a decorated tree. Used for connecting uses to declarations (e.g., variable resolution). The reference points to a specific decoration context. Threaded Attribute A pair of inherited + synthesized attributes that thread state sequentially through children: threaded attribute stIn, stOut :: State;. With propagate, auto-generates chaining: child1.stIn = top.stIn; child2.stIn = child1.stOut; top.stOut = child2.stOut. Monoid Attribute A collection attribute with explicit identity and operator: monoid attribute errors :: [Message] with [], ++;. Propagation uses identity for leaves and combines children's values. Functor Attribute Reconstructs the tree with transformed children: functor attribute host;. Propagation rebuilds the production with each child's functor value. Type is automatically the nonterminal it occurs on. Strategy Attribute Defines tree rewriting strategies combining rules with traversal combinators (all, some, <* sequence, <+ choice). Stratego-inspired term rewriting at the attribute level. Annotation A value supplied at tree construction time (not computed). Like a named constructor parameter: annotation location :: Location;. The parser auto-supplies location. Closed Nonterminal A nonterminal that disallows new attribute occurrences but permits new productions. Used for concrete syntax types where attributes are fixed. Aspect Production Adds equations to an existing production without modifying its definition. Enables modular extension: aspect production plus ... { e.newAttr = ...; }. Flow Type Declares which inherited attributes a synthesized attribute may depend on: flowtype Expr = forward {env}, pp {decorate};. Informs the MWDA about what must be supplied. InhSet A type-level set of inherited attributes. Controls what a Decorated type requires: Decorated Expr with {env, frame}. Supports subset constraints for flexible typing.

The Two-Stage Model

Silver's Undecorated → Decorated Pipeline Stage 1: Construction Stage 2: Decoration production(child1, child2) + {env = ..., frame = ...} | | v v Undecorated Tree — decorate —→ Decorated Tree (pure data, no attributes) (all attributes available) Type: Expr Type: Decorated Expr with {env} Can be decorated multiple Lazily evaluates attributes times with different inh attrs on demand with caching

Forwarding: The Extension Mechanism

How Forwarding Works

An extension production can define equations for some attributes and forward to a host-language tree for the rest:

abstract production gte
e::Expr ::= l::Expr  r::Expr
{
  e.pp = l.pp ++ " >= " ++ r.pp;
  forwards to not(lt(l, r));
  -- e.value, e.type, etc. come
  -- from not(lt(l,r))
}

Why It Matters

The expression problem, solved: Extensions can add new productions (new syntax) to existing nonterminals. The host language defines attributes; extensions forward to host-language trees for those attributes. Multiple independently-developed extensions compose safely.

Silver's Modular Well-Definedness Analysis (MWDA) statically guarantees that any composition of well-defined extensions produces a well-defined whole — no missing equations, no conflicts.

Silver Syntax Example

-- Nonterminal with attributes
nonterminal Expr with pp, value, env, errors;

-- Attribute declarations
synthesized attribute pp :: String;
synthesized attribute value :: Integer;
inherited attribute env :: Map<String Integer>;
synthesized attribute errors :: [Message] with ++;

-- Production with equations
abstract production add
e::Expr ::= l::Expr  r::Expr
{
  e.pp = "(" ++ l.pp ++ " + " ++ r.pp ++ ")";
  e.value = l.value + r.value;
  l.env = e.env;
  r.env = e.env;
  propagate errors;  -- auto-collect from children
}

What is the Same

Shared Foundations

All Three Share These Foundational Ideas

  • Trees as the central data structure. All three operate over tree-structured data — whether ASTs (JastAdd, Silver) or derivation trees / program structure (Cousot).
  • Properties computed over trees. Each system attaches computed values to tree nodes — called "attributes" (JastAdd, Silver) or "abstract properties" (Cousot).
  • Directional information flow. All distinguish between information flowing bottom-up (synthesized / post-state properties) and top-down (inherited / pre-state context).
  • Fixpoint computation for circularity. When definitions are mutually recursive, all three use iterative fixpoint computation to reach a stable solution.
  • Demand-driven / lazy evaluation. Properties are computed only when needed, avoiding unnecessary work.
  • Caching / memoization. Once computed, values are stored to avoid recomputation.
  • Compositionality. Specifications are modular — different aspects/domains can be defined separately and combined.

What is Different

Divergences

Concepts Unique to Each System

Cousot Only

  • Galois connections — the formal bridge between precision levels. Neither AG system has an equivalent mathematical construct.
  • Widening (∇) — an extrapolation operator to force convergence in infinite-height domains. AG systems use simpler fixpoint iteration.
  • Narrowing (Δ) — a refinement operator to recover precision after widening. No AG counterpart.
  • Soundness proof obligation — every abstract domain comes with a proof that it soundly approximates concrete semantics.
  • Hierarchy of abstractions — the idea that different analyses are systematically related by abstraction. AGs define attributes independently without this lattice structure between analyses.
  • Collecting semantics — the "most precise" abstract analysis that still operates point-wise.

JastAdd Only

  • Rewrites — AST node replacement rules that fire on first access. Neither Cousot nor Silver has this (Silver uses forwarding instead).
  • Inter-type declarations — injecting arbitrary Java methods/fields into generated AST classes.
  • Aspect refinement — one aspect overriding methods from another via refine + refined().
  • getParent() — mutable parent pointers on all nodes. Silver's trees are immutable; Cousot's framework doesn't address tree mutability.
  • .jrag vs .jadd distinction — separating declarative (attribute) aspects from imperative (method) aspects.
  • CRAG formalism — Circular Rewritable Reference Attribute Grammars: the unified formalism combining references, circularity, rewrites, NTAs, and collections in one demand-driven framework.

Silver Only

  • Forwarding — extension productions delegate to host-language trees for undefined attributes. Neither Cousot nor JastAdd has this.
  • Undecorated / Decorated type distinction — a first-class type-level distinction between raw trees and attributed trees. JastAdd conflates them; Cousot doesn't have tree types.
  • InhSet types — type-level sets of inherited attributes with subset constraints.
  • Modular Well-Definedness Analysis (MWDA) — static guarantee that independently-developed extensions compose safely.
  • Closed nonterminals — nonterminals open to new productions but closed to new attributes.
  • Autocopy / propagate — automatic inherited attribute copying via propagate.
  • Threaded / monoid / functor / strategy attributes — specialized attribute kinds with auto-propagation semantics that eliminate boilerplate equations.
  • Annotations — values supplied at construction time (not computed), like location. Distinct from attributes.
  • Type classes — Haskell-style type classes with class, instance, deriving.
  • Flow types — declarations of which inherited attributes each synthesized attribute depends on, enabling modular non-circularity checking.
  • Grammar-as-module — Silver "grammars" are modules (like Java packages), not CFGs. Identified by colon-separated paths: edu:umn:cs:melt:exts:ableC.
  • Copper integration — composable parser generator tightly integrated with the AG system.

Equivalent but Implemented Differently

Same Concept, Different Mechanism

1. Fixpoint Computation

Cousot

Iterate transfer functions over abstract lattice. Use widening (∇) to force convergence in finite steps, then narrowing (Δ) to refine. Works on any complete lattice. May need multiple widening points.

General-purpose, mathematically grounded.

JastAdd

Magnusson-Hedin algorithm: mark attribute as circular [bottom]. Iterate the equation from bottom value, re-evaluating all dependent attributes each iteration, until no value changes. Operates on individual attribute instances.

Attribute-level, practical, no widening needed (assumes finite-height domains).

Silver

Silver supports circular attributes but with less emphasis than JastAdd. The evaluation is similar (iterate to fixpoint) but the mechanism is less exposed in the language design.

Supported but not a central design concern.

2. Gathering Distributed Information

Cousot

Collecting semantics: the set of all reachable states at each program point. A single fixpoint computation gathers information from the entire program. The abstract domain determines how collected info is represented.

JastAdd

Collection attributes: coll T A.c() [init] with combine; + N contributes v to M.c();. Any node can contribute to any target node's collection. Uses a tree traversal to gather all contributions.

Silver

Collection attributes: synthesized attribute errors :: [Message] with ++; + propagate errors;. Contributions are combined via the declared operator. propagate auto-collects from children.

3. Handling Missing / Default Definitions

Cousot

If an abstract domain doesn't model a property, that property is implicitly ⊤ (all values possible). The "default" is maximal imprecision — sound but useless for that property.

JastAdd

Default equations on synthesized attributes: syn T A.x() = defaultVal;. Subclasses inherit the default unless they override. Rewrites can also transform nodes to provide missing behavior.

Silver

Forwarding is the primary mechanism. Extension productions delegate to host-language trees for any attributes they don't define. Also supports default productions providing fallback equations for an entire nonterminal.

4. Modularity and Composition

Cousot

Reduced product: combine two abstract domains into one that is at least as precise as either alone. Tensor product: track correlations between domains. Formal operations with soundness guarantees via Galois connections.

JastAdd

Aspects are the modularity unit. Multiple .jrag/.jadd files are woven together at code-generation time. No formal composition guarantee — missing equations are compile-time errors in generated Java.

Silver

Grammar modules with imports. Extensions can be independently developed and composed. MWDA provides a formal guarantee: if each extension is well-defined on its own, their composition is well-defined too.

5. Tree Transformation

Cousot

Not directly applicable. Abstract interpretation is about analyzing a fixed program, not transforming it. However, abstract-interpretation-guided transformations exist (e.g., partial evaluation, constant folding via AI results).

JastAdd

Rewrites: replace AST nodes in-place. NTAs: synthesize entirely new subtrees. Both happen lazily on first access. Rewrites are destructive (mutate the tree).

Silver

Higher-order attributes: synthesize new undecorated trees as attribute values. Forwarding: construct an equivalent tree without modifying the original. Both are non-destructive (original tree unchanged).

Deep Dives: Key Concepts Explored

Architecture

Galois Connections: The Heart of Abstract Interpretation

A Galois connection is a pair of functions (α, γ) between two lattices (C, ≤) and (A, &sqle;) satisfying:

α(c) &sqle; a ⇔ c ≤ γ(a) for all c ∈ C, a ∈ A Equivalently: c ≤ γ(α(c)) "abstracting then concretizing gives a superset" α(γ(a)) &sqle; a "concretizing then abstracting gives at most a"

Why it matters: The Galois connection guarantees that α produces the best possible approximation — the most precise abstract value that is still sound. This is what distinguishes abstract interpretation from ad-hoc approximation.

Example — Interval domain:

Concrete: {3, 5, 7, 11} — α —→ Abstract: [3, 11] Concrete: {1, 2, ..., 100} — α —→ Abstract: [1, 100] Abstract: [3, 11] — γ —→ Concrete: {3, 4, 5, ..., 11}

AG parallel: JastAdd and Silver don't have Galois connections because they compute exact attribute values, not approximations. There's no "precision vs. computability" trade-off in standard AG evaluation. However, when an AG computes a type for a program, that type system can be viewed as an abstract interpretation with a Galois connection between concrete values and types.

Silver's Decorated/Undecorated Distinction

Silver makes the distinction between "tree structure" and "tree with computed values" explicit at the type level:

Expr undecorated — just data Decorated Expr decorated with ALL known inh attrs Decorated Expr with {env} decorated with specific inh attrs Decorated Expr with {env, frame} decorated with more context

Key insight: The same undecorated tree can be decorated multiple times with different inherited attributes, yielding different attribute values each time. This models the reality that an expression's type depends on its context (environment).

JastAdd's approach: No type-level distinction. AST nodes are always "potentially decorated" Java objects. Attribute values are cached in mutable fields. A node can only have one set of attribute values (one evaluation context) because it has mutable parent pointers that fix its position.

Cousot's parallel: Closest analog is the distinction between program syntax (static) and an annotated program (with abstract values at each point). Different abstract domains give different annotations of the same program — analogous to different decorations.

JastAdd Rewrites vs. Silver Forwarding

Both address the problem of providing default behavior for new syntax, but they do it in fundamentally different ways:

Rewrites

Mechanism: When a node is first accessed, conditional rewrite rules fire. The node is replaced in the tree with the rewritten node. The original is gone.

rewrite ForEach {
  when (shouldDesugar())
  to WhileLoop {
    return new WhileLoop(
      getCond(), getBody());
  }
}

Nature: Destructive, in-place, eager on first access. Iterative (rewrites re-fire until stable).

Forwarding

Mechanism: A production declares a semantically equivalent tree. Attributes not explicitly defined on the production are looked up on the forwarded-to tree. The original production remains.

abstract production forEach
e::Stmt ::= v::Name l::Expr b::Stmt
{
  e.pp = "for " ++ v.pp ++ " in " ++
         l.pp ++ " " ++ b.pp;
  forwards to whileLoop(cond, body);
}

Nature: Non-destructive, lazy, transparent. Original tree preserved; forwarded tree computed on demand.

Collection Attributes: Three Approaches

Cousot

The collecting semantics gathers the set of all reachable states at each program point. This is the "maximal collection" — every fact from every execution path. It's a single global fixpoint.

Individual analyses (interval, pointer, etc.) are abstractions of this collecting semantics — they "collect" only what their domain can represent.

JastAdd

// Declaration
coll Set<String> Program.allNames()
  [new HashSet<>()] with add;

// Contributions (anywhere in tree)
VarDecl contributes getName()
  to Program.allNames();

FunDecl contributes getName()
  to Program.allNames();

Traverses the entire subtree to gather all matching contributions into the collection.

Silver

-- Declaration with operator
synthesized attribute
  errors :: [Message] with ++;

-- In a production (base value):
e.errors := l.errors ++ r.errors;
-- := means "base equation"

-- In an aspect (contribution):
e.errors <- [mkError(...)];
-- <- means "contribute"

-- Auto-collect from children:
propagate errors;

Uses := for base equations, <- for contributions, and with on the declaration for the combining operator.

Modular Well-Definedness Analysis

Silver's signature contribution to the field. The key problem: if Alice writes extension A and Bob writes extension B, can they be composed without conflicts?

The Composition Problem Host Language (defines Expr with value, type, pp) | +--- Extension A (adds forEach production) | defines: pp for forEach | forwards to: whileLoop(...) for value, type | +--- Extension B (adds attribute "complexity") defines: complexity for all host productions What about forEach? It's from Extension A! Answer: forwarding handles it. forEach.complexity is looked up on whileLoop(...).complexity which Extension B already defined (it's a host production).

MWDA guarantee: If each extension is "well-defined" according to Silver's rules (all new attributes have equations on all host productions, all new productions forward to host trees), then any combination of extensions produces a well-defined whole. Checked at compile time, per-extension.

No AG analog in JastAdd: JastAdd relies on the Java compiler to catch missing equations (as abstract methods). No compositional guarantee.

Cousot parallel: The soundness guarantee of Galois connections serves a similar role — ensuring that abstract domains compose correctly (via reduced product). But it's a proof obligation, not a static analysis.

Meta-Comparison: What Each System Optimizes For

Philosophy

Cousot Optimizes for...

Mathematical rigor. Every component has a formal definition. Soundness is proved, not tested. The framework tells you what questions you can and cannot answer for a given level of abstraction.

Generality. The same framework applies to imperative programs, logic programs, grammar analysis, parsing, type systems, model checking. "Types as abstract interpretations" unifies type theory and static analysis.

Trade-off: Not an engineering tool. You need substantial mathematical sophistication to design a new abstract domain. No "just run it" experience.

JastAdd Optimizes for...

Practical compiler construction. Built for real compilers (ExtendJ for Java, JModelica). Rich Java interop. Rewrites enable desugaring. NTAs enable higher-order transformations. Collection attributes handle name binding and error reporting.

Incrementality and performance. Generated Java code with fine-grained caching. Support for incremental re-evaluation in IDEs. Mutable tree model enables efficient in-place updates.

Trade-off: Tight coupling to Java. No formal composition guarantee. Mutable state makes reasoning about evaluation order subtle.

Silver Optimizes for...

Reliable extensibility. The entire design serves one goal: independently-developed language extensions that compose safely. Forwarding, MWDA, and the decorated/undecorated distinction all serve this purpose.

Clean semantics. Immutable trees, explicit decoration, type-level tracking of inherited attribute requirements. The functional flavor (Haskell-like syntax, algebraic data types) encourages equational reasoning.

Trade-off: Less mature tooling than JastAdd. Self-hosting adds complexity. The extension-focused design may be overkill for non-extensible languages.

The Fundamental Triangle

RIGOR (Cousot) /\ / \ / \ / \ / sound \ / approx. \ / theory \ /________________\ ENGINEERING EXTENSIBILITY (JastAdd) (Silver) fast, practical, safe composition, Java-integrated, forwarding, mutable, rewrites MWDA, immutable Each system lives at a different vertex. Cousot provides the theory that JastAdd and Silver implement (partially). JastAdd and Silver are engineering tools that approximate the rigor Cousot demands.

Flashcard Quiz

Test Yourself
What is the "weak interpreter" in Cousot's framework?
Click to reveal
The abstract interpreter — a systematically weakened version of the concrete interpreter that operates on an abstract domain instead of concrete values. It's "weak" because it trades precision for computability: it can answer questions about all possible executions but may produce false alarms. The weakening is formalized by a Galois connection between concrete and abstract domains.
What is the JastAdd equivalent of Silver's "undecorated tree"?
Click to reveal
JastAdd doesn't have a clean equivalent. The closest concept is a freshly constructed AST node before any attributes have been accessed. But JastAdd nodes are always mutable Java objects that can hold cached attribute values, so there's no type-level distinction between "has attributes" and "doesn't have attributes" like Silver's Expr vs Decorated Expr.
What is the Cousot equivalent of a JastAdd circular attribute?
Click to reveal
Fixpoint iteration in an abstract domain, possibly with widening. Both compute iteratively from a bottom value until stability. The difference: Cousot's widening can force convergence in infinite-height lattices (at the cost of precision), while JastAdd's circular attributes assume finite-height domains where simple iteration always terminates.
Why does Silver need forwarding but JastAdd doesn't?
Click to reveal
Silver optimizes for modular extensibility — independently-developed extensions must compose without coordination. Forwarding lets extension productions delegate to host-language trees for attributes they don't know about. JastAdd doesn't need this because it's designed for monolithic compilation — all aspects are available at code-generation time, so missing equations are caught by the Java compiler.
What do Galois connections, MWDA, and JastAdd's Java compiler all have in common?
Click to reveal
They all provide soundness/completeness guarantees for their respective systems. Galois connections guarantee that abstract interpretations are sound (never miss real behaviors). MWDA guarantees that Silver extensions compose without missing equations. JastAdd's Java compiler catches missing equations as compile errors. They're all "safety nets" — but at different levels of formality and with different guarantees.
Name three concepts that exist in JastAdd but not in Silver (or vice versa).
Click to reveal
JastAdd only: (1) Rewrites (in-place node replacement), (2) Inter-type declarations (injecting arbitrary Java into AST classes), (3) Aspect refinement (refine/refined()).

Silver only: (1) Forwarding (delegation to equivalent trees), (2) Decorated/undecorated type distinction, (3) MWDA (modular well-definedness analysis).
What is the Cousot term for what JastAdd calls a "collection attribute"?
Click to reveal
The closest Cousot concept is the collecting semantics — gathering all reachable states (or all relevant facts) at a program point. A JastAdd collection attribute gathers contributions from across the tree into a single value, which is conceptually similar to how collecting semantics gathers all states that can reach a given point. Both involve aggregation from distributed sources.
In what way is Silver's "decorate" operation analogous to Cousot's abstraction function α?
Click to reveal
Both transform a "raw" input into a "processed" form. Decoration takes an undecorated tree + inherited attributes and produces a decorated tree (with all attribute values). Abstraction (α) takes a concrete property and produces an abstract property. Both are lossy in a sense: decoration fixes one set of inherited attributes (losing the generality of other contexts), while abstraction fixes one level of precision (losing concrete detail). But the analogy is loose — decoration computes exact values, while α computes approximations.
What is Cousot's "protoderivation semantics" and how does it relate to attribute grammars?
Click to reveal
The protoderivation semantics is a fixpoint semantics for context-free grammars that models all possible derivation steps as transitions of a pushdown automaton. It is the "most concrete" grammar semantics. By successive abstractions, it yields: derivational semantics → equational language semantics → grammar flow analysis (FIRST/FOLLOW) → parsing algorithms. This hierarchy shows that classical grammar algorithms are all "weak interpreters" of the same underlying grammar semantics — each operating at a different level of abstraction.
