01The Arc: 2015 – 2025

A decade of iteration, pivots, and hard-won lessons. Click each event for details.

2015
Inception: Semantic Diffing in Swift, Then Haskell
A small team at GitHub builds a semantic diff engine. Swift prototype ported to Haskell in less than a day -- 2x faster.
The original goal was structural diffing: understanding what changed in code at a semantic level, not just line-by-line. The Swift prototype (Swift was still in beta) used an O(n) comparison for fixed-arity tree nodes, O(n²) for variable-arity nodes. Porting to Haskell was the team's first production use of the language. The Haskell version was immediately faster and more concise, beginning a long relationship with functional programming in production.
2016
Data Types a la Carte Adopted
The team adopts Swierstra's technique to share syntax types across languages via open unions of functors.
The insight: constructs like IfStatement, FunctionDeclaration, and Literal appear in many languages. Rather than one AST type per language, define small, reusable syntax "functors" and compose them via coproducts. A language's syntax is the open union of the functors it uses. This would later be identified as the root architectural problem.
2017
First Production Feature: Table of Contents
The "table of contents" feature ships on GitHub.com, powered by a Go RPC server calling the semantic binary.
A Go-based RPC server called semiotic executed the Haskell semantic binary out-of-band, returning results via Protocol Buffers. This was the first Semantic-powered feature visible to GitHub users -- an outline of definitions in a source file.
2018
Pivot: Diff to Code Navigation
Semantic diffs abandoned for non-technical reasons. Team pivots to code navigation and comprehension.
The diff feature was dropped because users expected git diff output -- having structurally different diffs on GitHub.com was "deemed an insurmountable barrier." The team built semanticd, an HTTP service using the Haskell Servant web library that wrote tag information to MySQL on repository push. Diff summaries also proved intractable: "no formulation substantially improved on reading a textual diff."
2019 (June)
Open-Sourced
The project is open-sourced, generating significant discussion on Hacker News and in the Haskell community.
At this point the codebase was ~20,000 lines of Haskell handling tens of thousands of requests per minute with thousands of simultaneous connections. The team published "Why Haskell" and "Why tree-sitter" design documents explaining their technical choices.
2020 (August)
CodeGen Pivot: Abandoning a la Carte
The team replaces hand-written Assignment grammars with auto-generated per-language types from tree-sitter's node-types.json.
The a la carte approach had become a maintenance crisis. Assignment grammars broke at runtime when tree-sitter grammars changed. The team contributed the field() function to tree-sitter's grammar API, enabling child nodes to be retrieved by semantic name. Template Haskell then generated strongly-typed, per-language Haskell datatypes from the schema. Java and CodeQL were the first languages shipped using the new system.
2021 (October)
Stack Graphs Replace Semantic
A Rust-based successor, stack graphs, begins replacing the Haskell code navigation pipeline. Presented at Strange Loop.
Stack graphs analyze files in isolation, producing subgraphs of definitions and references. At query time, subgraphs merge and path-finding with a "symbol stack" resolves name bindings. Language rules are written in a declarative DSL on tree-sitter, not in Haskell. The team acknowledged needing "to enable external contributors to maintain their own code navigation rules without having to write Haskell code."
2022 (September)
ICFP Retrospective Paper
"Fusing Industry and Academia at GitHub" -- the team's candid post-mortem. Semantic was "no longer part of the code navigation pipeline."
Published by Patrick Thomson, Rob Rix, Nicolas Wu, and Tom Schrijvers at ICFP 2022. Despite the project's end, the authors considered it "a successful application of FP techniques, which allowed us to iterate quickly on a solution, draw from well-researched avenues of thought, and express complex thoughts concisely."
2025 (April)
Repository Archived
"This repository is no longer supported or updated by GitHub." Stack graphs also archived 5 months later.
The repository was officially archived on April 1, 2025. The README's archival notice was the final update. Stack graphs (the Rust successor) was itself archived in September 2025, suggesting GitHub moved to yet another approach internally or integrated the technology differently.

02The Vision

What was GitHub Semantic actually trying to do?

Semantic was an attempt to solve the m × n problem in code analysis: you want m analysis features (semantic diffing, code navigation, type checking, dead code detection) across n languages (Python, Ruby, JavaScript, Go, TypeScript, Java, PHP...). Without sharing, you write m × n implementations. With a universal representation, you write m + n.

Without sharing (m × n):
Python Diff Python Nav Python Types
Ruby Diff Ruby Nav Ruby Types
JS Diff JS Nav JS Types
With universal representation (m + n):
Python parser Ruby parser JS parser → Universal AST → Diff Nav Types

The m + n approach has an obvious appeal. A single diff algorithm works across all languages. A single code navigation engine handles every repository. The team pursued this vision with deep Haskell type-level programming, algebraic effects, abstracting definitional interpreters, and recursion schemes.

What made it credible: The team was staffed with world-class Haskell programmers, had academic collaborators (Imperial College London, KU Leuven), and published at top venues (ICFP). The Haskell codebase handled production traffic at GitHub scale -- tens of thousands of requests per minute.

03Data Types a la Carte

The type-level technique at the heart of Phase 1 -- and the root of its problems.

The Idea

Based on Wouter Swierstra's 2008 paper: instead of one monolithic AST type per language, define small reusable syntax functors, each representing one kind of node. A language's syntax is the coproduct (open union) of the functors it uses.

Each language is defined as a union of reusable syntax pieces. If Python and Ruby both have IfStatement, they share that exact functor. Language-specific constructs get their own unique functors.

-- Shared functors (used by many languages)
data IfStatement a = IfStatement a a (Maybe a)   -- cond, then, else
data FuncDecl a   = FuncDecl a [a] a            -- name, params, body
data Literal a    = TextLit Text | IntLit Int   -- literals

-- Language-specific functors
data RubyZSuper a = RubyZSuper                  -- Ruby-only
data PythonWith a = PythonWith a a              -- Python-only

-- A language = coproduct (open union) of its functors
type PythonSyntax = Sum '[IfStatement, FuncDecl, Literal, PythonWith, ...]
type RubySyntax   = Sum '[IfStatement, FuncDecl, Literal, RubyZSuper, ...]

The two fixed points: one for the tree, one for diffs. Annotations tracked source locations and diff status.

-- Each functor is parameterized by an annotation type
-- The tree is a fixed-point of the coproduct of functors
type Term syntax ann = Fix (syntax :+: Annotation ann)

-- Diff is a separate fixed-point
type Diff syntax ann = Free
  (Patch (Term syntax ann))  -- insert, delete, replace
  (Term syntax ann)           -- copy (unchanged)

-- The constraint: a functor f is "in" the union g
class (f :<: g) where
  inject  :: f a -> g a
  project :: g a -> Maybe (f a)

JSON's syntax, composed entirely from reusable Literal functors:

type JSONSyntax = Sum
  '[ Literal.Null
   , Literal.List
   , Literal.Boolean
   , Literal.Hash
   , Literal.Decimal
   , Literal.KeyValue
   , Literal.TextElement
   , Syntax.Error
   ]

-- 8 functors. Simple. Reusable.
-- But JSON is the easy case...

The Scaling Problem: fastsum

With 100+ syntax constructors possible across all languages, the standard Haskell typeclass resolution for the f :<: g constraint became catastrophically slow -- recursive type-level search through the union. Patrick Thomson built the fastsum library (with Rob Rix's ElemIndex type family) to solve this: a closed type family that resolves membership in constant time.

A hint of problems ahead: When you need to build a custom library just to make your type-level machinery compile in reasonable time, the abstraction may be fighting the problem rather than solving it.

Source Code

Any supported language file. At its peak, Semantic handled Python, Ruby, JavaScript, TypeScript, Go, Java, PHP, and Haskell. JSON was also supported as a simpler test case.

tree-sitter Parsing

tree-sitter produces an untyped concrete syntax tree (CST) with positional access to children. The CST preserves all syntactic details (comments, whitespace), but children are accessed by index, not by semantic name. This meant the next stage had to know the exact child ordering of every grammar production.

Assignment Grammars (The Bottleneck)

A hand-written Haskell parser per language that maps tree-sitter's untyped CST nodes to typed a la carte syntax functors. This was, in the team's description, "a language-specific anamorphism mapping a tree-sitter tree to a coproduct of functors."

In practice, it was a second grammar per language, tightly coupled to tree-sitter's output format. Every tree-sitter grammar update required a corresponding Assignment update -- but failures were only detectable at runtime.

The fundamental failure point: Comments and heredocs could appear at arbitrary positions in the CST, causing runtime crashes that were invisible at compile time. "Any update in a language grammar required modifications" to the assignment code.

Typed AST

A tree of a la carte syntax functors, each annotated with source locations. The tree is the fixed-point of the coproduct of all functors used by the language. Analysis code (diffing, evaluation) operates on this representation using recursion schemes and typeclass dispatch.

Analysis

Multiple analysis features operated on the typed AST:

  • • Diffing: RWS-Diff algorithm comparing two typed ASTs
  • • Code navigation: Extracting definitions and references
  • • Abstract interpretation: Based on Abstracting Definitional Interpreters (Darais et al. 2017)
  • • Table of contents: File-level outlines of definitions

05What Broke

Five categories of failure, from technical to organizational.

1 The Assignment Maintenance Burden ▼

Every supported language required a hand-written Assignment grammar in Haskell that exactly mirrored the tree-sitter grammar's structure. This meant:

  • ✗ Two grammars per language (tree-sitter + Assignment)
  • ✗ Tight coupling to tree-sitter's child ordering
  • ✗ Runtime failures on grammar updates, no compile-time safety
  • ✗ "Inadvertently became incentivized against iterative improvement" -- improving a grammar risked breaking the Assignment layer
2 Insufficient Type Safety ▼

Despite using Haskell's powerful type system, the a la carte representation was "essentially untyped" at the structural level. A function's parameter list could contain any syntax node, not just parameters. The type system enforced that nodes were valid syntax kinds, but not that they appeared in valid positions.

Many Evaluatable typeclass instances had to handle error conditions that couldn't actually occur in practice -- the types were too permissive to rule them out.

3 Language-Specific Variants Undermined Sharing ▼

The promise of a la carte was reuse: IfStatement shared across Python, Ruby, JavaScript. But languages have unique semantic quirks:

  • • Ruby's zsuper keyword -- calls the parent method with the same arguments implicitly
  • • Python's with statement -- context managers with no equivalent elsewhere
  • • Go's goroutines, JavaScript's async/await semantics

These required language-specific functors, and as languages grew in number, the "shared" portion of the coproduct shrank relative to the language-specific portion.

4 Semantic Diffs Were Unwanted ▼

The project's original raison d'être was abandoned for a non-technical reason: users expected git diff output. Having structurally different diffs on GitHub.com was "deemed an insurmountable barrier."

Diff summaries (high-level descriptions of what changed) also failed: "no formulation substantially improved on reading a textual diff." The most ambitious feature was killed by UX expectations, not technical limitations.

5 Haskell Was a Contributor Barrier ▼

Despite being excellent for the core team, Haskell was unsuitable for external language maintainers. The team needed the community to add and maintain language support, but "leveraging community support for adding languages has been difficult" because of Haskell's "considerable ecosystem, large compiler, and learning curve."

This drove the eventual move to stack graphs, where language rules are written in a declarative DSL rather than Haskell.

"The Assignment stage proved a significant maintenance burden: any update in a language grammar required modifications. The a la carte datatypes were essentially untyped; enforcing only a minimal structure on the tree."
-- GitHub Engineering, CodeGen announcement (2020)

06The CodeGen Pivot

How the team replaced hand-written grammars with auto-generated per-language types.

⚠ Phase 1: A La Carte

  • Hand-written Assignment grammars per language
  • Shared syntax functors across languages
  • Runtime failures on grammar changes
  • Children accessed by position index
  • Weeks of effort per new language
  • Community contributions nearly impossible

✓ Phase 2: CodeGen

  • Auto-generated types from node-types.json
  • Per-language types (not shared)
  • Compile-time errors on grammar changes
  • Children accessed by named fields
  • Largely automated per new language
  • Contribution barrier significantly lowered

The Four-Step Pipeline

1 Named Fields in tree-sitter ▼
The team contributed the field() function to tree-sitter's grammar API. Before this, tree-sitter children were accessible only by index. After: children could be retrieved by semantic name (condition, body, alternative) rather than position. This was a prerequisite for reliable automated unmarshaling.
2 node-types.json Schema ▼
tree-sitter grammars now produce a JSON schema file describing four node shapes: sums (union types), products (record types), named leaves (identifiers, literals), and anonymous leaves (operators, punctuation). This file is the single source of truth for all AST types.
3 Template Haskell Code Generation ▼
The astDeclarationsForLanguage function uses Template Haskell (compile-time metaprogramming) to read the JSON schema and generate strongly-typed, per-language Haskell datatypes. A function's name is typed as an identifier, a switch's children as case statements. No more "anything can be anywhere."
4 Generic Unmarshaling ▼
Using GHC.Generics, the system automatically constructs typed ASTs from tree-sitter's cursor API. The parseByteString function accepts source code and a language spec, producing a complete annotated AST. Non-generic instances handle special cases like Maybe, [], and NonEmpty.
The key reversal: Phase 1 tried to share syntax types across languages. Phase 2 generated separate types per language. The team discovered that per-language types generated from a schema were more practical than hand-crafted shared abstractions. Sharing moved from the data representation to the processing infrastructure.

07The Effects System Journey

One of the project's most significant technical contributions -- the fused-effects library.

The team went through three effect system architectures, each solving problems the previous one couldn't:

Phase 1
MTL (Monad Transformers)
Classic approach. Failed due to the O(n²) instances problem.
Phase 2
Freer Monads
Solved n² problem. Failed on scoped effects.
Phase 3
Fused-Effects
Solved both problems. Open-sourced, widely adopted.

MTL: The n² Problem

MTL requires a separate monad transformer per effect, and each transformer must implement an instance for every other effect. With n effects, you need n² instances. For Semantic's numerous effects (filesystem, telemetry, concurrency, parsing, etc.), this was infeasible.

Freer Monads: The Scoped Effects Problem

A freer monad approach (based on Dev and King, 2016) solved n² by encoding effects as data rather than type class instances. But Wu et al. (2014) identified a critical limitation: operations like local (modifying an environment for a scoped computation) produce "surprising, and often undesired behavior" when combined with other effects.

Fused-Effects: The Solution

Based on Schrijvers et al. (2019), fused-effects eliminated the freer monad and used type class methods for effect handlers, supporting scoped effects with multiple interpretations. A concrete example: a Telemetry effect had three handlers -- production (aggregator upload), development (CLI reporting), and testing (discard) -- without altering application code. The library was open-sourced, achieved 10,000+ downloads, and is still actively used outside GitHub.

Key takeaway: The effects system was arguably the project's most lasting contribution. While the universal AST failed, the effect handling pattern produced a library that outlived the project itself.

09Lessons Learned

What the decade-long arc of GitHub Semantic teaches about universal ASTs.

Share Infrastructure, Not Representation

The Phase 1 approach shared the data representation (syntax functors) across languages. Phase 2 shared the infrastructure (Template Haskell code generation, generic unmarshaling, cursor API) while keeping data representation per-language. This mirrors the broader industry lesson: CodeQL shares the query language, Infer shares the analysis framework, GraalVM shares the compiler -- none of them share the AST.

Generate, Don't Abstract

The team's own words: they prioritized "compile-time safety over runtime flexibility." Rather than clever type-level Haskell to abstract over language differences, mechanically generating per-language types from node-types.json was more maintainable, more type-safe, and easier to update when grammars changed. The schema is the single source of truth, not the Haskell type hierarchy.

Contributor Accessibility Matters

Haskell's power was the team's advantage and the project's limitation. Adding a new language required understanding monad transformers, recursion schemes, open unions, and Template Haskell -- a vanishingly small contributor pool. Stack graphs replaced this with a declarative DSL that language experts (not Haskell experts) could write.

User Expectations Trump Technical Elegance

The semantic diff engine -- the project's founding feature -- was abandoned because users expected git diff. Diff summaries "substantially improved" on nothing. The table-of-contents feature (much simpler) is what actually shipped and stayed. Sometimes the less ambitious feature is the right one.

The 65/35 Split Is Real

YASA's research quantifies what Semantic experienced qualitatively: about 65% of syntax constructs can be universalized, but ~35% resist. Ruby's zsuper, Python's with, Go's goroutines -- these don't map to shared functors without semantic loss. Any universal AST design must explicitly account for this irreducible language-specific fraction rather than treating it as a corner case.

Extract Reusable Libraries

fused-effects, fastsum, and the tree-sitter field() contribution all outlived the project. The team's investment in solving hard problems (algebraic effects, open union performance, parser API design) produced reusable artifacts even when the application they were built for didn't survive. Good engineering leaves a residue of useful components.

"A successful application of FP techniques, which allowed us to iterate quickly on a solution, draw from well-researched avenues of thought, and express complex thoughts concisely, all the while retooling our approach in the face of evolving business requirements and use cases."
-- Thomson, Rix, Wu, Schrijvers, ICFP 2022

They contributed the field() function to tree-sitter's grammar DSL, enabling child nodes to be retrieved by semantic name (e.g., condition, body) rather than by positional index.
