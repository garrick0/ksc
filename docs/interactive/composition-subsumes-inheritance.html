Composition ⊇ Inheritance

01The Core Question

Is there anything inheritance can express that composition cannot?

The "favor composition over inheritance" design principle is one of the most repeated in software engineering. But is it merely good advice — or is there a mathematical proof that composition is strictly more powerful?

The answer: yes, composition subsumes inheritance, but with a precise nuance. The key concept that makes the proof work is open recursion — the one thing inheritance gives you "for free" that naive composition does not.

Key insight: Composition without delegation is almost as powerful as inheritance — but it misses open recursion. Adding explicit self-passing (delegation) closes the gap entirely. Since delegation is a natural part of the composition toolkit, composition strictly subsumes inheritance.

02The Three Pillars of Inheritance

Inheritance provides exactly three things. Composition provides all three — plus more.

Pillar 1: Code Reuse

A subclass inherits all methods of its parent. With composition, we achieve the same via delegation — forwarding calls to an inner object.

Inheritance
class Base {
  greet() { return "hello" }
  shout() { return "HEY!" }
}

class Child extends Base {
  // greet() and shout() inherited free
  wave() { return "*waves*" }
}
Composition
class Inner {
  greet() { return "hello" }
  shout() { return "HEY!" }
}

class Outer {
  inner = new Inner()
  greet() { return this.inner.greet() }
  shout() { return this.inner.shout() }
  wave()  { return "*waves*" }
}

The composition version is more verbose (it forwards each method), but it works identically. Languages like Kotlin (by keyword) and Go (embedding) automate this forwarding.

Pillar 2: Subtype Polymorphism

Inheritance creates an is-a relationship: a Child can be used wherever a Base is expected. With composition, we achieve the same via interface conformance.

Inheritance
class Animal {
  speak(): string { ... }
}
class Dog extends Animal { ... }

// Dog IS-A Animal
function pet(a: Animal) { ... }
pet(new Dog()) // works
Composition
interface Speaker {
  speak(): string
}
class Dog implements Speaker {
  speak() { return "woof" }
}
// Dog CONFORMS-TO Speaker
function pet(s: Speaker) { ... }
pet(new Dog()) // works

Interface-based polymorphism is strictly more flexible than inheritance-based polymorphism: a class can implement multiple interfaces but (in most languages) can only extend one parent. In structurally-typed languages like Go or TypeScript, even the implements declaration is optional.

Pillar 3: Open Recursion (the hard one)

This is the only feature unique to inheritance. When a base class method calls this.method(), it dispatches to the most-derived override — even though the base class was written first.

class Base {
  greet() { return "hello" }
  speak() { return this.greet() + "!" }
  //                 ^^^^ late-bound: resolves to CHILD's greet()
}

class Child extends Base {
  greet() { return "hi" }  // override
}

new Child().speak()  // "hi!" — Base.speak sees Child.greet
This is non-trivial to replicate with composition. With naive composition, the inner object's this points to itself — it cannot see the outer wrapper's overrides. Section 3 shows exactly how delegation recovers this behavior.

What Composition Adds Beyond Inheritance

Runtime Swappability

Replace a composed object at runtime (Strategy pattern). Inheritance hierarchies are fixed at class definition time.

Independent Lifecycles

Composed objects can be created, replaced, or destroyed independently. An inherited base is inseparable from its subclass.

Arbitrary Topologies

Compose objects in DAGs, cycles, or any graph structure. Inheritance is constrained to trees (single) or DAGs with linearization (multiple).

Explicit Dependencies

Composed objects' coupling is visible in the code. Inherited behavior is implicit — you must read the entire hierarchy to understand dispatch.

03The Mathematical Formalization

Cook & Palsberg (1989): Inheritance as a fixed-point construction over generators.

William Cook's denotational semantics of inheritance revealed the precise mathematical structure hiding inside class extends. The key idea: a class is not an object — it is a generator function that takes a self-reference and returns a record of methods.

Classes as Generators

A class is a function G : Self → Record. It takes a reference to "self" (the final object) and produces method implementations:

// A "generator" — takes self, returns methods
const baseGen = (self) => ({
  greet: () => "hello",
  speak: () => self.greet() + "!"
  //           ^^^^ self is a PARAMETER, not yet bound
})

const childGen = (self) => ({
  greet: () => "hi"
  // speak is not redefined — inherited from baseGen
})

Inheritance = Override + Fixed Point

Instantiating a class hierarchy means:

  1. Merge generators with override semantics (child wins on conflict)
  2. Take the fixed point — tie the knot so self refers to the final merged object
object = fix(λself. merge(baseGen(self), childGen(self))) 
// merge: child overrides base (right-biased merge)
const mergedGen = (self) => ({
  ...baseGen(self),     // greet: "hello", speak: self.greet() + "!"
  ...childGen(self)     // greet: "hi" (overrides base)
})

// fix: the Y combinator — tie the recursive knot
const fix = (gen) => {
  const obj = {}
  Object.assign(obj, gen(obj))  // self = obj itself
  return obj
}

const object = fix(mergedGen)
object.speak()  // "hi!" — self.greet() resolves to child's "hi"
The fixed-point is what creates open recursion. By passing the final object as self to every generator (including base), the base generator's self.greet() resolves to the child's override. This is exactly what this does in an inheritance hierarchy.

Composition Without Self — No Open Recursion

With naive composition, each object has its own this. There is no shared self:

const inner = {
  greet: () => "hello",
  speak: () => inner.greet() + "!"
  //           ^^^^^ hardcoded to inner — NOT the outer wrapper
}

const outer = {
  greet: () => "hi",         // "override"
  speak: () => inner.speak()   // delegate
}

outer.speak()  // "hello!" — inner.greet() is called, not outer.greet()

Composition With Delegation — Full Simulation

Pass an explicit self parameter to recover open recursion. This is exactly Cook's generator encoding:

// Generator-style: take self as parameter
const makeInner = (self) => ({
  greet: () => "hello",
  speak: () => self.greet() + "!"
  //           ^^^^ self is the OUTER object
})

// Tie the knot (fixed point)
const outer = {}
Object.assign(outer, {
  ...makeInner(outer),   // base methods, with self = outer
  greet: () => "hi"     // override
})

outer.speak()  // "hi!" — self.greet() resolves to outer.greet()
Key theorem (Cook, 1989): For any class hierarchy using single inheritance, there exists a composition with explicit self-passing (delegation) that produces an observationally equivalent object. The encoding is mechanical: each class becomes a generator, extends becomes merge, new becomes fix.

05Algebraic Structure Comparison

Inheritance forms a partial order. Composition forms a monoid. The monoid is strictly richer.

PropertyInheritanceComposition
Algebraic structure Partial order (preorder on types) Monoid of type transformers
Binary operation N/A (extends is not associative) f ∘ g (function composition)
Combining topology Tree (single) or DAG (multiple) Arbitrary graph
Identity element Object / root type Identity function id(x) = x
Associativity N/A Yes: (f ∘ g) ∘ h = f ∘ (g ∘ h)
Commutativity No No (order matters)
Idempotent Yes: A extends A ≡ A No: f ∘ f ≠ f generally
Inverse No (cannot "un-extend") Sometimes (if transformer is bijective)

Inheritance: Partial Order (Lattice)

Composition: Free Monoid

Why the monoid subsumes the partial order: A partial order is a special case of a category with at most one morphism between any two objects. A monoid (endomorphism monoid) allows multiple morphisms and free composition. Every inheritance chain A → B → C can be encoded as a composition of transformers, but the reverse is not true — transformers can form structures (cycles, DAGs, runtime-varying chains) that no inheritance hierarchy can express.

06Category Theory Perspective

Types form a category. Inheritance and composition are different constructions within it.

The Category Type

Objects are types. Morphisms are functions (or subtype coercions). Composition of morphisms is function composition. The identity morphism on each type is the identity function.

Inheritance as Subtype Morphisms

If Dog <: Animal, there exists a unique morphism (coercion) from Dog to Animal in the subtype preorder category. This category has at most one morphism between any two objects.

A thin category (preorder): at most one arrow between any pair. Very constrained.

Composition as Products

Given types A and B, their product A × B comes with projections π1 and π2. Composition builds products freely — no subtype relationship required.

Product types compose any two types. No hierarchy needed. The universal property guarantees uniqueness.

Inheritance as Coalgebra, Composition as Algebra

There is a deeper categorical view. An object with methods is a coalgebra for the functor F(X) = Method1(X) × ... × Methodn(X). Inheritance is a coalgebra homomorphism. Composition is an F-algebra construction that builds new coalgebras from existing ones:

ConceptCategorical View
Object with methodsF-coalgebra X → F(X)
InheritanceCoalgebra homomorphism (structure-preserving)
Open recursionGreatest fixed point of F (final coalgebra)
CompositionProduct of coalgebras + arbitrary rewiring
Generator (Cook)Endofunctor G : Type → Type
Class hierarchyComposition of endofunctors + fixed point
The punchline: Inheritance is a special case of the endofunctor composition + fixed-point pattern. Composition encompasses the entire space of endofunctors, while inheritance is restricted to those that arise from subtype coercions. Categorically: the category of endofunctors is strictly richer than the subtype preorder.

07The Simulation Theorems

Formal proofs that composition subsumes inheritance, but not vice versa.

Theorem 1: Composition + Delegation ⊇ Inheritance

1. Let H be any single-inheritance hierarchy with classes C1 ≺ C2 ≺ ... ≺ Cn
2. For each Ci, define generator Gi : Self → Record containing only Ci's own methods (Cook encoding)
3. Define merged = λself. { ...G1(self), ...G2(self), ..., ...Gn(self) } (right-biased merge = override)
4. Let obj = fix(merged) (Y combinator)
5. Then obj is observationally equivalent to new Cn() (all method calls return identical results)
6. Since each Gi is a composed component with explicit self-passing, QED

Theorem 2: Inheritance ⊅ Composition

We prove this by exhibiting three counterexamples — patterns expressible with composition but not inheritance:

Counterexample 1: Runtime Strategy Swap

class Sorter {
  strategy: SortStrategy  // composed, can change at runtime

  sort(data) {
    return this.strategy.sort(data)
  }

  useQuickSort() {
    this.strategy = new QuickSort()   // swap at runtime!
  }
  useMergeSort() {
    this.strategy = new MergeSort()   // swap again!
  }
}

With inheritance, Sorter would need to extend one sort algorithm. You cannot change which base class an object extends after construction. Inheritance hierarchies are static; composition graphs are dynamic.

Counterexample 2: Independent Lifecycles

class Editor {
  spellChecker: SpellChecker | null = null
  autoSaver: AutoSaver | null = null

  enableSpellCheck() {
    this.spellChecker = new SpellChecker()  // created on demand
  }
  disableSpellCheck() {
    this.spellChecker?.dispose()
    this.spellChecker = null              // destroyed independently
  }
}

Each composed object has its own lifecycle: created when needed, destroyed when not. An inherited base is born and dies with the subclass — you cannot detach it at runtime.

Counterexample 3: Circular Delegation

class ViewA {
  partner: ViewB
  update() { this.partner.refresh() }
}
class ViewB {
  partner: ViewA
  refresh() { this.partner.update() }
}

// Circular: A delegates to B, B delegates to A
const a = new ViewA()
const b = new ViewB()
a.partner = b; b.partner = a

Inheritance is acyclic by definition (a class cannot be its own ancestor). Composition places no such restriction — mutual delegation is natural for observer patterns, event systems, and bidirectional bindings.

Corollary: Mixins ≈ Linearized Composition

Mixins and traits are the middle ground between inheritance and composition. They are essentially composable generators with automatic linearization (a deterministic override order when there are conflicts).

Smalltalk — The purest inheritance model. Everything is an object. Single inheritance from a linear chain. No interfaces, no mixins. Composition is possible but idiomatic code uses deep hierarchies. self is always late-bound. The original "message passing" vision where inheritance is the primary reuse mechanism.
Java — Single implementation inheritance + multiple interface inheritance. abstract class for partial implementations, interface (with default methods since Java 8) for contracts. The "Gang of Four" book was written in this context — hence "favor composition over inheritance". Still heavily inheritance-oriented in practice (Spring, Servlet API).
Python — Multiple inheritance with C3 linearization (MRO). Mixins are idiomatic (LoggingMixin, SerializableMixin). super() follows the MRO, not the lexical parent. Duck typing means interfaces are implicit. Sits in the middle: inheritance is easy, composition is easy, mixins bridge both.
Scala — Traits with stackable modifications (linearized mixin composition). trait can have abstract and concrete members, composed with with. Self-types (self: OtherTrait =>) enable dependency declaration without inheritance. Companion objects for non-inherited behavior. A sophisticated middle ground.
Kotlin — Classes are final by default (discouraging inheritance). First-class by keyword for delegation: class Wrapper(inner: Base) : Base by inner. This is Cook's delegation pattern built into the language syntax. Interfaces with default methods. Explicitly favors composition.
TypeScript — Structural typing means any object with matching shape satisfies an interface — no implements needed. Intersection types (A & B) are type-level composition. Mixins via generic constructors. Classes exist but interfaces + composed objects are more idiomatic in advanced TS. Structural subtyping > nominal inheritance.
Go — No inheritance at all. Struct embedding provides forwarding (like Kotlin's by). Interfaces are satisfied structurally — no implements keyword. Composition is the only reuse mechanism. This is a deliberate language design choice: "If C++ and Java are about type hierarchies and the taxonomy of types, Go is about composition." — Rob Pike
Rust — No inheritance at all. Traits define shared behavior, implemented explicitly per type. Trait objects (dyn Trait) for dynamic dispatch. Composition via struct fields. Generics with trait bounds for static dispatch. The most principled separation: traits for polymorphism, structs for data, composition for reuse. Zero cost abstractions.

09The Nuance: When Inheritance is More Ergonomic

Subsumption ≠ superiority in all contexts. Inheritance wins on ergonomics for specific patterns.

The mathematical fact that composition subsumes inheritance does not mean inheritance is useless. For certain patterns, inheritance provides the most natural, concise expression:

Template Method

Define algorithm skeleton in base, let subclasses override specific steps. With composition, you'd need to pass explicit strategy functions — more flexible, but more boilerplate for simple cases.

Framework Extension

React class components, Django views, JUnit test cases. The framework provides base behavior; you override hooks. Open recursion makes this natural. Composition equivalent: dependency injection (equally powerful, more explicit).

State Hierarchies

When types genuinely form a hierarchy (AST nodes, UI widgets), inheritance models the domain directly. Composition can express this but may obscure the hierarchical structure.

Free Open Recursion

With inheritance, this is implicitly late-bound everywhere. With composition + delegation, you must explicitly thread self. Same power, but inheritance has less ceremony.

The meta-insight: Composition and inheritance have identical expressive power (once delegation is included). The difference is purely ergonomic: inheritance is more concise for hierarchical extension with open recursion, composition is more flexible for everything else. Modern language design recognizes this — Kotlin and Rust give you composition by default with opt-in mechanisms for the patterns inheritance excels at.

Summary of the Mathematical Argument

1. Inheritance provides three things: code reuse, subtype polymorphism, open recursion
2. Composition provides: code reuse (via delegation), polymorphism (via interfaces), plus runtime flexibility, arbitrary topologies, independent lifecycles
3. Open recursion = shared self reference = fixed point of merged generators (Cook 1989)
4. Composition + explicit self-passing = Cook's generator model ⇒ can simulate any inheritance hierarchy
5. Inheritance cannot simulate: runtime swap, circular delegation, independent lifecycles
6. Therefore: Composition + Delegation ⊋ Inheritance (strict superset)
7. Algebraically: Endofunctor monoid ⊋ Subtype preorder
8. Categorically: End(Type) is strictly richer than Sub(Type)

Q

What are the three things inheritance provides? Which one is unique to inheritance (not trivially available with naive composition)?

A

Code reuse, subtype polymorphism, and open recursion. Open recursion (late binding of this/self) is the one unique feature — naive composition gives each object its own this, so the inner object cannot see the outer's overrides.

Q

In Cook's model, what is a "class"? What operation does extends correspond to?

A

A class is a generator function G: Self → Record that takes a self-reference and returns a record of methods. extends corresponds to right-biased merge of generators: λself. { ...base(self), ...child(self) }. Instantiation (new) takes the fixed point (Y combinator).

Q

Why does naive composition fail to replicate open recursion? What do you add to fix it?

A

Naive composition: the inner object's this points to itself, so inner.speak() calls inner.greet(), not the outer wrapper's override. Fix: explicit self-passing (delegation). Make the inner a generator (self) => { ... } and pass the outer object as self. This is exactly Cook's encoding.

Q

What algebraic structure does inheritance form? What about composition? Why is the latter strictly richer?

A

Inheritance: partial order (preorder) on types — reflexive, transitive, antisymmetric, at most one relationship between any pair. Composition: monoid of type transformers — associative, has identity, allows multiple composed transformers in any order. The monoid admits arbitrary DAGs and runtime variation; the partial order is limited to static trees/DAGs.

Q

Name three things composition can express that inheritance cannot.

A

1) Runtime strategy swap — change composed behavior after construction. 2) Independent lifecycles — create/destroy composed objects independently. 3) Circular delegation — A delegates to B, B delegates to A. Inheritance hierarchies are acyclic by definition; composition has no such restriction.

Q

In category theory, what is the categorical view of an object with methods? How do inheritance and composition differ categorically?

A

An object with methods is an F-coalgebra (X → F(X)). Inheritance is a coalgebra homomorphism in the subtype preorder (a thin category with at most one morphism per pair). Composition is product of coalgebras with arbitrary rewiring in the full endofunctor category, which is strictly richer than the subtype preorder.

Q

What is the relationship between mixins/traits and the composition-vs-inheritance spectrum?

A

Mixins/traits are linearized composition — composable generators with a deterministic override order. They sit between inheritance and composition: Python uses C3 linearization, Scala uses right-to-left trait linearization, Rust uses explicit impl (no linearization needed). They prove that the boundary is a spectrum, not a binary.

Q

If composition is strictly more powerful, why would you ever use inheritance?

A

Ergonomics. The two have identical expressive power (with delegation), but inheritance is more concise for: the Template Method pattern (override hooks in a skeleton), framework extension points, and any case with open recursion (free late-bound this, no explicit self-passing). It's a syntax difference, not a power difference.
