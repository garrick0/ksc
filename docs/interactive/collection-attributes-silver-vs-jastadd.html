Collection Attributes: Silver vs JastAdd

How should @ksc/ag aggregate contributions from distributed tree nodes? Comparing Silver's monoid propagation, JastAdd's two-phase coll/contributes, and our current subtree-walk approach.

1Current @ksc/ag Collection

What we have today: a simple subtree DFS walk

Our coll() function walks the subtree rooted at the evaluated node using a DFS stack, extracts contributions from each descendant via a contribute function, and folds them into an accumulator via combine. It's implemented as a syn() wrapper — the collection is just a synthesized attribute whose equation happens to walk the subtree.

// @ksc/ag — current coll() implementation
export function coll<N, V>(
  init: () => V,
  combine: (acc: V, contribution: V) => V,
  contribute: (node: N) => V | undefined,
): AttributeDef<N, V> {
  return syn((node) => {
    let acc = init();
    const stack = [node];
    while (stack.length > 0) {
      const current = stack.pop()!;
      const c = contribute(current);
      if (c !== undefined) acc = combine(acc, c);
      for (const kid of (current as any).$children ?? [])
        stack.push(kid);
    }
    return acc;
  });
}

What works

  • Simple error collection from subtrees
  • Sum/count aggregation over descendants
  • Any case where the target = the subtree root
  • O(subtree) per evaluation, cached via syn()

What breaks

  • Cannot contribute to a remote node (e.g., module-level errors from a deeply nested expression)
  • Cannot contribute to multiple targets
  • No conditional contribution (when clause)
  • O(subtree) per node — redundant work when parent and child both collect

2Silver's Approach

Monoid attributes with := base values, <- contributions, and propagate

-- Classic collection via 'with' clause
synthesized attribute errors :: [Message] with ++;

-- In a production:
production add
e::Expr ::= l::Expr r::Expr
{
  e.errors := [];          -- base value
  e.errors <- l.errors;   -- contribution from left child
  e.errors <- r.errors;   -- contribution from right child
}

The := operator sets the base value (identity element). The <- operator adds a contribution. Silver combines them using the with operator (++ for list append).

-- Modern: monoid declares empty + operator together
monoid attribute allErrors :: [Message] with [], ++;

production add
e::Expr ::= l::Expr r::Expr
{
  e.allErrors <- l.allErrors;
  e.allErrors <- r.allErrors;
  -- base is automatically []
}

monoid attribute combines the identity element and the operator in one declaration. The base value is implicit — it's the identity element of the monoid. Cleaner, but semantically equivalent.

-- Auto-generate combining equations for all children
propagate errors on Expr;

-- Equivalent to writing:
--   e.errors := [];
--   e.errors <- child1.errors;
--   e.errors <- child2.errors;
--   ... for every child in every Expr production

propagate auto-generates the combining equations for all children of all productions of a nonterminal. This is Silver's version of "default inherited attribute copy" applied to collection attributes. Massively reduces boilerplate.

Key Insight: Silver collections are synthesized

In Silver, collection attributes flow bottom-up along the tree structure, just like synthesized attributes. Each production explicitly combines its children's contributions. The <- syntax is sugar for folding with the monoid operator. Contributions always flow from child to parent along the tree edges.

This means Silver collections cannot target remote nodes. A deeply-nested Use node cannot directly contribute to a distant Module.errors — the contribution must bubble up through every intermediate production.

3JastAdd's Two-Phase System

Remote targeting via coll declarations and contributes clauses

// Phase 1: Declare the collection attribute on a target node type
coll Set<String> Program.errors()
  [new HashSet<>()]   // initial value (identity element)
  with add              // combine method
  root Program;         // survey root (where to start looking for contributors)

The coll declaration specifies: the attribute type, initial value, combining method, and the survey root (which node to start scanning for contributors).

// Phase 2: Contributions from ANYWHERE in the tree
Expr contributes errorMsg()
  when hasError()
  to Program.errors();

// 'each' for contributing multiple items
Stmt contributes each getWarnings()
  to Program.errors();

contributes clauses can appear anywhere — even in separate .jrag files (aspect-oriented). The when clause is a guard condition. each flattens an iterable of contributions.

// Target a SPECIFIC node (not just the root)
Use contributes "undefined: " + getName()
  to Block.errors()
  for enclosingBlock();    // reference attribute resolving to target

// Multiple targets via 'for each'
Decl contributes each getWarnings()
  to Module.warnings()
  for each getAffectedModules();  // contributes to multiple nodes

The for clause uses a reference attribute to resolve the specific target node. This is the killer feature: contributions can target any reachable node, not just ancestors or subtree roots.

// Generated code (simplified):
public Set<String> errors() {
  if (errors_computed) return errors_value;

  // Phase 1: Survey — walk tree, build contributor map
  if (state.IN_COLLECT != errors) {
    state.IN_COLLECT = errors;
    survey_errors(root());  // DFS from survey root
  }

  // Phase 2: Collect — aggregate contributions targeting THIS node
  errors_value = new HashSet<>();
  for (ASTNode contributor : contributorMap.get(this)) {
    contributor.contributeTo_errors(errors_value);
  }
  errors_computed = true;
  return errors_value;
}

The survey phase runs once per collection attribute (amortized across all nodes). It walks the tree and builds an IdentityHashMap mapping each target node to its list of contributors. The collect phase is then O(contributors) per target node, not O(tree).

Key Insight: JastAdd decouples contributor from target

The contributes...to...for pattern lets any node in the tree contribute to any other node. A Use deep in the tree can contribute an error directly to its enclosingBlock() without the intermediate nodes knowing about it. This is fundamentally more powerful than Silver's bottom-up-only flow.

The trade-off: it requires a global survey phase and code generation to wire up the contributor dispatch. It also requires reference attributes (for clause) that can resolve to arbitrary nodes, which demands the AG already has those reference attributes defined.

5Verdict: Which Is Better?

It depends on whether you need remote targeting

Short answer: JastAdd's is more powerful, Silver's is more practical for us

JastAdd's coll/contributes system is objectively more expressive. The ability to contribute to remote nodes via for clauses enables patterns that are simply impossible in Silver or our current approach: a Use node contributing an error to its enclosing Block, a Decl contributing to a distant Module's symbol table.

However, JastAdd achieves this through code generation — it emits custom Java classes with survey methods, contributor maps, and contributeTo_X() dispatch methods. This is fundamentally at odds with our design principle of being a pure runtime TypeScript library (no code generation, accept the Kiama ceiling).

Silver's approach — monoid attributes with bottom-up propagation — maps naturally to a runtime library. It's essentially what we already do, but with better ergonomics (monoid declaration) and auto-propagation (propagate). The limitation (no remote targeting) is the same as our current limitation.

The real question is: can we approximate JastAdd's remote targeting at runtime, or is the expressiveness gap acceptable?

Why JastAdd's two-phase approach is hard to replicate at runtime ▶

JastAdd's contributes...to...for works because the code generator:

  1. Knows all contribution declarations at compile time. It can enumerate every contributes clause and generate a contributeTo_X() method on each contributing node type.
  2. Generates survey methods. The survey phase walks the tree once and builds an IdentityHashMap<ASTNode, List<ASTNode>> mapping each target to its contributors. This is O(tree) but amortized across all accesses.
  3. Uses static dispatch. Each contributor's contributeTo_X() is a generated method that directly evaluates the contribution equation. No dynamic dispatch or callback registration.

In a runtime library, we'd need to either:

  • Walk the entire tree on every access (O(tree), not just O(subtree)), or
  • Build a contributor registry at install time (mimicking the survey phase), which requires a registration API that the user calls during attribute definition.

Both are possible but add complexity. The registry approach is the more practical one, and that's what Option B below proposes.

When you actually need remote targeting ▶

Remote targeting matters when:

  • Error collection: A deeply-nested Use node discovers an undefined variable and wants to report it to the enclosing Module, not to every intermediate node.
  • Symbol tables: A Decl wants to register itself in the nearest enclosing scope's symbol table. The scope node might be several levels up.
  • Cross-cutting concerns: A node wants to contribute to a node that isn't its ancestor at all (e.g., contributing to a node reachable via a reference attribute).

For most use cases in @ksc/ag today (KindScript checker), subtree-scoped collection is sufficient. The checker uses coll() for error collection from subtrees, and more complex patterns like scope-based error collection can be handled with inh() + manual aggregation.

Remote targeting becomes important when building larger analyses (full compilers, IDE tooling) where the "report to nearest enclosing X" pattern is common.

6Uplift Options

Three concrete paths for replacing the current coll()

Option A: Silver-style Monoid Attributes

Replace coll() with a monoid() factory that declares the identity element and combine operator. Add auto-propagation: by default, a monoid attribute on a parent node combines all children's values for the same attribute. Manual contributions via the existing contribute function for node-local additions.

Implementation: Still a syn() wrapper internally. The difference is ergonomic: instead of manually specifying a DFS walk, the attribute auto-combines children's values (like Silver's propagate). The user can add local contributions per node type.

  • Minimal API change — drop-in replacement for coll()
  • No new infrastructure (still a synthesized attribute)
  • Matches Silver's proven semantics
  • Auto-propagation reduces boilerplate significantly
  • Type-safe monoid declaration
  • Still no remote targeting (same expressiveness ceiling as Silver)
  • Still O(subtree) per access (though cached)
  • No aspect-oriented distribution of contributions

Option B: JastAdd-style Two-Phase with Registry

Implement a contributor registry that mimics JastAdd's survey phase. At install time, contributors register themselves. A separate coll() declaration on the target node type consumes registered contributions. Uses a WeakMap to track contributor → target mappings.

Implementation: Two new APIs: coll(init, combine) for declaring collection targets, and contributes(targetAttr, contribution, { when?, for? }) for declaring contributions. At applyAttributes time, a survey walk builds the contributor map. Collection attribute access iterates only registered contributors.

  • Full JastAdd expressiveness: remote targeting, conditional, multi-target
  • O(contributors) per access after O(tree) survey (better than O(subtree) repeated)
  • Aspect-oriented: contributions can be defined separately from the target
  • Significantly more complex implementation
  • Requires WeakMap infrastructure (breaks "no external data structures" principle)
  • Survey phase must be triggered explicitly (when? at apply time? on first access?)
  • The for clause requires reference attributes to already be computed — introduces ordering constraints
  • More complex API for the user

Option C: Hybrid — Monoid + Targeted Contributions

Keep Silver-style monoid as the default (Option A), but add an optional targeted contribution mechanism for cases that need it. Monoid attributes auto-combine children by default. For remote targeting, a contributeTo() helper collects contributions into a node-keyed registry that the monoid attribute drains.

Implementation: monoid() works as in Option A. Additionally, contributeTo(targetAttr, targetResolver, contribution) installs a side-effect during tree walk that populates a per-target collection. The monoid attribute checks this registry in addition to children's values.

  • Simple for simple cases (monoid), powerful when needed (contributions)
  • Gradual adoption — start with monoid, add contributions when needed
  • Remote targeting available for complex analyses
  • Two mechanisms for the same concept — conceptual overhead
  • Registry is effectively a side-channel — harder to reason about evaluation order
  • Still has the ordering constraint problem from Option B

7API Designs

What each option looks like in practice

// Option A: Silver-style monoid attribute
import { monoid } from '@ksc/ag';

// Declaration: identity element + combine operator
const errors = monoid<TNode, string[]>({
  empty: () => [],
  combine: (a, b) => [...a, ...b],
  // Optional: local contribution per node (like Silver's :=)
  local: (node) => {
    if (node.type === 'Use' && !node.resolved)
      return [`undefined: ${node.name}`];
    return [];  // identity element = no contribution
  },
});

// Usage: errors at any node = local contribution + all children's errors
applyAttributes(root, { errors });
// root.errors contains all errors from entire tree

// ─────────────────────────────────────────────
// API signature:
function monoid<N, V>(opts: {
  empty: () => V;                     // identity element factory
  combine: (a: V, b: V) => V;         // associative binary operator
  local?: (node: N) => V;              // node-local contribution (default: empty)
}): AttributeDef<N, V>;

Internally: monoid() returns a syn() that computes combine(local(node), combine(child1.attr, combine(child2.attr, ...))). Same as current coll() but with auto-propagation from children instead of a DFS walk.

// Option B: JastAdd-style coll/contributes
import { coll, contributes } from '@ksc/ag';

// Step 1: Declare the collection target
const errors = coll<TNode, string[]>({
  init: () => [],
  combine: (acc, item) => [...acc, item],
});

// Step 2: Declare contributions (can be in separate files)
const useContrib = contributes<TNode, string>({
  to: 'errors',
  value: (node) => `undefined: ${node.name}`,
  when: (node) => node.type === 'Use' && !node.resolved,
  for: (node) => (node as any).enclosingModule,  // remote target!
});

// Step 3: Apply both
applyAttributes(root, {
  errors,
  _useContrib: useContrib,  // contributes attrs are side-effects
});

// ─────────────────────────────────────────────
// API signatures:
function coll<N, V>(opts: {
  init: () => V;
  combine: (acc: V, item: V) => V;
}): AttributeDef<N, V>;

function contributes<N, V>(opts: {
  to: string;                        // target attribute name
  value: (node: N) => V;             // contribution value
  when?: (node: N) => boolean;       // guard condition
  for?: (node: N) => N;              // target resolver (default: root)
}): AttributeDef<N, void>;
// Option C: Hybrid — monoid() for most cases, contributeTo() for remote
import { monoid, contributeTo } from '@ksc/ag';

// Simple case: monoid with auto-propagation (like Option A)
const errors = monoid<TNode, string[]>({
  empty: () => [],
  combine: (a, b) => [...a, ...b],
});

// Advanced case: remote contribution
const reportToModule = contributeTo<TNode, string>({
  target: 'errors',
  resolve: (node) => (node as any).enclosingModule,
  value: (node) => `type error: ${node.msg}`,
  when: (node) => node.type === 'TypeErr',
});

applyAttributes(root, {
  errors,
  _reportToModule: reportToModule,
});
// Current @ksc/ag coll() — for comparison
import { coll } from '@ksc/ag';

const errors = coll<TNode, string[]>(
  () => [],                                          // init
  (acc, v) => [...acc, v],                            // combine
  (node) => {                                         // contribute
    if (node.type === 'Use' && !node.resolved)
      return `undefined: ${node.name}`;
    return undefined;  // skip
  },
);

applyAttributes(root, { errors });
// root.errors: all errors from root's subtree (= entire tree)
// child.errors: all errors from child's subtree only
Performance comparison ▶
Operation Current coll() Option A: monoid() Option B: coll/contributes
Install time O(1) per node O(1) per node O(tree) survey
First access (root) O(tree) DFS O(tree) via children recursion O(contributors)
First access (leaf) O(1) DFS O(1) O(contributors of leaf)
Subsequent access O(1) cached O(1) cached O(1) cached
Total work (all nodes) O(n^2) worst case O(n) total O(n) total

Key insight: Option A's auto-propagation via children recursion avoids the O(n^2) worst case of the current DFS approach. When a parent accesses .errors, it reads each child's .errors (which are cached from their own computation). Each node's equation runs exactly once — O(n) total, same as Option B's survey phase.

Migration path from current coll() ▶

Option A migration (straightforward):

// Before:
errors: coll(() => [], (a, v) => [...a, v], (n) => n.error ?? undefined)

// After:
errors: monoid({
  empty: () => [],
  combine: (a, b) => [...a, ...b],
  local: (n) => n.error ? [n.error] : [],
})

The key difference: coll's contribute returns a single value, monoid's local returns a value of the full monoid type (e.g., an array, not a single element). The combine operator must work on full values, not single contributions.

Option B migration would require splitting each coll() call into a coll() declaration and separate contributes() declarations. More work, but the API is conceptually cleaner for complex cases.

In JastAdd, what does the for clause in a contributes declaration do?
It specifies the target node that receives the contribution, resolved via a reference attribute. Without for, contributions go to the survey root. With for, they can target any reachable node in the tree.

Why is @ksc/ag's current coll() O(n^2) in the worst case when all nodes access the attribute?
Each node independently DFS-walks its entire subtree. If every node accesses the attribute, the root walks n nodes, each child walks ~n/k nodes, etc. The total work is O(n + (n-1) + (n-2) + ...) = O(n^2). A monoid approach avoids this because each node's equation only reads its children's cached values — O(n) total.

What is Silver's propagate keyword equivalent to?
propagate errors on Expr auto-generates equations that combine all children's errors attributes using the monoid's operator. It's like writing e.errors := []; e.errors <- child1.errors; e.errors <- child2.errors; ... for every production of Expr.

Why can't a pure runtime library fully replicate JastAdd's two-phase collection system?
JastAdd knows all contributes clauses at compile time and generates custom survey methods and contributor dispatch code. A runtime library must discover contributions dynamically, requiring either a registration API or a full tree walk. The for clause also requires reference attributes to already be computed, creating ordering constraints that code generation handles statically.

What algebraic structure does Silver's monoid attribute declaration encode?
