The Three Concerns and Their Mathematical Names

Foundation

Every attribute grammar system decomposes into three separable concerns. Each has a precise mathematical identity:

F Structure (Functor)

Defines the shape of one layer of recursion — what a node looks like, how children relate.

In category theory: an endofunctor F on a category C. The AST is the initial F-algebra — the "most general" structure any interpretation can consume.

createGrammar(getChildren)

A Behavior (Algebra)

Defines how to interpret the structure — what values to compute at each node.

In category theory: an F-algebra (A, α: F(A) → A). The carrier A is the result type; α is the structure map that collapses one layer.

AGSpecInput { attributes }

E Orchestration (Evaluator)

Defines how to apply the algebra to the structure — the traversal strategy.

In category theory: a catamorphism (or its generalizations). The unique homomorphism from the initial algebra to the target algebra.

grammar.evaluate() / evaluateAll()

The universal decomposition: Initial Algebra F-Algebra Catamorphism (the AST) (the interpretation) (the evaluator) μF ——— unique homomorphism ———> (A, α) KSNode tree ——— grammar.evaluate(spec) ———> KindDefinition[] CheckerDiagnostic[] The catamorphism IS the thing that "runs" an algebra over data. It exists by initiality — it's the ONLY structure-preserving map.

Why "Algebra" Isn't Quite Right for the Grammar

A common confusion: calling the Grammar a "carrier algebra." Precisely:

ConceptMathWhat it ISWhat it ISN'T
Grammar Endofunctor F The shape of recursion (how children connect) Not the carrier, not the operations
AGSpecInput F-algebra (A, α) Carrier type A + structure map α (equations) Not the tree shape, not the traversal
AST (KSNode tree) Initial F-algebra μF The data itself — the "most general" F-algebra Not an interpretation, just structure
evaluate() Catamorphism (cata) The unique homomorphism μF → A Not an algebra, not a functor

F-Algebras, Initial Algebras, and Lambek's Lemma

Category Theory
Definition: F-Algebra Given an endofunctor F : C → C, an F-algebra is a pair (A, α) where: • A is an object in C (the carrier) • α : F(A) → A is a morphism (the structure map) In Haskell: type Algebra f a = f a -> a

The Functor F Defines Shape

For a tree grammar, F describes one layer of nesting. If your nodes can be Leaf or Branch(left, right), the functor is:

-- One layer, with holes (a) where children go
data TreeF a = LeafF | BranchF a a

-- The actual tree is the fixed point: Tree = TreeF(Tree)
type Tree = Fix TreeF

In KSC, the functor is implicit in getChildren: it says "a KSNode has N children, each of which is a KSNode."

The Initial Algebra Is the Data Type Itself

Lambek's Lemma: If (μF, in) is an initial F-algebra, then in : F(μF) → μF is an isomorphism.

This means μF ≅ F(μF) — "the data type IS one layer of functor wrapped around itself." A tree IS a node containing trees.

The AST KSNode is μF. The constructors (CompilationUnit, TypeAliasDeclaration, ...) are the structure map in.

The Catamorphism (Universal Property) For any F-algebra (α : F(A) → A), there exists a unique morphism cata(α) : μF → A such that: cata(α) ∘ in = α ∘ F(cata(α)) This commuting square says: "unwrap one layer (in¹), apply the catamorphism to all children (F(cata α)), then combine results (α)." in F(μF) ————→ μF | | F(cata α) | | cata α | | v v F(A) ————→ A α

Beyond Catamorphisms: What AG Evaluation Actually Is

Recursion Schemes

A catamorphism is bottom-up only — it handles synthesized attributes. But KSC's AG evaluator also handles inherited (top-down) and circular (fixed-point) attributes. What mathematical objects cover these?

Catamorphism (Fold) — Bottom-Up

The Math

cata(α) : μF → A

Process children first, combine results. This is a pure fold over the tree structure. Each node sees only the computed results from its children, not the children themselves.

cata :: Functor f => (f a -> a) -> Fix f -> a
cata alg = alg . fmap (cata alg) . unFix

KSC Equivalent

syn(compute) where compute only reads $children attributes.

Example: allViolations in checker.ts — aggregates results from the subtree via DFS.

Covers: syn, match (per-production dispatch)

Doesn't cover: inh, circular

Paramorphism — Fold with Original Subtree Access

The Math

para(α) : μF → A

Like a catamorphism, but each equation sees both the computed result and the original subtree at each child position.

para :: Functor f => (f (Fix f, a) -> a) -> Fix f -> a
-- Each child gives you (original_subtree, folded_result)

KSC Equivalent

Every syn attribute in KSC is effectively a paramorphism, because the equation receives the full node (not just child results). The equation (node: N) => V can read both node.$children (structure) and node.childAttr (computed).

This is a consequence of stamping attributes directly on nodes — the node IS both the data and the computed result container.

Zygomorphism — Mutual Recursion (Paired Folds)

The Math

zygo(f, g) : μF → A

Two mutually recursive folds. Attribute B depends on attribute A; both are computed in a single bottom-up pass. The "banana split" theorem says any two catamorphisms can be fused into one pass.

zygo :: Functor f
     => (f b -> b)        -- auxiliary fold
     -> (f (b, a) -> a)   -- main fold (sees aux results)
     -> Fix f -> a

KSC Equivalent

When importViolation reads node.isReference and node.fileImports (both synthesized), this is a zygomorphism — multiple folds where one depends on another.

KSC handles this implicitly via demand-driven lazy evaluation. Accessing node.isReference inside importViolation's equation triggers the auxiliary fold on demand. No explicit fusion needed.

Hylomorphism — Unfold Then Fold

The Math

hylo(α, β) = cata(α) ∘ ana(β)

First unfold (anamorphism β) to build intermediate structure, then fold (catamorphism α) to collapse it. The intermediate tree can be "deforested" away.

hylo :: Functor f => (f a -> a) -> (b -> f b) -> b -> a
hylo alg coalg = alg . fmap (hylo alg coalg) . coalg

KSC Equivalent

The full KSC pipeline IS a hylomorphism:

  • Unfold (ana): convertNode() — builds the KSNode tree from TypeScript AST
  • Fold (cata): evaluate() — computes attributes over the tree

But the intermediate tree isn't deforested — it's materialized as the KSNode AST (needed for the dashboard, serialization, etc.).

Adjoint Folds (Hinze-Wu-Gibbons) — The Unifying Framework

The Math

Hinze, Wu, and Gibbons (2013/2015) showed that all structured recursion schemes arise as catamorphisms in a Kleisli category of a comonad.

Gibbons (2014) showed that inherited attributes = downward accumulations and synthesized attributes = upward accumulations, both expressible as conjugate hylomorphisms.

The key insight: an AG with both synthesized and inherited attributes is a catamorphism over a function type:

-- AG = a catamorphism where the carrier is (Inh -> Syn)
type AG inh syn = cata (f (inh -> syn) -> (inh -> syn))

The inherited attribute flows in via the function argument; the synthesized attribute flows out as the result. The evaluator is still a catamorphism — just over a more complex carrier.

KSC Equivalent

This is the deepest insight. KSC's inh attributes work by reading $parent, which is essentially threading a "context" parameter downward. Conceptually:

// What KSC does (imperative, demand-driven):
const defEnv = inh(rootValue, eq)
// node.defEnv reads node.$parent.defEnv

// What the math says (functional):
// defEnv is part of an Inh->Syn function
// The parent passes Inh downward
// The child computes Syn upward

KSC's demand-driven evaluation with $parent navigation is an imperative encoding of the categorical "catamorphism over Inh→Syn."

Bialgebras and Dialgebras — Algebra + Coalgebra

The Math

Jacobs & Uustalu (2012): "Semantics of Grammars and Attributes via Initiality" models attribute grammars categorically. The key idea:

  • Synthesized = F-algebra: F(Syn) → Syn (bottom-up fold)
  • Inherited = F-coalgebra: Inh → G(Inh) (top-down unfold)
  • Combined AG = a dialgebra or structure on a product: computation over (Inh × Syn) tuples

Hamana (2009): "Attribute Grammars as Recursion Schemes over Cyclic Representations of Zippers" — models AGs using zippers (comonads) with circular representations for parent access.

KSC Equivalent

KSC's stamped tree with $parent / $children is literally a zipper. Each node can navigate both up and down, which is exactly what Hamana's framework requires.

The circular() primitive adds fixed-point iteration on top — this is the Magnusson-Hedin solver for cyclic dependency graphs, which goes beyond standard recursion schemes into iterative fixpoint computation.

The Multi-Sorted Algebra Problem

Category Theory

KSC has 12 attributes with different types: kindDefs: KindDefinition[], fileImports: Set<string>, isReference: boolean, etc. A standard F-algebra has one carrier A. How do you handle many?

Option 1: Product of Carriers

The carrier is a product type (tuple/record) of all attribute types:

type Carrier = {
  kindDefs: KindDefinition[]
  defEnv: Map<string, KindDefinition>
  fileImports: Set<string>
  isReference: boolean
  // ... 8 more
}

The algebra α: F(Carrier) → Carrier computes all attributes simultaneously. This is the "banana split" approach — multiple folds fused into one.

Used by: Haskell's recursion-schemes (product of carriers), Fokkinga's tupling theorem.

Option 2: Many-Sorted Algebra

A many-sorted algebra has multiple carrier sets (one per sort) and operations that map between them. Each attribute is a sort.

This is the mathematically precise model: the attribute grammar IS a many-sorted initial algebra problem, where each sort has its own equations.

Used by: Knuth's original 1968 formulation, Jacobs-Uustalu 2012.

What KSC Does: Implicit Product via Lazy Properties

KSC stamps all 12 attributes as lazy properties on every node. The "carrier" is implicitly the product of all attribute types. But because evaluation is demand-driven, the product is never materialized as a tuple — each attribute is computed independently on first access.

This is actually the optimal implementation of the banana-split theorem: you get the fusion of multiple folds (all attributes share a single tree traversal during installation) without paying for the product construction (lazy evaluation means unused attributes are never computed).

How Real Systems Separate the Three Concerns

Prior Art

JastAdd (Generator)

Structure: .ast files

Declarative grammar defining node types, child relationships, and production names.

Program ::= CompUnit*;
CompUnit ::= ImportDecl* TypeDecl*;
TypeDecl ::= <Name:String> Member*;

Behavior: .jrag files

"Aspects" containing attribute equations. Multiple .jrag files contribute to the same grammar.

syn int TypeDecl.memberCount() =
  getNumMember();
inh Env TypeDecl.env();
eq CompUnit.getChild().env() =
  buildEnv();

Orchestration: Code Generator

The JastAdd compiler weaves .ast + .jrag into Java classes with caching, demand-driven evaluation, and circular attribute support. The evaluator is generated code.

Separation score: 10/10. Files are physically separate. The compiler enforces completeness. But: requires a build step.

Kiama (Scala Library)

Structure: Case Classes

Standard Scala case classes define the tree. No special grammar file.

sealed trait Exp
case class Num(v: Int) extends Exp
case class Add(l: Exp, r: Exp) extends Exp

Behavior: attr() Definitions

Attributes are first-class Scala functions, defined with attr, paramAttr, circular.

val eval: Exp => Int = attr {
  case Num(v) => v
  case Add(l, r) => eval(l) + eval(r)
}

Orchestration: Built-in Strategies

Kiama provides rewriting strategies (topdown, bottomup, etc.) and a Tree class that establishes parent-child navigation.

Separation score: 7/10. Structure is standard Scala. Behavior uses attr combinator. But orchestration is implicit (demand-driven caching inside attr). Most similar to KSC.

Ohm.js (Three Explicit Objects)

Structure: Grammar

const g = ohm.grammar(`
  Arith {
    Exp = Exp "+" Term  -- add
        | Term
    Term = number
  }
`);

Behavior: Semantics

const s = g.createSemantics()
  .addOperation('eval', {
    Exp_add(l, _, r) {
      return l.eval() + r.eval();
    },
    Term(n) { return parseInt(n.sourceString); }
  });

Orchestration: match + apply

const m = g.match(input);
s(m).eval();  // triggers evaluation

Separation score: 9/10. Three explicit objects. Semantics can be extended after creation. But limited to synthesized attributes (no inherited).

recursion-schemes (Haskell Library)

Structure: Base Functor

data ExprF a
  = LitF Int
  | AddF a a
  | MulF a a
deriving Functor

type Expr = Fix ExprF

Behavior: Algebra Functions

evalAlg :: ExprF Int -> Int
evalAlg (LitF n)   = n
evalAlg (AddF a b) = a + b
evalAlg (MulF a b) = a * b

Orchestration: Named Morphisms

cata evalAlg expr    -- bottom-up
para paraAlg expr    -- with subtree access
zygo aux mainAlg expr -- mutual recursion
hylo alg coalg seed  -- unfold then fold

Separation score: 10/10. The three concerns are different types. The Base functor (via Recursive typeclass) is the structure. Algebras are plain functions. Morphisms are explicit combinators. But: no native inherited attribute support.

Tagless Final (Haskell/Scala Pattern)

Structure: Typeclass / Trait

class ExpSYM repr where
  lit  :: Int -> repr
  add  :: repr -> repr -> repr
  mul  :: repr -> repr -> repr

No data type at all — the "AST" is defined by overloaded operations.

Behavior: Instances

instance ExpSYM Int where
  lit = id
  add = (+)
  mul = (*)

instance ExpSYM String where
  lit = show
  add a b = "(" ++ a ++ "+" ++ b ++ ")"

Orchestration: Implicit

The typeclass resolution IS the evaluator. When you write add (lit 1) (lit 2) :: Int, Haskell selects the evaluator instance automatically.

Separation score: 8/10. Clean separation of syntax (typeclass) and semantics (instances). But: no first-class tree to inspect, no inherited attributes, hard to add analyses that need the full tree.

Object Algebras (Oliveira & Cook 2012)

Structure: Generic Interface

interface ExpAlg<E> {
  lit(n: number): E;
  add(l: E, r: E): E;
  mul(l: E, r: E): E;
}

The generic parameter E is the carrier. The interface IS the functor.

Behavior: Implementations

const evalAlg: ExpAlg<number> = {
  lit: (n) => n,
  add: (l, r) => l + r,
  mul: (l, r) => l * r,
};
const printAlg: ExpAlg<string> = {
  lit: (n) => String(n),
  add: (l, r) => `(${l}+${r})`,
};

Orchestration: Composition via Merge

// Merge two algebras into one over a product
function merge<A, B>(
  a: ExpAlg<A>, b: ExpAlg<B>
): ExpAlg<[A, B]>

Separation score: 9/10. Solves the expression problem: new data variants AND new operations. Algebras compose via merge. Closest to what a redesigned KSC could look like.

Design Options for KSC: Algebraic Separation

Options

Each option maps to a different level of mathematical faithfulness vs. practical simplicity.

Option A: Minimal — Extract Evaluator from Grammar

Change: Grammar becomes pure functor; evaluate() is standalone

// Structure — pure functor, knows only tree shape
const grammar = createGrammar<KSNode>(getChildren);

// Behavior — unchanged
const binder = createBinderSpec();

// Orchestration — standalone function
const results = evaluate(grammar, [binder, checker], root);

Pros

  • Smallest change (move methods out of Grammar)
  • Grammar.getChildren is the only field — pure functor
  • evaluate() takes (functor, algebras, data) — clear roles

Cons

  • Doesn't address stamp/validate/apply fusion inside evaluate()
  • Internal architecture unchanged
Mathematical Identity Grammar = F (endofunctor) AGSpecInput = (A, α) (F-algebra) evaluate = cata (catamorphism) Just moves cata out of F, which is the correct placement.

Option B: Three Objects — Grammar, Semantics, Interpreter

Change: Add Semantics object (validated, sealed algebra composition)

// 1. Structure — the functor
const grammar = createGrammar<KSNode>(getChildren);

// 2. Behavior — merged, validated, sealed
const semantics = createSemantics(grammar, [
  createBinderSpec(),
  createCheckerSpec(),
]);
// semantics.order: ['ksc-binder', 'ksc-checker']
// semantics.attributes: ['kindDefs', 'defEnv', ..., 'allViolations']
// semantics.deps: DepGraph (from analyzeDeps)
// Validation has already run (duplicates, cycles, completeness)

// 3. Orchestration — interpret
const results = interpret(semantics, root);
// Stamps tree, applies attributes, projects results

Pros

  • Mirrors Ohm.js's proven three-object pattern
  • Semantics is inspectable before execution (dep graph, order, attributes)
  • Dashboard can render the Semantics without evaluating a tree
  • Validate once, run on many trees
  • The "sealed registry" pattern (Strategy 1)

Cons

  • Three concepts where one call sufficed
  • createSemantics does eager work (analysis, validation)
Mathematical Identity Grammar = F (endofunctor) Semantics = ∏i (Ai, αi) (product of F-algebras, validated & sorted) interpret = cata over the product algebra Semantics is the "compiled" multi-sorted algebra. Validation = checking the algebra is well-formed. interpret = applying the unique homomorphism.

Option C: Named Morphisms — recursion-schemes Style

Change: Make evaluation strategy explicit and named

// Structure
const grammar = createGrammar<KSNode>(getChildren);

// Behavior
const kindDefs = synAttr(match('kind', { ... }));
const defEnv  = inhAttr(rootValue, eq);

// Orchestration — explicit morphism choice
const results = para(grammar, { kindDefs, defEnv }, root);   // paramorphism
const reach   = fixpoint(grammar, { reachable }, root);      // fixed-point iteration

Pros

  • Makes the recursion scheme explicit — you choose cata/para/hylo/fixpoint
  • Maps directly to the mathematical taxonomy
  • Could optimize per-scheme (cata needs no $parent, etc.)

Cons

  • Exposes complexity users don't need to see
  • Most AG specs need multiple schemes (syn + inh + circular)
  • Would need auto-detection to avoid manual scheme selection
  • This is what recursion-schemes libraries struggle with — users find the morphism zoo intimidating

Option D: Object Algebras — Composable Interpretations

Change: Specs become typed algebra implementations; merge() composes them

// Structure — the "signature" (generic interface)
interface KSAlgebra<E> {
  CompilationUnit(children: E[]): E;
  TypeAliasDeclaration(name: E, type: E): E;
  Identifier(text: string): E;
  // ... per production
}

// Behavior — each spec implements the algebra
const binderAlg: KSAlgebra<BinderCarrier> = {
  CompilationUnit(kids) { /* collect kindDefs */ },
  TypeAliasDeclaration(name, type) { /* extract Kind<> */ },
  Identifier(text) { /* return text */ },
};

// Orchestration — merge and fold
const combined = merge(binderAlg, checkerAlg);
const results = cata(combined, root);

Pros

  • Maximum type safety: each production has a typed equation
  • Merge gives you banana-split (product of carriers) for free
  • Solves the expression problem: add new node types OR new analyses

Cons

  • KSC has 361 node types — the interface would be enormous
  • No inherited attributes in the standard formulation
  • Would need to extend object algebras with context passing (a research problem)
  • Dramatically different API from current
Mathematical Identity KSAlgebra<E> = The functor F expressed as a TypeScript interface binderAlg = An F-algebra (E, α) where E = BinderCarrier merge(a, b) = Product algebra (A × B, α × β) cata = The unique homomorphism from μF to A × B This IS the mathematical object, directly. But: AG = more than catamorphism (need inherited attrs).

Option E: Full Algebraic — Functor + Bialgebra + Fixpoint

Change: The nuclear option — model the full AG as a bialgebra

// Structure — explicit base functor
type KSNodeF<A> = {
  kind: string;
  children: A[];      // holes where children go
  props: Record<string, unknown>;
}

// Behavior — bialgebra (algebra + coalgebra)
type AGBialgebra<Inh, Syn> = {
  // Algebra: synthesize from children's results
  syn: (KSNodeF<Syn>) => Syn,
  // Coalgebra: distribute inherited to children
  inh: (Inh) => KSNodeF<Inh>,
}

// Orchestration — bidirectional fold (adjoint fold)
function agEval<Inh, Syn>(
  bialg: AGBialgebra<Inh, Syn>,
  root: Fix<KSNodeF>,
  rootInh: Inh,
): Syn

Pros

  • Mathematically exact — this IS the categorical model (Jacobs-Uustalu)
  • Cleanest possible separation of structure / behavior / evaluation
  • Could derive evaluation order from the bialgebra structure
  • Opens path to formal verification

Cons

  • Requires base functor (KSNodeF) — doubles the type definitions
  • Fixed-point types (Fix) are unergonomic in TypeScript
  • Circular attributes don't fit the bialgebra model cleanly
  • Beautiful math, painful engineering
  • No existing TypeScript library does this well
Mathematical Identity KSNodeF = The base functor F Fix KSNodeF = μF (the initial algebra / AST) AGBialgebra = Dialgebra (F(Syn)→Syn, Inh→F(Inh)) agEval = The unique morphism from the initial dialgebra Alternatively, following Gibbons 2014: agEval = cata over carrier (Inh → Syn) i.e., the AG is a catamorphism where the carrier is a function type.

Recommendation: Option B (Three Objects) with Algebraic Naming

The Sweet Spot

Option B gives you clean three-way separation without requiring base functors or bialgebra types. The key insight from the math:

  • The Grammar IS the functor — just make it pure (remove evaluate methods)
  • The Semantics IS the product algebra — validated, sorted, sealed
  • The interpreter IS the catamorphism (over Inh→Syn function types, implicitly)

You get the mathematical structure without paying the TypeScript ergonomics tax. And the Semantics object gives you a concrete, inspectable artifact that the dashboard can render.

// This is what it looks like in practice:
const grammar = createGrammar<KSNode>(getChildren);     // F
const semantics = createSemantics(grammar, [binder, checker]);  // ∏(Ai, αi)
const results = interpret(semantics, root);              // cata

The Deepest Insight: AGs as Catamorphisms Over Function Types

Key Insight

The most important result from the categorical AG literature (Backhouse & de Moor 2001, Gibbons 2014, Hinze-Wu-Gibbons 2013):

An attribute grammar with both inherited and synthesized attributes is a catamorphism where the carrier is a function type Inh → Syn.

Inherited attributes are not a separate mechanism. They are the function argument. The parent passes inherited values by calling the child's function. The child returns synthesized values as the function result.

Standard catamorphism (synthesized only): cata : μF → Syn Each node computes Syn from children's Syn values AG catamorphism (synthesized + inherited): cata : μF → (Inh → Syn) Each node computes a FUNCTION from inherited context to synthesized result Parent calls child's function, passing Inh Child returns Syn as the result What KSC does (imperative encoding): stampTree(root) establish $parent links (enables Inh flow) applyAttributes(root) install lazy getters (Inh→Syn, cached) node.defEnv reads $parent.defEnv (Inh, via $parent link) node.kindDefs computes from children (Syn, via $children)

KSC's $parent links are the imperative implementation of the function argument. Instead of explicitly passing inherited values through a function call, KSC makes them available via navigation. The lazy getter reads $parent.attrName — this IS the "call the parent's function to get the inherited value" step, just encoded as property access.

Why This Matters for Design

If AGs are catamorphisms over Inh → Syn, then:

  • You don't need a separate mechanism for inherited attributes — they're the function input
  • The evaluator is still "just" a fold — over a more complex carrier
  • All the recursion scheme combinators (zygo, para, hylo) still apply
  • The multi-sorted problem is just: the carrier is Inh → (Syn1 × Syn2 × ... × Syn12)

Circular attributes add fixed-point iteration on top, which takes you beyond pure recursion schemes into the domain of abstract interpretation (Cousot & Cousot 1977). This is where KSC's circular() primitive lives — it's the Magnusson-Hedin solver, which is a fixpoint iterator, not a fold.

If (μF, in) is an initial F-algebra, then in : F(μF) → μF is an isomorphism. This means μF ≅ F(μF) — "a tree IS one layer of functor around more trees." The AST type IS its own recursive definition, which is why you can fold over it.
Q: Why is a catamorphism insufficient for full AG evaluation?
Click to reveal
A catamorphism is bottom-up only (synthesized attributes). It can't handle: (1) inherited attributes (top-down context), (2) circular attributes (fixed-point iteration). You need either adjoint folds (catamorphism over Inh→Syn function types) or the bialgebra/dialgebra model.
Q: How does the Backhouse-de Moor/Gibbons insight unify synthesized and inherited attributes?
Click to reveal
An AG with both syn and inh is a catamorphism where the carrier is Inh → Syn (a function type). Inherited values flow in as the function argument; synthesized values flow out as the result. The evaluator is still a fold — just over a more complex carrier.
Q: What recursion scheme describes KSC's importViolation reading isReference and fileImports?
Click to reveal
A zygomorphism — mutual recursion between multiple folds where one depends on another. KSC handles this implicitly via demand-driven lazy evaluation: accessing node.isReference triggers the auxiliary fold on demand.
Q: What is the stamped $parent link, categorically?
Click to reveal
It's the imperative encoding of the zipper comonad (Hamana 2009). Each node can navigate both up ($parent) and down ($children), which is exactly a tree zipper. This enables inherited attributes by providing upward navigation — equivalent to passing a function argument in the Inh→Syn model.
Q: How does KSC handle the multi-sorted algebra problem (12 attributes with different types)?
Click to reveal
Implicitly via the banana-split theorem: all attributes are installed as lazy properties on every node, making the carrier an implicit product type. But because evaluation is demand-driven, unused attributes are never computed — optimal fusion without explicit product construction.
Q: What goes beyond recursion schemes in KSC's AG library?
Click to reveal
The circular() primitive. It uses the Magnusson-Hedin fixed-point iteration algorithm, which re-evaluates all cycle members until convergence. This is abstract interpretation (Cousot-Cousot 1977), not a recursion scheme — it handles cyclic dependencies that no fold can express.
Q: What does the Object Algebra pattern (Oliveira-Cook 2012) lack that KSC needs?
Click to reveal
Inherited attributes. Object algebras handle synthesized attributes beautifully (each implementation is an F-algebra, merge gives you product). But there's no standard mechanism for top-down context flow. You'd need to extend the pattern with function-type carriers or explicit context threading.
Q: Why is Option B (Three Objects) recommended over Option E (Full Algebraic)?
Click to reveal
Option E requires base functors (KSNodeF<A>), which doubles the type definitions and is unergonomic in TypeScript. Option B gives the same three-way separation (Grammar = F, Semantics = ∏algebras, interpret = cata) without requiring the type-level encoding. You get mathematical correctness with TypeScript-friendly ergonomics.
Q: What is the full KSC pipeline as a recursion scheme?
Click to reveal
A hylomorphism: unfold (ana) via convertNode() builds the KSNode tree from TypeScript's AST, then fold (cata) via evaluate() computes attributes over it. The intermediate tree is materialized (not deforested) because it's needed for the dashboard and serialization.
Q: In the current KSC architecture, where exactly are the three concerns (structure, behavior, orchestration) entangled?
Click to reveal
Three places: (1) Grammar owns both getChildren (structure) AND evaluate/evaluateAll (orchestration). (2) AttributeDef.install() fuses equation definition (behavior) with caching strategy (orchestration). (3) evaluateAll() mixes validation, topo sort, tree stamping, and attribute application in one method.
