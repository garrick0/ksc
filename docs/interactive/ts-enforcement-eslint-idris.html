Part 15 — KindScript Research Series

Type Enforcement, Lint Rules & Dependent Types

How TypeScript's type system enforces Effect-TS without compiler changes, how eslint-plugin-functional works and compares, and how Idris 2 compiles dependent types from source to machine code.

Part 1: How TypeScript Enforces Effect-TS Types

The key insight: Effect-TS requires zero compiler changes. TypeScript's type system is already Turing-complete. Effect-TS encodes effect tracking into generic type parameters, and the standard tsc type checker does the rest. No plugins, no custom transforms, no AST manipulation.

1.1 TypeScript's Type System Is Already Powerful Enough

TypeScript uses bidirectional type checking (not Hindley-Milner). It flows type information both top-down (from annotations) and bottom-up (from expressions). This already provides the machinery needed:

Conditional Types — Type-Level If/Else

// Distribute over unions automatically
type ExtractError<T> = T extends Effect<infer A, infer E, infer R> ? E : never;

// tsc evaluates this at compile time:
type Err = ExtractError<Effect<string, HttpError, never>>;
// → HttpError

// Union distribution:
type Err2 = ExtractError<
  Effect<string, HttpError, never> | Effect<number, DbError, never>
>;
// → HttpError | DbError  (distributed automatically)

This is how Effect-TS accumulates error types through pipelines — tsc distributes conditional types over unions, merging error channels automatically.

Mapped Types — Transform Object Shapes

// Iterate over keys, transform each property
type Readonly<T> = {
  readonly [K in keyof T]: T[K]
};

// Effect-TS uses this for service definitions:
type Services<R> = {
  [K in keyof R]: Tag<R[K]>
};

// tsc tracks which services are required
// and checks they're all provided by Layer

Mapped types let Effect-TS express "for each service in R, there must be a corresponding Layer that provides it."

Variance Annotations — Controlling Assignability

// TypeScript 4.7+ explicit variance
interface Effect<out A, out E, in R> {
  // out A = covariant: Effect<string> assignable to Effect<string|number>
  // out E = covariant: errors widen (accumulate)
  // in  R = contravariant: requirements narrow (intersect)
}

// This means when you pipe two effects:
// Effect<A1, E1, R1>  →  Effect<A2, E2, R2>
// Result: Effect<A2, E1 | E2, R1 & R2>
//                    ^^^^^^^^  ^^^^^^^^
//                    union     intersection
//                    (widen)   (narrow/combine)

Variance is the core mechanic. Without changing tsc, Effect-TS gets automatic error accumulation (union) and requirement accumulation (intersection) just from how TypeScript handles covariant and contravariant positions.

Overload Resolution — Precise pipe() Signatures

// Effect-TS defines ~20 overloads of pipe():
function pipe<A>(a: A): A;
function pipe<A, B>(a: A, ab: (a: A) => B): B;
function pipe<A, B, C>(a: A, ab: (a: A) => B, bc: (b: B) => C): C;
// ... up to ~20 arguments

// tsc tries each overload first-to-last, picks the first that matches.
// For: pipe(httpGet(url), Effect.map(parse), Effect.flatMap(validate))
// tsc resolves the 3-arg overload, threading types through each step.

Each step in a pipe gets its types inferred from the previous step's return type. tsc just does normal overload resolution — no magic needed.

Branded Types — Nominal Identity in a Structural System

// TypeScript is structural: { name: string } == { name: string }
// But Effect-TS needs nominal identity for services.

// The trick: phantom branded properties
interface Tag<Id, Service> {
  readonly _tag: unique symbol;  // makes each Tag unique
}

// Two services with identical shapes are still distinct:
class UserRepo extends Context.Tag("UserRepo")<UserRepo, {
  getUser: (id: string) => Effect<User>
}>() {}

class AdminRepo extends Context.Tag("AdminRepo")<AdminRepo, {
  getUser: (id: string) => Effect<User>
}>() {}

// UserRepo ≠ AdminRepo despite same shape, because unique symbol differs

unique symbol is a TypeScript feature that gives each declaration a distinct type. Effect-TS leverages this to create nominal service identifiers inside a structural type system.

1.2 How a Pipeline Gets Type-Checked (Step by Step)

Click each step to see what tsc does internally:

1 You write: pipe(fetchUser(id), Effect.flatMap(getOrders), Effect.map(summarize))
tsc sees a call to pipe with 3 arguments after the initial value. It looks up the overload list and selects the 4-parameter overload: pipe<A,B,C,D>(a: A, ab: A→B, bc: B→C, cd: C→D): D.
2 Resolve fetchUser(id) → Effect<User, HttpError, HttpClient>
tsc infers A = Effect<User, HttpError, HttpClient>. This is standard return-type inference. The three type parameters encode: success value (User), possible error (HttpError), required service (HttpClient).
3 Effect.flatMap(getOrders) takes Effect<User, HttpError, HttpClient>
flatMap signature: <A,B,E1,E2,R1,R2>(f: (a:A)→Effect<B,E2,R2>) → (self: Effect<A,E1,R1>) → Effect<B, E1|E2, R1&R2>

tsc unifies: A=User, B=Order[], E1=HttpError, E2=DbError, R1=HttpClient, R2=Database.
Result: Effect<Order[], HttpError | DbError, HttpClient & Database>

Errors widened via union (covariant out). Requirements combined via intersection (contravariant in). This is pure TypeScript generic resolution.
4 Effect.map(summarize) takes the chained result
map only transforms the success value, doesn't add errors or requirements.
Result: Effect<Summary, HttpError | DbError, HttpClient & Database>

tsc now knows the full type of the pipeline. If you try to run this without providing both HttpClient and Database via Layers, it's a compile error.
5 Final: compile error if requirements unmet 
// This would be a TS error:
Effect.runPromise(pipeline)
// TS2345: Argument of type 'Effect<Summary, HttpError | DbError, HttpClient & Database>'
//   is not assignable to parameter of type 'Effect<Summary, never, never>'
//   Type 'HttpClient & Database' is not assignable to type 'never'

// You must provide all services:
pipeline.pipe(
  Effect.provide(HttpClientLive),    // R shrinks
  Effect.provide(DatabaseLive),      // R becomes never
  Effect.runPromise                  // ✓ now R=never, allowed
)
Key point: Every step above is standard TypeScript type inference. The "magic" is that Effect-TS designed its types so that normal generic resolution produces exactly the behavior you'd want from an effect system. No compiler plugin required.

1.3 What Makes This Work (and Its Limits)

What tsc catches

✓ Wrong error type in catchTag

✓ Missing service in requirements

✓ Type mismatch between pipeline steps

✓ Forgetting to handle an error channel

✓ Providing wrong service implementation shape

✓ Using the wrong branded service tag

What tsc cannot catch

✗ Side effects inside Effect.sync(() => ...)

✗ Thrown exceptions (not in type signature)

✗ Mutation of shared state

✗ Calling unwrapped async functions

✗ as any type assertions

✗ Violating conventions outside Effect wrappers

The fundamental trade-off: Effect-TS gets surprisingly far by encoding effects as types. But it operates on an honor system — the runtime callbacks you pass in are opaque to tsc. If you write Effect.sync(() => { fs.unlinkSync('/'); return 42 }), the type says Effect<number, never, never> — pure, no errors, no requirements. tsc cannot look inside that lambda. This is exactly the gap KindScript fills.

1.4 Cost: Type Instantiation Depth

TypeScript has a recursion limit of 100 levels for type instantiation. Long pipelines can hit this:

// 50-step pipe works fine
pipe(a, f1, f2, f3, ..., f50)  // ✓

// But deeply nested conditional types can blow up:
type DeepMerge<A, B> = A extends object
  ? B extends object
    ? { [K in keyof A | keyof B]: K extends keyof A & keyof B
        ? DeepMerge<A[K], B[K]>   // recursive!
        : ... }
    : B
  : B;

// Error: Type instantiation is excessively deep and possibly infinite. ts(2589)

Effect-TS carefully designs its types to stay within this budget. The pipe overloads (max ~20 args) exist specifically to avoid recursive type computation.

Part 2: eslint-plugin-functional

What it is: An ESLint plugin that enforces functional programming patterns through lint rules. It works at the AST level via ESLint's parser infrastructure, optionally with TypeScript type information via @typescript-eslint/parser.

2.1 How It Works

Architecture

// ESLint's visitor pattern:
// 1. Parser (espree or @typescript-eslint/parser) → AST
// 2. Each rule registers node visitors
// 3. ESLint traverses AST, calls matching visitors

module.exports = {
  create(context) {
    return {
      // Visit every VariableDeclaration node
      VariableDeclaration(node) {
        if (node.kind === "let" || node.kind === "var") {
          context.report({
            node,
            messageId: "noLet",
          });
        }
      }
    };
  }
};

Each rule is a function that returns an object mapping AST node types to handler functions. ESLint walks the tree and calls the right handler. Rules cannot modify the AST — they only report diagnostics (and optionally suggest fixes).

AST-Level Analysis

// What the plugin sees (ESTree AST via @typescript-eslint/parser):
{
  "type": "VariableDeclaration",
  "kind": "let",         // ← no-let rule checks this
  "declarations": [{
    "type": "VariableDeclarator",
    "id": { "type": "Identifier", "name": "x" },
    "init": { "type": "Literal", "value": 5 }
  }]
}

// The immutable-data rule looks for AssignmentExpression
// with MemberExpression on the left:
{
  "type": "AssignmentExpression",
  "left": {
    "type": "MemberExpression",  // ← obj.prop = ... is mutation!
    "object": { "name": "user" },
    "property": { "name": "age" }
  }
}

The plugin matches syntactic patterns in the AST. It can see let, for, obj.prop = x, arr.push(), class, throw, etc. It cannot track values through function calls or across module boundaries.

Type-Aware Mode

// .eslintrc.js — enable type info
{
  parser: "@typescript-eslint/parser",
  parserOptions: {
    project: "./tsconfig.json",  // ← links to TS project
  },
  plugins: ["functional"],
  rules: {
    "functional/immutable-data": ["error", {
      ignoreClasses: true,           // allow mutation in class methods
      ignoreImmediateMutation: true, // allow arr.sort() after [...arr]
    }]
  }
}

// With type info, the plugin can check:
// - Is this variable's type readonly?
// - Does this function's return type match expectations?
// - Is this parameter typed as ReadonlyArray?

Type-aware mode is slower (requires a full TypeScript program) but can distinguish ReadonlyArray from Array. Most rules work without type info; a few benefit from it.

2.2 What It Catches vs What It Misses

Click each scenario to see the analysis:

CAUGHT Direct mutation: user.name = "new"
immutable-data sees an AssignmentExpression with a MemberExpression on the left. Simple AST pattern match. Works 100% of the time for direct property assignment.
CAUGHT Mutable declaration: let x = 5
no-let checks VariableDeclaration.kind. Trivial and reliable. Forces const everywhere.
CAUGHT Mutating methods: arr.push(item)
immutable-data has a list of known mutating methods (push, pop, shift, unshift, splice, sort, reverse, fill, copyWithin). Checks CallExpression where callee is a MemberExpression with a known mutating method name.
PARTIAL Mutation through function call: modify(user)
The plugin cannot look inside modify to see if it mutates user. It only checks the call site's syntax. If modify does user.name = "x" internally, the plugin catches it inside that function but cannot warn at the call site.
MISSED External library mutation: _.set(obj, 'path', value)
Lodash's _.set mutates objects, but from the AST it's just a function call. The plugin has no knowledge of third-party function semantics. This is fundamentally beyond syntactic analysis.
MISSED Aliased mutation: const ref = user; ref.name = "x"
immutable-data does catch ref.name = "x" — but it doesn't know that ref aliases user. If your rule config exempts certain variables, the aliasing can create gaps. No cross-reference tracking.

Part 3: How Idris 2 Compiles Dependent Types

Idris 2 is a dependently-typed functional programming language where types can contain values and values can appear in types. It's the most concrete implementation of the "programs as proofs" idea in a general-purpose language. Its compiler pipeline is radically different from TypeScript's.

PTerm — Parsed Surface Syntax

-- Source code:
append : Vect n a -> Vect m a -> Vect (n + m) a
append Nil       ys = ys
append (x :: xs) ys = x :: append xs ys

-- PTerm is a direct representation of what you wrote.
-- Implicit arguments are NOT yet filled in.
-- Pattern matching is NOT yet compiled.
-- Type annotations are taken at face value.

The parser produces PTerm — a faithful representation of the source. No type checking has happened yet. This is analogous to TypeScript's raw AST from the parser phase.

TTImp — Type-annotated Term with Implicits

-- The desugarer fills in implicit arguments:
append : {n : Nat} -> {m : Nat} -> {a : Type}
       -> Vect n a -> Vect m a -> Vect (n + m) a

-- Where you wrote just Vect n a, Idris inferred that
-- n : Nat, m : Nat, a : Type must be implicit parameters.
-- Pattern clauses get case trees.
-- do-notation is desugared to >>= chains.
-- interface methods get dictionary-passing.

TTImp adds holes (metavariables) where the elaborator will solve for implicit arguments. It's the bridge between human-friendly syntax and the core type theory.

TT — The Core Type Theory (QTT)

-- Core TT is Quantitative Type Theory (QTT).
-- Every binding has a MULTIPLICITY:
--   0 = erased at runtime (types, proofs)
--   1 = used exactly once (linear)
--   ω = used any number of times (unrestricted)

-- append becomes (showing multiplicities):
append : {0 n : Nat} -> {0 m : Nat} -> {0 a : Type}
       -> {ω} Vect n a -> {ω} Vect m a -> Vect (n + m) a

-- n, m, a have multiplicity 0: they're erased at runtime!
-- The Vect arguments have multiplicity ω: used freely.

-- ELABORATION happens here:
-- 1. Unification solves metavariables (fills in implicits)
-- 2. Type checking verifies dependent types
-- 3. Totality checking (coverage + termination)

This is where the real work happens. The elaborator converts TTImp into TT by solving unification problems. Each metavariable (hole) becomes a constraint. The unifier finds values that satisfy all constraints simultaneously. If it can't, you get a type error.

CExp — Compiled Expressions (Erased)

-- After type checking, erase everything with multiplicity 0:
append xs ys =
  case xs of
    Nil       => ys
    Cons x xs => Cons x (append xs ys)

-- n, m, a are GONE. They only existed for type checking.
-- Vect is now just a linked list at runtime.
-- The length guarantee is enforced at compile time,
-- then erased — zero runtime cost.

CExp is an untyped lambda calculus with case expressions and primitive operations. All type information, proofs, and 0-multiplicity arguments are erased. This is crucial: dependent types have no runtime overhead.

Backend Code Generation

-- Idris 2 supports multiple backends:

Chez Scheme (default)  — Fastest, best supported
   CExp → Scheme S-expressions → compiled by Chez

JavaScript / Node.js  — Web target
   CExp → JS functions, algebraic types become objects

C (RefC)              — Reference-counted C
   CExp → C code with explicit memory management

Racket                — Alternative Scheme
   CExp → Racket code

The backend is pluggable. CExp is backend-agnostic. Each backend translates the same untyped core into its target language. You could write a WASM or LLVM backend against the same CExp interface.

3.2 Elaboration: The Heart of Idris

Elaboration is where Idris does what no other mainstream compiler does — it solves equations between types and values simultaneously.

Unification — Solving Type Equations

-- You write:
example : Vect 3 Int
example = [1, 2, 3]

-- Idris sees [1,2,3] as syntactic sugar for:
-- 1 :: 2 :: 3 :: Nil
-- which has type: Vect (1 + (1 + (1 + 0))) Int

-- Unification must solve:
-- Vect (1 + (1 + (1 + 0))) Int  ≡  Vect 3 Int
-- So it reduces: 1 + (1 + (1 + 0)) → 1 + (1 + 1) → 1 + 2 → 3  ✓

-- If you wrote [1, 2]:
-- Vect 2 Int ≡ Vect 3 Int → ERROR: 2 ≠ 3

The unifier evaluates expressions inside types. This is the key difference from TypeScript. In TS, types and values are separate worlds. In Idris, the unifier can reduce 1 + 2 to 3 during type checking because + on Nat is a total function defined by pattern matching.

Critical constraint: The unifier can only evaluate total functions. If a function might loop forever, it can't be used in types (or the type checker itself might loop).

Totality Checking

-- Idris checks two things for totality:

-- 1. COVERAGE: are all patterns handled?
head : Vect (S n) a -> a    -- S n means "at least 1"
head (x :: _) = x             -- ✓ only possible pattern for S n
-- No Nil case needed! The type rules it out.

-- 2. TERMINATION: does recursion decrease?
-- Uses SIZE-CHANGE PRINCIPLE:
-- At least one argument must get structurally smaller
-- in every recursive call.

map : (a -> b) -> Vect n a -> Vect n b
map f Nil       = Nil
map f (x :: xs) = f x :: map f xs
--                             ^^ structurally smaller ✓

-- This would FAIL totality:
bad : Nat -> Nat
bad n = bad (n + 1)   -- n+1 is LARGER, not smaller! ✗

Totality checking ensures the type checker terminates. Functions marked total must handle all inputs and provably terminate. Functions used in types MUST be total — otherwise type checking could loop forever.

Quantitative Type Theory (QTT)

-- Every variable binding has a multiplicity:

-- 0: compile-time only, erased at runtime
id : {0 a : Type} -> a -> a
id x = x
-- 'a' is erased. At runtime, id is just: λx. x

-- 1: used exactly once (linear types)
consume : (1 _ : FileHandle) -> IO ()
consume fh = closeFile fh
-- fh must be used exactly once. Can't forget to close!
-- Can't use twice (no aliasing of resources)!

-- ω: unrestricted (normal usage)
duplicate : String -> String
duplicate s = s ++ s   -- s used twice, that's fine

QTT gives Idris something no other mainstream language has: the ability to enforce at the type level that resources are used exactly once. This subsumes Rust's borrow checker concept but as part of the type theory rather than a separate system.

Auto-Implicit Search

-- Idris can auto-search for values that fill implicit args:

interface Show a where
  show : a -> String

print : Show a => a -> IO ()
--      ^^^^^^^^^^ auto-implicit: Idris searches for an instance

-- When you write:
print 42
-- Idris elaborates to:
print {a = Int} @{showInt} 42
-- It found: a must be Int (from 42), and there's a Show Int instance.

-- This extends to proofs:
safeIndex : (i : Fin n) -> Vect n a -> a
-- Fin n can only hold values 0..n-1
-- So safeIndex is total: no out-of-bounds possible!

Auto-implicits are how Idris manages the complexity of dependent types. You write high-level code; the elaborator fills in the type-level bookkeeping automatically. It's like TypeScript inference but operating on values inside types.

3.4 Dependent Types: What They Enable

1 Length-indexed vectors: Vect n a 
append : Vect n a -> Vect m a -> Vect (n + m) a
-- The TYPE guarantees the output length is the sum of inputs.
-- This is checked at COMPILE TIME and erased at runtime.

zip : Vect n a -> Vect n b -> Vect n (Pair a b)
-- Both vectors MUST have the same length. Enforced by types.

-- In TypeScript you'd write:
// function zip(a: Array<A>, b: Array<B>): Array<[A,B]>
// No length guarantee. Could crash at runtime.
2 Type-safe printf: printf : (fmt : String) → Fmt fmt 
-- The return type DEPENDS on the format string value:
printf "%d"     : Int → String
printf "%s %d"  : String → Int → String
printf "%s %s"  : String → String → String

-- The type checker parses the format string AT COMPILE TIME
-- and generates the correct function type.
-- Wrong number/type of arguments = compile error.
3 Provably correct state machines 
data DoorState = Open | Closed

openDoor  : Door Closed -> Door Open
closeDoor : Door Open   -> Door Closed
knock     : Door Closed -> Door Closed

-- You literally CANNOT open an already-open door.
-- The type system rejects it at compile time.
-- openDoor (openDoor d) → TYPE ERROR
4 Proofs as programs 
-- Prove that addition is commutative:
plusCommutative : (n : Nat) -> (m : Nat) -> n + m = m + n
-- This is a TYPE. A value of this type IS the proof.
-- If you can write a total function with this signature,
-- you've proven the theorem.

-- The proof (simplified):
plusCommutative Z     m = plusZeroRightNeutral m
plusCommutative (S k) m =
  rewrite plusCommutative k m in Refl

-- This proof is ERASED at runtime (multiplicity 0).
-- Zero cost. The compiler verified it, then threw it away.

4.3 Where KindScript Fits

KindScript occupies a unique niche: it does AST-level semantic analysis without requiring code changes (unlike Effect-TS) and goes beyond pure syntax matching (unlike ESLint). The gap to fill:

eslint-plugin-functional — "you used let, that's not allowed"
KindScript — "this function claims to be pure, but it calls db.save()"
Effect-TS — "this function's type says it requires Database, provide it via Layer"
