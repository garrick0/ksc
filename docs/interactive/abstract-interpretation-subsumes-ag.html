1The Core Insight

Attribute grammars are abstract interpreters wearing a tree-shaped disguise.

Both frameworks share the same mathematical foundation: monotone functions over lattices computing fixpoints. The Knaster-Tarski theorem guarantees convergence in both cases. But abstract interpretation operates over arbitrary program semantics, while attribute grammars are constrained to tree-structured computations.

An attribute grammar can be viewed as an abstract interpreter where four things have been constrained:

ABSTRACT INTERPRETER

Operates over arbitrary program models — CFGs, call graphs, traces, transition systems. Transfer functions can reference any program point. Abstract domains can be of infinite height. Information flows along any dependency edge.

⊇

ATTRIBUTE GRAMMAR

Program model restricted to a tree. Transfer functions (equations) can only reference parent, children, and siblings. Domains are typically finite-height. Information flows in two directions: synthesized (up) and inherited (down).

Key insight: Every attribute grammar is an abstract interpretation — one where the program model is a tree, the abstract domain is the product lattice of all attribute value domains, and the transfer functions are tree-local semantic equations. The reverse does not hold: most abstract interpretations are not attribute grammars, because they require non-tree information flow.

The Shared Foundation

Both frameworks operate in the same mathematical universe:

ConceptIn Both Frameworks
Complete lattice (L, ≤)The value domain where computations happen
Monotone function F: L → LThe semantic equations / transfer functions
Least fixpoint lfp(F)The analysis result (Knaster-Tarski guarantees existence)
Ascending chain ⊥ ≤ F(⊥) ≤ F(F(⊥)) ≤ ...The iterative computation strategy
SoundnessThe abstract result safely approximates concrete behavior

The difference is not in what they compute (fixpoints of monotone functions), but in where and how the functions are organized. AGs organize them on a tree; AIs organize them on an arbitrary graph.

2The Cousot Semantic Hierarchy

Every program semantics is an abstraction of the one above it. Cousot showed the entire tower is connected by Galois connections.

Cousot's 2002 paper "Constructive Design of a Hierarchy of Semantics of a Transition System by Abstract Interpretation" constructs a complete hierarchy where each level is obtained from the previous by a composable Galois connection, and all semantics are presented in uniform fixpoint form. Click each level to see where AGs fit.

MOST CONCRETE Maximal Trace Semantics

The most concrete semantics. Records every possible execution trace of the program — every state transition, every branch taken. Perfect precision, but undecidable for most properties. This is what actually happens when the program runs.

ABSTRACTION Reachability / Collecting Semantics

Collects the set of all possible states at each program point, across all inputs. Still perfectly precise but generally uncomputable. This is the foundation for deriving abstract interpretations — every AI is an approximation of this layer.

AI LIVES HERE Abstract Interpretation

Abstract interpreters approximate the collecting semantics via a Galois connection (α, γ). They trade precision for decidability. The abstract domain can be anything — intervals, polyhedra, octagons, signs — and the program model can be a CFG, call graph, or any transition system. This is the broadest practical framework for static analysis.

AG LIVES HERE Denotational Semantics / Attribute Grammars

Attribute grammars sit here. They compute tree-structured meanings via fixpoint equations — a compositional, syntax-directed abstraction. Each AG is equivalent to a denotational semantics for the grammar. The key constraint: information flows along tree edges (parent-child), not arbitrary program edges. This makes AGs a subset of abstract interpretations — the subset restricted to tree-structured program models and tree-local transfer functions.

MORE ABSTRACT Type Systems

Cousot proved in 1997 ("Types as Abstract Interpretations", POPL) that type systems are abstract interpretations. Type inference is a fixpoint computation within the AI framework. This connects to AGs because type checking and inference are commonly implemented as attribute grammars.

MOST ABSTRACT Axiomatic Semantics (Hoare Logic)

The most abstract layer — reasoning about programs via pre/post-conditions and invariants. Each assertion abstracts away all execution details and captures only the logical properties that hold. This too is an abstract interpretation, connected to all layers above by Galois connections.

The composability principle: Every pair of adjacent levels is connected by a Galois connection (α, γ). These connections compose: you can chain them to derive any analysis from any semantics. AGs are obtained by composing the abstraction from collecting semantics to denotational semantics with the restriction to tree-structured models.

3Six Degrees of Freedom

An abstract interpreter has six parameters. Constraining each one moves you from AI toward AG.

Degree 1: Program Representation

AI ALLOWS

Arbitrary structures: control-flow graphs with back-edges and loops, call graphs for interprocedural analysis, execution traces, transition systems, Kripke structures. The program model can have cycles, cross-edges, and unbounded fan-out.

→

AG CONSTRAINS TO

A tree derived from a context-free grammar. No cycles, no cross-edges. Every node has exactly one parent (except the root). The structure is determined by the parser.

What you lose: The ability to express loop back-edges, interprocedural call/return edges, and non-local dependencies. Information that would flow through loops must be expressed as circular attributes or approximated away.

Degree 2: Abstract Domain

AI ALLOWS

Any complete lattice, including those of infinite height: intervals [a,b], convex polyhedra, octagons, symbolic expressions. Convergence ensured via widening (overapproximate to force termination) and narrowing (refine the overapproximation).

→

AG CONSTRAINS TO

Product of per-attribute domains. Classical AGs: any computable domain works (no iteration needed due to DAG deps). Circular AGs: must be finite-height lattices — iteration terminates naturally in at most height(L) steps, no widening needed.

The widening gap: AGs traditionally avoid infinite-height lattices entirely. This is a real expressiveness loss — you can't directly express interval analysis, octagon analysis, or polyhedra in a classical AG. Circular AGs recover some of this by allowing finite-height iteration, but still can't handle infinite ascending chains.

Degree 3: Transfer Functions

AI ALLOWS

Arbitrary monotone functions between abstract states at any program point. A transfer function at point p can reference the abstract state at any other point q — there's no locality constraint. This enables interprocedural, context-sensitive analysis.

→

AG CONSTRAINS TO

Tree-local semantic equations. An equation at node n can only reference attributes of n, its parent, its children, and its siblings. No long-range dependencies. Reference attributes (JastAdd) partially relax this.

In AG literature, this is called the locality principle. It's what makes AGs compositional and modular — each production can be analyzed independently. But it's also the key restriction that prevents AGs from expressing many classical data-flow analyses.

Degree 4: Information Flow Topology

AI ALLOWS

Information flows along any dependency edge: forward along control flow, backward along branches, across procedure boundaries, through aliased pointers, along data dependencies. The flow graph can be arbitrary.

→

AG CONSTRAINS TO

Exactly two directions on tree edges: synthesized (child → parent, bottom-up) and inherited (parent → child, top-down). Collection attributes add a third: fold over children (⊕ aggregation). All flows follow the tree structure.

This is perhaps the most fundamental constraint. A CFG has back-edges (loops), cross-edges (gotos), and interprocedural edges (calls). An AG's tree has none of these. Any analysis requiring non-tree flow must either (a) use circular attributes to simulate iteration, (b) use reference attributes to create virtual non-tree edges, or (c) accept that it cannot be expressed.

Degree 5: Fixpoint Strategy

AI ALLOWS

Chaotic iteration (Bourdoncle), worklist algorithms with priority queues, widening/narrowing for acceleration, abstract acceleration for loops, strategic iteration sequences based on weak topological ordering.

→

AG CONSTRAINS TO

Non-circular: single evaluation pass in dependency order (topological sort). Circular: iterative fixpoint over finite lattices (converges in finite steps). Demand-driven: lazy computation with memoization (e.g., KSC's AGNode.attr()).

The tree structure enables simpler evaluation strategies. Multi-visit AGs (Knuth, Kennedy-Warren) make fixed passes over the tree. Demand-driven AGs (like KSC) compute lazily. Neither needs widening because the dependency structure is either acyclic (non-circular) or bounded (finite-height circular).

Degree 6: Scope of Analysis

AI ALLOWS

Whole-program analysis (global), interprocedural (function summaries, call strings, call-site sensitivity), modular (analyze modules independently and compose). Can track properties across compilation units.

→

AG CONSTRAINS TO

Single tree instance. The AG processes one derivation tree at a time. Cross-module analysis requires either a single combined tree or external composition mechanisms. Each tree is evaluated independently.

This is a practical, not fundamental, limitation. Modern AG systems (JastAdd, Silver) support modular extensions via forwarding and aspects. But the core evaluation model is always one tree at a time.

The parameterization thesis: An abstract interpreter is parameterized by these six choices. An attribute grammar arises when you make the most constrained choice for each parameter: tree model, product domain, local equations, syn/inh flow, tree-walk fixpoint, single-tree scope. Relaxing any constraint moves you back toward general AI.

4The Spectrum

From concrete execution to attribute grammars — each step trades expressiveness for tractability.

LEVEL 0 Concrete Interpreter

Executes the program on concrete inputs. Perfect precision — you know exactly what happens. But undecidable for general properties (Rice's theorem). You can't check "does this hold for all inputs?" by running on one input.

+ Perfect precision + Concrete semantics - Undecidable for non-trivial properties - Only covers inputs you test
LEVEL 1 Collecting Semantics

Collects the set of all possible concrete states at each program point, across all inputs. Still perfectly precise but generally uncomputable. This is the "ideal" that abstract interpretation approximates.

+ Covers all inputs + Still perfectly precise - Uncomputable in general
LEVEL 2 General Abstract Interpreter

Approximates collecting semantics via a Galois connection (α, γ). Operates over abstract domains (lattices). Can handle arbitrary program structures: CFGs, interprocedural graphs, traces. May require widening/narrowing for convergence over infinite-height lattices.

This is where tools like Infer (Facebook), Astrée (Airbus), and Polyspace (MathWorks) operate.

+ Decidable / computable + Sound (safely approximates) + Arbitrary program structure - Some precision lost (overapproximation)
LEVEL 3 Tree-Restricted AI = Circular Attribute Grammar

Restricts the program model to a tree and transfer functions to tree-local equations. This is a circular attribute grammar. Still allows iterative fixpoint computation over finite-height lattices. Systems like JastAdd live here.

This is the critical boundary: by constraining to trees, you lose the ability to express loop back-edges, interprocedural flow, and non-tree dependencies. But you gain compositional, syntax-directed organization.

+ Syntax-directed / compositional + Modular equation specification + Finite iteration guaranteed - No non-tree dependencies - No interprocedural flow
LEVEL 4 Classical Non-Circular AG (KSC lives here)

Further restricts to acyclic attribute dependencies. No iteration needed — a single evaluation pass in dependency order suffices. Demand-driven evaluation with memoization (like KSC's AGNode.attr()) is equivalent. This is the most constrained point on the spectrum.

KSC is here: its _cyc set detects and forbids circular dependencies at runtime. Attributes form a DAG, and evaluation is single-pass demand-driven with caching.

+ Single-pass evaluation + Simple, fast, no iteration + No convergence concerns - No circular attribute dependencies - Can't express iterative analyses

5Galois Connections & Attribute Domains

The formal machinery connecting concrete program behavior to abstract attribute values.

A Galois connection (α, γ) between a concrete domain C and an abstract domain A provides the soundness guarantee that makes abstract interpretation work:

// The Galois connection:
for all c ∈ C, a ∈ A:
    α(c) ≤A a  ⇔  c ≤C γ(a)

// Equivalently:
α: C → A    // abstraction: maps concrete to abstract
γ: A → C    // concretization: maps abstract back to concrete
α ∘ γ ≤ id   // abstracting then concretizing loses nothing (or overapproximates)

Mapping to AG Concepts

Abstraction α AG equations (compute attrs from tree) eq_kindDefs_CompilationUnit, eq_violationFor_Identifier
Transfer function Semantic equation at a tree node (ctx: Ctx) => KindDefinition[]
Fixpoint lfp(F) Solution of all attribute equations The completed AGNode._cache after evaluation
Observation fn Projection projections.definitions(root), projections.diagnostics(root)

The Galois Connection in KSC's Kind-Checking

Consider KSC's violationFor attribute. The Galois connection is:

// Concrete domain: all possible runtime behaviors of a code fragment
// (does it import? does it mutate? does it have side effects?)

// Abstract domain: Diagnostic | null
// (either a violation exists or it doesn't)

// α: concrete behavior → Diagnostic | null
//   "this identifier resolves to an import"  →  Diagnostic("noImports violated")
//   "this identifier is local"               →  null (no violation)

// Soundness: if α says null, no violation truly exists
// Overapproximation: α might flag things that aren't true violations
//   (sound = no false negatives, may have false positives)

Key insight: The α function in KSC is implicit — it's the composition of the AST translator (which maps a TS program to a KSC tree, losing some runtime information) with the equation functions (which compute attribute values from the tree). The information loss from ts.Program → KSProgram → attribute values is the Galois abstraction.

KSC's Eight Attributes as Abstract Domains

Each of KSC's eight attributes defines an abstract domain. Click each to see the lattice structure:

S kindDefs : KindDefinition[] ▼

Direction: Synthesized (child → parent)

Abstract domain: Lists of kind definitions. Flat lattice — each node computes its value independently, no join/meet needed. Only CompilationUnit nodes produce non-empty values.

AI parallel: Like a collecting semantics restricted to "what type aliases does this subtree define?" — a syntactic property that requires no iteration.

I defEnv : Map<string, KindDefinition> ▼

Direction: Inherited (parent → child, copy-down)

Abstract domain: Environment map. The root computes it from child kindDefs; all other nodes inherit unchanged. This is a top-down propagation — the AI equivalent of propagating an abstract environment down the call stack.

AI parallel: Exactly like context propagation in a context-sensitive analysis, but constrained to tree-local copy-down (no call-string sensitivity).

S violationFor(property) : Diagnostic | null ▼

Direction: Synthesized, parameterized (JastAdd-style)

Abstract domain: Diagnostic | null — a two-point lattice {null ≤ Diagnostic}. Each node either has a violation for the given property or doesn't.

AI parallel: Like a must-not-happen analysis: "does this program point violate property P?" The parameterization by property is like a family of abstract interpretations indexed by a configuration parameter — one per property.

Dependency: Calls ctx.attr('contextFor', property) — the inherited attribute provides the kind context that determines which violations to check.

C nodeCount : number ▼

Direction: Collection (fold over children)

Abstract domain: Natural numbers under addition. init = 1, combine = (acc, child) => acc + child.

AI parallel: A counting abstraction — the simplest possible abstract domain. In AI terms, this is like a program-size metric computed by abstract interpretation over a trivial lattice (ℕ, ≤).

Pattern recognition: KSC's attributes fall into exactly the categories predicted by the AI-AG correspondence: synthesized = bottom-up data flow, inherited = top-down context propagation, collection = fold/reduce over children. The parameterized attributes (contextFor, violationFor) are a family of analyses indexed by a parameter — like running multiple abstract interpretations simultaneously.

7Constructing KSC by Constraining an AI

Start with a general abstract interpreter. Apply six constraints. Arrive at KSC.

This section walks through how you would start with a fully general abstract interpreter and progressively constrain it until you arrive at KSC's current attribute grammar system. At each step, a concrete code snippet from KSC shows the constraint in action.

Step 1: Choose Tree Representation

A general AI operates over a CFG or transition system. KSC constrains to an AST tree via Grammar<K>:

// The Grammar port constrains program model to a tree
interface Grammar<K extends string> {
  readonly fieldDefs: Record<string, FieldDef[]>;  // tree structure
  readonly allKinds: ReadonlySet<K>;              // finite node types
  readonly rootKind: K;                            // single root
}

// This means: the program IS a tree, each node has a kind from K,
// children are determined by fieldDefs, and there is one root.
// No cycles. No back-edges. No cross-edges. Just a tree.

What an AI could do instead: Operate on ts.Program directly via its control-flow graph, tracking state at each branch/join point. KSC discards this structure by converting to a KS tree first.

Step 2: Choose Attribute Domains

A general AI uses arbitrary lattices. KSC defines per-attribute types in AttrDecl:

// Each AttrDecl defines an attribute's domain
{
  name: 'kindDefs',
  direction: 'syn',
  type: 'KindDefinition[]',    // domain: lists of definitions
  default: eq_kindDefs_default  // ⊥ = empty list
}

{
  name: 'nodeCount',
  direction: 'collection',
  type: 'number',               // domain: natural numbers
  init: 1,                       // each node contributes 1
  combine: code('(a, b) => a + b') // join: addition
}

// No widening operator. No infinite ascending chains.
// Every domain is implicitly finite-height for KSC's spec.

What an AI could do instead: Use interval domains [lo, hi] with widening to track numeric ranges, or polyhedra to track linear relationships between variables.

Step 3: Constrain Transfer Functions to Tree-Local Equations

A general AI has transfer functions that can reference any program point. KSC equations can only see parent, children, and the node itself:

// An equation can only access: ctx.node, ctx.parent, ctx.children, ctx.attr()
const eq_violationFor_Identifier = withDeps(['contextFor'],
  function eq_violationFor_Identifier(
    ctx: KindCtx<KSIdentifier>,
    property: string
  ): Diagnostic | null {
    const kindCtx = ctx.attr('contextFor', property);  // can access attrs
    if (!kindCtx) return null;

    if (property === 'noImports' && ctx.node.resolvesToImport) {
      return diag(ctx, kindCtx, property, '...');
    }
    // Can't reach out to "the function where this identifier is defined"
    // Can't check "the call site that invoked this function"
    // Tree-local: only see this node + its attr dependencies
  }
);

Step 4: Constrain Information Flow to syn/inh/collection

A general AI flows information along any edge. KSC restricts to three directions, encoded in DispatchEntry:

// The DispatchConfig encodes the flow constraint:
type DispatchEntry =
  | { direction: 'syn',        compute: (ctx) => T }           // ↑ child → parent
  | { direction: 'inh',        computeRoot, computeParent }    // ↓ parent → child
  | { direction: 'collection', init, combine }                 // ⊕ fold children

// Three flow patterns, all tree-local:
//   syn:        node computes from own fields + child attrs
//   inh:        node receives from parent (or root default)
//   collection: node folds child values with combine

// A general AI would also have:
//   - backward flow (through loop back-edges)
//   - cross-procedure flow (call/return edges)
//   - alias-based flow (pointer dereference)

Step 5: Choose Demand-Driven Fixpoint with No Iteration

A general AI uses chaotic iteration with widening. KSC uses lazy demand-driven evaluation with cycle detection:

// From AGNode.attr() in engine.ts:
attr(name: string, ...args: unknown[]): any {
  // 1. Check cache (memoization, not iteration)
  if (this._cache.has(name)) return this._cache.get(name);

  // 2. Detect cycles (FORBID circular deps)
  if (this._cyc.has(key)) throw new Error(`Circular: ${name}`);
  this._cyc.add(key);

  // 3. Compute once, cache forever
  const result = dispatch(name, this, ...args);
  this._cache.set(name, result);
  return result;
}

// A general AI would instead:
//   while (!stable) {
//     for (point in worklist) {
//       newState = transfer(point, currentState);
//       if (newState != oldState) { propagate; add successors; }
//     }
//     if (diverging) { widen(); }
//   }

Step 6: Restrict to Single-Tree Scope

A general AI can analyze multiple compilation units, track inter-module flow, and compose results. KSC evaluates one tree at a time:

// From packages/core/index.ts:
function createProgram(rootNames, config, options) {
  // 1. Parse TS → one big AST tree
  const tsProgram = ts.createProgram(rootNames, options);

  // 2. Convert to KSC tree (still one tree)
  const ksTree = tsToAstTranslatorAdapter.convert(tsProgram);

  // 3. Evaluate all attributes on this single tree
  const result = evaluator.evaluate(ksTree.root);
  //                                  ^^^^^^^^ one root, one tree

  // No cross-module analysis. No incremental re-evaluation.
  // Each call creates a fresh tree and evaluates from scratch.
}

The construction is complete: Apply all six constraints and you arrive at exactly KSC's current architecture. Grammar<K> → tree model. AttrDecl → finite domains. Equation functions → tree-local transfer functions. syn/inh/collection → constrained flow. AGNode._cyc → no iteration. createProgram() → single tree scope.

8The Reverse Direction

Given a general abstract interpreter, how do you extract an attribute grammar from it?

The "forward" direction (constraining an AI to get an AG) is conceptually clean. The reverse direction — starting with an abstract interpreter and extracting/generating an AG from it — is harder and less explored, but the theory exists.

The Extraction Procedure

1 Identify Tree Structure ▼

Find the AST or parse tree inside the AI's program model. Most abstract interpreters start from an AST anyway — the question is whether the analysis can be factored to operate on the tree rather than the CFG derived from it.

The key question: Does the analysis require non-tree edges? If yes, it cannot be fully expressed as a classical AG. If it only uses tree edges + a bounded number of inherited/synthesized passes, extraction is possible.

2 Factor Transfer Functions ▼

Decompose per-program-point transfer functions into per-node-kind equations. Each transfer function at a tree node n must be expressible in terms of n's fields, its parent's attributes, and its children's attributes.

If a transfer function references a distant node (e.g., the entry of an enclosing function), you need to thread that information through inherited attributes.

3 Classify Information Flow ▼

Determine which facts flow bottom-up (→ synthesized) vs. top-down (→ inherited):

Bottom-up: Information computed from subtrees and passed to parents. Examples: computed types, collected declarations, node counts.

Top-down: Context information passed from parents to children. Examples: type environments, enclosing scope, configuration.

Collection: Aggregation of child contributions into a single value. Examples: total counts, gathered warnings.

4 Check Acyclicity ▼

Verify that inter-attribute dependencies form a DAG (for non-circular AGs) or are all over finite-height lattices (for circular AGs). If the original AI has cycles over infinite domains with widening, the extraction must either:

  • Replace infinite domains with finite approximations
  • Accept circular AGs (JastAdd-style)
  • Accept that the analysis cannot be expressed as an AG
5 Generate AG Spec ▼

Output the AnalysisSpec from the factored transfer functions. Each transfer function becomes an equation attached to a node kind, with a declared direction (syn/inh/collection) and declared dependencies.

In KSC terms, you would produce AttrDecl[] objects with equation functions, plus projections to extract the final results.

Who Has Done This?

1990 – 1992
Rosendahl
The most complete treatment. His PhD thesis "Abstract Interpretation and Attribute Grammars" showed that abstract interpretations can be implemented using circular AGs with a domain-theoretic evaluator. He showed how to take an AI and express it in AG form, and how AI correctness methods strengthen AG proofs.
1995 – 2011
Cousot & Cousot
Showed grammar-based and set-constraint-based analyses are abstract interpretations with iterative fixpoint computation. In 2011, constructed a hierarchy of grammar semantics as abstract interpretations, placing Knuth's grammar problem and classical grammar flow analysis in the AI framework.
2007
Magnusson & Hedin (JastAdd)
Practical demonstration: JastAdd's Circular Reference AGs implement data-flow analyses (nullable, first/follow, devirtualization) as circular attributes with fixpoint evaluation — effectively implementing AI-style analyses in AG form. The closest to "AI extracted into AG."

The hard part: Not every abstract interpretation can be cleanly extracted into an AG. Analyses that fundamentally require non-tree information flow (pointer analysis, interprocedural constant propagation, loop-sensitive analyses) cannot be expressed as classical AGs. They require either reference attributes (virtual non-tree edges) or must remain as general AIs.

A chronological map of the research connecting abstract interpretation and attribute grammars.

Foundational Work

1968
Knuth — Attribute Grammars
"Semantics of Context-Free Languages". The original paper defining synthesized and inherited attributes, semantic equations, and the evaluation problem. Established AGs as a formalism for defining language semantics.
1977
Cousot & Cousot — Abstract Interpretation
"Abstract interpretation: a unified lattice model for static analysis of programs" (POPL). The foundational paper establishing AI's lattice-theoretic framework. Introduced Galois connections as the soundness mechanism.
1977
Kam & Ullman — Monotone Frameworks
"Monotone Data Flow Analysis Frameworks". Proved that the maximal fixpoint solution exists for every instance of every monotone framework — the mathematical foundation shared by both AI and AG evaluation.

Direct AI–AG Connections

1990
Rosendahl — AI Using AGs
"Abstract Interpretation Using Attribute Grammars" (Springer LNCS). Conference paper showing AI can be implemented via circular AGs. Precursor to his 1992 thesis.
1992
Rosendahl — PhD Thesis
"Abstract Interpretation and Attribute Grammars". 134 pages establishing three results: (1) AIs can be implemented using circular AGs, (2) lazy fixpoint iteration specializes AGs for finite lattices, (3) AI methods strengthen AG correctness proofs.
1997
Cousot — Types as AI
"Types as Abstract Interpretations" (POPL). Proved type systems are abstract interpretations. Since type checking is often implemented via AGs, this closes a triangle: types ⊂ AI ⊃ AG.
2011
Cousot & Cousot — Grammar Semantics as AI
"Grammar Semantics, Analysis and Parsing by Abstract Interpretation" (TCS). Constructed a hierarchy of bottom-up and top-down grammar semantics, placing Knuth's grammar problem and classical grammar flow analysis as AIs of a fixpoint protoderivation semantics.

Systematic Abstraction

1993
Bourdoncle — Chaotic Iteration
"Efficient chaotic iteration strategies with widenings". Established practical algorithms for fixpoint iteration in AI. His weak topological ordering is used by every modern AI engine and applies to circular AG evaluation.
2010
Van Horn & Might — AAM
"Abstracting Abstract Machines" (ICFP). Demonstrated systematic derivation of static analyses from concrete interpreters: start with a CEK machine, apply store-allocated continuations, bound the store, get a sound analysis. Shows the concrete→abstract spectrum in action.

Practical Frameworks

2007
JastAdd
Circular Reference Attributed Grammars with demand-driven evaluation. Directly implements AI-style analyses as AG specifications. The closest practical system to "AI expressed as AG." Used for Java compilers (ExtendJ).
2009
DOOP + Soufflé
Declarative pointer analysis in Datalog. Shows that AI-style analyses can be expressed as declarative fixpoint computations over relations — similar in spirit to AGs but over relational structures rather than trees.
2016
Flix
"From Datalog to Flix" (PLDI). Extends Datalog with lattice-valued predicates and monotone transfer functions. Unifies relational (Datalog) and lattice-based (AI) approaches. Predicates carry lattice values joined during fixpoint.
Ongoing
Silver (MELT Group)
Extensible AG system supporting higher-order attributes, reference attributes, forwarding, collections, and aspects. Includes constructs for specifying CFG-based data-flow analyses — directly bridging AG and AI worlds.

10The Unified View

All roads lead to fixpoints over lattices.

The deepest insight from this analysis is that all major frameworks for static analysis are special cases of the same mathematical structure: monotone functions over complete lattices, computing fixpoints. The differences are in the constraints applied.

FrameworkProgram ModelValue DomainConstraint
Abstract Interpretation Arbitrary (CFG, traces) Any lattice None (most general)
Attribute Grammars Tree (AST) Product of attr domains Tree-local equations
Datalog Relations (facts/rules) Sets (powerset lattice) Monotone rules, set-valued only
Flix Relations + lattices Any lattice Declarative rules
Type Systems Tree (AST) + constraints Type lattice Syntax-directed, compositionality
Model Checking Kripke structure Temporal logic lattice Finite-state, temporal properties
KSC Tree (Grammar<K>) 8-attr product Non-circular, demand-driven

The unification: KSC's kind-checking system is a maximally constrained abstract interpretation — tree-structured, non-circular, demand-driven, single-tree scope. Every relaxation of a constraint moves it toward general AI and enables new analyses. The architecture (ports-and-adapters, Grammar<K>, AnalysisSpec<K,P>) is already structured to accommodate many of these relaxations without rewriting the core.

The practical takeaway: KSC's ports-and-adapters architecture, where generic machinery lives in libs/ and pluggable adapters in adapters/, is already the right shape for an abstract interpreter parameterized by domain choice. The AnalysisSpec port is essentially the "analysis configuration" that a parameterized AI takes as input. The evaluator engine is the fixpoint solver. The grammar provides the program model. The only things constraining KSC to the AG subset are design choices in the engine (_cyc throws instead of iterating) and the spec types (no widening operator on AttrDecl).

To move KSC toward general AI, you would:

  1. Replace _cyc detection with iteration (circular AGs)
  2. Add an optional widen operator to AttrDecl (infinite domains)
  3. Add a CFG edge type to AGNode (flow-sensitive analysis)
  4. Support multiple trees with inter-tree edges (interprocedural)

Each step is independent. You can adopt them incrementally without breaking existing specs.

Extended AG systems add features that recover capabilities lost in the AI → AG projection. Each feature re-opens a degree of freedom.

Classical AGs sit at Level 4 on our spectrum — the most constrained point. But real AG systems like JastAdd and Silver have independently discovered that practical analyses demand more. Each extension they added corresponds to partially relaxing one of the six AI → AG constraints, climbing back up the spectrum toward general abstract interpretation.

Deep Dive: Each Feature as a Constraint Relaxation

Circular Attributes (JastAdd) — Relaxing Constraint #5: Fixpoint Strategy

Classical AGs forbid circular dependencies. JastAdd allows them with a Kleene fixpoint iteration identical to abstract interpretation:

// JastAdd circular attribute declaration:
syn Set<String> Nonterminal.FIRST() circular [new HashSet<>()];

// Evaluation algorithm:
// 1. Initialize to bottom: FIRST = {} (empty set)
// 2. Evaluate equation → get new FIRST value
// 3. Compare with previous: changed?
//    Yes → re-evaluate with new value cached
//    No  → stable! Permanently cache.
// 4. Repeat until fixpoint.

// Lattice requirements (identical to AI):
// - Values form a lattice of FINITE HEIGHT
// - Bottom value declared in [...] brackets
// - All equations on cycles must be MONOTONE

This is exactly abstract interpretation's Kleene iteration over a finite-height lattice. The only difference is organizational: the lattice and transfer functions are declared as attribute equations on a tree, not as transfer functions on a CFG.

AI FIXPOINT

state[p] = ⊥
do { state[p] = transfer(p, state) } while (changed)
Converges because lattice has finite height.

=

JASTADD CIRCULAR

cache = bottom
do { cache = equation(ctx) } while (changed)
Converges because lattice has finite height.

What this recovers: The ability to express iterative analyses: NULLABLE/FIRST/FOLLOW sets, reaching definitions, live variables, constant propagation over finite domains. These are classical AI problems now expressible in AG form.

KSC comparison: KSC's AGNode._cyc throws on circular deps. Replacing the throw with JastAdd's iteration loop would move KSC from Level 4 to Level 3 on the spectrum — recovering the fixpoint degree of freedom without changing anything else.

Reference Attributes (JastAdd & Silver) — Relaxing Constraint #4: Information Flow

Classical AGs flow information only along tree edges (parent ↔ child). Reference attributes allow an attribute to point to an arbitrary node anywhere in the tree, creating virtual non-tree edges.

// JastAdd: an IdUse node's 'decl' attribute references its declaration
syn DeclNode IdUse.decl() = lookup(getName());

// Now: other attributes can follow the reference
syn Type IdUse.type() = decl().getType();

// This is a non-tree edge! The IdUse is deep in one subtree;
// its declaration is in a completely different subtree.
// In AI terms: this is like a CFG edge from a use to its def.

Without reference attributes, the use-to-declaration link would have to be threaded through inherited/synthesized attributes up and down the tree — expensive, verbose, and unnatural. With references, you directly follow the edge.

The key equivalence: A tree with reference attributes is semantically a graph. The tree provides the syntactic skeleton; references add semantic edges. This is exactly the relationship between an AST and a CFG: the AST is a tree, the CFG adds back-edges and cross-edges. Reference attributes simulate CFG-like connectivity within the AG framework.

What this recovers: Name resolution, type-checking with inheritance hierarchies, cross-file references, qualified access resolution — any analysis requiring non-local lookup.

Higher-Order Attributes / NTAs — Relaxing Constraint #1: Program Representation

Classical AGs operate on a fixed tree produced by the parser. NTAs (Nonterminal Attributes) allow the AG to construct fresh subtrees during evaluation:

// JastAdd: a MethodDecl creates a CFG as a nonterminal attribute
syn nta CFG MethodDecl.controlFlowGraph() =
    new CFG(buildCFGFromBody(getBody()));

// The CFG subtree is grafted into the tree.
// It gets inherited attributes from its parent (the MethodDecl).
// Its own synthesized attributes flow back out.
// You can define data-flow attributes ON the CFG nodes!

// In Silver: same concept with decorated/undecorated distinction
synthesized attribute cfg :: Decorated CFGNode;
// Undecorated = bare data; Decorated = has inherited attrs, all attrs evaluable

Why this matters for AI: Abstract interpreters often work on derived representations — CFGs, SSA form, call graphs. NTAs let you construct these representations as attributes and then run further analysis on them. The AG evaluator handles the whole pipeline: parse → build CFG (NTA) → compute data-flow attributes on CFG → propagate results back to AST.

This partially relaxes the "tree only" constraint: the original structure is still a tree, but the analysis can create and analyze additional structures derived from it.

Collection Attributes — Relaxing Constraint #4: Non-Local Aggregation

KSC's collection attributes fold over direct children with init + combine. JastAdd and Silver go further:

// JastAdd: any node can contribute to a distant ancestor's collection
ErrorStmt contributes errorMessage()
    when hasError()
    to CompilationUnit.errors();

// A survey traverses the subtree to find all contributors.
// Contributions can be conditional (when), targeted (to), and filtered (for).

// Silver: collection with commutative operator
synthesized attribute errors :: [Message] with ++;
// Base value assigned with :=
// Contributions added with <-
// Operator (++) must be commutative (order-independent)

AI parallel: This is analogous to join operations at CFG merge points — collecting abstract states from all predecessors and merging them. JastAdd's contributes...to...for syntax is particularly powerful: it allows targeted contributions via reference expressions, making it possible for a node deep in the tree to contribute to a collection on any other node it can reference.

KSC COLLECTION

Child-local fold: init + combine(acc, child). Only direct children contribute. Simple, predictable, but limited to parent-child aggregation.

⊂

JASTADD / SILVER

Non-local survey: any node in the subtree (or any node reachable by reference) can contribute. Conditional filtering. JastAdd adds reference-targeted contributions.

Forwarding (Silver) — A Galois Connection in Disguise

Forwarding is Silver's most distinctive feature and has a deep connection to abstract interpretation. A production can forward to a semantically equivalent tree:

// Silver: 'or' forwards to 'not(and(not(l), not(r)))'
production or
e::Expr ::= l::Expr  r::Expr
{
  forwards to not(and(not(l), not(r)));
  // Any synthesized attr NOT explicitly defined on 'or'
  // is computed on the forwarded-to tree instead.
  // Inherited attrs of forwarded-to tree are copied from 'or'.
}

// The key invariant (soundness condition):
//   or.attr == not(and(not(l), not(r))).attr
//   for any attr not explicitly overridden by 'or'

The Galois connection analogy:

GALOIS CONNECTION

α: concrete → abstract
γ: abstract → concrete
Soundness: α(γ(a)) ≥ a
"The abstract result safely approximates."

≅

SILVER FORWARDING

Host semantics ← extension semantics (copy-back)
Invariant: t.s = t.forward.s
"The extension's semantics agree with the host's."

Forwarding is dynamically computed — the forwarded-to tree can depend on attribute values. A plus can forward to intAdd or stringConcat depending on operand types. This is like a context-dependent Galois connection where the abstraction function itself depends on the abstract state.

What it solves: The expression problem for AGs. Two independent extensions (new syntax + new attribute) compose correctly because forwarding provides sound defaults. Neither extension needs knowledge of the other.

Rewrites (JastAdd) — Relaxing Constraint #3: Transfer Functions

JastAdd rewrites allow transparent node replacement during attribute access:

// JastAdd: desugaring as a rewrite
rewrite ForEachStmt {
    when (shouldDesugar())
    to WhileStmt {
        return new WhileStmt(
            buildIteratorCheck(),
            buildLoopBody()
        );
    }
}

// When a parent accesses this child via getChild():
// 1. Check rewrite condition (may depend on attributes!)
// 2. If true: replace node with rewrite result
// 3. Check new node for further rewrites
// 4. Repeat until stable
// 5. Return final, rewritten node

// The parent never sees the original ForEachStmt —
// it only sees the desugared WhileStmt.

AI parallel: Rewrites are abstract transformers — functions that transform the abstract state (the tree) during analysis. In AI, transfer functions transform abstract states at program points; in JastAdd, rewrites transform tree structure during evaluation. The rewrite condition can depend on attributes (type-directed desugaring), making it a context-sensitive transformation — exactly like an AI transfer function that depends on the abstract state.

Confluence concern: JastAdd does not statically check rewrite confluence. If multiple rewrites apply, the order matters. This is analogous to the non-determinism in AI's chaotic iteration — different worklist orderings may visit states in different orders, but the final fixpoint is the same (assuming monotonicity). Rewrite confluence is the AG analogue of this property.

Modular Well-Definedness (Silver) — Compositional Verification

Silver's landmark contribution: independently verify extensions, guarantee correct composition.

// Silver's modComplete(Host, Extension) checks six properties:

// 1. noOrphanOccursOn  — no duplicate attribute occurrence declarations
// 2. noOrphanAttrEqs   — no duplicate or orphaned equations
// 3. noOrphanProds     — extension productions MUST forward
// 4. synComplete       — every syn attr has equations for all non-fwd productions
// 5. modularFlowTypes  — extensions can't change host flow types
// 6. inhComplete       — inherited equations are complete given flow types

// If each extension passes modComplete independently,
// then ALL compositions of extensions are guaranteed:
//   ✓ Complete (no missing equations)
//   ✓ Non-circular (no circular dependencies)

AI parallel: compositional abstract interpretation. This is the AG analogue of modular AI — analyzing components separately and proving that their composition is sound. In AI, this corresponds to function summary computation: analyze each function independently, produce a summary, and verify that summaries compose correctly. Silver's flow types (computed via fixpoint iteration — itself a form of AI over the AG's dependency structure!) are the summaries.

Most AI frameworks lack this guarantee entirely. You can build an abstract interpreter for module A and module B independently, but there's no general mechanism to guarantee their composition is sound. Silver provides this for AGs.

How Extended AGs Map to the Spectrum

Adding these features moves AG systems back up the spectrum toward general AI. Here's where each system sits:

LEVEL 2 General Abstract Interpreter

Arbitrary program model, arbitrary lattices, widening/narrowing, interprocedural scope. Full AI power. No system below reaches this level completely.

LEVEL 3 JastAdd CRAGs — Circular + References + NTAs

JastAdd with all features enabled sits here. Circular attributes provide fixpoint iteration. Reference attributes provide non-tree edges. NTAs allow constructing derived representations (CFGs). Together, CRAGs can express most AI-style analyses that operate on tree-derivable structures. The remaining gap: widening over infinite-height lattices, and true interprocedural scope.

+ Fixpoint iteration (circular) + Non-tree edges (references) + Derived representations (NTAs) - No widening (finite-height only) - No guaranteed safe composition
LEVEL 3.5 Silver — Non-Circular but Compositionally Verified

Silver trades circular attributes for modular well-definedness. It can't express fixpoint analyses directly, but it can construct CFGs as NTAs and delegate fixpoint to external solvers. Its unique value: guaranteed safe composition of independent extensions.

+ Safe modular composition + Forwarding (Galois-like defaults) + Non-tree edges (references) - No circular attributes - No widening
LEVEL 4 KSC — Classical Non-Circular AG

KSC sits at the most constrained level. No circular attributes (_cyc throws), no reference attributes, no NTAs, no forwarding. Demand-driven evaluation with memoization on a single fixed tree. Maximally simple, maximally constrained.

+ Simplest evaluation model + No convergence concerns - No iterative analysis - No non-tree flow - No derived representations

12Deriving JastAdd & Silver from an Abstract Interpreter

Start with a general AI. Apply different constraint profiles. Arrive at JastAdd or Silver.

If you built a general abstract interpreter first, you could derive either JastAdd or Silver by applying different sets of constraints. The derivation is not just theoretical — it reveals what each system chose to keep from AI and what it chose to sacrifice.

The Shared Starting Point

// A general abstract interpreter has:
type AbstractInterpreter<S, A> = {
  programModel:   Graph<S>,            // arbitrary program graph
  abstractDomain: Lattice<A>,           // any complete lattice
  transfer:       (S, A) => A,          // monotone transfer functions
  widen:          (A, A) => A,          // acceleration for infinite chains
  narrow:         (A, A) => A,          // refinement after widening
  scope:          'intra' | 'inter',   // analysis scope
  strategy:       IterationStrategy,     // worklist, chaotic, etc.
}

Deriving JastAdd from an Abstract Interpreter

JastAdd keeps the most AI capabilities, sacrificing only infinite domains and interprocedural scope:

1 Factor the program model into tree + references ▼

Take your AI's program graph (CFG, call graph, etc.) and factor it into:

(a) The AST — the syntactic tree structure. This becomes the grammar in JastAdd terms. Define node types (nonterminals), their children (grammar productions), and the root.

(b) Non-tree edges — CFG back-edges, use-def chains, scope links. These become reference attributes. Each non-tree edge becomes a synthesized or inherited attribute whose value is a reference to another node.

// AI: program graph with nodes and edges
// JastAdd: tree (grammar) + reference attributes (non-tree edges)

// The IdUse→IdDecl edge in the AI's def-use graph becomes:
syn IdDecl IdUse.decl() = lookup(getName());
// ^ reference attribute = non-tree edge as a first-class value

If your AI constructs derived representations (e.g., a CFG from the AST), these become NTAs — nonterminal attributes that create fresh subtrees during evaluation:

syn nta CFG MethodDecl.cfg() = buildCFG(getBody());
// The CFG is constructed as a subtree, then attributed
2 Map abstract domains to attribute types ▼

Each abstract domain in your AI becomes an attribute type in JastAdd. The key constraint: finite-height lattices only.

AI DomainJastAdd Attribute TypeLattice
Sign domain {⊥, Neg, Zero, Pos, ⊤}enum SignHeight 3, finite
Powerset P(Vars)Set<Variable>Height |Vars|, finite
Boolean {false, true}booleanHeight 1, finite
Intervals [a,b]Cannot expressInfinite height — needs widening

For infinite domains: either discretize (lose precision) or accept that JastAdd can't express this analysis.

3 Express transfer functions as attribute equations ▼

Each AI transfer function at a program point becomes an equation on the corresponding tree node. Classify by direction:

Bottom-up flow (results flowing up from subtrees) → synthesized attributes. Example: computed types, collected declarations, error lists.

Top-down flow (context flowing down from parents) → inherited attributes. Example: type environments, enclosing scope, expected type.

Join-point aggregation (merging info from multiple sources) → collection attributes. Example: all errors from a subtree, all declarations in scope.

// AI: transfer function at an if-statement join point
//   state[join] = state[then_branch] ⊔ state[else_branch]

// JastAdd: synthesized attribute with explicit join
syn AbstractState IfStmt.exitState() =
    getThen().exitState().join(getElse().exitState());
4 Declare circular attributes for cycles ▼

Wherever your AI has a cycle (loop back-edge in the CFG, recursive function call), the corresponding attribute equations will form a dependency cycle. In JastAdd, declare these attributes circular:

// AI: liveVars at loop header depends on liveVars at loop end (back-edge)
// JastAdd: circular attribute with set lattice
syn Set<Variable> Stmt.liveVars() circular [new HashSet<>()];

// The empty set is ⊥ (bottom)
// Set union is the join operation
// Height ≤ |Variables| (finite)
// Monotone: each iteration can only ADD variables, never remove

JastAdd's evaluator will automatically iterate until the fixpoint — the same Kleene iteration your AI used, just driven by demand rather than a worklist.

5 Drop widening, drop interprocedural scope ▼

The two things you sacrifice in the derivation:

Widening/narrowing: JastAdd requires finite-height lattices. If your AI uses widening (e.g., interval analysis), you must either discretize the domain or accept precision loss. This is the fundamental gap between JastAdd and general AI.

Interprocedural scope: JastAdd evaluates one tree at a time. Cross-module analysis requires putting everything in one tree (possible but unwieldy for large programs). Your AI's interprocedural capabilities (call-string sensitivity, function summaries) must be expressed within the single-tree model, typically using reference attributes to follow call edges and NTAs to represent summaries.

The result: A JastAdd specification that is a declarative re-expression of your abstract interpreter. Same fixpoint semantics (Kleene iteration), same monotone transfer functions, same finite-height lattice convergence guarantee. The only structural difference: the analysis is organized around a tree (with reference edges for non-tree flow) rather than directly on the CFG.

Concrete Example: Deriving Live Variable Analysis

// STEP 1: AI version — worklist on CFG
for (node in cfg.reversePostOrder()) {
  live[node] = use[node] ∪ (live[successors(node)] \ def[node]);
  if (changed) worklist.add(predecessors(node));
}

// STEP 2: JastAdd version — circular attrs on AST + CFG NTA
syn nta CFG MethodDecl.cfg() = buildCFG(getBody());

syn Set<Variable> CFGNode.liveOut() circular [new HashSet<>()];
eq CFGNode.liveOut() {
    Set<Variable> result = new HashSet<>();
    for (CFGNode succ : successors())   // reference attrs!
        result.addAll(succ.liveIn());
    return result;
}

syn Set<Variable> CFGNode.liveIn() circular [new HashSet<>()];
eq CFGNode.liveIn() {
    Set<Variable> result = new HashSet<>(liveOut());
    result.removeAll(defs());
    result.addAll(uses());
    return result;
}

// Same analysis. Same fixpoint. Declarative syntax.
// NTA creates CFG; reference attrs follow CFG edges;
// circular attrs iterate to fixpoint.

Deriving Silver from an Abstract Interpreter

Silver makes a different trade: it sacrifices fixpoint iteration but gains modular compositional guarantees.

1 Factor into host analysis + extension analyses ▼

Silver's key design decision is modularity. Take your AI and identify:

The host — the core analysis that all extensions share. This becomes the host grammar with its base attributes.

Extensions — optional analysis components that can be composed independently. Each becomes a Silver extension module with aspect productions and new attributes.

// Host grammar: core language
nonterminal Expr;
synthesized attribute type :: Type;

// Extension A: adds 'or' syntax, forwards to host
production or  e::Expr ::= l::Expr r::Expr
{ forwards to not(and(not(l), not(r))); }

// Extension B: adds 'java' attr, uses aspect
synthesized attribute java :: String;
aspect production and
e::Expr ::= l::Expr r::Expr
{ e.java = l.java ++ "&&" ++ r.java; }

// Composition: A's 'or' gets B's 'java' via forwarding!
2 Express the core analysis without fixpoints ▼

Silver forbids circular attributes and enforces non-circularity via modular analysis. Your AI's fixpoint computations must be handled differently:

Option A: Unroll into passes. Express iterative analysis as a sequence of higher-order attributes, each creating an NTA representing the next iteration's state:

// Instead of circular fixpoint, create NTA chain:
synthesized attribute pass1 :: Decorated AnalysisState;
synthesized attribute pass2 :: Decorated AnalysisState;
// Each pass creates an NTA decorated with the previous pass's results
// Works for bounded-iteration analyses (k passes)

Option B: Delegate to external solver. Construct the fixpoint problem as a data structure (NTA) and pass it to an external solver:

// Build CFG as higher-order attribute, solve externally
synthesized attribute cfg :: Decorated CFGNode;
// Run fixpoint solver outside AG, feed results back via inherited attrs

Option C: Restrict to non-iterative analyses. Many useful analyses (type checking, name resolution, error collection, code generation) don't need fixpoints. Silver excels at these.

3 Use forwarding for analysis variants ▼

If your AI has multiple abstract domains (e.g., a family of analyses of increasing precision), each domain can be a Silver extension that forwards to the base analysis:

// Base analysis: simple constant propagation
production constPropCheck ...

// Extension: more precise interval analysis
production intervalCheck ... {
    forwards to constPropCheck(...);
    // Overrides specific attrs with more precise values
    // Non-overridden attrs get the base analysis's result (sound default)
}

// The forwarding invariant mirrors the Galois connection:
//   more-precise.attr ≥ less-precise.attr (for safety properties)
//   ≡ α(γ(a)) ≥ a
4 Use collection attributes for aggregation ▼

Your AI's join operations at merge points become Silver collection attributes with commutative operators:

// AI: join({error1, error2, error3}) at a merge point
// Silver: collection attribute with list concatenation
synthesized attribute errors :: [Message] with ++;

// Base production provides initial value
e.errors := [];

// Extension aspects contribute additional errors
e.errors <- if hasTypeError(e) then [mkError(e)] else [];

// Contributions from independent extensions are merged via ++
// Order doesn't matter (commutativity requirement)
5 Verify modular well-definedness ▼

This is Silver's unique contribution — something no general AI framework provides. Run modComplete(Host, Extension) on each extension independently:

// Silver verifies these properties for EACH extension:
// ✓ Every new production forwards to the host
//   (guarantees: host attrs always computable)
// ✓ Every syn attr has equations for all non-forwarding prods
//   (guarantees: no missing equations after composition)
// ✓ Flow types are preserved
//   (guarantees: no new circular dependencies)
// ✓ Inherited attrs are complete
//   (guarantees: no missing context)

// If each extension passes independently,
// ANY composition of extensions is guaranteed correct.
// This is compositional verification — the AG analogue of
// modular abstract interpretation with function summaries.

The result: A Silver specification that trades your AI's computational generality (fixpoints, infinite domains) for compositional safety guarantees that your AI never had. Each analysis extension is verified in isolation; all compositions are guaranteed correct. This is a trade that no other framework — AI or AG — currently offers.

Concrete Example: Deriving an Extensible Error Checker

// AI version: monolithic error checker
function checkErrors(program) {
  const errors = [];
  for (node of program.allNodes()) {
    if (isTypeError(node)) errors.push(typeError(node));
    if (isNameError(node)) errors.push(nameError(node));
    if (isFlowError(node)) errors.push(flowError(node)); // ← needs fixpoint
  }
  return errors;
}

// Silver version: modular, composable, verified

// HOST: base language with type checking
synthesized attribute errors :: [Message] with ++;
aspect production assign
s::Stmt ::= lhs::Expr rhs::Expr
{ s.errors <- if !assignable(lhs.type, rhs.type) then [typeErr(s)] else []; }

// EXTENSION A: name checking (verified independently)
aspect production varRef
e::Expr ::= n::Name
{ e.errors <- if isNull(e.lookup) then [nameErr(n)] else []; }

// EXTENSION B: purity checking (verified independently)
aspect production call
e::Expr ::= fn::Expr args::Exprs
{ e.errors <- if e.inPureContext && !fn.isPure then [purityErr(e)] else []; }

// ALL THREE compose safely. modComplete guarantees no conflicts.
// Flow error checking (needs fixpoint) would require an external solver.

Q

What mathematical theorem guarantees that both AI and AG fixpoint computations converge?

A

The Knaster-Tarski theorem: every monotone function on a complete lattice has a least fixpoint. Both AI transfer functions and AG semantic equations are monotone functions over lattices, so convergence is guaranteed.

Q

What is the key structural constraint that distinguishes an attribute grammar from a general abstract interpreter?

A

AGs restrict the program representation to a tree and transfer functions to tree-local (parent-child) dependencies. A general AI can operate over arbitrary graphs (CFGs, call graphs) with non-local transfer functions.

Q

What are the six degrees of freedom for parameterizing an abstract interpreter into an attribute grammar?

A

(1) Program representation (arbitrary → tree), (2) Abstract domain (any lattice → finite-height product), (3) Transfer functions (arbitrary monotone → tree-local), (4) Information flow (any edges → syn/inh), (5) Fixpoint strategy (chaotic iteration → demand-driven), (6) Scope (whole-program → single tree).

Q

What did Cousot prove about type systems in his 1997 POPL paper?

A

That type systems are abstract interpretations — type inference is a fixpoint computation within the AI framework. Since type checking is often implemented via AGs, this closes a triangle: types ⊂ AI ⊃ AG.

Q

In KSC, what does AGNode._cyc enforce, and what is its AI-theoretic significance?

A

It enforces the non-circularity constraint — it detects and throws on circular attribute dependencies at runtime. In AI terms, this is the constraint that restricts KSC from Level 3 (circular AG / tree-restricted AI) down to Level 4 (classical non-circular AG). Removing this throw and replacing it with iteration would move KSC up one level on the spectrum.

Q

What is the Galois connection in KSC's kind-checking analysis?

A

α is the composition of the AST translator (ts.Program → KSProgram) with the equation functions (tree → attribute values like KindDefinition[], Diagnostic[]). γ maps back to the set of programs consistent with those attribute values. The information loss in the translation is the abstraction.

Q

What did Rosendahl show in his 1992 PhD thesis?

A

Three results: (1) Abstract interpretations can be implemented using circular attribute grammars with a domain-theoretic evaluator. (2) "Lazy fixpoint iteration" generates specialized iterators for finite lattices. (3) AI methods strengthen AG correctness proofs, establishing a new class of AGs based on domain theory.

Q

What does "widening" in abstract interpretation do, and why don't attribute grammars need it?

A

Widening accelerates convergence over infinite-height lattices by jumping up the lattice to force termination of ascending chains. AGs don't need it because: non-circular AGs have no iteration (DAG deps), and circular AGs require finite-height lattices where ascending chains are naturally bounded.

Q

How does KSC's compileAnalysis() relate to abstract interpreter specialization?

A

compileAnalysis(Grammar, AnalysisSpec) is a form of staging / partial evaluation — it takes a declarative analysis specification and generates specialized dispatch code (switch/case per attribute per kind). This is analogous to specializing a generic AI engine for a particular abstract domain and set of transfer functions, producing a purpose-built analyzer.

Q

What practical framework comes closest to "expressing AI-style analyses in AG form"?

A

JastAdd (Circular Reference Attributed Grammars). It implements data-flow analyses like nullable/first/follow sets and devirtualization as circular attributes with fixpoint evaluation over finite lattices — effectively expressing abstract interpretations as AG specifications with demand-driven evaluation.

Q

What is the "pivot" in KSC (pivotToAttrCentric) and what is its AI analogue?

A

The pivot reshapes equations from production-centric (grouped by node kind: "what does an Identifier compute?") to attr-centric (grouped by attribute: "how is violationFor computed across all kinds?"). In AI terms, this is like reorganizing transfer functions from per-program-point to per-analysis-dimension — a transpose of the function table for efficient dispatch.

Q

Why can't every abstract interpretation be expressed as an attribute grammar?

A

Because AGs are restricted to tree-structured program models with tree-local transfer functions. Analyses that fundamentally require non-tree edges — loop back-edges (reaching definitions), interprocedural call/return edges (context-sensitive pointer analysis), or alias-based flow — cannot be expressed as classical AGs. They require reference attributes, circular AGs, or must remain as general AIs.

Q

How would you move KSC one level up the spectrum (from classical AG to circular AG)?

A

Replace the _cyc throw in AGNode.attr() with an iteration loop: start from ⊥, compute, check if the value changed, re-compute until stable. Require each circular attribute to declare a finite-height lattice (with ⊥, ≤, and optionally a height bound). This preserves all existing non-circular specs while enabling new iterative analyses.

Q

How do JastAdd's circular attributes relate to abstract interpretation's Kleene fixpoint iteration?

A

They are identical. JastAdd initializes circular attrs to ⊥ (bottom), evaluates the equation, compares with the previous iteration, and re-evaluates until stable. This is exactly Kleene's ascending chain: ⊥ ≤ F(⊥) ≤ F²(⊥) ≤ ... Convergence is guaranteed because JastAdd requires finite-height lattices and monotone equations — the same requirements as AI.

Q

What are reference attributes, and which AI constraint do they relax?

A

Reference attributes are attributes whose value is a pointer to another node in the tree, creating virtual non-tree edges. They relax Constraint #4 (Information Flow): instead of flowing only along parent-child tree edges, information can flow along arbitrary node-to-node references. This turns the semantic structure into a graph while keeping the syntactic structure as a tree. They enable name resolution, type hierarchies, and cross-file references.

Q

How is Silver's forwarding mechanism analogous to a Galois connection?

A

Forwarding maps extension syntax to host syntax (like α: concrete → abstract), and non-overridden attributes flow back from the forwarded-to tree (like γ: abstract → concrete). The soundness invariant t.attr = t.forward.attr mirrors the Galois connection soundness condition α(γ(a)) ≥ a — the extension's semantics safely agree with the host's.

Q

What does Silver's modular well-definedness (modComplete) guarantee, and why is it unique among AI/AG systems?

A

If each extension independently passes modComplete with respect to the host grammar, then any composition of those extensions is guaranteed to be complete (no missing equations) and non-circular. This is compositional verification — the AG analogue of modular abstract interpretation with function summaries. No other AG or AI framework provides this guarantee.

Q

What are NTAs (Nonterminal Attributes) and why do they matter for the AI–AG bridge?

A

NTAs allow an AG equation to construct a fresh subtree as its value. This subtree is grafted into the tree and fully attributed. They relax Constraint #1 (Program Representation): the AG can create derived structures like CFGs during evaluation and run further analysis on them. This is how JastAdd expresses data-flow analyses — build the CFG as an NTA, define circular attributes on CFG nodes, compute the fixpoint.
