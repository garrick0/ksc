04New Findings

The ports-and-adapters reorg revealed issues that the v1 audit missed.

NF-1 Analysis barrel mixes runtime and codegen exports deeper root cause ▼

The real root cause

libs/analysis/index.ts is a single barrel that re-exports both runtime-needed utilities (code, pivotToAttrCentric, withDeps) and codegen-only machinery (compileAnalysis, runCodegenPipeline, buildDepGraph). In Node.js ESM, importing any export from this barrel loads all source modules.

// libs/analysis/index.ts — 7 modules mixed together

// Needed at runtime by adapters:
export { code, isCodeLiteral } from './types.js';           // 196 lines
export { withDeps, collectDepsForAttr } from './equation-utils.js'; // 65 lines
export { pivotToAttrCentric } from './pivot.js';         // 33 lines

// Only needed at codegen time:
export { compileAnalysis, buildDepGraph, ... } from './compile.js';  // 487 lines
export { validateSpec } from './validate.js';                       // 38 lines
export { runCodegenPipeline, ... } from './pipeline.js';           // 126 lines (imports fs/path)

This makes every barrel consumer — even the spec adapter which only needs code and pivotToAttrCentric — load 651 lines of codegen machinery and Node.js filesystem modules.

This is deeper than v1's root cause. The v1 audit identified AP-4 (full spec import) as the root cause. But fixing AP-4 alone wouldn't help now: the spec adapter itself imports from the barrel (NF-2), creating an independent leak path that AP-4 can't close.
NF-2 Spec adapter imports from analysis barrel new leak path ▼

Second import path into the barrel

// adapters/analysis/ts-kind-checking/spec.ts:24-25
import type { AnalysisSpec, AttrDecl, Ctx } from '../../../libs/analysis/index.js';  // type-only ✓
import { code, pivotToAttrCentric } from '../../../libs/analysis/index.js';        // runtime import!

The spec adapter needs code (from types.ts, 196 lines) and pivotToAttrCentric (from pivot.ts, 33 lines). But by importing from the barrel instead of the specific modules, it loads all 976 lines of the analysis package including the codegen pipeline.

The fix is trivial

// Direct imports — bypass the barrel entirely
import { code } from '../../../libs/analysis/types.js';
import { pivotToAttrCentric } from '../../../libs/analysis/pivot.js';

Two changed import lines would isolate the spec adapter from the entire codegen module graph.

Dependency chain (updated)

The v1 audit showed a single chain from program.ts. v2 reveals two independent paths that both trigger the barrel:

Key insight: Fixing the barrel (NF-1 + NF-2) is now the real first step. AP-4 was the v1 keystone; the barrel is the v2 keystone. Split it or bypass it, and the codegen module graph exits the runtime entirely.

05What Improved

The ports-and-adapters reorg delivered real gains — three things that are better than v1.

1 CLI lazy-loads the codegen pipeline 
// apps/cli/cli.ts:240-242 — dynamic import, only runs for `ksc codegen`
async function runCodegen(_opts: CLIOptions): Promise<number> {
  const { runCodegenPipeline } = await import('../../libs/analysis/pipeline.js');
  const { tsKindCheckingTarget, mockTarget } = await import('./codegen-targets.js');

The codegen pipeline and target definitions are not imported at CLI startup. They're lazy-loaded only when ksc codegen is invoked. This was a deliberate improvement.

Undermined by NF-1: the barrel import for buildDepGraph (line 50 of cli.ts) loads pipeline.ts anyway via the barrel re-export. The lazy import is bypassed.
2 Zero-cost lightweight npm entry point 
// apps/kindscript/annotations.ts — zero heavyweight imports
export type { Kind, PropertySet } from '../../adapters/analysis/ts-kind-checking/types.js';
export { defineConfig } from './config.js';
export type { KindScriptConfig } from './config.js';

The kindscript root import (import { Kind } from 'kindscript') is perfectly clean. Only phantom types and an identity function. Zero evaluator, grammar, or analysis imports. Users annotating source code pay zero startup cost.

Clean boundary: This is the gold standard. apps/kindscript/annotations.ts proves the architecture can enforce clean separation when done right. The heavyweight kindscript/core is where the leaks live.
3 Composition roots are explicit and inspectable

The old program.ts mixed wiring with domain logic. Now apps/core/index.ts is a clear composition root: it imports adapters, wires them together, and exports the public API. The wiring is 3 lines:

// apps/core/index.ts:59-61 — all wiring visible in one place
const evaluator = wireEvaluator({ grammar, spec: analysisSpec, dispatch: dispatchConfig });
const depGraph = buildDepGraph(analysisSpec.attrs);

Port interfaces like EvaluationTarget<K, P> enforce K-linking at the type level — TypeScript prevents mismatched grammar/spec pairs. The composition root is the only place where concrete adapters meet.

Architectural clarity: the boundary violations are now visible. In v1 they were hidden in a monolithic program.ts. In v2 you can see exactly which imports cross the codegen/runtime line.

06Updated Removal Plan

Leveraging the ports-and-adapters architecture to seal the boundary. The new plan addresses the barrel as root cause rather than just the spec import.

  • 1
    NF-2: Direct imports in spec adapter (new root cause fix)
    Change spec.ts to import code from libs/analysis/types.js and pivotToAttrCentric from libs/analysis/pivot.js directly, bypassing the barrel. The same for any other adapter that imports runtime utilities from the barrel.
    ~4 LOC changed (2 import lines in each spec adapter). Eliminates Path A entirely.
  • 2
    AP-4 + AP-3: Use createEvaluator directly, skip wireEvaluator
    Have apps/core/index.ts call createEvaluator({ dispatch, grammar, projections: analysisSpec.projections, setup: analysisSpec.setup }) directly. No need for wireEvaluator which only adds validateDispatch (AP-3). Alternatively, export a slim { projections, setup } from the spec separately.
    ~10 LOC changed. Eliminates runtime validation and the spec.attrs dependency.
  • 3
    AP-1: Generate dep graph as static data
    Emit a third generated file (dep-graph.ts) from the codegen pipeline. The composition root imports the static data instead of computing it at runtime. Remove the buildDepGraph import from the analysis barrel.
    ~40 LOC in compile.ts + pipeline.ts. Eliminates Path B and buildDepGraph from runtime.
  • 4
    AP-5 + AP-2: Auto-resolved
    After steps 1–3, no runtime code imports from the analysis barrel. compile.ts, pipeline.ts, and validate.ts exit the runtime module graph entirely. pivotToAttrCentric only runs when codegen scripts import spec.ts.
    0 LOC. Free side-effect of steps 1–3.

Q

The v1 audit identified AP-4 (full spec import) as the root cause. What is the deeper root cause discovered in v2?

A

The libs/analysis/index.ts barrel mixes runtime-needed and codegen-only exports. Importing anything from it (even code or pivotToAttrCentric) loads compile.ts (487 lines), pipeline.ts (126 lines with fs/path), and validate.ts (38 lines). The spec adapter itself imports from this barrel, creating a leak path that fixing AP-4 alone can't close.

Q

Why does importing one export from an ESM barrel load all re-exported source modules?

A

In ECMAScript modules, re-export declarations like export { x } from './y.js' cause y.js to be linked and evaluated as part of the module graph. Node.js has no tree shaking — it evaluates all linked modules regardless of which bindings the importer actually uses. The re-exported module's top-level code runs, its imports are resolved, and so on.

Q

How many lines of codegen machinery are loaded at runtime in v2, and what changed from v1?

A

651 lines (v1 was 487). The increase comes from pipeline.ts (126 lines) and validate.ts (38 lines) being added to the analysis barrel. pipeline.ts also brings Node.js fs and path modules into the runtime module graph, which weren't loaded in v1.

Q

The CLI lazy-loads the codegen pipeline with await import(). Why is this improvement undermined?

A

The CLI also imports buildDepGraph from libs/analysis/index.js at the top level (line 50 of cli.ts). This barrel re-exports from pipeline.ts, which loads the codegen pipeline eagerly via the barrel — bypassing the deliberate lazy import. The lazy await import() loads an already-evaluated module.

Q

Why can't you fix the boundary by just removing buildDepGraph from the runtime composition root?

A

Because the spec adapter (adapters/analysis/ts-kind-checking/spec.ts) also imports from the analysis barrel (for code and pivotToAttrCentric). This is an independent import path (Path A) that loads the barrel regardless of what the composition root does. You need to fix BOTH paths: direct imports in the spec adapter AND removing buildDepGraph from runtime.

Q

What does wireEvaluator() do that createEvaluator() doesn't, and why does it force the full spec import?

A

wireEvaluator calls validateDispatch(opts.dispatch, opts.spec.attrs.map(a => a.name)) before delegating to createEvaluator. This validation reads spec.attrs — the full attribute declarations — which is the ONLY reason wireEvaluator needs the complete AnalysisSpec. createEvaluator only needs { dispatch, grammar, projections, setup } — no attrs, no equations.

Q

Name the 3 genuine improvements the ports-and-adapters reorg delivered for boundary enforcement.

A

(1) CLI lazy-loads codegen with await import() for pipeline and targets (though undermined by barrel). (2) Zero-cost lightweight entry point (apps/kindscript/annotations.ts) with only phantom types. (3) Explicit composition roots with port contracts and K-linking making the wiring and boundary violations visible and inspectable.

Q

After applying all fixes in the v2 removal plan, what would the composition root's runtime imports be?

A

Five imports: (1) grammar from grammar adapter (for tree building), (2) dispatchConfig from generated dispatch.ts, (3) projections + setup from a slim spec export or directly, (4) depGraph from a new generated static file, (5) createEvaluator from the evaluator. Zero analysis barrel, zero codegen machinery, zero dead attr declarations, zero fs/path.
