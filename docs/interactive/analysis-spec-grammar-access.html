What Should Analysis Specs Know?

Three layers of grammar data — raw inputs, generated types, and runtime schema — and which ones analysis specs should access

01Three Layers of Grammar Data

Grammar information exists at three distinct layers, each with different richness and coupling cost. Understanding these layers is essential to the architecture question.

Grammar Inputs

GrammarSpec: nodes Map, sumTypes Map. Field names as strings, types as tag unions. What the grammar declares.

→

Generated Types

KSIdentifier, KSNode, KindToNode. Full TypeScript interfaces with typed fields. What the grammar produces.

→

Runtime Schema

getChildFields(kind), allKinds. Queryable schema data. What you can ask the grammar at runtime.

Click each layer to see exactly what it contains:

1 Grammar Inputs (GrammarSpec) ▼

The raw schema declarations — the input to Functor 1. This is what the grammar author writes.

interface GrammarSpec {
  nodes: ReadonlyMap<string, NodeEntry>;
  sumTypes: ReadonlyMap<string, SumTypeEntry>;
}

interface NodeEntry {
  kind: string;                    // e.g. 'Identifier'
  fields: Record<string, FieldDesc>; // e.g. { escapedText: { tag: 'prop', propType: 'string' } }
  memberOf: string[];               // e.g. ['Expression', 'TypeNode']
}

// You know: field names, field types (as strings), sum type membership
// You don't get: TypeScript interfaces, type narrowing, IDE autocomplete

Key insight: This layer has all the schema information — every field name, every type, every sum type. But it's all strings. You can't write node.escapedText — you'd write (node as any)['escapedText'].

2 Generated Types (Functor 1 output) ▼

Full TypeScript interfaces generated from the grammar inputs. This is what Functor 1 produces.

// generated/ts-ast/grammar/node-types.ts
interface KSIdentifier extends KSNodeBase {
  kind: 'Identifier';
  escapedText: string;
  resolvesToImport: boolean;
  isDefinitionSite: boolean;
  resolvedFileName: string;
  symIsFunction: boolean;
  // ... 15 more typed fields
}

// generated/ts-ast/grammar/kind-map.ts
type KindToNode = {
  'Identifier': KSIdentifier;
  'BinaryExpression': KSBinaryExpression;
  // ... 364 entries
}

// You get: full TypeScript interfaces, type narrowing, IDE autocomplete
// Cost: your spec imports from generated/ — coupling to Functor 1 output
3 Runtime Schema (getChildFields, allKinds) ▼

Queryable schema data embedded in the generated output. Used by the evaluator at runtime.

// generated/ts-ast/grammar/schema.ts
const C: Record<string, readonly string[]> = {
  'Identifier': [],                                // no child fields
  'VariableDeclaration': ['name', 'exclamationToken', 'type', 'initializer'],
  'CompilationUnit': ['children'],
  // ... 364 entries
};

export function getChildFields(kind: string): readonly string[] {
  return C[kind] ?? [];
}

// Used by the evaluator's buildKSCTree to know which fields to descend into.
// This is the same info as NodeEntry.fields filtered to child/optChild/list tags.

Key point: The runtime schema is derivable from grammar inputs. It's just the child field names per kind. The evaluator could compute this from GrammarSpec.nodes instead of importing it.

Key insight: The three layers form a refinement chain. Layer 1 (grammar inputs) has all the information. Layer 2 (generated types) adds TypeScript type safety. Layer 3 (runtime schema) is a subset of Layer 1, pre-computed for runtime use. Anything in Layer 3 can be derived from Layer 1 — but Layer 2 adds something genuinely new: compile-time type checking.

02What Specs Actually Use Today

The analysis spec currently imports from Layer 2 (generated types). Here's every import and why it exists.

specs/ts-ast/kind-checking/spec.ts imports 10 generated node types:

import type {
  KSNode,                       // base type for 'as' casts
  KSIdentifier,                 // .escapedText in console check
  KSPropertyAccessExpression,   // .expression, .name in console check
  KSVariableDeclarationList,    // .declarationKind in immutable check
  KSBinaryExpression,           // .operatorToken in mutation check
  KSPrefixUnaryExpression,      // .operator in mutation check
  KSPostfixUnaryExpression,     // .operator in mutation check
  KSExpressionStatement,        // .expression in side-effect check
  KSVariableDeclaration,        // .type in kindAnnotations equation
  KSIntersectionType,           // .types in extractKindAnnotations
  KSTypeReference,              // .typeName, .typeArguments
} from '../../../generated/ts-ast/grammar/index.js';

Usage pattern: All used for as KSFoo type casts inside equation functions and violation predicates, to access node-specific fields with type safety.

For example, the noConsole violation:

violation({
  triggerKind: 'PropertyAccessExpression',
  predicate: (_ctx, node) =>
    node.expression.kind === 'Identifier' &&
    (node.expression as KSIdentifier).escapedText === 'console',
               // ↑ Without KSIdentifier, this would be (node.expression as any).escapedText
})

specs/ts-ast/kind-checking/binder.ts imports 6 generated types:

import type {
  KSNode,                   // base type
  KSTypeAliasDeclaration,   // .name, .type — extracting Kind<{...}> defs
  KSTypeReference,          // .typeName, .typeArguments
  KSTypeLiteral,            // .members — reading property signatures
  KSIdentifier,             // .escapedText — reading names
  KSPropertySignature,      // .name, .type — reading prop defs
} from '../../../generated/ts-ast/grammar/index.js';

Usage: The binder walks TypeAliasDeclaration nodes looking for type X = Kind<{...}> patterns. It needs to access .typeName.escapedText, .typeArguments[0], .members — all node-specific fields.

analysis/violation.ts — the only file in the analysis/ directory that imports generated code:

import type { KindToNode } from '../generated/ts-ast/grammar/kind-map.js';

export function violation<K extends string & keyof KindToNode>(config: {
  triggerKind: K;
  predicate: (ctx: Ctx, node: KindToNode[K]) => boolean;
  message: (ctx: Ctx, node: KindToNode[K], def: KindDefinition) => string;
}): ViolationRule { ... }

What this enables: When you write violation({ triggerKind: 'Identifier', predicate: (ctx, node) => ... }), TypeScript infers node: KSIdentifier — you get autocomplete and type checking on all 20+ fields of KSIdentifier.

Why it's problematic: This is a generic analysis utility importing from a specific grammar's generated code. If you added a Python grammar, this file would need to know about both.

The pattern: Every import from generated code exists for one reason — type-safe field access inside equation functions and violation predicates. The spec knows what fields it wants (e.g., escapedText), and the generated types let TypeScript verify those fields exist.

03The Core Question

The user's question distills to: what layer of grammar data should the spec consume, and who should create grammar-specific objects?

There are two sub-questions here:

Sub-question A: Does the spec need generated types at all?

The equation functions and violation predicates currently look like this:

// WITH generated types (current):
const varDecl = raw as KSVariableDeclaration;   // ← type cast to generated type
if (!varDecl.type) return [];                     // ← .type is typed, IDE helps
const defLookup = ctx.attr('defLookup');
return extractKindAnnotations(varDecl.type, defLookup);

// WITHOUT generated types:
const varDecl = raw as any;                       // ← no type info
if (!varDecl.type) return [];                     // ← .type is unchecked, hope it exists
return extractKindAnnotations(varDecl.type, defLookup);

In practice, the runtime behavior is identical. The generated types are erased at compile time — they only provide developer-time safety. The question is whether that safety is worth the coupling.

Sub-question B: Should grammar inputs replace generated types?

Grammar inputs (GrammarSpec.nodes) contain all the schema information as data:

// Grammar input data:
nodes.get('Identifier').fields
  // → { escapedText: { tag: 'prop', propType: 'string' },
  //     resolvesToImport: { tag: 'prop', propType: 'boolean' }, ... }

// You could validate at spec-construction time:
const idFields = nodes.get('Identifier')!.fields;
assert('escapedText' in idFields);   // ← runtime check instead of compile-time
assert('resolvesToImport' in idFields);

This would give runtime validation (catch errors when building the spec) instead of compile-time validation (catch errors when running tsc). Different tradeoff.

The fundamental tension: Generated types provide compile-time type safety (IDE autocomplete, typo detection, field existence checks). Grammar inputs provide schema data (field names, types, relationships). You can validate against either — but the developer experience is very different.

04Five Architectural Options

Each option represents a different answer to "what should the spec know about the grammar?"

Option A: Keep current approach (spec imports generated types)

Grammar Inputs

nodes.ts declares 364 kinds with fields

Functor 1 Output

KSIdentifier, KSNode, KindToNode, getChildFields

↓ spec imports generated types for type safety

Analysis Spec

Imports KSIdentifier etc. Uses as KSFoo casts. Full type safety.

Analysis Machinery

Generic. Takes AnalysisSpec, generates evaluator. node: unknown.

Pros: Maximum type safety. IDE autocomplete. Compile-time error detection. Familiar TypeScript patterns.

Cons: Spec coupled to generated output. violation.ts in analysis/ also imports generated (bad). Build order dependency: grammar must generate before spec can type-check.

The coupling is type-only: All imports are import type — erased at runtime. This is the lightest possible form of coupling.

Option B: Spec uses only grammar inputs (no generated types)

Grammar Inputs

Spec receives GrammarSpec or NodeEntry maps

↓ spec reads schema data as strings

Analysis Spec

All field access via (node as any).field. Can validate field names against NodeEntry at spec construction time.

Analysis Machinery

Same as today — generic.

// Spec would look like:
function buildAnalysisSpec(grammar: GrammarSpec): AnalysisSpec {
  // Validate at construction time:
  const idFields = grammar.nodes.get('Identifier')!.fields;
  assert('escapedText' in idFields, 'Grammar must have Identifier.escapedText');

  return {
    structural: [...],
    properties: [{
      name: 'noConsole',
      violations: [{
        triggerKind: 'PropertyAccessExpression',
        predicate: (_ctx, node: any) =>                    // ← any, no type safety
          node.expression.kind === 'Identifier' &&
          node.expression.escapedText === 'console',    // ← typo here? runtime error
      }],
    }],
  };
}

Pros: Spec decoupled from generated output. Can build spec before running codegen. No build order dependency.

Cons: Loses all compile-time type safety. Every field access is unchecked. Typos in field names become runtime bugs. IDE can't help you write predicates. 364 node kinds × many fields = huge surface area for errors.

Critical issue: The binder walks TypeAliasDeclaration → TypeReference → Identifier → TypeLiteral → PropertySignature, accessing .typeName, .typeArguments, .members, .name, .type. With 10+ field accesses across 6 node kinds, all as any, this becomes extremely fragile.

Option C: Spec creates all grammar-specific objects, passes them in

This is the user's suggestion: the spec has full access to generated types, creates all grammar-coupled objects, and passes them through the AnalysisSpec interface so analysis/ stays generic.

Generated Types

KSIdentifier, KSNode, etc.

↓ spec imports generated types

Analysis Spec

Imports generated types. Creates equation functions, violation predicates, grammar-specific config objects. Passes everything through AnalysisSpec.

↓ passes fully-formed objects

Analysis Machinery

Receives AnalysisSpec. Zero knowledge of grammar. No hardcoded kind names. No generated imports.

// Spec provides grammar-specific config that analysis/ currently hardcodes:
interface AnalysisSpec {
  structural: StructuralAttr[];
  properties: PropertyDecl[];
  projections: { ... };

  // NEW: spec-owned grammar-specific config
  grammarConfig: {
    rootKind: string;           // 'CompilationUnit' — spec decides
    fileNameField: string;      // 'fileName' — spec decides
    annotationCarrier: string; // 'VariableDeclaration' — spec decides
    annotationAttr: string;    // 'kindAnnotations' — spec decides
  };
}

Pros: analysis/ becomes 100% generic. The hardcodings ('CompilationUnit', 'VariableDeclaration') move to the spec where they belong. Spec keeps type safety. violation.ts moves to spec directory.

Cons: Spec still imports generated types (but this is fine — specs are grammar-specific). The AnalysisSpec interface gets slightly larger.

This is actually mostly what already happens. Equation functions and violation predicates are already spec-owned objects passed through the AnalysisSpec. The gap is just the few hardcodings in compile.ts that should be spec-provided config.

Option D: Grammar provides a typed "vocabulary" API

Instead of the spec importing raw generated types, the grammar generates a spec-building API — a typed vocabulary for writing analysis specs.

Generated Vocabulary

Grammar generates helpers: onIdentifier(pred), onBinaryExpr(pred) — typed wrappers for spec authoring.

↓ spec uses grammar-provided API

Analysis Spec

Uses typed API instead of raw imports. Same type safety, more structured.

// Generated by grammar — a typed vocabulary for spec authoring:
export function onIdentifier(
  pred: (ctx: Ctx, node: KSIdentifier) => boolean,
  msg: (ctx: Ctx, node: KSIdentifier, def: KindDefinition) => string,
): ViolationRule { ... }

// Spec uses it:
violations: [
  onIdentifier(
    (_ctx, node) => node.resolvesToImport,   // ← fully typed, node: KSIdentifier
    (_ctx, node, def) => `'${node.escapedText}' violates ${def.name}`,
  ),
]

Pros: Cleaner API than raw type imports. Type safety preserved. Grammar "owns" its analysis-facing API. The violation() builder becomes grammar-generated.

Cons: Significant over-engineering. 364 node kinds → 364 on* functions — most unused. The current violation() builder already gives the same type inference with one function. This duplicates what TypeScript's type system does naturally.

This is what violation<K>() already does — the KindToNode generic provides the same per-kind type narrowing without generating 364 wrapper functions. Option D is a more explicit but less elegant version of the existing pattern.

Option E: Current + fix the leaks (recommended)

Keep the current architecture but fix the specific problems: move grammar-specific config out of analysis/compile.ts, move violation.ts to the spec.

Generated Types

KSIdentifier, KSNode, KindToNode, getChildFields

↓ spec imports generated types (same as today)

Analysis Spec

Imports generated types. Creates equations + predicates. Also provides grammarConfig with rootKind, annotationCarrier. Owns violation() builder (moved from analysis/).

↓ passes AnalysisSpec (now with grammarConfig)

Analysis Machinery

Zero generated imports. Reads rootKind from spec. Reads annotationCarrier from spec. No hardcoded kind names. Fully generic.

Concrete changes:

// 1. Move analysis/violation.ts → specs/ts-ast/kind-checking/violation.ts

// 2. Add grammarConfig to AnalysisSpec:
interface AnalysisSpec {
  structural: StructuralAttr[];
  properties: PropertyDecl[];
  projections: { ... };
  grammarConfig: {
    rootKind: string;            // replaces hardcoded 'CompilationUnit' in _findFileName
    fileNameField: string;       // replaces hardcoded 'fileName'
    annotationCarrier: string;  // replaces hardcoded 'VariableDeclaration' in context body
    annotationAttr: string;     // replaces hardcoded 'kindAnnotations'
  };
}

Pros: Minimal change. Maximum type safety preserved. analysis/ becomes truly generic. No over-engineering.

Cons: Spec still imports generated types (but this is the right tradeoff).

Why "inputs only" (Option B) is the wrong tradeoff

Grammar inputs have all the schema information, but analysis specs don't need schema information — they need typed field access. The binder walks 5 levels deep through AST nodes (TypeAlias → TypeReference → Identifier checking .escapedText === 'Kind'). Making all of that untyped would be actively harmful.

Consider: if a grammar update renames escapedText to name, with generated types you get a compile error. With grammar inputs you get a silent runtime bug.

Why C and E converge

Options C and E are actually the same architecture — the spec creates grammar-specific objects and passes them through the AnalysisSpec. The difference is just scope:

  • C says "make this the explicit philosophy" — every grammar-specific thing is spec-created
  • E says "fix the specific leaks" — move the 3-4 hardcodings to spec config

In practice, the current system is already mostly option C. Equation functions and violation predicates are spec-created objects. The gap is just the few hardcodings in compile.ts.

Key realization: The spec should import generated types — specs are inherently grammar-specific (that's what "spec" means). The goal isn't to decouple specs from their grammar — it's to decouple the analysis machinery from any specific grammar. Option E achieves this with minimal change.

06Recommendation

The architecture should make a clear distinction between two kinds of coupling:

Correct coupling (keep)

Specs import generated types. This is intentional — a kind-checking spec for the TS grammar needs to know about KSIdentifier, KSVariableDeclaration, etc. That's what makes it a TS-grammar spec. The generated types are the grammar's "public API" for spec authors.

Incorrect coupling (fix)

Analysis machinery has grammar knowledge. compile.ts hardcodes 'CompilationUnit' and 'VariableDeclaration'. violation.ts imports from generated/ts-ast/. These should be spec-provided.

The fix (Option E)

Step 1: Move analysis/violation.ts to specs/ts-ast/kind-checking/violation.ts. It needs KindToNode from the TS grammar — it belongs with the spec.

Step 2: Add grammarConfig to AnalysisSpec. The spec provides the grammar-specific values that compile.ts currently hardcodes.

Step 3: Update compile.ts to read from spec.grammarConfig instead of hardcoded strings.

After these changes:

  • analysis/ has zero imports from generated/ or specs/
  • analysis/compile.ts has zero hardcoded kind names
  • Specs keep full type safety via generated type imports
  • Adding a new grammar+analysis only requires new spec files — no machinery changes
Bottom line: The spec should absolutely have access to generated grammar code — it needs the types for safe field access. The analysis machinery should not — it should receive all grammar-specific information through the AnalysisSpec interface. This is a clean separation: specs know their grammar, machinery is generic.

Q

What are the three layers of grammar data, and what does each provide that the layer below it doesn't?

A

Layer 1 (Grammar Inputs): Schema declarations — field names, types as strings, sum type membership. Layer 2 (Generated Types): Adds compile-time TypeScript type safety — interfaces, type narrowing, IDE autocomplete. Layer 3 (Runtime Schema): A subset of Layer 1 pre-computed for runtime use (getChildFields). Layer 2 is the only one that adds genuinely new capability (type safety).

Q

Why is "spec uses only grammar inputs" (Option B) a bad tradeoff for this system?

A

Because analysis specs need typed field access, not schema queries. The binder walks 5 levels deep through AST nodes accessing specific fields (.typeName, .escapedText, .typeArguments). Making all of that untyped (as any) would mean typos become silent runtime bugs, IDE can't help, and field renames in the grammar don't produce compile errors.

Q

What's the difference between "correct coupling" and "incorrect coupling" in this architecture?

A

Correct: Specs import generated types — specs are inherently grammar-specific, they need typed field access. Incorrect: Analysis machinery (analysis/) has grammar knowledge — compile.ts hardcoding 'CompilationUnit', violation.ts importing from generated/ts-ast/. The rule: specs know their grammar, machinery is generic.

Q

How does violation<K>() provide type safety, and why does Option D (grammar vocabulary) duplicate this?

A

violation<K> uses the KindToNode mapped type — when you write triggerKind: 'Identifier', TypeScript infers K = 'Identifier' and narrows node to KindToNode['Identifier'] = KSIdentifier. Option D would generate 364 per-kind functions like onIdentifier(pred) — but that's exactly what the generic violation() already does via type inference, without needing 364 functions.

Q

What is grammarConfig in the recommended approach, and what hardcodings does it replace?

A

A new field on AnalysisSpec where the spec declares grammar-specific values: rootKind (replaces hardcoded 'CompilationUnit' in _findFileName), fileNameField (replaces hardcoded 'fileName'), annotationCarrier (replaces hardcoded 'VariableDeclaration' in context body), and annotationAttr (replaces hardcoded 'kindAnnotations'). These move grammar knowledge from machinery to spec.

Q

All spec imports from generated code are import type. Why does this matter for the coupling assessment?

A

Type-only imports are erased at runtime — they're the lightest possible form of coupling. The spec doesn't depend on any generated runtime code, only on the TypeScript type definitions. This means the coupling is purely a development-time concern (IDE, tsc) and has zero runtime cost. It also means the spec could theoretically work without generated code if you accepted any everywhere.

Q

What three changes constitute the recommended fix (Option E)?

A

(1) Move analysis/violation.ts to specs/ts-ast/kind-checking/violation.ts — it needs KindToNode from the TS grammar. (2) Add grammarConfig to AnalysisSpec with rootKind, fileNameField, annotationCarrier, annotationAttr. (3) Update compile.ts to read from spec.grammarConfig instead of hardcoded strings. After this, analysis/ has zero generated imports and zero hardcoded kind names.
