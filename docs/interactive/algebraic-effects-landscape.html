Algebraic Effects: The Languages & the Theory

How Koka, Eff, Frank, Effekt, OCaml 5, Unison, and Effect-TS are building the future of effect tracking.

1 What algebraic effects actually are

Every program does two things: compute values and perform effects (I/O, mutation, exceptions, async, randomness). Algebraic effects separate these two concerns at the language level.

The core idea in one sentence: An effect is an operation you declare (the syntax), and a handler you define separately (the semantics). The operation says "I need to read from a file." The handler says "here's how reading actually works." They're independent — you can swap handlers without changing the code.

2 Why this is better than monads

Haskell uses monads to track effects. They work, but they have a composition problem. Algebraic effects solve it:

The slogan: monads are one way to implement effects. Algebraic effects are a more general framework that decomposes effects into operations + handlers, composes naturally, and lets you swap implementations without changing code.

5 The theoretical foundation

Algebraic effects aren't just an engineering pattern — they come from deep mathematics:

Timeline of key developments

2002
Plotkin & Power — "Notions of Computation Determine Monads." Show that algebraic operations generate common monads (exceptions, state, nondeterminism).
2009
Plotkin & Pretnar — "Handlers of Algebraic Effects." Introduce the handler abstraction — the key innovation that makes effects practical.
2012
Eff released — first language built around algebraic effects and handlers.
2014
Koka adds row-polymorphic effect types (Leijen, Microsoft Research). First language to track effects in the type system with polymorphism.
2017
Frank — "Do Be Do Be Do" published at POPL. Introduces multihandlers.
2020
fp-ts brings monadic effect tracking to TypeScript. Proves the ecosystem wants effect systems.
2022
OCaml 5.0 ships with algebraic effects. First mainstream language with native support.
2023
Effect-TS rapid growth begins. fp-ts announces merge into Effect.
2025
Effect-TS 3.0 — first stable release. 13k+ stars. Koka v3.2 with two PLDI distinguished paper awards. Effekt tutorial at Programming 2025.

6 What this means for your project

The connection to property analysis:

Algebraic effects are the most principled solution to the property-analysis problem from your property-space doc. If every function's effects are tracked in its type, then:

Purity: A function with empty effects <> is provably pure — no analysis needed.
Effect classification: The type tells you exactly which primitives the function uses. No AST scanning required.
Composability: Calling a function with effects <console, exn> from a function with effects <console> is a type error — the exn effect isn't handled.
Swappable implementations: Test with a mock handler, deploy with a real one — the function code doesn't change.

For KindScript / your TypeScript analysis tool:

You're doing at the static analysis level what Koka does at the language level and Effect-TS does at the library level: tracking which effects each function uses. The approaches are complementary. KindScript could detect Effect-TS usage patterns, or it could infer a Koka-like effect signature for plain TypeScript functions by scanning their ASTs. Either way, the theoretical framework is the same — you're projecting functions into the effect lattice.
