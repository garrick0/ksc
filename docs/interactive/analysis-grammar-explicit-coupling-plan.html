01Design Decisions

The 6 decisions selected, their rationale, and what each requires.

#DecisionChoiceWhat It Means
1C Domain types Use specific node type directly KindDefinition.node: KSTypeAliasDeclaration, Diagnostic.node: ASTNode with grammar node type. No generics, no unknown. Locks domain types to one grammar — which is the explicit intent.
2C Predicate constants Derive from grammar sum type tables Replace hand-maintained Set<string> with sets derived from grammar.sumTypeMembers or typed ReadonlySet<TSNodeKind>. Constants stay in sync when grammar changes.
3B Projections Use TypedAGNode<KSCAttrMap> Change AnalysisProjections port: projection functions receive typed root with known attr() return types. Requires port interface change in core-evaluator.
4C Equation type Full per-kind typed with KindToNode Equation records become Partial<Record<K, (ctx: KindCtx<KindToNode[K]>, ...args) => T>>. Requires grammar's KindToNode mapping at the port level.
5B Helpers Audit each; type where beneficial getKindCtx and diag stay generic (correct). String constants in predicates get grammar-typed. Future helpers that access node fields must use KindCtx<N>.
6B Adapter barrel Re-export grammar + grammar types Analysis adapter's index.ts re-exports grammar, TSNodeKind, KSNode, KindToNode. Makes dependency explicit and discoverable.
Thread connecting all 6: Each decision pushes coupling one layer deeper. 6B makes the grammar dependency visible. 1C makes domain types concretely typed. 2C derives constants from grammar metadata. 5B types helpers that touch nodes. 4C types equation records with the grammar's node map. 3B types projection functions with the generated attr map. Together they form a closed loop where every module in the analysis adapter knows exactly which grammar it serves.

Dependency ordering

These decisions have implementation dependencies:

  1. 6B (barrel re-exports) — no dependencies, purely additive
  2. 1C (domain types) — needs grammar types importable from the adapter, so 6B first is convenient
  3. 2C (grammar-derived predicates) — needs grammar import, independent of 1C
  4. 5B (helper audit) — depends on 2C for predicate typing; independent of 1C
  5. 4C (typed equations) — port change in core-codegen; independent of adapter changes but needs design alignment with 3B
  6. 3B (typed projections) — port change in core-evaluator; independent of 4C

02Architecture Decision Records

Three ADRs to create in docs/adr/. Each captures one cluster of related decisions.

ADR-001 PROPOSED Analysis adapters explicitly depend on their grammar adapter

Context: Each analysis adapter (e.g., ts-kind-checking) is bound to exactly one grammar (e.g., Grammar<TSNodeKind>). The current architecture mixes direct coupling (equation functions import concrete node types) with generic abstraction (domain types use node: unknown, predicates use Set<string>). This creates a gradient where some layers have full type safety and others don't.

Decision: Analysis adapters will explicitly and directly depend on their grammar adapter at every layer. This means: (a) the adapter barrel re-exports grammar types, (b) domain types use concrete node types, (c) predicate constants are derived from grammar metadata or typed with the grammar's kind union, (d) helpers use grammar-typed parameters where they access node fields.

Rationale: The Expression Problem analysis shows that when rules are scoped to one grammar, a generic abstraction layer has cost but no payoff. Direct coupling provides exhaustiveness checking, type narrowing, and refactoring safety that generic interfaces cannot. The typescript-eslint case study demonstrates the concrete costs of abstraction layers whose original justification has evaporated.

Consequences: Analysis adapters become non-portable across grammars (intentionally). Domain types like KindDefinition and Diagnostic gain concrete node types, making downstream consumers grammar-aware. New analysis adapters for the same grammar can import grammar types directly; new analysis adapters for a different grammar must define their own domain types.

ADR-002 PROPOSED Type-safe equation records with KindToNode mapping

Context: AttrDecl<K>.equations is typed Partial<Record<K, Function>>. The Function type erases equation signatures: you can assign a function with wrong parameters or return type, and TypeScript won't complain. The error only surfaces at runtime or in generated dispatch code.

Decision: Replace Function with a typed equation callback. The equation records will accept (ctx: KindCtx<KindToNode[K]>, ...args) => T where KindToNode is the grammar adapter's kind-to-node-type mapping. This requires AttrDecl to be parameterized by the KindToNode mapping, making core-codegen aware of the grammar's structural types at the port level.

Rationale: The compile-time machinery only reads fn.name from equation functions, but the declaration site is where authors work. Typing equations at declaration prevents class-of-bugs where wrong ctx type, wrong return type, or wrong parameter arity pass silently. The generated dispatch already casts to KindCtx<KindToNode[K]> — this just moves the type constraint upstream to the declaration.

Consequences: AttrDecl gains a second type parameter or a KindToNode constraint. withDeps return type needs adjustment. All existing equation functions already conform to the required signatures, so the migration is a type annotation change, not a behavior change. The compile pipeline reads fn.name at runtime (unaffected by type changes).

ADR-003 PROPOSED Typed projection functions via TypedAGNode

Context: AnalysisProjections<P>.projections are typed (root: Ctx) => P[Key]. The Ctx interface's attr() method returns any, so projection functions have no type checking on attribute access. root.attr('kindDefs') returns any despite KSCAttrMap knowing the return type is KindDefinition[].

Decision: Change the AnalysisProjections port so projection functions receive a TypedAGNode<M> instead of Ctx, where M is the generated attr map type. This parameterizes AnalysisProjections by both P (projection result) and M (attr map), and changes the engine's invocation to pass the typed tree node.

Rationale: Projection functions are the final extraction step — they read computed attributes and build the analysis result. If attr('kindDefs') returns KindDefinition[] instead of any, projection logic gets compile-time verification. The TypedAGNode<M> type already exists and is already used in tests (test/helpers/fixtures.ts); this extends its usage to the port level.

Consequences: AnalysisProjections<M, P> gains the M parameter. EvaluationTarget already has M. The engine's projection invocation changes from (projection as (root: Ctx) => unknown)(tree) to projection(tree as TypedAGNode<M>). Mock projections that ignore the root parameter are unaffected (arrow function parameter is unused).

Step 3: Type domain types with concrete node types (1C)

Change KindDefinition.node and Diagnostic.node from unknown to concrete grammar node types.

// BEFORE (types.ts)
export interface KindDefinition {
  node: unknown;
  ...
}
export interface Diagnostic {
  node: unknown;
  ...
}

// AFTER (types.ts)
import type { KSTypeAliasDeclaration, KSNode } from '../../../grammar/grammar/ts-ast/index.js';

export interface KindDefinition {
  node: KSTypeAliasDeclaration;   // was: unknown
  ...
}
export interface Diagnostic {
  node: KSNode;                   // was: unknown
  ...
}

Why KSTypeAliasDeclaration for KindDefinition? Kind definitions are extracted exclusively from TypeAliasDeclaration nodes in tryExtractKindDef(node: KSTypeAliasDeclaration). The node stored is always that type.

Why KSNode for Diagnostic? Violations can occur at any node kind (Identifier, BinaryExpression, DeleteExpression, etc.). KSNode is the union of all TS AST grammar nodes.

Cascade: consumers that reference .node

Most consumers don't access .node directly — they use .pos, .end, .message, .fileName. The consumers that DO access .node:

  • equations/definitions.ts:73 — assigns node in tryExtractKindDef. Already passes KSTypeAliasDeclaration. No change needed.
  • equations/predicates.ts:49 — assigns node: ctx.node in diag(). ctx.node is ASTNode; needs assertion ctx.node as KSNode. One-line change.
  • apps/dashboard/extract.ts — reads def.node for position info. Will gain typed access. No change needed.
  • test/helpers/fixtures.ts — doesn't access .node on KindDefinition or Diagnostic.

function evaluate(root: ASTNode): P {

Step 6: Typed equation records with KindToNode (4C)

Replace Function in AttrDecl<K> with typed equation callbacks. This requires AttrDecl to know about a KindToNode-style mapping.

6a. Define the equation function type

// packages/core-codegen/src/ports.ts — new types

/** Equation function: receives typed context, returns attribute value. */
export type EquationFn<T = unknown> =
  ((ctx: any, ...args: any[]) => T) & { deps?: string[] };

/** Equation map: per-kind equation functions. */
export type EquationMap<K extends string, T = unknown> =
  Partial<Record<K, EquationFn<T>>>;

6b. Update AttrDecl to use EquationFn

// packages/core-codegen/src/ports.ts — updated SynAttr, InhAttr

export interface SynAttr<K extends string = string> extends AttrBase {
  direction: 'syn';
  default?: AttrExpr;   // AttrExpr stays as-is (Function | null | ... | CodeLiteral)
  equations?: EquationMap<K>;  // was: Partial<Record<K, Function>>
}

export interface InhAttr<K extends string = string> extends AttrBase {
  direction: 'inh';
  rootValue: AttrExpr;
  parentEquations?: EquationMap<K>;  // was: Partial<Record<K, Function>>
}
Why any for ctx parameter? The compile pipeline only reads fn.name from equations — it never inspects the parameter types. Making the ctx parameter any in the port type allows concrete equation functions (which use KindCtx<KSIdentifier>) to be assigned without a cast. The type safety for ctx narrowing is enforced at the equation declaration site (where the author writes KindCtx<KSIdentifier>), not at the AttrDecl assignment site.

6c. Update withDeps return type

// packages/core-codegen/src/equation-utils.ts
export function withDeps<F extends EquationFn>(
  deps: string[], fn: F
): F & { deps: string[] } {
  (fn as EquationFn).deps = deps;
  return fn as F & { deps: string[] };
}

6d. Update compile.ts and pivot.ts consumers

The compile pipeline uses typeof v === 'function' checks and reads fn.name. These runtime patterns work identically with EquationFn since it extends Function. The changes are type annotation updates, not behavior changes.

  • compile.ts: collectEquationFunctions already checks typeof v === 'function' — no change needed in the runtime code. Type annotations in helper functions may need updating.
  • pivot.ts: Input/output types change from Record<string, Function> to Record<string, EquationFn>.
  • equation-utils.ts: collectDepsForAttr already casts to internal EquationFn type — align with the exported EquationFn.

Under Key Conventions, add:

## Explicit Grammar Coupling

Analysis adapters are explicitly coupled to their grammar adapter.
This is an intentional architectural choice (see ADR-001), not an accident:

- Analysis adapter barrels re-export their grammar (`grammar`,
  `TSNodeKind`, `KSNode`, `KindToNode`)
- Domain types use concrete node types (`KindDefinition.node: KSTypeAliasDeclaration`)
- Predicate constants are typed with the grammar's kind union
- Equation functions use `KindCtx<KSIdentifier>` for type-safe node access
- New analysis adapters should follow the same pattern:
  import concrete grammar types directly, don't abstract through generics

Each analysis adapter is for exactly one grammar. Multiple analyses
can target the same grammar, but each is explicitly bound to one.
The `K` type parameter on `AnalysisDecl<K>` links them at the type level.
The grammar re-export from the adapter barrel makes the binding discoverable.
