Trees that Grow & How GHC Works

A visual journey through the Haskell compiler's architecture and its extensible AST design pattern

1. The Big Picture: GHC's Pipeline

GHC transforms Haskell source code through eight major phases, each producing a different intermediate representation. Click any phase to learn more.

Parser

The parser reads Haskell source text and produces the HsSyn AST parameterised by the GhcPs phase ("Parsed Source"). At this stage:

Identifiers

Names are RdrName — raw strings from source code, not yet resolved.

Extension fields

XApp GhcPs = EpAnn — stores source locations and exact print annotations for formatting preservation.

No types yet

Type information doesn't exist. The typechecker hasn't run. Expression nodes carry no type annotations.

Renamer

The renamer resolves all names to unique Name values, catching scope errors. The AST becomes HsSyn GhcRn.

Identifiers

RdrName → Name. Every identifier now has a Unique — a globally unique integer. Shadowing and scope are resolved.

Free variables

Extension fields like XMatch GhcRn now carry NameSet — the set of free variables used by each sub-expression.

Import resolution

Qualified names, re-exports, and fixity are all resolved. Error messages report the original user-written name.

Typechecker

The typechecker infers and checks types, producing HsSyn GhcTc — the fully elaborated AST.

Identifiers

Name → Id (a.k.a. Var). Every binder now knows its full type. Id = Name + Type + IdInfo.

Type annotations everywhere

XApp GhcTc = Type — every application node stores the result type. The wrapper HsWrapper records coercions and dictionary evidence.

Evidence & dictionaries

Type class constraints become explicit dictionary-passing. HsWrap nodes insert coercions, type applications, and evidence bindings.

Desugarer

The desugarer transforms the rich HsSyn AST into Core — GHC's small, typed intermediate language.

Pattern match compilation

Nested patterns, guards, and view patterns are compiled into simple case expressions with flat alternatives.

do-notation → binds

do { x <- e1; e2 } becomes e1 >>= (\x -> e2). List comprehensions are similarly desugared.

Core has only 8 constructors

Var | Lit | App | Lam | Let | Case | Cast | Type — dramatically simpler than HsSyn's ~60 expression forms.

Core-to-Core Optimisation

GHC runs multiple passes over Core, each simplifying and optimising. This is where the magic happens.

Simplifier

Inlining, beta reduction, case-of-case, case-of-known-constructor. Runs multiple times.

Specialiser

Creates type-specialised versions of polymorphic functions to avoid dictionary overhead.

Float in/out

Moves let-bindings to reduce allocation (float-in) or enable sharing (float-out/full laziness).

Demand analysis

Strictness/absence analysis. Identifies which arguments are always forced, enabling unboxing and worker/wrapper.

STG — Spineless Tagless G-machine

Core is translated to STG — an IR that makes heap allocation and evaluation strategy explicit.

Every let = allocation

Each let binding in STG corresponds to a heap object (thunk, data constructor, or partial application).

Explicit closures

Free variables are listed explicitly. let f = \[x,y] \z -> body — free vars [x,y] and param z.

Update flags

Each closure is marked as updatable (thunk, evaluated at most once) or re-entrant (function, no update).

Cmm (C--)

Cmm is GHC's low-level imperative IR — a portable assembly language with an explicit stack and register model.

Basic blocks

Control flow is a graph of basic blocks. Each block ends with a branch, call, or return.

GC info tables

Every heap object and stack frame has an info table describing its layout for the garbage collector.

Platform-independent

Cmm abstracts over registers and calling conventions. The native code generator lowers it to actual machine instructions.

Code Generation

The final step: Cmm is lowered to machine code via one of three backends.

NCG (Native Code Gen)

GHC's own backend. Generates x86-64, AArch64 assembly directly. Fastest compile times.

LLVM backend

-fllvm emits LLVM IR. Better at loop optimisation and SIMD. Used for numeric code.

JavaScript backend

Targets JavaScript/WebAssembly for running Haskell in browsers. Added in GHC 9.6.

2. The Problem: One AST, Many Phases

GHC's AST (HsSyn) represents Haskell syntax. But each compiler phase needs different data on the same tree structure. Before Trees that Grow, GHC had two bad options:

Option A: Separate AST types

Massive code duplication. Every function processing the AST needs three versions. Bug fixes must be applied three times.

Option B: One type with Maybe fields

data HsExpr = HsApp
  { appFun  :: HsExpr
  , appArg  :: HsExpr
  , appType :: Maybe Type  -- Nothing before TC
  , appFVs  :: Maybe NameSet -- Nothing before Rn
  , appAnn  :: Maybe EpAnn  -- Nothing after parse
  }

Littered with Maybe. No static guarantee that types exist after typechecking. Runtime crashes from unexpected Nothing. Wastes memory.

What we actually want

One AST definition with type-safe, phase-indexed extension points — so the type system guarantees that type info exists after typechecking and source locations exist after parsing, with zero runtime overhead.

3. Trees that Grow — The Pattern

The key insight: parameterise the AST by a phase type, and use type families to vary the data stored at each node per phase.

Step 1: Define extension type families

-- A "phase" type: just a tag, no runtime values
data GhcPass (c :: Pass) where ...
data Pass = Parsed | Renamed | Typechecked

type GhcPs = GhcPass 'Parsed
type GhcRn = GhcPass 'Renamed
type GhcTc = GhcPass 'Typechecked

-- Extension point: what does HsApp carry in each phase?
type family XApp p

Step 2: The AST uses the type family

data HsExpr p
  = HsApp (XApp p) (HsExpr p) (HsExpr p)
  | HsVar (XVar p) (LIdP p)
  | HsLam (XLam p) (MatchGroup p (LHsExpr p))
  | ...
  | XExpr  (XXExpr p)       -- extension constructor!

Step 3: Instantiate for each phase

GhcPs (Parsed)
GhcRn (Renamed)
GhcTc (Typechecked)
type instance XApp GhcPs = EpAnn             -- source location + print info
type instance XVar GhcPs = EpAnn
type instance LIdP GhcPs = LocatedN RdrName  -- raw, unresolved name
type instance XXExpr GhcPs = DataConCantHappen -- no extra constructors

After parsing, identifiers are raw strings (RdrName) and nodes carry exact-print annotations for round-tripping source code.

type instance XApp GhcRn = NoExtField        -- no extra data needed
type instance XVar GhcRn = NoExtField
type instance LIdP GhcRn = LocatedN Name     -- resolved unique Name
type instance XXExpr GhcRn = DataConCantHappen

After renaming, every identifier has a globally unique Name. Scope errors have been caught. Exact-print info is dropped (or moved).

type instance XApp GhcTc = Type             -- result type of application!
type instance XVar GhcTc = Type             -- type of the variable
type instance LIdP GhcTc = LocatedN Id      -- Id = Name + Type + IdInfo
type instance XXExpr GhcTc = XXExprGhcTc    -- HsWrap, etc.

After typechecking, every expression node carries its Type. The XXExpr extension constructor adds HsWrap for coercions and evidence application.

6. Core — The Heart of GHC

After desugaring, the AST changes entirely. Core has only 8 expression forms — making it easy to optimise.

▶ What gets desugared?

do-notation

do { x <- e1; e2 } → e1 >>= \x -> e2

where clauses

Become let bindings in Core

if/then/else

if c then t else f → case c of { True -> t; False -> f }

Guards

Multi-way guards become nested case

List comprehensions

[x | x <- xs, even x] → recursive build/foldr

Type classes

Constraints become explicit dictionary arguments

8. The Uninhabited Type Trick

One elegant detail: when a phase doesn't need an extension constructor, TTG uses an uninhabited type to make it impossible to construct.

The Pattern

-- An empty type: no constructors!
data DataConCantHappen
  -- (no constructors at all)

-- In GhcPs and GhcRn, no extra
-- expression forms exist:
type instance XXExpr GhcPs
  = DataConCantHappen

-- So this pattern is impossible:
-- case e of XExpr x -> ...
-- x :: DataConCantHappen
-- can't exist!

9. GHC Runtime: Heap Objects

GHC's runtime represents every Haskell value as a heap object with a uniform layout. Understanding this helps explain why STG and Cmm exist.

▶ What are these closure types?
  • CONSTR — A fully evaluated data constructor (e.g., Just 42, (:) 1 [])
  • FUN — A function closure. Payload = free variables. Entry code = the function body.
  • THUNK — An unevaluated expression (lazy!). Payload = free vars. Entry code evaluates it, then overwrites the thunk with an indirection to the result.
  • PAP — Partial application. A function applied to some but not all arguments.
  • AP — A generic application node. Used by the GC and the STG machine.
  • IND — Indirection. Points to the real value. Left behind when a THUNK is evaluated. Cleaned up by GC.

10. TTG Beyond GHC

The Trees that Grow pattern generalises to any multi-phase compiler. Here's how it maps to other architectures:

TypeScript Compiler

TSC uses a single Node type with optional fields (.symbol, .type). This is "Option B" — no type safety between phases. TTG would give compile-time guarantees.

Rust (rustc)

Rust uses separate AST types: ast::Expr → hir::Expr → mir::Statement. Clean but involves conversion functions. TTG offers a middle ground.

Attribute Grammars

AG systems (JastAdd, Silver) solve the same problem differently: attributes are externally computed properties on nodes, not fields inside them. The tree shape stays fixed; analysis results live separately.

Roslyn (C#)

Roslyn's SyntaxNode is immutable and phase-independent. Semantic info lives in a separate SemanticModel. Clean separation, but loses the "one tree with everything" benefit.

11. Test Your Understanding

Quiz 1: What problem does Trees that Grow solve?

TTG parameterises an AST by a phase index and uses type families to vary the data at each node. This avoids both code duplication (separate types) and runtime unsafety (Maybe fields).

Quiz 2: What is DataConCantHappen?

DataConCantHappen has zero constructors, so a field of this type can never be constructed. When XXExpr GhcPs = DataConCantHappen, the XExpr constructor is statically impossible to create.

Quiz 3: How many expression forms does Core have?

Core is a small typed lambda calculus with just 8 constructors. Everything else (do-notation, guards, where-clauses, list comprehensions) is desugared into these 8 forms.

Quiz 4: What type does an identifier have in each GHC phase?

GHC uses RdrName (raw from source), Name (globally unique after renaming), and Id (Name + Type + IdInfo after typechecking). This is controlled by the IdP type family.

Quiz 5: What is a THUNK in GHC's runtime?

THUNKs implement laziness. They store free variables as payload. When evaluated, the entry code runs, computes the result, and overwrites the THUNK with an IND (indirection) pointing to the result. The GC later short-circuits indirections.

12. Summary

Key takeaways

  • GHC compiles Haskell through 8 phases: Parse → Rename → Typecheck → Desugar → Core optimisation → STG → Cmm → Code gen
  • Trees that Grow lets the first 3 phases share one AST type (HsSyn p) while varying the data at each node per phase
  • Type families (XApp p, IdP p) resolve to different concrete types depending on the phase parameter
  • Extension constructors (XXExpr p) add phase-specific node variants — locked out in other phases via uninhabited types
  • Core is a separate, tiny (8 constructor) IR where all the serious optimisation happens
  • STG makes allocation and evaluation explicit, mapping directly to GHC's runtime heap objects
