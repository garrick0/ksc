The entire pipeline lives in src/program.ts and fits in a few lines:

export function createProgramFromTSProgram(
  tsProgram: ts.Program,
  config?: KindScriptConfig,
): KSProgramInterface {
  // 1. Convert TS AST → KSC AST (raw tree, no navigation)
  const ksTree = buildKSTree(tsProgram);

  // 2. Evaluate the binder spec — stamps tree + installs attributes + projects
  const allDefs = evaluate(createBinderSpec(), ksTree.root) ?? [];

  return {
    getTSProgram: () => tsProgram,
    getKindDefinitions: () => allDefs,
    getKSTree: () => ksTree,
    // ... other accessors
  };
}
Key insight: The consumer (KSC) defines what to compute via an AGSpec. The AG library handles how — tree stamping, attribute installation, orchestration, caching. The consumer never calls stampTree or applyAttributes directly.

02TS → KSC AST Conversion

buildKSTree() mirrors the entire TypeScript AST into typed KSC nodes.

1 buildKSTree(tsProgram) ▼

Entry point in src/pipeline/convert.ts. Iterates over all source files in the TypeScript program, converts each one to a KSCompilationUnit, wraps them in a synthetic KSProgram root node.

const root: KSProgram = {
  kind: 'Program',
  compilationUnits,   // KSCompilationUnit[]
  children: compilationUnits,
  pos: 0, end: 0, text: '',
};
return { root };
2 convertSourceFile(sf) ▼

Converts a single ts.SourceFile into a KSCompilationUnit. Calls convertNode() on each top-level statement to build the children array. Stores fileName, sourceText, and lineStarts.

3 convertNode(node, sf) → dispatch ▼

Central recursive dispatch. For each TS node:

  1. Collect child nodes via ts.forEachChild(), recursively converting each
  2. Look up specificConverters[node.kind] to find the typed converter function
  3. Call the converter, which returns a strongly-typed KSNode

Every ts.SyntaxKind has a specific converter — no generic fallback. The converter registry has ~350 entries covering all declarations, statements, expressions, type nodes, tokens, and keywords.

4 KSNode output shape ▼

Every node in the KSC AST has this common shape (defined by KSNodeBase):

{
  kind: 'TypeAliasDeclaration',  // string discriminant
  pos:  42,                       // source position
  end:  108,                      // source end
  text: 'type NoImports = ...',   // source text
  children: [...],               // child KSNodes
  tsNode: <ts.Node>,             // back-reference
  // + typed accessors per node kind
  name: { kind: 'Identifier', escapedText: 'NoImports' },
  type: { kind: 'TypeReference', ... },
}

The getChildren(node) function from ast.ts simply returns node.children. This is passed to the AG library as the tree structure accessor.

Output: buildKSTree() returns { root: KSProgram } — a raw tree with no navigation properties and no attributes. That's the AG library's job.

03The AGSpec — Cousot Phase 1

A complete attribute grammar is a single AGSpec object. The binder defines one via createBinderSpec().

AGSpec Interface

interface AGSpec<N extends object, R = unknown> {
  name: string;                      // Unique identifier ('ksc-binder')
  domain: AGDomain<N, R>;           // Abstract domain + result extraction
  children: (node: N) => N[];     // Tree structure accessor
  attributes: AttributeMap<N>;     // Named attribute definitions (equations)
  deps?: string[];                  // Dependencies on other specs
}

AGDomain Interface

interface AGDomain<N extends object, R = unknown> {
  name: string;                      // Human-readable name
  project?: (root: N) => R;       // Extract results from attributed tree
}

The Binder's Spec

Here is the complete createBinderSpec() showing how the binder defines its analysis as an AGSpec:

export function createBinderSpec(): AGSpec<KSNode, KindDefinition[]> {
  let nextDefId = 0;

  const kindDefs = match<KSNode, KindDefinition[]>('kind', { ... });
  const defLookup = lookup<KSNode, string, KindDefinition>((node) => { ... });

  return {
    name: 'ksc-binder',
    domain: { name: 'Kind Definitions', project: ... },
    children: getChildren,        // from ast.ts: (node) => node.children
    attributes: { kindDefs, defLookup },
  };
}

The spec packages everything: tree structure, attribute equations, domain, and result extraction. The AG library handles the rest.

const kindDefs = match<KSNode, KindDefinition[]>('kind', {
  // For CompilationUnit nodes: scan children for Kind<...> type aliases
  CompilationUnit: (cu) => {
    const defs: KindDefinition[] = [];
    for (const stmt of cu.children) {
      if (stmt.kind !== 'TypeAliasDeclaration') continue;
      const def = tryExtractKindDef(stmt, () => `kdef-${nextDefId++}`);
      if (def) defs.push(def);
    }
    return defs;
  },
  // For all other node kinds: empty array
  _: () => [],
});

match dispatches on node.kind. CompilationUnit nodes scan their children for type X = Kind<{ ... }> declarations. All other nodes get an empty array.

const defLookup = lookup<KSNode, string, KindDefinition>((node) => {
  if (node.kind !== 'CompilationUnit') return [];
  // Accesses kindDefs attribute — triggers lazy computation
  const defs: KindDefinition[] = (node as any).kindDefs;
  return defs.map(d => [d.name, d]);
});

lookup walks the entire tree once (from root), collecting [name, definition] pairs from CompilationUnit nodes into a Map. Every node then gets a lookup function: (name: string) => KindDefinition | undefined.

Cross-attribute dependency: defLookup accesses node.kindDefs inside its collect function. Since both are lazy getters, this triggers the kindDefs computation on demand. No explicit ordering needed. 
domain: {
  name: 'Kind Definitions',
  project: (root) => {
    const allDefs: KindDefinition[] = [];
    for (const cu of (root as any).compilationUnits
                       ?? (root as any).$children ?? []) {
      allDefs.push(...((cu as any).kindDefs ?? []));
    }
    return allDefs;
  },
},

domain.project(root) is called after all attributes are installed. It walks the CompilationUnits, accessing .kindDefs on each — which triggers the lazy computation. Returns the aggregated KindDefinition[] as the final analysis result.

Separation of concerns: The binder never calls stampTree, applyAttributes, or any orchestration function. It only defines equations and a projection. The AG library's evaluate() handles everything else.

04The Evaluator — Cousot Phase 2

evaluate(spec, root) is the weak interpreter. It takes a specification and executes it over the AST.

stampTree(root, spec.children)

BFS traversal of the tree. On every node, stamps six navigation properties via Object.defineProperty:

PropertyTypeDescription
$parentN | undefinedParent node (undefined for root)
$childrenN[]Child nodes array
$indexnumberPosition in parent's children (-1 for root)
$rootbooleantrue only for the root node
$prevN | undefinedPrevious sibling
$nextN | undefinedNext sibling

These are non-enumerable data properties — they don't show up in JSON.stringify or for...in but are directly accessible. The spec.children function (which is getChildren from ast.ts) tells the stamper how to find children of each node.

applyAttributes(root, spec.attributes)

DFS post-order traversal using $children. For every node and every attribute:

for (const [name, def] of entries) {
  def.install(node, name);  // installs a lazy getter as node[name]
}

After this walk, every node in the tree has lazy getters for kindDefs and defLookup. No computation has happened yet. Values are computed on-demand when the property is first accessed.

For a tree with 500 nodes and 2 attributes, this installs 1,000 lazy getters — all via Object.defineProperty.

spec.domain.project(root)

The final step: extract results from the attributed tree. The projection function accesses attributes on nodes, which triggers lazy computation and caching.

For the binder, project accesses .kindDefs on each CompilationUnit. This triggers the match dispatch, which scans children for Kind<...> type aliases. The results are cached as data properties — subsequent accesses are free.

The return type R of AGSpec<N, R> determines the projection return type. For the binder, R = KindDefinition[].

evaluate() implementation

export function evaluate<N, R>(
  spec: AGSpec<N, R>,
  root: N,
): R | undefined {
  stampTree(root, spec.children);      // 1. Navigation properties
  applyAttributes(root, spec.attributes); // 2. Lazy getters

  if (spec.domain.project) {
    return spec.domain.project(root);   // 3. Extract results
  }
  return undefined;
}

evaluateAll() — Multi-Spec Composition

For multiple analyses over the same tree, evaluateAll(specs, root) topologically sorts specs by their deps field, stamps the tree once, then applies each spec's attributes in order:

export function evaluateAll<N>(
  specs: AGSpec<N, any>[],
  root: N,
): Map<string, unknown> {
  const sorted = topoSort(specs);    // order by deps
  const results = new Map();
  let treeStamped = false;

  for (const spec of sorted) {
    if (!treeStamped) {
      stampTree(root, spec.children);
      treeStamped = true;
    }
    applyAttributes(root, spec.attributes);
    if (spec.domain.project)
      results.set(spec.name, spec.domain.project(root));
  }
  return results;
}
Demand-driven evaluation: applyAttributes only installs getters — no values are computed until accessed. The project call triggers the cascade: accessing one attribute may trigger others via cross-attribute dependencies, all resolved lazily.

05Attribute Primitives

The AG library provides factories that return AttributeDef objects. Each has an install(node, key) method.

syn(compute)
Synthesized
Cached lazy getter. Computes once, replaces with data property.
match(discrim, eqs)
Per-Production
Dispatches on a discriminant field. JastAdd-style per-production equations.
inh(rootVal, eq?)
Inherited
Parent-defined. eq(parent, child, index) provides value; undefined auto-propagates.
coll(init, combine, fn)
Collection
DFS subtree walk. Fold contributions from descendants.
lookup(collect)
Reference
Full-tree scan to build Map. Returns lookup function shared by all nodes.
uncached · paramSyn · circular · chain
Advanced
Recomputing, parameterized, fixed-point, and L-attributed flow.

syn(compute) → AttributeDef

export function syn<N, V>(compute: (node: N) => V): AttributeDef<N, V> {
  return {
    install(node: N, key: string): void {
      installLazy(node, key, compute);
    },
  };
}

The fundamental building block. All other primitives ultimately call syn internally. The compute function receives the node and returns the attribute value. It runs exactly once per node — subsequent accesses read the cached data property.

match(discriminant, equations) → AttributeDef

match<KSNode, KindDefinition[]>('kind', {
  CompilationUnit: (cu) => /* scan for Kind<> types */,
  TypeAliasDeclaration: (ta) => /* extract definition */,
  _: () => [],  // default fallback
})

Dispatches on node[discriminant] to select the right equation. The _ key provides a default. Under the hood, wraps syn with a dispatch layer. This is the JastAdd pattern of per-production attribute equations.

inh(rootValue, eq?) → AttributeDef

// Inherited attribute: parent-defined (JastAdd/Silver-style)
inh(
  new Set(),                           // value at root
  (parent, child, childIndex) => {  // parent equation
    if (parent.kind === 'Module')
      return parent.localScope;
    return undefined;                  // auto-propagate
  },
)

Parent-defined inherited attribute. Root gets rootValue (or rootValue(root) if a function). Non-root: eq(parent, child, childIndex) returns the value, or undefined to auto-propagate (use parent's own value).

If eq is omitted entirely, the root value propagates unchanged to all descendants (broadcast).

coll(init, combine, contribute) → AttributeDef

coll<KSNode, string[]>(
  () => [],                                   // init accumulator
  (acc, names) => [...acc, ...names],          // combine
  (node) => {                                 // contribute
    if (node.kind === 'ImportDeclaration')
      return [node.moduleSpecifier.text];
    return undefined;                          // skip
  },
)

DFS walks the subtree rooted at the evaluation node. For each descendant, calls contribute. Non-undefined results are folded via combine. Approximation of JastAdd's coll/contributes pattern.

lookup(collect) → AttributeDef

lookup<KSNode, string, KindDefinition>((node) => {
  if (node.kind !== 'CompilationUnit') return [];
  const defs = (node as any).kindDefs;  // triggers kindDefs computation
  return defs.map(d => [d.name, d]);
})

On first access from any node: walks $parent to find root, DFS walks the full tree via $children, calls collect on every node, builds a Map<K,V>. Returns a lookup function. The map is built once and shared across all nodes.

Other Primitives

PrimitiveBehavior
uncached(compute)Like syn but the getter never replaces itself. Recomputes on every access.
paramSyn(compute)Installs a function (param: P) => V. Results cached by (node, param) pair in an internal Map.
circular(init, compute)Fixed-point iteration. Uses Symbol-keyed properties on nodes for state. Iterates until values converge.
chain(init, update)L-attributed flow. Left-to-right sibling propagation via $prev/$next. Returns { inAttr, outAttr } pair.
Common contract: Every primitive returns an AttributeDef<N, V> with a single method: install(node, key). This is the only interface between the primitives and the evaluator. The evaluator doesn't know or care whether an attribute is synthesized, inherited, or a collection.

06The Property-Stamp Mechanism

How installLazy implements JastAdd's _computed/_value pattern in pure JavaScript.

installLazy() Implementation

export function installLazy<N, V>(
  node: N, key: string, compute: (node: N) => V,
): void {
  Object.defineProperty(node, key, {
    configurable: true,   // allows replacement
    enumerable: true,
    get() {
      const value = compute(node);
      // Replace getter with plain data property
      Object.defineProperty(node, key, {
        value,
        writable: false,      // immutable after computation
        configurable: false,   // can't be replaced again
        enumerable: true,
      });
      return value;
    },
  });
}

Before & After Property Access

// node.kindDefs is a getter (configurable: true)
Object.getOwnPropertyDescriptor(node, 'kindDefs')
// → { get: [Function], configurable: true, enumerable: true }

// Property is a getter — no value stored yet
// First access triggers the getter:
const defs = node.kindDefs;  // computes + caches

Object.getOwnPropertyDescriptor(node, 'kindDefs')
// → { value: [...], writable: false, configurable: false, enumerable: true }

// Property is now a plain data property — zero-cost reads forever
node.kindDefs === defs;  // true, same reference, no computation
// JastAdd generates this Java pattern:
private boolean kindDefs_computed = false;
private List<KindDef> kindDefs_value;

public List<KindDef> kindDefs() {
  if (kindDefs_computed) return kindDefs_value;
  kindDefs_computed = true;
  kindDefs_value = equation();
  return kindDefs_value;
}

// installLazy does the same thing, but using
// Object.defineProperty to swap getter → data property.
// Subsequent accesses are plain property reads (V8 optimized).
Performance: After first access, attribute reads are direct property lookups — the same cost as reading node.kind or node.pos. No function call overhead, no Map lookup, no WeakMap dereference. V8 treats data properties as inline slots.

07Concrete Walkthrough

Follow a type NoImports = Kind<{ noImports: true }> declaration through the entire pipeline.

1 TypeScript parses source ▼

TypeScript's parser produces a ts.TypeAliasDeclaration node with SyntaxKind.TypeAliasDeclaration. The type is a ts.TypeReferenceNode pointing to Kind with one type argument: a ts.TypeLiteralNode containing a property signature noImports: true.

2 convertNode produces KSTypeAliasDeclaration ▼

The typed converter for TypeAliasDeclaration creates a KSC node with structured accessors:

{
  kind: 'TypeAliasDeclaration',
  name: { kind: 'Identifier', escapedText: 'NoImports' },
  type: {
    kind: 'TypeReference',
    typeName: { kind: 'Identifier', escapedText: 'Kind' },
    typeArguments: [{
      kind: 'TypeLiteral',
      members: [{ kind: 'PropertySignature', name: ..., type: ... }]
    }]
  },
  children: [...], pos, end, text, tsNode,
}
3 evaluate() stamps tree + installs getters ▼

evaluate(createBinderSpec(), root) runs. stampTree installs $parent, $children, etc. on every node. applyAttributes installs lazy getters for kindDefs and defLookup on every node. No computation happens yet.

4 domain.project triggers computation ▼

domain.project(root) accesses cu.kindDefs on each CompilationUnit. This triggers the match dispatch: node.kind === 'CompilationUnit' selects the CompilationUnit equation.

That equation iterates cu.children, finds the TypeAliasDeclaration, calls tryExtractKindDef():

  • Checks node.type.kind === 'TypeReference' ✔
  • Checks typeRef.typeName.escapedText === 'Kind' ✔
  • Checks typeArguments.length === 1 ✔
  • Checks typeArguments[0].kind === 'TypeLiteral' ✔
  • Extracts properties: { noImports: true }
5 Result returned ▼

evaluate() returns the projection result:

[{
  id: 'kdef-0',
  name: 'NoImports',
  properties: { noImports: true },
  node: <KSTypeAliasDeclaration>,
}]

The kindDefs value is now cached as a data property on the CompilationUnit node. The defLookup function is installed but hasn't been triggered yet — it will compute lazily when first accessed (e.g., by a future checker spec).

  KSC Code                    AG Library                    AST Nodes

    └─ lookup(...) ──────────→ returns AttributeDef ──────→ (stored in spec)

  evaluate(spec, root) ──────→ stampTree(root, children) ──→ node.$parent = ...

Three touchpoints: KSC only touches the AG library in three places: (1) match() and lookup() to define attributes, (2) packaging them into an AGSpec, and (3) calling evaluate(spec, root). Everything else is internal to the library.

Q

What are the three steps that evaluate(spec, root) performs internally?

A

1. stampTree(root, spec.children) — installs $parent, $children, $index, $root, $prev, $next on every node.
2. applyAttributes(root, spec.attributes) — DFS walk installing lazy getters.
3. spec.domain.project(root) — extracts final results from the attributed tree.

Q

What is an AGSpec and what five fields does it contain?

A

A complete attribute grammar specification (Cousot Phase 1). Fields: name (unique identifier), domain (AGDomain with name and optional project function), children (tree structure accessor), attributes (named AttributeDef map), and optional deps (names of prerequisite specs).

Q

What does installLazy(node, key, compute) do, and why is it efficient?

A

It installs a getter via Object.defineProperty that, on first access, calls compute(node), then replaces itself with a plain data property holding the result. Subsequent reads are direct property lookups with zero function-call overhead — V8 treats them as inline slots. This is JastAdd's _computed/_value pattern.

Q

What is domain.project for and when does it get called?

A

domain.project(root) extracts the final analysis results from the attributed tree. It's called as the last step of evaluate(), after tree stamping and attribute installation. For the binder, it walks CompilationUnits and accesses .kindDefs on each, which triggers the lazy computation cascade and returns the aggregated KindDefinition[].

Q

How does match('kind', equations) work, and what is the _ key?

A

match creates a per-production dispatched attribute. It reads node[discriminant] (e.g., node.kind), looks up the matching equation in the map, and calls it. The _ key is the default fallback when no specific equation matches. Under the hood it wraps syn() with dispatch logic, so results are cached per node.

Q

How does the binder's defLookup attribute depend on kindDefs, and how is this dependency resolved?

A

defLookup uses lookup() which, inside its collect function, accesses (node as any).kindDefs. Since both are lazy getters installed on the same nodes, accessing .kindDefs inside defLookup's computation triggers its lazy getter, computing the kind definitions on demand. No explicit ordering is needed — demand-driven evaluation handles it naturally.

Q

What is the common interface that all attribute primitives (syn, match, down, coll, lookup) share?

A

They all return an AttributeDef<N, V> which has a single method: install(node: N, key: string): void. This is the only interface the evaluator needs — it calls def.install(node, name) for every node and every attribute during applyAttributes(). The evaluator doesn't know whether an attribute is synthesized, inherited, or a collection.

Q

What does evaluateAll() do differently from evaluate(), and when would you use it?

A

evaluateAll(specs, root) applies multiple AG specifications to the same tree. It topologically sorts specs by their deps field, stamps the tree only once (reusing it across specs), then applies each spec's attributes in dependency order. Use it when you have multiple analysis passes (e.g., binder then checker) that build on each other's results. Returns a Map<string, unknown> of spec name to projection result.
