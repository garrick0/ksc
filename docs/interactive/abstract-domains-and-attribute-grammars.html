01Abstract Interpretation Domains

In abstract interpretation, a domain is a simplified universe in which you execute a program. Different domains offer different trade-offs between cost and precision — the core idea is to use the weakest (cheapest) domain that still gives you the answers you need.

The Domain Hierarchy

Domains are organized from cheap-but-imprecise to expensive-but-precise. Click each domain to explore it.

Polyhedra domain — Fully relational. Represents arbitrary conjunctions of linear inequalities: a&sub1;x&sub1; + ... + aₙxₙ ≤ c. Tracks all linear relationships between variables. Most precise, but exponential worst-case cost. Used when you need exact inter-variable relationships (e.g., x ≤ y + 3).
Domain Type Constraints Cost Precision
Sign Non-relational x ∈ {+, −, 0} O(1) Very low
Interval Non-relational l ≤ x ≤ u per variable O(n) Low
Octagon Weakly relational ±X ± Y ≤ c O(n³) Medium
Polyhedra Fully relational Σaᵢxᵢ ≤ c Exp High
Key insight: Moving up the hierarchy, each domain can express strictly more relationships between variables, but at increasing cost. The art of abstract interpretation is choosing the cheapest domain that gives sufficient precision for your particular analysis question.

Lattice Structure

Every abstract domain is organized as a complete lattice — a partially ordered set with:

  • ⊥ (bottom) — least element, representing "no information" or "unreachable"
  • ⊤ (top) — greatest element, representing "all possible values"
  • ⊔ (join) — least upper bound, combining information from two paths
  • ⊓ (meet) — greatest lower bound

The lattice structure ensures that iterating monotone transfer functions will converge to a fixed point — a stable solution where no further iteration changes the result.

Widening and Narrowing

On infinite-height lattices (like intervals over integers), iteration may not converge. Two operators handle this:

Operator Symbol Purpose Direction
Widening ∇ Forces convergence by over-approximating — "jumps ahead" past oscillating chains Ascending (up)
Narrowing Δ Recovers precision lost by widening — iterates downward while staying above the true fixpoint Descending (down)
Watch out: Widening trades precision for termination. Narrowing partially recovers that precision. Together they let you analyze programs with unbounded loops — something impossible with finite-height lattices alone.

Domain Composition

Domains can be combined to get richer abstract states:

Composition How it works Information flow
Direct (Cartesian) product Run two analyses independently, take the pair of results None — domains are isolated
Reduced product Run two analyses, then let them mutually refine each other via reduction operators Bidirectional — each domain can sharpen the other

02Galois Connections

The formal bridge between concrete program execution and abstract analysis. This is what makes abstract interpretation sound — guaranteeing that if the abstract analysis says "safe," the concrete program really is safe.

The Definition

A Galois connection is a pair of monotone functions between a concrete lattice (C, ≤) and an abstract lattice (A, ≤):

// Abstraction: concrete → abstract (best approximation)
α : C → A

// Concretization: abstract → concrete (what values does this represent?)
γ : A → C

// The defining property:
α(c) ≤ a   ⇔   c ≤ γ(a)

α(x) gives the most precise abstract element whose concretization contains x. This guarantees soundness: any property proven in the abstract domain truly holds in the concrete.

Concrete: {-7, 0, 3, 42}
α →
Abstract: [-7, 42]
Concrete: all integers in [-7, 42]
← γ
Abstract: [-7, 42]
Key insight: α over-approximates (may include extra values), but never under-approximates (never misses real values). This is why abstract interpretation proofs are sound — false positives are possible, false negatives are not.

Galois Connections Between Domains

The domain hierarchy from Section 1 forms a chain of Galois connections. Each more precise domain can be abstracted down to a less precise one:

Polyhedra  —α→  Octagon  —α→  Interval  —α→  Sign
    ←γ—              ←γ—              ←γ—

Each abstraction step loses information but reduces cost. "Apply at the weakest possible domain" means: walk this chain leftward until you find the cheapest domain where the analysis still tells you what you need to know.

What Subsumes What?

Abstract interpretation is the strictly more general framework. Every AG-based analysis can be seen as an abstract interpretation: the tree is the "program," attributes are abstract values, and evaluation is fixpoint iteration. AI can also handle arbitrary graph-structured dependencies (CFGs), infinite-height lattices (via widening), and provides formal soundness guarantees via Galois connections.

AI's generality comes at a cost: you need to manually specify domain structure, transfer functions, widening strategies, and iteration order. There is less built-in structure to guide you.

AGs add structural discipline that AI lacks. The tree structure, the synthesized/inherited distinction, and demand-driven evaluation give you a restricted but more tractable framework. You don't have to think about iteration order, widening placement, or convergence strategies — the AG evaluator handles it.

This restriction is also a feature: the tree topology prevents many of the pathological cases that make AI hard (aliasing, complex join points, non-terminating widening sequences).

The sweet spot would combine AG's structural discipline with AI's domain engineering. You'd get:

  • Tree-structured specification (easy to write, reason about, compose)
  • Domain-polymorphic attributes (write once, run at different precision levels)
  • Soundness proofs via Galois connections (formal guarantees)
  • Widening support for infinite-height domains (loop analysis)

No existing system provides this combination. This is the research opportunity.

The deep correspondence: Circular AG attributes over a finite-height lattice with monotone equations are abstract interpretation — restricted to tree topology and finite domains. The question is whether the AG framework should make this correspondence explicit and provide domain engineering infrastructure.

Synthesized Attributes

Standard bottom-up computation. Each node computes a value from its children's attributes and passes it up. Example: type() of an expression synthesized from operand types.

Inherited Attributes

Context passed downward. The parent (or an ancestor) provides a value that children use. Example: env() mapping names to types, inherited from enclosing scope.

Reference Attributes

Values are references to other AST nodes, creating a graph structure overlaid on the tree. Enables name resolution (reference → declaration), type hierarchy links, and scope chains. This is what makes JastAdd's RAGs (Reference Attribute Grammars) powerful.

Circular Attributes — The AI Connection

An attribute may depend on itself (directly or transitively). The evaluator:

  1. Initializes the attribute to its bottom value
  2. Evaluates the equation (which may read the attribute's own current value)
  3. If the result changed, iterates until stable
  4. Terminates when a fixed point is reached

Requirement: values must form a lattice of finite height and equations must be monotone. This is exactly the same contract as abstract interpretation on finite domains.

Example: computing FIRST and FOLLOW sets for a grammar. Start with empty sets, iterate until no new symbols are added. The lattice is 𝓁(power set of terminals), height = |terminals|.

Collection Attributes

Gather contributions from anywhere in the tree. Each contributing node adds a value, and the collection attribute combines them (e.g., set union). Useful for error collection, declaration gathering, and global analyses.

Higher-Order Attributes

The attribute value is itself an AST subtree (a "nonterminal attribute"). This is powerful for intermediate representations: you can compute a desugared or simplified form as an attribute, then analyze that. It's a form of abstraction — translating to a simpler representation for further analysis.

CRAGs: Circular + Reference

The combined formalism (Magnusson & Hedin, 2007) merges reference attributes (graph overlay) with circular attributes (fixpoint). This enables analyses like:

  • Transitive closure over a reference graph (circular + reference)
  • Dataflow analysis on CFGs built from reference attributes
  • Reaching definitions / live variables (Söderberg et al., 2013)
The gap: JastAdd has no abstract domain combinators, no Galois connections, no widening, and no automatic refinement. The lattice is implicit — the programmer manually ensures finite height and monotonicity. There is no framework-level concept of "domain" that you could swap for a more or less precise one.

05Silver

An extensible AG system from the University of Minnesota focused on modular language composition, not analysis power.

Key Concepts

1 Forwarding ▼

A production can forward to another production, providing default semantics. New language constructs get semantics automatically by forwarding to host-language constructs. This is a form of abstraction: the forward target is a "simpler view" used for any attributes not explicitly defined on the new construct.

-- A new 'foreach' construct forwards to a while loop
abstract production forEach ...
forwards to whileLoop(init, cond, body, update)
2 Higher-Order Attributes ▼

Like JastAdd, attribute values can be entire syntax trees. Used to build abstract syntax from concrete syntax, or to create intermediate representations. A form of domain construction — you compute a different representation as an attribute.

3 Modular Well-Definedness ▼

Silver's distinguishing contribution: a modular analysis that guarantees independently developed language extensions compose into a well-defined AG. If extension A and extension B both pass Silver's modular analysis, then A+B is guaranteed to be well-defined — no diamond problem, no missing equations.

This is about composability guarantees, not analysis precision or abstract domains.

4 Strategy Attributes ▼

Defined with strategy expressions for term rewriting: rules, fail, identity, sequence, choice, and traversals. These combine rewriting with attribute evaluation — useful for optimization passes and normalization.

Silver vs. JastAdd for Analysis

Feature JastAdd Silver
Circular attributes Yes — iterate to fixpoint No
Reference attributes Yes — graph overlay Yes
Modular composition Limited Formally guaranteed
Forwarding No Yes
Abstract domains No No
Dataflow support Via circular attrs + reference attrs Domain-specific constructs
Key insight: Silver focuses on language extensibility (composing independent extensions safely). JastAdd focuses on analysis power (circular attributes for fixpoints). Neither provides abstract interpretation infrastructure. For domain-related work, JastAdd is the more relevant reference.

06Facebook Infer

A production abstract interpretation platform. This is what full domain engineering looks like in practice.

The Infer.AI Framework

To build a checker, you provide two things:

(* OCaml module signature for an Infer checker *)
module type AbstractDomain = sig
  type t                              (* abstract state type *)
  val leq : t → t → bool            (* partial order *)
  val join : t → t → t               (* least upper bound *)
  val widen : t → t → t              (* widening for convergence *)
  val bottom : t                       (* least element *)
end

module type TransferFunctions = sig
  val exec_instr : Domain.t → instr → Domain.t
end

In exchange, Infer gives you: cross-language support, compositional interprocedural analysis, incremental re-analysis, and procedure summaries.

Concrete Checkers

Pulse — Memory Safety

The flagship checker. Uses Incorrectness Separation Logic to find null dereferences, use-after-free, data races. Replaced the original biabduction analysis. The domain tracks heap shapes and ownership.

Inferbo — Buffer Overrun

Uses an interval-based abstract domain. Abstract locations are tuples of (allocation site, size interval, offset interval). Classifies findings by confidence:

  • L1 (must-bug): array size [3,3], offset [5,5] → definitely out of bounds
  • L2 (may-bug): array size [3,3], offset [0,5] → possibly out of bounds

This is a textbook example of an interval domain applied to a real problem.

RacerD — Race Detection

Abstract domain tracks thread-safety properties: which locks are held, which accesses are protected. Uses ownership and lockset analysis. Deployed at scale across Meta's mobile codebases.

SPARTA — Reusable Domain Components

SPARTA (Semantics, PARTitioning and Abstraction) is a C++ header-only library providing reusable abstract domain building blocks:

  • Finite lattices, power set domains, abstract environments
  • Map-based domains, lifted domains (with top/bottom)
  • 60% speedup and 90% memory reduction vs conventional frameworks

This is the domain combinator library that AG frameworks lack.

Domain Combinators in Infer

(* AbstractDomain.ml — composing domains *)
module TopLifted(D : S)      (* Add top element to domain D *)
module FiniteSet(E : Element) (* Set-based domain over element type *)
module Map(K)(V : S)          (* Map-based domain: keys → abstract values *)
module Pair(A : S)(B : S)     (* Product of two domains *)
Key insight: Infer separates the domain interface (lattice + widening) from the transfer functions (how instructions transform states) from the fixpoint engine (iteration strategy). This three-way separation is what enables domain swapping — change the domain, keep the transfer functions, get a different cost/precision tradeoff.

The Circular Detection Gap

KSC detects circular access but throws an error instead of iterating. This is the key architectural difference from JastAdd:

// KSC generated evaluator — throws on cycle
kindDefs(): KindDefinition[] {
  if (this._cyc.has('kindDefs'))
    throw new Error(`Circular attribute access: 'kindDefs'`);
  this._cyc.add('kindDefs');
  try { ... }
  finally { this._cyc.delete('kindDefs'); }
}

// JastAdd approach — iterates to convergence
kindDefs(): KindDefinition[] {
  if (this._computing_kindDefs)
    return this._kindDefs;  // return current approximation
  this._computing_kindDefs = true;
  this._kindDefs = bottom;  // start at ⊥
  do {
    prev = this._kindDefs;
    this._kindDefs = equation(this);
  } while (!equal(prev, this._kindDefs));
  this._computing_kindDefs = false;
  return this._kindDefs;
}
Key insight: The code path from "detect cycle + throw" to "detect cycle + iterate" is architecturally small. The hard part isn't the iteration loop — it's the lattice contract: ensuring attributes declare their bottom values, join operations, and that equations are monotone.

08Four Options for KSC

Each option represents a different point on the cost/power spectrum. They are not mutually exclusive — each builds on the previous.

Option A: Circular Attributes (JastAdd-style)

What: Add lattice-valued circular attributes that iterate to convergence instead of throwing.

What Changes in KSC

  • New CircularAttr direction in the AttrDecl union
  • Lattice declaration: bottom, equal (or leq)
  • Generated evaluator: cycle detection becomes iteration loop
  • Monotonicity is a contract, not checked (same as JastAdd)
// Hypothetical CircularAttr declaration
{
  name: 'reachable',
  direction: 'circular',
  type: 'Set<string>',
  bottom: code('new Set()'),
  equal: code('setEqual'),
  equation: eq_reachable,         // monotone: only adds to set
}

What It Enables

Fixed-point dataflow analyses on the AST: reaching definitions, live variables, reachability, transitive closures over reference graphs.

Limitation

Only works on finite-height lattices. No widening means no unbounded iteration (e.g., can't compute interval abstractions of loop-bound variables).

Implementation effort: Small. The circular detection infrastructure already exists in generated evaluators — the change is from "throw" to "iterate." The main work is adding CircularAttr to the type system and adjusting codegen.

Option B: Domain-Polymorphic Attributes

What: Parameterize analysis over the domain type, not just values. Write equations generically over a domain interface. Select the domain at instantiation time.

// Domain interface
interface NumericDomain<T> {
  bottom: T;
  top: T;
  join: (a: T, b: T) => T;
  meet: (a: T, b: T) => T;
  widen?: (prev: T, next: T) => T;
  leq: (a: T, b: T) => boolean;
  // Abstract operations
  abstractAdd: (a: T, b: T) => T;
  abstractMul: (a: T, b: T) => T;
  fromConst: (n: number) => T;
}

// Same equation, different domains
const spec1 = makeSpec(SignDomain);        // cheap, imprecise
const spec2 = makeSpec(IntervalDomain);    // moderate
const spec3 = makeSpec(OctagonDomain);     // expensive, precise

What It Enables

Write one analysis spec, run it at different precision levels. "Apply at the weakest possible domain" becomes a runtime configuration choice.

Requires

Option A as a prerequisite (circular attributes for fixpoint iteration). Plus: a domain interface type, equations written generically over that interface, and domain implementations.

Key insight: This is the AG equivalent of Infer's separation between AbstractDomain and TransferFunctions. The equations become generic transfer functions; the domain is injected.

Option C: CEGAR-style Adaptive Refinement

What: Automatically select precision. Start with the cheapest domain, refine when results are insufficient.

CEGAR (Counterexample-Guided Abstraction Refinement):

  1. Analyze with a coarse domain (e.g., signs)
  2. If the analysis reports a potential violation — generate a counterexample
  3. Check if the counterexample is real (concretizable) or spurious (artifact of imprecision)
  4. If spurious: refine the domain (e.g., switch to intervals) and re-analyze
  5. Repeat until either a real bug is found or the analysis proves safe

AG-Specific Variant: Domain as Inherited Attribute

// The domain itself propagated down the tree
{
  name: 'analysisDomain',
  direction: 'inh',
  type: 'NumericDomain',
  rootValue: SignDomain,  // start cheap
  parentEquations: {
    // Refine for subtrees that need more precision
    FunctionDeclaration: eq_refineDomain
  }
}

Different subtrees get analyzed at different precisions. A hot function with complex arithmetic gets the interval domain while simple code stays on the sign domain.

Complexity: High. Requires Option A + B as prerequisites, plus: spurious counterexample detection, refinement strategy, subtree re-evaluation, and convergence guarantees for the refinement loop.

Option D: Higher-Order Domain Construction

What: The domain lattice itself is computed as an attribute. Subtree structure determines what domain is needed.

// Synthesized: what domain does this subtree need?
{
  name: 'requiredDomain',
  direction: 'syn',
  type: 'DomainSpec',
  default: code('SignDomain'),  // default: cheapest
  equations: {
    // Code with pointer arithmetic? Need intervals
    BinaryExpression: eq_needsIntervalIfPointerArith,
    // Code with coupled variables? Need octagons
    ForStatement: eq_needsOctagonIfCoupledBounds,
  }
}

// Inherited: propagate the chosen domain down
{
  name: 'activeDomain',
  direction: 'inh',
  type: 'DomainSpec',
  rootValue: code('SignDomain'),
  parentEquations: {
    FunctionDeclaration: eq_selectDomainFromChildRequirements
  }
}

This is syntax-directed domain selection: the tree structure tells you what precision is needed before the analysis even runs.

Domain Products via Attributes

Multiple attributes reading each other already approximates a reduced product. If intervalResult and signResult are two circular attributes, and each equation reads the other's current approximation, the fixpoint iteration naturally performs mutual refinement.

Key insight: This is the most "AG-native" approach. It leverages the tree structure to make domain selection a declarative attribute computation rather than an external optimization loop. The tree guides the domain, and the domain guides the analysis.

Concrete Example: Null-Pointer Analysis

Imagine writing a null-pointer analysis as AG equations, then running it at different domain precisions:

Boolean Domain — "might be null / definitely not null"

// Domain: { ⊥=unreachable, NotNull, MaybeNull, ⊤=both }
// Cost: O(1) per operation
// Precision: can't track which path leads to null

const x = getUser();  // → MaybeNull
if (x !== null) {
  x.name;              // → NotNull ✓
}
x.name;                // → MaybeNull ⚠ (may report false positive)

Interval Domain — tracks numeric guards

// Domain: intervals + nullability per variable
// Cost: O(n) per operation
// Precision: tracks range constraints from guards

const arr = getArray();   // → MaybeNull, length ∈ [0,∞)
if (arr && arr.length > 0) {
  arr[0];                  // → NotNull, index ∈ [0,0], length ∈ [1,∞) ✓
}
arr[0];                    // → MaybeNull ⚠

Polyhedra Domain — tracks inter-variable relationships

// Domain: linear constraints over variables + nullability
// Cost: exponential worst case
// Precision: can prove i < arr.length across loops

for (let i = 0; i < arr.length; i++) {
  // Polyhedra knows: 0 ≤ i < arr.length
  arr[i];                  // → safe ✓ (no false positive!)
}

Same equation functions, same AG specification, three different tradeoffs. The domain is the only variable.

What It Would Take

1 Circular attributes (Option A) ▼

Add CircularAttr to the AttrDecl union. Change codegen to emit iteration loops instead of throwing on cycles. Small change — the infrastructure is 90% there.

2 Domain interface + library ▼

Define a Domain<T> interface (bottom, join, leq, widen) in analysis/. Build domain implementations (sign, interval, set-of-values) in a new domains/ directory. This is KSC's equivalent of SPARTA.

3 Domain-parameterized specs ▼

Spec factories that take a domain as input: buildNullAnalysis(SignDomain) vs buildNullAnalysis(IntervalDomain). Equations written in terms of the domain interface. Same equations, different precision.

4 Galois connection verification (optional) ▼

Property-based tests that check: α(γ(a)) ≤ a and c ≤ γ(α(c)) for random inputs. Not a formal proof, but a strong empirical soundness check. Fits naturally into KSC's existing test infrastructure.

The opportunity: KSC already has the hardest parts — the two-functor architecture, the spec/machinery separation, parameterized attributes, and the AttrDecl type system. Adding circular attributes + a domain interface would make KSC the first AG system that speaks the language of abstract interpretation natively.

Q

What is a Galois connection and why does it matter for analysis soundness?

A

A pair of monotone functions α (abstraction) and γ (concretization) between concrete and abstract lattices, satisfying α(c) ≤ a ⇔ c ≤ γ(a). It guarantees soundness: abstract analysis results always over-approximate concrete behavior. False positives are possible, but false negatives are not.

Q

What's the difference between widening and narrowing?

A

Widening (∇) forces convergence on infinite-height lattices by over-approximating — it "jumps ahead" in the ascending chain. Narrowing (Δ) then iterates downward to recover lost precision while staying above the true fixpoint. Widening guarantees termination; narrowing improves the result.

Q

How do circular attributes in JastAdd relate to fixed-point iteration in abstract interpretation?

A

They are the same computation restricted to tree topology. Both start at a bottom value, iterate a monotone function, and converge on a lattice of finite height. The difference: AI iterates over CFG-structured equations with chaotic iteration; JastAdd iterates over tree-structured attributes with demand-driven evaluation.

Q

What does JastAdd's CRAG (Circular Reference Attributed Grammar) provide that standard AGs don't?

A

CRAGs combine reference attributes (graph overlay on the tree — e.g., name resolution links) with circular attributes (fixpoint iteration). This lets you build graph structures with references and then compute fixpoint analyses over those graphs — like transitive closure, dataflow, or reachability — all within the AG framework.

Q

Why can't standard AGs (without widening) handle infinite-height lattices?

A

Without widening, iteration produces an ascending chain in the lattice. On an infinite-height lattice (like integer intervals), this chain may never stabilize — each iteration produces a strictly larger value. Widening forces convergence by deliberately over-approximating, collapsing the chain to a fixpoint in finite steps. No widening = no termination guarantee on infinite domains.

Q

What is the reduced product of two abstract domains?

A

Running two domains in parallel with bidirectional information flow via reduction operators. Unlike the simple Cartesian product (where domains are isolated), in a reduced product each domain can sharpen the other. Example: an interval domain knowing x ∈ [0,10] can tell a sign domain that x is non-negative, eliminating the negative case.

Q

How does Infer separate domain from transfer functions from fixpoint engine, and why does this matter?

A

Infer.AI requires you to provide an AbstractDomain module (lattice ops + widening) and TransferFunctions module (per-instruction transformers) separately. The fixpoint engine is Infer's responsibility. This three-way separation enables domain swapping: change the domain, keep the transfer functions, and get a different cost/precision tradeoff without rewriting analysis logic.

Q

What would domain-polymorphic attributes give KSC that it doesn't have today?

A

The ability to write one analysis spec and run it at multiple precision levels by injecting different domains. Today, KSC equation functions work with concrete types. With domain-polymorphic attributes, equations would be generic over a Domain<T> interface, and you'd choose sign/interval/octagon at instantiation. This is "apply at the weakest possible domain" made practical.
