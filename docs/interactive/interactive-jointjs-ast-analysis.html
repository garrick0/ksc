Analysis What JointJS Does

The JointJS+ AST Visualizer is a demo application that parses JavaScript source code into an AST using Esprima, then renders each AST node as a colored rectangle in an SVG canvas managed by JointJS+. Here's what makes it notable:

Key Features Observed

  • Node-link tree diagram — each AST node is a 120×30 rounded rectangle with white text, connected by orthogonal L-shaped edges with arrow markers
  • Color-coded by node type — 7 distinct colors map to AST categories (declarations, identifiers, literals, operators, arguments, etc.)
  • Click-to-collapse — clicking a node hides all descendants and re-layouts the tree; clicking again restores them
  • Bidirectional source linking — hovering an AST node highlights the corresponding source range in the code editor; clicking a token in the token list highlights it in source
  • Zoom & pan — PaperScroller component provides viewport navigation with pinch-to-zoom
  • Export — built-in SVG and PNG export

Architecture & Data Flow

The demo has a clean 4-step pipeline:

// 1. Parse with Esprima const syntax = esprima.parseScript(code, { loc: true, range: true, tokens: true }); // 2. Build JointJS graph cells from AST const cells = graphUtils.constructTree(syntax, { children: getChildren, // node → child nodes[] makeElement: (node) => new Node({...}), makeLink: (parent, child) => new Link({...}), }); // 3. Layout with TreeLayout (direction: 'BR' = bottom-right) graph.resetCells(cells); treeLayout.layout(); // 4. Animate viewport to fit scroller.transitionToRect(graph.getBBox(), { visibility: .9 });

The graphUtils.constructTree() function is the key — it recursively walks the AST and creates a flat array of Element and Link JointJS cells. The TreeLayout class then computes positions using a Reingold-Tilford-style algorithm.

SVG Node Anatomy

Each AST node in the SVG is rendered as:

<!-- JointJS renders each node as a <g> group --> <g data-type="ast.Node" transform="translate(240,150)"> <rect width="120" height="30" rx="5" ry="5" fill="#ff5246" stroke="none" cursor="pointer"/> <text x="60" y="15" text-anchor="middle" fill="white" font-size="10">a</text> </g> <!-- Links use L-shaped orthogonal paths with arrow markers --> <g data-type="ast.Link"> <path d="M 60 30 L 60 65 L 80 65" stroke="#666" stroke-width="2" marker-end="url(#arrowhead)"/> </g>

Key measurements: 120×30px per node, 50px vertical gap between levels, 80px horizontal indent per depth level. All nodes are the same size (no variable sizing). The layout direction is "BR" (bottom-right), producing the distinctive staircase pattern.

Interactions

The collapse mechanism stores subtrees in a subtrees map keyed by cell ID:

// On click: toggle collapse if (subtrees[cell.id]) { // Expand: re-add stored cells graph.addCells(subtrees[cell.id]); delete subtrees[cell.id]; } else { // Collapse: remove all successors const successors = graph.getSuccessors(cell); subtrees[cell.id] = [].concat( graph.getSubgraph(successors), graph.getConnectedLinks(cell, { outbound: true }) ); successors.forEach(s => s.remove()); } treeLayout.layout(); // re-layout after change

After collapse/expand, treeLayout.layout() recomputes all positions, producing a smooth transition to the new layout.

Each AST node stores its range: [start, end] from Esprima. On hover, the range is applied as a CodeMirror decoration:

// element:mouseenter → highlight source range paper.on('element:mouseenter', (view) => { const range = view.model.get('node').range; highlightRange(range); }); // Uses CodeMirror StateEffect to add/remove decorations const highlightMark = StateEffect.define(); editorView.dispatch({ effects: highlightMark.of([ highlightDecoration.range(start, end) ]) });

This creates the green background highlight over the corresponding source code. The token list uses the same mechanism via data-range attributes.

JointJS+ provides a PaperScroller component that wraps the paper (SVG canvas):

const scroller = new ui.PaperScroller({ paper, baseWidth: 1, baseHeight: 1, contentOptions: { padding: 10 }, autoResizePaper: true }); // Blank area drag = pan paper.on('blank:pointerdown', (evt) => scroller.startPanning(evt)); // Pinch-to-zoom with clamping paper.on('paper:pinch', (evt, ox, oy, scale) => { scroller.zoom((scale - 1) * 1.2, { min: 0.2, max: 5, ox, oy }); });

The scroller provides: scroll-based panning, pinch-to-zoom, auto-resize of the SVG canvas, and animated transitions between zoom levels.

Layout Algorithm

JointJS+ uses layout.TreeLayout with direction: 'BR' (bottom-right). From the extracted node positions:

  • Horizontal offset per depth level: 80px (each child is 80px right of parent)
  • Vertical gap between siblings: 50px (node is 30px tall + 20px gap)
  • Node size: fixed 120×30px for all nodes
  • Edge routing: orthogonal L-shaped paths (down from parent center-bottom, then right to child left-center)
  • Y Combinator example: 81 nodes, canvas spans 1640 × 4030px, max depth ~19 levels

This produces a distinctive staircase diagonal layout that grows down-right. Unlike a traditional top-down tree where children spread horizontally, this layout is more like a cascading file explorer.

Cost JointJS Pricing & Licensing

JointJS+ is Commercial Software

  • $2,990 per developer (one-time perpetual license)
  • Update subscription: $1,490/year after first year
  • Enterprise plan: custom pricing (includes source code, SOC 2)
  • The open-source @joint/core exists but TreeLayout, PaperScroller, graphUtils.constructTree, and format.openAsSVG/PNG are all JointJS+ exclusive

The features that make the AST demo impressive (automatic layout, zoom/pan scroller, tree construction utility, SVG/PNG export) are all in the commercial package. Using just the open-source core would require building all of this from scratch anyway.

Options How to Build This Ourselves

Given that JointJS+ is $2,990 per developer and we only need a small subset of its features, here are four approaches ranked by practicality:

Recommended

▶

Option A: D3.js Tree — Implementation Detail

This is the clear winner. We already have D3 v7 loaded and a renderTree() function in compiler-dashboard.html that handles hierarchy, zoom, pan, collapse, and tooltips. The AST graph view would:

  1. Convert the ASTNode tree to a d3.hierarchy()
  2. Apply d3.tree().nodeSize([24, 160]) for compact layout
  3. Render nodes as <rect> + <text> groups (exactly like JointJS)
  4. Draw L-shaped orthogonal links with arrow markers
  5. Auto-collapse at depth 3, click to expand/collapse
  6. Add a fullscreen modal for the graph (too wide for 420px panel)
// Pseudocode for AST graph view function renderASTGraph(astNode, container) { const root = d3.hierarchy(astNode, d => d.children); root.eachBefore((d, i) => { d.id = i; }); // Auto-collapse deep nodes root.each(d => { if (d.depth > 2 && d.children) { d._children = d.children; d.children = null; } }); const tree = d3.tree().nodeSize([24, 160]); const svg = d3.select(container).append('svg'); const g = svg.append('g'); // Zoom + pan svg.call(d3.zoom().on('zoom', (e) => g.attr('transform', e.transform))); function update(source) { const treeData = tree(root); // ... render nodes as rect+text, links as L-paths // ... click handler toggles children/_children } update(root); }

The D3 d3.tree() layout uses the Reingold-Tilford algorithm (O(n)), which is the same class of algorithm JointJS+ uses internally. The visual output would be very similar to JointJS, just with a top-down layout instead of the diagonal staircase.

▶

Option B: Custom SVG Engine — Implementation Detail

Build a minimal tree layout from scratch. The algorithm is simple for trees:

// Simple tree layout: assign x,y to each node function layoutTree(node, depth = 0, state = { y: 0 }) { node.x = depth * 180; // horizontal: 180px per level node.y = state.y; state.y += 36; // vertical: 36px per node if (node.children && !node.collapsed) { node.children.forEach(child => layoutTree(child, depth + 1, state)); } } // SVG render: rect + text per node, path per edge function renderNode(node) { return `<g transform="translate(${node.x},${node.y})"> <rect width="140" height="28" rx="5" fill="${color(node)}"/> <text x="70" y="17" text-anchor="middle" fill="white" font-size="10">${node.kind}</text> </g>`; }

Pros: Zero dependencies, full control, smallest possible code. The demo above uses this approach.
Cons: Must implement zoom/pan/collapse yourself (~80 lines). No built-in smooth transitions.

▶

Option C: Cytoscape.js — Implementation Detail

Cytoscape.js is a Canvas-based graph visualization library. It provides zoom, pan, selection, and layout out of the box.

// Cytoscape setup const cy = cytoscape({ container: document.getElementById('graph'), elements: flattenAST(astNode), // [{data:{id,label,parent}}] layout: { name: 'dagre', rankDir: 'TB', nodeSep: 20 }, style: [{ selector: 'node', style: { 'label': 'data(label)', 'background-color': 'data(color)', 'shape': 'round-rectangle', 'width': 120, 'height': 30, } }] });

Pros: Rich built-in interactions, good for large graphs, easy hover/click handlers.
Cons: +380KB, Canvas rendering (text less crisp), no native tree collapse, requires flattening the recursive tree into Cytoscape's flat format.

▶

Option D: ELK.js — Implementation Detail

ELK (Eclipse Layout Kernel) provides the most sophisticated layout algorithms, including Sugiyama for DAGs and Mr. Tree for trees. Layout-only — you render with D3 or raw SVG.

// ELK layout is async const elk = new ELK(); const graph = { id: 'root', layoutOptions: { 'elk.algorithm': 'mrtree' }, children: flattenNodes(ast), edges: flattenEdges(ast), }; const layout = await elk.layout(graph); // layout.children now have x, y positions // Render with D3 or raw SVG...

Pros: Best layout quality, handles edge cases, used by VS Code.
Cons: +330KB (Java-to-JS transpilation), async API, layout-only (still need renderer), overkill for simple trees.

Superseded Original Recommendation (Pre-Implementation)

Note: This recommendation was written before implementation. We ultimately chose Option D (ELK+D3) and significantly enhanced it. See Updated Verdict below for the current assessment.

Go with Option A: D3.js Tree Layout

Why:

  • D3 v7 is already loaded in the dashboard (→ 0 KB added)
  • The existing renderTree() already has zoom, pan, collapse, animated transitions, and tooltips
  • D3's d3.tree() uses the same Reingold-Tilford algorithm class as JointJS+ TreeLayout
  • We get the same visual output (colored rect+text nodes, orthogonal or curved links) with SVG export for free
  • The collapse/expand pattern (children/_children) is simpler than JointJS's getSuccessors/remove/addCells approach

What we'd add:

  1. A "Graph" button alongside the existing Source/AST tabs in the file viewer
  2. A fullscreen modal for the graph (necessary — a node-link tree for 250+ nodes won't fit in a 420px panel)
  3. AST-specific color mapping (reuse existing astKindColor() function)
  4. Orthogonal L-shaped links to match the JointJS aesthetic
  5. Source-position highlighting (click AST graph node → scroll source view to that line)

Update Post-Implementation Review — Our ELK+D3 vs JointJS

Date: 2026-02-23 — We ultimately went with Option D (ELK.js + D3) rather than the originally recommended Option A (D3 tree). After building and iterating, we then added significant interaction improvements (animated transitions, inspect/collapse separation, smart collapse, layout direction toggle, outline mode). This section compares what we built against JointJS capabilities.

What We Built

Our implementation in compiler-dashboard.html (~620 lines of graph code) includes:

  • ELK.js layered/Sugiyama layout with orthogonal edge routing (BRANDES_KOEPF placement, LAYER_SWEEP crossing minimization)
  • D3.js SVG rendering with zoom/pan (d3.zoom, scale range 0.05–5)
  • Animated transitions (400ms, d3.easeCubicOut) — existing nodes interpolate position, new nodes fade in from parent, edges fade in
  • Click = inspect, double-click = collapse — single-click shows persistent detail panel; double-click toggles branch collapse
  • Persistent detail panel (top-right) showing Kind, Name, Pos, Children, Hidden descendants, Text preview
  • Smart collapse heuristics — Expression subtrees at depth ≥2, structural nodes at depth ≥4, large subtrees (>8 descendants) at depth ≥3
  • Outline mode — collapses all non-structural nodes, showing only SourceFile/Declaration/Statement/Block skeleton
  • Layout direction toggle (LR left-right / TD top-down) with active state highlighting
  • 6-color node classification by AST category with legend
  • Zoom/pan preservation across re-renders via saved d3.zoomTransform
  • Namespaced D3 events preventing handler overwrite bugs
  • Tooltip cleanup on re-render and across file changes

Gaps What JointJS Does That We Don’t

These are features present in JointJS (free or paid) that our current implementation lacks. Ordered by impact for our AST visualization use case.

High Impact

Gap 1: Source-Code Linking

JointJS+ AST demo highlights the corresponding source range in a CodeMirror editor when hovering a graph node. We store pos/end on every node but don’t link back to the Source tab.

JointJS+: bidirectional Us: data present, UI missing
High Impact

Gap 2: Search / Filter Nodes

No way to search for a specific AST node by kind or name. For a 39-node tree this is manageable, but for 200+ nodes it becomes essential. JointJS doesn’t provide this either — but their graph model API (graph.getElements().filter(...)) makes it trivial.

Neither has built-in search JointJS model API helps
Medium Impact

Gap 3: Minimap / Navigator

JointJS+ provides ui.Navigator — a small overview rectangle showing your viewport position within the full graph. We rely on Fit View to reorient. For large expanded trees, a minimap provides crucial spatial awareness.

JointJS+: built-in Us: not implemented
Medium Impact

Gap 4: SVG / PNG Export

JointJS+ has format.openAsSVG() and format.openAsPNG(). Our graph is already SVG in the DOM, so export is straightforward to implement — serialize the SVG element and open in a new tab or trigger download.

JointJS+: one-liner Us: ~20 lines to add
Low Impact

Gap 5: Virtual Rendering

JointJS 4.2 renders only visible cells (viewManagement: true) — 19,999 elements in 0.24s. We render all SVG nodes. For our current scale (39–250 nodes), this isn’t a bottleneck. Only matters at 500+ nodes.

JointJS: 19,999 nodes in 0.24s Us: fine up to ~300 nodes
Low Impact

Gap 6: Node Drag / Manual Layout

JointJS allows dragging nodes to adjust positions. For an AST visualization (read-only tree structure), this is rarely useful — the layout algorithm should determine positions. Could be interesting for custom annotation workflows.

JointJS: built-in Us: not needed for ASTs
Low Impact

Gap 7: Graph Model API

JointJS has graph.bfs(), graph.dfs(), graph.getSuccessors(), graph.toJSON(). We walk the raw AST tree directly. A model abstraction would help if we added features like “highlight all ancestors” or “focus subtree”.

JointJS: rich graph API Us: raw tree walk
Low Impact

Gap 8: Multi-Select

JointJS+ ui.Selection allows selecting multiple nodes (shift-click, rubber-band). We support single selection only. Multi-select would enable batch collapse, comparison, or “highlight these nodes in source”.

JointJS+: built-in Us: single select

Uplift Opportunities for Improvement

Concrete improvements we can make, inspired by JointJS patterns but achievable within our ELK+D3 stack. Ordered by impact-to-effort ratio.

▶

Uplift 1: Source-Code Linking (High Impact, Low Effort)

What: Clicking an AST graph node scrolls the Source tab to the corresponding line and highlights the range. Hovering a node briefly flashes the source range.

How: We already store pos and end on every node. The dashboard has a Source tab with line numbers. On click:

// In the click handler for graph nodes: const { line } = sf.getLineAndCharacterOfPosition(node.pos); // Close graph modal, switch to Source tab, scroll to line scrollSourceToLine(line + 1); highlightRange(node.pos, node.end);

Effort: ~30 lines. The Source tab already renders with line numbers; adding a temporary highlight class and scroll-into-view is straightforward.

JointJS parallel: Their AST demo uses CodeMirror StateEffect decorations for the same feature. Our approach would be simpler (CSS class + scrollIntoView).

▶

Uplift 2: Node Search (High Impact, Low Effort)

What: A search input in the toolbar that filters/highlights nodes matching a query (by kind name, identifier name, or text content).

How: Add <input> to the toolbar. On input, walk the AST, find matches, and either:

  • Option A (Highlight): Add a pulsing border to matching nodes, dim non-matching ones
  • Option B (Expand-to-reveal): Auto-expand collapsed ancestors of matching nodes, then highlight
// Walk AST and collect matching node IDs function searchNodes(node, query) { const matches = []; const q = query.toLowerCase(); (function walk(n) { if (n.kind.toLowerCase().includes(q) || (n.name && n.name.toLowerCase().includes(q))) matches.push(n._elkId); if (n.children) n.children.forEach(walk); })(node); return matches; }

Effort: ~40 lines including UI.

▶

Uplift 3: SVG Export (Medium Impact, Very Low Effort)

What: A toolbar button that exports the current graph view as an SVG file.

How: Serialize the existing SVG element, inject inline styles, and trigger a download.

function elkExportSVG() { const svg = document.querySelector('#ast-graph-container svg'); const clone = svg.cloneNode(true); // Inline critical styles for standalone viewing clone.setAttribute('xmlns', 'http://www.w3.org/2000/svg'); const blob = new Blob([clone.outerHTML], { type: 'image/svg+xml' }); const a = document.createElement('a'); a.href = URL.createObjectURL(blob); a.download = `ast-graph.svg`; a.click(); }

Effort: ~15 lines. JointJS+ charges $2,990 for this among other features.

▶

Uplift 4: Minimap (Medium Impact, Medium Effort)

What: A small (200×150px) overview panel in the bottom-right showing the full graph with a viewport rectangle you can drag.

How: Clone the SVG into a smaller container, scale it to fit, overlay a draggable rectangle representing the current viewport. Sync with d3.zoom transform.

Effort: ~80–100 lines. The tricky part is keeping the minimap viewport rectangle in sync with zoom/pan transforms bidirectionally.

When to add: Only worth it when graphs regularly exceed 100 visible nodes. For our current 39-node test cases, Fit View is sufficient.

▶

Uplift 5: Focus Mode / Breadcrumb Navigation (Medium Impact, Medium Effort)

What: Click a node + press a “Focus” button to make it the new root. A breadcrumb bar shows the path back up: SourceFile › FunctionDeclaration: add › Block. Click any breadcrumb segment to zoom back out.

How: Set a elkFocusRoot variable. In buildELKGraph(), start from the focus root instead of the AST root. Render breadcrumbs by walking up the parent chain.

Effort: ~60 lines. JointJS doesn’t have this either — it would be a differentiator.

▶

Uplift 6: Ancestor Highlight Path (Low Impact, Low Effort)

What: When a node is selected, highlight the path from the root to that node (thicker/brighter edges and ancestor node borders). Gives immediate context of “where am I in the tree?”

How: Walk up the _elkId path (e.g., n0-1-0-2 → n0-1-0 → n0-1 → n0) and add a CSS class to those nodes/edges.

Effort: ~20 lines. Uses the same ID-path structure already in place for findParentPosition().
