Weak Interpreters & Abstract Analysis

Cousot's abstract interpretation framework reframes KSC's property analysis as a lattice of abstractions — from concrete execution to the weakest type-level checks. What does each property actually need?
▶

1. The Big Idea: Every Analysis Is an Approximation

Theory

In 1977, Patrick and Radhia Cousot published a paper that unified all of static analysis under one framework: abstract interpretation. The core insight:

Every program analysis is a "weak interpreter" — it executes the program abstractly, in a simplified universe, trading precision for decidability.

A real interpreter computes exact values. An abstract interpreter computes approximate properties. The key question is: how much do you approximate?

The Spectrum of Interpretation Strength Concrete Execution Run the program. Exact answers. Undecidable in general. | | abstract (lose precision, gain decidability) v Strong Abstract Interpreter Track relationships between all variables. | Polyhedra, shape analysis. Expensive. | abstract further v Medium Abstract Interpreter Track per-variable ranges, some relations. | Intervals, octagons. Moderate cost. | abstract further v Weak Abstract Interpreter Track only types/signs/flags. Type checking, lint rules. Very cheap. All are sound: if the weak interpreter says "safe", it IS safe. But weaker interpreters may cry wolf (false alarms) more often.

Cousot showed that these aren't ad-hoc tools — they form a mathematical lattice. Each analysis is an abstraction of a more concrete one, connected by Galois connections that guarantee soundness is preserved. A type system is just a very weak abstract interpreter. A model checker is a stronger one. They're all points on the same spectrum.

Concrete Semantics

The set of all possible execution traces of a program. Infinitely precise, infinitely expensive. Not computable in general (halting problem).

Example: "Variable x takes values 3, 7, 12, 42 across all possible inputs."

Abstract Semantics

A computable overapproximation of the concrete semantics. Sound: includes all real behaviors, plus maybe some spurious ones (false alarms).

Example: "Variable x is in range [0, 100]" or "x has type number" or just "x is positive."

2. Galois Connections: The Bridge Between Worlds

Theory

The formal machinery that connects concrete and abstract worlds is the Galois connection — a pair of functions that translate between domains:

Galois Connection (L, α, γ, L#) Concrete Domain (L) Abstract Domain (L#) ┌─────────────────────┐ ┌─────────────────────┐ │ Sets of integers │ α │ Intervals │ │ {3, 7, 12, 42} │ ────────► │ [3, 42] │ │ │ │ │ │ {3, 7, 12, 42} │ <──────── │ [0, 100] │ │ ... plus {0,1,2,..} │ γ │ │ └─────────────────────┘ └─────────────────────┘ α(x) ≤# a ⇔ x ≤ γ(a) ← The Galois connection property α (abstraction): maps concrete sets to their best abstract approximation γ (concretization): maps abstract values to all concrete values they represent

Soundness guarantee: If you prove a property in the abstract world, it holds in the concrete world. The abstract semantics is a superset of the concrete semantics. So if the abstract interpretation says "no error", there really is no error. If it says "possible error", it might be a false alarm (overapproximation).

Why This Matters for Analysis Design

Every design choice in an analyzer is a choice of abstraction:

  • What do you track? (the abstract domain)
  • How do you transform it? (the abstract transformer)
  • When do you stop iterating? (widening for termination)

A "weak interpreter" is one that uses a coarse abstraction — it loses more information but is faster and simpler. A "strong interpreter" uses a fine abstraction — more precise but more expensive.

3. The Hierarchy: From Traces to Types

Theory

Cousot showed that all program semantics form a hierarchy of abstractions. Each level throws away some information from the level above. Click each level to see what it keeps and what it loses:

1 Maximal Trace Semantics CONCRETE

The full sequence of every state transition in every possible execution. Contains everything: values, timing, ordering, non-termination. Not computable. This is what your program "really does."

Loses: Nothing (this is ground truth).

↓ α
2 Collecting Semantics (Reachable States) STRONG

At each program point, the set of all possible states that can be reached from any initial configuration. Still tracks which values appear together (relational). The "strongest postcondition."

Loses: Order of states, intermediate steps, timing.

Keeps: All reachable value combinations, correlations between variables.

↓ α
3 Relational Abstractions (Polyhedra, Octagons) MEDIUM

Approximate the collecting semantics with constraints between variables. Polyhedra: arbitrary linear constraints (Ax ≤ b). Octagons: ±x ± y ≤ c. Track relationships but with limited expressiveness.

Loses: Exact reachable state sets; some relational precision (non-linear, disjunctive).

Keeps: Relationships between variables — "x ≤ y + 5", "i < array.length".

↓ α
4 Non-Relational Abstractions (Intervals, Signs) WEAK

Track each variable independently. Intervals: x ∈ [lo, hi]. Signs: x is positive/negative/zero. No information about how variables relate to each other.

Loses: All correlations between variables. "x = y" becomes "x ∈ [0,10] and y ∈ [0,10]" — no knowledge that they're equal.

Keeps: Per-variable bounds or categories.

↓ α
5 Type Abstractions WEAKEST

The coarsest useful abstraction. Each variable mapped to a type (number, string, boolean). Cousot proved (POPL 1997) that type systems are abstract interpretations of the untyped lambda calculus.

Loses: All value information. "x: number" could be 0, -1, or 10^100.

Keeps: Category membership. Enough for basic safety (no "string + object" errors).

The precision-cost tradeoff: Moving down the hierarchy makes analysis cheaper and more decidable, but also less precise (more false alarms). The art of static analysis is choosing the weakest abstraction that still answers your question.

4. The Critical Split: Non-Relational vs Relational Domains

Domains

Within abstract interpretation, the most important distinction for KSC is between non-relational and relational abstract domains. This maps directly to our intrinsic vs relational property split.

Non-Relational Domains

Analyze each entity independently. No information about relationships between entities.

Classic examples:

  • Sign domain: x ∈ {+, -, 0, ⊤}
  • Interval domain: x ∈ [lo, hi]
  • Constant propagation: x = 5 or x = ⊤

Fast. Simple. Per-variable.

Relational Domains

Track relationships between entities. Know how changing one affects others.

Classic examples:

  • Octagons: ±x ± y ≤ c
  • Polyhedra: Ax ≤ b (arbitrary linear constraints)
  • Dependency graphs: module A imports module B

Expensive. Complex. Multi-entity.

The Domain Choice Determines What You Can Ask Non-Relational Domain (per-file analysis) ┌────────────┐ ┌────────────┐ ┌────────────┐ │ file-a.ts │ │ file-b.ts │ │ file-c.ts │ │ noConsole? │ │ noConsole? │ │ noConsole? │ │ immutable? │ │ immutable? │ │ immutable? │ │ noMutation? │ │ noMutation? │ │ noMutation? │ └────────────┘ └────────────┘ └────────────┘ Each file analyzed independently. No cross-file info. Relational Domain (cross-file analysis) ┌────────────┐ ┌────────────┐ ┌────────────┐ │ file-a.ts │────►│ file-b.ts │────►│ file-c.ts │ │ │ │ │◄────│ │ ← cycle! └────────────┘ └────────────┘ └────────────┘ Files analyzed as a graph. Relationships visible.

This is the exact same split that Cousot identified for numeric analysis (intervals vs. polyhedra), applied to module analysis (per-file vs. dependency graph). The abstraction framework is the same. The domain just changes.

5. KSC Is Already an Abstract Interpreter

KSC

KSC's AG evaluator, viewed through Cousot's lens, is a weak abstract interpreter with a non-relational domain. Let's map the concepts precisely:

Abstract Interpretation Concept KSC Equivalent
Concrete Semantics All possible runtime behaviors of the TypeScript program
Abstract Domain Attribute values: KindDefinition | null, CheckerDiagnostic | null, boolean flags
Abstract Transformer AG equations: eq_noConsoleContext, eq_consoleViolation, etc.
Abstraction α buildKSTree(): TypeScript AST → KSC decorated tree (abstracts away runtime values, keeps structure + kinds)
Concretization γ "Any program matching this AST with these kinds" — implicit
Fixpoint Demand-driven evaluation with caching (no iteration needed because KSC's attribute dependencies are acyclic)
Soundness If KSC reports no violations, the property truly holds for the analyzed scope (no false negatives for implemented checks)
Incompleteness Overapproximation: e.g., flagging let in immutable scope even if never reassigned

KSC's Abstraction Level

KSC operates at approximately Level 4 in the hierarchy — a non-relational abstract domain. It analyzes each AST node's properties independently (with inherited context from ancestors, but no cross-file relations). This is analogous to using the interval domain for numeric analysis: fast, per-entity, no cross-entity relationships.

For intrinsic properties (noConsole, immutable, etc.), this abstraction level is exactly right. These properties are definitionally per-scope. A non-relational domain loses nothing relevant.

6. What Abstraction Level Does Each Property Need?

KSC

This is the key reframing. Instead of asking "can AG handle this?", Cousot's framework asks: "what abstract domain does this property require?"

Property Required Domain Strength Why
noConsole Non-relational (per-node) Weak Check: does node match console.*? Only needs local AST.
immutable Non-relational (per-node) Weak Check: is binding let/var? Syntactic, per-node.
noMutation Non-relational (per-node) Weak Check: is there an assignment operator? Syntactic, per-node.
static Non-relational (per-node) Weak Check: is there a dynamic import? Syntactic, per-node.
noSideEffects Non-relational (per-node) Weak Check: statement-level call expression? Syntactic, per-node.
noIO Weakly relational (per-file) Medium Needs import resolution to know if a call targets an IO module. Cross-scope within single file.
noImports Weakly relational (per-file) Medium Needs to see import declarations at file scope, not just within annotated subtree.

The Cousot-Informed Answer

KSC's intrinsic properties need only a non-relational domain (Level 4 — weak interpreter). This is what our AG evaluator already is. Relational properties need a relational domain (Level 3) or even fixpoint computation (Level 2). These require a fundamentally different abstract domain — not just "more attributes", but a different mathematical structure.

7. Designing Two Abstract Domains for KSC

Architecture

Cousot's framework tells us: if you need two different levels of abstraction, you need two abstract domains, connected by a Galois connection. Here's how KSC should be structured:

Abstract Domain 1: The AST Domain (Non-Relational)

This is KSC's current AG evaluator. Each file's AST is an independent abstract state. Attributes flow within the tree (syn up, inh down).

// The abstract "state" for each AST node
interface TreeAbstractState {
  kindAnnotations: KindDefinition[];       // what kinds apply here
  noConsoleContext: KindDefinition | null;  // inherited from above
  consoleViolation: Diagnostic | null;     // synthesized from node
  // ... 22 attributes total
}

// Abstract transformer: equation function
// Maps parent's abstract state to child's
function eq_noConsoleContext(parentState): KindDefinition | null {
  if (parentState.kindAnnotations.some(k => k.properties.noConsole))
    return matchingKind;
  return undefined; // inherit from above
}

What this domain can express:

  • Per-node properties (is this a console call? a let binding?)
  • Inherited context (is this inside a noConsole scope?)
  • Aggregated results (collect all violations in subtree)

What it cannot express: anything about other files. Each tree is a closed world.

Abstract Domain 2: The Module Graph Domain (Relational)

A separate abstract domain where the "states" are modules and the "transitions" are import edges. This domain captures cross-file relationships.

// The abstract "state" for the entire program's module structure
interface GraphAbstractState {
  modules: Map<string, ModuleInfo>;      // file path → module info
  edges: Map<string, Set<string>>;       // import graph
  sccs: string[][];                       // strongly connected components
  layerViolations: Diagnostic[];          // cross-layer imports
}

// Abstract transformer: graph algorithms
function detectCycles(state: GraphAbstractState): Diagnostic[] {
  const sccs = tarjanSCC(state.edges);  // fixpoint computation
  return sccs
    .filter(scc => scc.length > 1)       // cycles have >1 node
    .map(scc => ({ property: 'noCycles', modules: scc }));
}

What this domain can express:

  • Import relationships between modules
  • Transitive reachability (which modules can reach which)
  • Cycles in the dependency graph
  • Layer membership and cross-layer violations

The Galois Connection Between Domains

The two domains are connected by an abstraction function that extracts graph structure from the tree analysis results:

Domain Connection (analogous to Galois connection) Domain 1: AST Trees (non-relational) ┌─────────────────────────────────────────────┐ │ file-a.ts: { importEdges: ["b.ts","c.ts"], │ │ kindDefs: [...], │ │ violations: [...] } │ │ file-b.ts: { importEdges: ["c.ts"], │ │ kindDefs: [...], │ │ violations: [...] } │ │ file-c.ts: { importEdges: ["a.ts"], │ ← creates cycle │ kindDefs: [...], │ │ violations: [...] } │ └──────────────────┬──────────────────────────┘ │ α (extract import edges) v Domain 2: Module Graph (relational) ┌─────────────────────────────────────────────┐ │ a.ts ──► b.ts ──► c.ts ──► a.ts │ │ │ │ SCC: {a.ts, b.ts, c.ts} ← CYCLE! │ │ noCycles violation reported │ └─────────────────────────────────────────────┘ α: For each file, read its importEdges collection attribute, build the graph. This is the abstraction function. γ: The graph represents all programs with these import relationships.

The α function extracts import edges from per-file AG evaluation results. The graph domain then applies its own abstract transformers (SCC, reachability, layer checks). Finally, violations from both domains are merged.

This is exactly how Cousot says to compose analyses: use the output of one abstract interpretation as input to another. The soundness of the composition follows from the soundness of each part.

8. How This Compares to AG System Approaches

Compare

The previous interactive doc analyzed JastAdd and Silver's mechanisms. Through Cousot's lens, we can now understand why they work:

Mechanism AI Interpretation Abstraction Level
syn/inh attributes Non-relational abstract domain. Each node's state is independent (modulo inherited context from parent). Level 4 (Weak)
Reference attributes Introduce relational edges into the domain. The abstract state of one node can depend on the abstract state of a non-ancestor node. This moves from non-relational to weakly relational. Level 3 (Medium)
Collection attributes A collecting semantics over a subtree. Gathers contributions from distributed nodes into a single abstract value. Analogous to Cousot's collecting semantics for reachable states. Level 3 (Medium)
Circular attributes Fixed-point iteration — the core computation of abstract interpretation. Start from bottom, iterate monotonically until stable. This is literally Cousot's algorithm, embedded in an AG. Level 2 (Strong)
Nonterminal attributes Domain change: construct a new abstract domain (the NTA tree) from the current one. Like building a graph domain from a tree domain. Level 2-3
KSC post-AG graph A second abstract interpretation over a relational domain, using the output of the first (tree) interpretation as input. Level 2-3

The Key Insight

JastAdd's genius was embedding multiple abstraction levels within a single AG framework (references for Level 3, circulars for Level 2). Silver kept the AG at Level 4 and used higher-order attributes + forwarding to construct new domains when needed.

KSC's post-AG graph phase is closer to Silver's philosophy: keep the AG simple (Level 4), construct the relational domain separately. Both approaches are valid instances of Cousot's framework — they just place the domain boundary in different places.

9. Widening, Soundness, and False Alarms

Theory

Cousot's framework makes explicit what every static analysis tool deals with implicitly: the tradeoff between soundness, completeness, and cost.

Sound (No False Negatives)

If the analyzer says "safe", it is safe. Abstract semantics ⊇ concrete semantics. KSC is sound: if it reports no noConsole violations, there truly are no console calls in that scope.

Incomplete (False Positives)

The analyzer may report violations that aren't real. Overapproximation. Example: KSC flags let in an immutable scope even if the variable is never reassigned. This is the price of decidability.

Widening (Force Termination)

For fixpoint computations (circular attrs, cycle detection), you may need to widen — coarsen the abstraction to ensure the iteration terminates. KSC doesn't need this yet (no circular attrs), but a future transitive purity check would.

Cousot's fundamental theorem: "All non-trivial questions about the concrete semantics of a program are undecidable." Therefore every automatic analysis must either (a) be incomplete (miss some errors), (b) have false alarms (report non-errors), or (c) not terminate. Sound analyses choose (b) — they overapproximate.

KSC's Soundness Profile

Property Sound? Possible False Alarms
noConsole Yes None — console.log detection is exact (syntactic)
immutable Yes let x = 5; flagged even if x is never reassigned (overapprox: any let ⊆ mutable)
noMutation Yes Assignment in dead code flagged. Any = operator, even if benign.
noSideEffects Approx Pure function calls flagged as side effects (can't distinguish pure from impure calls without type info)
noCycles (future) Yes None — cycle detection on the actual import graph is exact
pure (future) Approx Would need transitive analysis. Without it, any function calling another function is "possibly impure" — massive overapproximation.

10. Industrial Precedent: Astree and SLAM

Prior Art

Two real-world tools demonstrate Cousot's framework at scale, and their architecture choices parallel KSC's situation:

Astree (Airbus A380 Flight Software)

Developed by Cousot's group at ENS. Analyzes 132,000 lines of C code for the A380's flight control system. Uses:

  • Intervals for per-variable bounds (Level 4, weak)
  • Octagons for pairwise variable relations (Level 3, medium)
  • Polyhedra for critical code sections (Level 2, strong)

Key insight: Astree uses different domain strengths for different code regions. Critical loops get polyhedra (expensive, precise). Most code gets intervals (cheap, sufficient). This is the "right tool for the job" principle.

Result: Zero false alarms on A380 flight software.

SLAM (Microsoft Driver Verifier)

Uses abstract interpretation with counterexample-guided abstraction refinement (CEGAR): start with a weak abstraction, check a property, and if the result is inconclusive, automatically refine to a stronger abstraction.

  • Start with boolean programs (Level 5, weakest)
  • If spurious counterexample found, refine (add predicates)
  • Iterate until proven or real bug found

Key insight: don't pay for precision you don't need. Start weak, strengthen only when forced to.

The KSC Parallel

Like Astree and SLAM, KSC should use different abstraction levels for different properties:

  • Weak (per-node): noConsole, immutable, noMutation, static — syntactic checks, cheapest possible
  • Medium (per-file with imports): noIO, noImports — need TS checker data but still single-file scoped
  • Strong (cross-file graph): noDependency, noCycles, layers — need the module graph, separate domain
  • Strongest (fixpoint): pure (transitive) — needs call graph + fixed-point iteration, future work

This isn't an architectural compromise — it's exactly what Cousot's theory recommends: use the weakest abstraction that answers your question.

11. Types as Abstract Interpretations (Cousot 1997)

Theory

One of Cousot's most striking results: type systems are abstract interpreters. Specifically, a type checker is a very weak abstract interpreter that uses types as its abstract domain.

The Type System as Weakest Abstract Interpreter Concrete: x can be 3, 7, -1, 42, ... | | α (abstract to interval) v Interval: x ∈ [-1, 42] | | α (abstract to sign) v Sign: x ∈ {-, 0, +} | | α (abstract to type) v Type: x : number ← weakest useful abstraction

Cousot derived the entire Hindley-Milner type system by successive abstractions of a denotational semantics. Each abstraction loses information but the composition remains sound.

Relevance to KSC

KSC's kind annotations (Kind<{ noConsole: true }>) are themselves an abstract domain — a refinement type system. They enrich TypeScript's type system with behavioral constraints. In Cousot's terms: KSC adds a slightly stronger abstract domain on top of TypeScript's type domain, one that tracks "effect properties" in addition to data types.

The power of this view: it tells us that adding noCycles requires not just a richer type, but a fundamentally different domain structure (graph vs. tree). You can't get there by refining the type system further — you need a domain change.

12. Formal Architecture: KSC as Composed Abstract Interpreters

Architecture

Putting it all together, here's KSC's analysis architecture expressed in Cousot's framework:

KSC: Three Composed Abstract Interpretations Concrete: TypeScript program runtime behavior | | α1: TS compiler (parsing, name resolution, type checking) v ┌──────────────────────────────────────────────────────────────┐ │ AI-1: TypeScript Type Checker (weak interpreter) │ │ Domain: types, symbol flags, resolved paths │ │ Output: typed AST with stamped fields │ └──────────────────────────────┬───────────────────────────────┘ | | α2: buildKSTree + AG evaluation v ┌──────────────────────────────────────────────────────────────┐ │ AI-2: KSC AG Evaluator (weak, non-relational) │ │ Domain: kind annotations, context attrs, violations │ │ Output: per-file violations + import edges (collection) │ │ Properties: noConsole, immutable, noMutation, static, │ │ noSideEffects, noIO, noImports │ └──────────────────────────────┬───────────────────────────────┘ | | α3: extract import edges, build graph v ┌──────────────────────────────────────────────────────────────┐ │ AI-3: Module Graph Analyzer (strong, relational) │ │ Domain: module dependency graph (nodes + edges) │ │ Transformers: SCC, reachability, layer checking │ │ Properties: noDependency, noCycles, layerEnforcement │ └──────────────────────────────┬───────────────────────────────┘ | | merge diagnostics v ┌──────────────────────────────────────────────────────────────┐ │ Final Output: all CheckerDiagnostic[] │ └──────────────────────────────────────────────────────────────┘

Each AI layer is sound. The composition of sound abstractions is sound. Therefore the complete pipeline is sound.

Why This Is Theoretically Principled

This isn't just "run two tools and merge results." Cousot's framework gives us:

  • Soundness composition: α3 ˆ α2 ˆ α1 is a valid Galois connection from concrete to graph domain
  • Precision guarantees: each layer uses the weakest abstraction sufficient for its properties
  • Clear extension path: need transitive purity? Add AI-4 with a call graph domain and fixpoint iteration
  • Independent refinement: each AI can be made more precise without affecting the others (finer abstract domain, same interface)

13. Test Your Understanding

Quiz
Q: What is a "weak interpreter" in Cousot's framework?
Click to reveal answer
An abstract interpreter with a coarse abstraction. It uses a very simplified domain (like types or signs instead of exact values), trading precision for speed and decidability. A type checker is the canonical example of a weak interpreter — it only tracks type membership, losing all value information.
Q: What is the difference between a non-relational and a relational abstract domain?
Click to reveal answer
Non-relational domains analyze each entity independently. Intervals: x ∈ [0,10], y ∈ [0,10] — no information about how x and y relate. Relational domains track relationships between entities. Octagons: x - y ≤ 3 — knows that x and y are correlated. For KSC: per-file analysis is non-relational; the import graph is relational.
Q: Why can't KSC's AG evaluator check noCycles?
Click to reveal answer
Because the AG evaluator uses a non-relational abstract domain. It analyzes each file's tree independently. Cycle detection requires a relational domain that tracks edges between modules — a fundamentally different abstract structure. In Cousot's terms: cycle detection needs Level 2-3 abstraction (relational + fixpoint), but the AG operates at Level 4 (non-relational).
Q: What does "soundness" mean for an abstract interpreter?
Click to reveal answer
No false negatives. If the abstract interpreter says a property holds (no violations), then the property truly holds for the concrete program. Formally: the abstract semantics is a superset of the concrete semantics (γ(lfp(F#)) ⊇ lfp(F)). The cost of soundness is possible false positives (reporting violations that aren't real).
Q: How does Cousot's framework justify KSC's post-AG graph phase?
Click to reveal answer
Composition of sound abstractions. The AG evaluator (AI-2) is a sound abstract interpretation over a non-relational domain. The graph analyzer (AI-3) is a sound abstract interpretation over a relational domain. Cousot proved that composing sound abstractions (via Galois connections) yields a sound composite. So the pipeline TS → AG → Graph is mathematically principled, not an ad-hoc hack.
Q: What does "use the weakest abstraction that answers your question" mean?
Click to reveal answer
Don't pay for precision you don't need. noConsole only needs to check if a node is a console call — this is a trivial syntactic check (Level 5, type-level). noCycles needs transitive closure of the import graph — this needs a relational domain with fixpoint computation (Level 2-3). Using the stronger domain for noConsole would be wasteful. Using the weaker domain for noCycles would be insufficient.
Q: How did Cousot prove that type systems are abstract interpretations?
Click to reveal answer
By deriving Hindley-Milner types through successive Galois connection-based abstractions of the untyped lambda calculus's denotational semantics. Starting from the collecting semantics (all possible values), abstracting to type sets, then to polytypes, then to monotypes — each step is a Galois connection. Type inference is the abstract fixpoint computation. This places types in the same lattice as intervals, polyhedra, etc.
Q: What is widening and when would KSC need it?
Click to reveal answer
Widening (∇) forces fixpoint iteration to terminate by deliberately coarsening the abstract value when it grows too large. KSC doesn't need it today (no circular attributes, no fixpoint iteration). It would be needed if KSC added transitive purity analysis (iterating over the call graph until all functions' purity status stabilizes) or if the import graph were analyzed with circular attributes instead of Tarjan's SCC algorithm.
Q: How does JastAdd's circular attribute mechanism relate to Cousot's fixpoints?
Click to reveal answer
They are the same thing. JastAdd's circular attributes: start from a bottom value, iterate the equation until stable, require lattice values and monotonic equations. This is exactly Cousot's fixpoint approximation: start from ⊥, apply the abstract transformer F# repeatedly, stop when F#(x) = x. The lattice requirement (finite height + monotonicity) is the same in both. JastAdd just embeds Cousot's algorithm inside an AG framework.

14. Summary: What Cousot Teaches Us About KSC

Summary

The Framework

  • Every program analysis is an abstract interpretation
  • Analyses are ordered by precision in a lattice
  • A "weak interpreter" = coarse abstraction (fast, less precise)
  • A "strong interpreter" = fine abstraction (slow, more precise)
  • Soundness composes: chain of sound abstractions is sound

The Reframing

  • KSC's AG = weak interpreter (non-relational domain)
  • Intrinsic properties = need only weak interpretation
  • Relational properties = need a stronger, relational domain
  • Not "can AG do this?" but "what domain does this need?"
  • Two-domain architecture is mathematically principled

The Bottom Line

Cousot's abstract interpretation framework provides the theoretical foundation for KSC's architecture. The split between per-file AG analysis (non-relational domain) and cross-file graph analysis (relational domain) isn't an engineering compromise — it's the correct application of the principle: use the weakest abstraction that answers your question.

JastAdd embeds both domains in one AG. Silver constructs the relational domain via higher-order attributes. KSC constructs it in a post-AG phase. All three are valid realizations of the same Cousot framework. The question is only where you draw the domain boundary — and for KSC, drawing it between phases (tree AG, then graph analysis) is the most pragmatic choice.
