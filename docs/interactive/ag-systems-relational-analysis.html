How JastAdd & Silver Handle Relational Properties

Deep analysis of the mechanisms real AG systems use for cross-tree analysis, dependency graphs, and cycle detection — and what KSC can learn from them.
▶

1. The Core Problem: Trees vs Graphs

Concept

Attribute grammars are defined over trees. But many useful program analyses require reasoning about graphs — import graphs, call graphs, dependency graphs, inheritance hierarchies. The question every AG system must answer:

How do you analyze graph structure when your evaluation model is tree-shaped?

Tree-Local (Intrinsic) Properties

Each node's property depends only on its subtree or inherited context. Examples: noConsole, immutable, noMutation.

Standard AG handles these perfectly.

Cross-Tree (Relational) Properties

Properties that depend on relationships between files/modules. Examples: noDependency, noCycles, layerEnforcement.

Requires graph construction — the hard part.

JastAdd and Silver took fundamentally different approaches to this problem. JastAdd extended the attribute types to support graph edges and fixed-point iteration. Silver extended the tree itself with forwarding and modular composition. Both solve it. Let's see how.

▶

2. JastAdd's Five Mechanisms

JastAdd

JastAdd extends classical AGs with five mechanisms that, when combined, can express graph-based analyses declaratively. Each adds a specific capability that plain syn/inh attributes lack.

Reference Attributes — Graph Edges Over Trees

What: Attributes whose values are pointers to other AST nodes, not data values. A variable-use node can have a decl() attribute that points directly to its declaration node.

// JastAdd: reference attribute for name resolution
syn Decl IdUse.decl() = lookup(getName());

// The lookup() is an inherited parameterized attribute
inh Decl IdUse.lookup(String name);
eq Block.getStmt(int i).lookup(String name) {
  // Search declarations in this block, then delegate to parent
  for (Decl d : localDecls())
    if (d.getName().equals(name)) return d;
  return lookup(name); // inherited from parent scope
}

Key insight: Once decl() is evaluated, it creates a graph edge from use-site to declaration. You can then access attributes on the target node: myVar.decl().type(). This is how JastAdd "escapes" the tree — reference attributes create a DAG/graph overlay.

AST (tree) + Reference Attrs (graph) Program Program ├── ClassA ├── ClassA │ ├── method foo() │ ├── method foo() │ │ └── x.bar() │ │ └── x.bar() ──ref──► ClassB.bar() │ └── field x: ClassB │ └── field x ──ref──► ClassB └── ClassB └── ClassB └── method bar() └── method bar() Tree structure stays intact. References add cross-tree edges.

For import/dependency analysis: An ImportDecl can have a resolvedModule() reference attribute pointing to the CompilationUnit node of the imported file. This creates the import graph edge within the AG framework.

Collection Attributes — Gather From Distant Nodes

What: Attributes whose values are built by contributions from arbitrary nodes throughout the tree. Instead of flowing up/down, information is gathered from wherever it exists.

// JastAdd: collect all errors across the entire program
coll ArrayList<String> Program.errors();

// Any Div node in the tree can contribute an error
Div contributes
    "Division by zero!"
    when getRight().isZero()
    to Program.errors();

// An import node contributes its resolved target to the dependency set
ImportDecl contributes
    resolvedModule()
    to CompilationUnit.dependencies()
    for enclosingCompilationUnit();

Evaluation: JastAdd uses a survey phase — it walks a subtree collecting all contributions before computing the final value. The scope is controlled by a root declaration:

// Scoped to nearest Module ancestor (not whole program)
coll Set<CompilationUnit> Module.dependencies()
    [new HashSet<>()] with add
    root Module;

For dependency analysis: This is the natural way to build a dependency graph. Each ImportDecl contributes its target to the containing module's dependency set. The graph emerges from distributed contributions.

Circular Attributes — Fixed-Point Iteration

What: Attributes that may depend on themselves (directly or transitively). JastAdd evaluates them by iterating from a bottom value until a fixed point is reached.

// JastAdd: circular attribute for liveness analysis
syn Set<Var> Stmt.liveVars() circular [new HashSet<>()];

// Equation: live-out = union of live-in of all successors
eq Stmt.liveVars() {
  Set<Var> result = new HashSet<>();
  for (Stmt succ : successors())    // follows reference attrs!
    result.addAll(succ.liveVars()); // circular: depends on self
  result.removeAll(defs());
  result.addAll(uses());
  return result;
}
Lattice requirement: For termination, values must form a lattice of finite height, and all equations must be monotonic. Common lattice: sets with union (bottom = empty set, top = all variables).
Circular Evaluation (Fixed-Point Iteration) Iteration 0: liveVars = {} (bottom) ← start from bottom Iteration 1: liveVars = {x} ← found direct use Iteration 2: liveVars = {x, y} ← propagated from successor Iteration 3: liveVars = {x, y} ← fixed point reached!

For cycle detection: Circular attributes can compute reachability sets. If module A's reachability set (following import edges) contains A itself, there's a cycle. The fixed-point iteration naturally discovers transitive closures.

// Transitive dependencies via circular + reference attrs
syn Set<Module> Module.transitiveDeps() circular [new HashSet<>()];
eq Module.transitiveDeps() {
  Set<Module> result = new HashSet<>(directDeps());
  for (Module dep : directDeps())
    result.addAll(dep.transitiveDeps()); // circular!
  return result;
}

// Cycle detection falls out naturally
syn boolean Module.hasCycle() = transitiveDeps().contains(this);

Nonterminal Attributes (NTAs) — Computed Subtrees

What: Attributes whose values are entire AST subtrees, computed on demand. Unlike parser-built nodes, NTAs are "virtual children" that can represent derived structure — like a dependency graph node.

// JastAdd: NTA that constructs a dependency graph as a virtual child
syn nta DepGraph Program.dependencyGraph() {
  DepGraph g = new DepGraph();
  for (CompilationUnit cu : getCompilationUnits())
    for (ImportDecl imp : cu.getImports())
      g.addEdge(cu, imp.resolvedModule());
  return g;
}

// The NTA is a full AST node — it can have its OWN attributes!
syn boolean DepGraph.hasCycles() = ...;
syn List<List<Module>> DepGraph.stronglyConnectedComponents() = ...;

Key insight: NTAs let you construct new data structures (graphs, tables, indices) as part of the AG evaluation. The constructed structure is itself an AST node that can have attributes evaluated on it. This is how you bridge from tree-shaped AG to graph algorithms.

For relational analysis: Build a dependency graph NTA, then define cycle-detection as a synthesized attribute on the graph node. The AG evaluator handles caching, demand-driven evaluation, and incrementality automatically.

Rewrites — Transform the Tree Itself

What: Conditional tree transformations applied transparently during evaluation. A node can be rewritten to a different node type based on computed attributes.

// JastAdd: rewrite a generic method call to a specific one
rewrite MethodCall {
  when (decl().isStatic())
  to StaticMethodCall {
    return new StaticMethodCall(getName(), getArgList());
  }
}

For relational analysis: Less directly applicable, but rewrites can normalize import structures or flatten complex module references before the dependency analysis phase runs. They ensure the tree is in a canonical form for graph construction.

3. Silver's Three Mechanisms

Silver

Silver takes a different philosophical approach. Where JastAdd extends the attribute types, Silver focuses on modular composition and tree transformation. Its key innovations are forwarding, modular collection attributes, and a well-definedness analysis that guarantees composition safety.

Forwarding — Virtual Tree Translation

What: A production can declare that it "forwards to" another tree. Any attribute not explicitly defined on the production is automatically evaluated on the forwarded-to tree instead. This enables language extensions to define new syntax that translates to existing constructs.

-- Silver: a "forEach" loop forwards to a while loop
abstract production forEach
top::Stmt ::= var::Name list::Expr body::Stmt
{
  forwards to
    seq(
      assign(iterVar, call("iterator", [list])),
      while(call("hasNext", [iterVar]),
        seq(assign(var, call("next", [iterVar])), body)
      )
    );
}

How it helps with relational analysis: Forwarding lets you create analysis-specific views of the program. A module-level production could forward to a flattened dependency representation, inheriting all the dependency-checking attributes from the target view.

Forwarding is essentially a lazy, attribute-preserving macro. The original tree is kept; the forwarded-to tree is computed on demand. Attributes defined on the original production override; everything else delegates to the translation.

Collection Attributes — Fold-Based Aggregation

Silver's collection attributes work differently from JastAdd's. They use a fold operator to combine a base value with contributions.

-- Silver: error collection with fold operator
synthesized attribute errors :: [String] with ++;

-- Base value (identity for ++)
abstract production add
e::Expr ::= l::Expr r::Expr
{
  e.errors := l.errors ++ r.errors;  -- base value
}

-- Aspect production contributes additional errors
aspect production add
e::Expr ::= l::Expr r::Expr
{
  e.errors <-                         -- contribute
    if isNumeric(l.type)
    then []
    else ["Left operand of + must be numeric"];
}

Key difference from JastAdd:

  • JastAdd: contributions come from any node in a surveyed subtree
  • Silver: contributions come from aspect productions on the same node

Silver's model is more modular (contributions are scoped to productions) but less flexible for gathering information from distant nodes. For dependency analysis, you'd typically combine collection attributes with forwarding.

Modular Well-Definedness Analysis

What: Silver statically verifies that independently-developed grammar modules can be safely composed without circular dependencies or missing attribute definitions. This is unique to Silver.

The guarantee: If modules A and B each pass the modular well-definedness check independently, then their composition A+B is guaranteed to be well-defined (no circular attribute dependencies, all inherited attributes supplied).

This is enforced through a modular non-circularity analysis that tracks which attributes may depend on which, conservatively, at the module boundary. Each extension is analyzed in isolation against the host language spec.

For relational analysis: This mechanism is about ensuring the AG itself doesn't have dependency cycles — it's meta-level. But the same principle (tracking dependencies between modules, detecting cycles) is exactly what we want to do at the program level for noCycles enforcement.

4. Solving Relational Problems: Step by Step

Compare

Let's trace how each system would implement the three relational analyses KSC needs: dependency tracking, cycle detection, and layer enforcement.

Building the Import Graph

JastAdd Approach

1 Reference attrs resolve imports
Each ImportDecl has a resolvedModule() reference attribute that points to the target CompilationUnit node. This uses inherited lookup() to search the program's compilation units.
2 Collection attrs gather edges
Each ImportDecl contributes its resolved target to the enclosing Module.dependencies() collection attribute. The survey mechanism automatically walks the subtree to collect all contributions.
3 NTA constructs the graph
Optionally, a nonterminal attribute on Program assembles all module dependency sets into a formal graph structure. This NTA can then have its own attributes (cycle detection, topological sort).

Silver Approach

1 Synthesized attrs compute deps per module
Each module production synthesizes its list of imported module names. This is a standard synthesized attribute — each production equation concatenates its children's imports.
2 Higher-order attr constructs graph
A higher-order attribute on the root Program nonterminal constructs a dependency graph tree. This virtual tree represents the module graph and can have attributes defined on it.
3 Collection attrs gather errors
Aspect productions on the dependency graph tree contribute errors to a collection attribute using the fold operator. Multiple independently-developed checking aspects can contribute.

Detecting Import Cycles

JastAdd Circular Attrs

The most elegant solution. Define a circular attribute transitiveDeps() on each module that follows reference edges to compute the transitive closure:

syn Set<Module> Module.transitiveDeps()
  circular [new HashSet<>()];

eq Module.transitiveDeps() {
  Set<Module> r = new HashSet<>(directDeps());
  for (Module d : directDeps())
    r.addAll(d.transitiveDeps());
  return r;
}

// Self-membership = cycle
syn boolean Module.inCycle()
  = transitiveDeps().contains(this);

The evaluator automatically iterates until the sets stabilize. No explicit worklist algorithm needed. The lattice is Set<Module> with union (finite height because there are finitely many modules).

Silver HOA + Algorithm

Silver would construct the graph as a higher-order attribute, then run a standard cycle detection algorithm as a synthesized attribute on that graph:

synthesized attribute depGraph :: DepGraph;
synthesized attribute cycles :: [[Module]];

abstract production program
p::Program ::= ms::[Module]
{
  p.depGraph := buildGraph(ms);
  p.cycles := findSCCs(p.depGraph);
}

Less declarative than JastAdd's circular approach — you need to implement the graph algorithm. But Silver's modular analysis ensures this won't create attribute circularities in the AG itself.

Enforcing Architectural Layers

Layer enforcement (e.g., "UI layer cannot import from data layer") requires mapping each file to a layer, then checking each import edge against allowed layer relationships.

Both Systems: Inherited + Reference + Check

1 Config defines layers and allowed dependencies
A configuration (external to the AG or as NTA/HOA) maps file patterns to layers and specifies which layers can depend on which. This is inherited context, passed down to each compilation unit.
2 Each import resolves its target's layer
Using the reference attribute for import resolution, look up the target module, determine its layer assignment, and compare against the current module's layer.
3 Violation if edge crosses forbidden boundary
A synthesized attribute on each ImportDecl checks allowedDeps(myLayer, targetLayer). Violations are contributed to the error collection.

Layer enforcement is actually the simplest relational property. It only needs direct edges (no transitive closure), so circular attributes aren't needed. The hard part is getting the reference attribute for import resolution right.

6. Why KSC Can't Just Copy JastAdd

Architecture

Even knowing exactly how JastAdd solves relational analysis, we can't simply add reference + circular + NTA attributes to KSC. Here's why:

Problem 1: Single-File AST Trees

KSC builds one AST per file via buildKSTree(). JastAdd builds a single AST for the entire program (a Program node with CompilationUnit children). Reference attributes work because all nodes live in one tree.

KSC would need a multi-file root node to enable cross-tree references.

Problem 2: Compiled Evaluator

KSC's evaluator is generated code with static dispatch. JastAdd uses a demand-driven runtime that resolves circular dependencies dynamically. Adding circular attributes would require changing the evaluator generator to emit iteration loops.

This is tractable but non-trivial.

Problem 3: No Node Identity

JastAdd's reference attributes rely on object identity — transitiveDeps().contains(this). KSC's KSCDNode wrappers don't have stable identity for cross-file references. We use string paths (resolvedModulePath) instead of node refs.

Problem 4: TS Checker Dependency

JastAdd computes its own name resolution via AG equations. KSC delegates to the TypeScript compiler for import resolution and type checking. This means our "reference attributes" are really just stamped fields from an external system, not AG-computed values.

This is actually an advantage in some ways — we get battle-tested resolution for free.

JastAdd: One Tree, Graph Overlaid KSC: Separate Trees, No Links Program file-a.ts tree file-b.ts tree ├── CompUnit (a.java) ├── ImportDecl ├── ExportDecl │ └── Import ──ref──► CompUnit(b) │ (path:"b.ts") │ └── func bar └── CompUnit (b.java) └── func foo └── ... └── class Bar └── bar() JastAdd: import resolves to a node KSC: import has a string path in the same tree. Graph edge. No node reference. No graph.

8. Recommended Evolution Path

KSC

Based on the analysis, here's a phased approach that starts pragmatic and evolves toward JastAdd-level capability as needed:

Phase 1: Option D — Collection + Post-AG Graph (Now)

Add importEdges collection attribute. Build dependency graph in program.ts after AG evaluation. Implement noCycles (Tarjan's SCC) and noDependency (edge check) as imperative post-phases. Minimal changes to evaluator.

Phase 2: Option B — Multi-File Root (When needed)

When we need cross-file inherited context (e.g., "what kind annotations are visible from the importing file?"), add a Program root node. This enables inherited attributes to flow across file boundaries. Prerequisite for transitive call-graph analysis.

Phase 3: Option C — Circular Attributes (Research/v2)

If KSC grows to need dataflow analysis, points-to analysis, or transitive purity checking, implement circular attribute evaluation. This is the JastAdd endgame — fully declarative fixed-point computation. Requires Phase 2 as prerequisite.

Key Insight from This Analysis

JastAdd didn't build all five mechanisms at once. It started with reference attributes (1999), added circular attributes (2003), then collection attributes (2007), then circular collections (2009). Each was added when real compiler use cases demanded it.

KSC should follow the same evolutionary pattern: add capabilities as concrete use cases justify the complexity.

9. Case Study: ExtendJ (JastAdd Java Compiler)

JastAdd

ExtendJ (formerly JastAddJ) is a full Java compiler built entirely with JastAdd. It handles multi-file compilation, cross-package name resolution, and type checking for Java 4–11. Here's how it uses the mechanisms we've discussed:

Cross-File Name Resolution

ExtendJ has a Program root node containing all CompilationUnit children. Each CompilationUnit has inherited attributes like lookupType(String packageName, String typeName) that search all compilation units for matching declarations.

Import declarations resolve via:

// Simplified from ExtendJ source
syn TypeDecl ImportDecl.importedType() =
  lookupType(getPackageName(), getTypeName());

// lookupType is inherited from Program, searches all CUs
inh TypeDecl CompilationUnit.lookupType(String pkg, String name);
eq Program.getCompilationUnit(int i).lookupType(String pkg, String name) {
  for (CompilationUnit cu : getCompilationUnits())
    if (cu.packageName().equals(pkg))
      for (TypeDecl td : cu.getTypeDecls())
        if (td.name().equals(name)) return td;
  return unknownType();
}

What This Means for KSC

ExtendJ demonstrates that a single-tree, all-files-as-children architecture works at scale (Java has thousands of files in real projects). The lookupType() pattern — an inherited parameterized attribute from the root — is the canonical way to do cross-file resolution in an AG.

However, KSC already gets import resolution from the TS compiler. We don't need to reimplement it. We need the graph structure that results from it, which is a much simpler problem.

ExtendJ's Analysis Extensions

Beyond basic compilation, ExtendJ has been extended with:

  • CAT (Call Graph Analysis Tool) — builds call graphs using Class Hierarchy Analysis, implemented with collection + reference attrs
  • Intraprocedural dataflow — control flow, live variables, null pointer analysis using circular + reference attrs
  • Devirtualization — points-to analysis for optimizing virtual calls, using circular collection attrs
  • Metrics — code complexity, coupling metrics using collection attrs across the program tree

Each extension was added as a separate aspect module, demonstrating the composability of JastAdd's approach.

10. Advanced: RelRAG (Bidirectional Relations)

JastAdd

Relational Reference Attribute Grammars (RelRAG) are a 2020 extension to JastAdd that directly models bidirectional noncontainment relations — exactly the kind of structure needed for dependency graphs.

The Problem RelRAG Solves

In standard RAGs, if module A imports module B, you can navigate A→B via the reference attribute. But navigating B→"who imports me?" requires either a collection attribute (expensive survey) or maintaining two opposing references manually (error-prone redundancy).

RelRAG adds first-class bidirectional relations that the AG runtime maintains consistently:

// RelRAG: bidirectional import relation
rel CompilationUnit.imports* <-> CompilationUnit.importedBy*;

// Setting one direction automatically updates the other
cuA.imports().add(cuB);
// Now cuB.importedBy() contains cuA automatically

// Both directions are navigable as attributes
syn Set<CompilationUnit> CompilationUnit.allImporters() = importedBy();
Relevance to KSC: RelRAG shows where the AG field is heading — toward native graph structure support. We don't need this yet, but it validates that the "dependency graph as relations" concept is a recognized research direction, not a KSC-specific hack.

11. Test Your Understanding

Quiz
Q: What JastAdd mechanism creates graph edges over a tree structure?
Click to reveal answer
Reference attributes. They are attributes whose values are pointers to other AST nodes, creating a graph overlay on the tree. Example: IdUse.decl() points to the declaration node.
Q: How does JastAdd detect import cycles declaratively?
Click to reveal answer
Circular attributes computing transitive closure. Define transitiveDeps() as circular (starts from empty set, iterates until fixed point). A module is in a cycle if transitiveDeps().contains(this). The evaluator automatically handles the iteration.
Q: What is the key difference between JastAdd and Silver collection attributes?
Click to reveal answer
Contribution scope. JastAdd collections use a survey phase to gather contributions from any node in a subtree. Silver collections use a fold operator where contributions come from aspect productions on the same node. JastAdd's is more flexible for distant gathering; Silver's is more modular and composable.
Q: What are nonterminal attributes (NTAs) and why do they matter for graph analysis?
Click to reveal answer
Computed subtrees as virtual children. NTAs are AST nodes computed by equations rather than built by the parser. They let you construct derived structures (like a dependency graph) inside the AG. The NTA itself can have attributes (e.g., hasCycles()), getting demand-driven evaluation and caching for free.
Q: Why can't KSC simply add JastAdd-style reference attributes?
Click to reveal answer
Three reasons: (1) KSC builds separate trees per file, so there's no single tree to host cross-file references. (2) KSC delegates import resolution to the TS compiler, so references would need to bridge two systems. (3) KSC's compiled evaluator uses static dispatch, not the dynamic demand-driven runtime that JastAdd uses for reference traversal.
Q: What is Silver's "forwarding" and what problem does it solve?
Click to reveal answer
Lazy, attribute-preserving tree translation. A production can declare it "forwards to" another tree. Attributes not defined on the original production are evaluated on the forwarded-to tree instead. This enables language extensions to define new syntax that transparently reduces to existing constructs, inheriting all their analysis attributes.
Q: What lattice property must circular attribute values satisfy for termination?
Click to reveal answer
Finite height and monotonicity. Values must form a lattice of finite height (can only grow so far), and all equations must be monotonic (values only increase). Common example: sets with union — bottom is empty set, values grow monotonically, height is bounded by the number of elements in the universe.
Q: What is the recommended first step for adding relational properties to KSC?
Click to reveal answer
Option D: Collection-based edge gathering + post-AG graph phase. Add an importEdges collection attribute to gather import paths per file. After AG evaluation, build a dependency graph in program.ts and run Tarjan's SCC for cycles / edge checks for noDependency. Minimal architecture changes, proven approach (similar to Silver's strategy).
Q: How does ExtendJ handle cross-file name resolution?
Click to reveal answer
Inherited parameterized attribute from the Program root. A lookupType(pkg, name) attribute is inherited from the Program node (which contains all CompilationUnits as children). The equation searches all compilation units for matching type declarations. This makes the AST itself the symbol table.
Q: What does RelRAG add to standard reference attribute grammars?
Click to reveal answer
First-class bidirectional noncontainment relations. Standard RAGs give you A→B navigation via reference attributes, but B→A requires manual collection attributes. RelRAG adds rel A.r* <-> B.s* declarations that maintain both directions automatically, with consistent incremental updates.

12. Summary & Key Takeaways

Summary

JastAdd's Strategy

Extend the attribute system itself. Reference attributes create graph edges. Circular attributes compute fixed points. Collection attributes gather distributed data. NTAs construct derived structure. Everything stays inside the AG framework — declarative, cached, demand-driven.

Best for: systems that want maximum declarativeness and are willing to invest in runtime infrastructure.

Silver's Strategy

Transform and compose trees. Forwarding creates analysis views. Higher-order attributes construct derived trees. Collection attributes aggregate modularly. Modular well-definedness ensures safe composition. Graph algorithms run as synthesized attributes on derived trees.

Best for: systems that prioritize modular composition and language extension.

KSC's Best Path

KSC has a unique advantage: the TS compiler already resolves imports. We don't need to reimplement name resolution with reference attributes like ExtendJ does. We just need to:

  1. Collect the import edges that TS already resolved (collection attrs)
  2. Build a graph from those edges (post-AG phase, like Silver)
  3. Analyze the graph with standard algorithms (SCC, layer checks)

This hybrid approach — AG for per-file analysis, imperative graph algorithms for cross-file relations — is pragmatic and well-supported by the literature. Both JastAdd and Silver ultimately reduce to running graph algorithms on collected data; they just embed those algorithms inside the AG framework. We can run them outside it and get the same results.

Bottom Line

JastAdd shows the theoretical ideal: everything declarative inside the AG. Silver shows the pragmatic middle: construct derived structures, then analyze. KSC should follow Silver's philosophy for v1 (post-AG graph phase), with the option to evolve toward JastAdd's mechanisms (multi-file root, circular attrs) if and when use cases demand it.
