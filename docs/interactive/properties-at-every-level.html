Properties at Every Syntactic Level

KindScript currently assigns properties to function declarations. But what if we could assign properties to every syntactic category — expressions, statements, patterns, types, modules? What would that look like, and what would it buy us?

Part 21 — KindScript Research Series

1. The Vision: A Property for Every Level

TypeScript assigns types to expressions and symbols to declarations. KindScript could assign behavioral properties at every level of the syntax tree, creating a parallel property environment that flows alongside the type environment.

// Today: KindScript assigns properties to declarations
// @ksc pure, noIO
function add(a: number, b: number): number {
  return a + b;
}

// The vision: properties at EVERY level
// Module:      [pure, treeshakeable, noSideEffectsOnImport]
// Declaration: [pure, noIO, immutable, total]
// Statement:   [effect-free, terminates]
// Expression:  [pure, deterministic, constant]
// Pattern:     [irrefutable, exhaustive]
// Type:        [serializable, sendable, comparable]

2. Expression-Level Properties

This is the most fundamental level — the atomic building blocks from which all higher-level properties are derived. Every expression in the program gets a property set.

2.1 The Properties

⟦⟧ pure — No observable side effects

A pure expression computes a value without reading or writing external state, performing I/O, or modifying anything beyond its return value.

Math.sin(x)          // [pure] — computes, returns, nothing else
a + b                  // [pure] — arithmetic is pure
[...arr].sort()       // [pure] — spread creates new array, sort is on the copy
arr.sort()             // [impure] — mutates arr in place!
console.log(x)        // [impure] — I/O side effect
fetch("/api")          // [impure] — network I/O
Date.now()             // [impure] — reads global clock state

Bugs caught: Unintended mutations in map/filter callbacks, side effects in constructors, I/O in pure computation paths.

Optimizations: Common subexpression elimination, memoization, parallel evaluation, dead code elimination (if result unused and expression is pure, the entire expression can be removed).

⟦⟧ deterministic — Same inputs always produce same output

Deterministic expressions produce identical results when called with identical arguments. Note: deterministic ≠ pure. A function that logs AND returns a value is impure but deterministic (in its return value).

Math.sqrt(4)           // [deterministic] — always 2
"hello".toUpperCase()  // [deterministic] — always "HELLO"
JSON.parse(s)          // [deterministic] — same string → same object
Math.random()           // [non-deterministic] — different each call
Date.now()              // [non-deterministic] — depends on when called
await fetch("/api")    // [non-deterministic] — server may return different data

Bugs caught: Using Math.random() where determinism is required (tests, reproducible builds), cache key mismatches.

Optimizations: Safe memoization (only deterministic functions can be memoized), test reproducibility guarantees.

⟦⟧ total — Never throws or fails

A total expression always returns a value — it never throws an exception, never returns undefined unexpectedly, never panics. The opposite is partial: it may fail for some inputs.

a + b                       // [total] — arithmetic never throws
arr.length                  // [total] — always a number
JSON.parse(userInput)       // [partial] — throws on invalid JSON!
obj.deeply.nested.prop      // [partial] — throws if any level is null
arr[getIndex()]             // [partial] — may be out of bounds
new Map().get("key")!     // [partial] — non-null assertion may lie

Bugs caught: Uncaught exceptions in production, null pointer errors, array out-of-bounds, JSON parse failures.

Optimizations: Remove try/catch overhead when all expressions in the block are total.

⟦⟧ constant — Value known at compile time

A constant expression can be fully evaluated during compilation. Rust calls this const fn — expressions that are "promotable" to compile-time evaluation.

42                         // [constant] — literal
2 + 3 * 4                   // [constant] — arithmetic on constants
"hello" + " world"         // [constant] — string concat on constants
config.MAX_RETRIES         // [non-constant] — runtime value
arr.length                  // [non-constant] — depends on runtime state

Optimizations: Constant folding, dead branch elimination (if (false) → removed), precomputed lookup tables, zero-cost configuration.

⟦⟧ referentially transparent — Can be replaced by its value

The strongest property: the expression can be substituted with its result everywhere it appears, without changing program behavior. This implies pure + deterministic + total.

// Referentially transparent: these are interchangeable
const x = add(2, 3);
use(x, x);
// ≡ (equivalent to)
use(add(2, 3), add(2, 3));

// NOT referentially transparent:
const y = readLine();
use(y, y);
// ≢ (NOT equivalent to)
use(readLine(), readLine()); // reads TWO lines!

Why this matters: Enables equational reasoning about programs. You can understand code by substitution, just like algebra. This is the holy grail of program reasoning.

2.2 Expression Property Composition

When expressions combine, their properties compose via lattice intersection — the result has only the properties that ALL sub-expressions share:

// Property composition rules:
pure    + pure    = pure           // both pure → result pure
pure    + impure  = impure         // one impure → result impure
total   + total   = total          // both total → result total
total   + partial = partial        // one partial → result partial
const   + const   = const          // both const → result const
const   + runtime = runtime        // one runtime → result runtime
determ. + determ. = determ.        // both deterministic → result det.

// In practice:
Math.sin(x) + Math.cos(x)  // [pure, deterministic, total, !constant]
Math.sin(x) + Date.now()   // [impure, non-deterministic, total, !constant]
2 + 3                        // [pure, deterministic, total, constant] = ref. transparent
The lattice: Each property forms a two-element lattice with ⊤ (has property) and ⊥ (doesn't). Composition is meet (∧). This means impurity is infectious — one impure sub-expression contaminates the whole expression. Same for partiality, non-determinism, and non-constancy.

3. Statement-Level Properties

Statements don't produce values — they transform state. Their properties describe how they transform state and what guarantees hold about the transformation.

3.1 The Properties

→ effect-free — This block doesn't modify external state 
// Effect-free: no observable state changes
if (x > 5) {
  const y = x * 2;    // local binding only
  return y + 1;        // return is control flow, not mutation
}

// NOT effect-free:
if (x > 5) {
  globalCount++;        // modifies external state
  console.log(x);       // I/O side effect
  arr.push(x);          // mutates arr (from outer scope)
}

What it catches: Accidental mutations in branches that should be read-only, I/O in computation-only paths. Enables reasoning about what a branch does vs what it computes.

→ terminates — This loop is guaranteed to finish 
// Terminates: i decreases toward 0
for (let i = n; i > 0; i--) { // [terminates] — i strictly decreases
  process(i);
}

// Terminates: array is finite, forEach visits each once
items.forEach(item => process(item)); // [terminates]

// MAY NOT TERMINATE:
while (true) {                          // [may-diverge]
  if (shouldStop()) break;             // depends on external state
}

// Dafny-style annotation for KindScript:
// @ksc terminates(decreases: arr.length - i)
for (let i = 0; i < arr.length; i++) { ... }

What it catches: Infinite loops, accidental server-blocking loops, unbounded recursion in rendering loops.

How Dafny does it: Require a "decreases expression" — a value that gets strictly smaller each iteration and is bounded below. If the analysis can verify the decrease, termination is proven. Common patterns (for-i, forEach, map, reduce) can be auto-proven.

→ exhaustive — All cases handled 
// Exhaustive: all cases of the discriminated union handled
type Shape = { kind: "circle" } | { kind: "square" } | { kind: "triangle" };

switch (shape.kind) {                // [exhaustive]
  case "circle": return ...;
  case "square": return ...;
  case "triangle": return ...;
}

// NOT exhaustive:
switch (shape.kind) {                // [non-exhaustive]
  case "circle": return ...;
  case "square": return ...;
  // Missing "triangle"! Falls through silently.
}

What it catches: Missing cases after adding a new variant to a union type — the classic "I added a new enum value but forgot to update 15 switch statements" bug.

Note: TypeScript already checks this with never in the default branch, but only if you opt in. KindScript could make it a first-class property.

→ idempotent — Executing twice = executing once 
// Idempotent: running again has no additional effect
user.active = true;              // [idempotent] — already true? no change
element.classList.add("open"); // [idempotent] — add is set-like
cache.set(key, value);           // [idempotent] — same key+value → no change

// NOT idempotent:
counter++;                        // [non-idempotent] — each run adds 1
arr.push(item);                  // [non-idempotent] — each run adds element
sendEmail(to, body);             // [non-idempotent] — sends AGAIN

What it catches: Double-execution bugs in retry logic, React useEffect running twice in StrictMode, duplicate API calls, double-counting in analytics.

→ commutative — Order doesn't matter 
// Commutative: these can be reordered safely
const x = computeA();   // independent
const y = computeB();   // independent — can run in any order (or parallel!)

// NOT commutative:
const x = readLine();   // order matters!
const y = readLine();   // this reads the SECOND line, not the first

What it enables: Automatic parallelization — if two statements are commutative, they can run on separate threads. Also enables statement reordering for cache optimization.

4. Declaration-Level Properties

This is where KindScript already operates — annotating function declarations with properties. But the space is much richer than just @ksc pure.

4.1 The Properties

≡ immutable — Binding never changes after initialization 
const x = 5;                    // [immutable] — const keyword
const obj = { a: 1 };            // [NOT deeply immutable] — obj.a can change!
const obj = { a: 1 } as const;  // [deeply immutable] — readonly + literal types

// KindScript could distinguish:
// @ksc immutable         — shallow (binding doesn't change)
// @ksc deeplyImmutable   — deep (nothing reachable changes)
// @ksc frozen            — Object.freeze verified at compile time

What it catches: Accidental reassignment, property mutation of "constant" objects, shared mutable state between modules.

Rust parallel: Rust's default is immutable (let x = 5). You must opt into mutability (let mut x = 5). KindScript could enforce this convention.

≡ pure — Function has no side effects (the current KindScript property) 
// @ksc pure
function add(a: number, b: number): number { // [pure]
  return a + b;
}

// With expression-level properties, the checker verifies:
// 1. Every expression in the body is [pure]
// 2. No expression reads/writes external state
// 3. No expression calls impure functions
// If all pass → declaration gets [pure]

The key insight: declaration purity is derived from expression purity. A function is pure iff every expression in its body is pure. This is where the multi-level system pays off — you don't need separate logic for declaration-level purity, you just aggregate expression-level purity.

≡ local — Binding doesn't escape its scope 
function process() {
  const temp = [];              // [local] — never returned, never assigned out
  temp.push(1, 2, 3);
  return temp.reduce((a,b) => a+b); // returns a number, not temp itself
}

let leaked;
function leaky() {
  const data = createData();    // [escapes] — assigned to outer scope below
  leaked = data;
}

What it enables: If a mutable binding is local, mutations are invisible outside the function — meaning the function can still be considered "externally pure" even with internal mutation. This is exactly how Haskell's ST monad works.

≡ bounded-recursion — Recursive function has bounded call depth 
// Bounded: depth limited by input size
function factorial(n: number): number { // [bounded(n)]
  if (n <= 1) return 1;
  return n * factorial(n - 1);  // n strictly decreases each call
}

// Unbounded: no clear termination
function collatz(n: number): number { // [unbounded?]
  if (n === 1) return 0;
  if (n % 2 === 0) return 1 + collatz(n / 2);
  return 1 + collatz(3 * n + 1); // increases before decreasing!
}

What it catches: Stack overflows from unbounded recursion, accidental infinite recursion in tree traversals.

MetricRatingNotes
Bugs caught
Mutation bugs, stack overflow, scope leaks, impure functions
Optimizations
Inlining, dead code elimination, escape analysis
Ease of implementation
KindScript already does this! Extending is incremental
False positive risk
Low — conservative analysis is safe for declarations
Developer value
Highest immediate value — this is KindScript's core

5. Pattern-Level Properties

Patterns deconstruct values. Their properties describe coverage (do you handle all cases?) and reliability (does the pattern always match?).

5.1 The Properties

⟨⟩ exhaustive — All possible values handled 
type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

// Exhaustive handling:
function handle<T>(r: Result<T>) {    // [exhaustive]
  if (r.ok) return r.value;
  else throw new Error(r.error);
}

// Non-exhaustive — what if someone adds { ok: "pending" }?
function handle(r: Result<T>) {       // [non-exhaustive]
  if (r.ok) return r.value;
  // implicit undefined return for error case!
}

Why this is high-value: Exhaustiveness checking is one of ML/Rust's greatest features. When you add a new variant to a union, the compiler tells you every place that needs updating. TypeScript's never trick approximates this, but KindScript could make it first-class.

⟨⟩ irrefutable — Pattern always matches 
// Irrefutable: always succeeds
const { x, y } = point;          // [irrefutable] if point: { x: T, y: T }
const [first, ...rest] = arr;    // [irrefutable] (rest may be empty)

// Refutable: may fail
const { data } = await response.json(); // [refutable] if response is opaque
const [a, b] = getItems();       // [refutable] array may have < 2 items

What it catches: Runtime destructuring failures, undefined values from incomplete patterns.

⟨⟩ non-overlapping — No redundant cases 
// Overlapping: first case catches everything, rest is dead code
switch (action.type) {
  default: return state;            // catches everything!
  case "increment": return ...;    // [unreachable] — dead code
  case "decrement": return ...;    // [unreachable] — dead code
}

What it catches: Dead code in switch/case, accidentally broad patterns hiding specific cases, order-dependent bugs in conditional chains.

5.2 Practical Value Assessment

MetricRatingNotes
Bugs caught
Missing cases, dead code, destructuring failures
Optimizations
Jump table generation, dead branch removal
Ease of implementation
Well-studied algorithms; TS already does some of this
False positive risk
Nearly zero — exhaustiveness algorithms are complete
Developer value
High for union-heavy codebases (Redux, state machines)

6. Type-Level Properties

This is the most speculative level for KindScript — assigning properties to types themselves, not just the values they classify. Rust does this with marker traits (Send, Sync, Copy). Haskell does it with type classes (Eq, Ord, Show).

6.1 The Properties

: serializable — Values of this type can be JSON.stringify'd 
// Serializable types:
number                    // [serializable]
string                    // [serializable]
{ x: number, y: string }  // [serializable] — all fields serializable
number[]                  // [serializable] — array of serializable

// NOT serializable:
(x: number) => number    // [non-serializable] — functions can't be serialized
Map<string, number>      // [non-serializable] — Map serializes to {}
Set<number>              // [non-serializable] — Set serializes to {}
{ handler: () => void }   // [non-serializable] — contains function

// KindScript could verify API boundaries:
// @ksc serializable
type ApiResponse = {
  data: UserData,          // checker verifies UserData is serializable
  meta: Metadata,          // checker verifies Metadata is serializable
};

Huge practical value: Any type that crosses a serialization boundary (API response, localStorage, postMessage, worker communication) MUST be serializable. Catching a function in an API response type at compile time prevents silent data loss at runtime.

: sendable — Safe to transfer across worker/thread boundaries 
// Sendable (safe for postMessage / worker transfer):
number, string, boolean      // primitives are always sendable
ArrayBuffer                    // transferable (moves, doesn't copy)
{ x: number, y: number }      // plain objects with sendable fields

// NOT sendable:
HTMLElement                     // DOM nodes can't cross worker boundary
() => void                     // closures capture scope, not transferable
WeakRef<T>                    // weak references are thread-local

// Rust parallel: Send trait
// In Rust: T: Send means T can be transferred to another thread
// KindScript: @ksc sendable on a type means it can cross worker boundaries

What it catches: Passing DOM nodes to workers (crashes), closures in postMessage (silent serialization to null), shared mutable state across workers.

: comparable — Supports meaningful equality 
// Comparable: === gives meaningful results
number    // [comparable] — value equality
string    // [comparable] — value equality
enum Color { Red, Green }  // [comparable] — enum equality

// NOT meaningfully comparable:
{ x: 1 } === { x: 1 }     // false! reference equality, not structural
[1, 2] === [1, 2]         // false! same problem
NaN === NaN               // false! NaN is not equal to itself

// KindScript could warn when using === on non-comparable types:
if (objA === objB) { ... } // ⚠️ reference equality on objects — intended?

What it catches: Object equality bugs (the #1 source of subtle bugs in React), NaN comparison traps, comparing objects by reference when structural comparison is intended.

: sized — Has a known maximum memory footprint 
// Sized: known memory usage
number                   // [sized: 8 bytes]
{ x: number, y: number } // [sized: ~32 bytes + overhead]
[number, number]         // [sized: fixed tuple]

// Unsized / unbounded:
number[]                 // [unsized] — array can grow
string                   // [unsized] — strings can be any length
Map<string, T>          // [unsized] — map can grow

What it enables: Memory budget enforcement, stack allocation guarantees (sized values can go on the stack), embedded/IoT constraints.

6.2 Auto-Derivation

Like Rust's auto-traits, type properties could be automatically derived from structure:

// Composition rules (like Rust's auto-trait derivation):
// A type is [serializable] if ALL its fields are [serializable]
// A type is [sendable] if ALL its fields are [sendable]
// A type is [comparable] if ALL its fields are [comparable]
// A type is [sized] if ALL its fields are [sized]

type Point = { x: number, y: number };
// Auto-derived: [serializable, sendable, comparable, sized]
// Because number has all four properties

type Handler = { name: string, fn: () => void };
// Auto-derived: [NOT serializable, NOT sendable]
// Because () => void is not serializable or sendable

7. Module-Level Properties

Module properties describe the behavior of entire files. These are the highest-level properties and the ones most directly useful for architectural enforcement and build optimization.

7.1 The Properties

{ } pure — No side effects on import 
// Pure module: importing it does nothing except make bindings available
// @ksc pure
export function add(a: number, b: number) { return a + b; }
export function mul(a: number, b: number) { return a * b; }
export const PI = 3.14159;
// No top-level code that "does stuff"

// IMPURE module: importing has side effects
registerPlugin("my-plugin");           // runs on import!
polyfill(globalThis);                   // modifies global!
export const db = createConnection();  // I/O on import!

Massive practical value: Pure modules are safe to tree-shake (bundlers can remove unused exports), safe to lazy-load (importing at any time is safe), safe to re-order (import order doesn't matter). This is exactly what webpack's "sideEffects": false flag means, but verified at compile time instead of declared manually.

{ } treeshakeable — Exports are independently importable 
// Treeshakeable: each export is independent
// @ksc treeshakeable
export function sort(arr) { ... }    // independent
export function filter(arr) { ... }  // independent
export function map(arr) { ... }     // independent

// NOT treeshakeable: exports have hidden coupling
const shared = createSharedState();     // shared state!
export function increment() { shared.count++; }
export function getCount() { return shared.count; }
// Can't tree-shake increment without breaking getCount!

What it catches: Hidden coupling between exports through shared module-level state, which defeats tree-shaking and creates subtle ordering dependencies.

{ } deterministic-init — Module initialization is deterministic 
// Deterministic: same result every time the module loads
export const CONFIG = {
  maxRetries: 3,
  timeout: 5000,
};

// Non-deterministic: result depends on when/how module loads
export const startTime = Date.now();     // different each load!
export const id = crypto.randomUUID();  // different each load!
export const env = process.env.NODE_ENV; // depends on environment

What it enables: Safe module caching, reproducible builds, predictable hot module replacement.

{ } self-contained — No imports from outside the project 
// Self-contained: only uses language builtins
// @ksc selfContained
export function deepClone<T>(obj: T): T {
  return JSON.parse(JSON.stringify(obj));
}

// NOT self-contained:
import _ from 'lodash';              // external dependency!
export const clone = _.cloneDeep;

What it enables: Dependency auditing, security boundary enforcement, identifying which modules can be extracted into standalone packages.

7.2 Practical Value Assessment

MetricRatingNotes
Bugs caught
Side-effect-on-import, hidden coupling, non-deterministic init
Optimizations
Tree-shaking, lazy loading, parallel module loading, bundle size
Ease of implementation
Top-level statement analysis is tractable
False positive risk
Side effects in module init are common and sometimes intentional
Developer value
Architectural enforcement, build optimization, dependency auditing

8. Cross-Level Property Flow

The real power of multi-level properties is how they flow between levels. Properties propagate in two directions:

8.1 Bottom-Up Inference

Expression properties bubble up to determine higher-level properties automatically:

// Bottom-up: expression → declaration → module

// Expression level: each sub-expression gets properties
export function distance(a: Point, b: Point): number {
  const dx = a.x - b.x;            // expr: [pure, det, total, !const]
  const dy = a.y - b.y;            // expr: [pure, det, total, !const]
  return Math.sqrt(dx*dx + dy*dy); // expr: [pure, det, total, !const]
}
// Declaration level (inferred): [pure, deterministic, total]
//   ↑ because ALL expressions in body have these properties

export const ORIGIN: Point = { x: 0, y: 0 };
// Declaration level: [immutable, constant]

// Module level (inferred): [pure, treeshakeable, deterministic-init]
//   ↑ because ALL declarations are pure and all top-level code is constant
Expression Props
→ aggregate →
Declaration Props
→ aggregate →
Module Props
Properties flow upward through aggregation: if all sub-components have a property, the container has it too

8.2 Top-Down Annotation

Annotations at higher levels propagate requirements downward:

// Top-down: module annotation → declaration requirement → expression check

// @ksc pure                          ← MODULE-LEVEL annotation
//   ↓ means: every export must be pure

export function add(a: number, b: number) {
  // ↓ means: every expression must be pure
  return a + b;                    // ✓ pure expression
}

export function logAndAdd(a: number, b: number) {
  console.log(a, b);                // ✗ IMPURE expression!
  // Error: module is annotated @ksc pure but logAndAdd
  // contains impure expression console.log() at line N
  return a + b;
}
Module Annotation
→ requires →
Declaration Check
→ requires →
Expression Check
Annotations flow downward through requirements: a module-level annotation means every sub-component must satisfy it

8.3 The Dual-Flow System

In practice, both flows operate simultaneously:

// The full picture: annotation + inference working together

// @ksc pure                         ← top-down: sets requirement
export function transform(data: Data[]): Data[] {
  return data
    .filter(d => d.active)          // ← bottom-up: [pure] ✓
    .map(d => ({                     // ← bottom-up: [pure] ✓
      ...d,
      score: computeScore(d)        // ← bottom-up: is computeScore [pure]?
    }))                               //   → look up computeScore's declaration
    .sort((a, b) => b.score - a.score); // ← ⚠️ sort creates new array via spread?
}                                     //   actually .sort() mutates! but we're on
                                      //   a chain from .filter() which creates a
                                      //   new array, so the mutation is local.

// The checker needs to determine:
// 1. [top-down] module says pure → transform must be pure
// 2. [bottom-up] .filter() creates new array → local mutation OK
// 3. [cross-reference] computeScore must be [pure] → check its declaration
// 4. [bottom-up] all expressions check out → transform is [pure] ✓

The key insight: Expression properties are the foundation. Everything above is derived. Declaration properties aggregate expression properties. Module properties aggregate declaration properties. Type properties are orthogonal — they describe the data rather than the computation. Pattern properties are structural — they describe coverage of case analysis.

Test Your Understanding

1. If a function contains 10 pure expressions and 1 impure expression, what is the function's purity?

Properties compose via lattice meet (∧). Pure ∧ impure = impure. One impure expression makes the entire function impure. (The dead branch case is interesting — if the impure expression is provably unreachable, you could argue the function is effectively pure, but conservatively it's marked impure.)

2. Which property level would catch the bug: "I passed a function closure to a Web Worker via postMessage and it silently became null"?

This is a type-level property. The sendable property, auto-derived structurally, would flag any type containing functions as non-sendable. When you call postMessage(value), KindScript would check that the value's type is [sendable] and report an error if it contains function types, DOM references, or other non-transferable values.

3. What is the relationship between module-level "pure" and webpack's "sideEffects: false"?

They express the same semantic property: "importing this module has no observable side effects." The difference is enforcement. webpack's sideEffects: false is a manual assertion in package.json that developers can get wrong (marking an impure module as side-effect-free breaks tree-shaking). KindScript's module-level pure would actually verify this by checking all top-level statements for side effects.

4. In the bottom-up inference model, how does KindScript determine if a module is "pure"?

Module purity flows bottom-up: expression-level purity → statement-level effect-freedom → module-level purity. The checker walks all top-level code (not inside functions, just the module body) and verifies that every statement is effect-free, which means every expression within those statements must be pure. Function declarations are fine (they don't execute on import). Calls to impure functions at the top level would fail the check.

5. Why is "serializable" on types potentially more impactful than "pure" on expressions?

Serialization bugs are insidious because they fail silently. JSON.stringify on a Map returns "{}". A function in an object becomes undefined. Dates become strings. These bugs cross system boundaries (API responses, localStorage, worker messages) where they're hardest to trace. A type-level serializable check catches them at compile time, at the exact boundary where the type crosses a serialization point.
