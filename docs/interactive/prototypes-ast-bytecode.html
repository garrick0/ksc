Prototypes, AST Comparison & V8 Bytecode

How [[Prototype]] chains work, how V8's AST differs from TypeScript's, how V8 walks the tree to emit bytecode, and what register machines and accumulators actually are.

1. The [[Prototype]] Chain

Every JavaScript object has a hidden internal slot called [[Prototype]]. It's a pointer to another object (or null). When you access a property that doesn't exist on the object itself, the engine follows this pointer to look on the next object in the chain. It keeps following pointers until it either finds the property or reaches null.

1.1 [[Prototype]] vs .prototype

These are two completely different things, and the naming is confusing:

[[Prototype]] (internal slot)

Every object has this. It's the actual link in the chain. You can't access it directly — it's an engine-internal slot, not a JavaScript property.

const obj = { x: 1 };

// obj.[[Prototype]] points to Object.prototype
// You can READ it via:
Object.getPrototypeOf(obj)
// → Object.prototype

// Or the deprecated accessor:
obj.__proto__
// → same thing (don't use this)

.prototype (regular property)

Only functions have this. It's a normal property that holds the object that will become the [[Prototype]] of instances created with new.

function Dog(name) {
  this.name = name;
}
Dog.prototype.bark = function() {
  return "Woof!";
};

const d = new Dog("Rex");
// d.[[Prototype]] === Dog.prototype
// d.bark() → follows [[Prototype]] → "Woof!"

1.2 Property Lookup: Walking the Chain

Click each object in the chain to see what happens when you access dog.toString():

dog

→ [[Prototype]] →
Dog.prototype
{ bark: fn }
→ [[Prototype]] →
Object.prototype
{ toString, hasOwnProperty, ... }
→ [[Prototype]] →
null
end of chain
Step 1: Check dog itself. Does dog have an own property called toString? No — dog only has { name: "Rex" }. Follow dog.[[Prototype]] to the next object.
Step 2: Check Dog.prototype. Does it have toString? No — it only has { bark: fn }. Follow Dog.prototype.[[Prototype]] to the next object.
Step 3: Check Object.prototype. Does it have toString? Yes! Found it. Return Object.prototype.toString. This is where built-in methods like toString, hasOwnProperty, and valueOf live. Every object inherits them because every chain eventually reaches here.
End: null. If the engine reaches null, the property doesn't exist anywhere in the chain. Return undefined. null is the sentinel that terminates every prototype chain. Object.prototype.[[Prototype]] is always null.
Classes are sugar for this. When you write class Dog extends Animal, JavaScript sets Dog.prototype.[[Prototype]] = Animal.prototype. That's it. No special class machinery. The chain does all the work.

2. V8's AST vs TypeScript's AST

They're both ASTs of the same language, but designed for different purposes. V8's AST is optimized for fast bytecode generation and is discarded immediately after. TypeScript's AST is designed for rich analysis, type checking, and tooling — it preserves far more information and lives longer.

2.1 Side-by-Side Comparison

For the code: function add(a: number, b: number): number { return a + b; }

V8 AST

// V8's internal AST (C++ structs)
FunctionLiteral {
  name: "add"
  params: [Variable("a"), Variable("b")]
  body: ReturnStatement {
    expression: BinaryOperation {
      op: Token::ADD
      left: VariableProxy("a")
      right: VariableProxy("b")
    }
  }
  // NO type annotations
  // NO decorators
  // NO generics
  // Minimal source positions
}

Stripped down. Only what's needed to emit bytecode. Type annotations are syntax errors in V8 (it parses JS, not TS).

TypeScript AST

// TypeScript's AST (ts.Node objects)
FunctionDeclaration {
  name: Identifier("add")
  parameters: [
    Parameter {
      name: Identifier("a")
      type: TypeReference("number")
    },
    Parameter {
      name: Identifier("b")
      type: TypeReference("number")
    }
  ]
  returnType: TypeReference("number")
  body: Block {
    statements: [ReturnStatement {
      expression: BinaryExpression {
        operator: PlusToken
        left: Identifier("a")
        right: Identifier("b")
      }
    }]
  }
  modifiers: []
  typeParameters: undefined
  // Full source positions for every node
  // Trivia (comments, whitespace)
}

Rich. Preserves types, modifiers, generics, comments, exact positions. Designed for analysis, not execution.

Neither uses ESTree. ESTree is a community standard used by ESLint, Babel, Acorn, etc. V8 has its own internal C++ AST. TypeScript has its own ts.SyntaxKind-based AST. KindScript uses TypeScript's AST via ts.createProgram() — it gets the same rich nodes that the TS checker uses.

3. How V8 Walks the AST

V8's BytecodeGenerator is an AST visitor (like TypeScript's forEachChild) that does a single-pass depth-first traversal. As it visits each node, it emits bytecode instructions directly. The AST goes in, bytecode comes out, and the AST is discarded.

3.1 The Walk, Step by Step

For: function add(a, b) { return a + b; }

1 Visit FunctionLiteral
The BytecodeGenerator sees a function. It allocates a register file (slots for local variables and temporaries) and starts a new bytecode sequence. Parameters a and b are assigned to argument registers a0 and a1.
2 Visit ReturnStatement
The generator sees a return. It needs the return value, so it recurses into the expression child. After the expression is generated, it will emit a Return instruction.
3 Visit BinaryOperation(ADD)
The generator sees a + b. It needs both operands. It recurses: first visit the left child (a), then the right child (b), then emit the add.
4 Visit VariableProxy("a") — emit Ldar a0
The generator resolves a to argument register a0 and emits Ldar a0 (Load Register into Accumulator). The accumulator now holds the value of a.
5 Back in BinaryOperation — emit Add a1, [0]
The left operand is in the accumulator. The right operand (b) is in register a1. Emit Add a1, [0] which adds a1 to the accumulator. [0] is the feedback slot index for type profiling. Result stays in the accumulator.
6 Back in ReturnStatement — emit Return
The return value is in the accumulator (result of the add). Emit Return, which returns whatever is in the accumulator. Done. The bytecode for this function is complete.

Single-pass matters. V8 can't afford multiple passes because parsing + bytecode generation happens on the main thread before any code can run. Every millisecond of parsing is startup latency. That's why V8 uses lazy parsing (only fully parse functions when called) and single-pass bytecode generation. TypeScript can afford multiple passes because it runs as a build step or background process.

4. V8 Bytecode Deep Dive

4.1 Is V8 Bytecode Universal?

No. V8 bytecode is proprietary to V8. It's not standardized, not versioned for compatibility, and changes between V8 releases. SpiderMonkey has its own bytecode. JavaScriptCore has its own. Python has its own. Java has its own (.class files). None of them are interchangeable.

BytecodeEngine / VMPortable?Standardized?
V8 BytecodeV8 (Chrome, Node)Across OS/CPU within same V8 versionNo — changes between releases
Java BytecodeJVM (any impl)Yes — "write once, run anywhere"Yes — JVM spec
CIL / .NET IL.NET CLRYes — across .NET implementationsYes — ECMA-335
Python BytecodeCPythonSame CPython version onlyNo — CPython internal
WebAssemblyAny browser / runtimeYes — designed for portabilityYes — W3C standard

V8 bytecode IS portable across operating systems and CPU architectures within the same V8 version, because it runs in V8's virtual machine, not on bare hardware. But unlike Java bytecode or WebAssembly, it's not designed to be distributed — it's an internal representation that V8 generates fresh from source each time.

4.2 Register Machine vs Stack Machine

Stack Machine (JVM, Python, .NET)

Operands live on a stack. Instructions pop inputs and push results.

// Computing: a + b (stack machine)

iload 0     // push a onto stack          stack: [a]
iload 1     // push b onto stack          stack: [a, b]
iadd        // pop two, push sum          stack: [a+b]
ireturn     // pop and return             stack: []

After iload 0; iload 1:

b   ← top
a

After iadd: stack = [a+b]. The ADD instruction implicitly knows its operands are the top two stack values.

Advantage: Instructions are small (no operand encoding needed — everything is implicit "top of stack"). Bytecode is compact.

Disadvantage: Lots of push/pop overhead. Values get shuffled on and off the stack constantly. More memory traffic.

Register Machine (Lua, Dalvik)

Operands live in numbered registers. Instructions name their operands explicitly.

// Computing: a + b (pure register machine)

ADD  r2, r0, r1     // r2 = r0 + r1  (a + b)
RET  r2              // return r2

Register state:

r0 = a
r1 = b
r2 = a+b

The ADD instruction explicitly names all three operands. No stack, no push/pop, no implicit state.

Advantage: Fewer instructions needed. Values stay in registers (fast). Less memory traffic.

Disadvantage: Each instruction is larger (must encode register names). More bytes per instruction.

Accumulator Machine (V8's Choice)

A hybrid: one implicit register (the accumulator) + explicit register operands. Most instructions implicitly read/write the accumulator.

// Computing: a + b (V8's accumulator style)

Ldar  a0       // Load a into accumulator      acc = a
Add   a1, [0]  // acc = acc + a1               acc = a+b
Return         // Return accumulator           return a+b

After Ldar a0:

acc = a
a0 = a
a1 = b

Add a1, [0] means "add a1 to the accumulator." Only one operand is explicit. The other (acc) is implicit. This saves ~25-50% bytecode size compared to pure register encoding.

Why V8 chose this: Bytecode size matters for memory and cache. The accumulator pattern keeps most instructions to 1-2 bytes of operands instead of 3. Since expressions naturally flow left-to-right (each sub-expression produces a result used by the next), the accumulator maps naturally to expression evaluation.

Comparison: The Same Operation

// Expression: (a + b) * (c - d)

═══ STACK MACHINE (JVM-like) ═══
iload 0       // push a
iload 1       // push b
iadd          // pop 2, push a+b
iload 2       // push c
iload 3       // push d
isub          // pop 2, push c-d
imul          // pop 2, push (a+b)*(c-d)
// 7 instructions, each 1-2 bytes

═══ REGISTER MACHINE (Lua-like) ═══
ADD r4, r0, r1   // r4 = a+b
SUB r5, r2, r3   // r5 = c-d
MUL r6, r4, r5   // r6 = (a+b)*(c-d)
// 3 instructions, each 3-4 bytes

═══ ACCUMULATOR (V8-like) ═══
Ldar  a0         // acc = a
Add   a1, [0]    // acc = a+b
Star  r0         // r0 = acc (save temp)
Ldar  a2         // acc = c
Sub   a3, [1]    // acc = c-d
Mul   r0, [2]    // acc = (a+b) * (c-d)
// 6 instructions, each 1-3 bytes

Stack: most instructions, smallest per instruction. Register: fewest instructions, largest per instruction. Accumulator: in between on both axes. V8 chose the accumulator because for JavaScript's typical expression patterns, it gives the best overall bytecode density and dispatch efficiency.

4.4 Complete Bytecode Examples

Click each example to see the full bytecode with line-by-line explanation:

function abs(x) { if (x >= 0) return x; else return -x; }

    Ldar             a0            // acc = x
    TestGreaterThanOrEqual Smi(0), [0]  // acc >= 0 ?
    JumpIfFalse      @else         // if not, jump to else
    Ldar             a0            // acc = x (true branch)
    Return                          // return x
  @else:
    Ldar             a0            // acc = x
    Negate           [1]           // acc = -x
    Return                          // return -x

The if/else becomes JumpIfFalse. True branch falls through (no jump). False branch jumps to a label. Both branches end with Return.

function sum(n) { let s = 0; for (let i = 0; i < n; i++) s += i; return s; }

    LdaZero                        // acc = 0
    Star            r0             // s = 0
    LdaZero                        // acc = 0
    Star            r1             // i = 0
  @loop_header:
    Ldar            r1             // acc = i
    TestLessThan    a0, [0]        // acc = (i < n)
    JumpIfFalse     @loop_end      // if not, exit loop
    Ldar            r0             // acc = s
    Add             r1, [1]        // acc = s + i
    Star            r0             // s = acc
    Ldar            r1             // acc = i
    Inc             [2]            // acc = i + 1
    Star            r1             // i = acc
    JumpLoop        @loop_header, [0] // back to test
  @loop_end:
    Ldar            r0             // acc = s
    Return                          // return s

JumpLoop creates the back-edge. V8 uses this to detect hot loops for OSR (On-Stack Replacement) — promoting a loop to optimized code while it's still running.

function getName(obj) { return obj.name; }

    LdaNamedProperty  a0, [0], [1]  // acc = a0.name
                                     // [0] = constant "name"
                                     // [1] = IC feedback slot
    Return                           // return acc

Just two instructions! LdaNamedProperty does the full property lookup (check hidden class, find offset, load). The feedback slot [1] caches the hidden class and offset for inline caching. On subsequent calls with the same-shaped object, this is one comparison + one memory read.

function outer(x) { return function inner() { return x; }; }

// outer's bytecode:
    CreateClosure    [0], [0], #pretenured
                              // Create inner function
                              // [0] = SharedFunctionInfo for inner
                              // Captures current context (has x)
    Return                     // Return the closure

// inner's bytecode:
    LdaCurrentContextSlot  [2]  // acc = context slot 2
                                // (x is in slot 2 of the captured context)
    Return                      // return x

CreateClosure allocates a new function object that captures the current scope context. LdaCurrentContextSlot reads the captured variable by its pre-computed slot index — no name lookup, no chain walking. The compiler already resolved x to slot 2 at bytecode generation time.

5. The Feedback Loop: Runtime Type Inference

Every arithmetic, comparison, property access, and call instruction has a feedback slot — a small data structure where the interpreter records what types it actually sees at runtime. This is the bridge between interpretation and JIT compilation.

1 Interpreter executes Add a1, [0] with values 2 and 3
Feedback slot [0] records: { type: Smi, count: 1 }. "I've seen small integers here."
2 Called 100 more times, always with integers
Feedback slot [0]: { type: Smi, count: 101 }. Monomorphic. "This is always integers."
3 Maglev/TurboFan reads feedback, compiles specialized code 
// TurboFan generates:
CheckSmi(a)    // assert a is Smi, deopt if not
CheckSmi(b)    // assert b is Smi, deopt if not
Int32Add(a, b) // raw integer add (no type dispatch)
CheckOverflow  // deopt if overflow

The feedback told TurboFan "always Smi", so it generates code that assumes Smi and inserts guards. If the assumption holds (which it will 99.9% of the time), execution is as fast as C.

4 Suddenly called with a string — deoptimize
CheckSmi fails. TurboFan discards the machine code. V8 falls back to interpreter bytecode. Feedback slot updates: { type: Smi|String, count: 102 }. Now polymorphic. If recompiled, TurboFan will generate code that handles both types.
This is runtime type inference. V8 does what TypeScript does (figure out what types flow where), but at runtime through observation instead of at compile time through analysis. TypeScript says "I analyzed the code and this is always a number." V8 says "I ran the code 100 times and it was always a number." Same conclusion, different method. KindScript could do the same for behavioral properties.
