What Actually Is a "Property"?

And why "properties of computations" are a special (and weird) case.

1 A property is just a yes/no question

Start with the simplest possible definition. You have a universe of objects. A property is any yes/no question you can ask about those objects. That's it.

Mathematically: a property is a subset. The objects that answer "yes" are in the set; the objects that answer "no" are out.

Property = Subset. If your universe is U, then a property P is just some subset P ⊆ U. An object x "has the property" when x ∈ P.

2 This works for any universe

The definition "property = subset" is completely general. It works for numbers, strings, people, triangles, programs — anything.

Numbers

"Is prime"

Universe: ℕ. Property: {2, 3, 5, 7, 11, 13, …}

Strings

"Contains the letter 'z'"

Universe: all strings. Property: {"zoo", "fizz", "pizza", …}

Triangles

"Is equilateral"

Universe: all triangles. Property: those where all sides are equal

Functions

"Is monotonically increasing"

Universe: ℝ → ℝ. Property: {f : x ≤ y ⟹ f(x) ≤ f(y)}

So far, nothing is special about programs. Every mathematical object can have properties, and every property is just a subset. The interesting question is: how hard is it to check whether something is in the subset?

3 Properties of programs are special

Now make the universe all programs (or equivalently, all Turing machines, all computable functions, all computations). A property of programs is still just a subset — but now something weird happens.

Rice's Theorem (1953): Every non-trivial property of the function a program computes is undecidable. There's no algorithm that always correctly answers the yes/no question.

This doesn't apply to simple objects like numbers (you can check "is 7 prime?" — just try dividing). It's specific to programs, because programs can run forever, simulate other programs, and exhibit behavior that requires infinite observation to fully determine.

Properties of programs are subsets — same as any other property. But most of them are subsets you can never fully compute membership in. That's what makes them special.

4 "Property of a computation" vs "property of a program"

Your document distinguishes these. Here's the difference:

Program

Static object

The source code itself. A text file. You can look at its syntax, its structure, count its lines.

Example property: "Uses a while loop" — just scan the text.

Computation

Dynamic process

What happens when you run the program. An (possibly infinite) sequence of states, outputs, and effects.

Example property: "Terminates" — you'd have to actually run it (forever, potentially).

A computation is the behavior that a program produces when executed. It's the full trace of everything that happens — values read, outputs written, states visited, possibly going on forever. Properties of computations are the hard ones, because you're asking about the behavior of an infinite process, not a finite text.

The spectrum: Properties of programs sit on a scale from "purely syntactic" (easy, decidable) to "purely semantic" (hard, undecidable). Your property-space document is about classifying where properties fall on this scale.

6 So what was the property-space doc about?

Your document asks: can we make a complete checklist of all properties of programs? Meaning: can we define a finite number of categories (safety, liveness, purity, etc.) such that every possible property falls neatly into one?

The answer is no, and here's the visual intuition for why:

No matter how many bubbles (categories) you draw, there are always properties that don't fit in any of them — and the arithmetic and Wadge hierarchies prove this rigorously. The space of subsets of "all programs" is just too large to capture with finitely many labels.

The punchline of your document: The space of all program properties is a mathematically infinite object. But we have powerful decomposition theorems (Alpern-Schneider, Clarkson-Schneider, etc.) that carve it along useful axes. Each theorem is provably complete for the slice it covers. It's like how you can't list every possible color with a finite palette, but RGB gives you three axes that span the visible spectrum — except for program properties, even three axes aren't enough.
