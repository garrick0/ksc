Attribute Dependency Management

How KSC coordinates inter-attribute dependencies at compile time and runtime — and how Silver, JastAdd, and Facebook Infer solve the same fundamental problem

01The Core Problem

In an attribute grammar system, every node in a syntax tree carries attributes — computed properties like types, scopes, diagnostics, or code. These attributes depend on each other: a type-check depends on a name lookup, which depends on a definition environment, which depends on declarations collected from the source.

Three directions of flow create the coordination challenge:

Synthesized (bottom-up)

Computed from a node's children. Example: kindDefs — collect all kind definitions from child compilation units, flowing results upward.

Inherited (top-down)

Provided by a node's parent. Example: defEnv — the definition environment flows from the root downward to every node that needs it.

Collection (fold)

Aggregated across the tree with an init + combine. Example: nodeCount — each node contributes 1, folded recursively over all children.

Key insight: The fundamental question every AG system must answer is: in what order do you evaluate attributes when they form a web of mutual dependencies across hundreds of tree nodes? The four systems here give strikingly different answers.

Three Strategies

Strategy How It Works Used By
Static Ordering Analyze dependency graph at compile time, produce fixed visit sequences. All attributes evaluated in prescribed order. Classical OAGs (Kastens 1980)
Demand-Driven Evaluate lazily on first access. No pre-scheduling — the call chain determines order. Cache results. KSC, Silver, JastAdd
Summary-Based Analyze each procedure independently. Produce summaries. Apply summaries at call sites. Fixpoint over SCCs. Facebook Infer

02How KSC Does It

KSC uses compile-time dependency validation combined with demand-driven runtime evaluation. Dependencies are declared explicitly, validated at codegen time, but resolved lazily at runtime through the attribute call chain.

Compile-Time Dependency Graph

Every equation function declares its dependencies via withDeps(deps, fn), which attaches a .deps: string[] property to the function:

export const eq_defEnv_root = withDeps(['kindDefs'],
  function eq_defEnv_root(ctx: Ctx): Map<string, KindDefinition> {
    for (const cu of ctx.children)
      for (const def of cu.attr('kindDefs'))
        map.set(def.name, def);
    return map;
  }
);

At codegen time, collectDepsForAttr(attr) reads .deps from all equation functions in an AttrDecl. Then buildDepGraph(attrs) in analysis/compile.ts performs a DFS topological sort with cycle detection:

// analysis/compile.ts — buildDepGraphFromAttrs
if (inStack.has(name))
  throw new Error(`Cycle in dep graph: ${name}`);
inStack.add(name);
for (const dep of depMap[name])
  visit(dep);
inStack.delete(name);
order.push(name);

The result is an AttributeDepGraph with attributes, edges, order, and declarations. But critically: the topological order is for validation and visualization only — the runtime evaluator never reads it.

Demand-Driven Evaluation

The AGNode class in evaluator/engine.ts implements the Ctx interface. When any code calls ctx.attr(name), the engine:

1 Dispatch Lookup ▼

Looks up this._config.dispatch[name] to find the generated dispatch entry. This maps the attribute name to a {direction, compute, computeRoot, computeParent} object.

2 Cache Check ▼

Non-parameterized: check _cache.get(name). Parameterized: check _paramCache.get(name)?.get(param). If cached, return immediately — no re-computation ever.

3 Compute by Direction ▼

Route to syn/inh/collection evaluation (see Attr Directions tab). The equation function may call ctx.attr() on children or parent, triggering recursive demand-driven evaluation.

4 Cache & Return ▼

Store the computed value in _cache or _paramCache. Return to caller. Every (node, attr, param) tuple is computed exactly once across the entire tree.

Key insight: There is no global scheduling. Evaluation order is determined entirely by which attributes are demanded first. Projections on the root node trigger the cascade: root.attr('allViolations') pulls on violationFor, which pulls on contextFor, which pulls on defLookup, and so on.

Three Direction Strategies

Synthesized (syn) — bottom-up

case 'syn':
  result = entry.compute(this, ...args);
  // Dispatch function switches on node.kind
  // Each kind gets its own equation or uses default

The generated dispatch function contains an exhaustive switch over all node kinds. Each kind with a specific equation gets routed; others fall through to the default.

Inherited (inh) — top-down with copy-down

case 'inh':
  if (!this.parent) {
    result = entry.computeRoot(this, ...args);
  } else {
    result = entry.computeParent(this, ...args);
    if (result === undefined)
      result = this.parent.attr(name, ...args); // copy-down!
  }

The root gets a special initial value. Non-root nodes ask their parent for an override. If the parent returns undefined, the node copies down its parent's value — inheriting the same attribute value. This means most nodes don't need explicit equations for inherited attributes.

Collection — fold over children

case 'collection':
  let acc = entry.init;
  for (const child of this.children)
    acc = entry.combine(acc, child.attr(name, ...args));
  result = acc;

Each node starts with init (e.g., 1) and folds over children recursively via combine (e.g., (a, b) => a + b). The recursive child.attr() call triggers the same fold pattern on each child.

Generated Dispatch Functions

The only codegen in KSC produces dispatch.ts, which maps attribute names to per-kind equation functions:

// generated/ts-ast/kind-checking/dispatch.ts (auto-generated)
function dispatch_kindDefs(ctx: Ctx): KindDefinition[] {
  const _kind = (ctx.node as KSNode).kind;
  switch (_kind) {
    case 'CompilationUnit':
      return eq_kindDefs_CompilationUnit(
        ctx as unknown as KindCtx<CompilationUnitNode>
      );
    default:
      return eq_kindDefs_default(ctx);
  }
}

export const dispatchConfig: DispatchConfig = {
  kindDefs: { direction: 'syn', compute: dispatch_kindDefs },
  defEnv:   { direction: 'inh', computeRoot: dispatch_defEnv_root,
              computeParent: dispatch_defEnv_parent },
  // ... 8 entries total
};

The evaluator engine is hand-written and generic. Only the dispatch functions are generated — they provide the spec-specific wiring between node kinds and equation functions.

Two Layers of Cycle Detection

Layer 1 — Compile Time

DFS in buildDepGraph()

The topological sort uses an inStack set to detect back-edges. If attribute A depends on B and B depends on A, the codegen throws Cycle in dep graph before any dispatch code is generated.

This catches static attribute-level cycles — cycles in the declared dependency structure.

Layer 2 — Runtime

_cyc Set in AGNode.attr()

Each AGNode tracks currently-evaluating attributes in a _cyc Set. Before computing, it checks _cyc.has(key). For parameterized attrs, the key is "attrName:param".

This catches dynamic self-referential cycles — e.g., an equation accidentally calling itself on the same node. Cleared in finally to allow retry.

Current limitation: KSC rejects all cycles. It cannot express analyses that require fixed-point iteration (e.g., dataflow analysis, nullable/FIRST/FOLLOW sets). This is the biggest capability gap compared to JastAdd and Silver.

03How Silver Does It

Silver (University of Minnesota / MELT group) uses demand-driven evaluation with memoization, like KSC, but adds forwarding, higher-order attributes, circular attributes, and a modular well-definedness analysis that guarantees independently-developed grammar extensions compose safely.

Demand-Driven + Memoization

Like KSC, Silver evaluates attributes lazily on first access. It compiles to Java: each nonterminal becomes a class, each production a subclass. The runtime uses Lazy and Thunk wrappers for deferred computation.

The key distinction from classical ordered AGs (Kastens): no static visit sequences. Silver and KSC share this property — evaluation order is determined entirely by runtime access patterns.

Silver adds a decorated vs. undecorated type distinction. An undecorated node is a bare data structure (Expr). A decorated node (Decorated Expr with {env}) is fully attributed with specific inherited attributes supplied. This enables precise tracking of which inherited attrs are available.

Modular Well-Definedness

Silver's signature innovation for extensibility: each grammar module is checked independently for well-definedness. If every module passes, the composed grammar is guaranteed safe.

Flow types define, for each nonterminal, which inherited attributes each synthesized attribute may depend on. These are stitched together to form flow graphs that are checked for non-circularity.

Three independent checks per module:

No Orphans

Every attribute declaration and equation has a corresponding grammar element.

Completeness

Non-forwarding productions must define all synthesized attributes on their nonterminal.

Non-Circularity

Flow types + stitched flow graphs prove no circular dependencies exist within or across modules.

Key insight: This means two independent teams can build grammar extensions, and as long as each passes the modular analysis, the combined grammar is provably well-defined. No whole-program analysis needed.

Forwarding — The Expression Problem Solution

A production can designate a semantically-equivalent "forwarded-to" tree:

abstract production gte
e::Expr ::= l::Expr r::Expr
{
  e.pp = "(" ++ l.pp ++ " >= " ++ r.pp ++ ")";
  forwards to not(lt(l, r));
}

Attributes explicitly defined on the production (pp here) compute normally. Attributes not defined are automatically forwarded — the query delegates to the forwarded-to tree (not(lt(l, r))).

This solves the expression problem: extension A adds a new attribute, extension B adds a new production. The new attribute on the new production evaluates via the forward chain, which eventually reaches host-language constructs where the attribute IS defined.

Higher-Order Attributes

An attribute whose value is an undecorated syntax tree. The created tree can itself be decorated and have its own attributes evaluated:

synthesized attribute desugared :: Stmt;  -- undecorated nonterminal type

aspect production enhancedFor
top::Stmt ::= var::VarDecl iter::Expr body::Stmt
{
  top.desugared = regularFor(...);  -- builds a new subtree
}

A reference attribute (decorated nonterminal type) points to an existing node elsewhere in the tree — useful for name binding (variable use → declaration).

Silver includes a termination analysis ensuring only finitely many trees are created during evaluation.

Collection Attributes

synthesized attribute errors :: [Message] with ++;
-- ++ is the combine operator for list concatenation

aspect production someProduction
top::Expr ::= ...
{
  top.errors := [];                   -- base value
  top.errors <- [errMsg("oops")];    -- contribution
}

:= sets the base value. <- contributes. Contributions from separate aspect productions in different modules are folded together.

Modern Silver uses monoid attributes: monoid attribute errors :: [Message] with [], ++; — specifying both empty value and operator.

Circular Attributes

Silver supports circular attributes via successive approximation:

1 Initialize to bottom ▼

The circular attribute starts at its declared bottom value on the lattice (e.g., empty set for FIRST sets).

2 Evaluate equation ▼

Compute the equation. It may read its own previous value (the bottom, first time through), producing a new value.

3 Check convergence ▼

If the new value differs from the old, repeat from step 2 with the updated value. If unchanged, the fixed point is reached.

Requires: finite-height lattice + monotonic equations → guaranteed convergence to the least fixed point. Classic use case: nullable, FIRST, and FOLLOW sets for parser construction.

04How JastAdd Does It

JastAdd is the most feature-rich demand-driven AG system. It supports circular attributes, reference attributes, non-terminal attributes, rewrites, and collection attributes — all unified under the ReCRAG formalism (Rewritable Circular Reference Attributed Grammars).

Demand-Driven Evaluation

Like KSC and Silver, JastAdd evaluates lazily. Each attribute declaration compiles to a Java accessor method with memoization. No static scheduling — order determined entirely at runtime by access patterns.

The JastAdd team describes this as "roll your own laziness for Java" — the metacompiler generates the caching infrastructure that a lazy functional language would provide natively.

Key advantage: only attributes actually needed for a given input are evaluated. If you're only doing name resolution, type-checking attributes are never computed.

Key insight: JastAdd chose demand-driven evaluation specifically because reference attributes break the tree structure. With references, the dependency graph is not known until runtime — classical static ordering can't handle it.

Circular Attributes — Fixed-Point Iteration

JastAdd's signature feature (Magnusson & Hedin, SCP 2007). When a dependency cycle is detected at runtime:

// Pseudocode of JastAdd's circular evaluation
if (IN_CIRCLE) {
  return currentValue;  // return intermediate value
} else {
  value = BOTTOM;       // initialize to lattice bottom
  IN_CIRCLE = true;
  do {
    CHANGE = false;
    newValue = evaluate();
    if (newValue !== value) {
      value = newValue;
      CHANGE = true;
    }
  } while (CHANGE);
  IN_CIRCLE = false;
}

Convergence is guaranteed when:

Finite-Height Lattice

Values form a lattice with finite height (e.g., finite subsets of terminals). This bounds the number of iterations.

Monotonic Equations

Each iteration can only move up the lattice (add elements, never remove). This guarantees convergence to a unique least fixed point.

Reference Attributes (RAGs)

An attribute whose value is a reference to another AST node. This enables non-local dependencies without environment threading:

// A variable use points to its declaration node
syn VarDecl VarUse.decl() = lookup(getName());

// The type of an expression follows the reference
syn TypeDecl VarUse.type() = decl().type();

Once you have a reference, you can access attributes on the remote node. VarUse.type() navigates to the declaration node and reads its type attribute. This is how real compilers do name resolution and type analysis.

Key insight: Reference attributes + circular attributes = CRAGs. You can define a control-flow graph via reference attributes and compute data-flow properties via circular attributes over that graph. This is how ExtendJ (JastAdd's Java compiler) works.

Non-Terminal Attributes (NTAs)

An attribute that is simultaneously a node and an attribute:

  • Like a node: has children, can have its own attributes, can be rewritten
  • Like an attribute: defined by an equation, computed on demand

The equation must return a freshly built AST subtree. This subtree is attached as a child and fully decorated — inherited attributes flow in, synthesized attributes flow out.

Key use: desugaring. ExtendJ uses NTAs to desugar Java 5+ features (generics, enhanced for-loops, autoboxing) into Java 1.4 equivalents. The desugared tree exists alongside the original and can be analyzed independently.

Conditional Rewrites

Nodes can be replaced in-place when conditions (which may depend on inherited attributes) are met:

rewrite Dot {
  when (rightSide().isArrayAccess())
  to ArrayAccess {
    return new ArrayAccess(...);
  }
}

Rewrites fire on first access to any attribute or child. After a rewrite, all rules are re-checked — enabling rewrite chains. The replacement is transparent: code accessing the node sees only the final form.

Interacts with caching: attributes cannot be cached until the node reaches its final rewritten form.

Configurable Caching

Special interactions: circular attributes don't cache intermediate values during iteration. Rewritable nodes don't cache until final form. Cache flushing is available for incremental scenarios.

05How Facebook Infer Does It

Infer is not an attribute grammar — it's an abstract interpretation framework. But it solves the same fundamental problem of coordinating interdependent analyses across a codebase. Understanding its radically different approach illuminates the design space.

Summary-Based Compositional Analysis

Where AG systems attach attributes to tree nodes, Infer attaches summaries to procedures. Each procedure is analyzed independently:

1 Intraprocedural Fixpoint ▼

Compute abstract state at each point in the procedure's CFG using transfer functions. Iterate over loops with Bourdoncle's weak topological ordering (WTO), applying widening at loop headers to guarantee termination even on infinite-height lattices.

2 Summary Production ▼

Convert the procedure's final abstract state into a summary — a set of Hoare triples {P} proc {Q} in separation logic. The summary captures the procedure's behavior for all possible callers.

3 Summary Application ▼

When analyzing a caller and encountering a call to g, look up g's summary and apply it at the call site. No re-analysis of g's body.

Analysis writers provide three things to the Infer.AI framework:

Abstract Domain

A lattice with <=, join, widen, and bottom. Must form a valid abstract interpretation.

Transfer Functions

exec_instr(state, instr) → state for each CFG instruction. Must maintain the Hoare triple contract.

Summary Converter

How to convert intraprocedural state to a summary, and how to apply callee summaries at call sites.

Bi-Abduction — Automatic Spec Discovery

Infer's core innovation (Calcagno et al., POPL 2009): automatically discover procedure specifications from bare code with no annotations.

At each call site, given current state A and callee precondition B, solve:

A * ?antiframe  |-  B * ?frame

// antiframe = what must be ADDED to the caller's precondition
//             (heap state the callee needs but isn't available)
// frame     = what SURVIVES unchanged through the call
//             (heap state the callee doesn't touch)

The * is separation logic's separating conjunction — P * Q means P and Q hold on disjoint heap regions. This enables precise local reasoning about pointer-manipulating code.

Call Graph Scheduling

Infer processes procedures in bottom-up order on the call graph:

SCC Decomposition

Find strongly connected components in the call graph. Non-recursive procedures form singleton SCCs.

Topological Sort

Sort the condensed DAG (SCCs as nodes). Leaf SCCs (no outgoing calls) are analyzed first.

Fixpoint for Recursion

Within a multi-member SCC, iterate: start with bottom summaries, analyze each procedure, check convergence, apply widening if needed.

Three scheduler implementations: file (per-file), callgraph (per-procedure, respects call deps), restart (lock-based for parallel workers).

Incremental / Differential Analysis

Infer runs on diffs, not whole codebases. At Meta, this achieves 15–20 minute analysis times on 10M+ line codebases.

1 Identify changed files ▼

Accept a list of changed source files from the CI system.

2 Invalidate summaries ▼

Remove all summaries that transitively depend on changed procedures, using the persisted dependency graph.

3 Re-analyze affected only ▼

Re-analyze only the affected procedures. Reuse cached summaries for everything else.

Fundamental Differences

Dependency Structure

AG systems (KSC, Silver, JastAdd) work over trees — the AST has well-defined parent-child relationships. Dependencies follow the tree (syn up, inh down) plus optional references. This structure makes dependency analysis tractable.

Infer works over the call graph — an arbitrary directed graph with cycles (recursion). There are no parent-child semantics. SCCs must be identified and processed via fixpoint iteration.

Granularity

AG systems: The unit is a single (node, attribute, param) tuple. Fine-grained — each node can have different values for the same attribute. Evaluation is a cascade of accessor calls.

Infer: The unit is an entire procedure. Computing a single "attribute" (summary) requires running a full intraprocedural abstract interpretation with its own fixpoint computation. Coarser-grained, but each unit is self-contained.

Exactness

AG systems: Typically compute exact values. The equations define the value precisely. Circular attributes on finite lattices converge to the exact least fixed point.

Infer: Must approximate. The exact semantics of programs are undecidable. Widening operators ensure termination but introduce over-approximation. Narrowing can partially recover precision, but results are inherently approximate.

08Our Options — What KSC Could Adopt

Each option below is inspired by a specific system's approach. Click to expand for details, implementation complexity, and trade-offs.

1. Circular Attributes (Fixed-Point Iteration)
JastAdd ▼
Add fixed-point evaluation to the engine for attributes that reference themselves. Enables dataflow analysis, nullable/FIRST/FOLLOW, type inference.
Complexity Medium
Impact High
Files Changed evaluator/engine.ts, analysis/types.ts

What Changes

Modify AGNode.attr() to detect cycles and iterate instead of throwing. Add bottom and isEqual to AttrDecl for circular attributes. Replace compile-time cycle rejection with lattice convergence checking.

Trade-offs

Pro: Enables a whole class of analyses (flow analysis, type inference). Con: Harder to debug (values change across iterations), potential non-termination if lattice constraints are violated, more complex engine.

2. Reference Attributes
JastAdd ▼
Allow attributes whose values are references to other AGNodes. Enables direct name binding (variable use → declaration) without environment threading.
Complexity Low-Medium
Impact Medium
Files Changed evaluator/types.ts, spec equations

What Changes

Already partially possible — ctx.attr() can return any value including Ctx references. The main change is adding a typed wrapper (RefAttr<TargetKind>) and validation that references are well-formed.

Trade-offs

Pro: Simplifies name resolution, eliminates some environment-passing boilerplate. Con: Breaks tree-locality assumptions, complicates any future incrementality work (reference targets may change).

3. Higher-Order / Non-Terminal Attributes
Silver ▼
Allow attributes that produce new AST subtrees. The created subtrees get fully decorated and evaluated. Enables desugaring and tree transformations.
Complexity Medium-High
Impact High
Files Changed evaluator/engine.ts, grammar/derive.ts

What Changes

The evaluator must handle dynamically-created tree nodes, decorate them (supply inherited attrs), and manage their lifecycle. Need a termination analysis to ensure finite tree creation.

Trade-offs

Pro: Very powerful for desugaring, lowering, and multi-pass transformations without separate passes. Con: Adds significant complexity to the engine. Termination becomes non-trivial.

4. Forwarding (Expression Problem Solution)
Silver ▼
Productions delegate undefined attributes to a semantically-equivalent "forwarded-to" tree. Solves the expression problem for extensible grammars.
Complexity High
Impact Medium
Files Changed grammar/, analysis/, evaluator/

What Changes

Grammar nodes need a forward-to declaration. The evaluator must check for explicit equations first, then delegate to the forwarded tree. Inherited attrs must be auto-copied to forwarded trees.

Trade-offs

Pro: Enables a plugin ecosystem where extensions compose safely. Con: Requires grammar/spec infrastructure changes across all layers. Only valuable if we want third-party grammar extensions.

5. Incremental Evaluation
All Systems ▼
Track runtime attribute dependencies. On tree edit, invalidate only affected attributes and re-evaluate. Essential for IDE integration and watch mode.
Complexity High
Impact High
Files Changed evaluator/engine.ts (major rework)

What Changes

During evaluation, record which (node, attr) pairs were accessed by each computation. Build a runtime dependency graph. On edit, walk the dependency graph to find invalidated attributes. Re-evaluate only those. This is what Silver's "dynamic dependency graph for efficient re-evaluation after local tree edits" provides.

Trade-offs

Pro: Essential for responsive tooling (LSP, watch mode). Con: Major engine rework. Runtime overhead from dependency tracking. Interaction with caching, parameterized attrs, and potential future circular attrs is complex.

6. Compositional / Summary-Based Analysis
Infer ▼
Analyze files/modules independently and produce summaries. Apply summaries at import sites. Enables diff-time analysis for CI.
Complexity High
Impact High
Files Changed Fundamental model change

What Changes

Replace whole-tree evaluation with per-file analysis that produces a summary (e.g., "this module exports kind X with properties Y"). At import sites, apply summaries instead of re-analyzing the imported module. Track which summaries depend on which.

Trade-offs

Pro: Scales to huge codebases, enables diff-time CI like Infer's 15-min analysis on 10M+ LOC. Con: Fundamentally changes the evaluation model. Loses whole-program precision. Cross-file kind-checking becomes approximate.

7. Parallel Evaluation
JastAdd ▼
Independent attribute computations run concurrently. JastAdd uses @Parallel annotations for collection surveys.
Complexity Medium
Impact Medium
Files Changed evaluator/engine.ts

What Changes

Identify independent subtrees (e.g., separate compilation units) and evaluate their attributes concurrently. Collection attribute surveys can traverse children in parallel. Cache access needs to be thread-safe (or use per-thread caches).

Trade-offs

Pro: Performance gain on large ASTs with many independent subtrees. Con: JS single-threaded model limits this to worker threads. Concurrency bugs. Cache contention.

8. Modular Well-Definedness Analysis
Silver ▼
Check completeness and non-circularity per grammar module independently. Enable a safe extension/plugin ecosystem for KSC.
Complexity Medium
Impact Medium
Files Changed analysis/validate.ts, analysis/types.ts

What Changes

Introduce flow types: for each grammar kind, declare which inherited attributes each synthesized attribute may depend on. At spec validation time, check completeness (all syn attrs defined) and non-circularity via flow type analysis. Each spec module is checked independently.

Trade-offs

Pro: Enables safe composition of independently-developed analysis specs. Con: Constrains spec authors (must declare flow types). Only valuable if we want a plugin/extension ecosystem.

Q

What evaluation strategy does KSC use, and does the compile-time topological order drive runtime evaluation?

A

KSC uses demand-driven (lazy) evaluation. The topological order computed by buildDepGraph() is for validation and visualization only — the runtime evaluator never reads it. Evaluation order is determined entirely by which attributes are accessed first via ctx.attr().

Q

How does KSC detect dependency cycles, and at how many levels?

A

Two levels: (1) Compile-time — DFS in buildDepGraph() with an inStack set detects static attribute-level cycles. (2) Runtime — a per-node _cyc Set in AGNode.attr() catches dynamic self-referential cycles (cleared in finally to allow retry).

Q

How does inherited attribute "copy-down" work in KSC?

A

When evaluating an inh attribute on a non-root node, the engine calls entry.computeParent(ctx). If it returns undefined, the node copies down its parent's value via this.parent!.attr(name). This means most nodes inherit attributes automatically without needing explicit equations.

Q

What is JastAdd's key innovation for handling circular attribute dependencies?

A

Fixed-point iteration on finite lattices. When a cycle is detected at runtime, the attribute starts at a bottom value and iterates: evaluate, check if changed, repeat. Convergence is guaranteed by requiring a finite-height lattice and monotonic equations, reaching the least fixed point.

Q

What is Silver's "forwarding" mechanism and what problem does it solve?

A

Forwarding lets a production delegate undefined attributes to a semantically-equivalent tree. It solves the expression problem: extension A adds a new attribute, extension B adds a new production. The new attribute on the new production evaluates by forwarding through to host-language constructs where it IS defined.

Q

What is bi-abduction in Facebook Infer, and why is it important?

A

Bi-abduction simultaneously discovers the antiframe (missing precondition) and frame (unchanged state) at each call site: A * ?antiframe |- B * ?frame. It enables Infer to automatically synthesize procedure specifications from bare code with no annotations — the core mechanism behind compositional analysis at scale.

Q

How does Infer handle recursive / mutually-recursive procedures?

A

Infer identifies strongly connected components (SCCs) in the call graph. Within an SCC, it starts with bottom summaries for all procedures, analyzes each using current summaries of co-recursive partners, checks convergence, and repeats with widening to ensure termination. This is analogous to JastAdd's circular attribute fixed-point but at procedure granularity.

Q

What is the biggest capability gap in KSC compared to Silver and JastAdd, and what would adopting it require?

A

KSC rejects all circular dependencies, while both Silver and JastAdd support fixed-point iteration. Adopting this requires: modifying AGNode.attr() in evaluator/engine.ts to iterate instead of throw on cycles, adding bottom and isEqual to AttrDecl, and ensuring equations are monotonic on finite lattices for convergence.
