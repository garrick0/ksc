Sharing Without a Shared AST

Three production-proven architectures that achieve multi-language code infrastructure without forcing languages into a universal representation.

01The Thesis

You don't need a universal AST to build multi-language tooling.

The instinct when building multi-language tools is to create a single, shared AST that all languages compile into. But the most successful multi-language systems in production don't do this. Instead, they share infrastructure — protocols, algorithms, compiler passes — while keeping language representations distinct.

Three systems prove this at scale, each with a different sharing mechanism:

Truffle: Language-specific AST nodes → Interop Protocol → Shared compiler
Languages never see each other's trees. They speak a common protocol.
MLIR:   Domain-specific dialects → Coexistence in one IR → Progressive lowering
Multiple abstraction levels live side-by-side. No forced normalization.
CodeQL: Language-specific extractors → Adapter Interface → Shared algorithm
The algorithm asks questions. Each language answers in its own way.
The common thread: All three define a boundary between shared and language-specific concerns. The boundary is a protocol (Truffle), a type system for operations (MLIR), or a module signature (CodeQL). None require languages to surrender their native representation.

Truffle: "I don't know what you are, but I know what to ask you"

Each language builds its own AST nodes. Cross-language interaction happens through the InteropLibrary — a protocol of ~150 standardized messages. When JavaScript receives a Ruby hash, it doesn't need to understand Ruby's internal representation. It just asks hasMembers(), readMember("key"). The Ruby hash responds according to its own semantics.

The Graal compiler then partially evaluates the interpreter + AST structure together, compiling the interop dispatch into direct calls. After JIT compilation, the protocol overhead disappears entirely.

MLIR: "Everything lives together, nothing is forced to lower early"

MLIR doesn't have one IR — it has an infinite number of IRs, called dialects. A dialect is a namespace of operations, types, and attributes. Operations from different dialects coexist in the same function at the same time. A matrix multiply function can mix affine.for loops, arith.mulf arithmetic, and memref.load memory ops.

Lowering is progressive: each pass converts one dialect to another, but only when the information loss is acceptable. High-level structure stays high-level as long as it's useful for analysis.

CodeQL: "I'll find the paths — you tell me the edges"

The dataflow algorithm lives in DataFlowImplCommon.qll — completely language-independent. Each language implements a DataFlowPrivate adapter that answers questions: "What are the local flow steps?" "How do you resolve calls?" "What counts as a field access?" The shared algorithm never knows which language it's analyzing.

Python provides ESSA (Enhanced SSA) edges. Java provides standard SSA edges. The algorithm consumes both identically.

A foreign object that represents "nothing" across languages. JS null, Python None, Ruby nil all respond isNull() = true.
Boolean coercion protocol. Languages can export boolean-like values that other languages check with isBoolean() before using asBoolean().
isExecutable() asks "can I call this?" Functions, lambdas, closures, procs all respond true. execute(args...) invokes the callable. After JIT compilation, this cross-language call is inlined — zero overhead.
isInstantiable() asks "can I new this?" Classes, constructors respond true. instantiate(args...) creates a new instance.
Number coercion with lossless checking. fitsInInt(), fitsInDouble(), etc. verify the value can be represented without loss before conversion. Languages with different numeric representations (Python arbitrary-precision, JS double-only) safely interoperate.
String protocol. isString() checks if the object represents text. asString() or asTruffleString() extracts it. TruffleString is a shared, encoding-aware string implementation across languages.
Object property access. hasMembers() asks "do you have named properties?" Then readMember("name") retrieves one. Both JS objects and Ruby hashes respond — each according to their own semantics.
Direct method invocation. invokeMember("sort", args) calls a method by name. This is more efficient than readMember + execute because the object can dispatch directly to the method without creating an intermediate function value.
Mutation protocol. isMemberModifiable() and isMemberInsertable() distinguish between changing an existing property and adding a new one. writeMember() performs the mutation. removeMember() deletes a property.
Indexed collection protocol. hasArrayElements() checks if indexed access is supported. getArraySize() + readArrayElement(i) provide iteration. JS arrays, Python lists, Ruby arrays all respond.
Key-value collection protocol (added later). hasHashEntries(), readHashValue(key), getHashEntriesIterator(). Ruby hashes, Python dicts, Java HashMaps all respond. Separate from members because hash keys can be arbitrary objects, not just strings.
Lazy iteration protocol. hasIterator() checks if the object is iterable. getIterator() produces an iterator object that responds to hasIteratorNextElement() and getIteratorNextElement().
Reflective type information. getMetaObject() returns the type/class of a value. isMetaInstance() checks instanceof-style relationships. getMetaQualifiedName() returns a human-readable type name.
Variable scope protocol. isScope() identifies lexical scopes. hasScopeParent() / getScopeParent() walk the scope chain. Used by debuggers and code analysis tools to inspect variables without understanding each language's scoping rules.

Self-Modifying AST & Specialization

Truffle AST nodes rewrite themselves at runtime. A node starts generic, then specializes based on the actual types it encounters:

@NodeChild("leftNode")
@NodeChild("rightNode")
public abstract class AddNode extends EasyScriptNode {

    @Specialization                       // Fast path: both ints
    protected int addInts(int left, int right) {
        return left + right;
    }

    @Specialization(replaces = "addInts") // Fallback: doubles
    protected double addDoubles(double left, double right) {
        return left + right;
    }
}
// When addInts sees a double, the AST node literally
// rewrites itself to use addDoubles instead.
The compilation magic: Graal's partial evaluator takes the interpreter + specific AST structure and collapses them into a single flat computation graph. @Child fields become constants. Specialization guards become dead code. The result: language-specific semantics compiled to native speed, with cross-language calls inlined across language boundaries.

04MLIR: Dialect Coexistence

Multiple abstraction levels in the same IR, lowered progressively. Click levels to explore.

Four Dialects in One Function

This matrix multiplication uses func, arith, memref, and affine ops interleaved freely:

func.func @matmul(%A: memref<?x?xf32>, %B: memref<?x?xf32>, %C: memref<?x?xf32>) {
  %c0 = arith.constant 0 : index
  %c1 = arith.constant 1 : index
  %M  = memref.dim %A, %c0 : memref<?x?xf32>
  %N  = memref.dim %B, %c1 : memref<?x?xf32>
  %K  = memref.dim %A, %c1 : memref<?x?xf32>

  affine.for %i = 0 to %M {              // polyhedral loop
    affine.for %j = 0 to %N {
      affine.for %k = 0 to %K {
        %a = affine.load %A[%i, %k]
        %b = affine.load %B[%k, %j]
        %c = affine.load %C[%i, %j]
        %p = arith.mulf %a, %b : f32     // arithmetic op
        %s = arith.addf %c, %p : f32
        affine.store %s, %C[%i, %j]
      }
    }
  }
  return
}

SSA values flow freely between dialects: %c0 (an arith.constant) is used as an operand to memref.dim. No conversion needed.

Progressive Lowering Pipeline

Click each level to see what's happening:

High linalg / tosa / stablehlo Domain-specific ops
↓
Mid affine Polyhedral loop optimization
↓
Mid scf (structured control flow) for, while, if — still structured
↓
Low cf (control flow) Branches & jumps — unstructured
↓
Target llvm dialect → LLVM IR → machine code Final code generation
High-level dialects represent domain concepts directly. linalg.matmul is a single op for matrix multiplication. tosa.conv2d is a 2D convolution. At this level, the compiler can apply domain-specific optimizations (tiling, fusion) that would be invisible at lower levels.
affine.for loops carry polyhedral analysis information: loop bounds are affine expressions over induction variables and symbols. The compiler can auto-parallelize, tile, and vectorize using polyhedral techniques. Once lowered, this information is lost forever.
scf.for, scf.while, scf.if — structured control flow without the polyhedral constraints. Loop optimizations like unrolling and pipelining still work here. Block arguments (not PHI nodes) carry values between iterations.
cf.br, cf.cond_br — raw branches. All structured control flow is gone. The compiler works with a CFG of basic blocks. This is close to LLVM's level but still in MLIR's type system.
The llvm dialect wraps LLVM's type system and operations. llvm.call, llvm.load, llvm.getelementptr. Translation to actual LLVM IR is mechanical — one-to-one op mapping. Then LLVM's backend handles register allocation and machine code emission.
The MLIR principle: "Op semantics declare and transport information that is traditionally obtained by compiler analyses. This information is not lost by lowering prematurely." Each pass converts only the ops it understands, leaving others untouched. After all passes, only the target dialect remains.

05CodeQL: The Adapter Interface

One algorithm, many languages. The algorithm asks questions — each language answers differently.

The Layer Architecture

1 DataFlowImplCommon.qll — The Shared Algorithm ▼

The language-independent core. Implements:

  • • Matching call-sites with return values
  • • Matching field writes with field reads
  • • Context-sensitive interprocedural flow tracking
  • • Lambda/higher-order function flow
  • • Access path tracking through field accesses
  • • Path graph construction for reporting

This file is identical whether analyzing Java, Python, C++, or Go. It never knows which language it's running on.

2 DataFlowPrivate.qll — The Language Adapter ▼

Each language implements this module. The required predicates form the "contract" between the shared algorithm and the language:

// Core edge relation (SSA-based)
predicate simpleLocalFlowStep(Node n1, Node n2)

// Call graph
DataFlowCallable viableCallable(DataFlowCall c)
predicate isParameterNode(ParameterNode p, DataFlowCallable c, ParameterPosition pos)

// Field flow
predicate storeStep(Node n1, ContentSet f, Node n2)
predicate readStep(Node n1, ContentSet f, Node n2)

// Inter-callable jumps (globals, captures)
predicate jumpStep(Node n1, Node n2)

// Lambda support
predicate lambdaCreation(Node creation, LambdaCallKind kind, DataFlowCallable c)
predicate lambdaCall(DataFlowCall call, LambdaCallKind kind, Node receiver)
3 ConfigSig — The User Query ▼

End users write queries by implementing ConfigSig — declaring what they're looking for:

module SqlInjectionConfig implements DataFlow::ConfigSig {
  predicate isSource(DataFlow::Node source) {
    source instanceof RemoteFlowSource
  }
  predicate isSink(DataFlow::Node sink) {
    sink = any(SqlQuery q).getAnArgument()
  }
  predicate isBarrier(DataFlow::Node node) {
    node.getType() instanceof NumericType
  }
}
module SqlFlow = TaintTracking::Global<SqlInjectionConfig>;

The Global<> parameterized module instantiates the shared algorithm with this config. The same query pattern works across all supported languages.

4 Consistency Checks — The Safety Net ▼

DataFlowImplConsistency.qll contains automated invariant checks that verify each language's adapter satisfies the contracts expected by the shared algorithm:

  • ✓ Every PostUpdateNode has a valid getPreUpdateNode()
  • ✓ Every ArgumentNode has a valid DataFlowCall
  • ✓ Parameter positions match argument positions
  • ✓ Every node has an enclosing callable
  • ✓ Return nodes connect properly to out-nodes

These checks are themselves CodeQL queries that run against the language adapter, catching broken invariants before they cause incorrect analysis.

Java vs Python: Same Algorithm, Different Edges

Java's DataFlowPrivate uses standard SSA form. simpleLocalFlowStep follows def-to-first-use and use-to-next-use chains. Java's strong static type system means CastNode provides effective pruning — if a value is typed as int, it can't flow to a String sink.

File: java/ql/lib/semmle/code/java/dataflow/internal/DataFlowPrivate.qll

Python's DataFlowPrivate uses Enhanced SSA (ESSA), which treats anything that may change how an object is viewed as a definition — not just assignments, but attribute writes, passing to possibly-mutating functions, etc. SynthDictSplatParameterNode handles Python's **kwargs, which has no Java equivalent. Dynamic typing means CastNode provides almost no pruning.

File: python/ql/lib/semmle/python/dataflow/new/internal/DataFlowPrivate.qll

The architectural insight: The shared algorithm in DataFlowImplCommon.qll is identical regardless of language. Java produces SSA edges; Python produces ESSA edges; the algorithm consumes both the same way. Language differences are entirely encapsulated in the adapter layer.

07Synthesis: The Shared Pattern

What all three systems have in common — and what it means for designing multi-language tools.

Despite operating in different domains (runtimes, compilers, static analysis), all three systems follow the same structural pattern:

1 Define a boundary contract ▼

Each system defines a precise interface between the shared infrastructure and the language-specific parts:

  • Truffle: InteropLibrary — "what messages can I send to a foreign object?"
  • MLIR: ODS dialect definition — "what operations, types, and attributes does this dialect provide?"
  • CodeQL: DataFlowPrivate signature — "what predicates must a language implement?"

The boundary is explicit, typed, and verifiable. It's not just a convention — violating it causes compilation errors or automated consistency check failures.

2 Share the hard algorithm, not the data ▼

The expensive, carefully-engineered component — JIT compiler, progressive lowering pipeline, interprocedural dataflow — is written once and shared. The language-specific part is the relatively easy work: defining nodes, defining ops, implementing adapter predicates.

This inverts the universal-AST approach, where the easy part (representation) is shared and the hard part (analysis) must be duplicated or impoverished.

3 Eliminate abstraction overhead ▼

All three systems ensure the sharing mechanism doesn't cost performance:

  • Truffle: Partial evaluation + JIT inlines cross-language calls → zero overhead at runtime
  • MLIR: Progressive lowering removes high-level abstractions → only target dialect ops remain
  • CodeQL: QL's Datalog evaluation merges adapter predicates with shared algorithm → single evaluation pass
4 Language fidelity is preserved ▼

No language surrenders its native representation. Truffle Ruby keeps its own AST nodes with Ruby semantics. MLIR's affine dialect keeps polyhedral information that would be lost in a generic loop representation. Python's ESSA captures mutation semantics that Java's SSA doesn't need.

This is the critical advantage over universal ASTs: no information is lost. The 35% of constructs that resist universal representation are simply kept in their native form.

The lesson for multi-language tools: Don't build a universal AST. Build a universal interface that languages implement according to their own semantics. The interface says what questions the shared infrastructure needs answered. Each language answers those questions using its own native representation.

Q

How does Truffle achieve cross-language interop without a shared AST?

A

Through the InteropLibrary message protocol (~150 messages). Language objects implement protocol methods (hasMembers, readMember, execute, etc.) according to their own semantics. A JavaScript function doesn't know it's being called from Ruby — it just responds to isExecutable() / execute() messages.

Q

What is "self-modifying AST" in Truffle and why does it matter?

A

Truffle AST nodes rewrite themselves at runtime based on observed types via @Specialization. An AddNode starts with an addInts specialization; when it encounters a double, the node replaces itself with addDoubles. This matters because Graal's partial evaluator treats the stabilized AST as constants, compiling the specialized interpreter into native-speed code with no interpretation overhead.

Q

What is an MLIR dialect and how can multiple dialects coexist in one function?

A

A dialect is a namespaced set of operations, types, and attributes. Multiple dialects coexist because SSA values flow freely between ops from different dialects — an arith.constant result can be used as an operand to memref.dim. MLIR explicitly supports this: operations are tagged with their dialect prefix (affine.for, arith.mulf), and a single function can contain ops from any number of dialects simultaneously.

Q

What does "progressive lowering" mean in MLIR, and why is it better than immediate lowering?

A

Progressive lowering converts one dialect at a time: affine → scf → cf → llvm. Each pass only transforms ops it understands, leaving others intact. It's better than immediate lowering because high-level information (like polyhedral loop structure in affine.for) is preserved as long as it's useful for optimization. Lowering too early destroys information that later passes could have exploited.

Q

In CodeQL's architecture, what is DataFlowPrivate.qll and what must it provide?

A

It's the language-specific adapter module that each language implements. It must provide: simpleLocalFlowStep (SSA-based flow edges), viableCallable (call resolution), storeStep/readStep (field flow), jumpStep (inter-callable flow for globals/captures), and lambda support predicates. The shared algorithm in DataFlowImplCommon.qll consumes these predicates without knowing which language produced them.

Q

How does CodeQL's Python adapter differ from its Java adapter?

A

Java uses standard SSA (definitions are assignments). Python uses Enhanced SSA (ESSA), where anything that might change how an object is viewed counts as a definition — including attribute writes and passing to possibly-mutating functions. Python also needs SynthDictSplatParameterNode for **kwargs (no Java equivalent), and gets less CastNode pruning due to dynamic typing. But the shared algorithm handles both identically.

Q

What is the common structural pattern across all three systems?

A

All three: (1) define an explicit boundary contract (protocol, dialect spec, module signature), (2) share the hard algorithm (JIT compiler, lowering pipeline, dataflow algorithm) while keeping language-specific representation, (3) eliminate abstraction overhead (via partial evaluation, progressive lowering, or Datalog evaluation), and (4) preserve full language fidelity — no information is lost to normalization.
