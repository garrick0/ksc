Expressions — The Core Category

Expressions are the most fundamental syntactic category. An expression denotes a value — when evaluated, it produces a result. Expressions are the category with the richest recursive structure: expressions contain sub-expressions, nested arbitrarily deep.

Γ ⊢ e₁ : number    Γ ⊢ e₂ : number
Γ ⊢ e₁ + e₂ : number
T-Add

The typing judgment Γ ⊢ e : τ reads: "In type environment Γ, expression e has type τ." Every expression gets a type.

Denotational view: ⟦ e ⟧ : Env → Value — an expression denotes a function from environment to value. ⟦ 2 + 3 ⟧ = 5. ⟦ x ⟧ = lookup(x, ρ).

Examples across languages:

TypeScript

2 + 3 * 4          // arithmetic
f("hello")           // call
x > 0 ? "yes" : "no" // conditional expr
[1, 2, 3].map(x => x*2) // method + arrow

Haskell

map (*2) [1, 2, 3]  -- application
if x > 0 then "yes" else "no"
let y = 5 in y + 3    -- let expression
\x -> x * 2           -- lambda

Rust

if x > 0 { "yes" } else { "no" }
match n { 0 => 1, _ => n * fact(n-1) }
vec![1,2,3].iter().map(|x| x*2)
{ let t = compute(); t + 1 } // block expr

C

2 + 3 * 4        // arithmetic only
f("hello")         // call
x > 0 ? 10 : 20   // ternary (limited)
// no if-expr, no match-expr
// no block-as-expression
Key property: Expressions compose. Anywhere you can put an expression, you can put a compound expression. This is why expression-oriented languages are more composable — more positions in the grammar accept expressions.

Statements — Commands That Transform State

Statements are executed for their effects, not for a value they produce. A statement transforms the program state — memory, control flow position, or the external world.

Γ ⊢ e : τ
Γ ⊢ return e   ok
T-Return

The judgment Γ ⊢ s ok reads: "In type environment Γ, statement s is well-formed." Statements don't get types — they get well-formedness checks.

Denotational view: ⟦ s ⟧ : Store → Store⊥ — a statement denotes a partial function from states to states. ⟦ x = 5 ⟧(σ) = σ[x ↦ 5]. The ⊥ (bottom) represents non-termination — a loop that runs forever.

What statements do:

1 Modify memory — assignment: x = 5;
The assignment statement takes the current store σ, evaluates the expression on the right to a value v, and produces a new store σ[x ↦ v] where the location bound to x now holds v. This is the fundamental state-transforming operation.
2 Control flow — branching: if (...) {...} else {...}
An if-statement evaluates its condition expression, then executes one of two sub-statements. It doesn't produce a value — it selects which state transformation to apply. In TypeScript, this also narrows the type environment differently in each branch.
3 Iteration — loops: for (...) {...}, while (...) {...}
Loops apply a state transformation repeatedly. Denotationally, a while loop is the least fixed point of a recursive state transformer: ⟦ while(c) s ⟧ = fix(λf.λσ. if ⟦c⟧(σ) then f(⟦s⟧(σ)) else σ). This is why loops can diverge — the fixed point may not exist.
4 Transfer control — jumps: return, break, throw
These statements don't just transform state — they abort the current continuation. A return transfers control to the caller. A throw transfers control to the nearest enclosing catch. In the formal model, these are handled by extending the state with a "control tag" indicating normal vs exceptional flow.
5 Sequence — composition: s₁; s₂
The semicolon is the fundamental statement combinator. ⟦ s₁; s₂ ⟧ = ⟦ s₂ ⟧ ∘ ⟦ s₁ ⟧ — apply s₁'s transformation, then s₂'s. This is function composition of state transformers, and it's where the "imperative = monadic" insight comes from (Moggi 1989).

Declarations — Binding Names to Meanings

Declarations introduce bindings — they associate names with entities (values, types, modules) and extend the environment. The key theoretical distinction from statements: a declaration extends the environment (name → meaning mapping), while a statement modifies the store (location → value mapping).

Γ ⊢ e : τ
Γ, x : τ ⊢ let x = e   ok
T-Let

Notice how the conclusion extends Γ with a new binding x : τ. This is what makes declarations special — they modify the environment available to subsequent code.

Denotational view: ⟦ let x = e ⟧ : Env → Env — a declaration denotes an environment extension. ⟦ let x = 5 ⟧(ρ) = ρ[x ↦ 5]. Subsequent code sees the extended environment.

Kinds of declarations:

Value Declarations

const x = 5;          // TS
let x = 5;            // Rust
val x = 5             // ML
x = 5                 // Haskell
Bind a name to a runtime value

Function Declarations

function f(x) {...}   // TS
fn f(x: i32) {...}    // Rust
fun f x = ...          // ML
f x = ...              // Haskell
Bind a name to a function

Type Declarations

type Alias = string | number; // TS
type Alias = i32;     // Rust
type alias = int      // ML
type Alias = Int      // Haskell
Bind a name to a type (erased)

Module Declarations

import { x } from 'm'; // TS
use m::x;              // Rust
open M                 // ML
import M (x)           // Haskell
Link a name to an export elsewhere
In TypeScript: Declarations create symbols. A symbol stores: name, declaration site, type, scope, visibility (exported?), mutability (const/let). When TS later sees the expression x, it resolves the identifier to its symbol.

Patterns — The Mirror of Expressions

Patterns are the dual of expressions. Where expressions construct values (build up), patterns deconstruct them (take apart). They have their own grammar — they look like expressions but the rules for what's valid are completely different.

Γ ⊢ e : { x: τ₁, y: τ₂ }
Γ, a : τ₁, b : τ₂ ⊢ let { x: a, y: b } = e   ok
T-Destructure

A single pattern can introduce multiple bindings simultaneously — it distributes the scrutinee's structure across named parts.

Pattern richness varies enormously by language:

TypeScript has destructuring patterns only — no full pattern matching.

// Object destructuring
const { name, age } = person;

// Array destructuring
const [first, ...rest] = items;

// Nested destructuring
const { address: { city, zip } } = user;

// In function parameters
function greet({ name, age }: Person) { ... }

// Default values
const { x = 0, y = 0 } = point;

// NO algebraic pattern matching
// Must use discriminated unions + if/switch instead
if (result.kind === 'ok') { result.value } // ← not a pattern

Rust has full algebraic patterns — one of the richest pattern systems.

// Enum patterns
match result {
    Ok(value) => println!("{}", value),
    Err(e) => eprintln!("{}", e),
}

// Nested patterns with guards
match msg {
    Message::Move { x, y } if x > 0 => ...,
    Message::Color(r, _, _) => ...,
    _ => ...,  // wildcard
}

// Binding + matching simultaneously
match value {
    n @ 1..=5 => println!("small: {n}"),
    n => println!("large: {n}"),
}

// Patterns everywhere: let, if-let, while-let, for
if let Some(x) = optional { ... }
while let Some(item) = iter.next() { ... }

OCaml/ML — the origin of algebraic pattern matching.

(* Algebraic data type *)
type expr =
  | Num of int
  | Add of expr * expr
  | Mul of expr * expr

(* Pattern matching with exhaustiveness checking *)
let rec eval = function
  | Num x -> x
  | Add (e1, e2) -> eval e1 + eval e2
  | Mul (e1, e2) -> eval e1 * eval e2

(* Compiler warns if you miss a case! *)
(* This is "the ML way" — make illegal states *)
(* unrepresentable via algebraic types + patterns *)

Haskell — patterns are everywhere, including function definitions.

-- Patterns in function definitions
factorial 0 = 1
factorial n = n * factorial (n - 1)

-- Patterns in case expressions
eval expr = case expr of
  Num x     -> x
  Add e1 e2 -> eval e1 + eval e2

-- As-patterns: bind AND match
firstTwo xs@(a:b:_) = (xs, a, b)

-- View patterns (GHC extension)
f (length -> 0) = "empty"
f _              = "non-empty"
Duality: Expression { x: 1, y: 2 } constructs an object. Pattern { x, y } deconstructs one. Same syntax, opposite direction. Expressions are introduction forms; patterns are elimination forms.

Types — A Language Within the Language

Type expressions form their own sub-language with their own grammar, their own composition rules, and in TypeScript's case, their own Turing-complete computation. They denote classifiers — sets of values, predicates, constraints — and exist purely at the static level. They are erased before runtime.

⊢ τ₁ type    ⊢ τ₂ type
⊢ τ₁ → τ₂ type
T-Arrow

Types have their own well-formedness judgment: ⊢ τ type. They compose via type constructors (→, ×, +, etc.) just as expressions compose via expression constructors (+, *, function application).

TypeScript's type language is remarkably expressive:

// Primitive types (base cases)
number, string, boolean, null, undefined, void, never

// Composite types (type constructors)
A | B                  // union (set union)
A & B                  // intersection (set intersection)
T[]                    // array type constructor
[T₁, T₂]              // tuple
(x: A) => B           // function type (→)

// Type-level computation (!)
type IsString<T> = T extends string ? "yes" : "no";  // conditional
type Keys<T> = { [K in keyof T]: T[K] };             // mapped type
type Deep<T> = T extends object                       // recursive type
  ? { [K in keyof T]: Deep<T[K]> }
  : T;
Key insight: TypeScript's type language is Turing-complete — you can encode arbitrary computation in conditional types + recursive types. This makes it a genuine programming language embedded in the type annotation positions of another programming language. Most languages' type sub-languages are much simpler (Java's generics, C's type names, Haskell's type classes).

Modules — Organizing the Namespace

Modules are the top-level organizational category. A module is a collection of declarations with a visibility boundary. Some bindings are exported (public), others are internal (private). Modules create a two-level structure: the language of expressions/statements/declarations, and a "module language" describing how collections are composed and linked.

Denotational view: ⟦ module ⟧ : Env → Envpublic — a module denotes a function from the imported environment to the exported environment. The module hides its internal bindings.

Module systems vary significantly:

TypeScript / ES Modules

// File = module (implicit boundary)
import { sort } from './utils';
export function process() { ... }
export type Config = { ... };
// Can export values AND types

Rust

// Explicit module hierarchy
mod utils {
    pub fn sort() { ... }
    fn internal() { ... } // private
}
use utils::sort;

OCaml

(* First-class modules! *)
module type SORTABLE = sig
  type t
  val compare : t -> t -> int
end
(* Module signatures = type system for modules *)

Haskell

module Utils (sort, Config) where
-- Explicit export list
sort :: [a] -> [a]  -- exported
helper = ...          -- hidden
-- Type classes = module-like structures
For KindScript: Module-level properties are natural. @ksc pure on a module could mean "all exports of this module are pure." This is how the property system scales — instead of annotating every function, you annotate the module boundary.

2. The Expression/Statement Spectrum

The most significant design decision in language design is how much is an expression. Languages sit on a spectrum from "everything is a statement" to "everything is an expression."

Why does this matter? Expression-oriented design gives you three things:

Composability

If if is an expression, you can use it anywhere an expression is expected — inside another expression, as a function argument, in a return position. If if is a statement, you need to introduce a temporary variable and break the composition chain.

// Rust: if is expression — composes
let items = vec.iter()
    .map(|x| if *x > 2 { x * 2 } else { *x })
    .collect::<Vec<_>>();
// C: if is statement — breaks flow
int result[3];
for (int i = 0; i < 3; i++) {
    if (arr[i] > 2) result[i] = arr[i]*2;
    else result[i] = arr[i];
}

Type Safety

Every expression position has a type. If everything is an expression, the compiler can verify the type of every sub-component. Statement-oriented code leaves gaps — an uninitialized variable between its declaration and the if-statement that assigns it is a type hole.

// Type safe: y is always initialized with a number
let y: number = x > 5 ? 10 : 20;

// Less safe: what if you forget the else branch?
let y: number;
if (x > 5) { y = 10; }
// y might be undefined here!

Referential Transparency

In a pure expression language, any expression can be replaced by its value without changing meaning. This enables equational reasoning — you can understand programs by substitution, just like algebra. Statements break this because they depend on (and modify) external state.

-- Haskell: these are interchangeable
let x = expensive() in x + x
-- is equivalent to:
expensive() + expensive()
-- (assuming expensive is pure)

3. The Formal Framework: Sorts, Judgments, and ABTs

Robert Harper's Practical Foundations for Programming Languages provides the rigorous foundation. The key idea: each syntactic category is a distinct sort, with its own judgment forms — you can't substitute an expression where a statement is expected.

3.1 Abstract Binding Trees (ABTs)

ABTs extend ASTs to properly handle variable binding and scope. A regular AST treats variable names as opaque strings. An ABT understands that let x = 5 in x + 1 has a binding site (the x after let) and a scope (the x + 1 part where that binding is visible).

// AST representation (naive):
Let("x", Num(5), Add(Var("x"), Num(1)))
// Problem: "x" is just a string. How do we know which
// "x" refers to which binding? Renaming breaks things.

// ABT representation (proper):
Let(Num(5), x. Add(Var(x), Num(1)))
// The "x." notation means "x is bound in what follows."
// Now we know exactly which binding each variable refers to.
// Renaming is safe: Let(Num(5), y. Add(Var(y), Num(1))) is
// alpha-equivalent — the same program.
Why this matters for KindScript: When KindScript walks the AST to check properties, it's implicitly working with binding structure. Knowing that a variable references a pure function (vs a mutating one) requires resolving bindings — the ABT formalism makes this precise.

3.2 Sorts and Judgments

Each syntactic category gets a distinct sort (a formal classification) and its own judgment form (a statement about well-formedness or properties):

SortWhat it isJudgment FormReads as
Exp Expressions Γ ⊢ e : τ "e has type τ in context Γ"
Cmd Statements / Commands Γ ⊢ s ok "s is well-formed in context Γ"
Decl Declarations Γ ⊢ d ⇒ Γ' "d extends environment Γ to Γ'"
Pat Patterns Γ ⊢ p : τ ⇒ Γ' "p matches type τ, binding new names"
Type Type expressions Δ ⊢ τ type "τ is a well-formed type in kind context Δ"
Mod Modules Σ ⊢ M : S "module M has signature S"
Notice: Each sort has a different kind of output. Expressions produce types. Declarations produce extended environments. Patterns produce both bindings AND match conditions. Modules produce signatures. You can't interchange them — this is what "distinct sorts" means formally.

3.3 How Judgments Compose

The power of the framework is how judgments chain together. Click each rule to see how categories feed into each other:

1 Declaration creates binding, expression uses it
Γ ⊢ e₁ : τ₁     Γ, x : τ₁ ⊢ e₂ : τ₂
Γ ⊢ let x = e₁ in e₂ : τ₂
T-Let

The declaration let x = e₁ extends the environment with x : τ₁. The expression e₂ is then checked in this extended environment. The result type is τ₂.

2 Statement narrows environment for contained expressions
Γ ⊢ e : string | number     Γ, e : string ⊢ s₁ ok     Γ, e : number ⊢ s₂ ok
Γ ⊢ if (typeof e === "string") s₁ else s₂   ok
T-If-Narrow

TypeScript's control flow narrowing in action: the if-statement produces two different environments, one per branch. Inside the true branch, e is narrowed to string. Inside the false branch, e is narrowed to number. The statement transforms the type environment differently along each path.

3 Pattern distributes type across bindings
Γ ⊢ e : { name: string, age: number }
Γ, name : string, age : number ⊢ const { name, age } = e   ok
T-Destructure

The pattern takes one type ({name: string, age: number}) and produces multiple bindings (name: string AND age: number). It's a one-to-many binding operation — the type is "distributed" across the pattern's variables.

4 Module imports link environments across files
Σ ⊢ M : { sort: (a: T[]) → T[], Config: Type }     Σ(M) = "./utils"
Γ, sort : (a: T[]) → T[], Config : Type ⊢ import { sort, Config } from "./utils"   ok
T-Import

The import declaration looks up module M's signature in the module context Σ, then extends the local type environment Γ with the imported bindings. Cross-module type checking works by composing module-level and expression-level judgments.

4. Four Ways to See the Categories

Each syntactic category can be understood through four complementary lenses. Click a lens to see how it reveals the structure:

Syntactic View — What the Grammar Says

Each category is a non-terminal in the language's grammar. The grammar defines what syntactic forms are valid for each category.

// Grammar (simplified TypeScript)

Expr  ::= number | string | id          // atoms
        | Expr + Expr | Expr * Expr     // binary ops
        | Expr(Expr, ...)                  // function call
        | Expr ? Expr : Expr              // conditional expr
        | (Params) => Expr                // arrow function

Stmt  ::= Expr ;                           // expression stmt
        | if (Expr) Block [else Block]    // conditional
        | while (Expr) Block               // loop
        | return Expr ;                     // transfer

Decl  ::= const id [: Type] = Expr ;     // value binding
        | function id (Params) Block     // function binding
        | type id = Type ;                 // type binding

Pat   ::= id                               // simple binding
        | { id, id, ... }                  // object destructure
        | [ id, ...id ]                    // array destructure

Type  ::= number | string | boolean       // primitives
        | Type | Type                        // union
        | Type & Type                        // intersection
        | (id: Type) => Type               // function type

Notice: each non-terminal (Expr, Stmt, Decl, Pat, Type) is self-contained. Expressions refer to other expressions. Types refer to other types. The categories cross-reference each other at specific points (e.g., Stmt contains Expr, Decl contains Type).

Denotational View — What Each Category Means Mathematically

In denotational semantics, we assign mathematical objects to each category via semantic functions (⟦ · ⟧):

CategoryDenotesSemantic FunctionExample
ExpressionA value⟦ e ⟧ : Env → Value⟦ 2 + 3 ⟧ = 5
StatementA state transformer⟦ s ⟧ : Store → Store⊥⟦ x = 5 ⟧(σ) = σ[x↦5]
DeclarationAn environment extension⟦ d ⟧ : Env → Env⟦ let x=5 ⟧(ρ) = ρ[x↦5]
PatternA partial destructuring⟦ p ⟧ : Value → Env⊥⟦ {a,b} ⟧({a:1,b:2}) = [a↦1,b↦2]
TypeA set of values⟦ τ ⟧ : Set(Value)⟦ number ⟧ = ℝ
ModuleA qualified environment⟦ M ⟧ : Env → Envpub⟦ module ⟧ = exported bindings

Each category maps to a fundamentally different kind of mathematical object. Values, state transformers, environment extensions, partial functions, sets, and qualified environments are all distinct. This is why you can't interchange the categories.

Operational View — How Each Category Executes

In operational semantics, we define reduction rules showing how each category steps forward:

CategoryReduction FormWhat Happens
Expressione → e' → ... → vReduces step by step to a value
Statement⟨s, σ⟩ → ⟨s', σ'⟩ → ... → ⟨skip, σₙ⟩Transforms state until done
Declaration⟨d, Γ⟩ → ⟨skip, Γ'⟩Extends environment, then done
Patternmatch(p, v) = θ or failEither produces bindings or fails
TypeNo runtime behaviorErased before execution
Module⟨M, Σ⟩ → ⟨M', Σ'⟩Links dependencies, resolves exports

Notice that types have no operational semantics — they exist only at the static level and are completely erased before execution. This is the fundamental distinction between "static" and "dynamic" categories.

KindScript View — Properties as a New Judgment Layer

KindScript adds a parallel judgment to each category — a property judgment that runs alongside the type judgment:

CategoryType Judgment (TS)Property Judgment (KSC)
ExpressionΓ ⊢ e : τΠ ⊢ e [pure, noIO]
StatementΓ ⊢ s okΠ ⊢ s [effect-free]
DeclarationΓ ⊢ d ⇒ Γ'Π ⊢ d ⇒ Π'
Type⊢ τ type(not yet — but could constrain)
ModuleΣ ⊢ M : SΣ_P ⊢ M [pure, noSideEffects]

The property environment Π transforms in parallel with the type environment Γ. When a declaration introduces a binding, it extends both Γ (with a type) and Π (with properties). When a statement narrows types, it could also narrow properties.

Π ⊢ e₁ [P₁]     Π ⊢ e₂ [P₂]
Π ⊢ e₁ + e₂ [P₁ ∩ P₂]
P-Add (properties compose via intersection)

If both operands are pure, the addition is pure. If either has side effects, the combination has side effects. Properties compose through intersection — the result has only the properties that all sub-expressions share.

Cross-category embedding examples:

↗ Expression inside Statement — if (x > 5) { ... }
The condition x > 5 is an expression embedded inside a statement. The statement uses the expression's value to decide control flow, but the statement itself doesn't produce a value. This is the most common cross-category embedding.
↗ Statement inside Expression — Rust: { let x = 5; x + 1 }
In expression-oriented languages, a block (which contains statements) IS an expression — its value is the last expression in the block. This is why Rust can use if and match as expressions. TypeScript doesn't have this — you can't use a block as an expression value.
↗ Declaration inside Expression — let x = 5 in x + 1
The let...in construct embeds a declaration inside an expression. The declaration extends the environment, and the body expression is evaluated in that extended environment. TypeScript doesn't have this exact form, but arrow functions approximate it: ((x) => x + 1)(5).
↗ Type inside Expression — x as number, x satisfies T
Type assertions and satisfies expressions embed type syntax inside expression positions. The type annotation influences what the type checker believes about the expression, but is erased at runtime.
↗ Pattern inside Declaration — const { x, y } = point;
The destructuring pattern appears in the binding position of a declaration. Instead of binding one name, the pattern binds multiple names simultaneously. The pattern "distributes" the declaration across its components.
↗ Expression inside Type — TS: typeof x in type position
TypeScript's typeof in type position takes a runtime expression and extracts its type: type T = typeof myObject. This bridges from the value world to the type world — the expression is not evaluated, but its static type is used as a type expression.

Where TypeScript Blurs the Boundaries

TypeScript inherits JavaScript's messy boundary between categories:

// 1. EXPRESSION STATEMENT: expression in statement position
console.log("hi");   // This is a CallExpression wrapped in
                       // an ExpressionStatement node

// 2. DECLARATION STATEMENT: declaration that acts as a statement
function f() {}       // FunctionDeclaration IS a statement in TS's
                       // grammar — it can appear in statement position

// 3. EXPRESSION vs DECLARATION ambiguity
function f() {}       // declaration (in statement position)
const g = function() {} // expression (in expression position)
// Same syntax, different category based on position!

// 4. CLASS: same ambiguity
class Foo {}           // declaration
const Bar = class {}   // expression

// 5. FOR-OF destructuring: statement + pattern + declaration
for (const { x, y } of points) { ... }
// for-of = statement
// const = declaration keyword
// { x, y } = pattern
// points = expression
// All four categories in one line!
Why this matters: When KindScript walks the AST, it encounters these blurred nodes constantly. A function in expression position and a function in declaration position should get the same property analysis — the category boundary is a syntactic artifact, not a semantic one.

7. Beyond the Six: Meta-Categories

There are two additional categories that sit outside the program's semantics:

Pragmas / Directives

Instructions to the compiler or runtime that don't contribute to the program's meaning. They modify how the code is processed, not what it computes.

"use strict";            // JS directive
// @ts-ignore              // TS pragma
// @ts-expect-error        // TS pragma
#pragma once              // C pragma
{-# LANGUAGE GADTs #-}    // Haskell ext
@deprecated              // decorator/annotation

For KindScript: @ksc pure annotations are pragmas! They're meta-level instructions that tell KindScript's checker what to verify, without changing the program's runtime behavior.

Comments

Carry no semantic content in most languages, but are part of the source text. TypeScript's compiler preserves comments in its AST (unusual among compilers) — this is why TSDoc works, and why KindScript could read annotations from comments.

// Single-line comment
/* Multi-line comment */
/** TSDoc comment — structured metadata */
// TODO: fix this — informal pragma

The boundary between comments and pragmas is porous. JSDoc type annotations (/** @type {number} */) are technically comments but carry semantic weight. KindScript annotations could live in either space.

Test Your Understanding

1. In TypeScript, what is const { x, y } = point;?

Correct! This is a variable declaration (const) using an object destructuring pattern ({x, y}) on the right-hand side expression (point). It's a declaration + pattern + expression all in one. The pattern creates two bindings (x and y) by deconstructing the value of point.

2. Why can Rust use if in assignment (let y = if x > 5 { 10 } else { 20 };) but TypeScript can't?

Correct! This is the expression/statement distinction in action. In Rust, if is an expression — it evaluates to a value (the value of whichever branch is taken). In TypeScript, if is a statement — it performs control flow but doesn't produce a value. TypeScript uses the ternary operator (? :) as the expression form of conditionals instead.

3. What is the formal difference between a declaration and a statement?

Correct! In formal semantics, the distinction is precise: declarations extend the environment (mapping names to meanings), producing Γ → Γ'. Statements modify the store (mapping memory locations to values), producing σ → σ'. In practice, most languages blur this — let x = 5 both declares a binding (environment) and allocates storage (store) — but the conceptual distinction is fundamental.

4. What does the typing judgment Γ ⊢ e : τ mean?

Correct! The turnstile (⊢) means "entails." Γ is the type environment — a mapping of variable names to their types (e.g., {x: number, y: string}). The whole judgment reads: "Given that the variables in scope have the types listed in Γ, the expression e is well-typed with type τ." This is a static assertion — it's verified at compile time without running the program.

5. How could KindScript assign properties to a for...of statement like for (const item of items) { process(item); }?

Correct! Just as TypeScript models statements as type environment transformers (Γ → Γ'), KindScript would model them as property environment transformers (Π → Π'). The for-of statement's properties derive from: (1) the properties of the items expression (is the iteration pure?), (2) the properties of process (is the callback pure?), and (3) the statement's own structure. If both items and process are pure, the entire for-of is effect-free.
