Variance, Assignability & Inference

The visual link — how one concept controls two different things

§1 — The One Core Idea

Variance = which way does the subtype arrow flow through a generic wrapper?

You already know that Cat is a subtype of Animal. The question variance answers is: when you wrap them in Box<Cat> and Box<Animal>, does the subtype arrow between the boxes point the same direction, the opposite direction, or neither?

This one answer simultaneously determines two things:

Variance of T in Box<T>
→
Assignability direction
→
Inference constraint direction

Same concept, two consequences. That's the whole link.

§5 — Which Positions are Covariant vs Contravariant?

Here's the complete map. Anything in an "output" position is covariant; anything in an "input" position is contravariant:

Covariant Positions (output)

  • Function return type: (): T
  • Readonly property: readonly x: T
  • Promise resolution: Promise<T>
  • Array element (read): ReadonlyArray<T>
  • Getter: get value(): T
  • Conditional true branch: T extends U ? T : never

Contravariant Positions (input)

  • Function parameter: (x: T) => void
  • Setter: set value(x: T)
  • Write-only property (rare)
  • Callback parameter: on(cb: (x: T) => void)
  • Constructor parameter: new (x: T) => Obj
  • Argument to contravariant generic
// This interface is COVARIANT in T — T only flows OUT interface Producer<out T> { get(): T; // return type → covariant map<U>(f: (x: T) => U): Producer<U>; // T in callback param... } // Wait — isn't (x: T) contravariant? Yes! But it's inside // another function parameter, so contra × contra = covariant!

§8 — Inference: Second Consequence of Variance

When TypeScript infers type arguments, variance controls which direction the constraint goes. This is the less obvious but equally important link:

The rule

Covariant position: generates T ⊇ observed_type (lower bound — T must be at least this wide)
Contravariant position: generates T ⊆ observed_type (upper bound — T must be at most this wide)
Both: generates constraints in both directions → T is pinned to specific type

Step through: infer T in function transform<T>(p: Producer<T>, c: Consumer<T>): T

Called as: transform(catProducer, animalConsumer)

§9 — The Unified Picture

Here's the complete summary. Variance is the single knob that controls both assignability direction and inference constraint direction:

Variance T appears in... Assignability: Box<Sub> → Box<Super>? Inference: constraint on T Intuition
Covariant Output positions ✓ Same direction T ⊇ observed (lower bound) "I produce T — specific is safe"
Contravariant Input positions ✗ Must flip direction T ⊆ observed (upper bound) "I consume T — general is safe"
Invariant Both positions ✗ Must be exact T = observed (pinned) "I read and write T — must match"
Bivariant Special (unsound) ✓ Either direction Weakest constraint "TS legacy — allows both"

§10 — strictFunctionTypes: When TS Gets Variance Wrong

Without strictFunctionTypes, TypeScript treats function parameters as bivariant — it allows both directions. This is unsound:

type Handler = (x: Animal) => void; const catOnly = (x: Cat) => x.purr(); // expects Cat let h: Handler = catOnly; // ✓ Allowed without strict! (bivariant) h(new Dog()); // 💥 Runtime crash — Dog has no .purr()

With strictFunctionTypes: true, parameters become properly contravariant:

let h: Handler = catOnly; // ✗ Error! Cat is not assignable to Animal // (parameter is contravariant) const anyAnimal = (x: Animal) => console.log(x); let h2: (x: Cat) => void = anyAnimal; // ✓ Safe — Animal handler accepts Cat

The reason: if you need a handler for Cats specifically, a handler that accepts any Animal is safe (it handles Cats too). But a handler that only handles Cats is NOT safe when you might pass it a Dog.
