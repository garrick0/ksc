TypeScript Indexed Access Types

How T[K] works — from syntax to compiler internals, visually explained

1. The Concept: Property Lookup at the Type Level

You already know how to look up a property on a value at runtime. An indexed access type does the exact same thing, but on types at compile time.

Runtime (values)

const person = { name: "Alice", age: 30 };

person["name"];  // "Alice" — a value

JavaScript evaluates this at runtime. The result is the value "Alice".

Compile time (types)

type Person = { name: string; age: number };

type N = Person["name"];  // string — a type

TypeScript resolves this at compile time. The result is the type string.

The syntax is identical: Thing["key"]. The difference is whether Thing is a value or a type.

3. What Can You Index?

Indexed access types work with several kinds of index types. Click each to see an example.

String Literal
Union
keyof
number (arrays)
Template Literal
type Person = { name: string; age: number; active: boolean };

type A = Person["name"];    // string
type B = Person["age"];     // number

A string literal type picks out exactly one property's type. This is the simplest case.

type Person = { name: string; age: number; active: boolean };

type C = Person["name" | "age"];   // string | number

A union index distributes: the compiler looks up each member separately and combines the results into a union. Person["name" | "age"] = Person["name"] | Person["age"] = string | number.

type Person = { name: string; age: number; active: boolean };

type AllValues = Person[keyof Person];
// = Person["name" | "age" | "active"]
// = string | number | boolean

keyof T gives you a union of all key literal types. Indexing with it gives you a union of all value types.

type Names = string[];

type Elem = Names[number];   // string

type Tuple = [string, number, boolean];
type First = Tuple[0];       // string
type Second = Tuple[1];      // number
type Any = Tuple[number];    // string | number | boolean

Arrays have a numeric index signature. number gets the element type. Tuples support specific numeric literals for positional access.

type API = {
  getUser: () => User;
  getPost: () => Post;
  deleteUser: () => void;
};

type Getters = API[`get${string}`];
// = (() => User) | (() => Post)

Template literal types can match multiple keys at once. The compiler finds all keys matching the pattern and unions their types.

4. Inside the Compiler: Three-Phase Journey

TypeScript compiles code in three phases: Parser, Binder, and Checker. Here's how Person["name"] flows through each.

5. The AST Node

The parser creates an IndexedAccessTypeNode — a simple tree with two children.

▶ How does the parser know T[] vs T[K]?

After parsing the [ bracket, the parser checks if the content is a type expression via isStartOfType():

  • If brackets are empty (T[]) → creates ArrayTypeNode
  • If brackets contain a type expression (T[K]) → creates IndexedAccessTypeNode

This is a postfix operator, so A[B][C] nests left-to-right: (A[B])[C].

6. The Checker: Resolution Algorithm

This is where all the real work happens. When the checker encounters T[K], it follows this decision tree.

7. Deferred vs Eager Resolution

The key decision the checker makes: can we resolve T[K] right now, or must we wait?

Eager (concrete types)

type Person = {
  name: string;
  age: number;
};

// Both sides concrete — resolve now
type N = Person["name"];
//    ↓ immediately resolves to: string

The checker calls getPropertyOfType(Person, "name"), finds the property symbol, returns its type: string. Done in one step.

Deferred (generic types)

function get<T, K extends keyof T>(
  obj: T, key: K
): T[K] {       // T and K are unknown!
  return obj[key];
}

// At call site, T and K become concrete:
get(person, "name");  // NOW resolves to string

The checker creates a lazy IndexedAccessType object that stores references to T and K. When the function is called with concrete types, it instantiates and resolves.

8. Distribution Rules

When indexed access types involve unions or intersections, the checker distributes — and the direction matters for soundness.

Distribution Visualizer

▶ Why does writing use intersection?

Consider this scenario:

type T = { a: string; b: number };
declare const key: "a" | "b";
declare const obj: T;

obj[key] = ???;  // what type is safe to assign?

Since key could be "a" or "b", the value must be safe for both — that's an intersection: string & number (which is never). If we used a union (string | number), we could assign a number to the string property.

This soundness fix was added in TypeScript 3.5 (PR #30769).

9. Mapped Type Interaction

When you index into a mapped type, the checker uses a shortcut: it substitutes the index directly into the template, without expanding the full mapped type.

Basic Example
How the Checker Does It
Partial<T>[K]
type Mapped = { [P in "a" | "b"]: P extends "a" ? number : string };

// Indexing into it:
type R = Mapped["a"];  // number

// The checker substitutes "a" for P in the template:
//   "a" extends "a" ? number : string  →  number

For { [P in K]: E }[X], the checker calls substituteIndexedMappedType:

1

Recognize the object type is a mapped type with template E and key variable P

2

Create a type mapper: P → X (substitute the index for the key variable)

3

Instantiate E with that mapper → result type. No need to expand the full mapped type first.

// Partial<T> is a mapped type:
type Partial<T> = { [P in keyof T]?: T[P] };

// So:
type X = Partial<Person>["name"];
//     = { [P in keyof Person]?: Person[P] }["name"]
//     → substitute P = "name" into Person[P]
//     → Person["name"]
//     → string
//     + account for ?  →  string | undefined

The ? modifier in the mapped type adds | undefined to the result. The checker handles this as part of the substitution.

11. Real-World Patterns

Indexed access types power many of TypeScript's most useful patterns.

Deep Property Access
Event Maps
Trees that Grow
type Config = {
  db: { host: string; port: number };
  cache: { ttl: number };
};

// Chain indexed access for nested types:
type Host = Config["db"]["host"];  // string
type Port = Config["db"]["port"];  // number

// The checker resolves left-to-right:
//   Config["db"]          → { host: string; port: number }
//   { ... }["host"]       → string
interface EventMap {
  click: { x: number; y: number };
  keydown: { key: string; code: number };
  resize: { width: number; height: number };
}

function on<K extends keyof EventMap>(
  event: K,
  handler: (data: EventMap[K]) => void
): void;

on("click", (data) => {
  // data is { x: number; y: number } — fully typed!
  // EventMap[K] deferred until K="click", then resolved
});
// Trees that Grow — indexed access as type families
interface Extensions {
  App: object;
  Var: object;
}

type Expr<Ext extends Extensions> =
  | AppNode<Ext>
  | VarNode<Ext>;

type AppNode<Ext extends Extensions> = {
  type: "App";
  fun: Expr<Ext>;
  arg: Expr<Ext>;
} & Ext["App"];  // ← indexed access = type family application

// Phase 1: parsed (carry source locations)
type ParsedExpr = Expr<{
  App: { loc: SrcSpan };
  Var: { loc: SrcSpan };
}>;

// Phase 2: typed (carry result types)
type TypedExpr = Expr<{
  App: { resultType: Type };
  Var: { resolvedId: Symbol };
}>;

Ext["App"] is the TypeScript equivalent of Haskell's XApp p. One type parameter, one indexed access, same pattern.

12. Test Your Understanding

Quiz 1: What does Person["name" | "age"] resolve to?

type Person = { name: string; age: number; active: boolean }

Union index types distribute: Person["name" | "age"] = Person["name"] | Person["age"] = string | number. The checker maps over each union member and combines the results.

Quiz 2: What does the binder do with an IndexedAccessTypeNode?

The binder only processes declarations that create names in scopes. T[K] is a pure type-level computation — it doesn't introduce any bindings. All resolution happens in the checker.

Quiz 3: When does the checker defer resolving T[K]?

The checker defers when shouldDeferIndexedAccessType returns true — which happens when T or K are generic (still type parameters, not yet instantiated with concrete types). It creates a lazy IndexedAccessType object that gets resolved later when the generics are instantiated.

Quiz 4: Why does writing to obj[key] (where key: "a" | "b") use intersection instead of union?

Since key could be "a" or "b" at runtime, the assigned value must be valid for both property types. That's an intersection. Using a union would allow assigning a number where a string is expected. This fix was added in TypeScript 3.5 (PR #30769 by Anders Hejlsberg).

Quiz 5: What TypeScript mechanism is the equivalent of Haskell's type families?

Indexed access types (Ext["Key"]) are the closest equivalent. Like Haskell's type families, they map a type-level key to a type-level value. With declaration merging on the interface, they can even be open (extendable from other modules), just like open type family instances in Haskell. Conditional types also work but are closed — you must enumerate all cases upfront.

13. Summary

  • Indexed access types (T[K]) are property lookups at the type level — the same as obj[key] but resolved at compile time
  • The parser creates an IndexedAccessTypeNode with objectType and indexType children
  • The binder ignores them completely — they don't create symbols or bindings
  • The checker does all the work: resolve eagerly (concrete types) or defer (generic types)
  • Union distribution: T["a" | "b"] = T["a"] | T["b"] (reading) or T["a"] & T["b"] (writing)
  • Mapped type shortcut: { [P in K]: E }[X] substitutes X for P in E directly
  • They enable the Trees that Grow pattern in TypeScript: Ext["NodeName"] = type family application
