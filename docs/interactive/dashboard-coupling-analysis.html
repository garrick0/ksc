01Current State

The dashboard is already well-isolated — but understanding why requires tracing the full dependency chain.

The KindScript dashboard is a standalone Vite + React SPA that visualizes TypeScript AST trees. It lives in apps/dashboard/. The extraction logic that bridges the pipeline to the dashboard has been split into two layers: a generic tree walker in @kindscript/core-grammar and a TS-specific extraction adapter at src/adapters/grammar/extraction/ts-ast/.

Three Composition Paths

KindScript has three distinct composition paths — evaluation, codegen, and visualization. Each pulls in different adapter subsets:

EVAL CODEGEN DASHBOARD CORE PACKAGES @kindscript/core-grammar Yes Yes Yes* @kindscript/core-evaluator Yes No No @kindscript/core-codegen No Yes No GRAMMAR ADAPTERS Grammar<TSNodeKind> Runtime Runtime Runtime AstTranslatorPort Runtime No Types ANALYSIS ADAPTERS AnalysisProjections Runtime No No AnalysisDecl (spec.ts) No Runtime No DispatchConfig (generated) Runtime No No DepGraph (generated) Runtime No No * Dashboard imports grammar metadata via adapter barrel, not via @kindscript/core-grammar directly
Key finding: The dashboard's runtime dependencies are a strict subset of the evaluator's. It needs grammar metadata to understand AST structure — nothing more. It never imports the evaluator engine, analysis projections, dispatch config, or any codegen machinery.

03Data Flow Architecture

How data moves from TypeScript source to pixels on screen.

1 TypeScript Source Code ▼

User's .ts files. This is the input to the entire pipeline. The dashboard never sees raw source — it receives pre-converted AST trees.

2 tsToAstTranslatorAdapter.convert() ▼

The AST translator converts a ts.Program into a KSTree — KindScript's internal representation. This is a runtime operation using the grammar adapter's schema to map fields.

The evaluator calls this at runtime. The dashboard only needs the result.

3 extractASTData(ksTree) — The Bridge ▼

This is the sole coupling point between the KindScript pipeline and the dashboard. It walks the KSTree using grammar.fieldDefs to understand node structure, and produces a self-contained ASTDashboardData JSON object.

After this point, the pipeline is erased — the React SPA works purely with serialized JSON.

const { fieldDefs, sumTypeMembership } = grammar;
// ^^^ The only runtime values imported from the pipeline
4 ASTDashboardData (serializable JSON) ▼

A self-contained JSON structure with zero pipeline imports:

{ version: 2,
  analysisDepth: 'parse' | 'bind' | 'check',
  schema: { fieldDefs, sumTypes },
  files: [{ fileName, lineCount, source, ast }] }

The schema is embedded directly — the SPA doesn't need to import the grammar at all.

5 React SPA — Zero Pipeline Dependencies ▼

The SPA fetches ASTDashboardData from /__data__ (Vite dev) or falls back to SAMPLE_DATA. From here:

  • D3 TreeView — recursive AST tree visualization
  • ELK Graph — force-directed layout via elkjs
  • Source Tab — syntax-highlighted code with node mapping
  • Redux-like state — DashboardProvider with context + reducer

None of these components import from src/, packages/, or any adapter.

Clean serialization boundary: ASTDashboardData is the anti-corruption layer. The SPA could be deployed as a static site consuming pre-generated JSON — it has no runtime dependency on KindScript.

04The Split: Generic Walker + TS Adapter

Extraction was split into grammar-generic logic in core-grammar and TS-specific logic in an adapter.

The original apps/dashboard/extract.ts tangled two concerns. The refactored architecture separates them cleanly:

Layer 1: serializeNode() in core-grammar

A grammar-generic function that walks any AST node tree using fieldDefs. No knowledge of TypeScript, identifiers, or compilation units.

// packages/core-grammar/src/serialize-tree.ts

export function serializeNode(
  node: ASTNode,
  fieldDefs: Record<string, readonly FieldDef[]>,
  options?: SerializeNodeOptions,
): SerializedNode

interface SerializeNodeOptions {
  getName?: (node: ASTNode) => string | undefined;  // injected by adapter
  maxTextLength?: number;                          // default: 80
  skipProps?: ReadonlySet<string>;                    // e.g., sourceText
}
The getName callback is the key. The current code hardcodes KSIdentifier.escapedText — a TS-specific concern. With the callback, the generic walker delegates name extraction to whichever adapter is calling it. A Python AST adapter would inject its own name extraction logic.

Layer 2: TS extraction adapter

A new adapter at src/adapters/grammar/extraction/ts-ast/ that knows the TS-specific details:

Identifier name extraction
TS-SPECIFIC
KSIdentifier.escapedText — injected as the getName callback to the generic walker.
Declaration file filtering
TS-SPECIFIC
cu.isDeclarationFile — skips .d.ts files during extraction.
CompilationUnit shape
TS-SPECIFIC
fileName, lineStarts, sourceText — the structure of TS compilation units.

The generic serializeNode has no concept of "named" nodes. The TS adapter provides a callback that checks if a node has a name child of kind Identifier, then reads escapedText:

function getTSNodeName(node: ASTNode): string | undefined {
  const named = node as { name?: ASTNode };
  if (named.name?.kind === 'Identifier') {
    return (named.name as KSIdentifier).escapedText;
  }
}

TypeScript projects include .d.ts declaration files. These are structural type definitions that aren't user source code. The TS adapter filters them out via cu.isDeclarationFile — a property specific to TS compilation units that wouldn't exist in other grammars.

The TS adapter knows that KSTree.root.compilationUnits is an array, and that each unit has fileName: string, lineStarts: number[], and sourceText: string. These are TS-specific node properties. A Python grammar's equivalent would have a different root shape.

05What Changed & Why

The refactoring that separated generic and TS-specific concerns.

The problem

apps/dashboard/extract.ts was the sole definition of extractASTData, but it was also consumed by examples/programmatic-api.ts, examples/showcase.ts, and test/api/export.test.ts. These non-dashboard consumers had to reach into apps/dashboard/ — a dependency direction violation. Additionally, the function tangled grammar-generic tree walking with TS-specific concerns.

The solution: split by concern

serializeNode() — packages/core-grammar/

The generic tree walker moved to core-grammar alongside the existing nodeToJSON/treeToJSON serialization utilities. It uses Grammar<K>.fieldDefs to classify children vs props — works for any grammar, not just TS AST.

What moved hereWhat stayed out
Field classification via fieldDefs (child/prop/list) KSIdentifier.escapedText name extraction
Text truncation to first line isDeclarationFile filtering
Recursive child serialization CompilationUnit shape (fileName, lineStarts)
Props collection (booleans, arrays→length, skip empty) ASTDashboardData envelope (version, schema)

New types exported: SerializedNode, SerializedFieldEntry, SerializeNodeOptions.

extractASTData() — src/adapters/grammar/extraction/ts-ast/

A new extraction adapter following the convention: src/adapters/<lib>/<port>/<target>/. It calls serializeNode() from core-grammar with TS-specific options injected.

// src/adapters/grammar/extraction/ts-ast/extract.ts

serializeNode(cu as KSNode, fieldDefs, {
  getName: getTSNodeName,        // KSIdentifier.escapedText
  skipProps: new Set(['sourceText']),  // redundant with file.source
});

The adapter also defines ASTDashboardData and ASTSchemaInfo types (moved from apps/dashboard/app/types.ts).

Q

What does serializeNode() in core-grammar do, and why does it take a getName callback?

A

serializeNode() walks any AST node tree using fieldDefs to classify children vs props, producing a SerializedNode. The getName callback exists because "how to extract a display name" is grammar-specific. For TS, it reads KSIdentifier.escapedText. A Python grammar would inject different logic. The walker itself has zero TS knowledge.

Q

Name three TS-specific concerns in extractASTData() that don't belong in the generic walker.

A

1. KSIdentifier.escapedText for display names (injected via getName callback).
2. cu.isDeclarationFile filtering (skips .d.ts files).
3. CompilationUnit shape: fileName, lineStarts.length, sourceText.
All three depend on knowledge of the TS AST grammar's concrete node types.

Q

Does the React SPA (apps/dashboard/app/) import anything from the KindScript pipeline?

A

No. The React SPA has zero direct imports from src/ or packages/. Its types.ts re-exports SerializedNode as ASTNode from @kindscript/core-grammar and ASTDashboardData from the extraction adapter, but all SPA components import from the local types.ts re-export — they don't know where the types originate.

Q

Are the dashboard's dependencies a strict subset of the evaluator's?

A

Yes. The evaluator needs grammar + translator (runtime) + analysis projections + dispatch + depGraph + core-evaluator. The extraction adapter needs only grammar metadata + serializeNode() from core-grammar. The dashboard SPA needs nothing from the pipeline at all — it consumes serialized JSON.

Q

Why was it wrong for extractASTData to live in apps/dashboard/?

A

Because non-dashboard consumers needed it: examples/programmatic-api.ts, examples/showcase.ts, and test/api/export.test.ts all imported from apps/dashboard/extract.js. This violated the dependency direction rule — application code shouldn't reach into an app shell. The function is a shared extraction use case, not a dashboard concern.

Q

Where does ASTDashboardData live now, and where does SerializedNode live?

A

SerializedNode lives in @kindscript/core-grammar (it's the grammar-generic output of serializeNode()). ASTDashboardData lives in the TS extraction adapter (src/adapters/grammar/extraction/ts-ast/types.ts) because it's a TS-specific envelope (references analysisDepth, CompilationUnit fields). The dashboard's app/types.ts re-exports both under their original names.

Q

Name the three composition paths in KindScript and what core packages each requires.

A

Evaluation: core-grammar + core-evaluator (runs attribute grammars over ASTs).
Codegen: core-grammar + core-codegen (compiles AnalysisDecl into dispatch tables).
Extraction: core-grammar only (serializes trees using serializeNode() + grammar metadata).
Each path is isolated — ksc check never loads codegen, ksc codegen never creates the evaluator, and extraction never touches either.
