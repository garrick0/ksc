03Antipattern Deep-Dives

Click each card to expand the analysis with removal options. Select an option to see its trade-offs.

AP-1 buildDepGraph() called at runtime medium ▼

What happens

// app/user-api/lib/program.ts:27
import { buildDepGraph } from '../../../analysis/index.js';

// Line 37 — runs once at module load
const depGraph = buildDepGraph(analysisSpec.attrs);

This computes a topological sort of attribute dependencies by walking .deps metadata on equation functions. The identical computation already runs at codegen time in compile.ts:461. The result is deterministic from the spec — it never changes between codegen and runtime.

Why it's wrong

The dep graph is a static property of the analysis spec. Computing it at runtime is like re-parsing your grammar at startup. Worse, importing buildDepGraph from the analysis barrel pulls compile.ts (487 lines of codegen machinery) into the runtime module graph.

Removal Options

A: Generate static data
Emit dep-graph.ts as a third generated file alongside dispatch.ts and attr-types.ts. Export the dep graph as a const. Program.ts imports the static data.
One more generated file, but zero runtime computation. compile.ts exits the runtime bundle entirely.
B: Extract to own module
Move buildDepGraph to analysis/dep-graph.ts (standalone, no compile.ts import). Keep runtime computation but isolate bundle impact.
Still computes at runtime, but no longer pulls 487 lines of codegen code. Partial fix.
C: Remove from public API
If getAttributeDepGraph() is only for debugging/visualization, remove it from KSProgramInterface entirely. No dep graph needed at runtime.
Loses functionality. Users currently use this for dep graph visualization in the dashboard.
AP-2 pivotToAttrCentric() executes at module load low ▼

What happens

// specs/ts-ast/kind-checking/spec.ts:56-68
const pivoted = pivotToAttrCentric<TSKind>({
  CompilationUnit: CompilationUnitEquations,
  VariableDeclaration: VariableDeclarationEquations,
  Identifier: IdentifierEquations,
  // ... 8 more per-kind objects
});

This reshapes 11 per-kind equation objects into per-attr equation maps. It runs whenever spec.ts is imported — both at codegen time (needed) AND at runtime (wasted). At runtime, the evaluator uses dispatch.ts, not these raw attr declarations.

Why it's wrong

It's wasted work at runtime. The pivot result feeds into analysisSpec.attrs, which at runtime is only read by buildDepGraph() (AP-1) and validateDispatch() (AP-3). Neither of those should run at runtime either.

Removal Options

A: Split spec exports
Export a runtimeSpec with only { projections, setup }. The full analysisSpec is imported only by codegen roots. Program.ts imports the slim version.
Breaks the clean AnalysisSpec port abstraction — wireEvaluator would need a different signature. But perfectly clean separation.
B: Lazy init
Make pivotToAttrCentric lazy — only computed when .attrs is first accessed via a getter.
Adds complexity for marginal gain. The pivot is ~33 lines of simple reshaping on 11 objects.
C: Accept the cost
It's a small computation on 11 objects. The real issue is conceptual pollution, not performance.
No change required, but the architecture stays impure. Blocks future refactors that assume clean separation.
AP-3 validateDispatch() runs at startup low ▼

What happens

// evaluator/engine.ts:259-260 (inside wireEvaluator)
validateDispatch(opts.dispatch, opts.spec.attrs.map(a => a.name));
return createEvaluator<K, M, P>({ ... });

This iterates over attr names and dispatch keys to verify they match. It runs once at module load time (when wireEvaluator is called in program.ts:30).

Why it's wrong

The dispatch.ts file is generated from the spec. Any mismatch means codegen is broken — this should be caught at codegen time, not at runtime. It's like checksumming your compiler output at startup. Furthermore, this is the reason wireEvaluator needs the full spec (to read spec.attrs).

Removal Options

A: Move to codegen pipeline
Add a validation step in runCodegenPipeline() that verifies the generated dispatch config matches the spec. Runs once at build, never at runtime.
No runtime safety net. But codegen already validates exhaustiveness, kind refs, and function names.
B: TypeScript types
Make DispatchConfig generic: DispatchConfig<A extends string> keyed by attr names. TypeScript catches mismatches at compile time.
More complex generic types, but zero runtime cost. Requires generated dispatch to use the parameterized type.
C: Keep as defense-in-depth
It runs once, catches broken codegen before evaluation starts. In a young system, this safety net has value.
Accepting impurity for safety. Requires keeping the full spec import at runtime (blocks AP-4 fix).
AP-4 Full analysisSpec imported at runtime high ▼

What happens

// app/user-api/lib/program.ts:22
import { analysisSpec } from '../../../specs/ts-ast/kind-checking/spec.js';

// What wireEvaluator actually reads (engine.ts:264-265):
projections: opts.spec.projections,  // 2 functions
setup: opts.spec.setup,              // 1 function

// What the import also pulls in (unused at runtime):
//   - 8 AttrDecl objects with all equation function refs
//   - 16 equation function refs (with .deps metadata)
//   - 11 per-kind equation objects from pivotToAttrCentric
//   - code() helper, CodeLiteral values
//   - withDeps() metadata on all functions

Why it's wrong

This is the root cause antipattern. At runtime, wireEvaluator only extracts spec.projections (2 functions) and spec.setup (1 function). The remaining ~95% of the import — all attr declarations, equation references, dependency metadata — is dead weight. The generated dispatch.ts already imports the equation functions it needs independently.

Chain reaction: This import triggers AP-2 (pivotToAttrCentric runs at load), provides the data for AP-1 (buildDepGraph reads spec.attrs), and is required by AP-3 (validateDispatch reads spec.attrs). Fix this and AP-1/2/3 become easier or unnecessary.

Removal Options

A: Runtime-only export
Export runtimeSpec = { projections, setup } from spec.ts. Program.ts imports only that. The full spec is only imported by codegen roots.
Two exports from spec.ts. Clean separation. wireEvaluator signature must accept the slim shape instead of full AnalysisSpec.
B: Pass directly to createEvaluator
Skip wireEvaluator entirely. Have program.ts call createEvaluator({ dispatch, grammar, projections, setup }) directly, importing each piece separately.
Slightly more verbose composition root. wireEvaluator becomes codegen-only validation helper. Clean elimination of the spec dependency.
C: Generate projections into dispatch.ts
Include projection functions in the generated output. Codegen already has access to the spec. Generated file imports and re-exports them.
Projections are hand-written — they'd need to be imported into the generated file, adding import complexity. But spec.ts is no longer needed at runtime.
AP-5 compile.ts in the runtime module graph medium ▼

What happens

// analysis/index.ts:34 — barrel re-export
export { compileAnalysis, buildDepGraph, validateSpecConsistency } from './compile.js';

// app/user-api/lib/program.ts:27
import { buildDepGraph } from '../../../analysis/index.js';
// ↑ This import loads the entire 487-line compile.ts module

Because program.ts imports buildDepGraph from the analysis barrel, the barrel's re-export of compile.ts loads the entire codegen module into the runtime bundle: compileAnalysis(), emitExpr(), generateSynDispatch(), all the code generation functions.

Why it's wrong

In Node.js ESM, barrel re-exports load the entire source module regardless of which export you use. Tree shaking is not guaranteed. This violates the ports-and-adapters separation: runtime code now depends on codegen machinery.

Removal Options

A: Fix AP-1 (generate dep graph)
If the dep graph is static data in a generated file, program.ts never imports from analysis/ at all. compile.ts exits the runtime bundle entirely.
Requires solving AP-1 first. But eliminates the problem at the source.
B: Separate barrels
Create analysis/runtime.ts (types + ctx only) and analysis/codegen.ts (compile + validate + pivot). Program.ts imports from the runtime barrel.
Two barrels. Cleaner import graph, but more files to maintain. compile.ts stays out of runtime.
C: Direct module import
Instead of importing from the barrel, program.ts imports directly: import { buildDepGraph } from '../../../analysis/compile.js'
Breaks the barrel convention. Still loads compile.ts. Only helps if bundler can tree-shake the module.

04Dependency Chain

How the antipatterns chain together. Each import triggers the next.

program.ts imports analysisSpec from specs/.../spec.ts AP-4
spec.ts runs pivotToAttrCentric() at module load AP-2
spec.ts loads all equation functions + withDeps() metadata
program.ts imports buildDepGraph from analysis/index.ts AP-1
analysis/index.ts re-exports from compile.ts — 487 lines loaded AP-5
wireEvaluator() calls validateDispatch(spec.attrs) AP-3
reads spec.attrs — the only reason wireEvaluator needs the full spec
After fix: program.ts imports only { projections, setup } + dispatchConfig + grammar clean
Key insight: The entire chain collapses if program.ts stops importing the full analysisSpec. AP-4 is the keystone. Fix it and AP-1/2/3/5 either resolve automatically or become trivially fixable.

05Recommended Removal Order

Prioritized by impact and dependency order. Each step unlocks the next.

  • 1
    AP-4: Split the spec export (root cause)
    Export runtimeSpec = { projections, setup } from spec.ts. Change wireEvaluator or use createEvaluator directly. Program.ts stops importing the full spec.
    ~20 LOC changed across 3 files. Eliminates ~95% of dead runtime imports.
  • 2
    AP-1: Generate dep graph as static data
    Add dep-graph.ts to codegen output. Emit the dep graph computed by buildDepGraphFromAttrs() as a const export. Program.ts imports it statically.
    ~40 LOC in compile.ts + pipeline.ts. Program.ts import changes to generated file.
  • 3
    AP-5: Auto-resolved
    Once AP-1 is done, program.ts no longer imports from analysis/ at all. compile.ts exits the runtime module graph. No code changes needed beyond steps 1-2.
    0 LOC. Free side-effect of step 2.
  • 4
    AP-3: Move validation to codegen
    Move validateDispatch() into runCodegenPipeline(). Or make DispatchConfig generic for compile-time checking. Remove from wireEvaluator.
    ~15 LOC moved. Optional — once AP-4 removes the spec import, this becomes a minor cleanup.
  • 5
    AP-2: Auto-resolved
    Once program.ts doesn't import the full spec, pivotToAttrCentric() only runs at codegen time (when codegen scripts import spec.ts). No change needed.
    0 LOC. Free side-effect of step 1.
Total impact: ~75 LOC changed. 3 of 5 antipatterns resolve as free side-effects. The runtime bundle loses ~500+ lines of dead codegen machinery, all withDeps metadata, and all attr declaration overhead. The evaluation pipeline becomes: dispatchConfig + grammar + projections + setup — exactly what it needs, nothing more.

Q

What two files does the KSC codegen pipeline produce?

A

dispatch.ts (per-attribute switch/case routing functions + dispatch config) and attr-types.ts (TypeScript interface mapping non-parameterized attribute names to their types). These are the only generated files — there is no grammar codegen.

Q

Why is buildDepGraph() at runtime an antipattern?

A

The dep graph is 100% deterministic from the spec — it's computed identically at codegen time. Running it again at runtime is redundant. Worse, importing buildDepGraph from the analysis barrel pulls the entire 487-line compile.ts module into the runtime bundle.

Q

What does wireEvaluator() actually need from the analysisSpec?

A

Only spec.projections (2 functions: projectDefinitions and a diagnostics extractor) and spec.setup (1 function: resetCounter). The remaining ~95% of the spec — all 8 AttrDecl objects, 16 equation refs, dependency metadata — is dead weight at runtime.

Q

Which antipattern is the "root cause" and why?

A

AP-4 (full analysisSpec import). It triggers AP-2 (pivotToAttrCentric runs at load), provides data for AP-1 (buildDepGraph reads spec.attrs), is required by AP-3 (validateDispatch reads spec.attrs), and its barrel import causes AP-5. Fixing it resolves or simplifies all other antipatterns.

Q

What's the difference between the dispatch routing (legitimate runtime) and the dep graph computation (antipattern)?

A

Dispatch routing (the generated switch/case functions) produces different results per node — each AST node has a different kind, so the dispatch must happen at runtime. The dep graph is a fixed property of the spec — it's identical regardless of input, so it should be precomputed at build time.

Q

Why is validateDispatch() at runtime questionable?

A

Because dispatch.ts is generated from the spec. Any mismatch between dispatch config keys and spec attr names would mean codegen is broken — a bug that should be caught by the codegen pipeline, not discovered at user runtime. It's like checksumming compiler output every time you run the program.

Q

After applying all 5 fixes, what would program.ts import at runtime?

A

Only 4 things: (1) grammar from specs/.../grammar/ (for tree building), (2) dispatchConfig from generated/.../dispatch.ts, (3) projections and setup from a slim runtime export, (4) depGraph from a new generated static file. Zero codegen machinery, zero dead attr declarations.
