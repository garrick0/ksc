01KSC — Current System

KSC uses explicit dependency declaration via withDeps() on equation functions, combined with lazy on-demand evaluation in generated code. Dependencies are declared statically, validated at codegen time, but resolved dynamically at runtime through the attribute call chain.

The Kind-Checking Dependency Chain

The 8 attributes form a linear dependency chain (plus one independent attribute):

kindDefs → defEnv → defLookup → kindAnnotations → contextFor(p) → violationFor(p) → allViolations nodeCount (independent)

Legend: syn = synthesized • inh = inherited • collection = fold • dashed = parameterized

How It Works: Four Stages

1 Declare Dependencies ▼

Equation functions attach dependency metadata via withDeps(deps, fn). This mutates the function to add a .deps: string[] property naming the attributes it reads.

export const eq_defEnv_root = withDeps(['kindDefs'],
  function eq_defEnv_root(ctx: Ctx): Map<string, KindDefinition> {
    for (const cuCtx of ctx.children) {
      for (const def of cuCtx.attr('kindDefs')) {
        map.set(def.name, def);
      }
    }
    return map;
  }
);

Every equation function across the spec uses this pattern. The deps array names attribute-level dependencies, not node-level or function-level ones.

2 Collect & Validate ▼

collectDepsForAttr(attr) walks all function-typed values in an AttrDecl, reads .deps from each, and returns the union. Then validateSpec() checks every dep name corresponds to an actual attribute in the spec.

// collectDepsForAttr walks direction-specific fields:
switch (attr.direction) {
  case 'syn':  addFromValue(attr.default); addFromRecord(attr.equations);
  case 'inh':  addFromValue(attr.rootValue); addFromRecord(attr.parentEquations);
  case 'collection': addFromValue(attr.init);
}

Validation catches typos like withDeps(['kinDefs'], ...) at codegen time — before any user code runs.

3 Build Dep Graph & Topo Sort ▼

buildDepGraphFromAttrs() in analysis/compile.ts collects all edges as [dependent, dependency] pairs, then runs DFS post-order topological sort. Cycles are detected when a node is revisited while still on the DFS stack.

// Generated static dep graph (published for tooling):
const KSC_STATIC_DEP_GRAPH = {
  attributes: ['kindDefs', 'defEnv', 'defLookup', ...],
  edges: [
    ['defEnv', 'kindDefs'],
    ['defLookup', 'defEnv'],
    ['kindAnnotations', 'defLookup'],
    // ...
  ],
  order: ['kindDefs', 'defEnv', 'defLookup', ...],
};

The topological order is published in the generated evaluator for visualization/tooling, but not used at runtime — lazy evaluation determines the actual order.

4 Lazy Evaluation with Caching ▼

The generated evaluator produces one method per attribute on KSCDNode. Each method follows the same pattern:

violationFor(property: string): Diagnostic | null {
  if (this._pc_violationFor.has(property)) return ...;  // 1. cache hit
  if (this._cyc.has('violationFor:' + property))       // 2. circular guard
    throw new Error(`Circular attribute access`);
  this._cyc.add('violationFor:' + property);
  try {
    // 3. evaluate (triggers contextFor → kindAnnotations → ...)
    const _r = eq_violationFor_Identifier(this, property);
    this._pc_violationFor.set(property, _r);             // 4. cache result
    return _r;
  } finally { this._cyc.delete('violationFor:' + property); }
}

When allViolations is accessed on the root, the entire chain evaluates on demand: allViolations → violationFor → contextFor → kindAnnotations → defLookup → defEnv → kindDefs. Each result is cached per-node so subsequent accesses are O(1).

Key insight: KSC's approach is "explicit deps, implicit scheduling." The withDeps() calls declare what each equation needs, but lazy evaluation determines when things are computed. The static dep graph exists for validation and tooling, not for scheduling.

02JastAdd — Reference Attribute Grammars

JastAdd is the most mature AG system in active use. It takes a fully dynamic, demand-driven approach — dependencies are implicit in equations, discovered at runtime, and there is no static scheduling at all.

Demand-Driven Evaluation

JastAdd compiles AG specs to Java classes. Attributes become Java methods with automatic memoization. When you call an attribute method:

  1. Check cache → return if hit
  2. Evaluate equation → may trigger other attribute evaluations
  3. Cache result → return

No dependency declaration needed. No evaluation scheduling. Dependencies are discovered dynamically as equations call other attribute methods. This is essentially the same lazy model as KSC, but with zero explicit dep metadata.

Key insight: JastAdd proves that explicit dep declarations (like withDeps()) are not required for correct lazy evaluation. KSC's withDeps() exists for validation and tooling, not for scheduling.

Reference Attributes — The Key Innovation

In JastAdd, attribute values can be references to other AST nodes. This turns the AST into a semantic graph:

Name Resolution

A parameterized inherited attribute lookup("x") traverses scope chains via parent references. Use nodes link directly to declaration nodes.

Type Checking

An expression's type() attribute returns a reference to a type declaration node. Subtype checks follow the reference graph.

Non-Local Access

Once you have a reference to a distant node, you can access any of its synthesized attributes — no need to thread values through the whole tree.

KSC comparison: KSC handles non-local data via inherited attributes with copy-down (e.g., defEnv propagated from root to all nodes). Reference attributes would eliminate the need to copy-down large environments — nodes would follow a reference to look up what they need.

Circular Attributes — Fixed-Point Iteration

JastAdd uniquely supports circular attribute equations that reference themselves. The system iterates until a fixed point:

// Circular attribute with lattice-based iteration
syn Set<Variable> Block.liveVars circular [new HashSet<>()];  // bottom value
eq Block.liveVars() = use().union(
  succ().liveVars().minus(def())  // self-referential!
);

Termination guaranteed when:

  • Values form a lattice of finite height
  • Equations are monotonic (values only go up)

This enables data-flow analyses (reaching definitions, liveness, available expressions) to be expressed as attribute equations — no separate worklist algorithm needed.

KSC gap: KSC throws on circularity (both at codegen via topo sort and at runtime via _cyc guards). There is no fixed-point iteration support.

Collection Attributes — Survey + Collect

JastAdd collection attributes gather contributions from arbitrary nodes anywhere in the tree, not just direct children. Evaluation uses two phases:

  1. Survey phase: traverse the tree to find all contributing nodes
  2. Collection phase: evaluate and combine contributions

KSC comparison: KSC collection attributes only fold over direct children (for (const child of this.children) { result = combine(result, child.nodeCount()) }). JastAdd's model is more powerful — any node can contribute to a collection attribute on any other node.

03Silver — Static Well-Definedness

Silver (from the MELT group at U of Minnesota) adds compile-time static analysis on top of demand-driven evaluation. It can statically prove properties that JastAdd and KSC only check at runtime.

The Key Innovation: Modular Well-Definedness Analysis

Silver statically verifies two critical properties per module:

Completeness

Every attribute has a defining equation in every context where it's needed. No "missing equation" runtime errors.

Non-Circularity

No circular dependencies exist among non-circular attributes. Guaranteed at compile time, not just detected at runtime.

These checks are modular — each grammar module is verified using only its declared dependencies. Independently developed language extensions can be composed safely without rechecking the whole grammar.

Key insight: Silver trades expressiveness for safety. By disallowing circular attributes entirely, it can provide static guarantees that JastAdd and KSC cannot. KSC sits between them: it does static topo-sort cycle checking (like Silver) but doesn't verify completeness.

Flow Types

Silver's static analysis is powered by flow types: a mapping from each nonterminal showing which inherited attributes each synthesized attribute may depend on.

-- Silver builds a flow graph per production:
-- "For production P, synthesized attr S depends on inherited attrs {I1, I2}"
-- These are stitched into a complete dependency graph
-- and checked for circularity + completeness

KSC comparison: KSC's withDeps() serves a similar purpose to flow types — both declare what an attribute depends on. But Silver's flow types are automatically inferred from equations, while KSC's deps are manually declared. Silver verifies them exhaustively; KSC only checks that dep names exist.

Forwarding — Safe Modular Extension

Forwarding lets a production specify a "semantically equivalent" tree that provides default attribute values for any attributes not explicitly defined:

abstract production forEach
top::Stmt ::= iter::Name coll::Expr body::Stmt
{
  forwards to whileLoop(...);  -- desugar to while loop
  -- Any attribute not defined here is computed on the while loop
  -- Inherited attrs automatically copied to forwarded tree
}

This enables independently developed extensions to compose: one extension adds new attributes, another adds new productions with forwarding. Neither needs to know about the other.

KSC comparison: KSC has no forwarding mechanism. New node kinds must have equations defined for every attribute, or rely on the default clause.

Higher-Order Attributes

In Silver, attributes can have nonterminal types — their values are entire trees:

Undecorated (Higher-Order)

A bare tree without inherited attrs supplied. Used for transformations: concrete-to-abstract syntax, desugaring.

Decorated (Reference)

A tree with inherited attrs already supplied. Functions like JastAdd's reference attributes — provides a semantic link to another node.

Higher-order attributes enable multi-level compilation within the AG framework: parse tree → desugared tree → IR, all as attribute equations.

04Facebook Infer — Compositional Analysis

Infer takes a fundamentally different approach. Instead of tree-based attribute evaluation, it uses procedure-level compositional analysis with summaries. The unit of analysis is a function/method, not an AST node.

Bottom-Up Procedure Summaries

Infer's analysis pipeline:

  1. Capture: Translate source to Infer's IR (SIL)
  2. Analysis: Build call graph → analyze bottom-up (callees before callers)
  3. Report: Emit diagnostics

Each procedure produces a summary (pre/post specification via bi-abduction). Callers read callee summaries — never re-analyze callees. This is the key to scaling to 100M+ LOC at Meta.

Key difference from AG systems: AG attributes are computed per-node on a single AST. Infer summaries are computed per-procedure across the whole program's call graph. The "tree" is the call graph, and "attributes" are procedure summaries.

Scheduling Strategies

Callgraph (default)

One job per procedure, following syntactic call graph bottom-up. Usually fastest.

File-level

One job per file. Coarser granularity, simpler scheduling.

Restart

One job per procedure with locking to avoid duplicate work across parallel workers.

For mutually recursive functions, Infer iterates analysis until summaries stabilize — essentially fixed-point iteration on the call graph, analogous to JastAdd's circular attributes.

Multi-Checker Dependencies

Infer runs multiple checkers (analyses) simultaneously. Checkers declare dependencies through:

  • Payload fields: Payloads.Fields.<checker_name> to read another checker's summaries
  • Registration order: list order in registerCheckers.ml determines execution sequence
  • Interprocedural callbacks: configure which payload fields a checker reads

This is somewhat manual and fragile compared to AG dep tracking, but it works at massive scale.

Trade-off: Infer's ordering-based dependency system in registerCheckers.ml is analogous to KSC's composition roots — both are manual composition points where the developer wires analyses together. Neither system automatically discovers inter-analysis dependencies.

06Evaluation Models Compared

The fundamental question: how do you decide which attribute to compute next?

Lazy / Demand-Driven

Used by: KSC, JastAdd, Silver (at runtime)

No scheduling at all. The caller drives evaluation: when you access an attribute, it evaluates its equation, which may trigger other attribute evaluations. Caching ensures each attribute per node is computed at most once.

// Pseudocode for lazy evaluation
attr(name) {
  if (cache[name]) return cache[name];     // already computed
  if (computing[name]) throw "circular"; // cycle detection
  computing[name] = true;
  cache[name] = equation[name]();              // evaluate — may trigger other attrs
  computing[name] = false;
  return cache[name];
}

Pros: Only computes what's needed. No scheduling complexity. Natural for tree-structured data.
Cons: Stack depth proportional to dep chain length. No parallelism within a single tree. Hard to predict evaluation order.

Static Schedule

Used by: Classical AG systems (Knuth's original), Silver (for verification)

Analyze the dependency graph at compile time, compute a fixed evaluation order (e.g., topological sort), then execute passes in that order. Each pass computes one "level" of attributes across all nodes.

// Pseudocode for static multi-pass
for (const pass of scheduledOrder) {
  for (const node of traversal(tree, pass.direction)) {
    node.attrs[pass.attr] = pass.equation(node);
  }
}

Pros: Predictable. Enables parallelism (all nodes in a pass are independent). Static error checking.
Cons: May compute unnecessary attributes. Requires global analysis. Less flexible for incremental updates.

Key insight: KSC publishes a topological order (KSC_STATIC_DEP_GRAPH.order) but doesn't use it for scheduling. This is the bridge point — KSC could switch to pass-based evaluation using the graph it already computes.

Bottom-Up Composition

Used by: Facebook Infer

Build a dependency graph (call graph), topologically sort it, analyze each node (procedure) bottom-up. Results are summaries that abstract away implementation details.

// Pseudocode for bottom-up analysis
for (const proc of topoSort(callGraph)) {
  const calleeSummaries = proc.callees.map(c => summaries[c]);
  summaries[proc] = analyze(proc, calleeSummaries);
}

Pros: Scales massively (parallel workers). Natural incrementality (re-analyze only changed procedures). Clear abstraction boundaries.
Cons: Summary precision limits. Not natural for tree-structured data. Information loss at abstraction boundaries.

Fixed-Point Iteration

Used by: JastAdd (circular attrs), Infer (mutual recursion), data-flow frameworks

When dependencies are circular, iterate until values stabilize. Requires a lattice (finite height + monotonic equations) to guarantee termination.

// Pseudocode for fixed-point iteration
for (const attr of circularAttrs) {
  node.attrs[attr] = bottom;  // start at bottom of lattice
}
do {
  changed = false;
  for (const attr of circularAttrs) {
    const newVal = attr.equation(node);
    if (newVal !== node.attrs[attr]) {
      node.attrs[attr] = newVal;
      changed = true;
    }
  }
} while (changed);

Pros: Handles mutual dependencies. Enables data-flow analyses. Mathematically well-founded.
Cons: Convergence speed depends on lattice height. Hard to debug. Requires careful lattice design.

07Improvement & Alignment Opportunities

Based on the comparison, here are concrete opportunities for KSC, ordered from most practical to most ambitious.

1. Completeness Checking from Silver
▼
Verify that every attribute has an equation (or default) for every node kind at codegen time.

Current state: If a node kind triggers a syn attribute with no matching equation and no default, the generated switch statement falls through to the default case. If the default returns null or an empty value, this may silently produce wrong results.

What Silver does: Statically proves completeness per module — every attribute has a defining equation in every context.

Implementation for KSC: Cross-reference attr.equations keys against the grammar's node kinds at codegen time. Warn (or error) when a syn attr has no equation for a node kind and the default seems insufficient. The grammar is already available in the composition root.

Effort: Low. KSC already has the grammar node list and attr equations in scope during compilation.

2. Dep Accuracy Verification from Silver
▼
Verify that withDeps() declarations are accurate — not under- or over-declared.

Current state: withDeps() is trusted. If an equation calls ctx.attr('defEnv') but doesn't declare 'defEnv' in its deps, the dep graph is incomplete. If it declares deps it doesn't use, the graph is over-constrained.

What Silver does: Auto-infers flow types from equations — no manual declaration needed.

Implementation for KSC: A runtime verification mode that intercepts ctx.attr() calls during evaluation and compares actual accesses against declared deps. Run during tests to catch mismatches. Alternatively, static analysis of equation function bodies (harder in TS due to dynamic ctx.attr(name)).

Effort: Medium. Runtime mode is straightforward; static analysis is harder.

3. Richer Collection Attributes from JastAdd
▼
Allow contributions from arbitrary nodes, not just direct children.

Current state: KSC collection attrs fold over direct children only: for (const child of this.children) { result = combine(result, child.attr()) }.

What JastAdd does: Any node can contribute to a collection attr on any other node. Uses a survey phase to find all contributors.

Implementation for KSC: Add a contributions field to CollectionAttr — equation functions that select which nodes contribute and what they contribute. The generated evaluator would add a survey traversal before the collection fold.

Effort: Medium. Requires extending AttrDecl, compile.ts codegen, and generated evaluator pattern.

4. Reference Attributes from JastAdd
▼
Allow attribute values to be references to other AST nodes, enabling direct non-local access.

Current state: Non-local data flows via inherited attributes with copy-down. defEnv is a Map copied from root to every node. Nodes look up definitions from the map, not from a reference to the definition site.

What JastAdd does: lookup("x") returns a reference to the declaration node. The use node then directly accesses the declaration's synthesized attributes.

Implementation for KSC: Not a new attr direction — synthesized attrs can already return KSCDNode values. The change is in the equations: instead of returning data, return a node reference. May need generated type support for KSCDNode references in attr type signatures.

Effort: Low-Medium. Conceptually simple (just return node references), but may need type system support in the generated evaluator.

5. Circular / Fixed-Point Attributes from JastAdd
▼
Support lattice-based fixed-point iteration for self-referential attribute equations.

Current state: Circularity throws at both codegen time (topo sort) and runtime (_cyc guard).

What JastAdd does: Circular attrs have a bottom value and iterate their equation until convergence.

Implementation for KSC: New attr direction 'circular' with bottom: AttrExpr and equation: Function. Generated method: set to bottom, evaluate, compare, repeat until stable. Topo sort must allow circular attrs in SCCs. The spec must guarantee monotonicity and finite lattice height.

Effort: High. New direction, new codegen pattern, SCC handling in dep graph, and the spec author must ensure lattice properties.

6. Multi-Analysis Composition from Infer
▼
Allow one analysis to depend on another analysis's results (e.g., type-checking depends on kind-checking).

Current state: Each analysis is independent. Separate composition roots, no inter-analysis dependencies. If a type-checking analysis needs kind-checking results, the user must manually wire them together.

What Infer does: Checkers read other checkers' payload fields. Registration order determines execution sequence.

Implementation for KSC: An AnalysisSpec could declare dependencies on other analyses' projections. The composition root would evaluate analyses in dep order, passing projection results forward. The generated evaluator for analysis B would receive analysis A's results as an inherited attribute or constructor parameter.

Effort: High. Requires new composition patterns, cross-analysis type sharing, and careful API design.

7. Incremental Evaluation from all systems
▼
Cache results across evaluations, only recompute attributes on changed subtrees.

Current state: Full re-evaluation on each evaluate() call. resetCounter() + buildKSCTree() rebuilds everything from scratch.

What the research says: JastAdd has published work on incremental RAGs with dynamic dependency tracking. Infer's incremental mode re-analyzes only changed procedures. Silver's static analysis enables determining which attributes are affected by a change.

Implementation for KSC: KSC's explicit dep graph (KSC_STATIC_DEP_GRAPH) is the foundation. Track which AST nodes changed, use the dep graph to determine which cached attributes to invalidate, re-evaluate only those. Requires stable node identity across edits.

Effort: Very high. Requires stable node identity, change tracking, cache invalidation, and careful handling of inherited attributes (a change to root kindDefs invalidates defEnv everywhere).

Q

How does KSC declare attribute dependencies, and what are they used for?

A

withDeps(deps, fn) attaches a .deps: string[] array to equation functions. These deps are used for (1) validation at codegen time (validateSpec checks names exist), (2) building the static dep graph (buildDepGraphFromAttrs), and (3) cycle detection via topological sort. They are not used for scheduling — lazy evaluation handles that.

Q

How does JastAdd determine attribute evaluation order without explicit dependency declarations?

A

Purely demand-driven: when you access an attribute, its equation executes, which may call other attributes. Dependencies are discovered dynamically at runtime. Caching ensures each attribute per node is computed at most once. There is no static schedule — the call stack is the schedule.

Q

What is Silver's "modular well-definedness analysis" and what two properties does it verify?

A

A compile-time static analysis that verifies (1) completeness — every attribute has a defining equation in every required context, and (2) non-circularity — no circular dependencies among non-circular attributes. It's checked modularly, so independently developed extensions compose safely.

Q

What can JastAdd do with circular attributes that KSC cannot, and why does it matter?

A

JastAdd supports fixed-point iteration on circular attribute equations: starting from a bottom value, it iterates until convergence on a finite-height lattice with monotonic equations. This enables data-flow analyses (liveness, reaching definitions) to be expressed directly as attribute equations — no separate worklist algorithm needed. KSC throws on circularity.

Q

How does Facebook Infer achieve scalability to 100M+ lines of code?

A

Compositional analysis: each procedure is analyzed independently, producing a summary (pre/post spec via bi-abduction). Callers read callee summaries, never re-analyze callees. This enables (1) parallel workers analyzing different procedures, (2) incremental re-analysis of only changed procedures, and (3) clear abstraction boundaries that prevent analysis blowup.

Q

What are reference attributes, and how do they differ from KSC's approach to non-local data?

A

Reference attributes (JastAdd/Silver) are attributes whose values are references to other AST nodes. A use node's lookup("x") returns a reference to the declaration node, enabling direct access to its attrs. KSC instead uses inherited attrs with copy-down: defEnv copies a full Map from root to every node. Reference attrs avoid copying large environments but make the dep graph harder to analyze statically.

Q

KSC publishes KSC_STATIC_DEP_GRAPH with a topological order, but doesn't use it for scheduling. Why not, and what would change if it did?

A

KSC uses lazy on-demand evaluation, which naturally respects the dependency order through the call chain. The published order exists for tooling and visualization. Switching to pass-based scheduling would enable (1) predictable traversal order, (2) potential parallelism within a pass, but would lose (1) laziness (computing only what's needed) and (2) simplicity of the current generated code.

Q

What is the most practical improvement KSC could adopt from Silver, and why?

A

Completeness checking: verify at codegen time that every synthesized attribute has an equation (or adequate default) for every node kind in the grammar. KSC already has both the grammar's node list and the attr equations available during compilation. This catches "missing equation" bugs statically, before any user code runs — with low implementation effort.
