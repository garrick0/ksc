Beyond stampTree

Five strategies for tree navigation in attribute grammar systems — what KSC does today, what JastAdd and Silver do, and how to evolve

The Problem: Why AG Systems Need Tree Navigation

Foundation

Attribute grammars need to flow information in two directions:

Synthesized (bottom-up)

Child results flow up to parents. Only requires getChildren() — the parent already knows its children.

// syn: parent reads from children
const totalSize = node.children
  .reduce((sum, c) => sum + c.size, 0);

Inherited (top-down)

Context flows down from parents. Requires a way for children to look up at their parent.

// inh: child reads from parent
const scope = node.$parent.scope;

Without $parent, inherited attributes are impossible. The fundamental question is: how and when do you install parent pointers?

The Core Tension Trees are naturally built bottom-up (leaves first, root last). But inherited attributes flow top-down (root first, leaves last). Someone has to bridge the gap by connecting children back to parents. JastAdd: constructor sets parent navigation is part of construction Silver: decoration captures context navigation is part of evaluation KSC: stampTree BFS pass navigation is a separate phase

KSC's stampTree stamps six properties on every node via BFS: $parent, $children, $index, $root, $prev, $next. They're non-enumerable (hidden from JSON.stringify, Object.keys, spread) because $parent/$prev/$next create circular references.

Why stampTree Exists

The AG engine (ag-interpreter/) was designed as a generic, reusable library. It doesn't know how trees are constructed — it just takes any tree + a getChildren function and stamps navigation on it. This is maximally flexible: you could use it with any tree shape, not just KSC's AST.

But KSC already controls tree construction. The codegen in ast-schema/codegen.ts generates both the node types and the convert.ts that builds nodes. This means the "genericity" of stampTree is unused — there's only one tree shape, and we build it ourselves.

Five Approaches to Tree Navigation

Comparison

A Post-hoc BFS Stamping (Current KSC)

Build the tree as plain objects. Then walk the entire tree via BFS and Object.defineProperty six non-enumerable navigation properties onto every node.

// ag-interpreter/interpret.ts
export function stampTree<N extends object>(
  root: N,
  getChildren: (node: N) => N[],
): void {
  const stamp = (node: N, key: string, value: unknown) => {
    Object.defineProperty(node, key, {
      value, writable: false, configurable: true, enumerable: false,
    });
  };

  const queue: N[] = [root];
  stamp(root, '$parent', undefined);
  stamp(root, '$index', -1);
  stamp(root, '$root', true);
  // ... BFS continues through all descendants
}

Advantages

  • Generic — works on any tree shape
  • Tree construction stays simple (plain objects)
  • Non-enumerable hides circular refs from JSON
  • Already working and tested

Disadvantages

  • Extra O(n) BFS pass over entire tree
  • Mutates nodes after construction
  • Object.defineProperty x6 per node is not free
  • Genericity is unused — only one tree shape exists
  • Gap between "raw" and "stamped" tree is error-prone

B Construct-Time Navigation (JastAdd-style)

Set navigation properties inside the factory that creates each node. Since KSC builds nodes bottom-up in convert.ts, children already exist when the parent is created. The parent constructor links everything.

How JastAdd does it

// JastAdd-generated Java code
public AddExpr(Expr left, Expr right) {
  setChild(left, 0);   // left.parent = this; left.childIndex = 0;
  setChild(right, 1);  // right.parent = this; right.childIndex = 1;
}

// No stampTree. No separate pass. Navigation exists from birth.

What it would look like in KSC

// A helper used by every generated converter
function attachChildren(parent: KSNode, children: KSNode[]): void {
  defProp(parent, '$children', children);
  for (let i = 0; i < children.length; i++) {
    defProp(children[i], '$parent',  parent);
    defProp(children[i], '$index',   i);
    defProp(children[i], '$root',    false);
    defProp(children[i], '$prev',    children[i - 1]);
    defProp(children[i], '$next',    children[i + 1]);
  }
}

// In codegen'd converter:
register(ts.SyntaxKind.FunctionDeclaration, (node, sf, children, pos, end, text) => {
  const ksNode = {
    kind: 'FunctionDeclaration',
    name: findChild(children, n.name),
    body: findChild(children, n.body),
    pos, end, text, children,
  };
  attachChildren(ksNode, children);  // ← navigation installed here
  return ksNode;
});

// Final step: mark root
const root = { kind: 'Program', compilationUnits, children: compilationUnits, ... };
attachChildren(root, compilationUnits);
defProp(root, '$root', true);
defProp(root, '$parent', undefined);

Advantages

  • Eliminates separate BFS pass — one fewer O(n) traversal
  • No "naked node" state — navigation always available
  • Matches JastAdd model closely
  • Minimal change: add attachChildren call per converter
  • Codegen can emit the call automatically

Disadvantages

  • Couples tree construction to navigation
  • $root must be set as a final step (not known during bottom-up build)
  • AG engine loses genericity — can't stamp arbitrary trees anymore
  • deserializeTree still needs its own stamping (but already does this)
$root edge case: During bottom-up construction, you don't know which node will be root. Two options: (a) set $root = true on the final node after construction, or (b) derive it: get $root() { return this.$parent === undefined; }. Option (a) is simpler and matches current behavior.

C External WeakMap Navigation

Store all navigation relationships in external WeakMaps rather than on the nodes. Nodes stay completely clean — zero mutation, zero hidden properties.

class TreeContext<N> {
  private parents   = new WeakMap<N, N>();
  private childMap  = new WeakMap<N, N[]>();
  private indexMap  = new WeakMap<N, number>();

  parent(node: N): N | undefined  { return this.parents.get(node); }
  children(node: N): N[]          { return this.childMap.get(node) ?? []; }
  index(node: N): number          { return this.indexMap.get(node) ?? -1; }
  prevSibling(node: N): N | undefined {
    const p = this.parents.get(node);
    if (!p) return undefined;
    const sibs = this.childMap.get(p)!;
    const i = this.indexMap.get(node)!;
    return i > 0 ? sibs[i - 1] : undefined;
  }
}

Advantages

  • Zero node mutation — purest separation of data and navigation
  • No circular references at all
  • Multiple independent contexts for same tree (NTAs, rewrites)
  • No Object.defineProperty tricks
  • WeakMap allows GC of unreferenced subtrees

Disadvantages

  • Every equation needs a ctx parameter — large API change
  • ctx.parent(n) instead of n.$parent is more verbose
  • WeakMap lookup is ~3-5x slower than property access
  • Still needs an O(n) pass to populate the maps
  • Threading ctx through everything adds complexity

D Decorated Tree Wrapper (Silver-style)

Wrap each raw node in a decoration object that carries navigation, context, and attributes. This is how Silver works: a "decorated tree" is distinct from a "raw tree." The raw AST is never touched.

interface Decorated<N> {
  raw:      N;                         // the original node data
  parent:   Decorated<N> | undefined;  // navigation on wrappers
  children: Decorated<N>[];            // child wrappers
  index:    number;
  prev:     Decorated<N> | undefined;
  next:     Decorated<N> | undefined;
  attrs:    Map<string, unknown>;      // computed attributes live here too
}

function decorate<N>(root: N, getChildren: (n: N) => N[]): Decorated<N> {
  const wrap = (node: N, parent?: Decorated<N>): Decorated<N> => {
    const d: Decorated<N> = {
      raw: node, parent, index: 0,
      children: [], prev: undefined, next: undefined,
      attrs: new Map(),
    };
    d.children = getChildren(node).map((c, i) => {
      const dc = wrap(c, d);
      dc.index = i;
      return dc;
    });
    // link siblings
    for (let i = 0; i < d.children.length; i++) {
      d.children[i].prev = d.children[i - 1];
      d.children[i].next = d.children[i + 1];
    }
    return d;
  };
  return wrap(root);
}

Advantages

  • Perfect separation — raw tree is never modified
  • Can decorate same tree multiple times (different AG specs)
  • Attributes and navigation co-located on wrapper
  • This is exactly what Silver does — proven design
  • No non-enumerable tricks needed

Disadvantages

  • Doubles memory — one wrapper object per node
  • d.raw.kind instead of node.kind in every equation
  • Most invasive change — every spec, equation, and test must change
  • Still needs an O(n) pass (just building wrappers instead of stamping)
  • Wrapper indirection adds cognitive overhead

E Proxy-Based Lazy Navigation

Use JavaScript Proxy to intercept property access on nodes and compute navigation on demand. The node itself is never modified.

function proxyTree<N extends object>(root: N, getChildren: (n: N) => N[]): N {
  const parentMap = new WeakMap<N, N>();
  // ... build parentMap via BFS ...

  const wrap = (node: N): N =>
    new Proxy(node, {
      get(target, prop) {
        if (prop === '$parent')   return parentMap.get(target);
        if (prop === '$children') return getChildren(target).map(wrap);
        if (prop === '$index')    return computeIndex(target);
        return (target as any)[prop];
      }
    });
  return wrap(root);
}

// Usage looks identical to stampTree from the equation's perspective:
node.$parent.scope  // intercepted by Proxy

Advantages

  • Zero mutation of original nodes
  • API is identical — node.$parent just works
  • Lazy — only computes what's accessed

Disadvantages

  • 5-10x slower property access (Proxy overhead)
  • Debugging nightmare — Proxies obscure objects in devtools
  • Poor TypeScript support (proxy types are tricky)
  • Every property access goes through handler, not just navigation
  • Still needs the parent WeakMap built via BFS
  • Clever but impractical for a hot path like AG evaluation
Not recommended. Proxy overhead on every property access makes this unsuitable for AG evaluation, where attributes are accessed thousands of times during a single pass. Included for completeness.

Deep Dive: KSC's Tree Construction Pipeline

Implementation

Understanding why Option B (construct-time) is particularly natural for KSC requires seeing how convert.ts builds nodes.

Current convertNode() flow

// ast-schema/generated/convert.ts (generated by codegen.ts)

function convertNode(node: ts.Node, sf: ts.SourceFile): KSNode {
  // 1. Convert all children first (bottom-up!)
  const children: KSNode[] = [];
  ts.forEachChild(node, (child) => {
    children.push(convertNode(child, sf));  // ← recursive, children exist first
  });

  // 2. Build this node with references to converted children
  const converter = specificConverters[node.kind];
  const ksNode = converter(node, sf, children, pos, end, text);
  // ← at this point, ksNode has .children, .name, .body, etc.
  // ← but NO $parent, $prev, $next, $index, $root

  return ksNode;
}

Why this is perfect for construct-time navigation

The bottom-up recursion means children always exist before their parent. When the parent node is created at step 2, all its children are already fully constructed. This is exactly the precondition attachChildren() needs — the same order that JastAdd constructors use.

The codegen (ast-schema/codegen.ts) generates every register() call. Adding an attachChildren(ksNode, children) call to the template is a one-line change in the code generator.

What the codegen change would look like

// In ast-schema/codegen.ts, the converter template gains one line:

register(ts.SyntaxKind.FunctionDeclaration, (node, sf, children, pos, end, text) => {
  const n = node as any;
  const ksNode = {
    kind: 'FunctionDeclaration',
    name: findChild(children, n.name),
    typeParameters: findChildrenOf(children, n.typeParameters),
    parameters: findChildrenOf(children, n.parameters),
    type: findChild(children, n.type),
    body: findChild(children, n.body),
    modifiers: findChildrenOf(children, n.modifiers),
    pos, end, text, children,
  } as KS.KSNode;
  attachChildren(ksNode, children);  // ← NEW: one added line per converter
  return ksNode;
});

// Actually, even simpler — add it once in convertNode():
function convertNode(node: ts.Node, sf: ts.SourceFile): KSNode {
  const children: KSNode[] = [];
  ts.forEachChild(node, (child) => {
    children.push(convertNode(child, sf));
  });
  const converter = specificConverters[node.kind];
  const ksNode = converter(node, sf, children, pos, end, text);
  attachChildren(ksNode, children);  // ← one line, covers ALL node kinds
  return ksNode;
}

How JastAdd and Silver Handle This

Systems

JastAdd: Navigation at Construction

JastAdd generates Java classes from a .ast grammar. Every generated class has:

// Generated by JastAdd
public class ASTNode {
  private ASTNode parent;
  private ASTNode[] children;
  private int childIndex;

  public void setChild(ASTNode child, int i) {
    children[i] = child;
    child.parent = this;       // ← parent set immediately
    child.childIndex = i;       // ← index set immediately
  }

  public ASTNode getParent() { return parent; }
}

// No stampTree. No separate pass.
// Navigation is a fundamental part of the node type.

Key insight: JastAdd nodes are intrinsically navigable. There's no "raw" vs "stamped" distinction. A node is born connected.

Silver: Decoration as a Concept

Silver distinguishes between undecorated (raw data) and decorated (with inherited attribute values supplied) trees.

-- Silver: nonterminal declaration
nonterminal Expr;
abstract production addExpr
  top::Expr ::= l::Expr  r::Expr
{
  -- 'l' and 'r' are automatically "decorated"
  -- Inherited attributes flow via decoration context
  -- No explicit parent pointer needed
  l.env = top.env;  -- inherited attr passed down
  top.value = l.value + r.value;
}

-- Decoration happens implicitly when you access
-- a child's inherited attributes.
-- The "decorated tree" wraps the raw tree.

Silver's approach is closest to Option D (Decorated Tree). The raw tree is never mutated; instead, decoration provides context. Silver does this at the language level, so the programmer never sees the wrapping.

Timeline: When navigation becomes available JastAdd: ──── constructor ────┤ node has parent ────────────────────── ^ immediate Silver: ──── production ──── decoration ────┤ context available ──── ^ on first inh access KSC (current): ──── buildKSTree ──── gap ──── stampTree ────┤ $parent works ── ^ naked nodes ^ after BFS KSC (Option B): ──── buildKSTree + attachChildren ────┤ $parent works ────── ^ during construction

Recommendation

Decision

Option B (Construct-time) is the right move for KSC

It eliminates the extra BFS pass, matches JastAdd's proven model, and requires minimal change — just adding attachChildren() to convertNode() and buildKSTree(). The codegen already controls all node creation, so this is a one-line template change.

Why not the others?

C (WeakMap) — too invasive

Every equation would need a ctx parameter. The entire spec API changes. Performance regresses. The purity gain doesn't justify the cost for a system with one tree shape.

D (Decorated) — wrong abstraction level

Silver needs decoration because it's a language that manages this automatically. In a library like KSC, the wrapper indirection (d.raw.kind) adds friction without the language-level ergonomics that make Silver's version pleasant.

Migration path for Option B

  1. Add attachChildren() helper to convert.ts (or a shared util)
  2. Call it in convertNode() after the converter returns — one line
  3. Call it in convertSourceFile() and buildKSTree() for the root
  4. Remove stampTree() call from interpret()
  5. Keep stampTree() for deserializeTree() and ad-hoc test trees
  6. Update codegen template so future regeneration includes attachChildren()
  7. Update tests that build raw trees and rely on separate stampTree() calls
Before vs After Before (current): buildKSTree() raw objects, no navigation ↓ stampTree(root, gC) BFS pass, O(n), 6 defineProperty per node ↓ applyAttributes(root) install lazy getters ↓ project(root) trigger evaluation After (Option B): buildKSTree() navigable from birth, attachChildren in each converter ↓ stampTree eliminated from hot path ↓ applyAttributes(root) install lazy getters (unchanged) ↓ project(root) trigger evaluation (unchanged)

Knowledge Check

Flashcards
Why does stampTree use enumerable: false?
Click to reveal
$parent, $prev, and $next create circular references (parent → child → parent). If they were enumerable, JSON.stringify would throw a circular reference error, Object.keys loops could infinite-loop, and spread {...node} would capture back-pointers. Non-enumerable hides them from iteration while keeping direct access (node.$parent) working.
Why doesn't JastAdd need a stampTree?
Click to reveal
JastAdd generates Java classes with setChild() methods. The constructor calls setChild() for each child, which immediately sets child.parent = this and child.childIndex = i. Navigation is part of the node type, not bolted on after construction. There's no "raw" vs "stamped" distinction.
What does Silver's "decoration" do that KSC's stampTree doesn't?
Click to reveal
Silver's decoration supplies inherited attribute values, not just parent pointers. When you access a child in a Silver production, the child is automatically "decorated" with the inherited attributes flowing from the parent. The decorated tree is a separate object from the raw tree. KSC's stampTree only installs navigation; attributes are a separate step (applyAttributes).
Why is the "construct-time" approach (Option B) uniquely suited to KSC?
Click to reveal
Because KSC already controls tree construction via generated code (ast-schema/codegen.ts → convert.ts). The convertNode() function builds nodes bottom-up (children first, then parent), which is exactly the order needed for attachChildren(). Adding one line to the codegen template covers all 364 node kinds automatically.
If we adopt Option B, can we delete stampTree entirely?
Click to reveal
No. stampTree is still needed for: (1) deserializeTree() — reconstructing a tree from JSON needs navigation stamped after parsing, and (2) ad-hoc test trees that are built manually without going through convert.ts. But it's removed from the hot path (the interpret() pipeline).
What's the $root edge case with construct-time navigation?
Click to reveal
During bottom-up construction, you don't know which node will be root until the very end. attachChildren() sets $root = false on every child, but the actual root node needs $root = true set after buildKSTree() creates the Program node. Alternatively, derive it: $root is just $parent === undefined.
Why is the WeakMap approach (Option C) slower than stampTree?
Click to reveal
WeakMap lookups are 3-5x slower than direct property access. In AG evaluation, navigation properties like $parent are accessed thousands of times per tree pass (every inherited attribute equation uses them). The constant-factor overhead of ctx.parent(node) vs node.$parent adds up significantly. Plus, Option C still needs an O(n) pass to populate the maps.
