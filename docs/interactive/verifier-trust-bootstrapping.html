The Core Problem

You build a tool that claims to mathematically prove properties about code. A customer asks: “How do I know your proof is correct?” This is not a hypothetical — it is the #1 objection in every formal verification sales conversation.

The Naive Answer

“Trust us — we’re mathematicians, the theory is sound, and we tested it.”

This is what most SAST tools do. Coverity, SonarQube, and Checkmarx all say “trust our implementation.” No independently checkable artifact is produced. For pattern-matching bug-finders this is fine — they find some bugs, miss some, whatever. But for a tool claiming mathematical proof of absence, this is a fatal credibility gap.

The Real Answer

“Here is the proof. Check it yourself.”

The tool produces a proof certificate — a structured artifact that a much simpler, independently verifiable checker can validate. You don’t need to trust the analyzer. You only need to trust the checker. And the checker can be small enough to audit, or even formally verified itself.

The fundamental asymmetry: Computing a fixpoint (the proof) is expensive — it requires iterative widening, narrowing, and complex domain operations. But checking that a candidate is a valid fixpoint only requires a single pass: apply the abstract transfer function once and verify the inclusion holds. This means the checker can be orders of magnitude simpler than the analyzer.

Why This Matters Commercially

First Client Problem

Your first 2–3 customers are taking a risk on an unproven tool. They need something more than your word. Every subsequent customer asks “who else uses this?” You need to break the chicken-and-egg cycle.

Regulated Industries

Banks (APRA), medtech (TGA/FDA), telcos (SOCI Act) need evidence they can show auditors. “We ran a tool” is not evidence. A checkable proof certificate is evidence.

Competitive Moat

If you produce independently checkable proofs and your competitors don’t, you have a trust advantage that’s almost impossible to replicate quickly. This is a genuine differentiator, not marketing.

Benchmark Performance

Shows the tool works on known test cases but doesn’t prove it works on your code. Good for initial credibility and comparative evaluation. NIST SATE has a special Ockham Sound Analysis Criteria track where a single incorrect finding disqualifies — only Astrée and Frama-C have passed.

The most powerful commercial trust signal. Certora found a fundamental flaw in MakerDAO’s DAI equation that multiple manual audits missed. The Csmith random testing campaign found zero wrong-code bugs in CompCert’s verified portions (vs 79 in GCC, 202 in LLVM). This proves the tool works on real code where competitors fail.

Open-Source Core Engine
Anyone can inspect, audit, and test the analysis implementation.
Examples: Meta Infer (2015), Certora Prover (Feb 2025, GPLv3), Frama-C
Moderate

Open-sourcing invites community scrutiny. Certora open-sourced their prover after 7+ years and $100B+ TVL protected — a deliberate trust-building move. Meta open-sourced Infer in 2015. Frama-C has always been open source. Open source enables independent evaluation but doesn’t guarantee anyone actually audits the code.

5
Peer-Reviewed Publications
Soundness arguments validated by academic peer review at top conferences.
Examples: Astrée (PLDI, ESOP, SAS), Infer (POPL via separation logic), seL4 (SOSP 2009)
Good

Academic publication at PLDI, POPL, CAV, or SAS is table stakes for a formal verification tool. The academic community has rejected Astrée’s paper zero times because the soundness argument is mathematically rigorous. But peer review validates the theory, not the implementation — a gap that higher levels address.

6
Tool Qualification / TÜV Certification
Independent certification body evaluates the tool against safety standards.
Examples: Polyspace (TÜV SÜD for ISO 26262), AbsInt QSK (DO-330, IEC 61508)
Good

TÜV SÜD, TÜV Nord, and Bureau Veritas evaluate tools against DO-330 (aviation), ISO 26262 (automotive), and IEC 61508 (industrial). AbsInt provides Qualification Support Kits with test cases verifying every specified requirement. MathWorks provides IEC Certification Kits and DO Qualification Kits. Cost: months of effort but creates hard evidence for regulated customers.

7
Independent Red-Teaming / IV&V
Hire third parties to try to break the tool or validate its claims.
Examples: NASA IV&V facility, Galois Inc., DARPA PROVERS red team, Trail of Bits
Strong

Independent Verification & Validation (IV&V) typically costs 5–10% of project cost. NASA operates a dedicated IV&V facility. Galois (Portland, OR) specializes in formal methods evaluation. DARPA’s PROVERS program (launched April 2023, 42 months) includes a dedicated red team. In smart contracts, the norm is 2–4 independent audits with public reports. Trail of Bits builds open-source tools (Slither, Echidna) specifically for this.

8
Proof Certificates (Fixpoint Witnesses)
Analyzer outputs a certificate checkable by a simpler, independent program in a single pass.
Examples: Abstraction-Carrying Code (Hermenegildo), certified result checking (Besson/Jensen/Pichardie)
Strong

The sweet spot for commercial viability. The analyzer computes a fixpoint (expensive), then outputs it as a certificate. A simpler checker validates the certificate in a single pass (cheap). The checker can be independently built, audited, or even formally verified. Besson, Jensen, and Pichardie demonstrated this with a Coq-verified checker for interval analysis. Hermenegildo’s Abstraction-Carrying Code further reduces certificate size.

9
Verified Analyzer (Coq/Isabelle/Lean4)
The analyzer itself is formally verified in a proof assistant.
Examples: Verasco (Coq, POPL 2015), CompCert (~100K lines Coq, 6 person-years)
Very Strong

The gold standard for soundness. Verasco (INRIA, POPL 2015) is a certified abstract interpreter built on CompCert’s C semantics — entirely specified and proved sound in Coq. But it took 4+ years, only handles small programs, and excludes recursion and dynamic allocation. CompCert took ~6 person-years of proof effort. Practically: this approach costs 5–10 person-years for a real language subset.

10
Machine-Checked Proofs with Minimal Trusted Kernel
Proofs checked by a tiny kernel satisfying the de Bruijn criterion.
Examples: seL4 (Isabelle, 200K lines proof), Lean4 kernel (~6,700 lines C++), Lean4Lean self-verifying checker
Strongest

The de Bruijn criterion (1960s): proof assistants produce proof objects checkable by a tiny kernel. The checker can be written by someone who completely distrusts the proof producer — they only trust their own checker. Lean 4’s trusted kernel is ~6,700 lines of C++. Lean4Lean (2024) is a self-verifying typechecker for Lean written in Lean. seL4 uses 200,000 lines of Isabelle proof, all passing through Isabelle’s small trusted kernel.

Practical note: You don’t need to verify the entire analyzer at this level. You can verify only the checker (level 8 + level 10), which is dramatically smaller and achievable in months, not years.

The practical recommendation: Aim for Level 8 (proof certificates) with a Level 10 checker. The analyzer does the hard work and doesn’t need to be verified. The checker is simple, small, and can be verified in Lean4/Coq. Customer trusts the ~6,700-line Lean4 kernel — not your analyzer.

The Certificate Approach

Our recommended architecture: the analyzer produces proof certificates that an independently verifiable checker validates. This is the best balance of trust, cost, and practicality.

The key insight: Computing a fixpoint is expensive (iterative, non-deterministic). Checking a fixpoint is cheap (single pass, deterministic). This asymmetry means the checker can be orders of magnitude simpler than the analyzer. A 100K-line analyzer produces a certificate that a 2K-line verified checker validates.

Academic Foundation

▶ Abstraction-Carrying Code (Hermenegildo et al.)

The abstract interpretation analogue of Proof-Carrying Code (George Necula, 1996). The producer runs full abstract interpretation and sends the resulting abstract fixpoint as a certificate. The consumer validates in a single pass using a simplified abstract interpreter. Reduced certificates transmit only the minimal subset the checker needs, significantly reducing size.

▶ Certified Result Checking (Besson, Jensen, Pichardie)

Certificates take the form of “strategies for reconstructing a fixpoint.” The checker, itself verified in Coq, reconstructs and validates the certificate from a compressed form. Demonstrated for interval analysis certifying absence of array-out-of-bounds errors. Uses Farkas lemma-based inclusion certificates that are easily checkable.

▶ SV-COMP Correctness Witnesses (2024)

SV-COMP now uses a standardized YAML v2.0 witness format for exchanging invariants between tools. In 2024, 8 verifiers and 4 validators supported the format. Tools like CPAchecker extract invariants from witnesses to validate correctness proofs. An “unassume” abstract operation incorporates witness invariants into abstract program states. This is the certificate approach in practice, in the largest software verification competition.

Why This Is Better Than Verifying the Whole Analyzer

Verify the Whole Analyzer

  • Cost: 5–10 person-years (Verasco: 4+ years for subset of C)
  • Scope: Severe limitations (no recursion, small programs only)
  • Maintenance: Every change requires re-verification
  • Flexibility: Locked into verified abstract domains
  • Who did it: Verasco (POPL 2015) — academic achievement, never commercially deployed

Verify Only the Checker

  • Cost: Months, not years (checker is ~100x simpler)
  • Scope: No limitations on analyzer complexity
  • Maintenance: Checker rarely changes; analyzer evolves freely
  • Flexibility: Any abstract domain, any optimization
  • Who did it: Besson/Jensen/Pichardie (Coq-verified checker, deployed in SV-COMP ecosystem)

How Existing Players Built Trust

Every successful verification tool followed a trust-building playbook. The patterns are consistent and instructive.

▶ Astrée — The “A340 Moment” Gold Standard
2001–2003 — Academic Foundation
Developed at ENS by Patrick Cousot’s CNRS group. Soundness grounded in Galois connections, each abstract domain individually proven. Published at PLDI, ESOP, SAS.
November 2003 — The Dramatic Proof Point
Proved complete absence of runtime errors in A340 fly-by-wire flight control software: 132,000 lines of C, analyzed in 1h20m, with zero false alarms. First time any analyzer achieved this on industrial-scale safety-critical code.
January 2004 — Flagship Extension
Extended to A380 electric flight control codes. Analysis completed before A380’s maiden flight (April 27, 2005). Also proved ESA’s Jules Verne ATV docking software (2008).
2009 — Commercialization
Licensed to AbsInt by ENS/CNRS. Now deployed in defense, aerospace, medical, industrial control, telecom, and transportation.
Ongoing — Qualification
Qualification Support Kits for DO-330, ISO 26262, IEC 61508, EN 50128. TÜV certified. Passed NIST SATE Ockham Sound Analysis Criteria.
Key lesson: Astrée’s soundness proof is NOT machine-checked. It rests on mathematical arguments validated through peer review and decades of theoretical work. The trust came from a dramatic, verifiable result (zero false alarms on A340) + academic credentials + industry certification. We need our own “A340 moment.”
▶ CompCert — Machine-Checked Verification Verified in Coq
2005 — Project Start
INRIA, led by Xavier Leroy. ANR-funded. Goal: optimizing C compiler with machine-checked semantic preservation proof.
2006–2011 — Proof Effort
~100,000 lines of Coq proof, ~6 person-years. 90% of compiler verified; 10% (elaboration, assembly, linking) unverified.
2011 — External Validation (Csmith)
Yang, Chen et al. used ~6 CPU-years of random testing. CompCert: zero wrong-code bugs in verified portions. GCC: 79 bugs. LLVM: 202 bugs. The most powerful third-party validation of any compiler.
2021 — ACM Software System Award
Highest recognition in systems software. Commercially licensed through AbsInt.
Key lesson: CompCert’s trust comes from two independent sources: the Coq proof (mathematical) and the Csmith results (empirical). The Csmith validation was done by researchers who wanted to break the tool — the most credible form of evidence. We should invite similar adversarial evaluation.
▶ seL4 — The Transparency Gold Standard Isabelle/HOL

seL4 proved functional correctness of 8,700 lines of C + 600 lines of assembly: 20 person-years total proof effort (11 seL4-specific), producing 200,000 lines of Isabelle proof. All derivations pass through Isabelle’s small trusted kernel.

▶ Meta Infer — Scale as Trust Pragmatic

Infer deliberately does not claim traditional soundness. It uses bi-abduction (O’Hearn, Gödel Prize 2016) for compositional analysis. Acquired from Monoidics (2013). Open-sourced 2015. Meta reports “hundreds of bugs per month” fixed before production. Trust metric: fix rate (proportion developers actually fix), not false-positive rate. The “soundiness manifesto” (Livshits et al., CACM 2015) acknowledges that virtually all practical tools make deliberate unsound choices.

Key lesson: Infer proves that scale of deployment can substitute for formal soundness proofs. “We run on every commit across Facebook, Messenger, Instagram, and WhatsApp” is a trust signal. But this path requires having Meta-scale infrastructure. For a startup, the certificate approach provides stronger guarantees without requiring scale.

Benchmarks & Evaluation Frameworks

Standardized ways to demonstrate that a verification tool works, and how to achieve maximum credibility through third-party evaluation.

▶ SV-COMP (Software Verification Competition) Primary Target

14th edition (2025): 62 verification tools, 18 witness validators. 33,353 C verification tasks across 6 specification categories, 674 Java tasks. Organized by Dirk Beyer (LMU Munich). The de facto academic standard for software verification tools.

Key feature: Standardized witness exchange format (YAML v2.0) enables cross-validation — one tool validates another’s proofs. This is exactly the certificate approach in practice.

Why we should participate: Demonstrates our tool on a curated set of programs with verified ground truth. Puts us alongside the best verification tools in the world. Scores are public and comparable.

▶ NIST SATE (Static Analysis Tool Exposition) Credibility

6 editions since 2008. Uses production software as test subjects + Juliet Test Suite (~64,000 C/C++ synthetic test cases covering 118 CWE categories). Explicitly does NOT rank tools — focuses on relevance, correctness, and prioritization.

Ockham Sound Analysis Criteria: A special track where a single incorrect finding disqualifies. Must produce correct findings for 75%+ of sites. In SATE V, only Frama-C passed. In SATE VI, only Astrée and Frama-C/Eva passed. Being in this company is a massive trust signal.

▶ OWASP Benchmark Java-Specific

Open-source Java web application with 2,740 test cases covering 11 vulnerability categories. Scored using Youden Index (TPR + (1 − FPR) − 1). Limitation: Java only; synthetic tests don’t fully represent real-world complexity. Tools score high on synthetic but only 12.7% detection on real-world vulnerabilities.

Relevant if we build Java analysis. The gap between synthetic and real-world performance is exactly where formal verification should excel.

▶ Bug Bounty on the Tool Itself Most Credible

The ultimate trust signal: Pay people to find bugs in your tool. If your analyzer says “property X holds” and someone demonstrates a counterexample, they get paid. This is the approach used in the smart contract space (Certora community competitions, Immunefi bug bounties). The fact that nobody collects the bounty IS the evidence.

Firms for adversarial evaluation: Galois (Portland, formal methods specialists), Trail of Bits (smart contract tools), Mandiant/Google Cloud (general security red team).

Our benchmark strategy: (1) Enter SV-COMP for academic credibility + witness exchange. (2) Aim for NIST SATE Ockham track (being alongside Astrée and Frama-C is extraordinary positioning). (3) Launch a bug bounty on the tool itself. (4) Commission Galois or Trail of Bits for independent adversarial evaluation.

Q

What is the fundamental asymmetry between computing a fixpoint and checking one?

Q

What is the de Bruijn criterion, and why does it matter for trust?

A

Named after Nicolaas de Bruijn (Automath project, 1960s): proof assistants should produce proof objects checkable by a small kernel. The checker can be written by someone who completely distrusts the proof producer — they only trust their own checker and the logical calculus. Lean 4’s kernel is ~6,700 lines of C++. This means trust reduces to auditing a tiny, well-understood program.

Q

What is “Abstraction-Carrying Code” and how does it differ from Proof-Carrying Code?

A

Abstraction-Carrying Code (Hermenegildo) is the abstract interpretation analogue of Necula’s PCC. Instead of a logical proof, the certificate is the abstract fixpoint itself. The producer runs full abstract interpretation; the consumer validates in a single pass using a simplified abstract interpreter. “Reduced certificates” further minimize size by transmitting only the minimal subset needed by the checker.

Q

Why is verifying only the checker dramatically better than verifying the whole analyzer?

A

Verifying the whole analyzer (Verasco approach) took 4+ years, handles only small programs, and excludes recursion and dynamic allocation. Every analyzer change requires re-verification. Verifying only the checker takes months because the checker is ~100x simpler (single-pass inclusion check). The analyzer can evolve freely — add domains, optimizations, new languages — without re-verification. The trust guarantee is identical: the certificate is valid or it isn’t.

Q

How did Astrée establish trust without machine-checked proofs?

A

Through a “dramatic proof point”: in November 2003, Astrée proved complete absence of runtime errors in the A340 fly-by-wire software (132,000 lines C) with zero false alarms. This was backed by: peer-reviewed publications at top venues, decades of theoretical work (Cousot’s abstract interpretation theory), TÜV certification, and passing NIST SATE’s Ockham Sound Analysis Criteria. Q

What is the NIST SATE Ockham Sound Analysis Criteria, and which tools have passed it?

A

A special track within NIST SATE where (1) the analyzer’s findings must be claimed always correct (sound), (2) it must produce findings for 75%+ of sites, and (3) even one incorrect finding disqualifies. In SATE V, only Frama-C passed. In SATE VI, only Astrée and Frama-C/Eva passed. Being in this exclusive company is an extraordinary positioning signal for a new tool.

Q

How did the Csmith campaign validate CompCert, and why was this so powerful?

A

Yang, Chen et al. (2011) used ~6 CPU-years of random C program generation (Csmith) to test multiple compilers. Results: GCC had 79 wrong-code bugs. LLVM had 202. CompCert had zero in its verified portions (bugs were found only in unverified parts). This was done by independent researchers trying to break things — the most credible form of validation. It earned CompCert the 2021 ACM Software System Award.
