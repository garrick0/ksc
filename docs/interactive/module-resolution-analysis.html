1 What Is Module Resolution?

The process of turning an import specifier into a concrete file on disk

When TypeScript encounters import { helper } from './helpers', it needs to figure out which actual file './helpers' refers to. This is module resolution — the bridge between a string specifier and a real file path.

What It Must Decide

Extension Resolution

Does './helpers' mean helpers.ts, helpers.tsx, helpers.js, helpers.d.ts, or helpers/index.ts?

Path Mapping

Does '@app/utils' map to src/utils via tsconfig.json paths? What about baseUrl?

Package Resolution

Does 'lodash' resolve from node_modules/lodash? Which entry point? Does package.json exports field apply?

Declaration Files

If no .ts source exists, is there a .d.ts file? What about @types/ packages in typeRoots?

2 TypeScript's Resolution Algorithms

Five strategies, each modelling a different runtime environment

Classic (Legacy)

The original TypeScript algorithm from before Node.js module conventions were standard. Almost never used in modern code.

Relative imports: Tries .ts, .d.ts extensions at the specified path.

Non-relative imports: Walks up the directory tree, trying each directory. Does not look in node_modules.

// import { x } from './helpers'
// Tries:
./helpers.ts
./helpers.d.ts

// import { x } from 'utils'
// Walks up from current file:
/src/utils.ts
/src/utils.d.ts
/utils.ts
/utils.d.ts

Obsolete

Only useful for AMD/SystemJS projects. Does not understand node_modules or package.json.

Node10 (formerly "Node")

Models Node.js's CommonJS require() resolution. The most widely used strategy historically.

Relative imports: Tries extensions, then /index files.

Non-relative imports: Walks up node_modules directories.

// import { x } from './helpers'
// Tries:
./helpers.ts
./helpers.tsx
./helpers.d.ts
./helpers/package.json  // "types" field
./helpers/index.ts
./helpers/index.tsx
./helpers/index.d.ts

// import { x } from 'lodash'
// Walks up node_modules:
./node_modules/lodash/package.json  // "types"
./node_modules/lodash/index.ts
./node_modules/@types/lodash/index.d.ts
../node_modules/lodash/...  // keep walking up

This is what most projects used before TypeScript 5.x. It does not understand the exports field in package.json.

Node16 / NodeNext

Models Node.js's native ESM resolution (v16+). The most "correct" strategy for Node.js projects.

  • Requires file extensions in import specifiers: import './helpers.js' (not './helpers')
  • Understands package.json "exports" field for package entry points
  • Differentiates CJS (.cts) and ESM (.mts) files
  • "type": "module" in package.json determines file interpretation
// import { x } from './helpers.js'
// Resolves to: ./helpers.ts (replaces .js -> .ts)

// import { x } from 'lodash'
// Reads: node_modules/lodash/package.json
// Uses: "exports" field (conditional exports)
// Fallback: "types", "main" fields

Bundler

Added in TypeScript 5.0. Models how bundlers (webpack, esbuild, Vite) resolve imports.

  • Like Node16, understands package.json "exports"
  • Unlike Node16, does not require file extensions
  • No CJS/ESM distinction — bundlers handle that
  • The most pragmatic choice for modern web projects
// import { x } from './helpers'   -- OK, no extension needed
// import { x } from './helpers.js' -- also OK
// import { x } from 'lodash'
//   -> uses "exports" if present
//   -> falls back to "types" / "main" / index lookup
The resolution API: ts.resolveModuleName()

TypeScript exposes its resolution machinery via a public API:

const result = ts.resolveModuleName(
  './helpers',       // the specifier
  '/src/main.ts',   // containing file
  compilerOptions,   // includes moduleResolution
  ts.sys,            // the host (file system)
);

// result.resolvedModule?.resolvedFileName => '/src/helpers.ts'
// result.resolvedModule?.isExternalLibraryImport => false

Internally, ts.createProgram() calls this for every import statement in every source file, building the full dependency graph. This is how TypeScript discovers which files belong to the program.

CompilerHost: the filesystem abstraction

ts.CompilerHost is the interface TypeScript uses to interact with the filesystem. It allows custom implementations (e.g., in-memory filesystems for tests, or virtual filesystems for language services).

interface CompilerHost {
  fileExists(path: string): boolean;
  readFile(path: string): string | undefined;
  getSourceFile(fileName: string, ...): SourceFile;
  getDefaultLibFileName(options: CompilerOptions): string;
  getCurrentDirectory(): string;

  // Optional: override module resolution
  resolveModuleNames?(...): (ResolvedModule | undefined)[];
  resolveModuleNameLiterals?(...): ResolvedModuleWithFailedLookupLocations[];
}

If you don't provide a CompilerHost, ts.createProgram() uses ts.sys which delegates to the real filesystem via fs.readFileSync etc.

Key Code Paths

src/program.ts — The entry point

export function createProgram(
  rootNames: string[],
  config?: KindScriptConfig,
  options?: ts.CompilerOptions,
): KSProgramInterface {
  // THIS IS WHERE MODULE RESOLUTION HAPPENS
  // ts.createProgram resolves ALL imports transitively
  // starting from rootNames, building the full file graph
  const tsProgram = ts.createProgram(rootNames, options ?? {});
  return createProgramFromTSProgram(tsProgram, config);
}

We pass rootNames (file paths) and CompilerOptions to ts.createProgram(). TypeScript handles everything: parsing every root file, resolving every import, discovering transitive dependencies, loading declaration files, and building the complete program.

We do zero module resolution ourselves. TypeScript does it all.

src/cli/cli.ts — File discovery (not module resolution)

function findRootFiles(rootDir: string): string[] {
  // Walk filesystem to find *.ts files (not .d.ts)
  // This is FILE DISCOVERY, not MODULE RESOLUTION
  // We find the "entry points" — tsc resolves imports from there
  const results: string[] = [];
  function walk(dir: string) {
    for (const entry of fs.readdirSync(dir)) {
      if (entry === 'node_modules') continue;
      // ... walk and collect .ts files
    }
  }
  walk(path.join(rootDir, 'src'));
  return results;
}

The CLI finds root files by walking the directory tree. This is file discovery, not module resolution. The distinction matters: file discovery finds entry points; module resolution finds dependencies.

src/pipeline/convert.ts — Uses resolved source files

export function buildKSTree(tsProgram: ts.Program): KSTree {
  const compilationUnits: KSCompilationUnit[] = [];

  // getSourceFiles() returns ALL files tsc resolved
  // including transitive imports from rootNames
  for (const sf of tsProgram.getSourceFiles()) {
    if (sf.isDeclarationFile) continue;
    compilationUnits.push(convertSourceFile(sf));
  }
  // ...
}

tsProgram.getSourceFiles() returns every file TypeScript resolved — the root files plus all their transitive imports. We skip declaration files and convert each real source file to our KSC AST. We never need to know how they were resolved.

Current Status: Full Piggyback

KSC does zero module resolution. We call ts.createProgram() which internally resolves all imports, then we consume the result via getSourceFiles(). This is identical to how typescript-eslint works — create a ts.Program and piggyback on its resolution.

What typescript-eslint does (for comparison)

typescript-eslint creates a ts.Program from the project's tsconfig.json and uses it for type-aware linting. Its architecture is strikingly similar to ours:

// typescript-eslint (simplified)
const tsProgram = ts.createProgram(
  configParseResult.fileNames,    // from tsconfig
  configParseResult.options,       // compiler options
  host,                            // custom compiler host
);

// Then for each file being linted:
const sourceFile = tsProgram.getSourceFile(filePath);
const checker = tsProgram.getTypeChecker();
// Use checker for type-aware lint rules

The main differences: typescript-eslint uses a custom CompilerHost for caching and uses the TypeChecker for type-aware rules. KSC does neither (yet).

4 Do We Need Module Resolution?

Analyzing when and why different tools need it

Who Needs Module Resolution?

Tool TypeNeeds Resolution?Why?
Type checker (tsc) Yes, always Must resolve imports to check types across file boundaries
Bundler (webpack, esbuild) Yes, always Must resolve imports to build the dependency graph and bundle
Type-aware linter (typescript-eslint) Yes, via tsc Needs TypeChecker which requires resolved program
Syntax-only linter (ESLint without TS) No Only looks at single-file syntax, no cross-file analysis
Formatter (Prettier) No Only reformats syntax, never crosses file boundaries

Analyzing KSC's Actual Needs

Today: noImports Property

The only property KSC currently checks is noImports — enforcing that a value's initializer does not reference imported bindings.

Here's the critical insight: checking noImports does NOT require knowing what the imports resolve to. The checker only needs to know:

  1. Which names in this file were imported (from the import declarations' syntax)
  2. Whether those names are referenced in an annotated value's initializer

This is entirely syntactic and file-local. We look at import { helper } from './helpers' and extract the name helper. We never need to know what file './helpers' actually points to.

noImports: No resolution needed

The checker only inspects import declaration syntax (what names are imported), not the resolved targets (what file they come from). Module resolution is irrelevant for this property.

Future Properties That Might Need Resolution

Consider properties KSC might check in the future:

noIO

No filesystem/network calls. Needs to know if fs.readFile is from the fs module. Could need resolution to identify the source module, or could work purely with TypeChecker type information.

noMutation

No object mutation. Needs type info to know if an assignment target is mutable. Needs TypeChecker (which requires resolution), not resolution directly.

pure

No side effects. Needs to know if called functions are pure. Might need cross-file Kind annotation lookup — if a function from another file is annotated as Pure.

noExternalDeps

Only local imports allowed. Would need resolution to distinguish node_modules imports from local imports. But could also use syntactic heuristics (bare specifiers vs relative paths).

Most future properties fall into two categories:

  1. Need TypeChecker info (types, symbols) — which we already get from tsProgram.getTypeChecker()
  2. Need cross-file Kind annotations — which we already have since buildKSTree() processes all resolved files

Cross-File Kind Propagation

The most compelling reason to care about resolution is cross-file kind checking:

// kinds.ts
export type Pure = Kind<{ noImports: true }>;

// helpers.ts
import type { Pure } from './kinds';
export const add: Pure & (number, number) => number = ...

// main.ts
import { add } from './helpers';
// Q: Does main.ts know that `add` is Pure?

Today this already works because:

  1. ts.createProgram() resolves all imports and gives us all source files
  2. buildKSTree() converts ALL source files (kinds.ts + helpers.ts + main.ts)
  3. The binder AG spec finds Kind<...> definitions across all files and propagates them via inherited attributes

So we already get cross-file analysis for free — because we piggyback on tsc's resolution to get all the files.

Bottom Line

KSC itself does not need to perform module resolution. It needs the results of module resolution (the complete set of source files), and it gets those by delegating to ts.createProgram(). This is the right architecture.

5 Our Options

Four strategies for handling module resolution, from simplest to most complex

Option A: Keep Piggybacking on tsc

Continue using ts.createProgram() exactly as we do today. TypeScript handles all resolution; we consume the result.

RECOMMENDED

Option B: Use tsc's Language Service

Use ts.createLanguageService() for incremental resolution and caching. Better for IDE integration and watch mode.

FUTURE OPTION

Option C: Standalone Parsing (No tsc)

Parse files individually without creating a ts.Program. Implement our own file discovery. Lose cross-file type info.

NOT RECOMMENDED

Option D: Custom Module Resolution

Implement our own resolution via CompilerHost.resolveModuleNames. Could support virtual files, custom aliases.

NICHE USE CASE

Option A: Keep Piggybacking (Current Approach)

Pros

  • Zero code to maintain for resolution
  • Always correct — uses the same algorithm as tsc
  • Handles all tsconfig.json settings automatically
  • Gives us TypeChecker for free
  • Cross-file kind definitions work automatically
  • Matches typescript-eslint's proven approach

Cons

  • Full program creation on every run (no incrementality)
  • Startup cost scales with project size
  • Depends on TypeScript as a heavy runtime dependency
  • No support for non-TS files
// What we do today (program.ts:24)
const tsProgram = ts.createProgram(rootNames, options ?? {});
// That's it. Resolution is done. All source files are available.

Option B: Language Service (Incremental)

A ts.LanguageService wraps a ts.Program but adds incremental updates, caching, and watch support. This is what VS Code uses internally.

Pros

  • Incremental: only re-resolves changed files
  • Cached: repeated runs are fast
  • Built-in watch mode support
  • Still gets full TypeChecker

Cons

  • More complex setup (custom LanguageServiceHost)
  • Memory overhead for persistent state
  • Only worthwhile in long-lived processes (watch/IDE)
// Language Service approach (for future watch mode)
const host: ts.LanguageServiceHost = {
  getScriptFileNames: () => rootNames,
  getScriptVersion: (f) => versions.get(f) ?? '0',
  getScriptSnapshot: (f) => {
    if (!fs.existsSync(f)) return undefined;
    return ts.ScriptSnapshot.fromString(fs.readFileSync(f, 'utf-8'));
  },
  getCompilationSettings: () => options,
  getCurrentDirectory: () => process.cwd(),
  getDefaultLibFileName: ts.getDefaultLibFilePath,
};
const service = ts.createLanguageService(host);
const program = service.getProgram();  // Incremental!

This is the right choice when we add a ksc --watch mode or IDE integration. Not needed for one-shot CLI usage.

Option C: Standalone Parsing (No ts.Program)

Parse each file individually using ts.createSourceFile() without creating a full program. No resolution at all.

Pros

  • Extremely fast — no resolution overhead
  • Files can be processed independently in parallel
  • No dependency graph computation

Cons

  • No TypeChecker — cannot resolve types
  • No cross-file analysis — Kind definitions in one file invisible to others
  • Must implement our own file discovery
  • Import declarations are just syntax with no semantic meaning
  • Breaks our current architecture fundamentally
// Standalone parsing (NO resolution)
const source = fs.readFileSync(filePath, 'utf-8');
const sf = ts.createSourceFile(
  filePath,
  source,
  ts.ScriptTarget.Latest,
  true,  // setParentNodes
);
// We have the AST but no type info, no resolved imports
// Cannot look up: what does 'NoImports' resolve to?

Why This Won't Work for KSC

KSC's core value proposition is checking Kind annotations across an entire program. Kind definitions like type Pure = Kind<{ noImports: true }> are in separate files from the values they annotate. Without cross-file resolution, we can't connect annotations to definitions.

Even our current simple noImports check needs to find Kind definitions across files. Our binder's defEnv inherited attribute propagates definitions from ALL CompilationUnits to ALL nodes — this fundamentally requires all files to be present.

Option D: Custom Module Resolution

Provide a custom CompilerHost that overrides resolveModuleNames. This gives us control over how imports are resolved while still using ts.createProgram().

Use Cases

  • Virtual/in-memory file systems (testing, playground)
  • Custom resolution for KSC-specific imports
  • Filtering: exclude certain imports from resolution
  • Caching resolution results across runs

Downsides

  • Must re-implement or delegate to tsc's default resolution
  • Easy to get wrong (many edge cases in Node resolution)
  • Maintenance burden as TypeScript adds new features
  • No benefit unless we have custom resolution needs
// Custom CompilerHost with resolution override
const host = ts.createCompilerHost(options);
const defaultResolve = host.resolveModuleNames?.bind(host);

host.resolveModuleNames = (moduleNames, containingFile, ...) => {
  return moduleNames.map((name) => {
    // Custom logic: virtual files, caching, filtering
    if (name.startsWith('virtual:')) {
      return { resolvedFileName: resolveVirtual(name) };
    }
    // Fall back to default resolution
    return ts.resolveModuleName(name, containingFile, options, host)
      .resolvedModule;
  });
};

const program = ts.createProgram(rootNames, options, host);

Only pursue this if we need non-standard resolution (virtual files for a playground, custom module aliases beyond tsconfig paths, etc.).

7 Recommendation

A phased strategy aligned with KSC's evolution

Phase 1: Stay With Option A (Now)

Our current approach is exactly right. We call ts.createProgram(), TypeScript handles all module resolution, and we get the complete set of source files plus a TypeChecker for free. This is the same proven approach used by typescript-eslint. There is zero reason to implement our own module resolution.

Why This Is Correct

  1. KSC is a Kind checker, not a module resolver. Our domain is enforcing property annotations on values. We have no business reimplementing what tsc already does perfectly.
  2. Our AG pipeline works at the program level. The binder's defEnv attribute propagates Kind definitions across all CompilationUnits. This inherently requires all files to be present — which ts.createProgram() guarantees.
  3. Future properties will need the TypeChecker. Properties like noMutation or noIO will need type information, which only comes from a resolved ts.Program.
  4. Zero maintenance cost. No resolution code to write, test, or keep compatible with new TypeScript versions.

  • Multi-language support: If KSC ever analyzes non-TS files, we'd need custom resolution for those
What about createProgramFromTSProgram()?

We already expose createProgramFromTSProgram() which accepts a pre-built ts.Program. This is our extensibility hook — it means:

  • A user can create their own ts.Program with custom resolution and pass it to KSC
  • An IDE plugin can reuse its existing ts.Program instead of creating a new one
  • Tests can use mock programs

This API design means we never need to implement custom resolution ourselves — we let the caller handle it if they need to.

Test your understanding of module resolution and KSC's architecture

Q1: Does KSC implement its own module resolution?
Click to reveal answer
No. KSC does zero module resolution. It calls ts.createProgram() which handles all resolution internally, then consumes the result via tsProgram.getSourceFiles(). This is piggybacking on TypeScript's resolution.
Q2: Where in KSC's code does module resolution actually happen?
Click to reveal answer
At src/program.ts:24 — the line const tsProgram = ts.createProgram(rootNames, options ?? {}). Internally, ts.createProgram parses root files, resolves all their imports, recursively discovers transitive dependencies, and builds the full program. All of this is inside TypeScript's code, not ours.
Q3: What's the difference between file discovery (CLI) and module resolution?
Click to reveal answer
File discovery (in cli.ts:findRootFiles()) walks the filesystem to find entry point .ts files. Module resolution takes an import specifier (like './helpers') and finds the actual file it refers to. File discovery finds starting points; module resolution finds dependencies from those starting points.
Q4: Does the noImports property check need module resolution?
Click to reveal answer
No. The noImports check only needs to know which names were imported (from import declaration syntax) and whether they're referenced in an annotated value. It never needs to know what file an import resolves to. However, finding the Kind definition (type NoImports = Kind<...>) across files does require all files to be present — which we get from ts.createProgram().
Q5: What TypeScript moduleResolution strategy is "Bundler" and why was it added?
Click to reveal answer
Bundler (added in TS 5.0) models how bundlers like webpack, esbuild, and Vite resolve imports. Unlike Node16, it doesn't require file extensions in import specifiers ('./helpers' is fine). Unlike Node10, it understands package.json "exports" field. It's the most pragmatic choice for modern web projects that use bundlers.
Q6: Why is Option C (standalone parsing) wrong for KSC?
Click to reveal answer
KSC's core feature is checking Kind annotations, which are defined in separate files from the values they annotate. The binder's defEnv attribute propagates Kind definitions across ALL CompilationUnits. Without module resolution, we wouldn't have all the files, so Kind definitions from one file would be invisible to annotated values in another file. We'd also lose the TypeChecker, which future properties will need.
Q7: What is createProgramFromTSProgram() and why is it architecturally important?
Click to reveal answer
createProgramFromTSProgram() accepts a pre-built ts.Program and creates a KSC program from it. It's important because it decouples KSC from any specific resolution strategy. A user, IDE plugin, or test can create a ts.Program with whatever resolution they need (custom host, virtual files, etc.) and KSC just works. We never need to implement custom resolution ourselves.
Q8: When should KSC move from Option A to Option B (LanguageService)?
Click to reveal answer
When we add a --watch mode or IDE integration. A LanguageService provides incremental updates (only re-resolves changed files), which matters for long-lived processes. For one-shot CLI checks (our current use case), ts.createProgram() is simpler and sufficient.
