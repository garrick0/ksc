01The Boundary

KSC is a two-functor compilation system. Functor 1 (grammar/compile.ts) produces grammar output files from a GrammarSpec. Functor 2 (analysis/compile.ts) produces an evaluator from an AnalysisSpec. The central design question: how does Functor 2 get the grammar information it needs?

The analysis/ directory is designed to be generic — it has no imports from specs/ or from grammar source code. But in practice, the boundary has several leak points where grammar-specific assumptions are baked into the "generic" machinery.

Key insight: The analysis functor does not consume the generated grammar code at compile time. It reads the AnalysisSpec data structure and emits evaluator source code that will import from the generated grammar at runtime. The functor is a code generator — it produces import statements, not imports of its own.

The fundamental difference

This is the single most important concept: compileAnalysis is a code generator, not a code consumer. It takes declarative spec data (attribute definitions, violation rules) and emits TypeScript source code. That emitted code will import from grammar output — but the compiler itself only needs to know the import paths, not the actual grammar types.

// What compileAnalysis DOES:
function compileAnalysis(spec: AnalysisSpec, opts?) → {
  evaluatorFile: { path: 'evaluator.ts', content: /* generated source code */ },
  attrTypesFile: { path: 'attr-types.ts', content: /* generated source code */ },
}

// What it emits (in the generated evaluator.ts):
import type { KSNode } from '../grammar/index.js';      // ← path configured via opts
import { getChildFields } from '../grammar/index.js';  // ← runtime schema access

02Four Channels of Grammar Info

Grammar information flows into the analysis through exactly four channels. Click each stage to explore the details.

1 Codegen Import Paths ▼

The CompileAnalysisOpts interface lets each composition root configure where the generated evaluator should import from. These are string paths emitted into generated source code, not actual TypeScript imports.

interface CompileAnalysisOpts {
  specImportPath?: string;     // default: '../specs/ts-ast/kind-checking/spec.js'
  grammarImportPath?: string;  // default: './ast/index.js'
  analysisImportPath?: string; // default: '../analysis'
  binderImportPath?: string;   // default: derived from analysisImportPath
}

Note: the default values hardcode TS-specific paths — this is one of the leak points. A truly generic system would require these, not default them.

2 Cross-Functor Validation ▼

analysis/validate.ts checks that triggerKind values in violation rules and equation cases keys reference valid grammar kinds. This is the only place where analysis machinery actually receives a CompiledGrammar object.

validateSpec(spec: AnalysisSpec, grammar: CompiledGrammar): ValidationDiagnostic[]

// Checks: grammar.hasKind(rule.triggerKind)
// Checks: grammar.hasKind(caseName) for equation cases

This is cleanly generic — it only uses the hasKind() method on CompiledGrammar. No grammar-specific knowledge needed.

3 Spec Type Imports (Generated Code) ▼

The spec files (which live in specs/, not analysis/) import generated grammar types for their equation functions and violation predicates. This is the intended coupling point — specs are allowed to know about their grammar.

// specs/ts-ast/kind-checking/spec.ts:
import type {
  KSNode, KSIdentifier, KSVariableDeclaration, ...
} from '../../../generated/ts-ast/grammar/index.js';

// specs/ts-ast/kind-checking/binder.ts:
import type { KSCompilationUnit } from '../../../generated/ts-ast/grammar/index.js';

This is by design — analysis specs declare violation predicates like (ctx, node: KSIdentifier) => node.resolvesToImport. They need the generated types to be type-safe.

4 Runtime Schema Access (getChildFields) ▼

The generated evaluator imports getChildFields from the grammar output. This is the runtime bridge between grammar and analysis — the evaluator uses it to build the decorated tree by discovering which fields of each node kind are child nodes.

// In the generated evaluator.ts:
import { getChildFields } from '../grammar/index.js';

// Used in buildKSCTree:
const fields = getChildFields(raw.kind);  // ← asks the grammar schema
for (const field of fields) { ... }

This is also cleanly generic — the grammar exposes a schema-querying function, the evaluator uses it. The evaluator doesn't need to know the actual node kinds.

Summary: Channels 2 and 4 are clean. Channel 3 is intentional (specs are allowed to know their grammar). Channel 1 has hardcoded defaults but is configurable. The real problems are inside the codegen logic of compile.ts, which we'll explore next.

Generic AG machinery (no issues)

These parts of compile.ts are truly generic and reusable across any grammar+analysis:

deriveAllAttrs(spec)          // Derives context + violation + allViolations from properties
buildDepGraphFromAttrs(attrs)  // Topological sort of attribute dependencies
generateAttrTypes(attrs)      // Emits KSCAttrMap interface
generateStructuralBody(...)   // Switch dispatch for structural attributes
KSCDNode class skeleton        // Typed cache, attr(), hasAttr(), navigation
buildKSCTree                   // Schema-aware tree construction via getChildFields

This is the majority of the 739-line file — the generic parts work well.

Kind-checking pattern hardcodings

The analysis compiler knows it's generating a kind-checking system. Three functions embed this pattern:

generateContextBody() (line 586) — hardcodes 'VariableDeclaration':

if ((this.parent!.node as any).kind === 'VariableDeclaration') {
  const kinds = this.parent!.kindAnnotations();
  const match = kinds.find((k: KindDefinition) =>
    (k.properties as any)['${prop.annotationKey}']);
  if (match) return this._c_${a.name} = match;
}

This makes VariableDeclaration the only "annotation carrier" — a complexity analysis might want annotations on FunctionDeclaration instead.

generateViolationBody() (line 600) — hardcodes the CheckerDiagnostic object shape:

{
  node: this.node,
  message: rule.message(this, this.node, kindCtx),
  kindName: kindCtx.name,
  property: '${prop.name}',
  pos: this.node.pos, end: this.node.end,
  fileName: this._findFileName(),
}

A different analysis might produce different diagnostic shapes.

generateAllViolationsBody() (line 665) — the collection-of-violations pattern is hardcoded rather than being driven by the spec.

TS grammar assumptions

_findFileName() (line 364) — walks up the tree looking for 'CompilationUnit':

private _findFileName(): string {
  let current = this.parent;
  while (current && current.node.kind !== 'CompilationUnit') {
    current = current.parent;
  }
  return current ? (current.node as any).fileName : '<unknown>';
}

This hardcodes the TS grammar's root node kind and its fileName field. A Python AST might have 'Module' with a filepath field.

analysis/violation.ts — imports directly from generated TS grammar output:

import type { KindToNode } from '../generated/ts-ast/grammar/kind-map.js';

This is the most direct coupling — a supposedly generic analysis utility imports from a specific grammar's generated code. The violation() builder provides type-safe node narrowing, but at the cost of binding analysis/ to generated/ts-ast/.

Watch out: The violation.ts import means you can't create a second grammar target without either (a) moving the violation builder to the spec, or (b) parameterizing it with a generic KindToNode map.

04Grammar Side Comparison

The grammar side achieved clean separation through three moves. The analysis side hasn't had the same treatment yet. Here's the comparison:

Aspect Grammar Side Analysis Side
Spec interface GrammarSpec — minimal: nodes, sumTypes, convertGenerator? AnalysisSpec — minimal: structural, properties, projections
Code generation 6 files generic + convert.ts delegated to spec's convertGenerator() Evaluator + attr-types generated entirely by compile.ts — no delegation
Spec-specific logic Spec owns its convert generator via closure — grammar/compile.ts just calls spec.convertGenerator() Pattern templates (context, violation, allViolations) are hardcoded in compile.ts
Grammar coupling Zero — grammar/ has no TS-specific imports violation.ts imports from generated/ts-ast/; compile.ts hardcodes 'CompilationUnit'
Shared helpers convert-skeleton.ts, convert-codegen.ts, field-extractors.ts — all generic No shared helper libraries (all inline in compile.ts)
Multiple targets Works — mock grammar uses same machinery with zero changes Would require fixing hardcodings
Key insight: The grammar side's clean separation was achieved by moving the spec-specific convert generation into a closure on GrammarSpec.convertGenerator. The equivalent move for analysis would be to let the spec provide code generation templates for the context, violation, and allViolations patterns.

05Cleanup Options

Four approaches to fixing the boundary leaks, ranging from minimal to comprehensive:

Option A: Fix the grammar leaks only

Effort: Small. Impact: Removes TS grammar assumptions while keeping the kind-checking patterns.

Two changes:

// 1. Parameterize _findFileName() — add to AnalysisSpec or CompileAnalysisOpts:
interface CompileAnalysisOpts {
  rootKind?: string;         // default: 'CompilationUnit'
  fileNameField?: string;    // default: 'fileName'
}

// 2. Move violation.ts to specs/ts-ast/kind-checking/violation.ts
//    The type-safe builder is spec-specific (it needs KindToNode)

This keeps the kind-checking patterns in compile.ts but removes the grammar-specific assumptions. Pragmatic for a single-grammar system.

Option B: Parameterize the pattern templates

Effort: Medium. Impact: The context/violation/allViolations patterns become configurable rather than hardcoded.

interface AnalysisSpec {
  structural: StructuralAttr[];
  properties: PropertyDecl[];
  projections: { ... };

  // NEW: pattern configuration
  patterns?: {
    annotationCarrier?: string;      // default: 'VariableDeclaration'
    annotationAttr?: string;        // default: 'kindAnnotations'
    diagnosticShape?: string;       // template for violation body
  };
}

This makes the system configurable without fully delegating code generation. A complexity analysis could set annotationCarrier: 'FunctionDeclaration'.

Option C: Full spec-owned code generation (like grammar side)

Effort: Large. Impact: The analysis spec gets its own code generator closure, like GrammarSpec.convertGenerator.

interface AnalysisSpec {
  structural: StructuralAttr[];
  properties: PropertyDecl[];
  projections: { ... };

  // NEW: spec-owned method generators
  methodGenerators?: {
    contextBody?: (attr: DerivedAttr) => string[];
    violationBody?: (attr: DerivedAttr) => string[];
    allViolationsBody?: (spec: AnalysisSpec) => string[];
  };
}

This mirrors the grammar side's architecture — the spec owns its patterns entirely. But the current kind-checking patterns are reasonable defaults for most analyses, making full delegation potentially over-engineered.

Option D: Fix violation.ts only

Effort: Minimal. Impact: Removes the most egregious coupling.

// Move analysis/violation.ts → specs/ts-ast/kind-checking/violation.ts
// It imports from generated/ts-ast/grammar/kind-map.js anyway,
// so it already belongs in the spec directory.
//
// The spec.ts file already imports and uses violation().
// Only the import path changes.

This is the lowest-risk change. The violation() builder is fundamentally spec-specific (it needs a KindToNode map from a specific grammar), so it belongs alongside its spec. The analysis machinery doesn't need it.

Recommendation: Start with D (move violation.ts), then A (parameterize _findFileName). These two changes remove all grammar coupling from analysis/ with minimal risk. Consider B later if a second analysis type emerges with different annotation patterns.

Q

Does compileAnalysis import or read the generated grammar files?

A

No. compileAnalysis is a code generator — it emits import statements that reference grammar output, but it never imports or reads the generated files itself. It only needs the import paths (configured via CompileAnalysisOpts).

Q

What are the four channels through which grammar information flows to the analysis?

A

(1) Codegen import paths — string paths emitted into generated code. (2) Cross-functor validation — validateSpec receives a CompiledGrammar to check triggerKinds. (3) Spec type imports — spec files import generated grammar types. (4) Runtime schema access — generated evaluator uses getChildFields to build the decorated tree.

Q

Why is analysis/violation.ts problematic from a separation-of-concerns perspective?

A

It imports KindToNode directly from generated/ts-ast/grammar/kind-map.js. This makes the supposedly generic analysis/ directory depend on a specific grammar target's generated code. It should live in specs/ts-ast/kind-checking/ alongside the spec that uses it.

Q

What grammar-specific kind name is hardcoded in _findFileName(), and what grammar-specific kind name is hardcoded in generateContextBody()?

A

_findFileName() hardcodes 'CompilationUnit' — it walks up the tree to find the root node with a fileName field. generateContextBody() hardcodes 'VariableDeclaration' — it checks if the parent is a VariableDeclaration to find kind annotations.

Q

How did the grammar side achieve clean separation for spec-specific code generation? What's the analysis equivalent?

A

The grammar side uses GrammarSpec.convertGenerator — a () => string closure that the spec owns. grammar/compile.ts just calls it with no args. The analysis equivalent would be spec-owned method generators (Option C) — but unlike convert.ts, the kind-checking patterns are reasonable defaults for most analyses, so full delegation may be over-engineered.

Q

What does deriveAllAttrs produce from an AnalysisSpec with 5 structural attributes and 8 property declarations?

A

22 total attributes: the 5 structural attrs + 8 {name}Context (inherited) + 8 {violationAttr} (synthesized) + 1 allViolations (collection). The 17 derived attrs are mechanically produced from the property declarations — no spec-specific logic needed.

Q

If you wanted to create a complexity analysis over the TS grammar (not kind-checking), what would break in the current analysis/compile.ts?

A

The generateContextBody() template assumes annotations live on VariableDeclaration nodes — a complexity analysis might want annotations on FunctionDeclaration. The CheckerDiagnostic shape is hardcoded in violation bodies. The _findFileName() hardcoding of 'CompilationUnit' would be fine (same grammar), but violation.ts's import from generated/ts-ast/ is already coupled. The property-to-context-to-violation derivation pattern might not even apply.
