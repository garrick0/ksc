Pluggable Property Axes

The property space is infinite — but we don't need all of it. Pick the 5 most useful decomposition axes, implement them as pluggable modules, evaluate each independently, and make the whole system extensible.

Part 1: The RGB Analogy — And Why We Need Plugins

Where RGB works

Human vision has exactly 3 cone types. Three axes (R,G,B) span all visible colors. There's no color a human can see that can't be represented as an (r,g,b) triple.

Where it breaks down

Program properties don't have a fixed number of "cone types." The arithmetic hierarchy has infinitely many strict levels. Each decomposition axis carves off one dimension, but there are always more remaining. It's like electromagnetic radiation — it extends beyond any finite number of axes.

The solution: pluggable axes

Ship with the N most useful axes built in. Make each axis a self-contained plugin with its own properties, analyzers (static + dynamic), and inference rules. Let users and the community add new axes without changing the core.

// The core type: an Axis is a plugin
interface PropertyAxis {
  name: string;
  theorem: string;              // mathematical backing
  properties: Property[];        // what this axis measures
  staticAnalyzer: StaticAnalyzer;   // AST → abstract values
  dynamicAnalyzer: DynamicAnalyzer; // runtime → observations
  unifier: Unifier;                 // combine static + dynamic
}

// KindScript ships with 5 axes. Users can add more:
ksc.registerAxis(myCustomAxis);

Axis 1: Safety / Liveness

Theorem: Every trace property P = Safety ∩ Liveness (Alpern-Schneider). Safety = "nothing bad ever happens" (violated by a finite prefix). Liveness = "something good eventually happens" (can never be violated by a finite prefix).

Why this axis first: It's the most fundamental binary decomposition. Almost every practical property is either a safety property (checkable by monitoring finite prefixes) or has a safety component. Safety properties are where static analysis is most effective.

Properties on this axis

🛡️
Type Safety
No type errors at runtime (no "undefined is not a function")
SafetyStatic: TS compilerDynamic: runtime type checks

Static: TypeScript's type checker already does this (with unsoundness via any/as). KindScript can track where any contaminates and report "type safety confidence."

Dynamic: Zod/ArkType at boundaries. Proxy-based type monitoring in development mode.

Unified: Statically proven safe regions need no runtime checks. any-contaminated regions get runtime guards.

🚫
Null Safety
No null/undefined dereferences
SafetyStatic: strictNullChecks + narrowing

Static: TS strictNullChecks + control flow narrowing. KindScript can verify all ! (non-null assertions) are justified.

Dynamic: Trap null dereferences, log call stack, identify which function produced the null.

Unified: Flag every ! assertion. Prove it safe statically or insert runtime check.

📏
Bounds Safety
No array out-of-bounds access
SafetyStatic: refinement inferenceDynamic: bounds monitoring

Static: Infer array length refinements from guards. Track index variables through loops. Use interval abstract domain.

Dynamic: Wrap arrays with Proxy that traps out-of-bounds reads (returning undefined in JS is silent).

Unified: Report "87% of accesses proven safe, 13% need runtime bounds checks."

⏱️
Termination
Program halts on all inputs
LivenessStatic: structural recursion checkDynamic: timeout guards

Static: Check for structural recursion (decreasing argument on well-founded relation). Detect likely non-termination patterns (unbounded while(true) without break).

Dynamic: AbortController timeouts, iteration counters, call-depth limits.

Unified: Functions with structural recursion: proven. Functions with bounded loops: likely. Unknown: runtime timeout guard.

🔄
Responsiveness
Every request eventually gets a response
LivenessDynamic: monitor req/res pairs

Static: Track promise chains — does every fetch() have a corresponding .then() or await? Are there dangling promises?

Dynamic: Runtime monitor that pairs requests with responses. Alert on timeout or orphaned requests.

Unified: Static finds structural issues (unhandled promises). Dynamic catches timing issues (slow responses).

Analysis pipeline for this axis

Parse AST→ Type narrowing→ Control flow→ Prove safety→ Merge← Runtime monitor← Test suite

Axis 2: Effects / Purity

Backing: Effect systems (Gifford & Lucassen 1986, Koka, Effect-TS). Every function has an effect set — the set of observable side effects it may perform.

Why this axis: Effects are the single most useful property for TypeScript. Knowing which functions are pure enables caching, parallelism, testing, and refactoring. Effect tracking is decidable (intensional safety property) and directly actionable.

Properties on this axis

✨
Purity
Function has no observable side effects
EffectStatic: call graph + ASTDynamic: Proxy monitoring

Static: Walk call graph. Function is pure if: no global reads/writes, no I/O APIs, no this mutation, all callees are pure. Recursively infer.

Dynamic: Wrap in Proxy that intercepts all property access, mutation, and API calls. Run 1000x with diverse inputs. Any effect → impure.

Unified: Static proves purity for most utility functions. Dynamic confirms edge cases. Result: per-function purity label.

💾
State Mutation
Does the function modify external state?
EffectStatic: mutation tracking

Static: Track readonly types, as const, Immer usage. Detect assignments to non-local variables. Separate reads from writes.

Dynamic: Object.freeze + Proxy to detect mutation attempts at runtime.

Classification: No mutation → readonly. Local mutation only → local-mutable. External mutation → mutable.

🌐
I/O Effects
Does it read from or write to the external world?
EffectStatic: API call detection

Static: Detect calls to fetch, fs.*, console.*, XMLHttpRequest, DOM APIs, process.env. Track through call graph.

Classification: network, filesystem, console, dom, process, crypto-random.

⚡
Async Effects
Does it introduce asynchrony?
EffectStatic: async/await detection

Static: Detect async functions, Promise constructors, setTimeout, setInterval. Track async propagation through call graph.

Effect-TS parallel: Effect<A, E, R> already encodes this — the R (requirements) type tracks async dependencies.

🎲
Nondeterminism
Can the function return different outputs for the same input?
EffectDynamic: repeated execution

Static: Detect Math.random(), Date.now(), crypto.randomUUID(). Track through callees.

Dynamic: Run function 100x with same input. If outputs differ → nondeterministic.

Unified: Static catches obvious sources. Dynamic catches hidden ones (e.g., hash map iteration order).

Analysis pipeline for this axis

Detect effect APIs→ Walk call graph→ Propagate effects→ Merge← Proxy monitoring← Repeated execution

Axis 3: Trace / Hyper

Theorem: Every hyperproperty HP = Hypersafety ∩ Hyperliveness (Clarkson-Schneider 2010). Trace properties examine single executions. Hyperproperties relate multiple executions.

Why this axis: Security properties (noninterference, timing-channel freedom) are inherently hyperproperties — you can't check them by watching one execution. This axis tells you which properties need single-execution analysis vs. multi-execution comparison.

Properties on this axis

👁️
Determinism
Same input → same output (always)
2-Trace (Hyper)Dynamic: compare two runs

Determinism is a 2-safety hyperproperty: for any two executions with the same input, the output must be the same. You need to compare at least 2 runs.

Static: Absence of nondeterminism sources (Math.random, Date.now, etc.) implies determinism.

Dynamic: Run twice with same input, compare outputs. Scale to N runs for confidence.

🔒
Noninterference
Secret inputs don't affect public outputs
2-Safety (Hyper)Dynamic: vary secrets, compare outputs

The classic information-flow property: for any two runs that differ only in secret inputs, the public outputs must be identical.

Static: Taint analysis — track secret-labeled data through the program. If it reaches public outputs, flag.

Dynamic: Run with different secret values, same public values. Compare public outputs. Divergence = leak.

⏳
Timing-Channel Freedom
Execution time doesn't depend on secrets
2-Safety (Hyper)Dynamic: timing comparison

Static: Check that control flow doesn't branch on secret data (constant-time analysis).

Dynamic: Time executions with different secret values. Statistical test for timing differences.

📊
Observational Equivalence
Two implementations behave identically
∀-Trace (Hyper)Dynamic: differential testing

Static: Prove two ASTs compute the same function (very hard in general — Rice's theorem).

Dynamic: Differential testing: run both implementations on same random inputs, compare outputs. fast-check is perfect for this.

Analysis pipeline for this axis

Label secrets→ Taint propagation→ Control flow check→ Merge← Multi-run comparison← Timing analysis

Axis 4: Qualitative / Quantitative

Backing: Quantitative verification (RAML, PRISM, Quantitative Information Flow). Some properties are yes/no. Others are measured: how much time, memory, information leakage, complexity.

Why this axis: Boolean properties miss critical information. "Is it fast?" is less useful than "Its worst-case is O(n²) with constant 3.2." Resource analysis is increasingly important for cloud cost optimization.

Properties on this axis

⏰
Time Complexity
Asymptotic time bound: O(n), O(n log n), O(n²), ...
QuantitativeStatic: recurrence analysisDynamic: benchmarking

Static: Count loop nesting depth. Analyze recursive call structure. RAML-style amortized analysis for precise bounds on functional code.

Dynamic: Run on inputs of size n=10, 100, 1000, 10000. Fit curve to timing data. Infer O(n), O(n²), etc.

Unified: Static gives symbolic bound. Dynamic validates with concrete measurements. Disagreement triggers investigation.

📦
Space Complexity
Peak memory usage as a function of input size
QuantitativeDynamic: heap profiling

Static: Track allocation patterns, detect accumulator-style (O(1) space) vs. building collections (O(n) space).

Dynamic: V8 heap snapshots at varying input sizes. Fit curve to peak heap usage.

🔢
Fan-out / Coupling
How many other modules does this function depend on?
QuantitativeStatic: import/call graph

Static: Count unique callees, imports, type dependencies. Measure coupling (efferent/afferent).

Thresholds: fan-out > 10 → warning. fan-out > 20 → refactor candidate.

🔓
Information Leakage
How many bits of secret information leak?
QuantitativeDynamic: statistical estimation

Static: Quantitative information flow analysis — count bits leaked through public outputs.

Dynamic: Run with many different secret values, measure entropy of public outputs conditioned on secrets.

Metric: "0 bits leaked" = noninterference. "> 0 bits" = quantified leakage.

Analysis pipeline for this axis

Loop structure→ Symbolic bounds→ Merge← Benchmark suite← Curve fitting

Axis 5: Extensional / Intensional

Theorems: Rice (1953): every non-trivial extensional property is undecidable. Baldan et al. (2021): even abstract (intensional) properties have Rice-like limits.

Why this axis: It tells you what's possible to check. Extensional properties (about behavior) require either dynamic observation or static over-approximation. Intensional properties (about code structure) can be checked directly on the AST. This axis determines your analysis strategy.

Properties on this axis

🏗️
Code Patterns
Syntactic patterns: loops, recursion, exception handling style
IntensionalDecidableStatic only

Static: AST pattern matching. "Does this use for-of or forEach?" "Is recursion structural?" "Are exceptions caught locally?"

No dynamic needed: These are syntactic — you can see them directly in the code.

KindScript's current analysis is almost entirely intensional. This is why it's decidable and fast.

🔀
Functional Equivalence
Do two functions compute the same thing?
ExtensionalUndecidable (Rice)Dynamic: differential testing

Static: Undecidable in general (Rice's theorem). Can sometimes be proved for specific cases via SMT.

Dynamic: Differential testing with fast-check. Run both on 10,000 random inputs. Same outputs → likely equivalent.

This is the paradigmatic extensional property. It's about what the program computes, not how.

📐
Algebraic Laws
Does this monoid satisfy associativity? Is this functor lawful?
ExtensionalDynamic: property-based testing

Static: If types are strong enough (Lean, Agda), these can be proven. In TypeScript, generally undecidable.

Dynamic: fast-check is ideal: fc.assert(fc.property(fc.anything(), fc.anything(), fc.anything(), (a,b,c) => associativity(a,b,c)))

Laws are extensional — they're about behavior. But property-based testing gives high confidence.

📋
API Contract Adherence
Does this function satisfy its documented contract?
ExtensionalStatic: type checkingDynamic: contract monitoring

Static: Type signature is a partial contract. Refinement types (if available) encode preconditions/postconditions.

Dynamic: Zod/ArkType schemas as runtime contracts at API boundaries.

Unified: Type system proves structural contract. Runtime checks prove value contracts.

Analysis pipeline for this axis

AST patterns→ Type analysis→ Classify← PBT / differential← Contract monitors

Part 3: The Plugin Architecture

Each axis is a self-contained plugin. Here's the interface and how it works:

Core Plugin Interface

// Every axis plugin implements this interface
interface AxisPlugin {
  // Metadata
  id: string;                    // "safety-liveness"
  name: string;                  // "Safety / Liveness"
  theorem: string;               // "Alpern-Schneider 1985"
  description: string;

  // What this axis measures
  properties: PropertyDef[];

  // Analyzers
  createStaticAnalyzer(program: ts.Program): StaticAnalyzer;
  createDynamicAnalyzer(config: DynConfig): DynamicAnalyzer;
  createUnifier(): Unifier;
}

interface PropertyDef {
  id: string;                    // "purity"
  name: string;                  // "Function Purity"
  category: 'safety' | 'liveness' | 'hyper' | 'quantitative';
  decidability: 'decidable' | 'semi-decidable' | 'undecidable';
  valueType: 'boolean' | 'enum' | 'number' | 'interval' | 'set';
  description: string;
}

interface StaticAnalyzer {
  analyze(node: ts.Node): Map<string, AbstractValue>;
}

interface DynamicAnalyzer {
  instrument(code: string): string;   // add runtime monitors
  observe(trace: Trace): Observation;  // process runtime data
}

interface Unifier {
  combine(
    staticResult: Map<string, AbstractValue>,
    dynamicResult: Observation
  ): UnifiedResult;
}

Example: Registering a Custom Axis

Plugin: "Resource Bounds" axis (user-created)

// A community-contributed axis plugin
const ResourceBoundsAxis: AxisPlugin = {
  id: 'resource-bounds',
  name: 'Resource Bounds',
  theorem: 'RAML (Hoffmann et al.)',
  description: 'Tracks memory allocation, stack depth, and computational cost',

  properties: [
    {
      id: 'heap-bound',
      name: 'Heap Allocation Bound',
      category: 'quantitative',
      decidability: 'semi-decidable',
      valueType: 'interval',     // e.g., [0, O(n)]
      description: 'Upper bound on heap allocations'
    },
    {
      id: 'stack-depth',
      name: 'Maximum Stack Depth',
      category: 'safety',        // stack overflow is a safety violation
      decidability: 'decidable', // for non-recursive functions
      valueType: 'number',
      description: 'Maximum call stack depth'
    },
    // ... more properties
  ],

  createStaticAnalyzer(program) {
    return new ResourceStaticAnalyzer(program);
    // Analyzes loop nesting, recursion depth, allocation patterns
  },

  createDynamicAnalyzer(config) {
    return new ResourceDynamicAnalyzer(config);
    // Instruments V8 heap profiling, measures actual allocations
  },

  createUnifier() {
    return new ResourceUnifier();
    // Static bound ∩ dynamic measurement → refined bound
  }
};

// Register with KindScript:
ksc.registerAxis(ResourceBoundsAxis);

How axes compose

Each axis runs independently and produces its own results. The core then composes them into a unified property map:

// Core composition: run all axes, merge results
function analyzeFunction(fn: ts.FunctionDeclaration): PropertyMap {
  const results = new Map();

  for (const axis of registeredAxes) {
    const staticResult  = axis.createStaticAnalyzer(program).analyze(fn);
    const dynamicResult = axis.createDynamicAnalyzer(config).observe(traces);
    const unified       = axis.createUnifier().combine(staticResult, dynamicResult);

    for (const [prop, value] of unified.entries()) {
      results.set(`${axis.id}:${prop}`, value);
    }
  }

  return results;
  // Returns something like:
  // {
  //   "safety-liveness:null-safety": { value: true, confidence: 1.0, source: "static" },
  //   "effects:purity": { value: "pure", confidence: 0.95, source: "unified" },
  //   "trace-hyper:determinism": { value: true, confidence: 0.88, source: "dynamic" },
  //   "quantitative:time-complexity": { value: "O(n)", confidence: 0.72, source: "dynamic" },
  //   "ext-int:code-pattern": { value: "recursive-structural", confidence: 1.0, source: "static" },
  // }
}

The five built-in axes cover the most useful ground. But the plugin architecture means you can add more:

Axis 6: Concurrency

Race conditions, deadlocks, atomicity violations. Van Glabbeek's spectrum. Bisimulation equivalences.

Static: Detect shared mutable state + async access patterns. Dynamic: RacerD-style race detection.

Community plugin

ARIA compliance, keyboard navigation, screen reader compatibility. For UI component analysis.

Static: AST check for aria-* attributes, role attributes. Dynamic: Axe-core runtime audit.

Community plugin

Axis 8: Privacy / GDPR

PII handling, data retention, consent tracking. Information flow + temporal properties.

Static: Taint analysis on PII-labeled fields. Dynamic: Monitor data flow to storage/network.

Community plugin

Axis 9: Your Domain

Financial compliance, medical safety, game physics invariants — any domain-specific properties.

Static + Dynamic: Custom analyzers for domain rules. The plugin interface supports arbitrary property types.

User-defined

The key design constraint: axes must be independent

Each axis runs independently. The core composes results but never needs axis A to complete before axis B starts. This means:

// Each axis can be:
// - Added without modifying existing axes
// - Removed without breaking anything
// - Run in parallel (independent analysis)
// - Updated independently (semver per plugin)
// - Tested in isolation

// The composition is always:
// result = merge(axis1.analyze(fn), axis2.analyze(fn), ..., axisN.analyze(fn))
// Never: axis2 depends on axis1's output

If two axes DO need to share information (e.g., "purity" affects "termination analysis"), this is handled via cross-axis hints — optional, non-blocking suggestions that one axis can emit for another to consume.

Check Your Understanding

Q1: Why can't we just pick 3 axes like RGB and call it done?

Because TypeScript has more than 3 types
Because the property space is provably infinite-dimensional (arithmetic hierarchy, Wadge hierarchy) — every finite set of axes misses something
Because properties are subjective

Q2: What makes the Effects/Purity axis especially useful for TypeScript?

TypeScript has a built-in effect system
It's an intensional safety property — decidable, directly actionable, and enables caching/parallelism/testing
Because JavaScript is a pure language

Q3: Why must security properties like noninterference be on the Trace/Hyper axis?

Because they involve network traces
Because they're always unsafe
Because they inherently compare multiple executions — you can't check "secret doesn't leak" by watching one run

Q4: What's the advantage of axes being independent plugins?

They can be added/removed/run in parallel/updated independently, and the system is extensible without changing the core
Independent plugins are faster than integrated ones
Users can't break the system by writing bad plugins

The five axes: (1) Safety/Liveness — Alpern-Schneider. (2) Effects/Purity — Gifford-Lucassen effect systems. (3) Trace/Hyper — Clarkson-Schneider. (4) Qualitative/Quantitative — RAML, quantitative verification. (5) Extensional/Intensional — Rice's theorem. Each independently evaluated via static + dynamic analysis, composed in the core, extensible via plugins.
