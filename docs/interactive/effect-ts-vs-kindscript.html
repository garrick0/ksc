Effect-TS vs KindScript

Two fundamentally different approaches to the same goal: knowing what your TypeScript programs actually do.

1. The Core Question

Both Effect-TS and KindScript care about program properties — purity, effects, safety, mutation. But they answer the question in opposite directions:

Effect-TS: Encode & Trust

Make the programmer describe effects in the type system via monadic wrappers. Trust that they always use the wrappers. TypeScript's type checker does the rest.

Library Runtime Voluntary
vs

KindScript: Analyze & Verify

Walk the actual AST of ordinary TypeScript code and check whether declared properties hold. No wrappers needed. The analyzer does the work.

Analyzer Static Enforced

This is the difference between a library convention and a compiler guarantee. Let's see exactly how each one works under the hood.

What is Effect<R, E, A>?

It's a lazy description of a computation. Not a function. Not a promise. A plain immutable data structure — a tagged union of primitive instructions that the runtime pattern-matches on later.

// This does NOT execute anything. It builds a data structure.
const getUser = Effect.flatMap(
  Effect.service(Database),       // instruction: "get Database from context"
  (db) => Effect.tryPromise(      // instruction: "run this async thunk"
    () => db.query("SELECT ...")
  )
);

// Type: Effect<Database, UnknownException, User>
//              ^R           ^E               ^A
//         needs this    might throw this   returns this

The three type parameters:

R — Requirements

Services this effect needs. Uses TypeScript's intersection types — composing two effects unions their R's: Database & Logger.

E — Errors

What can go wrong. Uses union types — composing two effects unions their E's: DbError | TimeoutError.

A — Success

The value produced on success. Threaded through via map and flatMap, like Promise but with full type tracking.

What TypeScript tricks make this work?

Effect-TS uses no compiler plugins, no AST analysis, no custom transforms. It's pure TypeScript generics:

▶ Variance annotations (TypeScript 4.7+)
// Simplified — Effect's actual signature uses variance modifiers:
interface Effect<out R, out E, out A> {
  // 'out' = covariant: subtypes widen
  // R: covariant — can have fewer requirements
  // E: covariant — can fail with fewer errors
  // A: covariant — can return subtypes
}

This means Effect<never, never, string> (no requirements, no errors) is assignable to Effect<Database, Error, string>. Variance makes composition type-safe.

▶ Branded types
// Brand.nominal prevents structural confusion:
type UserId = string & Brand.Brand<"UserId">
type OrderId = string & Brand.Brand<"OrderId">
// Both are strings, but TypeScript won't let you mix them
▶ Generic inference in pipe
// pipe is overloaded with ~20 signatures:
function pipe<A>(a: A): A
function pipe<A,B>(a: A, ab: (a:A)=>B): B
function pipe<A,B,C>(a: A, ab: (a:A)=>B, bc: (b:B)=>C): C
// ...TypeScript infers each step's types through the chain

"Effects are Values" — What Does This Actually Mean?

In most TypeScript, calling a function does the thing:

// This EXECUTES immediately — side effect happens NOW
const data = await fetch("/api/users");  // network request fires
console.log("hello");                    // prints immediately
fs.writeFileSync("f.txt", "data");      // writes immediately

In Effect-TS, nothing executes until you explicitly run it:

// This DESCRIBES what to do — nothing happens yet
const program = pipe(
  Effect.tryPromise(() => fetch("/api/users")),
  Effect.flatMap(res => Effect.tryPromise(() => res.json())),
  Effect.tap(data => Effect.log(`Got ${data.length} users`)),
);
// program is just a VALUE. A data structure. An inert description.
// You can store it, pass it around, compose it, test it.

// Only THIS executes it:
await Effect.runPromise(program);

Think of it like the difference between a recipe and cooking. The Effect is the recipe — you can read it, combine it with other recipes, photocopy it. Only runPromise actually turns on the stove.

Why does this matter?

Referential Transparency

program above always means the same thing. You can substitute it anywhere. No hidden state changes.

Composability

Since effects are values, you can map, flatMap, zip, race, retry them — just like arrays.

Testability

You can inspect the description without running it. Swap real services for mocks via Layers.

The Fiber Runtime

When you call Effect.runPromise(program), the fiber runtime takes over. It's a continuation-passing interpreter — essentially a virtual machine for your Effect programs.

Effect value
→
FiberRuntime
→
instruction loop
→
continuation stack
→
Exit<A, E>

The runtime maintains an explicit continuation stack instead of using JavaScript's call stack. This gives it:

▶ Stack safety — deep recursion doesn't overflow

Like a trampoline. When flatMap completes, the runtime pops the next continuation from its own stack instead of returning through JavaScript frames. This means you can nest thousands of flatMaps without stack overflow — the runtime yields and reschedules itself.

▶ Fibers — lightweight green threads

A fiber is a lightweight thread managed by Effect's scheduler, not the OS. You can create thousands of them. Effect.fork(program) starts a fiber. Fibers can be interrupted, joined, raced. The scheduler uses work-stealing queues for concurrency.

▶ Structured concurrency — resources always cleaned up

When a parent fiber completes or is interrupted, all child fibers are automatically interrupted too. Combined with Scope (resource management), this means database connections, file handles, etc. are always released — even if an error occurs.

The Layer System — Dependency Injection via Types

The R parameter in Effect<R, E, A> tracks what services your effect needs. Layers are recipes for building those services:

// 1. Define a service tag
class Database extends Context.Tag("Database")<
  Database, { query: (sql: string) => Effect<never, DbError, Row[]> }
>() {}

// 2. Define a layer that builds it
const DatabaseLive = Layer.scoped(
  Database,
  Effect.gen(function*() {
    const pool = yield* acquirePool("postgres://...");
    return { query: (sql) => Effect.tryPromise(() => pool.query(sql)) };
  })
);

// 3. Use it — R tracks the requirement
const getUsers: Effect<Database, DbError, User[]> =
  Effect.flatMap(
    Effect.service(Database),
    db => db.query("SELECT * FROM users")
  );

// 4. Provide the layer — R becomes 'never' (satisfied)
const runnable = getUsers.pipe(Effect.provide(DatabaseLive));
// Type: Effect<never, DbError, User[]>  — no more requirements

Key insight: layers compose. Layer.merge(DatabaseLive, LoggerLive) builds both. Layers are memoized — each service is constructed exactly once. For testing, swap DatabaseLive for DatabaseTest.

Error Tracking via the E Parameter

In normal TypeScript, errors are invisible in types:

// What errors can this throw? Who knows!
async function getUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);  // NetworkError?
  if (!res.ok) throw new Error("Not found");   // Invisible in type
  return res.json();                           // ParseError?
}

In Effect-TS, errors are union-tracked in E:

const getUser = (id: string): Effect<HttpClient, NetworkError | NotFoundError | ParseError, User> =>
  pipe(
    HttpClient.get(`/api/users/${id}`),  // E includes NetworkError
    Effect.filterOrFail(                    // E adds NotFoundError
      res => res.status === 200,
      () => new NotFoundError(id)
    ),
    Effect.flatMap(res => parseJson(res)),  // E adds ParseError
  );
// TypeScript enforces you handle ALL of these before 'running'

You handle errors with typed combinators instead of try/catch:

pipe(
  getUser("123"),
  Effect.catchTag("NotFoundError", () => Effect.succeed(defaultUser)),
  // E is now: NetworkError | ParseError  (NotFoundError handled)
)

The Escape Hatches — Where Guarantees Break

Effect-TS is a convention, not a cage. Here's where it leaks:

1. Raw side effects bypass tracking

const sneaky = Effect.sync(() => {
  console.log("This side effect is invisible to the type");
  fs.writeFileSync("/tmp/oops", "data");
  return 42;
});
// Type: Effect<never, never, number>
// Claims: no requirements, no errors, pure number
// Reality: does I/O and console logging

2. as any breaks everything

const lie = someEffect as any as Effect<never, never, string>;

3. Mixing Effect and non-Effect code

If half your codebase uses Effect wrappers and half doesn't, the boundary is unpoliced. Nothing stops a colleague from writing fetch() directly.

4. Effect.runSync can throw at runtime

If you runSync an async effect, it throws at runtime — the types don't prevent this.

KindScript is a Compiler Extension, Not a Library

KindScript follows TypeScript's own multi-phase compiler pattern. It delegates parsing to ts.createProgram(), then runs its own bind → link → check phases on top:

ts.createProgram()
→
ts.SourceFile[]
→
ksBind(config)
→
ksLink(symbols, tsProgram)
→
ksCheck(resolved)
→
KSDiagnostic[]

The key files:

config.ts

Type definitions + defineConfig() helper. Properties are declared here as plain objects.

binder.ts

Converts config entries into KindSymbol[] — internal representations of analysis targets.

linker.ts

Maps KindSymbols to actual ts.SourceFile objects by matching paths to files on disk.

checker.ts

The heavy lifting (~520 lines). Walks ASTs using ts.forEachChild(), runs property checks, emits diagnostics.

program.ts

Top-level coordinator. Creates the TS program, runs binder/linker/checker.

types.ts

The type system: PropertySpec, KindSymbol, KSDiagnostic, etc.

The Four-Phase Pipeline

▶ Phase 1: Parse — delegated entirely to TypeScript
const tsProgram = ts.createProgram(rootNames, options);
// TypeScript does: scan → parse → bind → type-check
// KindScript gets: ts.SourceFile[] + ts.TypeChecker

KindScript doesn't re-parse anything. It piggybacks on TypeScript's full AST.

▶ Phase 2: Bind — config → symbols
// Input: your kindscript.config.ts
export default defineConfig({
  domain: {
    path: './src/domain',
    rules: { pure: true, noImports: true }
  },
  infra: {
    path: './src/infra',
    rules: { noIO: true }
  }
});

// Output: KindSymbol[]
// [
//   { id:0, name:"domain", path:"./src/domain",
//     declaredProperties: { pure:true, noImports:true } },
//   { id:1, name:"infra", path:"./src/infra",
//     declaredProperties: { noIO:true } }
// ]
▶ Phase 3: Link — symbols + files → resolved targets
// For each symbol, find matching ts.SourceFiles
// "domain" with path "./src/domain" → all .ts files under src/domain/
// "infra" with path "./src/infra/db.ts" → exactly that one file
//
// Output: ResolvedSymbol[]
// { symbol, sourceFiles: ts.SourceFile[], members? }

This is where config paths become real files. The linker handles file vs directory matching, composite members, and declaration file filtering.

▶ Phase 4: Check — walk ASTs, compare properties, emit diagnostics
// For each resolved symbol → for each source file → for each declared property:
const check = intrinsicChecks.get(propertyName);
const result = check(sourceFile, tsChecker);
if (!result.ok) {
  diagnostics.push(createDiagnostic(prop, sourceFile, result.violations[0]));
}

The checker uses a registry pattern: each property is an independent function (node, checker) → { ok, violations[] }. Diagnostics match TypeScript's format (file, start, length, message, code).

How AST Walking Works

KindScript uses TypeScript's own AST traversal. Here's what happens when you declare noMutation: true:

function checkNoMutation(node: ts.Node): { ok: boolean; violations: PropertyViolation[] } {
  const violations: PropertyViolation[] = [];

  function visit(n: ts.Node) {
    // Catch: x = 1, x += 1, obj.prop = val
    if (ts.isBinaryExpression(n)) {
      const op = n.operatorToken.kind;
      if (op === ts.SyntaxKind.EqualsToken ||
          op === ts.SyntaxKind.PlusEqualsToken || /* ... */) {
        violations.push({ property: 'noMutation', node: n, message: `Assignment` });
      }
    }
    // Catch: x++, x--
    if (ts.isPostfixUnaryExpression(n) || ts.isPrefixUnaryExpression(n)) {
      violations.push({ property: 'noMutation', node: n, message: `Increment/decrement` });
    }
    // Catch: delete obj.prop
    if (ts.isDeleteExpression(n)) {
      violations.push({ property: 'noMutation', node: n, message: `Delete expression` });
    }

    ts.forEachChild(n, visit);  // recurse into children
  }

  visit(node);
  return { ok: violations.length === 0, violations };
}

This is direct pattern matching on the AST. No monads, no wrappers — KindScript reads your actual source code and checks what it does.

The Property Registry

Every property is an independent check function registered in a map:

const intrinsicChecks = new Map<string, IntrinsicCheckFn>([
  ['noImports',     checkNoImports],     // import/require statements
  ['noConsole',     checkNoConsole],     // console.* access
  ['immutable',     checkImmutable],     // let/var at module scope
  ['static',        checkStatic],        // dynamic import()
  ['noSideEffects', checkNoSideEffects], // top-level non-declarations
  ['noMutation',    checkNoMutation],    // assignment, ++, delete
  ['noIO',          checkNoIO],          // fs/net/http imports
  ['pure',          checkPure],          // composite: noIO ∧ noMutation ∧ noSideEffects
]);

This is exactly the pluggable axis pattern from the previous doc — each property is independent, composable, and the registry is extensible.

Satisfaction Check

// Declared: { pure: true, noIO: true }  ← what you claim
// Computed: { pure: false, noIO: true }  ← what the AST shows
// Result:   pure: false ⊇ true → VIOLATION!
//           noIO: true ⊇ true  → OK

4. The Fundamental Difference

The Same Code, Two Perspectives

// src/domain/user.ts
export function processUser(user: User): Result {
  console.log(`Processing ${user.name}`);     // side effect
  const data = await fetch("/api/save");    // I/O
  user.processed = true;                     // mutation
  return { ok: true };
}

Effect-TS says...

Nothing. This is plain TypeScript. Effect-TS doesn't exist in this code. It only knows about code that uses Effect wrappers. This function is invisible to it.

To get Effect-TS tracking, you'd need to rewrite the function:

const processUser = (user: User) =>
  pipe(
    Effect.log(`Processing...`),
    Effect.flatMap(() =>
      HttpClient.post("/api/save")),
    Effect.map(() => ({ok:true}))
  );
vs

KindScript says...

Three violations (if pure: true is declared for this path):

KS70003 Pure violation: console.log
  → src/domain/user.ts:2:3

KS70007 No-IO violation: fetch
  → src/domain/user.ts:3:22

KS70013 Mutation violation: assignment
  → src/domain/user.ts:4:3

5. What Each Approach Can and Can't Do

Capabilities Grid
Depth vs Breadth
The Composability Gap

What Can Each System Tell You?

Depth vs Breadth

Effect-TS: Deep on Effects

Within code that uses Effect wrappers, you get an extraordinarily rich picture: every dependency tracked, every error typed, concurrency managed, resources scoped, retries declarative.

But it covers only one axis (Effects/Purity) and only for code that opts in.

Depth on effects axis: 95%

Breadth across codebase: varies (only wrapped code)

vs

KindScript: Broad Across Properties

Covers the entire codebase — every file in the declared paths is checked. Covers multiple property axes (purity, mutation, IO, imports, side effects, fan-out).

But each check is currently shallow — syntactic pattern matching, no transitive call graph, no deep semantic analysis.

Depth on effects axis: 40% (syntactic)

Breadth across codebase: 95% (all declared paths)

The Composability Gap

This is where "effects as values" gives Effect-TS something KindScript currently can't match:

Effect-TS: Composition IS the Program

Because effects are values, you compose them with pipe, flatMap, zip, race. The types flow through composition automatically. If A requires Database and B requires Logger, composing them gives A & B requiring Database & Logger. This is compositional reasoning — the properties of the whole follow from the properties of the parts, by construction.

KindScript: Analysis After the Fact

KindScript checks properties of code that's already written. It can see that function A is pure, and function B has mutation. But it doesn't currently track what happens when A calls B. There's no transitive property propagation yet — no call graph that says "A calls B which calls C which does IO, therefore A is impure."

This is the next frontier for KindScript: building a call graph and propagating properties through it, so that a pure annotation on a function actually checks that nothing it transitively calls violates purity.

What "Effects as Values" Gives You (That Static Analysis Doesn't)

Runtime Power

Retry, timeout, race, fork, interrupt, scope — these are runtime behaviors that static analysis can describe but not implement. Effect-TS gives you both the tracking and the execution.

First-Class Error Handling

Effect-TS makes errors visible and provides typed combinators to handle them. KindScript can tell you something might throw, but doesn't give you structured error handling.

What Static Analysis Gives You (That Effect-TS Doesn't)

Zero Code Changes

KindScript analyzes code as-is. No rewriting, no wrappers. Works on existing codebases immediately.

Verification of Claims

KindScript checks whether code actually is pure, not whether the programmer said it was pure. Effect-TS trusts the wrapper; KindScript reads the code.

6. Where They Could Converge

The interesting question is: what if KindScript understood Effect-TS code? Or what if they worked together?

Option 1: KindScript Verifies Effect-TS

KindScript could check that code inside Effect.sync(() => ...) actually matches what the Effect type claims. If Effect<never, never, number> contains a console.log, KindScript flags it. This bridges the trust gap.

Option 2: KindScript Generates Effect Types

After AST analysis, KindScript could emit inferred effect signatures. "This function does IO and mutation" → automatically generate Effect<FileSystem, IOError, Result> type annotations or documentation.

Option 3: Effect-TS as KindScript's Runtime Layer

KindScript handles static verification. Effect-TS handles runtime execution. Together: statically verified effects with a production-ready runtime.

Option 4: KindScript Adopts Compositional Tracking

Build a call graph, propagate properties transitively, and achieve the same compositional reasoning that Effect-TS gets from monadic composition — but through analysis rather than encoding.
