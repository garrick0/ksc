Library vs Generator: How Real Systems Bridge the Gap

Code generators (JastAdd, protoc, Hibernate) get a compile step that gives them validation, optimization, and separation of concerns. Libraries (Kiama, MobX, SQLAlchemy) don't. Here's how 12 real systems solved that problem — and what KSC can learn from them.

▶

The Fundamental Tension

Context

A code generator reads a specification in a DSL, analyzes it, and emits optimized target-language code. This gives it a "compile step" where it can:

  • Validate completeness — every declared thing has an implementation
  • Analyze dependencies — detect cycles, compute evaluation order
  • Optimize — inline, specialize, eliminate overhead
  • Separate concerns — schema vs implementation live in different files/languages
  • Weave aspects — merge contributions from multiple files into one class

A library lives in the host language. Users call functions and pass objects. There's no separate "compile step" — everything happens at runtime. This gives it:

  • Zero toolchain overhead — no build plugin, no codegen step
  • Full host-language power — closures, generics, debugging, IDE support
  • Incremental adoption — just import and use
  • Composability — mix with any other library
The question for KSC

Our AG library is a runtime library. JastAdd is a code generator. We identified several things generators get that we don't (completeness checking, grammar separation, dependency analysis, optimized caching). But this is a well-studied tension. Many systems have faced it before us. What patterns did they use?

Seven Strategies Libraries Use to Get Generator Benefits

Patterns

Across all the case studies, seven distinct strategies emerge. Most successful systems combine two or three.

1 Registration Phase (Class-Definition-Time Work)

The library provides a "define" or "register" step that runs once, at module load / class definition time, before any runtime use. This is the library's "compile step."

Core insight

Module initialization in JS/Python/Java is a compile step. Code that runs when classes are defined or modules are imported can validate, analyze, and optimize — all before any "real" work happens.

Django ORM (Python)

Uses a ModelBase metaclass. When you write class User(Model), the metaclass fires at class definition time:

  • Collects all Field instances from class attributes
  • Creates a _meta object with schema metadata
  • Registers the model in the global apps registry
  • Validates field configuration (completeness, constraints)
  • Installs descriptors for attribute access

By the time you instantiate User(), all validation is done. The metaclass is the compiler.

attrs / dataclasses (Python)

The @define decorator runs at class definition and does all heavy work:

  • Scans for attr.ib() sentinel objects
  • Generates __init__, __eq__, __repr__ methods
  • Validates types and defaults
  • Freezes the class configuration

Key quote from attrs docs: "Everything happens exactly once when the class is defined. attrs runtime impact is very close to zero because all the work is done when the class is defined."

What this gives you (that generators get)

Generator capabilityRegistration-phase equivalent
Completeness checkingValidate at .build() / .seal() that all declared attributes have equations
Dependency analysisBuild dep graph at registration time, detect cycles before first evaluation
Schema separationSchema is the registration calls; implementation is the equation functions
Aspect weavingMultiple modules call .contribute() before .seal()

2 Descriptor / Property Protocol

Use the host language's property interception mechanism to make ordinary property access trigger computation, caching, and tracking.

KSC AG Library (JavaScript)

Object.defineProperty with configurable getters. First access triggers computation, then replaces getter with a frozen data property. This is the JS equivalent of JastAdd's generated _computed/_value fields.

// installLazy: the core mechanism
Object.defineProperty(node, key, {
  configurable: true,
  get() {
    const value = compute(node);
    Object.defineProperty(node, key, {
      value, writable: false
    });
    return value;
  }
});

SQLAlchemy (Python)

Python descriptors (__get__/__set__) intercept attribute access on model instances. Column objects are replaced with "instrumented attributes" that track changes, lazy-load relationships, and integrate with the unit-of-work pattern.

Key insight: "When a class is constructed, Declarative replaces all the Column objects with special Python accessors known as descriptors — this is a process known as instrumentation."

Kiama (Scala)

Uses Scala's lazy val and custom CachedAttribute objects. Attributes are functions from node to value stored in attribute objects (not on the nodes themselves, unlike JastAdd/KSC). Caching happens inside the attribute object's internal map.

This is a different design choice: external caching (attribute objects hold the cache) vs node-stamping (KSC/JastAdd put values on nodes). Both achieve the same thing: compute-once semantics.

MobX (JavaScript)

Uses Object.defineProperty (MobX 5) or Proxy (MobX 6) to intercept reads and writes. When you read observable.value, MobX records the access in a dependency graph. When you write, it notifies all dependents.

This is exactly the same mechanism as KSC's lazy getters, but for a different purpose: reactive UI updates instead of AG evaluation.

3 Type-System Encoding (the TS Compiler as Your Generator)

Use the host language's type system to enforce constraints that generators normally check. The type checker becomes your compile step.

Prisma (TypeScript)

Prisma generates a fully-typed client from its schema. Every query is type-checked: prisma.user.findMany({ where: { age: { gt: 18 } } }) — TS knows which fields exist, their types, and valid filter operations. The generated types make the type checker enforce schema compliance.

Hybrid approach: Prisma uses codegen to produce types, then relies on TS to enforce them. The insight: you don't need codegen for the enforcement if you can express the schema in TS types directly.

Zod / ArkType (TypeScript)

Pure library. Defines schemas as TypeScript expressions, extracts types via z.infer<typeof schema>. No codegen, yet you get full type safety. The schema IS the type definition.

const User = z.object({
  name: z.string(),
  age: z.number().min(0),
});
type User = z.infer<typeof User>;
// { name: string; age: number }

This proves that TS's type system can substitute for codegen when the schema is expressible as types.

Application to KSC

We could use satisfies with mapped types to enforce equation completeness at tsc time. If the grammar is defined as a type, TS can check that every production has an equation. No build plugin needed — the TypeScript compiler is the "JastAdd compiler."

// Grammar as types
interface KSGrammar {
  CompilationUnit: { statements: KSNode[] };
  Identifier: { escapedText: string };
  VariableDeclaration: { name: KSNode; type?: KSNode };
}

// Force completeness or explicit default
type FullEquations<V> =
  { [K in keyof KSGrammar]: (n: KSGrammar[K]) => V }
  & { _?: (n: KSNode) => V };

// TS ERROR if you miss a production and have no _:
const kindDefs = match('kind', {
  CompilationUnit: cu => extractDefs(cu),
  // Error: missing Identifier, VariableDeclaration...
} satisfies FullEquations<KindDef[]>);

4 Host-Language Macros / Compile-Time Execution

Some languages let libraries run arbitrary code at compile time. This gives library authors the full power of a code generator without requiring external tooling.

Rust proc macros

Crepe is a Datalog engine embedded in Rust as a procedural macro. You write Datalog-like syntax inside crepe! { ... }, and the macro expands to optimized Rust code with semi-naive evaluation, automatic indexing, and stratified negation.

serde uses #[derive(Serialize)] to generate serialization code at compile time. No runtime reflection. The proc macro inspects the struct's fields and generates specific serialization code for each one. This is why kotlinx.serialization chose the same approach.

Performance is comparable to hand-written code and compiled Souffle, with fraction of compilation time.

Zig comptime

Zig's comptime lets you run any Zig code at compile time. No separate language, no special syntax. The same function can run at compile time or runtime depending on the call site.

Key quote: "Zig saw that generics is literally just generating code at compile time and decided to ask 'why on earth are we writing entirely separate turing complete programming languages for this?'"

No macros, no templates, no codegen tools — just regular code that happens to execute during compilation.

Swift macros

Swift 5.9 introduced @attached and #freestanding macros. They are type-safe, AST-aware, and integrated into the compiler. A library can ship a macro that generates code when your type is compiled.

Example: @Observable generates property-change tracking code. The developer writes a plain class; the macro adds all the observation boilerplate at compile time.

Availability in JavaScript/TypeScript

JS doesn't have proc macros or comptime. The closest equivalents are TC39 decorators (Stage 3, limited to class/method/field metadata) and TS compiler plugins (powerful but non-standard). This is why JS libraries lean harder on strategies 1-3 and 5-7 instead.

5 Proxy-Based Interception (Runtime Dependency Discovery)

Use runtime interception (Proxies, aspect weaving, or instrumented access) to automatically discover information that generators compute statically.

MobX (Dependency Tracking)

MobX wraps objects in Proxies. When a reactive computation reads obj.x, MobX records the dependency. When obj.x changes, MobX knows exactly which computations to re-run.

This is runtime dependency analysis — the same thing JastAdd computes statically from equations. MobX discovers it dynamically by observing actual access patterns.

Trade-off: "MobX's only downside is that it tracks data access and not the data itself, which can be a source of bugs." Static analysis (Svelte's compiler) avoids this class of bugs entirely.

Application to KSC: Dry-Run Analysis

We could wrap a sample tree in Proxies, run each equation once, and record all property accesses. The access log is the dependency graph:

const deps = recordDeps(spec, sampleTree);
// {
//   importViolation: ['noImportsContext',
//     'isReference', 'fileImports',
//     'enclosingLocals'],
//   ...
// }

This works because match with a _ default guarantees all dispatch paths are covered. One sample per node type is sufficient.

6 Optional Compiler (Progressive Enhancement)

Ship a library that works at runtime. Optionally provide a compiler/build-plugin that makes it faster or adds validation. The library is the source of truth; the compiler is a performance optimization.

React Compiler (React 19+)

React works fine without the compiler — it's a runtime library using virtual DOM diffing. The React Compiler analyzes components at build time and auto-inserts useMemo/useCallback to skip unnecessary re-renders.

Key quote: "React is saying: keep the programming model the same, we'll just use a compiler to make it faster."

The library semantics are canonical. The compiler is a transparent optimization. You can disable it and everything still works.

Svelte (Compiler-First)

Svelte took the opposite path: compiler-first from day one. Components are compiled to imperative DOM operations. No virtual DOM, no runtime diffing.

But Svelte 5 introduced "runes" — a signal-based runtime that works without compilation. The compiler optimizes, but the reactivity model is understandable and debuggable at runtime too.

Convergence: Runtime libraries add optional compilers (React). Compiler frameworks add runtime understanding (Svelte 5). They're meeting in the middle.

kotlinx.serialization

Uses a Kotlin compiler plugin to generate serialization code at compile time. No runtime reflection. The generated code knows exactly which properties exist, their types, and how to read/write them.

Contrast with Jackson/Gson: These use runtime reflection — scanning class structures, bypassing access modifiers. Much slower, and can cause runtime NPEs that the compiler plugin prevents.

Key insight: The compiler plugin is mandatory, not optional. But it's a library-scoped compiler plugin — it doesn't require a whole new build system, just a Gradle dependency.

Application to KSC

We could provide an optional TS transformer or Vite plugin that:

  • Replaces installLazy with inlined caching code
  • Converts match dispatch to switch statements
  • Validates equation completeness at build time
  • Emits dependency graphs as build artifacts

Without the plugin, everything works (our current behavior). With it, you get JastAdd-level optimized code. Progressive enhancement.

7 Schema Object (Shared Substrate)

Factor out the shared schema/grammar into a first-class object that multiple specs reference. This is the simplest and most common pattern.

Protocol Buffers

The .proto file is the schema. Multiple code generators (protoc for Java/C++/Go, protobuf.js, prost for Rust) all consume the same schema and generate language-specific code.

Runtime libraries like protobuf.js can also parse .proto files at runtime and create message classes dynamically, or use pre-generated static code. The schema exists independently of any particular generator.

GraphQL

The GraphQL schema defines types and relationships. Resolvers (the equivalent of AG equations) are provided separately. The schema is the grammar; resolvers are the rules. Multiple tools (Apollo, graphql-codegen, Pothos) all work with the same schema concept.

Application to KSC

Factor children out of AGSpec into a shared grammar object:

const grammar = createGrammar<KSNode>(getChildren);

// Specs are now pure domain + rules:
const binder = { attributes: { kindDefs, defEnv, defLookup } };
const checker = { attributes: { ... }, deps: ['ksc-binder'] };

grammar.evaluate(root, [binder, checker]);

This is the .ast file equivalent. The grammar knows the tree structure. Specs just provide attributes. Zero magic, zero build step, immediately achievable.

12 Case Studies: Generator vs Library Pairs

Case Studies

Each pair shows a generator and a library solving the same problem. The interesting question is: what strategies did the library use to close the gap?

Domain: Parsing
yacc / ANTLR (generator) nom / parsec (library)

This is the original, canonical instance of the library-vs-generator tension.

Generator: yacc, ANTLR, tree-sitter

  • Read a .y or .g4 grammar file
  • Analyze grammar (check for ambiguity, compute parse tables)
  • Generate optimized parser code in target language
  • Guarantee linear-time parsing for restricted grammar classes

Compile-step benefits: grammar validation, ambiguity detection, performance guarantees, generated table-driven parsers

Library: nom, parsec, Parsimmon

  • Parsers are ordinary functions/values in the host language
  • Composed using combinators: seq, alt, many
  • Full host-language power in semantic actions
  • IDE support, type checking, debugging "for free"

How they bridge the gap:

  • Type system (Strategy 3): Haskell/Scala parsec uses types to ensure parser composition is well-typed
  • Runtime analysis (Strategy 5): some combinator libs detect left-recursion at first parse
  • No ambiguity checking — this is genuinely lost. The user is responsible for correct grammar
The outcome

Handwritten/combinator parsers have won for production compilers: GCC, Clang, V8, Go, Rust, Swift, TypeScript — all use handwritten recursive descent. Parser generators are used for prototyping, DSLs, and IDE tooling (tree-sitter). The library approach traded validation/guarantees for flexibility/debuggability, and the flexibility won.

Domain: Attribute Grammars
JastAdd / Silver (generator) Kiama / KSC (library)

Our direct domain. This is the exact tension we're trying to resolve.

Generator: JastAdd

  • Reads .ast (grammar) + .jrag (attributes) files
  • Generates Java classes with _computed/_value caching fields
  • Validates: every inh attr has equations on all parents
  • Merges aspects from multiple .jrag files
  • Static dependency analysis for circular attributes

Library: Kiama (Scala)

  • "Attributes are defined as functions by normal Scala code"
  • Uses Scala traits/mixins for modular specification (host-language modules, not a custom system)
  • Caches values in attribute objects (external cache, not node-stamped)
  • No completeness checking — missing equations are runtime errors
  • Leverages Scala's type system for some safety

Strategies used: Descriptor protocol (cached attribute objects), host-language modules (Scala traits for aspect-like composition), type system

What Kiama genuinely loses

Completeness checking. JastAdd statically verifies that every inherited attribute has an equation for every child in every production. Kiama and KSC discover missing equations at runtime (when a node tries to read an attribute that no one computed). This is the hardest gap to close without a compile step.

Domain: Object-Relational Mapping
Prisma (schema + codegen) TypeORM / SQLAlchemy (runtime)

Generator: Prisma

Dedicated .prisma schema language. Generates a fully-typed client with method per model, type-safe where clauses, and validated migrations.

Key advantage: "Prisma offers strong, automatic type safety through its generated client, with all queries — simple or complex — being fully type-checked based on your schema."

Library: TypeORM (decorators) / SQLAlchemy (metaclass)

TypeORM: Uses TS experimental decorators (@Entity, @Column) for schema definition. Less type-safe for complex queries.

SQLAlchemy: Uses Python metaclass + descriptors. Model registration at class-definition time. The metaclass is the "compiler" — validates fields, creates schema metadata, installs descriptors.

Strategies used: Registration phase (metaclass), descriptor protocol, type-level encoding (TypeORM generics)

Lesson

SQLAlchemy is the gold standard of "library that gets generator benefits." Its metaclass protocol gives it class-definition-time validation. Its descriptor protocol gives it lazy loading. Its MetaData object is a first-class schema (Strategy 7). It combines three strategies and is widely considered superior to the Java generators it competes with.

Domain: Serialization
protoc / kotlinx.serialization (codegen) Jackson / Gson (runtime reflection)

Generator: protoc, kotlinx.serialization

protoc: Reads .proto schema, generates language-specific classes with optimized serialize/deserialize methods.

kotlinx.serialization: Kotlin compiler plugin. Analyzes @Serializable classes at compile time, generates specific serialization code. "The generated code knows exactly which properties exist, their types, and how to read and write them."

Advantages: No runtime reflection, cross-platform support (KMP), catches schema violations at compile time.

Library: Jackson, Gson

Use Java runtime reflection to discover fields, their types, and access modifiers at runtime. Flexible but slower — reflection is "a heavy operation that involves scanning class structures and bypassing access modifiers."

Hybrid: Moshi — offers both reflection mode (library, slow) and codegen mode (annotation processor, fast). Same API, choose your compilation model.

The Moshi pattern

Moshi is the clearest example of Strategy 6 (Optional Compiler): same library API, works with runtime reflection, optionally use annotation processing for zero-reflection codegen. This is exactly the progressive-enhancement model we could use for KSC.

Domain: UI Reactivity
Svelte (compiler) MobX / Solid (runtime signals) React Compiler (hybrid)

Compiler: Svelte

Compiles components to imperative DOM operations. No virtual DOM, no runtime reactivity overhead. "Svelte is one of the fastest frameworks because it's a compiler that optimizes itself away."

Library: MobX

Proxy-based observable tracking. "If you're using MobX, you don't need the React Compiler. MobX already does this for you." Runtime signals achieve the same result as compiler analysis — fine-grained dependency tracking and minimal re-renders.

Hybrid: React Compiler

"Keep the programming model the same, we'll just use a compiler to make it faster." Library semantics + optional compiler optimization. The canonical Strategy 6 example in the JS ecosystem.

Convergence

The industry is converging on TC39 Signals — a standard proposal with support from Angular, Ember, MobX, Preact, Qwik, Solid, Svelte, and Vue. This represents the runtime-library side winning the programming model, with compilers as optional optimizers. The same trajectory KSC could follow.

Domain: Datalog / Logic Programming
Souffle (compiler) Crepe (proc macro) Datafrog (pure library)

This trio perfectly illustrates the full spectrum:

Generator: Souffle

Standalone Datalog compiler. Reads .dl files, compiles to parallel C++ with sophisticated optimizations (semi-naive evaluation, magic sets). Separate toolchain, maximum performance.

Macro: Crepe

Rust proc macro. Write Datalog inside crepe! { ... }, macro generates Rust code at compile time. "Comparable speed to compiled Souffle, fraction of the compilation time." Host-language macro is the generator.

Library: Datafrog

"Has no runtime, and relies on you to build and repeatedly apply the update rules." Types are wrappers around Vec<Tuple>. No DSL, no macro — just call functions to define relations and rules. Used by rustc's borrow checker (Polonius).

Lesson

All three coexist because they serve different use cases. Souffle for large-scale program analysis. Crepe for embedded Datalog with good ergonomics. Datafrog for minimal-dependency embedding in compilers. The right point on the spectrum depends on your constraints, not on any inherent superiority.

The meta-lesson across all 12 case studies

The most successful library-based systems (SQLAlchemy, MobX, Zod, attrs) combine 2-3 strategies from the middle of the spectrum: registration phase + descriptor protocol + type encoding. They don't try to replicate everything a generator does. They pick the highest-value capabilities (validation, type safety, schema separation) and use library-native patterns to achieve them. The pure-codegen capabilities (optimized output, cross-language support) are left to optional tooling or accepted as trade-offs.

For KSC: Strategies 7 + 1 + 3 + 5 (schema object + registration/validation + type-level checking + runtime dep analysis) are now implemented, closing ~80% of the gap with JastAdd, with no build tooling required.
