KSC Codebase Architecture

A visual guide to KindScript's two-functor compilation pipeline — from grammar specification through compiled evaluator to kind-checked diagnostics.

At a Glance

KSC (KindScript Compiler) is a meta-compiler for TypeScript that type-checks behavioral properties (no side effects, immutability, no I/O, etc.) against user-defined kind annotations. It uses an attribute grammar evaluation model, but compiles all equations into a static evaluator at build time — no generic interpreter in production.

The Big Idea

Two pure compilation functors take declarative specifications and produce fully-typed TypeScript code. This separates what you define (grammar, analysis) from how it runs (generated code), creating clean decoupling boundaries.

Functor 1 Grammar Compilation

compileGrammar(GrammarSpec) → CompiledGrammar — a pure function that takes the declarative grammar definition and produces 7 TypeScript files.

Input: GrammarSpec

Built from grammar/nodes.ts which calls the builder API (~360 node()/leaf() calls).

interface GrammarSpec {
  nodes: ReadonlyMap<string, NodeEntry>      // All node definitions
  sumTypes: ReadonlyMap<string, SumTypeEntry> // Sum type hierarchy
  fieldExtractors: Record<string, Record<string, string>>  // Per-kind field overrides
  skipConvert?: ReadonlySet<string>
  syntaxKindOverrides?: Record<string, number>
}

Output: 7 Generated Files

~3,000 lines — All KSC AST types.

  • One interface per node kind: KSIdentifier, KSBinaryExpression, ...
  • KSNode = union of all ~360 node types
  • Sum type unions: KSExpression, KSStatement, KSDeclaration, ...
  • Type guard functions: isExpression(), isStatement(), etc.
// Example generated interface
export interface KSBinaryExpression {
  kind: 'BinaryExpression';
  left: KSExpression;
  operatorToken: KSToken;
  right: KSExpression;
  pos: number; end: number;
}

~2,500 lines — Schema metadata for every node kind.

  • getChildren(node) — Walk named child fields (schema-aware)
  • getChildFields(kind) — Field names for a given kind
  • fieldDefs — Full field metadata per kind (type, optional, list)
  • allKinds: Set<string> — All registered node kinds
  • sumTypeMembership — Which sum types each kind belongs to

~2,500 lines — Central AST conversion dispatch.

  • convertNode(tsNode) → KSNode — Per-kind converters
  • Symbol lookup, import resolution, type checking
  • Comment attachment and JSDoc extraction
  • buildKSTree(tsProgram, depth) — Main entry point

~3,600 lines — Factory functions for all node types.

// Example generated builder

~750 lines — Schema-aware JSON serialization.

  • nodeToJSON() / nodeFromJSON() — Single node
  • treeToJSON() / treeFromJSON() — Full tree
  • Uses schema fieldDefs to determine which fields to serialize

~700 lines — Type-level map from kind strings to node types.

Functor 2 Analysis Compilation

compileAnalysis(AnalysisSpec) → CompiledAnalyzer — takes structural attributes + property declarations and generates the evaluator with all equations statically wired.

Input: AnalysisSpec

Structural Attributes (5)

immutable — Flags var/let bindings

noSideEffects — Flags side-effect statements

noMutation — Flags assignment/mutation ops

noIO — Flags I/O module imports

Attribute Derivation: 5 + 8 → 22

Each of the 8 properties mechanically generates 3 attributes: a context (inh), a violation (syn), and a contribution to allViolations (collection).

The Generated Evaluator

▶ KSCDNode class — the decorated tree node
class KSCDNode implements Ctx {
  readonly node: KSNode;          // Raw AST node (never mutated)
  readonly parent: KSCDNode | undefined;
  readonly children: readonly KSCDNode[];

  // Typed cache fields (one per attribute)
  private _c_kindDefs: KindDefinition[] | undefined;
  private _c_defEnv: Map<string, KindDefinition> | undefined;
  // ... 20 more typed cache fields

  // Generic accessor
  attr<K extends keyof KSCAttrMap>(name: K): KSCAttrMap[K];

  // Typed methods — one per attribute
  kindDefs(): KindDefinition[] { /* switch on node.kind, call equations */ }
  defEnv(): Map<string, KindDefinition> { /* root case vs. parent */ }
  noImportsContext(): KindDefinition | null { /* check parent annotations */ }
  importViolation(): CheckerDiagnostic | null { /* context → violation */ }
  allViolations(): CheckerDiagnostic[] { /* aggregate all violations + children */ }
  // ...
}

Key design: Each attribute method is generated with a specific evaluation strategy:

  • Synthesized (syn): switch on node.kind, dispatch to equation function
  • Inherited (inh): root has special case, children read from parent
  • Collection: aggregate across children + self
  • All attributes are lazy-cached: computed once, then memoized
▶ evaluate() — the public entry point
export function evaluate(root: KSNode): EvaluationResult {
  const counter = createDefIdCounter();
  const dnode = buildKSCTree(root, counter);
  const definitions = project_binder(dnode);        // Extract KindDefinitions
  const diagnostics = analysisSpec.projections.checker(dnode);  // Extract violations
  return {
    definitions,
    diagnostics,
    getDepGraph: () => KSC_STATIC_DEP_GRAPH,
  };
}

Decoupling Boundaries

The architecture has 4 clean interfaces that serve as decoupling boundaries. Components on either side can evolve independently as long as the boundary contract holds.

Why Boundaries Matter

Each boundary is a typed interface (GrammarSpec, AnalysisSpec, KSProgramInterface). Components on either side can change freely as long as the interface contract holds.

No Direct Coupling

Grammar and Analysis don't import each other. They're connected only through string literal kind names, validated at build time by validateSpec(). Swapping the grammar doesn't break the analysis.

How a Property Works

The violation() Builder

▶ Type-safe violation rule factory
// compiler/violation.ts
export function violation<K extends string & keyof KindToNode>(config: {
  triggerKind: K;                                    // Narrows to literal
  predicate: (ctx: Ctx, node: KindToNode[K]) => boolean;
  message: (ctx: Ctx, node: KindToNode[K], def: KindDefinition) => string;
}): ViolationRule

// Usage in analysis/spec.ts:
violation<'ImportDeclaration'>({
  triggerKind: 'ImportDeclaration',
  predicate: () => true,                           // Always triggers
  message: (_, node, def) =>
    `Kind '${def.name}' forbids imports: '${node.moduleSpecifier?.value}'`,
})

KindToNode from generated/ast/kind-map.ts provides compile-time type narrowing — the node parameter is automatically typed to e.g. KSImportDeclaration.

All 8 Properties at a Glance

noImports

Flags ImportDeclaration nodes within annotated scope.

noConsole

Flags CallExpression where callee is console.*.

immutable

Flags VariableDeclaration with var/let (not const).

static

Flags CallExpression for dynamic import() calls.

noSideEffects

Flags CallExpression/AwaitExpression/YieldExpression as statements.

noMutation

Flags assignment/mutation operators (+=, ++, delete).

noIO

Flags imports from I/O modules (fs, http, net, ...).

pure

Placeholder — transitive call-graph purity (Phase 6).

Codegen Scripts

Thin wrappers that build a spec, call the pure functor, and write files. The actual logic lives entirely in the compiler.

The full pipeline: compile grammar, validate, compile analysis.

// scripts/codegen-all.ts
// Step 1: Functor 1 — grammar → CompiledGrammar
const grammar = compileGrammar(grammarSpec);
writeFiles(grammar.files, 'generated/ast/');

// Step 2: Cross-validate
const diags = validateSpec(analysisSpec, grammar);
if (diags.length > 0) throw new Error(...);

// Step 3: Functor 2 — analysis → CompiledAnalyzer
const analyzer = compileAnalysis(analysisSpec);
writeFile(analyzer.evaluatorFile, 'generated/');
writeFile(analyzer.attrTypesFile, 'generated/');

Runs Functor 1 only.

// scripts/codegen-grammar.ts
loadNodeDefinitions();              // Execute grammar/nodes.ts
const spec = buildGrammarSpec();    // From registry → GrammarSpec
const result = compileGrammar(spec);
for (const file of result.files) {
  writeFileSync(`generated/ast/${file.name}`, file.content);
}

Runs Functor 2 only.

// scripts/codegen-analysis.ts
import { analysisSpec } from '../analysis/spec';
const result = compileAnalysis(analysisSpec);
writeFileSync('generated/evaluator.ts', result.evaluatorFile.content);
writeFileSync('generated/attr-types.ts', result.attrTypesFile.content);
// Reports: 22 attributes, 27 dep edges

Adding a New Property (3 Steps)

  1. Add PropertyDecl to analysis/spec.ts with violation rules
  2. Add to PropertySet + PROPERTY_KEYS in analysis/types.ts
  3. Run npx tsx scripts/codegen-analysis.ts — context/violation/allViolations attrs auto-generated

No boilerplate context equations needed. All mechanical derivation happens in the compiler.
