Two-Functor Architecture

KSC's specialization pipeline as two typed functors: Grammar compilation and Analysis compilation. Each stage takes explicit inputs, produces explicit outputs, and the dependency between them is a function parameter — not an import path.

2. Three Inputs, Two Stages

KSC has three inputs that determine its behavior. The two-functor design processes these inputs at different times.

Build-time Grammar

AST node kinds, fields, sum type hierarchy. Defined in schema.ts. Processed by Functor 1 at build time.

364 node kinds, each with typed fields and parent-child relationships. Changes rarely.

Build-time Analysis

AG equations: structural attributes (binder) + property enforcement (checker). Defined in spec.ts. Processed by Functor 2 at build time.

22 attributes, 8 properties, each bundling its violation rules. Changes when adding new checks.

Runtime Program

TypeScript source files to analyze. Provided at runtime by the user. Not known at build time.

The "dynamic input" in partial evaluation terms. Each run gets a different program.

The Core Insight

Grammar and Analysis are static inputs — they don't change between runs. The two-functor architecture specializes the evaluator on these static inputs at build time, producing a residual program (the generated evaluator) that only needs the dynamic input (the TS program) at runtime.

This is exactly the first Futamura projection: mix(interpreter, static_input) → residual_program.

3. Functor 1: Grammar Compilation

Takes a grammar spec, produces a CompiledGrammar — typed runtime capabilities and generated source files.

interface GrammarSpec {
  nodes:           ReadonlyMap<string, NodeEntry>;
  sumTypes:        ReadonlyMap<string, SumTypeEntry>;
  fieldExtractors: Record<string, Record<string, string>>;
}

interface CompiledGrammar {
  files:       GeneratedFile[];          // source files to emit
  kinds:       ReadonlySet<string>;      // all 364 kind strings
  childFields: (kind: string) => readonly string[];
  convert:     (program: ts.Program, depth: AnalysisDepth) => KSTree;
  hasKind(kind: string): boolean;        // validation helper
}

function compileGrammar(spec: GrammarSpec): CompiledGrammar;

The grammar spec is the existing schema.ts — the DSL that defines all TypeScript AST node kinds, their fields, and sum type hierarchies.

// schema.ts — the grammar definition (unchanged)

const PropertyAccessExpression = node('PropertyAccessExpression', {

4. Functor 2: Analysis Compilation

Takes a CompiledGrammar (from Functor 1) and an AnalysisSpec, produces a CompiledAnalyzer. The grammar dependency is an explicit function parameter.

Key Design: Properties and Rules Are One Concern

A property is its rules. noConsole means nothing without "detect console.x on PropertyAccessExpression." The context equation, violation detection, and collection wiring are all aspects of enforcing one behavioral constraint. They should be declared together in a single PropertyDecl.

interface AnalysisSpec {
  structural: StructuralAttr[];   // binder + kindAnnotations (4 attrs)
  properties: PropertyDecl[];     // checker properties (8 decls)
}

interface StructuralAttr {
  name:      string;
  direction: 'syn' | 'inh' | 'collection';
  type:      string;
  deps:      string[];            // explicit (only ~4 of these)
  equations: EquationSet;         // functions, not data
}

interface PropertyDecl {
  name:          string;          // 'noConsole'
  annotationKey: string;          // key in PropertySet to check
  violations:    ViolationRule[]; // the detection rules
}

function compileAnalysis(
  grammar: CompiledGrammar,       // ← grammar is explicit input!
  spec:    AnalysisSpec,
): CompiledAnalyzer;

4 Attributes The structural attributes are the binder system — they resolve kind definitions and annotations. Their dependencies are explicit in the spec.

These 4 attributes form the binder chain. They parse kind definitions from the source, build a definition environment, and resolve annotations on variable declarations. All 8 property context attributes depend on kindAnnotations.

8 Properties Each property bundles its name, annotation key, and violation rules into one declaration. The functor generates all AG plumbing from this.

{
  name: 'noConsole',
  annotationKey: 'noConsole',
  violations: [
    violation({
      triggerKind: 'PropertyAccessExpression',
      predicate: (ctx, pae) => {
        if (pae.expression.kind !== 'Identifier') return false;
        return (pae.expression as KSIdentifier).escapedText === 'console';
      },
      message: (ctx, pae, def) =>
        `console.${(pae.name as KSIdentifier).escapedText} violates ${def.name} (noConsole)`,
    }),
  ],
}

The developer only writes the violation predicate — the domain-specific detection logic. Everything else is generated.

6. Dependency Graph Derivation

The dep graph is fully derived from spec structure — zero regex scanning. 3 edges are explicit in the structural spec. 24 edges are mechanically derived from property declarations.

The dep graph is built by two rules — no scanning needed:

function buildDepGraph(spec: AnalysisSpec): DepEdge[] {
  const edges: DepEdge[] = [];

  // Rule 1: Structural deps — explicit in spec (3 edges)
  for (const attr of spec.structural) {
    for (const dep of attr.deps) {
      edges.push([attr.name, dep]);
    }
  }

  // Rule 2: Property deps — mechanical derivation (24 edges)
  for (const prop of spec.properties) {
    const contextAttr = `${prop.name}Context`;

    // Context always depends on kindAnnotations
    edges.push([contextAttr, 'kindAnnotations']);

    // Each violation depends on its context
    for (const rule of prop.violations) {
      const violationAttr = violationAttrName(prop, rule);
      edges.push([violationAttr, contextAttr]);

      // allViolations collects each violation
      edges.push(['allViolations', violationAttr]);
    }
  }

  return edges; // 3 + 24 = 27 edges, zero regex
}

7. Type Safety via KindToNode

Functor 1 generates a type-level mapping from kind strings to node types. The violation() builder uses this for full type inference — no unsafe casts on trigger nodes.

Generated by Functor 1: kind-map.ts

// ast-schema/generated/kind-map.ts
export type KindToNode = {
  'Identifier':                  KSIdentifier;
  'PropertyAccessExpression':     KSPropertyAccessExpression;
  'BinaryExpression':             KSBinaryExpression;
  'VariableDeclarationList':      KSVariableDeclarationList;
  // ... all 364 kinds
};

The violation() Builder — Type Inference in Action

function violation<K extends string & keyof KindToNode>(config: {
  triggerKind: K;
  predicate:   (ctx: Ctx, node: KindToNode[K]) => boolean;
  message:     (ctx: Ctx, node: KindToNode[K], def: KindDefinition) => string;
}): ViolationRule;

// Usage — TypeScript infers K from triggerKind:
violation({
  triggerKind: 'PropertyAccessExpression',
  //           ↑ K inferred as 'PropertyAccessExpression'
  predicate: (ctx, pae) => {
    //              ↑ pae: KSPropertyAccessExpression (auto-narrowed!)
    return pae.expression.kind === 'Identifier';
    //         ↑ .expression is typed — full IDE autocomplete
  },
  message: (ctx, pae, def) => `...`,
});

No unsafe casts on trigger nodes. The type parameter K flows from triggerKind through KindToNode[K] to narrow the node parameter in both predicate and message.

10. Futamura Projection Mapping

The two-functor architecture maps cleanly to the Futamura projections from partial evaluation theory.

1st Projection

mix(interp, spec) → program

KSC: compileAnalysis(grammar, analysisSpec) → evaluator

Specializes the generic AG evaluation strategy on a specific set of equations. The evaluator is the residual program.

2nd Projection

mix(mix, interp) → compiler

KSC: compileGrammar(grammarSpec) then compileAnalysis(compiledGrammar, ·)

The analysis functor itself is a compiler — takes any analysis spec and produces an evaluator.

3rd Projection

mix(mix, mix) → cogen

KSC: The pipeline function itself

Takes any grammar spec + any analysis spec and produces an analyzer. Applied to KSC's inputs, it produces KSC.

Fixpoint

mix(mix, mix) → mix

KSC: Self-application

The pipeline applied to its own structure would reproduce the pipeline. The for (item of spec) emit(template(item)) loop is the irreducible unit.

▶ The Pipeline: Putting It All Together 
// ksc-compiler/pipeline.ts

import { compileGrammar } from './compile-grammar.js';
import { compileAnalysis } from './compile-analysis.js';
import { TS_GRAMMAR } from '../ast-schema/schema.js';
import { KSC_ANALYSIS } from '../ksc-analysis/spec.js';

// Functor 1: specialize on grammar
const grammar = compileGrammar(TS_GRAMMAR);

// Functor 2: specialize on analysis (grammar is explicit input)
const analyzer = compileAnalysis(grammar, KSC_ANALYSIS);

// Build time: emit generated files
grammar.emit('./ast-schema/generated');
analyzer.emit('./ksc-generated');

// Runtime: analyze a program
export function analyze(
  program: ts.Program,
  depth?: AnalysisDepth
): AnalysisResult {
  return analyzer.analyze(program, depth);
}
▶ Build-Time Validation

Because Functor 2 receives the compiled grammar as a parameter, it can validate all kind references at build time:

// Inside compileAnalysis():
for (const prop of spec.properties) {
  for (const rule of prop.violations) {
    if (!grammar.hasKind(rule.triggerKind)) {
      throw new Error(
        `Property '${prop.name}': violation references ` +
        `unknown kind '${rule.triggerKind}'`
      );
    }
  }
}

11. Quiz

Click each question to reveal the answer.

What are the two functors in the architecture?
Functor 1: compileGrammar() — takes a GrammarSpec (AST node definitions), produces a CompiledGrammar with generated type files and runtime capabilities.
Functor 2: compileAnalysis() — takes the CompiledGrammar + an AnalysisSpec (structural attrs + property declarations), produces a CompiledAnalyzer with the generated evaluator.
How does the dep graph avoid regex scanning?
The spec structure implies the dep graph. Structural attributes declare their deps explicitly (3 edges). Property declarations mechanically derive 24 edges: each context depends on kindAnnotations, each violation depends on its context, and allViolations depends on all violations. Zero regex scanning needed.
What is a PropertyDecl and what does the functor generate from it?
A PropertyDecl bundles a property name, annotation key, and violation rules. From it, the functor generates: (1) an inherited context equation, (2) a null root value, (3) a synthesized violation equation wrapping the predicate, (4) allViolations wiring, (5) KSCAttrMap type entries, and (6) 3 dependency edges. The developer only writes the violation predicate.
How does KindToNode provide type safety for violation predicates?
Functor 1 generates a type-level map from kind strings to node types. The violation<K>() builder infers K from the triggerKind string literal, then uses KindToNode[K] to narrow the node parameter in predicate and message. This gives full IDE autocomplete and eliminates unsafe casts on trigger nodes.
How many files/lines does adding a new property require before vs after?
Before: 5 files, ~25 lines — add to PropertySet, PROPERTY_KEYS, write context equation (boilerplate), write violation equation, add to allViolations, add attr-types entries.
After: 1 file, ~8 lines — add a single PropertyDecl to spec.ts with the violation predicate. Everything else is generated.
Why is the grammar dependency a function parameter rather than an import path?
Making the grammar an explicit parameter to compileAnalysis(grammar, spec) means: (1) the coupling is typed and visible, (2) the functor can validate kind references at build time, (3) the grammar can be swapped (different language grammars, test grammars), and (4) the data flow matches the Futamura projection structure where mix(program, static_input) → residual.
What Futamura projection does compileAnalysis correspond to?
The 1st Futamura projection: mix(interpreter, spec) → residual_program. The analysis functor specializes the generic AG evaluation strategy on a specific set of equations (the AnalysisSpec), producing the evaluator as a residual program that only needs the runtime input (the TS program).
What happens if a violation references a non-existent AST kind?
Build-time error. The functor validates all triggerKind values against the compiled grammar's kind set. If you typo 'ProeprtyAccessExpression', you get an immediate error: "Property 'noConsole': violation references unknown kind 'ProeprtyAccessExpression'". In the old system, this would silently miss violations at runtime.
