1. The Theory: What Cousot Actually Said

Foundations

Patrick and Radhia Cousot didn't define 5 levels. They defined a mathematical framework in which abstractions form a complete lattice — meaning there are infinitely many valid abstractions, and any two can be composed, compared, or combined.

Key Papers & What They Established

1977 — POPL
"Abstract Interpretation: A Unified Lattice Model for Static Analysis of Programs by Construction or Approximation of Fixpoints"
The founding paper. Programs compute fixpoints of monotone operators on complete lattices. An abstract interpretation approximates these fixpoints in a simpler lattice connected to the concrete one via a Galois connection (α, γ). Proved: if the abstract domain satisfies the ascending chain condition, the analysis terminates. If not, widening forces convergence.
1979 — POPL
"Systematic Design of Program Analysis Frameworks"
The lattice of abstractions: all sound abstract interpretations of a given program form a complete lattice, ordered by precision. The reduced product combines two domains into one that is at least as precise as either. This is the paper that makes the "hierarchy" continuous rather than discrete.
1992 — Journal of Logic and Computation
"Abstract Interpretation Frameworks"
Generalized the theory beyond Galois connections to upper closure operators, handling infinite-height lattices more cleanly.
1997 — POPL
"Types as Abstract Interpretations"
Showed that type systems are a specific instance of abstract interpretation — type checking is an abstract interpreter using a very coarse domain. This bridges PL theory and static analysis into one framework.
2002 — VMCAI
"Constructive Design of a Hierarchy of Semantics of a Transition System by Abstract Interpretation"
The hierarchy of program semantics itself is a chain of abstractions: trace semantics → relational semantics → denotational semantics → etc. Each is an abstraction of the one above.
2019 — POPL (Distinguished Paper)
"A2I: Abstract² Interpretation"
Higher-order abstract interpretation — abstracting the abstract interpreter itself.
2021 — MIT Press
Principles of Abstract Interpretation (832 pages)
The definitive textbook. Covers trace semantics, fixpoints, abstract domains, widening/narrowing, relational/symbolic analysis. Exercise solutions on GitHub.

The Actual Lattice (Not 5 Levels)

The "5 levels" from the earlier diagram are pedagogical landmarks, like base camps on a mountain. The real structure is:

The Lattice of Abstract Interpretations (Cousot & Cousot 1979) Given a concrete domain C, the set of all abstract interpretations of C forms a complete lattice (Abs(C), ⊑) where:
• ⊤ = the trivial abstraction (maps everything to one value)
• ⊥ = the concrete semantics itself (identity Galois connection)
• ⊑ = "is at least as precise as"
• Meet = reduced product (most precise common refinement)
• Join = most abstract common abstraction
Between any two points, there are infinitely many other abstractions.

Why "5 Levels" Is Still Useful

The continuum has natural clusters based on computational complexity:

1

Maximal Trace Semantics

All execution paths, all intermediate states. Undecidable.

Keeps: Every possible execution trace, including non-terminating ones.
Loses: Nothing — this is the concrete semantics.
Complexity: Undecidable (equivalent to the halting problem in general).
In between levels 1→2: Natural semantics, small-step SOS, denotational semantics, big-step trace semantics, angelic/demonic variants — each discards some ordering or branching info from traces while keeping more than just reachable states.
↓ α (forget ordering within traces)
2

Collecting Semantics (Reachable States)

Set of all reachable states at each program point. Still undecidable, but the standard "concrete" target.

Keeps: Which values each variable can take at each program point.
Loses: The order in which states are visited, correlations across time.
In between 2→3: Disjunctive completions, trace partitioning, powerset domains — each adds back some of the lost relational information using different strategies. The number of possible partitioning strategies is itself infinite.
↓ α (forget some inter-variable relations)
3

Relational Abstractions

Polyhedra, Octagons, Zones, Pentagons — track relations between variables.

Keeps: Relationships between pairs or groups of variables (e.g. x ≤ 2y + 3).
Loses: Exact sets of reachable values; only knows constraint shapes it can represent.
The sub-hierarchy within level 3 is well-studied:
• Polyhedra (Cousot & Halbwachs 1978): arbitrary linear constraints. Exponential worst-case.
• Two Variables Per Inequality (TVPI): constraints on at most 2 variables. Polynomial.
• Octagons (Miné 2001): constraints of form ±x ± y ≤ c. Cubic.
• Zones / DBMs: constraints x - y ≤ c. Cubic.
• Pentagons (Logozzo & Fähndrich): intervals + strict upper bounds.
Each is a distinct point in the lattice, with a strict precision ordering.
↓ α (forget all inter-variable relations)
4

Non-Relational Abstractions

Intervals, Congruences, Signs — track each variable independently.

Keeps: Bounds or properties of individual variables.
Loses: All relationships between variables (x and y analyzed independently).
Sub-hierarchy:
• Intervals (Cousot & Cousot 1976): x ∈ [a, b]. Very fast, very common.
• Congruences (Granger 1989): x ≡ a (mod m).
• Signs: x ∈ {neg, zero, pos}.
• Parity: x ∈ {even, odd}.
• Interval × Congruence reduced product: strictly more precise than either alone.
↓ α (forget numeric bounds, keep only type shape)
5

Type Abstractions

Only track the "type" of values: int, string, null, etc.

Keeps: The coarsest shape of data — "is this an integer or a pointer?"
Loses: All numeric information.
Cousot 1997: Type systems are abstract interpretations. A type checker is an abstract interpreter running over a very coarse domain. But even here there's a sub-hierarchy: simple types < generic types < refinement types < dependent types, each a different point in the lattice with different precision/cost tradeoffs.

Key Mechanisms

Galois Connection (α, γ)

The formal bridge between concrete and abstract. α (abstraction) maps concrete values to abstract ones; γ (concretization) maps back. The pair must satisfy: α(c) ⊑ a ⇔ c ⊑ γ(a).

Composition: if (α₁,γ₁) and (α₂,γ₂) are Galois connections, so is (α₂∘α₁, γ₁∘γ₂). This is how you chain abstractions.

Widening ▽

When the abstract domain has infinite ascending chains (like intervals [0,1], [0,2], [0,3], ...), iteration may not converge. A widening operator over-approximates the limit: x ▽ y ⊇ x ∪ y, and guarantees termination in finite steps.

Trade-off: widening loses precision to gain termination. Delayed widening and narrowing can recover some precision afterward.

Reduced Product

Given domains D₁ and D₂, their reduced product is the most precise domain that encodes the conjunction of information from both. It's not just the Cartesian product — it includes reduction steps where each domain's information constrains the other.

Astrée uses ~20 domains in a reduced product. This is the key to "right precision for each code region."

CEGAR (Counterexample-Guided Abstraction Refinement)

Start with a coarse abstraction. If it reports a potential bug, check if the counterexample is real. If spurious, refine the abstraction to eliminate that particular false alarm and re-analyze.

This is abstract interpretation's answer to "how do I pick the right abstraction?" — let the analysis failures guide you.

Why Zero False Alarms? Astrée doesn't use one domain for everything. It uses intervals for most code (cheap, sufficient), octagons for pairwise variable relations, polyhedra only for critical sections, and domain-specific domains like the digital filter domain (designed specifically for the signal processing patterns in avionics). This selective precision is what Cousot's framework enables — you compose exactly the abstractions you need.

How the Reduced Product Works in Astrée

// Conceptual architecture (simplified)

type AstreeState = {
  intervals:    IntervalDomain,      // [lo, hi] per variable
  octagons:     OctagonDomain,       // ±x ± y ≤ c
  polyhedra:    PolyhedraDomain,     // arbitrary linear constraints
  filters:      DigitalFilterDomain, // IIR/FIR filter bounds
  gauges:       GaugeDomain,         // loop counter relations
  trees:        TreeDomain,          // pointer/struct shapes
  // ~15 more...
}

// After each abstract transfer function, reduce:
function reduce(state: AstreeState): AstreeState {
  // Intervals inform octagons: if x ∈ [0,5], tighten ±x constraints
  // Octagons inform polyhedra: if x+y ≤ 10, add polyhedron constraint
  // Polyhedra inform intervals: if 2x+3y ≤ 15, tighten x and y bounds
  // All domains collaborate via reduction to eliminate spurious states
  return iterateReductions(state);
}

The CEGAR Loop

// SLAM's CEGAR algorithm (conceptual)

loop {
  // 1. ABSTRACT: Create boolean program (weakest abstraction)
  boolProg = predicateAbstraction(driver, predicates);

  // 2. CHECK: Model-check the boolean program
  result = modelCheck(boolProg, safetyProperty);

  if (result === "SAFE") return "VERIFIED";

  // 3. VALIDATE: Is the counterexample real?
  cex = result.counterexample;
  if (isFeasible(cex, driver)) return "BUG FOUND";

  // 4. REFINE: Spurious! Add new predicates to eliminate it
  newPredicates = interpolate(cex);
  predicates = predicates.union(newPredicates);
}
CEGAR vs. Astrée: Opposite Strategies Astrée starts with a rich combination of domains and aims for zero false alarms. SLAM starts with the weakest possible abstraction and only refines when forced. Both are valid instantiations of Cousot's framework — they just explore the lattice of abstractions in opposite directions.

Key Innovation: Bi-Abduction

Traditional abstract interpretation analyzes whole programs. Infer uses bi-abduction (from separation logic) to analyze each procedure independently:

// Bi-abduction discovers both:
//   PRECONDITION: what the function needs (anti-frame)
//   POSTCONDITION: what the function establishes (frame)

// Given a function that does: x->data = 42
// Bi-abduction infers:
//   PRE:  x ≠ null ∧ x ↦ {data: _}
//   POST: x ↦ {data: 42}

// This lets Infer analyze each function once,
// then compose summaries for interprocedural analysis.
Compositional = Incremental Because Infer analyzes functions independently and stores summaries, it only needs to re-analyze functions whose dependencies changed. This is how it handles Meta's scale — on a typical diff, it re-analyzes only the affected procedures, not the entire codebase.

What Makes MOPSA Different

MOPSA decomposes analysis into the smallest possible domains: each domain handles one construct or one abstraction. An analysis is assembled from these micro-domains via reduced products. This means:

  • Swapping in a different numerical domain = change one module
  • Adding support for a new language feature = add one micro-domain
  • Multi-language analysis (C calling Python) = compose the respective domain sets
Sound by Design MOPSA is sound: if it reports no errors, there truly are none. It also features an interactive abstract debugger for stepping through the analysis to understand imprecisions — important for research use.

5. Abstract Interpretation for Neural Networks

Neural Networks

The newest frontier: using Cousot's framework to prove properties of neural networks. Can you guarantee that a self-driving car's vision model won't misclassify a stop sign if pixels are perturbed by ε?

The Key Idea

A neural network is a function f: Rn → Rm. Each layer applies affine transforms + nonlinear activations (ReLU, sigmoid, tanh). Abstract interpretation propagates abstract shapes (boxes, zonotopes, polyhedra) through each layer to bound the output region.

// Neural network as a chain of abstract transformers

// Input: pixel values perturbed by ε
abstract_input = Box([x1 ± ε, x2 ± ε, ..., xn ± ε])

// Layer 1: affine transform (exact in any linear domain)
after_affine_1 = abstract_affine(abstract_input, W1, b1)

// Layer 1: ReLU activation (this is where precision matters)
after_relu_1 = abstract_relu(after_affine_1)  // key approximation step

// ... repeat for each layer ...

// Output: if abstract output region doesn't cross decision boundary
// ⇒ PROVEN ROBUST: no perturbation within ε can change the classification

Tools & Domains

Custom abstract domain for neural networks: combines floating-point polyhedra with intervals. Each neuron gets an upper and lower affine bound in terms of the input.

Same Framework, New Domain Neural network verification is exactly Cousot's framework applied to a new program class. The "program" is the neural network, the "concrete semantics" is the set of all possible outputs for all inputs in a region, and the "abstract domain" (DeepPoly, CROWN, Zonotopes) over-approximates that set. Widening isn't needed (networks have fixed depth), but the precision/cost tradeoff is the same: boxes are fast but imprecise; DeepPoly is tighter but more expensive; and branch-and-bound refines further at exponential cost.

7. Design Patterns Across Implementations

Patterns

Looking across all these implementations, several architectural patterns emerge:

Pattern 1: Domain Parameterization

Every serious tool makes the abstract domain a parameter, not hardcoded. This is the single most important lesson from Cousot's framework.

// APRON: swap domain in one line
let man = Box.manager_alloc ()     (* intervals: fast *)
let man = Oct.manager_alloc ()     (* octagons: medium *)
let man = Polka.manager_alloc_strict () (* polyhedra: precise *)

(* Rest of the analyzer code is identical *)

Pattern 2: Reduced Product Composition

Don't pick one domain. Combine many, and let them strengthen each other via reduction.

  • Astrée: ~20 domains in collaborative reduced product
  • Frama-C/EVA: modular stack of domains with reduction
  • MOPSA: micro-domain constellation with pairwise reduction

Pattern 3: Selective Precision

Apply expensive domains only where needed. This is what makes Astrée practical on 1M+ LoC:

  • Most code: intervals (cheap, usually sufficient)
  • Loop bounds: gauges + octagons
  • Critical computations: polyhedra (expensive but precise)
  • Domain-specific patterns: custom domains (e.g., digital filters)

Pattern 4: CEGAR vs. Fixed Domain

Fixed Rich Domain (Astrée-style)CEGAR (SLAM-style)
StrategyStart precise, hope it's fast enoughStart cheap, refine on demand
False alarmsZero (if domain rich enough)Zero (if refinement succeeds)
PerformancePredictable but heavyVariable (depends on refinement)
Best forKnown domain with domain-specific patternsGeneric properties on diverse code

Pattern 5: Compositionality for Scale

Infer's key insight: analyze functions independently, compose summaries. This turns abstract interpretation from a whole-program analysis into an incremental, diff-time analysis. Without this, abstract interpretation can't handle Meta-scale codebases.

Check Your Understanding

Astrée achieved zero false alarms on the A380 flight software. What was the key technique?

Correct! Astrée combines ~20 abstract domains (intervals, octagons, polyhedra, digital filter domains, gauges, etc.) in a reduced product, applying expensive domains only to code regions that need them. This selective precision is what achieves zero false alarms at practical cost.
Not quite. Polyhedra everywhere would be too expensive, and CEGAR/bi-abduction are techniques from SLAM and Infer respectively. Astrée's key is the reduced product of ~20 domains with domain-specific abstractions (like digital filter domains) applied selectively.

Check Your Understanding

What does ELINA provide that APRON doesn't?

Correct! ELINA is API-compatible with APRON but uses online decomposition and Intel SIMD instructions to achieve 10-100× speedups on the same domains (octagons, polyhedra, zones). Same abstractions, much faster computation.
Not quite. ELINA provides roughly the same domains as APRON, with the same API. Its innovation is dramatically faster implementations (10-100×) through online decomposition and vectorization (Intel SSE/AVX).

Check Your Understanding

According to Cousot, how many "levels" exist in the hierarchy of abstractions?

Correct! Cousot & Cousot (1979) proved that all sound abstractions of a given program form a complete lattice. Between any two abstractions, there are infinitely many others. The "5 levels" are pedagogical landmarks (like base camps on a mountain), not a fixed structure from the theory.
The "5 levels" (or any fixed number) is a pedagogical simplification. Cousot & Cousot proved in 1979 that abstractions form a complete lattice with infinitely many points. You can always find a new useful abstraction between two existing ones (e.g., Pentagons sit between intervals and octagons).
