From AG Interpreter to AG Compiler

Where KSC stands today with its DNode architecture, what remains to be generated, and the concrete stages to get to a fully typed, schema-driven AG compiler.

Where We Are: The DNode Architecture

KSC has already completed a major architectural shift: the raw AST is never mutated. Instead, a separate DNode wrapper tree provides navigation and attribute access. This is Silver-style Strategy 4 — the same approach used by production AG systems.

What's already solved: The raw AST is never mutated. Navigation (parent, children, siblings, fieldName) lives in separate DNode wrappers. No Object.defineProperty hacks. No enumerable: false tricks. No $parent, $agAttrs, or stampTree(). Attribute caching uses simple Maps. The raw tree can be evaluated multiple times with different specs.
What remains: Everything is still generic and dynamic. Attribute access is ctx.attr<T>('name') where T must be manually specified (returns any if you forget). Equations are Record<string, unknown>. Dispatch is runtime hash-map lookup. The cache is Map<string, unknown>. Two serializers exist, neither complete. The type system doesn't know what attributes exist or what types they have.

The Four-Module Architecture

KSC separates concerns into four layers. Understanding these is essential for seeing where codegen fits in.

What Remains Dynamic (and Shouldn't Be)

The DNode architecture solved the mutation problem. But the system is still a pure interpreter: everything is wired up at runtime from generic data structures. Here's what could be pre-computed:

Operation Current Could Be Impact
Attribute dispatch (kind → equation) Runtime hash-map in makeDispatch() Generated switch(node.kind) Faster, typo-safe
Attribute type information ctx.attr<T>('name') — T is manual Typed: ctx.attr('kindDefs') → KindDefinition[] IDE support, safety
Cache structure Map<string, unknown> per DNode Typed cache object with one field per attr V8 optimization, types
Equation typing equations: Record<string, unknown> Typed per-attribute equation signatures Spec author safety
Attribute registration Walk entire DNode tree, register each attr Baked into DNode construction Faster startup
Serialization (attributes) Two serializers, one doesn't know attrs One generated serializer, full knowledge Correctness, simplicity
Grammar + Semantics creation Rebuilt every evaluate() call Generated as static code Smaller, faster
Actual attribute values Computed on demand (lazy) Still computed on demand Must stay runtime
Key insight: Only the concrete tree and actual attribute values must be computed at runtime. Everything else — which attributes exist, their types, how dispatch works, how caching works, how serialization works — is known statically from the schema (364 kinds) and specs (12 attributes).

Partial Evaluation & Staging

What is partial evaluation?

Partial evaluation (or specialization) is pre-computing everything that depends only on known inputs, producing a simpler program that only needs the remaining input.

The Futamura projections formalize this: specializing an interpreter with respect to a program produces a compiled version. Specializing the AG interpreter with respect to the schema + specs produces a compiled AG evaluator.

What JastAdd does (the reference AG compiler)

JastAdd generates Java classes at build time from grammar + attribute declarations. Every attribute becomes a real method with the correct return type:

// Grammar declaration
abstract Statement;
IfStatement : Statement ::=
  Condition:Expression
  Then:Statement
  [Else:Statement];

// Attribute declaration
syn List<KindDef> CompilationUnit.kindDefs();
inh Map<String, KindDef> ASTNode.defEnv();

// Equation
eq CompilationUnit.kindDefs() {
  return extractKindDefs(this);
}

// GENERATED by JastAdd at build time
public class IfStatement extends Statement {
  // Schema fields (from grammar)
  private Expression condition;
  private Statement then;
  private Statement else_;

  // Navigation (typed, explicit)
  private ASTNode parent;
  private int childIndex;

  // Binder attribute: kindDefs (typed, cached)
  private boolean kindDefs_computed = false;
  private List<KindDef> kindDefs_value;
  public List<KindDef> kindDefs() {
    if (kindDefs_computed) return kindDefs_value;
    kindDefs_computed = true;
    kindDefs_value = /* equation body */;
    return kindDefs_value;
  }
}

// KSC today: DNode-based interpreter
const dnode: DNode<KSNode> = ...;

// Navigation: typed via DNode
dnode.parent       // DNode | undefined  (typed!)
dnode.children     // readonly DNode[]   (typed!)
dnode.fieldName    // string | undefined (typed!)

// Data access: typed via .node
dnode.node.kind    // string
dnode.node.pos     // number

// Attribute access: NOT typed
dnode.attr('kindDefs')       // returns any!
dnode.attr<KindDefinition[]>('kindDefs')  // manual cast

// What we want:
dnode.attr('kindDefs')       // => KindDefinition[]  (auto!)

Current Interpreter vs Target Compiler

How it works today (DNode architecture)

The DNode architecture is clean — no mutation — but all wiring (dispatch, caching, registration) happens at runtime from generic data structures. Every evaluate() call rebuilds Grammar, Semantics, compiles all 12 attributes, walks the tree to register them.

Target: generated evaluator

Build time handles all structural wiring. Runtime only runs equation bodies over the concrete tree. Equations are still hand-written — codegen handles dispatch, caching, types, and serialization.

AspectCurrent (DNode Interpreter)Target (Generated Evaluator)
Tree mutation None (DNode wrappers) None (generated DNode)
Navigation typing Typed via DNode class Typed via generated DNode
Attribute types attr<T>('name') — T manual attr('name') → typed automatically
Equation safety Record<string, unknown> Typed per-attribute signatures
Dispatch Runtime hash-map closure Generated switch/if
Caching Map<string, unknown> Typed cache object
Serialization Two serializers, incomplete One generated serializer
IDE support No autocomplete for attrs Full autocomplete
Startup cost Rebuild Grammar+Semantics each call Static generated code
Flexibility Specs can change at runtime Regenerate on spec change

Recommended Stages

Each stage is independently valuable and builds toward full codegen. The stages are ordered by impact and complexity — each one unlocks the next.

Stage 0: DNode Architecture Done

Immutable raw tree. DNode wrappers for navigation. Map-based attribute caching. No Object.defineProperty, no $parent mutation, no $agAttrs heuristic. This is complete.

Achieved

Clean separation of data/navigation/attributes. Raw tree reusable. No JS hacks.

Remaining Gap

Attribute dispatch and caching are still generic/dynamic. Next: generated dispatch tables and cache slots.

Stage 1: Attribute Type Registry Done

Created KSCAttrMap interface mapping all 12 attribute names to their value types. Added second type parameter A to DNode<N, A> with overloaded attr() for typed access. Binder and checker equations use type Ctx = DNode<KSNode, KSCAttrMap> — all manual <T> casts removed. This is complete.

Stage 2: Typed Spec Declarations Done

Added defineSpec() helper that validates equation keys match declaration keys and equation shapes match declaration directions. Uses const type parameter inference to derive declaration types, then EquationsFor<D> mapped type for validation. This is complete.

// ag-behavior/spec.ts
type EquationFor<D extends AttrDecl> =
  D extends SynDecl ? Function | ProductionMap
  : D extends InhDecl ? Function | undefined  // auto-propagate
  : D extends CollectionDecl ? Function | ProductionMap
  : ...;

type EquationsFor<D> = {
  [K in keyof D]: EquationFor<D[K]>;

export function defineSpec<const D, R>(spec: {
  declarations: D;
  equations: EquationsFor<D>;  // key alignment + shape validation
  ...
}): SpecInput;

// Catches: misspelled equation names, wrong equation shape (e.g. undefined for syn)

Achieved

Key alignment: misspelled equation names are compile errors. Shape validation: undefined rejected for syn, allowed for inh. Both binder and checker use defineSpec(). Zero runtime cost — it’s just a passthrough.

Files changed

ag-behavior/spec.ts (EquationFor, EquationsFor, defineSpec), ag-behavior/index.ts (export), ksc-behavior/binder.ts & checker.ts (use defineSpec).

Stage 3: Generated Dispatch Skipped

Decision: not worth it at current scale. Only 6 production-dispatched attributes across both specs, each handling 1–3 kinds out of 364. makeDispatch() creates a closure doing equations[key] — V8 inline-caches this after the first few calls. The performance difference vs switch is unmeasurable at this scale.

Why skip

Equation bodies contain closures and local state (e.g. nextDefId++) that resist extraction. Codegen would only generate the dispatch shell, not the equations — same indirection as makeDispatch, just with switch.

When it would matter

If the project grew to dozens of attributes × hundreds of handled kinds. Or for per-kind fused evaluation (compute ALL attributes for a given kind in one pass) — but that’s Stage 4/5 territory.

Stage 3: Unified Serialization Done

The generated schema serializer (ast-schema/generated/serialize.ts) already handles all 364 node kinds with typed fields. Rather than extending codegen to consume attribute declarations (which would break the ast-schema ↔ ksc-behavior boundary), we use composition: the schema serializer handles fields, attribute values are added at the ksc layer if needed.

Schema fields are per-kind (364 kinds × varying fields) — codegen pays off. Attributes are uniform (every node has the same 12) — a simple loop suffices.

Stage 4: Shared Attribute Definitions Done

Instead of generating a typed cache (12 attributes don’t justify it), the real win was eliminating the O(N×A) registerAttributes walk. All DNodes now share a single attribute definitions reference, passed at tree construction time. This is complete.

// Before (O(N×A)): walk every node, copy every attr def
const dnodeRoot = buildContextTree(root, getChildren, getChildFields);
registerAttributes(dnodeRoot, compiled);  // walks N nodes × A attrs

// After (O(0) extra): defs shared at construction time
const dnodeRoot = buildContextTree(root, getChildren, getChildFields, compiled);
// No registerAttributes walk needed!
// All DNodes point to the same shared defs ReadonlyMap.
// Per-node cache (Map<string, unknown>) stays — 12 attrs don't justify a typed cache.

Achieved

Eliminated O(N×A) registerAttributes walk. All DNodes share a single ReadonlyMap<string, AttrComputed> reference. interpret() passes compiled defs directly to buildContextTree. registerAttributes kept for backward compat (legacy path).

Files changed

ag-interpreter/dnode.ts (shared defs via constructor), ag-interpreter/interpret.ts (pass defs at build time).

Stage 5: Full Generated Evaluator Done

A codegen script (scripts/gen-ksc-evaluator.ts) introspects the binder and checker specs, reads attribute type info from ksc-behavior/attr-types.ts, and generates a fully specialized evaluator (ksc-generated/evaluator.ts, 454 lines). All 5 compilation targets achieved:

1. Pre-compiled attribute defs

No compile() at runtime. Equations are extracted from specs and wired directly into KSCDNode getters. The generic compilation layer (compileSyn, compileInh, makeDispatch) is bypassed entirely.

2. Switch-based dispatch

Production equations use switch(node.kind) instead of runtime hash-map closures. 6 attributes with production equations generate switch statements with 1–3 cases each.

3. Typed cache

KSCDNode has one typed field per attribute (_c_kindDefs: KindDefinition[] | undefined) instead of Map<string, unknown>. V8 optimizes fixed-shape objects better than Maps.

4. Typed accessors

attr() is overridden with a switch that dispatches to typed getter methods (_get_kindDefs(), etc.). No string-keyed Map lookup. Each getter returns the correct type with proper TypeScript narrowing.

5. Generated tree builder

buildKSCTree() creates KSCDNode instances (not generic DNode) using schema-aware traversal via getChildFields(). No generic getChildren fallback needed for schema-known nodes.

// GENERATED: ksc-generated/evaluator.ts (454 lines)
// KSCDNode extends DNode with typed cache + switch dispatch
class KSCDNode extends DNode<KSNode, KSCAttrMap> {
  // Typed cache — one field per attribute
  private _c_kindDefs: KindDefinition[] | undefined;
  private _c_defEnv: Map<string, KindDefinition> | undefined;
  // ... all 12 attributes, each with correct type

  // Switch dispatch (replaces Map lookup + AttrComputed closures)
  attr(name: string): any {
    switch (name) {
      case 'kindDefs': return this._get_kindDefs();
      case 'defEnv': return this._get_defEnv();
      // ...
    }
  }

  // Typed getter with switch-based production dispatch
  private _get_kindDefs(): KindDefinition[] {
    if (this._c_kindDefs !== undefined) return this._c_kindDefs;
    switch (this.node.kind) {
      case 'CompilationUnit':
        return this._c_kindDefs = this._eqs.kindDefs_CompilationUnit(this, this.node);
      default:
        return this._c_kindDefs = this._eqs.kindDefs_default(this, this.node);
    }
  }
}

What’s preserved

Equation bodies are still hand-written in binder.ts and checker.ts. The generic ag-interpreter module is untouched — still available for prototyping and tests. Lazy dep graph falls back to generic compile() + analyzeDeps() on demand.

Files

scripts/gen-ksc-evaluator.ts (codegen script, introspects specs at build time), ksc-generated/evaluator.ts (454 lines, generated output), ksc-interpreter/evaluate.ts (re-exports from generated module).

Status: All stages complete (0–5). The production evaluate() uses a fully generated evaluator: KSCDNode with typed cache (object fields, not Map), switch-based dispatch (not hash-map closures), typed accessor methods, and a schema-aware tree builder. No createGrammar, createSemantics, compile, or interpret on the hot path. The generic ag-interpreter remains available for prototyping. 131 tests passing.

Review

What did the DNode architecture solve?
DNode eliminated AST mutation. The raw tree is never modified — navigation (parent, children, siblings, fieldName) lives in separate DNode wrappers. Attribute caching uses Maps instead of Object.defineProperty. No $parent, $agAttrs, or enumerable: false hacks. This is Silver-style Strategy 4.
What problems remain after Stages 0–2?
Attribute access and spec declarations are now type-safe (Stages 1–2). Remaining dynamic aspects were addressed in later stages: unified serialization (Stage 3), shared attribute definitions (Stage 4), and the specialized evaluator (Stage 5) that eliminated Grammar/Semantics/interpret overhead. All stages are now complete.
What is partial evaluation / staging?
Pre-computing everything that depends only on known (static) inputs, producing specialized code that only needs the remaining (dynamic) inputs. In KSC: the schema (364 kinds) and specs (12 attributes) are static. Only the concrete tree is dynamic. Codegen can pre-compute all dispatch, caching, types, and serialization.
What’s the current status of the staging roadmap?
All stages complete. Stages 0–5 are done: DNode architecture, type registry, typed specs, unified serialization, shared attribute definitions, specialized evaluator. Generated dispatch was skipped (not worth it at current scale). The generic ag-interpreter remains available for prototyping; production uses the specialized evaluate() pipeline.
What does the full generated evaluator (Stage 5) do?
A codegen script (scripts/gen-ksc-evaluator.ts) generates KSCDNode — a DNode subclass with typed cache (object fields, not Map), switch-based dispatch (not hash-map closures), and typed accessor methods. Plus a schema-aware tree builder and specialized evaluate(). No compile(), createGrammar, createSemantics, or interpret on the hot path. Equation bodies stay hand-written; all wiring is generated.
Why keep the generic interpreter alongside generated code?
The interpreter is valuable for rapid prototyping. When developing a new spec or adding attributes, you don’t want to regenerate code on every change. The interpreter lets you iterate quickly. Once the spec stabilizes, run npx tsx scripts/gen-ksc-evaluator.ts to regenerate. Tests use the generated evaluator (via evaluate()) and still pass — proving the generated code matches the generic interpreter’s semantics.
How does unified serialization work?
Today there are two serializers: the schema serializer knows node fields but not attributes, the AG serializer can dump attributes but doesn't know field types. With codegen that reads both schema (364 kinds with fields) AND spec declarations (12 attributes with types), a single serializer handles everything: named child fields, scalar props, and typed attribute values with per-attribute serialization logic.
What's the key property of the migration path?
Each stage is independently valuable and keeps the generic interpreter working. Generated code is an optimization, not a replacement. The four-module architecture (ag-behavior, ag-interpreter, ksc-behavior, ksc-interpreter) localizes changes to specific modules. All 131 existing tests must continue to pass at each stage.
