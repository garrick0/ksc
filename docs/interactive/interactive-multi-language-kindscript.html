01TypeScript Coupling Analysis

Before designing multi-language support, we need to understand exactly where KindScript is coupled to TypeScript and where it's already language-agnostic.

KindScript's current architecture has four distinct layers, each with different levels of TypeScript coupling. The key insight is that the property concepts are universal, but the extraction machinery is deeply TS-specific.

Layer-by-Layer Coupling

100%
Entirely delegated to ts.createProgram(). Zero custom scanning or parsing.
AST Conversion

convert.ts maps every ts.SyntaxKind to a KSNode variant. 60+ specific converters reference TS-specific node types.

The KSNode union type is a direct mirror of TypeScript's AST. Node types like KSTypeAliasDeclaration, KSAsExpression, KSArrowFunction are TypeScript-specific constructs with no equivalents in Python or Go.

Searches for type X = Kind<{...}> patterns. The algorithm is generic; the pattern matching is TS-specific.

Key takeaway: KindScript's core value proposition — "define behavioral properties, check them at compile time" — is inherently language-agnostic. The libs/ag infrastructure and property concepts are ready for multi-language use. But the IR layer presents a significant challenge.

02The IR Problem: KSNode is a TS Mirror

The current KSNode intermediate representation looks language-neutral at first glance, but it's actually a 1:1 copy of TypeScript's AST shape. This creates fundamental problems for multi-language support.

TS-Specific Constructs in KSNode

Many KSNode variants have no equivalent in other languages:

KSNode TypeTS-Specific?Problem for Other Languages
KSTypeAliasDeclarationYesPython/Go/Rust have no type aliases of this form. This is the Kind<{}> carrier.
KSAsExpressionYesTypeScript's x as T syntax. No equivalent in Python or Go.
KSArrowFunctionPartiallyPython has lambda, Go has func literals, Rust has closures — but all have different semantics.
KSEnumDeclarationPartiallyRust has enums (but algebraic). Python has enum.Enum. Go uses iota. All differ.
KSInterfaceDeclarationPartiallyGo has interfaces but they're structural. Python uses Protocol/ABC. Rust uses traits.
KSImportDeclarationUniversalAll languages have imports, but the syntax differs wildly.
KSFunctionDeclarationUniversalAll languages have function declarations.

The Mapping Challenge

Where does a Python decorated_definition go? What about Go's goroutine (go func())? Rust's lifetime annotations (&'a T)? The current KSNode has no home for these. You'd need to either:

  1. Force everything into the TS-shaped mold (lossy, awkward)
  2. Add language-specific node types to KSNode (combinatorial explosion)
  3. Design a genuinely universal IR from scratch
The core problem: KSNode's node names sound generic (FunctionDeclaration, ImportDeclaration) but their fields are TypeScript-shaped. KSFunctionDeclaration has asteriskToken (generators), typeParameters, and modifiers — concepts that don't map cleanly to def in Python or func in Go. Using this IR for other languages would mean either leaving fields empty/undefined or shoehorning foreign concepts into TS-shaped holes.

What We Actually Need

A truly universal IR for KindScript doesn't need to represent every syntactic construct in every language. It only needs to represent constructs that are relevant to property checking: imports, function boundaries, variable declarations, mutation operators, I/O calls, and module structure. Everything else can be a generic "opaque node" that property checking traverses but doesn't inspect.

03Existing Universal AST Formats

Others have tried solving the multi-language AST problem. Here's what exists, what works, and what we can learn from.

1 Semgrep's AST_generic Production ▼

The most proven universal AST in existence. Semgrep uses tree-sitter to parse 35+ languages, then maps each language's CST to a shared AST_generic representation. Core categories:

  • stmt — statements (ExprStmt, Block, DefStmt)
  • expr — expressions (Call, N (name), L (literal))
  • definition — definitions (FuncDef, VarDef, ClassDef)
  • name / tok — identifiers and tokens with location info

Each language has a dedicated mapper (e.g., python_to_generic.ml) that converts tree-sitter output to AST_generic. Language-specific constructs that don't fit the generic model are either mapped to the closest equivalent or wrapped in a language-specific extension node.

Relevance to KSC: This is exactly the pattern KSC needs. A slim generic IR focused on the constructs that matter for analysis, with per-language mappers from tree-sitter CSTs.

Limitation: Written in OCaml. No JS/TS library to import the AST_generic types directly. KSC would need to design its own IR inspired by this approach.

2 Tree-sitter Node Types Ubiquitous ▼

Critical finding: tree-sitter node types are NOT standardized across languages. Each grammar defines its own names independently:

3 Babelfish UAST Archived 2020 ▼

Source{d}'s Universal AST used a two-tier approach: raw language-specific nodes annotated with ~120 universal "roles" (Identifier, Expression, Statement, Function, Import, etc.), plus a higher-level "semantic UAST" that normalized common constructs into language-independent types.

Key design: Semantic node types included Identifier, QualifiedIdentifier, Import, Function, FunctionType, Argument, Block, Alias, Comment, String. Language-specific constructs (like Python decorators) remained as annotated native nodes.

Status: Project archived. But the design is instructive — especially the roles-based annotation approach where you don't try to normalize everything.

4 CodeQL, Joern, ast-grep Active ▼

CodeQL (GitHub): Uses language-specific AST representations per language. Cross-language sharing happens at the analysis level (DataFlow, TaintTracking libraries), not the AST level. Each language has its own .dbscheme.

Joern / CPG: Merges AST + control-flow + program-dependence into a unified "Code Property Graph" with standardized node types across languages. Supports C/C++, Java, Python, JS/TS, Kotlin, Rust. JVM-based (Scala).

ast-grep: A high-level TypeScript API over tree-sitter with pattern matching. Does not normalize node types — you write language-specific patterns. But it's the best JS/TS tool for tree-sitter-based code search.

5 SrcML, unist, CQFN Niche ▼

SrcML: XML-based AST for C/C++/C#/Java/Python. Uses shared tag names where constructs overlap (<function>, <if>, <import>). Lossless (preserves whitespace/comments). C++ CLI tool.

unist (unified.js): Not for programming languages, but its architecture — shared node interface (type, children, position), pluggable parsers, unified traversal — is an excellent design reference for a JS/TS-native universal AST. 30M monthly npm downloads.

CQFN UAST: Academic project classifying nodes as "green" (universal: conditionals, functions, loops) or "red" (language-specific: Java's synchronized, Python's slice syntax). Inactive since 2021, but the green/red taxonomy is a useful mental model.

The consensus across all these projects: Nobody has succeeded at creating a single universal AST that perfectly represents all languages. The approaches that work in production (Semgrep, CodeQL) either (a) build a slim generic IR covering only common constructs and accept some loss, or (b) keep language-specific ASTs and share analysis libraries. For KSC's purposes — where we only need to detect structural properties — option (a) is the right fit.

04Kind Annotations Across Languages

TypeScript uses phantom types for kind annotations. Other languages need different mechanisms.

Current: Phantom Type Approach

import type { Kind, PropertySet } from 'kindscript';

type Pure = Kind<{ noImports: true }>;

const add: Pure & ((a: number, b: number) => number) = (a, b) => a + b;

Strengths: Zero runtime cost, IDE autocomplete. Weakness: TS-only.

Decorator Annotations (Recommended)

from kindscript import kind

@kind("Pure", no_imports=True)
def add(a: int, b: int) -> int:
    return a + b

Decorators are idiomatic Python and easy to detect in any AST (tree-sitter: decorated_definition).

Comment Directive Annotations

//kindscript:kind Pure {noImports: true}
func Add(a, b int) int {
    return a + b
}

Follows Go conventions (//go:generate, //go:embed).

Attribute Macro Annotations

#[kind(Pure, no_imports = true)]
fn add(a: i32, b: i32) -> i32 {
    a + b
}

Rust's attribute macros are the natural annotation mechanism.

External Configuration (Any Language)

# kindscript.yaml - works for any language
kinds:
  Pure:
    noImports: true
rules:
  "src/core/**":
    kind: Pure

Language-agnostic. Coarser granularity (file/directory level).

Design decision: These approaches aren't mutually exclusive. Config-file definitions work universally, while each language frontend adds idiomatic in-code annotations as an overlay.

06Architecture Options

Four approaches, from reusing the existing TS-shaped IR to designing a new Semgrep-inspired generic AST.

A Shoehorn into Current KSNode ▼

Reuse the existing KSNode union as-is. Force Python/Go/Rust constructs into the TS-shaped node types. Map def add() to KSFunctionDeclaration, import os to KSImportDeclaration, etc.

Problems:

  • Many fields will be undefined (Python functions have no asteriskToken or typeParameters)
  • TS-specific nodes like KSAsExpression, KSTypeAliasDeclaration are unused by other languages
  • Python decorators, Go goroutines, Rust lifetimes have nowhere to live
  • Back-reference tsNode: ts.Node on every node makes no sense for non-TS code

Verdict: Not recommended. Quick to start but creates a fragile, confusing codebase.

B Language Adapter Pattern (No Shared IR) ▼

Skip the shared IR entirely. Define a LanguageAdapter interface where each language implements parsing, kind discovery, and property checking against its own native AST (tree-sitter nodes).

interface LanguageAdapter {
  parse(files: string[]): ParsedProgram;
  findKindAnnotations(program: ParsedProgram): KindAnnotation[];
  checkProperty(node: SyntaxNode, prop: string): boolean;
}

Pro: Lowest effort to add a new language. No IR design needed.

Con: Every property check is duplicated per language. Doesn't leverage libs/ag. Adapters diverge over time.

C Tree-sitter Queries Only (No IR) ▼

Use tree-sitter for all languages and write property checks as tree-sitter queries (S-expression patterns). No intermediate representation at all. Each property has per-language query files.

; noImports.py.scm
(import_statement) @violation
(import_from_statement) @violation

; noImports.ts.scm
(import_statement) @violation
(import_clause) @violation

; noImports.go.scm
(import_declaration) @violation

Pro: Tree-sitter queries are fast and declarative. Low per-language effort for simple properties.

Con: Queries can't express complex analysis (transitive purity, call graph). Doesn't use libs/ag. Still duplicates queries per language (different node names).

D New Generic IR (Semgrep-Inspired) Recommended ▼

Design a purpose-built universal IR inspired by Semgrep's AST_generic and CQFN's green/red taxonomy. The IR covers only constructs relevant to property checking. Each language gets a tree-sitter frontend that maps to this IR. TypeScript can optionally retain ts.createProgram() for type-level analysis.

The IR design (sketch):

// Universal node types — only what property checking needs
type KSGenericNode =
  | { kind: 'Program'; units: KSGenericNode[] }
  | { kind: 'Module'; name: string; body: KSGenericNode[] }
  | { kind: 'Import'; source: string; names: string[] }
  | { kind: 'FuncDef'; name: string; params: KSGenericNode[]; body: KSGenericNode }
  | { kind: 'ClassDef'; name: string; members: KSGenericNode[] }
  | { kind: 'VarDef'; name: string; mutable: boolean; init?: KSGenericNode }
  | { kind: 'Call'; target: KSGenericNode; args: KSGenericNode[] }
  | { kind: 'MemberAccess'; object: KSGenericNode; member: string }
  | { kind: 'Assignment'; target: KSGenericNode; value: KSGenericNode }
  | { kind: 'Name'; id: string }
  | { kind: 'Literal'; value: string }
  | { kind: 'Block'; stmts: KSGenericNode[] }
  | { kind: 'Annotation'; name: string; args: Record<string, unknown> }
  | { kind: 'Other'; lang: string; type: string; children: KSGenericNode[] };

The Other node is the "red" escape hatch — any language-specific construct that doesn't fit the generic model gets preserved with its original type name. Property checking traverses through Other nodes but doesn't pattern-match on them.

Pro: Clean IR designed for the problem at hand. Full libs/ag integration. Write each property check once. Each language adds only a mapper (~200-400 lines).

Con: Requires upfront IR design work. Must maintain mappers per language.

08Recommendation

A purpose-built generic IR with tree-sitter frontends, following the proven Semgrep pattern.

Recommended: Option D — New Generic IR (Semgrep-Inspired) + tree-sitter frontends

Design a slim, purpose-built KSGenericNode IR covering only property-relevant constructs (~15 node types). Use tree-sitter to parse all languages, with per-language mappers (~200-400 lines each) from tree-sitter CST to the generic IR. TypeScript optionally retains ts.createProgram() for type-level properties. All property checking and attribute grammar analysis operates on the generic IR.

Why Not Reuse KSNode?

The current KSNode has 60+ node types that are 1:1 copies of TypeScript's AST. Forcing Python's def into KSFunctionDeclaration (which has asteriskToken, typeParameters, and modifiers) would leave most fields undefined and create confusing, error-prone code. A generic IR designed from scratch for property checking is smaller (~15 types vs 60+), cleaner, and avoids this mismatch.

Phased Rollout

Phase 1: Design the Generic IR

Define KSGenericNode with ~15 universal node types: Program, Module, Import, FuncDef, ClassDef, VarDef, Call, MemberAccess, Assignment, Name, Literal, Block, Annotation, Other. Integrate with libs/ag's createTree().

Phase 2: TypeScript Mapper + Migrated Binder

Write the TS mapper (from existing convert.ts or from tree-sitter-typescript). Port the binder to operate on KSGenericNode. Verify existing tests still pass. The old KSNode can be kept alongside during migration.

Phase 3: Python Frontend

Write map-python.ts (~300 lines) using tree-sitter-python. Map function_definition → FuncDef, import_statement → Import, decorated_definition → Annotation + wrapped node. Implement noImports and noConsole checks on the generic IR.

Phase 4: Go, Rust, and Beyond

Add more language mappers. Each is a self-contained ~200-400 line file. All property checking is already written against the generic IR and works automatically for new languages.

function mapNode(node: SyntaxNode): KSGenericNode {
  switch (node.type) {
    case 'function_definition':
      return {
        kind: 'FuncDef',
        name: node.childForFieldName('name')?.text ?? '',
        params: mapChildren(node.childForFieldName('parameters')),
        body: mapNode(node.childForFieldName('body')!),
      };
    case 'import_statement':
    case 'import_from_statement':
      return {
        kind: 'Import',
        source: extractImportSource(node),
        names: extractImportNames(node),
      };
    case 'decorated_definition':
      return {
        kind: 'Annotation',
        name: extractDecoratorName(node),
        args: extractDecoratorArgs(node),
      };
    case 'call':
      return {
        kind: 'Call',
        target: mapNode(node.childForFieldName('function')!),
        args: mapChildren(node.childForFieldName('arguments')),
      };
    default:
      return {
        kind: 'Other', lang: 'python',
        type: node.type, children: mapChildren(node),
      };
  }
}

Q

Why can't the current KSNode IR be reused as-is for Python and Go?

A

KSNode is a 1:1 mirror of TypeScript's AST. Its node types have TS-specific fields (KSFunctionDeclaration has asteriskToken, typeParameters) and back-references (tsNode: ts.Node). Many nodes like KSAsExpression and KSTypeAliasDeclaration have no equivalents in other languages. Mapping Python/Go into this shape would leave most fields undefined.

Q

Are tree-sitter node type names standardized across languages? Give an example.

A

No. Each grammar defines its own names independently. For example, a function definition is called function_definition in Python, function_declaration in TypeScript/Go, and function_item in Rust. A mapping layer is required on top of tree-sitter to normalize these.

Q

What is Semgrep's AST_generic and why is it relevant to KindScript?

A

Semgrep's AST_generic is the most production-proven universal AST, used across 35+ languages. It uses tree-sitter for parsing, then per-language mappers convert CSTs to a shared generic representation with types like stmt, expr, FuncDef, Call. KSC should follow this same pattern: tree-sitter parsing + per-language mappers + slim generic IR.

Q

What is the purpose of the Other node type in the proposed generic IR?

A

Other is the "red" escape hatch for language-specific constructs that don't fit the generic model (Python's with statement, Go's go keyword, Rust's lifetime annotations). The construct is preserved with its original type name and children. Property checking traverses through Other nodes but doesn't pattern-match on them, so no information is lost.

Q

Why does the recommended approach use ~15 generic node types instead of KSNode's ~60?

A

KindScript only needs to detect constructs relevant to property checking: imports, function boundaries, variable declarations, mutation operators, I/O calls, and module structure. Everything else can be an Other node that is traversed but not inspected. A slim IR is easier to map to, easier to understand, and reduces the per-language mapper from 1000+ lines to ~200-400 lines.

Q

Which layer of KindScript's architecture has zero coupling to TypeScript and requires no changes for multi-language support?

A

The libs/ag attribute grammar library. Functions like createTree(), syn(), inh(), and circular() work with any tree of any node type. They have no imports from the typescript package and can be used directly with the new generic IR.

Q

What approach did Babelfish (UAST) take for language-specific constructs, and what can KSC learn from it?

A

Babelfish used a two-tier approach: raw language-specific AST nodes annotated with ~120 universal "roles" (Function, Import, Assignment, etc.), plus a higher-level "semantic UAST" normalizing common constructs. Language-specific constructs remained as annotated native nodes. KSC's Other node with lang and type fields follows a similar philosophy — preserving what you can't normalize.

Q

Why is checking noMutation harder in Python than in Rust, and how does the generic IR handle this?

A

Rust's ownership system makes mutation explicit (let mut, &mut), so the mapper can set VarDef.mutable = true directly. Python has no immutability primitives and uses mutating methods (.append()) that the Python mapper must recognize as Call nodes targeting known mutating methods. The generic IR's Call + MemberAccess nodes provide enough structure for this analysis, but the property checker needs a per-language list of mutating builtins.
