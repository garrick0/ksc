Key Restructuring: Spec as Data

The big change: checker equations moved from hand-written functions (checker.ts with 60+ exported eq_* functions) into a declarative AnalysisSpec in analysis/spec.ts. Each property is now a PropertyDecl with typed violation rules:

// analysis/spec.ts — one property declaration
{
  name: 'noConsole',
  annotationKey: 'noConsole',
  violationAttr: 'consoleViolation',
  violations: [
    violation({
      triggerKind: 'PropertyAccessExpression',
      predicate: (_ctx, node) =>
        node.expression.kind === 'Identifier' &&
        (node.expression as KSIdentifier).escapedText === 'console',
      message: (_ctx, node, def) =>
        `'console.${(node.name as KSIdentifier).escapedText}' violates ${def.name}`,
    }),
  ],
}

The compiler mechanically derives 17 attrs from 8 property declarations: 8 *Context (inh) + 8 *Violation (syn) + 1 allViolations (collection).

3. Building Layer 0: The Generic AG Evaluator

Layer 0 is the interpreter that the other layers specialize away. Building it validates that the specialization chain is correct: if eval_ag(ksc_spec, tree) produces the same output as generated/evaluator.ts, then the compilation is faithful.

Option A: Minimal
Option B: Spec-Driven
Option C: Indexed

Option A: Minimal Generic Interpreter (~100 lines)

A bare-minimum AG evaluator that takes the existing AnalysisSpec directly and evaluates it with Map-based caching. No code generation involved.

import type { AnalysisSpec, StructuralAttr, PropertyDecl } from '../compiler/types.js';
import type { KSNode } from '../generated/ast/index.js';
import { getChildFields } from '../generated/ast/index.js';

class GenericDNode {
  readonly node: KSNode;
  readonly parent: GenericDNode | undefined;
  readonly children: GenericDNode[];
  readonly isRoot: boolean;
  readonly fieldName: string | undefined;
  private cache = new Map<string, any>();
  private spec: AnalysisSpec;
  private counter: { value: number };

  attr(name: string): any {
    if (this.cache.has(name)) return this.cache.get(name);

    const structural = this.spec.structural.find(s => s.name === name);
    if (structural) {
      const val = this.evalStructural(structural);
      this.cache.set(name, val);
      return val;
    }

    // Context attr?
    const prop = this.spec.properties.find(
      p => name === p.name + 'Context'
    );
    if (prop) {
      const val = this.evalContext(prop);
      this.cache.set(name, val);
      return val;
    }

    // Violation attr?
    const vprop = this.spec.properties.find(
      p => p.violationAttr === name
    );
    if (vprop) {
      const val = this.evalViolation(vprop);
      this.cache.set(name, val);
      return val;
    }

    if (name === 'allViolations') {
      const val = this.evalAllViolations();
      this.cache.set(name, val);
      return val;
    }
    throw new Error(`Unknown attr: ${name}`);
  }

  // evalStructural, evalContext, evalViolation, evalAllViolations
  // ~60 more lines implementing the three AG directions generically
}

Advantages

  • Directly consumes AnalysisSpec — same spec object as compileAnalysis
  • Bit-for-bit output comparison validates the compiler
  • ~100 lines, no new abstractions
  • Can serve as a reference implementation for testing

Disadvantages

  • Map-based caching, linear scans — slow
  • No type safety on attribute values
  • Tightly coupled to KSC's AnalysisSpec shape
  • find() on every attr access — O(n) per access

Option B: Spec-Driven with Generic AGSpec (~150 lines)

Define a truly generic AGSpec type that isn't tied to KSC's domain. Build a converter from AnalysisSpec → AGSpec, then a generic evaluator over AGSpec.

// The generic AG specification — domain-independent
interface AGSpec {
  attrs: AGAttrDecl[];
}

interface AGAttrDecl {
  name: string;
  direction: 'syn' | 'inh' | 'collection';
  // Equation dispatch: kind string → equation function
  cases: Map<string, (ctx: GenericCtx, ...args: any[]) => any>;
  defaultEq?: (ctx: GenericCtx) => any;
  rootValue?: any;
  rootEq?: (ctx: GenericCtx) => any;
  inhEq?: (parentCtx: GenericCtx) => any | undefined;
  contributeEq?: (ctx: GenericCtx) => any;
  combineEq?: (acc: any, contrib: any) => any;
}

// The generic evaluator
function eval_ag(
  spec: AGSpec,
  root: any,
  getChildren: (node: any) => { node: any; field: string }[],
): GenericDNode { ... }

// KSC-specific: convert AnalysisSpec to AGSpec
function kscSpecToAGSpec(spec: AnalysisSpec): AGSpec {
  const attrs: AGAttrDecl[] = [];

  // Map structural attrs
  for (const s of spec.structural) {
    attrs.push({
      name: s.name,
      direction: s.direction,
      cases: new Map(Object.entries(s.equations.cases ?? {})),
      defaultEq: s.equations.default,
      rootValue: s.equations.rootValue,
      rootEq: s.equations.root,
      contributeEq: s.equations.contribute,
      combineEq: s.equations.combine,
    });
  }

  // Mechanically derive context+violation attrs from properties
  for (const prop of spec.properties) {
    // ... same derivation as compileAnalysis but interpreted at runtime
  }

  return { attrs };
}

Advantages

  • Truly generic — AGSpec could evaluate any AG, not just KSC's
  • Clean separation: kscSpecToAGSpec is the "semantic" bridge
  • Could be reused for other AG-based tools
  • Makes the Futamura structure maximally explicit

Disadvantages

  • More code (~150 lines for evaluator + converter)
  • Two layers of indirection to debug
  • AGSpec may drift from what compileAnalysis actually supports
  • Risk of building an unnecessary abstraction layer

Option C: Pre-indexed Interpreter (~120 lines)

Like Option A, but pre-computes lookup tables at construction time. Array-indexed caching instead of Map. Halfway between generic and compiled.

class IndexedDNode {
  private cache: (any | undefined)[];
  private dispatch: AttrDispatch[];

  constructor(node, parent, spec: CompiledSpec) {
    this.cache = new Array(spec.attrCount);
    this.dispatch = spec.dispatch;
  }

  attr(name: string): any {
    const idx = this.dispatch.ordinal(name); // O(1) lookup
    if (this.cache[idx] !== undefined)
      return this.cache[idx];
    return this.cache[idx] = this.dispatch[idx].evaluate(this);
  }
}

// Pre-computed at spec load time:
interface CompiledSpec {
  attrCount: number;
  dispatch: AttrDispatch[];  // [idx] → {evaluate, direction, ...}
  ordinal: (name: string) => number;  // name → index
}

Advantages

  • Much faster than Option A (array index vs Map lookup)
  • Still generic — works for any spec
  • Shows the "partial evaluation" structure: pre-indexing = specializing on the spec
  • Demonstrates what mix would do at a fine granularity

Disadvantages

  • Blurs the line between Layer 0 and Layer 1
  • Pre-indexing is partial evaluation — so this isn't purely "Layer 0"
  • More complex to implement correctly

Recommendation

Option A is the right choice. The purpose of Layer 0 is validation, not performance. You want to prove that eval_ag(ksc_spec, tree) === evaluate(tree). The simpler the interpreter, the more trustworthy the proof. Option A is also the most honest representation of "what does the compiled evaluator specialize away?" — every Map lookup, every linear scan, every untyped cache field is exactly the overhead that compileAnalysis eliminates.

Option C is interesting as a pedagogical halfway point, but it muddies the layers — pre-indexing is itself a form of partial evaluation.

4. What Is Mix? Building a Partial Evaluator

Mix is one function applied three ways

mix : (Program, StaticInput) → ResidualProgram

// It evaluates everything in Program that depends ONLY on StaticInput,
// and emits a residual program that handles the remaining dynamic input.

mix is the same function/algorithm in all three projections. The three projections are just feeding it different arguments:

ProjectionCallMeaningWhat You Get
1st mix(interp, program) Specialize interpreter on program Compiled program
2nd mix(mix, interp) Specialize mix on interpreter's structure Compiler
3rd mix(mix, mix) Specialize mix on itself Compiler generator

Is mix real or theoretical?

Theoretical
Real Systems
KSC's Hand-Mix

A fully automatic mix for a language as dynamic as TypeScript/JavaScript doesn't exist. The theoretical mix requires:

  • Binding-time analysis: classify every expression as "static" (known at specialization time) or "dynamic" (depends on runtime input)
  • Polyvariant specialization: generate different code for different static contexts
  • Termination guarantees: ensure specialization doesn't loop forever

For languages with dynamic typing, eval, computed property access, and closures, this is an open research problem. Real partial evaluators work on restricted subsets (RPython, Truffle DSLs).

Systems that implement mix (or close to it):

PyPy / RPython

Write interpreter in RPython (restricted Python). The meta-tracing JIT is mix: it observes the interpreter running a specific program and compiles hot traces. First projection in action.

GraalVM / Truffle

Write interpreter using Truffle API (Java). Graal's partial evaluator specializes the interpreter AST per-program. Explicitly designed around Futamura projections.

Similix (Scheme)

Academic partial evaluator for a Scheme subset. Can self-apply: mix(mix, interp) produces a compiler. Demonstrated all three projections in the 1990s.

Staging (MetaOCaml)

Programmer-directed partial evaluation via quasi-quotation. Not automatic, but gives manual control over what gets specialized when.

KSC's compileAnalysis is a hand-written mix

compiler/analysis.ts is not a generic partial evaluator — it's a hand-coded specializer that knows the specific structure of AG evaluators. It performs exactly the transformations that mix(eval_ag, spec) would perform, but by template-based code generation rather than automatic program analysis:

This is the pragmatic approach. You get the same output quality as automatic partial evaluation, with the cost being that compileAnalysis must be manually updated if the AG evaluator pattern changes (new attribute directions, different caching strategies, etc.).

Could we build a real mix for KSC?

Yes, but only for a restricted subset. The key insight: the AnalysisSpec is already in a form where binding-time analysis is trivial:

  • Static (known at compile time): attribute names, directions, dep edges, equation function references, trigger kinds, violation rule counts
  • Dynamic (depends on user program): this.node, this.node.kind, this.children, equation return values

Everything in the AnalysisSpec is static. Everything about the AST is dynamic. The boundary is sharp. This is why compileAnalysis works so cleanly — it can enumerate all static data and emit code that only handles dynamic data.

A "real" mix for KSC would be a function that takes the source code of eval_ag (Option A) + the AnalysisSpec and produces evaluator.ts. In practice, this would be a TypeScript-to-TypeScript partial evaluator, which is a research project, not a weekend task.

5. Validating All Four Layers

Layer 0: Generic AG Evaluator

Status: To Build

What: A function eval_ag(spec, tree) → result that interprets any AnalysisSpec on any KSNode tree using Map-based caching and generic dispatch.

Validation: For every test fixture, eval_ag(analysisSpec, tree) must produce identical definitions and diagnostics as evaluate(tree) from generated/evaluator.ts.

// Validation test
import { evaluate } from '../generated/evaluator.js';
import { eval_ag } from './generic-evaluator.js';
import { analysisSpec } from '../analysis/spec.js';

  const compiled = evaluate(tree);
  const interpreted = eval_ag(analysisSpec, tree);
  expect(interpreted.diagnostics).toEqual(compiled.diagnostics);
  expect(interpreted.definitions).toEqual(compiled.definitions);
});

Layer 1: Compiled Evaluator

Status: Exists — generated/evaluator.ts

What: mix(eval_ag, analysisSpec) — the AnalysisSpec baked into typed fields, switch dispatch, and direct function calls.

Validation: 179 passing tests across 13 test files. Cross-validated against TS checker in integration tests. This is the production artifact.

Futamura check: evaluate(tree) should equal eval_ag(analysisSpec, tree). Building Layer 0 lets us verify this directly.

Layer 2: Analysis Compiler

Status: Exists — compiler/analysis.ts

What: mix(mix, eval_ag) — a compiler from analysis specs to specialized evaluators.

Validation: If you change analysis/spec.ts (add a property, modify an equation) and re-run compileAnalysis(analysisSpec), it produces a new evaluator.ts that correctly handles the changed spec.

Futamura check: compileAnalysis is parameterized by spec structure, not by specific attribute names. It derives context/violation attrs mechanically from PropertyDecl[]. Adding a 9th property (just append to the array) produces a 24-attribute evaluator with no changes to the compiler.

compileAnalysis doesn't know about "noConsole" or "immutable". It processes:

  • spec.structural[] → emit typed cache fields + methods with kind dispatch
  • spec.properties[] → derive *Context (inh) + *Violation (syn) + allViolations (collection)
  • Dep edges → derived from deps[] arrays and structural relationships

The specific property names, trigger kinds, and violation predicates are all data flowing through — the compiler just knows the shapes (syn/inh/collection, cases/default/universal, contribute/combine). This is exactly what mix(mix, eval_ag) would produce: a function that knows "how AG evaluators work" and can specialize them for any specific spec.

Layer 3: Compiler Generator

Status: Not Built

What: mix(mix, mix) — given the structure of an AG evaluator (cache strategy, dispatch strategy, attribute kinds), produce a compiler like compileAnalysis.

Validation: Feed it "typed fields, switch dispatch, syn/inh/collection, cycle detection" and get back a function equivalent to compileAnalysis.

  cycleDetection: boolean;
  treeBuilder: 'schema-driven' | 'generic';
  inhPropagation: 'parent-chain' | 'copy-down';
}

function generateAGCompiler(
  shape: EvaluatorShape
): (spec: AnalysisSpec) => CompiledAnalyzer {
  // Returns a function equivalent to compileAnalysis,
  // but configured for the given evaluator architecture
}

This would let you explore alternative evaluator designs systematically. For example, generateAGCompiler({ cacheStrategy: 'array', dispatchStrategy: 'vtable' }) would produce a compiler that emits evaluators using array caching and virtual dispatch instead of typed fields and switch statements.

In practice, this is rarely worth building unless you're designing a multi-target AG compiler framework (like JastAdd's code generation for Java/C++/etc.). KSC targets TypeScript exclusively, so the evaluator shape is fixed.

Building a configurable compiler generator would be over-engineering unless you plan to target multiple languages or radically different evaluator architectures.

The Architecture Already Validates Layers 1–2

The restructuring has already made the Futamura structure concrete:

  • Layer 1 (generated/evaluator.ts): validated by 179 passing tests
  • Layer 2 (compiler/analysis.ts): validated by the fact that it produces Layer 1 output that passes all tests, and by compiler/validate.ts which cross-checks trigger kinds against the grammar
  • The two-functor split is the Futamura structure: compileGrammar = first projection on grammar, compileAnalysis = first projection on analysis spec

Building Layer 0 completes the picture by providing the reference interpreter that the projections specialize away. The key validation it adds: proof that the compiled evaluator is a faithful specialization, not just "a thing that produces the right output for tested inputs" but "the same function as the generic evaluator, just faster."
