Removing config.ts

Options for eliminating the config module and simplifying the type pipeline

The Problem

1. RuleSet and PropertySpec are duplicates

config.ts:RuleSet and types.ts:PropertySpec define the exact same fields. The binder bridges them with an unsafe cast:

declaredProperties: (entry.rules ?? {}) as PropertySpec

If you add a rule to one and forget the other, there's no compiler error — just a silent mismatch.

2. defineConfig is a no-op

export function defineConfig(config: KindScriptConfig): KindScriptConfig {
  return config;  // literally identity
}

It only exists for IDE autocompletion. TypeScript's satisfies operator (TS 4.9+) does this without a function.

3. Unnecessary module boundary

config.ts is 72 lines. 5 files import from it. The types are core to the compiler — they belong with the other compiler types in types.ts.

How TypeScript Does It

Aspect TypeScript Compiler KindScript (current)
Config format JSON (tsconfig.json) TypeScript code (kindscript.config.ts)
Config types location In main types.ts alongside all other types Separate config.ts module
Duplicate types? No — CompilerOptions is used everywhere Yes — RuleSet + PropertySpec
Config helper None needed (JSON) defineConfig() identity function
Type guard Inline checks or utilities.ts isCompositeEntry() in config.ts

Key insight: TypeScript keeps one CompilerOptions interface that flows from tsconfig.json parsing all the way through to the checker. There is no "config CompilerOptions" vs "internal CompilerOptions".

The Options

Option A: Merge into types.ts, delete config.ts Recommended

Kill RuleSet, keep PropertySpec as the single source of truth. Move TargetEntry, CompositeEntry, KindScriptConfig into types.ts. Move isCompositeEntry into binder.ts (its only consumer). Drop defineConfig.

types.ts changes:
// Config entry types (moved from config.ts)
export interface TargetEntry {
  readonly path: string;
  readonly rules?: PropertySpec;     // was RuleSet — same type, no cast needed
}

export interface CompositeEntry {
  readonly members: Record<string, TargetEntry>;
  readonly rules?: PropertySpec;
}

export type ConfigEntry = TargetEntry | CompositeEntry;
export type KindScriptConfig = Record<string, ConfigEntry>;
binder.ts changes:
- import type { KindScriptConfig } from './config.js';
- import { isCompositeEntry } from './config.js';
- declaredProperties: (entry.rules ?? {}) as PropertySpec,
+ import type { KindScriptConfig, ConfigEntry, CompositeEntry } from './types.js';
+ function isCompositeEntry(e: ConfigEntry): e is CompositeEntry { return 'members' in e; }
+ declaredProperties: entry.rules ?? {},
User config files:
- import { defineConfig } from 'kindscript';
- export default defineConfig({ ... });
+ import type { KindScriptConfig } from 'kindscript';
+ export default { ... } satisfies KindScriptConfig;
Pros
  • Eliminates the RuleSet/PropertySpec duplication entirely
  • Removes the unsafe as PropertySpec cast
  • One fewer file in src/
  • Follows TypeScript's own pattern
  • satisfies is standard TS — no custom helper
Cons
  • types.ts grows ~20 lines
  • Requires TS 4.9+ for satisfies (already required by the project)
  • Breaking change for users importing defineConfig
src/
  config.ts
  types.ts ← gains config types
  binder.ts ← imports from types, gets isCompositeEntry
  index.ts ← re-exports from types instead of config
  checker.ts, linker.ts, program.ts ← untouched

Option B: JSON config like TypeScript TS Pattern

Switch to kindscript.json (or a "kindscript" key in tsconfig.json). Parse it in the CLI like TypeScript parses tsconfig.json.

// kindscript.json
{
  "domain": { "path": "./src/domain", "rules": { "pure": true, "noIO": true } },
  "app": {
    "members": {
      "domain": { "path": "./src/domain" },
      "infra":  { "path": "./src/infra" }
    },
    "rules": { "noDependency": [["domain", "infra"]] }
  }
}
Pros
  • Closest to TypeScript's actual pattern
  • No dynamic import / tsx/ts-node dependency for config loading
  • JSON Schema gives IDE autocompletion without any helper function
  • Can be embedded in tsconfig.json (like Jest, ESLint flat configs started in JSON)
Cons
  • Loses programmatic config (computed paths, env-based rules)
  • No comments in JSON (JSONC would help)
  • Need to write/maintain a JSON Schema for autocompletion
  • Bigger migration — changes the entire config surface

Option C: Merge types + keep defineConfig as re-export Conservative

Same as Option A but keep defineConfig as a one-liner in index.ts for backwards compatibility. Deprecate it.

// index.ts
/** @deprecated Use `satisfies KindScriptConfig` instead. */
export const defineConfig = (c: KindScriptConfig): KindScriptConfig => c;
Pros
  • Non-breaking — existing configs keep working
  • Still eliminates the duplicate types and config.ts file
  • Gradual migration path
Cons
  • Keeps dead code around (the identity function)
  • Users won't migrate unless forced

Option D: tsconfig.json plugin field Ambitious

Piggyback on tsconfig.json's plugins array, like typescript-eslint and ts-sql-plugin do:

// tsconfig.json
{
  "compilerOptions": {
    "plugins": [{
      "name": "kindscript",
      "targets": {
        "domain": { "path": "./src/domain", "rules": { "pure": true } }
      }
    }]
  }
}
Pros
  • Zero extra config files — lives in tsconfig.json
  • Enables Language Service Plugin for real-time IDE feedback
  • Established TypeScript pattern
Cons
  • Plugin field is typed as any — no autocompletion without a schema
  • Mixes concerns (TS compiler options + KS architectural rules)
  • Requires building a TS Language Service Plugin (significant work)
  • Locks you into TypeScript's config parsing

Recommendation: Option A (or C if you want a soft landing)

Option A is the right call. It's the smallest change that fixes all three problems. It follows TypeScript's own pattern of keeping one canonical type that flows through the whole pipeline. The as PropertySpec cast disappears because rules is already typed as PropertySpec.

defineConfig can go — satisfies is the modern TS way to get type-safe object literals. If backwards compat matters right now, Option C keeps it as a deprecated re-export.

Option B (JSON) is worth considering later if you want to support Language Service Plugin integration or eliminate the tsx/ts-node runtime dependency. But it's orthogonal to the type duplication problem.
